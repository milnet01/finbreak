# FIBR-0138 — Expandable dashboard drill-down (Income / Spending / Transfers → categories → merchant → transactions)

> **Status:** ✍️ **DRAFT — pending `/cold-eyes`** (project cap **7** loops,
> correctness-critical override, `finbreak/CLAUDE.md`). Not yet cleared for code.
> **Phase:** enhancement of the FIBR-0012 dashboard. **Kind:** enhancement.
> **Source:** roadmap FIBR-0138 (designed in-session 2026-07-14; three
> user-approved brainstorm decisions — D1 an **expanding tree** below the existing
> donut + trend charts, not a click-to-drill donut; D2 Spending/Income drill the
> **category tree** → leaf → **merchant** (×count) → the individual transactions;
> D3 merchant = **smart cleanup** of the free-text `description`; D4 Transfers drill
> by **account pair** (from→to, ×count) → the individual moves). This spec numbers
> its own **D1–D9** independently of that bullet's D1–D4.

> All new work reads/aggregates metadata inside the already-encrypted vault,
> reachable only while unlocked; the merchant cleanup runs in-memory on a
> `description` already held. The FIBR-0054 `_network_offenders` source scan stays
> green (no new `urllib` importer).
> **Roadmap:** [ROADMAP.md → FIBR-0138](../../ROADMAP.md).
> **Governed by:** `docs/standards/{coding,naming,testing,commits}.md` (no new pip
> dependency — stdlib `re` only — so `dependencies.md` is n/a) (`naming.md § 4` the

> [ADR-0008](../decisions/0008-qtcharts-for-reporting.md) (unchanged — the tree is
> a `QTreeWidget`, not a chart; ADR-0008 still governs the donut + trend it sits
> below).
> Builds directly on [FIBR-0012](FIBR-0012.md) (the `ReportingService` +
> `HomeView` dashboard this extends — its `resolve_period`, `_excluded`,
> `ReportingRepository.rows_in_range`, `ReportPrefs`, the period + account
> selectors, and the `Summary` the top-line totals must equal), [FIBR-0011](FIBR-0011.md)

> [FIBR-0007](FIBR-0007.md) (the signed-`amount_minor` debit-negative /
> credit-positive convention income/spending rests on).

**Theme:** the Home dashboard's headline figures become **openable**. Today you see
income, spending, and net as tiles plus a donut and a trend. This adds an **expanding
tree** below those charts with three openable rows — **Income**, **Spending**, and
**Transfers** (the *net* figure is not drilled; *transfers*, which never count toward a
tile, get their own openable total). Click
**Spending** and it opens into your categories; open a category and it breaks into
its sub-categories down to a single category; open that and it groups the
transactions **by shop** with a count (e.g. *"Woolworths ×3"*); open a shop and you
see each individual purchase (date + amount). **Income** opens the same way;
**Transfers** opens by account pair (*"Current → Savings ×2"*) into the individual
moves. Every number in the tree is summed from the real stored amounts — the shop
grouping only decides *which line a transaction sits under*, never *how much* it is.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| **INV-1** | **Money-safety (the headline).** Every node's `amount` is summed from integer `amount_minor` and every `count` is a real row count; the merchant cleanup (D3) and the category/pair grouping only ever decide **which node a transaction is placed under** — they can **never** change a total, a subtotal, or a count. The **Income** and **Spending** top-line totals **equal** their dashboard tiles for the same `(prefs, account_ids)`: Income node total = `Summary.income`, Spending node total = `Summary.expenditure`. (The **Transfers** node has no tile — transfers are excluded from every FIBR-0012 tile, INV-2 — so it is not part of this tile equality; its total is the sum of the drilled transfers.) Every category subtree sums to its parent, which sums to Spending/Income. | FIBR-0012 INV-4/INV-13 |
| **INV-2** | **Transfers excluded from Income / Spending; shown only under Transfers.** The Income and Spending branches drop every id in `TransferDetectionService.confirmed_transfer_txn_ids()` (the same `_excluded()` set the tiles use), so a confirmed self-transfer never appears under a category or a merchant. It appears **only** under the Transfers branch. An **un**confirmed pair is still ordinary income/spending (the exclusion keys on *confirmed*). | FIBR-0011; FIBR-0012 INV-1 |
| **INV-3** | **The selectors drive the tree; live on view.** The tree is built for the **same** `(ReportPrefs, account_ids)` the tiles/charts use — `resolve_period` for the date window, the `account_ids: frozenset[int] \| None` set for the account scope (`None` ⇒ all). It re-reads the vault on every `refresh()` (no cached tree), so an import / edit / transfer-confirm elsewhere is reflected on next view. | FIBR-0012 INV-3/INV-10 |
| **INV-4** | **Category branch follows the real tree, grouped by id; every row is placed.** Under Income (positive rows) and Spending (negative rows), the branch's top-level children are the distinct **top-of-chain** categories of the sign-filtered rows — climbing each row's `category_id` up its `parent_id` chain to the category whose parent is a **root** (D4a). Because grouping is by that climbed category, **no row is ever dropped**, so the branch total always equals its tile (INV-1) — including a mis-set row whose category sits under the *other* root. Each category node's `amount`/`count` is the sum of its **own** directly-assigned rows plus all descendants' (post-order); a node's children are its non-empty child-category nodes **and** the merchant nodes for its own directly-assigned rows (so a category that has both sub-categories and its own rows shows both). Grouping is by **`category_id`**, so two same-named leaves under different parents stay distinct (FIBR-0006 sibling-unique names). Rows with `category_id is None` form **one** synthetic **Uncategorised** node (identified by the `None` id, never a display-name string), a sibling of the top-level categories, drilling straight to merchants. A category node with **zero** matching rows in the period is **omitted** (only non-empty branches render). | FIBR-0006; FIBR-0012 INV-5 |
| **INV-5** | **Merchant grouping is display-only and total.** At **any** category holding directly-assigned rows (a leaf, or a parent that also has its own rows — D4a), those rows are grouped by `merchant_name(description)` — a **pure, total** cleanup that never raises for any `str`; each merchant node carries `×count` and expands to the individual transactions (date + amount magnitude). The cleanup is **fuzzy by design** and refined over releases; because of INV-1 a mis-grouped shop is only cosmetic — the leaf's total is unchanged. A description that cleans to empty (e.g. all-digits) falls back to its trimmed raw text, never an empty label (stored descriptions are non-empty — `parse_transaction`, D3). | roadmap D3; coding.md § 1.3 (reuses `normalise_text`) |
| **INV-6** | **Transfers drill by account pair.** The Transfers node groups the **confirmed** transfers whose window/scope match (a transfer belongs to the period when its **debit** leg's `occurred_on` is in `[start, end]`, and to the account scope when **either** leg's `account_id` is in `account_ids`, or `account_ids is None`) by their **(from_account → to_account)** name pair, with `×count`; each pair expands to the individual moves (the debit leg's date + the positive `display_amount`). A transfer is attributed **wholly to its debit-leg month**: one whose debit leg falls outside the period is absent from the Transfers node even if its credit leg falls inside (it is also already excluded from Income/Spending by id, INV-2), so it is not double-counted across two periods — it belongs to exactly the period holding its debit leg. Transfers have **no categories**; the Transfers total is the sum of those `display_amount`s (never mixed into Income/Spending, INV-2). | FIBR-0011 `confirmed_transfers()` |
| **INV-7** | **Biggest-amount-first at every level.** Every sibling list — top nodes' children, category children, merchants, account pairs, and the individual-transaction leaves — is ordered by **descending magnitude**, then the display **label ascending**, then a **final total key** that is a **uniform string** built per node so a mixed sibling list (a node's children can be both category and merchant nodes, D4a) never compares an `int` against a `str` (a `TypeError`): `f"cat:{id}"` for a category, `"cat:none"` for Uncategorised, `f"mer:{grouping_key}"` for a merchant, `f"txn:{id}"` for a transaction (move) leaf's `pair_id`/`txn_id`, and — for an aggregated **account-pair** node (which spans several `pair_id`s) — `f"pair:{min pair_id in the group}"`. This makes the order **total** even when magnitude *and* label tie — two same-named leaves under different parents, or two same-date same-amount transactions (the read has no `ORDER BY`, so a lesser key would be unstable; the donut's `(magnitude desc, category_id asc)` is the totality precedent). The tie-break label is the bare `node.label`, **not** the widget's rendered `"{label} ×{count}"`. The sort runs at **build time** (the service still has the ids/keys); `DrillNode` deliberately carries **no** id field (it is a display model), so sorting precedes node construction. The three top nodes are always in the fixed order **Income · Spending · Transfers** (not amount-sorted — they are the fixed headings). | roadmap ("matches the donut") |
| **INV-8** | **Money stays exact.** Every `amount`/subtotal is computed on integer `amount_minor` (transfers on the already-scaled `display_amount` `Decimal` from `confirmed_transfers()`); the only crossing to a display string is the shared `_format_amount`. No aggregation touches `float`. | FIBR-0012 INV-13; coding.md § 5 |
| **INV-9** | **No schema change; no network; i18n.** No migration (read-only aggregation over existing tables); `LATEST_SCHEMA_VERSION` unchanged. The `_network_offenders` AST scan in `test_INV8_no_network_imports_under_src` (the FIBR-0054 INV-12 network-confinement gate) stays green — the new modules import no network. Every fixed user string is `tr()`-wrapped: the four `DrillLabels` strings (the three headings + "Uncategorised") travel in from the `QObject` `HomeView` (the non-`QObject` `ReportingService` cannot `tr()`, D2), while the column headers `tr("Name")`/`tr("Amount")` (D1) and the `tr("{label} ×{count}")` count template (D7) are `tr()`-ed **directly** in `HomeView`; merchant / category / account names and dates are **data**, not `tr()` strings; every widget sits in a layout manager. | coding.md § 5.2; FIBR-0054 INV-12 |

## Context

Verified against source 2026-07-14 (see **Verified code basis**):

- **The dashboard aggregator (`services/reporting.py`).** `ReportingService(vault)`
  (`:110`) exposes `summary` (`:139`), `spending_by_category` (`:168`),
  `monthly_trend` (`:219`), `base_currency` (`:121`), `transaction_count` (`:129`).
  Each of the three aggregations takes `(prefs: ReportPrefs, account_ids:
  frozenset[int] | None, today: date | None = None)`, calls `resolve_period(prefs,
  today)` (`:69`, pure) for the `[start, end]` window, builds the exclusion set once
  via `_excluded()` (`:135` → `confirmed_transfer_txn_ids()`), reads rows via
  `ReportingRepository(self._conn).rows_in_range(...)`, and scales integers to a
  display `Decimal` with `read_minor_unit_exponent` + `to_display_decimal`
  (`services/transactions.py:26/79`). This is the shape `drill_down` mirrors so its
  totals **provably equal** the tiles' (INV-1). **Signature note:** the account
  parameter is now the **`account_ids: frozenset[int] | None`** *set* — FIBR-0013 D4
  widened it from FIBR-0012's original singular `account_id: int | None`, so the FIBR-0012
  spec's prose lags the current code; this spec describes the **current** (post-0013)
  signature. `summary`'s sign split is `if amount_minor > 0 … else …`; since
  `parse_transaction` **rejects a zero amount** (`transactions.py:74`), no row is ever
  `amount_minor == 0`, so the drill's strict `> 0` / `< 0` split (D4) is **total** (it
  places every row `summary` counts).

- **The transfer feed (`services/transfer_detection.py`).** `confirmed_transfers()

  `debit`/`credit` `Transaction`s, the positive `display_amount` (`Decimal`), and the
  `from_account`/`to_account` names (`models.py:284`). `confirmed_transfer_txn_ids()

## Design decisions

- **D1 — A `QTreeWidget` below the charts (roadmap D1).** The drill is an
  **expanding tree**, not a click-to-drill donut. In `_build_dashboard`, a
  `QTreeWidget` (`objectName "dashboard_drilldown"`) is added to the dashboard
  `QVBoxLayout` **after** `dashboard_trend_chart`. Two columns: **Name** and
  **Amount** (`setHeaderLabels([tr("Name"), tr("Amount")])` — "Name" covers every node
  kind: category, merchant, account-pair, and transaction-date). Because
  the charts + tree together make a tall page, the dashboard page's content is wrapped
  in a **`QScrollArea`** (`setWidgetResizable(True)`) so it stays usable on small
  windows — a pure layout change; the existing chart/tile object names are unchanged
  (the qtbot lookups still resolve through the scroll area's child widget). The tree's
  own column **sorting is disabled** (`setSortingEnabled(False)`): the service returns
  every level **pre-sorted** (INV-7), so the widget inserts in order — no
  `SortableItem` needed (unlike the flat Transactions table, this tree is
  display-only, never user-reordered).

- **D2 — Read model: one recursive `DrillNode` (`models.py`).** A single frozen
  dataclass renders every level uniformly (coding.md § 1.1 — no per-level type
  zoo):

  ```python
  @dataclass(frozen=True)
  class DrillNode:
      label: str              # display text: category/merchant/pair name, or a txn date
      amount: Decimal         # summed positive magnitude (display Decimal)
      count: int              # underlying transactions; ≥ 0 (0 only for an empty top branch)
      children: tuple[DrillNode, ...]   # () at the individual-transaction leaf
  ```

  `children` is a `tuple` (frozen-friendly, immutable). The count is carried as
  **data** — the widget decides how to show it (D7 appends `×N` for merchant/pair
  nodes with `count > 1`), so the model never bakes presentation into `label`. The
  node kinds and their field mapping: a **top** node (`label` = `labels.income` /
  `labels.spending` / `labels.transfers`, always present, `count` may be **0** for an
  empty branch); a **category** node (`label` = the stored category name, data); a
  **merchant** node (`label` = the cleaned shop name, D3, data; `count` = the group
  size); an **account-pair** node (`label` = `f"{from} → {to}"`, data; `count` = pair
  size); and an **individual-transaction** leaf (`label` = the row's `occurred_on`
  date, `amount` = that row's display magnitude, `count` = 1, `children` = `()`).

  Because `ReportingService` is **not** a `QObject`, it cannot `tr()` (INV-9). The
  five fixed user strings the tree needs are therefore **passed in** by the caller as a
  `DrillLabels` value — the exact `build_donut_chart(spending, tr("Uncategorised"),
  tr("Other"), …)` precedent (`ui/home.py:337-343`):

  ```python
  @dataclass(frozen=True)
  class DrillLabels:
      income: str           # tr("Income")        — the Income top node
      spending: str         # tr("Spending")      — the Spending top node
      transfers: str        # tr("Transfers")     — the Transfers top node
      uncategorised: str    # tr("Uncategorised") — the None-category node label
  ```

  Every other label is **data** (a category name, a shop name, an account name, a
  date), so the service produces a fully-labelled tree while emitting **no**
  untranslated user string. (There is no "no description" label: `parse_transaction`
  rejects a blank `description` (`transactions.py:49`), so a stored row's description is
  never empty and the merchant cleanup never yields a blank group — D3.)

- **D3 — Merchant cleanup (`text.py`), pure + total (INV-5).** A new
  `merchant_name(description: str) -> str` produces a best-guess shop name and
  **never raises** for any `str`. Steps (documented as **fuzzy, refined per release** —
  like the FIBR-0139 category library, it improves with zero call-site change):
  1. Strip; if empty → return `""` (defensive totality — a **stored** `description` is
     never blank, since `parse_transaction` rejects it (`transactions.py:49`), so this
     branch is exercised only by the direct unit test of `merchant_name`, never the
     pipeline).
  2. Strip a small documented set of leading noise **prefixes**, matched as whole
     words/phrases at the **start** of the string (case-insensitive), **longest-match
     first** (so a future overlapping prefix stays deterministic), **not** per-token,
     repeating while any remains: `POS`, `PURCHASE`, `CARD`, `PMT`, `PAYMENT`, `EFT`,
     `DEBIT ORDER`, `DR` — so the two-word `DEBIT ORDER` matches as a phrase, and
     `POS CARD WOOLWORTHS` strips both leading prefixes.
  3. Remove digit-heavy reference tokens: any run of **≥ 3 digits** (card fragments,
     dates, ref numbers) and any token that is **majority digits**.
  4. Strip punctuation from the token edges; collapse internal whitespace.
  5. If the result is non-empty → **title-case** it for display; else fall back to the
     **trimmed raw** description (never lose the row's identity).
  Company suffixes (`PTY`, `LTD`, `INC`, `CC`) are **not** stripped in this first
  version (they rarely split a real shop and stripping them risks merging distinct
  merchants); noted as a future refinement, like the FIBR-0139 library.
  The grouping **key** is `normalise_text(merchant_name(desc))` (reusing `text.py`'s
  primitive, coding.md § 1.3) so `"WOOLWORTHS 12345"` and `"Woolworths"` collapse to
  one node; the node's **display `label`** is the **lexicographically smallest**
  `merchant_name(desc)` among the group's rows — a deterministic choice independent of
  row order (the read has no `ORDER BY`), so the label is stable and testable. The rules
  are stdlib `re` only — **no new dependency**. Cleanup runs **only** at the
  leaf-merchant level, on rows already in hand; it is O(rows) with a couple of cheap
  regexes — no perf budget beyond FIBR-0012's applies (coding.md § 1.1, no index/cache).

- **D4 — `drill_down` on `ReportingService` (`services/reporting.py`).** A new method
  building the whole tree in one pass, reusing every FIBR-0012 primitive so its
  numbers **cannot drift** from the tiles (INV-1):

  ```python
  def drill_down(
      self,
      prefs: ReportPrefs,
      account_ids: frozenset[int] | None,
      today: date | None = None,
      *,
      labels: DrillLabels,   # tr()-ed fixed strings, passed in (D2/INV-9)
  ) -> list[DrillNode]:   # exactly 3: [Income, Spending, Transfers], in that order
  ```

  Steps: `resolve_period(prefs, today)` → `[start, end]`; `_excluded()` once; read the
  **richer** rows via `drill_rows_in_range` (D5); split by sign into income
  (`amount_minor > 0`) and spending (`amount_minor < 0`) after dropping excluded ids
  (INV-2). **Sign of the accumulated minor (load-bearing for INV-1):** every node sums
  a **positive magnitude** — the Income branch accumulates `amount_minor` (already
  positive), the **Spending** branch accumulates `-amount_minor` (negating, exactly as
  `summary` does at `reporting.py:161`), so `to_display_decimal(sum_of_magnitude,
  exponent)` yields the **positive** Decimal INV-1 equates to `Summary.expenditure` /
  `Summary.income`. Build the **Income** and **Spending** category subtrees (D4a); build
  the **Transfers** subtree from `confirmed_transfers()` filtered to the window/scope
  (D4b). The top three nodes are **always present** — an empty branch is a
  `DrillNode(labels.income/…, amount=0, count=0, children=())` (the one place `count` is
  0, D2), so the tree always shows the three headings, INV-7.

  - **D4a — Category subtree (INV-4), ONE algorithm.** Built **per sign branch**
    (Income = positive rows, Spending = negative rows), each with its **own**
    row buckets, so the two trees never share a total. Setup from
    `CategoryRepository.list_all()`:
    - `roots` = the categories with `parent_id is None` (the two Type roots; their
      `kind` — `CategoryKind.INCOME` / `CategoryKind.EXPENDITURE`, the lowercase
      `"income"`/`"expenditure"` tokens, `models.py` — is **not** used to place rows;
      it is decorative here, since the sign bucket already decides the branch);
    - `children_by_parent: dict[int, list[Category]]` from every non-root category;
    - `own_rows: dict[int | None, list[Row]]` — this branch's sign-filtered rows
      bucketed by their **exact** `category_id` (`None` ⇒ the Uncategorised bucket).
    - **`top_of_chain(category_id)`** = climb `parent_id` until the parent is a root;
      that ancestor (a **direct child of a root**) is the top-of-chain. (In today's
      depth-1 tree every leaf's parent is already a root, so top-of-chain is the leaf
      itself; the climb makes the algorithm correct at any depth.) A live
      `transactions.category_id` is **never a root** — transactions bind only to
      non-root categories (the rule engine `_require_leaf`s, the picker offers leaves
      only, the library binds by leaf name; the two Type roots are never assignable) — so
      the climb always reaches a child-of-root. **Defensively total:** if the node handed
      to `top_of_chain` is *itself* a root, it is its own top-of-chain (the loop
      terminates on bad data, never spins).
    - A recursive **`_category_node(category)`** (a closure over this branch's
      `own_rows` + `children_by_parent` maps — not a standalone method, so it needs no
      map parameters) returns a `DrillNode` whose `amount`/`count` sum this category's
      **own** bucket **plus** every descendant's (post-order), and whose `children` are
      **both**: (i) the non-empty `_category_node(child)` for each child category, **and**
      (ii) the **merchant** nodes (D3) built from this category's **own** bucket — so a
      category with *both* sub-categories *and* directly-assigned rows shows both, and a
      category's own rows are **never invisible** (INV-1). A pure leaf yields only merchant
      children; a pure branch yields only child nodes. Every sibling list is sorted by the
      INV-7 total key — `(magnitude desc, label asc, uniform-string-key asc)` — computed at
      build time before the `DrillNode`s are constructed (`DrillNode` carries no id/key).
    - The **branch node's** top-level `children` are the `_category_node(...)` for each
      **distinct `top_of_chain`** of this branch's rows (one node per top-of-chain
      category that has ≥ 1 matching row), **plus** an **Uncategorised** node (`label` =
      `labels.uncategorised`) from the `None` bucket when non-empty (its children are
      merchants directly — Uncategorised has no sub-tree). This single "group by
      top-of-chain" rule places **every** row,
      so the branch total always equals its tile (INV-1) — there is **no** separate
      "children directly under the root" pass to disagree with it.
    - **Mis-set edge (rare, handled not dropped):** a negative row tagged to a category
      whose top-of-chain sits under the *Income* root still surfaces under **Spending**
      (its sign branch) via that top-of-chain node — so no spending row is ever lost from
      the Spending total. The realistic case (a category's rows match its root's sign) is
      the common path; the edge just guarantees completeness.
    - **A category may appear under both branches (intended).** A category id holding
      *both* positive and negative rows produces a node under Income (its positive rows +
      total) *and* a separate node under Spending (its negative rows + total); each node
      carries only its **own sign-bucket** rows and total, so neither double-counts.
  - **D4b — Transfers subtree (INV-6).** From `TransferDetectionService(self._vault)
    .confirmed_transfers()`: keep a `ConfirmedTransfer` iff its **debit** leg's
    `occurred_on` is in `[start, end]` **and** (`account_ids is None` **or** either
    leg's `account_id ∈ account_ids`). Group the kept transfers by the
    `(from_account, to_account)` name pair; each pair node's `amount` = Σ
    `display_amount`, `count` = number of transfers, `label` = `f"{from} → {to}"`; its
    `children` are the individual moves (`label` = the debit leg's `occurred_on`,
    `amount` = that transfer's `display_amount`, `count` = 1), sorted by the INV-7 total
    key (magnitude desc, label asc, `f"pair:{pair_id}"` asc — each move maps to one
    `pair_id`, so the key is unique). The pair **node** itself sorts among its siblings on
    `f"pair:{min pair_id in the group}"` (INV-7), since an aggregated pair spans several
    `pair_id`s. The Transfers top node's `amount` = Σ
    over the kept transfers. `ConfirmedTransfer.display_amount` is **already** the
    display-scaled positive `Decimal` (from `_make_candidate`, same exponent as
    `to_display_decimal`), so summing these Decimals needs no re-scaling and presents
    identically to the Income/Spending figures (INV-8). A transfer is attributed to **exactly one**
    period — the one holding its **debit** leg — so a debit-outside/credit-inside split
    is not shown here (INV-6); it is not lost data, just attributed to its debit month.
    Grouping is by the account-**name** pair, so on the (rare) two-accounts-same-name
    case the pairs collapse by name — cosmetic, and acceptable at personal-finance scale.

- **D5 — The richer read (`repositories/reporting.py`), a deliberate sibling.**
  `ReportingRepository.drill_rows_in_range(start_iso, end_iso, account_ids) ->
  list[tuple[int, str, int, int | None, str]]` returns `(id, occurred_on,
  amount_minor, category_id, description)` with the **same** window + `account_ids`
  semantics as `rows_in_range` (including the empty-set short-circuit). It adds exactly
  **one** column over `rows_in_range`'s 4-tuple — `description` (for the merchant
  cleanup, D3); it does **not** carry `account_id`, because the account **scope** is
  already applied in the SQL `WHERE account_id IN (…)` clause and the Transfers branch
  sources its account pair from `confirmed_transfers()` (D4b), not this read — so an
  `account_id` column would be dead weight (coding.md § 1.1). This is a **new** read, not
  an extension of `rows_in_range`, on purpose: the three hot aggregations
  (`summary`/`spending_by_category`/`monthly_trend`) run on every dashboard render and
  want the lean 4-tuple with no strings; the drill needs the `description` string and
  runs on the same refresh but is a distinct concern — extending the 4-tuple would ripple
  three unpack sites and pull an unused string into the hot path (coding.md § 1.3 —
  reuse is preferred, but here the column sets genuinely differ, so a focused sibling is
  the smaller, clearer change). Both reads share the identical `WHERE occurred_on
  BETWEEN ? AND ?` + `account_id IN (…)` skeleton.

- **D6 — `HomeView` wiring (`ui/home.py`).** `_build_dashboard` builds the tree
  (D1) and stores it as `self._drilldown`. `refresh()` builds the tr()-ed labels once —
  `labels = DrillLabels(income=self.tr("Income"), spending=self.tr("Spending"),
  transfers=self.tr("Transfers"), uncategorised=self.tr("Uncategorised"))` — and gains a
  `self._render_drilldown(self._reporting.drill_down(prefs, account_ids, labels=labels))`
  call placed **after** the trend render (alongside `_render_tiles` / `_render_donut` /
  `_render_trend` at `home.py:281-283`, **before** the `self._stack.setCurrentIndex(1)`
  page-switch at `:284`) — i.e. **guarded by** the existing
  `transaction_count() == 0` early-`return` (`:270-272`), so it runs only when the
  vault has data and the getting-started page still wins on an empty vault. It is
  **not** inside the `count == 0` block. Same `VaultLockedError` posture as the
  surrounding renders. `HomeView` (a `QObject`) owns the `tr()` calls, keeping the
  non-`QObject` `ReportingService` translation-free (D2/INV-9) — the exact pattern
  `_render_donut` already uses to pass `tr("Uncategorised")`/`tr("Other")` into
  `build_donut_chart` (`home.py:337-343`). No constructor signature change — `HomeView`
  already holds the `ReportingService` `drill_down` lives on.

- **D7 — `_render_drilldown` (`ui/home.py`).** Clears the tree and inserts the three
  top `DrillNode`s and their descendants via a recursive `_add_node(parent_item, node,
  branch_colour=None)` — the branch colour is threaded down the recursion (the top node
  sets it; children inherit) since `DrillNode` carries no colour. Each `QTreeWidgetItem`
  shows the `label` in column 0 and
  `_format_amount(node.amount, symbol, self._amount_prefs.negative_style)` in column 1
  (right-aligned; the shared FIBR-0105 formatter, so screen figures match the tiles).
  For a **merchant** or **account-pair** node with `count > 1`, column 0 shows
  `self.tr("{label} ×{count}").format(label=node.label, count=node.count)` (a single
  `tr()` template with data placeholders — the FIBR-0083 `.format(...)` idiom,
  `lupdate`-extractable); a `count == 1` node and the category/txn nodes show the bare
  `label`. Amount colouring reuses the tile rule when `amount_prefs.colour` is on:
  Income-branch amounts `_POSITIVE_TEXT`, Spending-branch `_NEGATIVE_TEXT` — passed
  down from the top node so a deep txn leaf inherits its branch colour (Transfers use
  the default text colour — they are neither income nor spending). The top three items
  are inserted collapsed (the user expands what they want); the tree has no editable
  cells (`QTreeWidget` is read-only by default — no `ItemIsEditable` flag set).

- **D8 — Empty states (INV-4/INV-7).** A category / merchant / pair with **zero**
  matching rows is **omitted** (parity with the donut, which drops empty categories);
  only the **three top nodes** are always present (a zero `DrillNode` with no
  `children` when a whole branch is empty). So a period with only income shows a
  Spending node reading the zero-formatted amount with nothing to expand, and vice
  versa; a vault with no confirmed transfers shows a zero Transfers node.

- **D9 — No new dependency; bundle-neutral.** `merchant_name` uses stdlib `re`;
  `DrillNode` is a dataclass; the tree is `PySide6.QtWidgets.QTreeWidget` (already
  imported elsewhere, LGPL, no new Qt module — unlike QtCharts it needs no `--self-test`
  bundle-travel leg). `pyproject.toml` gains **no** runtime dependency and **no**
  package-data (there is no data file — the merchant rules are code, deliberately, so
  they are unit-testable and versioned with the logic).

## Data model — no schema change

**No migration.** `LATEST_SCHEMA_VERSION` is **unchanged** (currently **8**). The
drill is **read-only** aggregation over the existing `transactions` / `categories` /
`transfer_pairs` tables; nothing new is written to the schema and no setting is
persisted (the tree state is not remembered between views — out of scope). `DrillNode`
is an in-memory read model, never stored.

## Test plan

`tests/features/dashboard_drilldown/` — all legs runnable headless except the qtbot
UI legs (which `pytest-qt` runs). Seeded two-account vault with income, categorised +
uncategorised expense, a nested category, a confirmed **and** an unconfirmed transfer,
and merchant-noisy descriptions, over a fixed `today`.

- **`merchant_name` totality + cleanup (INV-5, pure):** noisy inputs that the D3
  steps actually fold — `"POS 1234 WOOLWORTHS 5678"`, `"woolworths"`,
  `"WOOLWORTHS 09-08-2026"` — all clean to the **same grouping key**
  `normalise_text(...)`; a blank / whitespace-only string → `""` (never raises); a
  digits-only string (`"1234567"`) → falls back to the trimmed raw (never an empty
  label); punctuation-only edges trimmed. **Company suffix is retained** (D3): assert
  `"WOOLWORTHS PTY LTD"` keys **separately** from `"WOOLWORTHS"` (documents the v1
  behaviour, so a future suffix-strip is a deliberate change, not a silent one). The
  group **label** is the lexicographically smallest `merchant_name` in the group
  (deterministic regardless of row order). `merchant_name("")` / whitespace `== ""` is
  asserted for **totality** (the pure function never raises), noting it does not arise in
  the pipeline (stored descriptions are non-empty, D3). Table-driven.
- **`drill_rows_in_range` (INV-3, repo):** returns the 5-tuple `(id, occurred_on,
  amount_minor, category_id, description)`; window + `account_ids` semantics identical to
  `rows_in_range` (all / subset / empty-set → `[]`); the extra `description` column is
  populated.
- **`drill_down` totals equal the tiles (INV-1, the headline):** for the same
  `(prefs, account_ids)`, the Income node total == `summary(...).income`, the Spending
  node total == `summary(...).expenditure`; each category node == Σ of its children;
  the `amount_minor` multiset is untouched (a pure read). Computed on integers (INV-8).
- **Category branch (INV-4):** a **nested** category (root → parent → leaf) aggregates
  the leaf's rows into the parent and the branch; two same-named leaves under different
  parents stay **distinct** nodes (grouped by id); uncategorised rows form **one**
  Uncategorised node (by `None` id, even if a real leaf is named "Uncategorised"); an
  empty category is **omitted**; the branch total equals `Summary` (INV-1).
- **Category branch — the reachable edges (INV-1/INV-4, load-bearing):**
  - **Mis-set cross-root row:** a **negative** row tagged to an **Income-root** leaf
    still appears under **Spending** (via its top-of-chain node) and the Spending branch
    total **still equals** `summary(...).expenditure` — the row is never dropped. This is
    the exact case the single "group by top-of-chain" rule protects (D4a).
  - **Non-leaf category with its own rows:** a category that has **both** a child
    category **and** directly-assigned rows renders **both** child-category nodes **and**
    merchant nodes for its own bucket, and its subtree total = own + descendants (no rows
    made invisible). Built via `CategoryRepository.add` under a row-bearing category. This
    same node's children are a **mixed** category+merchant sibling list — assert it sorts
    **without raising** (the INV-7 uniform **string** key, not an int/str mix), the
    falsifier for the `TypeError` INV-7's key exists to prevent.
  - **Same id under both branches:** a category holding both a positive and a negative
    row yields a node under **Income** *and* under **Spending**, each carrying only its
    own sign-bucket total (neither double-counts).
- **Merchant grouping (INV-5):** at a leaf, three transactions at "Woolworths" (noisy
  descriptions) collapse into one merchant node with `count == 3` and the summed
  magnitude; expanding it yields three individual-transaction leaves (date + amount);
  a mis-grouped shop does **not** change the leaf total (INV-1 re-asserted).
- **Transfers exclusion + drill (INV-2/INV-6):** a **confirmed** self-transfer is
  absent from every category/merchant node and present under Transfers, grouped by the
  `from → to` pair with `×count`; an **un**confirmed pair is ordinary spending/income;
  the account filter keeps a transfer when either leg is selected; the period filter
  keys on the debit leg's date.
- **Ordering (INV-7):** every sibling list is descending by magnitude, then label asc,
  then the total `id`/key; the three top nodes are always `[Income, Spending,
  Transfers]` in that order. Falsifier for the **total** key: two same-named leaves
  under different parents with **equal** magnitude order deterministically by
  `category_id`; two transaction leaves with the **same date and amount** order by row
  `id` — a label-only tie-break would leave these unstable.
- **Empty states (INV-4/INV-7/D8):** an income-only period → a zero Spending node
  (`count == 0`) with no children; a vault with no confirmed transfers → a zero
  Transfers node; the three top nodes always present.
- **UI (qtbot, INV-3/INV-9):** `HomeView.refresh()` populates `dashboard_drilldown`
  with three top items whose labels are the passed-in `DrillLabels` (`tr("Income")` /
  `tr("Spending")` / `tr("Transfers")`) and a `None`-category node labelled the passed

## Out of scope

- **Jumping from a drill leaf to the Transactions tab** (select the row there) — a nice
  future affordance; the tree is read-only display now (coding.md § 1.1).
- **Remembering the tree's expanded state across views / relaunch** — the tree resets to
  collapsed on each `refresh()`; persisting expansion is a future polish.
- **A user-editable / configurable merchant-cleanup ruleset, or a merchant field on the
  transaction** — the cleanup is code, refined per release (D3); there is no stored
  merchant column and none is added.
- **Drilling the 12-month trend bars** — the drill hangs off the Income/Spending/
  Transfers totals for the selected period, not the per-month trend; trend drill is a
  separate future idea.
- **Editing a category / merchant from the tree** (rename, re-categorise) — the
  Transactions tab + Categories tab own those; the drill is a read-only lens.
- **Marking library-guessed rows in the tree** (the FIBR-0139 "~ guess" tag) — the tree
  is an aggregate view; the marker stays on the Transactions per-row cell (FIBR-0139 D7).
