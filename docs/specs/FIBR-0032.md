## Context

Users want to lift a transaction's amount or description out of finbreak — to
paste into a calculator, a search box, or a note. Today there is **no copy
affordance anywhere** in the app (grep of `src/` for `QClipboard` /
`clipboard()` returns zero app-wired hits — FIBR-0032 introduces the first).
The catch: the system clipboard is a **shared, plaintext, cross-application
surface**. Anything copied there lingers until overwritten — readable by any
other running app, and often captured by clipboard-history managers. For a
finance app that is a small but real exposure the moment a copy affordance
exists.

So this feature is deliberately **two halves that ship together**: (1) add the
copy affordance the user asked for — "Copy amount" / "Copy description" on the
transactions list — and (2) make it **self-cleaning**: the copied value is
wiped from the clipboard after a short, configurable timeout, **but only if the
clipboard still holds the value we put there** (never clobber something the user
copied in the meantime).

**Scope resolved by the user (2026-07-18):** the only copyable values are a
transaction's **amount** and **description**. The stored statement PDF password
stays **non-copyable** (no regression of FIBR-0128 INV-1 — that secret never
crosses into the UI), and account numbers are **not** copyable in this pass.
Adding a "copy the PDF password" affordance would be a new secret exposure and
is explicitly excluded.

## The mechanism

Two pieces: a tiny reusable clipboard helper, and the transactions-list wiring
that calls it.

**1. `ClipboardAutoClear` (`src/finbreak/ui/_clipboard.py`) — a `QObject`.**

```
ClipboardAutoClear(clipboard: QClipboard, *, seconds_provider: Callable[[], int], parent=None)
    .copy(text: str) -> None       # set clipboard + arm the auto-clear timer
    .clear_if_ours() -> None       # clear the clipboard IFF it still holds our value
```

- `copy(text)`:
  1. `self._clipboard.setText(text)` — writes to the **default `Clipboard`
     mode** (`QClipboard.Clipboard`; the X11 middle-click *Selection* is not
     touched — D6).
  2. `seconds = self._seconds_provider()` — read **live** at copy time (so a
     mid-session Settings change takes effect on the next copy).
  3. If `seconds > 0`: record `self._pending = text` and arm the timer —
     `self._timer.stop(); self._timer.start(seconds * 1000)` on a single owned
     `QTimer(self)` with `setSingleShot(True)`. If `seconds == 0` ("Never"):
     `self._timer.stop()` and set `self._pending = None` — copy still works, it
     is just never auto-cleared.
  A second `copy()` before the first fires simply replaces `_pending` and
  restarts the one timer, so the **newest** copied value always governs the
  full timeout and a stale pending clear can never fire against a newer value.
- `clear_if_ours()` — **the guard, in one place.** `if self._pending is not
  None and self._clipboard.text() == self._pending: self._clipboard.clear()`,
  then `self._pending = None`. Reading the current clipboard text and comparing
  to the exact string we set is what guarantees we only wipe *our* value — if
  the user copied anything else since (even from another app), the strings
  differ and we leave it alone.
- `_on_timeout()` (the single-shot timer's slot) **is** `clear_if_ours()` — the
  slot simply calls it. So the guard has exactly one implementation, it is
  exercised by the auto-clear path (and its tests, INV-3), and the deferred
  lifecycle-clear follow-up (Out of scope) reuses the **same** method rather
  than duplicating the comparison. `clear_if_ours()` is therefore **not** dead
  code this pass — the timeout is its caller.

**2. Transactions-list wiring (`src/finbreak/ui/transactions.py`).**

The transactions table is a `QTableWidget` that already has a custom context
menu (`_show_context_menu`, `transactions.py:345`, currently one action "Set
category…"). Extend that menu, for a selected row, with **"Copy amount"** and
**"Copy description"** above a separator, then the existing "Set category…":

- Row → data: the menu already early-returns unless `self._selected_txn()` is
  non-`None`. The selected row's tuple is
  `self._rows[selected_index(self._table)]` =
  `(Transaction, Decimal display_amount, account_name, category_name)`.
- **Copy amount** copies the **exact Amount-column display string** — reusing
  the one existing formatter: `_format_amount(display, self._transactions.
  base_currency(), self._amount_prefs.negative_style)` (the same call the
  column render makes at `transactions.py:268`). WYSIWYG — see D1.
- **Copy description** copies the raw `txn.description` (the Transaction's own
  field, exactly as the Description column shows it).
- Both route through the injected `ClipboardAutoClear.copy(...)`.

**3. The timeout setting (`clipboard_clear_seconds`).** Mirrors the existing
auto-lock-minutes pattern exactly (an `AuthService` typed getter/setter with a
default constant; the key is **defaulted in code**, not seeded in a migration —
`migrations.py` issues no `INSERT INTO settings`):

- `AuthService.clipboard_clear_seconds() -> int` — reads the
  `clipboard_clear_seconds` settings key; falls back to
  `DEFAULT_CLIPBOARD_CLEAR_SECONDS = 30` on an absent / non-int / out-of-set
  value. `ALLOWED_CLIPBOARD_CLEAR_SECONDS = (10, 30, 60, 0)` (0 = "Never").
- `AuthService.set_clipboard_clear_seconds(seconds: int) -> None` — **raises
  `ValueError`** for a value outside `ALLOWED_CLIPBOARD_CLEAR_SECONDS` (before
  any write), then `SettingsRepository(...).set("clipboard_clear_seconds",
  str(seconds))` — exactly as `set_auto_lock_minutes` raises `ValueError`
  (`auth.py:326`), minus that setter's `_arm_timer()` call (there is no live
  clipboard timer to re-arm; the timeout is read fresh per copy — D3).
- `SettingsDialog`: a `QComboBox` (`objectName="settings_clipboard_clear"`)
  populated from `ALLOWED_CLIPBOARD_CLEAR_SECONDS` with `addItem(label,
  seconds)`, preselected on open via `select_combo_data(self._combo, service.
  clipboard_clear_seconds())`, persisted in `_on_save` via
  `self._service.set_clipboard_clear_seconds(self._combo.currentData())`
  (inside the existing `try/except VaultLockedError`). Labels: "10 seconds",
  "30 seconds", "1 minute", "Never".

**4. Injection.** The view's `__init__` gains an optional `clipboard:
ClipboardAutoClear | None = None` that **defaults to a self-built fallback**
when `None` — exactly mirroring FIBR-0151's `transfers=None` parameter
(`transactions.py:74`; `self._transfers = transfers or
TransferDetectionService(...)`, `:85`). The fallback builds a
`ClipboardAutoClear` over `QGuiApplication.clipboard()` with a constant
`DEFAULT_CLIPBOARD_CLEAR_SECONDS` provider (enough for standalone / test
construction).

For the **production** path the shell injects a service-wired instance, so the
timeout is read from the *single* settings source (`AuthService`) rather than a
second reader: `main_window` builds `ClipboardAutoClear(QGuiApplication.
clipboard(), seconds_provider=self._service.clipboard_clear_seconds)` and passes
it as a new `clipboard=` argument at the existing `TransactionsView(...)`
construction (`main_window.py:608`, currently four positional args). **Note —
this shell injection is *new*:** FIBR-0151 added the `transfers` parameter but
left it on its internal fallback and did **not** inject at `main_window.py:608`.
So only the `clipboard=None` fallback mirrors FIBR-0151; the shell-side
`clipboard=` injection is the first of its kind at that call site.

## Design decisions

- **D1 — "Copy amount" is WYSIWYG (the displayed cell text), not a re-derived
  machine number.** Copy puts the exact Amount-column string on the clipboard —
  currency symbol, locale grouping, and the user's chosen negative style
  (`-$1,234.56` or `($1,234.56)`) — by reusing `_format_amount`, the *single*
  money formatter. Rationale: (a) copying a table cell copying its rendered text
  is the least-surprising behaviour everywhere else; (b) it introduces **no
  second money-format code path** to keep in sync with the display (§3 reuse,
  §2 shortest); (c) it is consistent with "Copy description" (raw shown text).
  **Considered and rejected for this pass:** copying a plain machine value
  (`1234.56`, unsymboled) for spreadsheet paste — a reasonable alternative, but
  it forks the number format and picks a paste target we can't know. If users
  ask for it, it is a trivial follow-up. Documented here so the choice is
  deliberate, not accidental.
- **D2 — Clear only if the clipboard still holds our value.** The timeout slot
  compares the live clipboard text against the exact string we set and clears
  only on an exact match. This is the roadmap's explicit requirement ("don't
  wipe something the user copied since"). Unavoidable edge: if the user
  independently copies a *byte-identical* string elsewhere, we cannot tell it
  apart from ours and will clear it on timeout — an accepted, negligible corner
  (identical content is indistinguishable, and the loss is only a re-copy).
- **D3 — Timeout read live, per copy.** `seconds_provider()` is called inside
  `copy()`, not cached at construction, so changing the setting in Settings
  applies to the very next copy without rewiring. Cheap (one settings `SELECT`).
- **D4 — 0 = "Never" means copy-without-auto-clear, not copy-disabled.** A user
  who sets "Never" still gets the copy actions; the value simply persists like
  any normal clipboard copy. The feature never *removes* the ability to copy.
- **D5 — The setting mirrors auto-lock exactly.** Same shape (a `QComboBox` of
  allowed values with an `AuthService` typed getter/setter and a code default),
  same `settings_*` objectName convention, same `_on_save`/`VaultLockedError`
  persistence path. No new persistence mechanism, no migration change (the key
  is defaulted in code, exactly as `auto_lock_minutes` and the amount prefs are).
- **D6 — Only the default `Clipboard` mode.** We `setText`/`clear` the standard
  clipboard (Ctrl-C/Ctrl-V). We deliberately do **not** write or clear the X11
  *Selection* (primary/middle-click) buffer: we never put the value there, so
  there is nothing of ours to clear, and touching a user's primary selection
  would be surprising cross-platform behaviour.
- **D7 — The helper is a testable seam.** `ClipboardAutoClear` takes its
  `QClipboard` and `seconds_provider` by injection, so tests drive it with a
  real `QGuiApplication.clipboard()` (qtbot) or a fake and assert the guard
  branches deterministically (interval + active-state after `copy`, and the
  clear / no-clear behaviour by invoking the timeout slot) — no 30-second waits,
  no global-clipboard pollution in the UI tests (which inject a recording fake).

## At a glance — invariants

_These are FIBR-0032's own local numbering, enforced by
`tests/features/clipboard/`._

| INV | Statement |
|-----|-----------|
| **INV-1** | **Copy affordances exist.** For a selected transaction row, the context menu offers **"Copy amount"** and **"Copy description"** (and still "Set category…"). With no row selected, the menu early-returns (unchanged). |
| **INV-2** | **What gets copied.** "Copy amount" places on the clipboard the exact Amount-column display string produced by `_format_amount(display, base_currency, negative_style)` (symbol + grouping + the configured negative style); "Copy description" places the raw `Transaction.description`. Nothing else is transformed. |
| **INV-3** | **Auto-clear, guarded.** After a copy with a positive timeout, once the timeout elapses the clipboard is cleared **iff** it still holds the exact value we set. If the clipboard content changed after the copy (the user copied something else), the timeout leaves it untouched. |
| **INV-4** | **Configurable + "Never".** The timeout is read live from `AuthService.clipboard_clear_seconds()` (default 30; allowed `(10, 30, 60, 0)`). With the setting at `0` ("Never"), a copy still writes the value and **no** clear timer is armed (the value persists). |
| **INV-5** | **Settings control.** `SettingsDialog` exposes the timeout as a `QComboBox` (`objectName "settings_clipboard_clear"`), preselected to the stored value on open and persisted through `AuthService.set_clipboard_clear_seconds` on Save; the setter **raises `ValueError`** on a value outside the allowed set (before any write), and the getter falls back to the default on an absent / non-int / out-of-set stored value. |
| **INV-6** | **No new secret exposure.** No "Copy password" / "Copy account number" affordance is added anywhere. The stored statement PDF password remains non-copyable and never crosses into the UI (FIBR-0128 INV-1 preserved); the transactions menu exposes only amount + description. |
| **INV-7** | **Clipboard mode + i18n + no network.** Copy and clear target `QClipboard`'s default `Clipboard` mode only (the X11 Selection buffer is never written or cleared). All user-facing strings — the "Copy amount" / "Copy description" menu items, the four combo item labels ("10 seconds" / "30 seconds" / "1 minute" / "Never"), and the settings field label — go through `tr()`; this is a **convention (ruff / reviewer-enforced), not a unit-test assertion** (asserting a *rendered* label can't prove `tr()` was called). The new modules import nothing network-related (the vault-suite static scan covers them). |

## Out of scope

- **Lifecycle-clear (on lock and on app-exit)** — wiping a still-pending
  clipboard value the instant the vault locks (manual / idle, via
  `main_window._lock`, `main_window.py:551`) **or the app exits** (the same
  lifecycle seam that wipes the key on exit — security-model T3) is a natural
  strengthening and cheap: both just call the existing
  `ClipboardAutoClear.clear_if_ours()`. Left out **this pass** to hold the
  resolved scope (copy + timed auto-clear). **Accepted residual:** on a
  mid-timeout lock *or quit*, the copied value can outlive its intended timeout
  on platforms where the clipboard survives the app (Windows / macOS /
  Wayland-with-manager); with the 30 s default far below the 10 min auto-lock
  default this window is small, and since `clear_if_ours()` already exists (it
  is the timeout guard, not new code) the follow-up is a one-line wire-up per
  event. Tracked as a roadmap follow-up.
- Copying account numbers or the statement PDF password (a deliberate
  non-goal — FIBR-0128 INV-1, INV-6 above).
- Copying arbitrary cells / whole rows, or a "Copy as CSV" export.
- Clearing or writing the X11 *Selection* (primary/middle-click) buffer (D6).
- A clipboard-history-manager opt-out (`x-kde-passwordManagerHint` MIME and the
  like) — a possible later hardening, not needed for the resolved scope.
