**Theme:** When you move money **between your own accounts** (pay a credit card
from your current account, shift cash to savings), two lines appear — money
**out** of one account and the same amount **in** to another. finbreak spots
these matched pairs and **asks** you to confirm; only pairs you confirm stop
counting as spending or income. Nothing is ever hidden without your say-so
(ADR-0006). Rejected pairs are remembered and never re-offered.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Suggest, never auto-apply.** Detection only **proposes** candidate pairs; a transaction is excluded from totals **only after the user confirms**. Nothing is hidden without confirmation. | ADR-0006 |
| INV-2 | **What matches.** A candidate is one **debit** (`amount_minor < 0`) and one **credit** (`amount_minor > 0`) of **equal magnitude** (`debit.amount_minor = -credit.amount_minor`), in **different** accounts, dated within **±`TRANSFER_WINDOW_DAYS` (= 3)** days **inclusive**. | ADR-0006; FIBR-0007 sign convention |
| INV-3 | **Decided pairs don't resurface.** A pair the user has **confirmed or rejected** is never offered again. Rejections are remembered (a `'rejected'` row) so a coincidental same-amount match stays dismissed. | ADR-0006 "remembers rejected pairs" |
| INV-4 | **One transfer per transaction.** A transaction belongs to **at most one confirmed transfer**; once confirmed it is excluded from all further candidate detection (an ambiguous debit that matched two credits resolves to the one the user picked). | ADR-0006 correctness |
| INV-5 | **Exclusion primitive.** `confirmed_transfer_txn_ids()` returns the set of transaction ids in a confirmed pair — the single primitive FIBR-0012 totals against to drop internal movement. (No totals consumer exists yet; this phase only creates the primitive.) | success criterion 3 |
| INV-6 | **Unlink is reversible.** Unlinking a confirmed transfer deletes its row; the pair becomes undecided and re-surfaces as a candidate — a mis-confirm is fully recoverable (the ADR-0006 "never silently hide a real expense" guard against **user** error, not just auto error). | ADR-0006 |
| INV-7 | **Live detection.** Candidates are recomputed from the vault **every time the Transfers tab is shown** — no stored candidate state — so imports / edits / deletes are always reflected without a manual scan. | FIBR-0052 D4 (tab self-refresh) |
| INV-8 | **Statement-delete cascade.** Deleting a statement (and its transactions, FIBR-0052 INV-9) removes any `transfer_pairs` rows referencing those transactions automatically (`ON DELETE CASCADE`), so there are no dangling links and the delete is never FK-blocked. | FIBR-0052 INV-9; `vault.py` `PRAGMA foreign_keys = ON` |
| INV-9 | **Schema v7 → v8.** One new `transfer_pairs` table; `LATEST_SCHEMA_VERSION` 7 → 8; the migration is one atomic unit. | FIBR-0005 INV-4 |
| INV-10 | **Transfers tab.** A 6th tab lists **suggested** pairs with **Confirm / Reject / Confirm all** and **confirmed** transfers with **Unlink**. Actions apply **directly** (no modal — they are reversible or low-harm); every slot catches `VaultLockedError`. | FIBR-0052 INV-3/4; FIBR-0065 |
| INV-11 | **No network; i18n.** No network. Every user string `tr()`, every widget in a layout manager. | coding.md § 5.2; security-model INV-8 |
| INV-12 | **Transactions untouched.** Transfer detection never inserts, deletes, or edits a transaction — it only records the user's confirm/reject decisions. Row existence stays import's concern. | FIBR-0010 INV-12 (same spirit) |

## Context

Verified against source 2026-07-12 (see **Verified code basis**):

- **`transactions`** (v2 rebuild; v6 `statement_period_id`; v7 category link) has
  `id, account_id, occurred_on, amount_minor, description, created_at,
  statement_period_id, category_id, category_source`. The `Transaction` dataclass
  is **eight** fields (`statement_period_id` is a table column **not** in the
  dataclass); `TransactionRepository.list_all` `SELECT`s the eight in order into
  `Transaction(*row)`.
- **Sign convention (FIBR-0007).** `amount_minor` is a **signed** integer: a
  **debit is negative** (money out), a **credit is positive** (money in) —
  `csv_importer._debit_credit_amount` returns `-magnitude if debit else magnitude`,
  and `parse_transaction` forbids a **zero** amount. So a real inter-account
  transfer is exactly *one negative row in account A and one positive row of the
  same magnitude in account B*.
- **Dates.** `occurred_on` is a `TEXT` ISO-8601 date (`YYYY-MM-DD`,
  `date.fromisoformat`-validated at the form boundary). SQLite's `julianday()`
  parses this directly, so the ±3-day window is `abs(julianday(a) - julianday(b))
  <= 3`.
- **Deletes.** Transactions are removed only by **statement delete**
  (`TransactionRepository.delete_for_statement`, run inside `StatementService`'s
  owned transaction with `PRAGMA foreign_keys = ON`). There is **no** per-row
  transaction-delete or transaction-edit surface today — so a confirmed pair can
  only be invalidated by its statement being deleted, which INV-8 handles.
- **Shell** (`main_window.py`): a **5-tab** `QTabWidget` workspace
  (Home/Statements/Accounts/Categories/Rules, index constants `_TAB_HOME = 0` …
  `_TAB_RULES = 4`). `_build_workspace` builds each tab once from `self._service`
  (an `AuthService`); `_refresh_tab(index)` refreshes the activated tab;
  `_clear_live` nulls every tab reference on lock. Tab actions are built once by
  `_make_action("action_rules", tr("Rules"), "rules", self._open_rules)` and added
  to the View menu + toolbar; `_open_rules` sets the current index. Tab widgets
  (`RulesWidget(service: AuthService)`) construct their own services from
  `service.vault` and guard each slot with `except VaultLockedError: return`.

## Design decisions

- **D1 — Schema v8, one decision table (INV-9).** `_migrate_to_v8`, one atomic
  unit mirroring `_migrate_to_v7` exactly (`with owned_transaction(conn):`, the
  `CREATE TABLE` then `UPDATE schema_version SET version = 8` as the last
  statement, `ROLLBACK` on exception via the context manager). The single table:

  ```sql
  CREATE TABLE transfer_pairs(
      id INTEGER PRIMARY KEY,
      txn_a_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
      txn_b_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
      status TEXT NOT NULL,                 -- 'confirmed' | 'rejected'
      created_at TEXT NOT NULL,
      UNIQUE(txn_a_id, txn_b_id));
  ```

  Both the **confirmed link** and the **remembered rejection** are the identical
  shape — an ordered transaction pair + a decision — so one table with a `status`
  column is the shortest correct model (coding.md § 1.1: no second table for
  no gain). `ON DELETE CASCADE` (**unlike** FIBR-0010's service-owned category
  cascade) is correct here because removing the row is the *entire* cleanup when a
  referenced transaction is deleted — there is no source-reset or re-apply to do —
  so the FK does it automatically and `delete_for_statement` needs **no** change
  (INV-8).

- **D2 — Models.** `TransferStatus(StrEnum)` (`CONFIRMED = "confirmed"`,
  `REJECTED = "rejected"`) alongside `AccountType`/`CategoryKind`/`CategorySource`
  in `models.py`; a `TransferPair` dataclass (`id, txn_a_id, txn_b_id, status,
  created_at`) whose field order matches the repository `SELECT` for
  `TransferPair(*row)`.

- **D3 — Detection is one SQL self-join (single source of truth for INV-2).**
  `TransferRepository.candidate_pairs()` returns `(debit_id, credit_id)` tuples:

  ```sql
  SELECT d.id, c.id
  FROM transactions d
  JOIN transactions c
    ON c.amount_minor = -d.amount_minor        -- equal magnitude, opposite sign
   AND c.account_id  <> d.account_id           -- different accounts
   AND abs(julianday(c.occurred_on) - julianday(d.occurred_on)) <= 3
  WHERE d.amount_minor < 0 AND c.amount_minor > 0   -- d = debit, c = credit (fixes roles)
    AND d.id NOT IN (SELECT txn_a_id FROM transfer_pairs WHERE status = 'confirmed'
                     UNION SELECT txn_b_id FROM transfer_pairs WHERE status = 'confirmed')
    AND c.id NOT IN (SELECT txn_a_id FROM transfer_pairs WHERE status = 'confirmed'
                     UNION SELECT txn_b_id FROM transfer_pairs WHERE status = 'confirmed')
    AND NOT EXISTS (SELECT 1 FROM transfer_pairs p
                    WHERE p.txn_a_id = min(d.id, c.id) AND p.txn_b_id = max(d.id, c.id))
  ORDER BY d.occurred_on, d.id, c.id;
  ```

  The `amount_minor < 0` / `> 0` split **fixes** `d` as the debit and `c` as the
  credit, so each unordered transfer appears **once** — no direction-dedup needed.
  The two `NOT IN` clauses enforce INV-4 (a txn already in a confirmed pair is
  consumed); the `NOT EXISTS` enforces INV-3 (a specific confirmed-or-rejected
  pair never re-surfaces). The rule lives in this one query — **no** duplicate
  pure predicate (coding.md § 1.3); it is tested against a seeded in-memory vault
  (window edges at day offsets 0/3/4, off-by-one-minor amounts, same-account,
  same-sign).

- **D4 — Canonical ordering.** `confirm`/`reject` store `min(a, b)` as `txn_a_id`
  and `max(a, b)` as `txn_b_id`, so a pair has one representation regardless of
  which side the caller passes and `UNIQUE(txn_a_id, txn_b_id)` holds. **Direction
  is not stored** — which side is "from" (debit) vs "to" (credit) is recovered
  from each transaction's sign at read time.

- **D5 — TransferService (`services/transfers.py`, vault-scoped) mirrors
  `CategorizationService`.** `_conn()` / a `TransferRepository` accessor, plus:
  - `candidates() -> list[TransferCandidate]` — runs D3, resolves each pair's two
    `Transaction` rows, the display magnitude (via `read_minor_unit_exponent` +
    `to_display_decimal`, reused), and both account names (an id→name map resolved
    once, mirroring `list_transactions`).
  - `confirm(debit_id, credit_id)` — insert a `'confirmed'` row (canonical order);
    guards that **neither** txn is already in a confirmed pair (`ValueError`,
    matching the house validation convention — the UI only offers valid
    candidates, so this is defensive).
  - `reject(debit_id, credit_id)` — insert a `'rejected'` row (canonical order).
  - `unlink(pair_id)` — delete the confirmed row (INV-6).
  - `confirmed_transfers() -> list[ConfirmedTransfer]` — for the tab's confirmed
    list + Unlink (pair id + both rows + display + from/to names).
  - `confirmed_transfer_txn_ids() -> set[int]` — the INV-5 exclusion primitive.

- **D6 — Repository writes are per-decision commits.** `confirm`/`reject`/`unlink`
  are each a **single** `INSERT`/`DELETE` — atomic on their own — so
  `TransferRepository` commits per write (like `TransactionRepository.add`), not
  service-owned (there is no multi-statement unit to wrap). `candidate_pairs`,
  `list_confirmed`, and `confirmed_txn_ids` are reads.

- **D7 — "Confirm all" is greedy + consumption-safe (INV-4).** Iterate the current
  `candidates()` in order; for each, skip if its debit or credit was **already
  consumed** by an earlier confirm in the same pass (tracked in a local set), else
  confirm; report the count confirmed in the status label. So a debit matching two
  credits is confirmed against the **first** and its second candidate is dropped —
  never a double-link.

- **D8 — No modal dialogs in the tab.** Confirm / Reject / Confirm-all / Unlink are
  **direct** button actions with status-label feedback — no `QMessageBox`
  confirmation. Rationale: **Unlink** makes a confirm reversible (INV-6), and a
  **Reject** is low-harm (it merely keeps a pair counting as income+expenditure —
  the safe direction of error, versus a mis-confirm which Unlink undoes). This
  keeps the auto-lock story trivial (no nested event loop to tear down — the
  FIBR-0065 class of bug cannot arise here); each slot still catches
  `VaultLockedError` and returns, exactly like `RulesWidget._on_delete`.

- **D9 — TransfersWidget (`ui/transfers.py`, `AuthService`) mirrors RulesWidget.**
  Two `QTableWidget`s in a `QVBoxLayout`:
  - **Suggested transfers** — columns Date / Amount / From → To / Description;
    buttons **Confirm**, **Reject** (enabled on a single-row selection) and
    **Confirm all**.
  - **Confirmed transfers** — the same columns; an **Unlink** button (enabled on
    selection).
  Parallel `self._candidates` / `self._confirmed` lists track the table rows (the
  `RulesWidget._rows` pattern); `_refresh()` re-reads both via the service;
  `_selected_row()` / selection-driven button enabling reused. A **status
  `QLabel`** reports outcomes ("Confirmed %n transfer(s).", "Unlinked."). A new
  **`transfers.svg`** glyph in the icon set (loaded by `_make_action`; travels via
  the existing `*.svg` package-data glob).

- **D10 — Shell wiring (`ui/main_window.py`).** Add `_TAB_TRANSFERS = 5`. In
  `_build_workspace`, build `self._transfers_tab = TransfersWidget(self._service)`
  and `addTab(..., self.tr("Transfers"))` as the **6th** tab; add the
  `_transfers_tab = None` line to `_clear_live`; add the `_refresh_tab` branch
  (`elif index == _TAB_TRANSFERS ... self._transfers_tab._refresh()`). Build
  `self._action_transfers = self._make_action("action_transfers",
  self.tr("Transfers"), "transfers", self._open_transfers)`, add it to
  `_menu_view` and the toolbar action list, and add `_open_transfers` (sets the
  index via `_ensure_workspace()`). **No** special post-import hook — INV-7's
  live-on-view detection means the new candidates appear the moment the user opens
  the tab.

## Invariants

- **INV-1 — Suggest, never auto-apply.** `candidates()` is the only producer of
  proposed pairs; a transaction enters `confirmed_transfer_txn_ids()` **only**
  through `confirm()`. No code path confirms without a user action. Test: seed a
  perfect debit/credit pair → it appears in `candidates()` but
  `confirmed_transfer_txn_ids()` is empty until `confirm()`.
- **INV-2 — What matches.** Exactly the D3 predicate. Tests (seeded vault):
  a matching pair at day offsets 0 and 3 **matches**; offset 4 **does not**;
  amounts off by one minor unit **do not**; two debits (same magnitude, both
  negative) **do not**; a same-account opposite pair **does not**;
  `TRANSFER_WINDOW_DAYS = 3` is the named constant.
- **INV-3 — Decided pairs don't resurface.** After `reject(d, c)`, the pair is
  absent from `candidates()`; after `confirm(d, c)`, likewise. A *different*
  same-amount credit still surfaces for the same debit **only while the debit is
  unconfirmed** (INV-4).
- **INV-4 — One transfer per transaction.** `confirm` on a txn already in a
  confirmed pair raises `ValueError`; after `confirm(d, c1)`, `d` (and `c1`) no
  longer appear in `candidates()`, so `(d, c2)` is gone. `confirm_all` on a debit
  with two candidate credits confirms exactly one.
- **INV-5 — Exclusion primitive.** `confirmed_transfer_txn_ids()` == the union of
  both ids of every `'confirmed'` row (and **only** confirmed — a rejection
  excludes nothing). Test: confirm one pair, reject another → the set is exactly
  the two confirmed ids.
- **INV-6 — Unlink reversible.** After `confirm` then `unlink`, the pair is back
  in `candidates()` and out of `confirmed_transfer_txn_ids()`.
- **INV-7 — Live detection.** `candidates()` reads live vault state each call (no
  cache); a newly-inserted matching pair appears on the next call with no other
  action. (`_refresh()` on tab activation is the UI expression of this.)
- **INV-8 — Statement-delete cascade.** Confirm a pair whose two txns belong to a
  statement, then delete the statement → the `transfer_pairs` row is gone (no FK
  error, no dangling row). Test drives `StatementService.delete` (or
  `delete_for_statement`) and asserts the count.
- **INV-9 — Schema v7 → v8.** A v7 vault migrates to v8 with `transfer_pairs`
  present and empty; `LATEST_SCHEMA_VERSION == 8`; a mid-step failure leaves a
  re-openable v7 (atomic). Test mirrors the existing migration tests (a
  `build_v7_vault` helper if needed).
- **INV-10 — Transfers tab.** The workspace has **6** tabs; the Transfers tab
  (`objectName "tab_transfers"`) shows the two tables + buttons; Confirm/Reject/
  Confirm-all/Unlink drive the service and refresh; an auto-lock mid-click
  (`VaultLockedError`) is caught, no crash.
- **INV-11 — No network; i18n.** No import under `src/` reaches the network
  (the existing `test_INV8_no_network_imports_under_src` AST scan still passes —
  `transfers.py`/`ui/transfers.py` add none); every user string is `tr()`,
  every widget sits in a layout manager.
- **INV-12 — Transactions untouched.** No transfer operation issues an
  `INSERT/UPDATE/DELETE` against `transactions`. Test: transaction rows (count,
  amounts, categories) are byte-identical before and after confirm/reject/unlink.

## Data model — schema v7 → v8

```sql
CREATE TABLE transfer_pairs(
    id INTEGER PRIMARY KEY,
    txn_a_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    txn_b_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    status TEXT NOT NULL,               -- 'confirmed' | 'rejected'
    created_at TEXT NOT NULL,
    UNIQUE(txn_a_id, txn_b_id));
```

Canonical ordering `txn_a_id < txn_b_id` (D4). `ON DELETE CASCADE` on both FKs so
a statement delete (with `PRAGMA foreign_keys = ON`) auto-clears links (INV-8).
No backfill (a fresh table). `LATEST_SCHEMA_VERSION` → 8.

## Out of scope

- **Near-amount / fee-tolerant matching** — exact magnitude only (ADR-0006 "same
  amount"; fees are separate statement lines). A tolerance is a clean future
  follow-up; **no unused column now** (coding.md § 1.1).
- **The dashboard totals** that consume `confirmed_transfer_txn_ids()` — FIBR-0012;
  this phase creates only the primitive.
- **Undo-reject in the UI** — a mis-reject is low-harm (the pair merely keeps
  counting); re-surfacing rejected pairs is a possible future "show rejected"
  view.
- **A pending-count badge** on the Transfers tab — polish.
- **Auto-confirm / transfer rules, multi-currency matching, N-way / split
  transfers** — one debit ↔ one credit, single base currency, always user-confirmed.
- **Marking transfers in the Home / Transactions table** — display is unchanged
  here; FIBR-0012 / FIBR-0109 may surface a transfer marker later.

## Exit criteria

1. All INV-1..INV-12 covered by `tests/features/transfers/` and green (INV-11's
   i18n half is convention/ruff-enforced; its no-network half is the existing AST
   scan).
2. `./scripts/ci-local.sh` green (ruff/format/bandit/pip-audit/gitleaks + pytest),
   **mypy 0** (gated, FIBR-0061).
3. `/audit` 0 actionable; `/indie-review` clean (or all findings folded).
4. Manual sanity: import two statements that contain a real inter-account transfer
   → the pair appears under **Suggested transfers**; **Confirm** it → it moves to
   **Confirmed transfers** and both ids appear in `confirmed_transfer_txn_ids()`;
   **Unlink** → it returns to Suggested; **Reject** a coincidental match → it does
   not re-appear; delete one statement → its links are gone.
