**Theme:** When you move money **between your own accounts** (pay a credit card
from your current account, shift cash to savings), two lines appear ‚Äî money
**out** of one account and the same amount **in** to another. finbreak spots
these matched pairs and **asks** you to confirm; only pairs you confirm stop
counting as spending or income. Nothing is ever hidden without your say-so
(ADR-0006). Rejected pairs are remembered and never re-offered.

## At a glance ‚Äî invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Suggest, never auto-apply.** Detection only **proposes** candidate pairs; a transaction is excluded from totals **only after the user confirms**. Nothing is hidden without confirmation. | ADR-0006 |
| INV-2 | **What matches.** A candidate is one **debit** (`amount_minor < 0`) and one **credit** (`amount_minor > 0`) of **equal magnitude** (`debit.amount_minor = -credit.amount_minor`), in **different** accounts, dated within **¬±`TRANSFER_WINDOW_DAYS` (= 3)** days **inclusive**. | ADR-0006; FIBR-0007 sign convention |
| INV-3 | **Decided pairs don't resurface.** A pair the user has **confirmed or rejected** is never offered again. Rejections are remembered (a `'rejected'` row) so a coincidental same-amount match stays dismissed. | ADR-0006 "remembers rejected pairs" |
| INV-4 | **One transfer per transaction.** A transaction belongs to **at most one confirmed transfer**; once confirmed it is excluded from all further candidate detection (an ambiguous debit that matched two credits resolves to the one the user picked). | ADR-0006 correctness |
| INV-5 | **Exclusion primitive.** `confirmed_transfer_txn_ids()` returns the set of transaction ids in a confirmed pair ‚Äî the single primitive FIBR-0012 totals against to drop internal movement. (No totals consumer exists yet; this phase only creates the primitive.) | success criterion 3 |
| INV-6 | **Unlink is reversible.** Unlinking a confirmed transfer deletes its row; the pair becomes undecided and re-surfaces as a candidate ‚Äî a mis-confirm is fully recoverable. A **spec-added** safety net consistent with ADR-0006's "never silently hide a real expense" spirit (the ADR does not itself mandate unlink). | ADR-0006 (spirit) |
| INV-7 | **Live detection.** Candidates are recomputed from the vault **every time the Transfers tab is shown** ‚Äî no stored candidate state ‚Äî so imports / edits / deletes are always reflected without a manual scan. | FIBR-0052 D4 (tab self-refresh) |
| INV-8 | **Statement-delete cascade.** Deleting a statement (and its transactions, FIBR-0052 INV-9) removes any `transfer_pairs` rows referencing those transactions automatically (`ON DELETE CASCADE`), so there are no dangling links and the delete is never FK-blocked. | FIBR-0052 INV-9; `vault.py` `PRAGMA foreign_keys = ON` |
| INV-9 | **Schema v7 ‚Üí v8.** One new `transfer_pairs` table; `LATEST_SCHEMA_VERSION` 7 ‚Üí 8; the migration is one atomic unit. | FIBR-0005 INV-4 |
| INV-10 | **Transfers tab.** A 6th tab lists **suggested** pairs with **Confirm / Reject / Confirm all** and **confirmed** transfers with **Unlink**. Actions apply **directly** (no modal ‚Äî they are reversible or low-harm); every slot catches `VaultLockedError`. | FIBR-0052 INV-3/4; FIBR-0065 |
| INV-11 | **No network; i18n.** No network. Every user string `tr()`, every widget in a layout manager. | coding.md ¬ß 5.2; security-model INV-8 |
| INV-12 | **Transactions untouched.** Transfer detection never inserts, deletes, or edits a transaction ‚Äî it only records the user's confirm/reject decisions. Row existence stays import's concern. | FIBR-0010 INV-12 (same spirit) |

## Context

Verified against source 2026-07-12 (see **Verified code basis**):

- **`transactions`** (v2 rebuild; v6 `statement_period_id`; v7 category link) has
  `id, account_id, occurred_on, amount_minor, description, created_at,
  statement_period_id, category_id, category_source`. The `Transaction` dataclass
  is **eight** fields (`statement_period_id` is a table column **not** in the
  dataclass); `TransactionRepository.list_all` `SELECT`s the eight in order into
  `Transaction(*row)`.
- **Sign convention (FIBR-0007).** `amount_minor` is a **signed** integer: a
  **debit is negative** (money out), a **credit is positive** (money in) ‚Äî
  `csv_importer._debit_credit_amount` returns `-magnitude if debit else magnitude`,
  and `parse_transaction` forbids a **zero** amount. So a real inter-account
  transfer is exactly *one negative row in account A and one positive row of the
  same magnitude in account B*.
- **Dates.** `occurred_on` is a `TEXT` ISO-8601 date (`YYYY-MM-DD`,
  `date.fromisoformat`-validated at the form boundary). SQLite's `julianday()`
  parses this directly, so the ¬±3-day window is `abs(julianday(a) - julianday(b))
  <= 3`.
- **Deletes.** Transactions are removed only by **statement delete**
  (`TransactionRepository.delete_for_statement`, run inside `StatementService`'s
  owned transaction with `PRAGMA foreign_keys = ON`). There is **no** per-row
  transaction-delete or transaction-edit surface today ‚Äî so a confirmed pair can
  only be invalidated by its statement being deleted, which INV-8 handles.
- **Shell** (`main_window.py`): a **5-tab** `QTabWidget` workspace
  (Home/Statements/Accounts/Categories/Rules, index constants `_TAB_HOME = 0` ‚Ä¶
  `_TAB_RULES = 4`). `_build_workspace` builds each tab once from `self._service`
  (an `AuthService`); `_refresh_tab(index)` refreshes the activated tab;
  `_clear_live` nulls every tab reference on lock. Tab actions are built once by
  `_make_action("action_rules", self.tr("Rules"), "rules", self._open_rules)` and added
  to the View menu + toolbar; `_open_rules` sets the current index. Tab widgets
  (`RulesWidget(service: AuthService)`) construct their own services from
  `service.vault` and guard each slot with `except VaultLockedError: return`.

## Design decisions

- **D1 ‚Äî Schema v8, one decision table (INV-9).** `_migrate_to_v8`, one atomic
  unit mirroring `_migrate_to_v7` exactly (`with owned_transaction(conn):`, the
  `CREATE TABLE` then `UPDATE schema_version SET version = 8` as the last
  statement, `ROLLBACK` on exception via the context manager). The single table:

  ```sql
  CREATE TABLE transfer_pairs(
      id INTEGER PRIMARY KEY,
      txn_a_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
      txn_b_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
      status TEXT NOT NULL,                 -- 'confirmed' | 'rejected'
      created_at TEXT NOT NULL,
      UNIQUE(txn_a_id, txn_b_id));
  ```

  Both the **confirmed link** and the **remembered rejection** are the identical
  shape ‚Äî an ordered transaction pair + a decision ‚Äî so one table with a `status`
  column is the shortest correct model (coding.md ¬ß 1.1: no second table for
  no gain). `ON DELETE CASCADE` (**unlike** FIBR-0010's service-owned category
  cascade) is correct here because removing the row is the *entire* cleanup when a
  referenced transaction is deleted ‚Äî there is no source-reset or re-apply to do ‚Äî
  so the FK does it automatically and `delete_for_statement` needs **no** change
  (INV-8).

- **D2 ‚Äî Models.** `TransferStatus(StrEnum)` (`CONFIRMED = "confirmed"`,
  `REJECTED = "rejected"`) alongside `AccountType`/`CategoryKind`/`CategorySource`
  in `models.py`; a `TransferPair` dataclass (`id, txn_a_id, txn_b_id, status,
  created_at`) whose field order matches the repository `SELECT` for
  `TransferPair(*row)`.

- **D3 ‚Äî Detection is one SQL self-join (single source of truth for INV-2).**
  `TransferRepository.candidate_pairs()` returns `(debit_id, credit_id)` tuples:

  ```sql
  SELECT d.id, c.id
  FROM transactions d
  JOIN transactions c
    ON c.amount_minor = -d.amount_minor        -- equal magnitude, opposite sign
   AND c.account_id  <> d.account_id           -- different accounts
   AND abs(julianday(c.occurred_on) - julianday(d.occurred_on)) <= :window  -- := TRANSFER_WINDOW_DAYS
  WHERE d.amount_minor < 0 AND c.amount_minor > 0   -- d = debit, c = credit (fixes roles)
    AND d.id NOT IN (SELECT txn_a_id FROM transfer_pairs WHERE status = 'confirmed'
                     UNION SELECT txn_b_id FROM transfer_pairs WHERE status = 'confirmed')
    AND c.id NOT IN (SELECT txn_a_id FROM transfer_pairs WHERE status = 'confirmed'
                     UNION SELECT txn_b_id FROM transfer_pairs WHERE status = 'confirmed')
    AND NOT EXISTS (SELECT 1 FROM transfer_pairs p    -- min/max are 2-arg SCALAR here,
                    WHERE p.txn_a_id = min(d.id, c.id) AND p.txn_b_id = max(d.id, c.id))  -- not the 1-arg aggregate
  ORDER BY d.occurred_on, d.id, c.id;
  ```

  The `:window` bind is `TRANSFER_WINDOW_DAYS` (D5 / Deliverable 4) ‚Äî the literal
  `3` **must not** be hard-coded in the query, so INV-2's "named constant" claim is
  load-bearing, not decorative (change the constant ‚Üí the window changes).
  The `amount_minor < 0` / `> 0` split **fixes** `d` as the debit and `c` as the
  credit, so each unordered transfer appears **once** ‚Äî no direction-dedup needed.
  The two `NOT IN` clauses enforce INV-4 (a txn already in a confirmed pair is
  consumed); the `NOT EXISTS` enforces INV-3 (a specific confirmed-or-rejected
  pair never re-surfaces). The rule lives in this one query ‚Äî **no** duplicate
  pure predicate (coding.md ¬ß 1.3); it is tested against a seeded in-memory vault
  (window edges at day offsets 0/3/4, off-by-one-minor amounts, same-account,
  same-sign).

- **D4 ‚Äî Canonical ordering.** `confirm`/`reject` store `min(a, b)` as `txn_a_id`
  and `max(a, b)` as `txn_b_id`, so a pair has one representation regardless of
  which side the caller passes and `UNIQUE(txn_a_id, txn_b_id)` holds. **Direction
  is not stored** ‚Äî which side is "from" (debit) vs "to" (credit) is recovered
  from each transaction's sign at read time.

- **D5 ‚Äî `TransferDetectionService` (`services/transfer_detection.py`,
  vault-scoped) mirrors `CategorizationService`** (the name design.md + ROADMAP
  already use for this component). `_conn` (a **property**, mirroring
  `CategorizationService._conn`) / a `TransferRepository` accessor, plus:
  - `candidates() -> list[TransferCandidate]` ‚Äî runs D3, resolves each pair's two
    `Transaction` rows via a `{txn.id: Transaction}` map built once per call from
    `TransactionRepository.list_all()` (there is **no** get-by-id; same resolve-once
    pattern as the account-name map, and shared by `confirmed_transfers()`), the
    display magnitude (via `read_minor_unit_exponent` + `to_display_decimal`,
    reused ‚Äî the **single base-currency** exponent from `settings`, **not**
    per-account, so both sides share it), and both account names (an id‚Üíname map
    resolved once, mirroring `list_transactions`).
  - `confirm(debit_id, credit_id)` ‚Äî insert a `'confirmed'` row (canonical order).
    **Guards to `ValueError`** (never a raw `sqlite3.IntegrityError`) when either
    txn is already in a confirmed pair **or** the canonical pair is already decided
    (confirmed *or* rejected ‚Äî the `UNIQUE(txn_a_id, txn_b_id)` spans both statuses,
    so an undecided-check must precede the insert). The UI only offers valid
    candidates, so these guards are defensive.
  - `reject(debit_id, credit_id)` ‚Äî insert a `'rejected'` row (canonical order);
    **same undecided-pair guard** as `confirm` (a re-reject or a reject of an
    already-confirmed pair raises `ValueError`, not `IntegrityError`).
  - `unlink(pair_id)` ‚Äî delete the confirmed row (INV-6).
  - `confirmed_transfers() -> list[ConfirmedTransfer]` ‚Äî for the tab's confirmed
    list + Unlink (pair id + both rows + display + from/to names).
  - `confirmed_transfer_txn_ids() -> set[int]` ‚Äî the INV-5 exclusion primitive.

- **D6 ‚Äî Repository writes are per-decision commits.** `confirm`/`reject`/`unlink`
  are each a **single** `INSERT`/`DELETE` ‚Äî atomic on their own ‚Äî so
  `TransferRepository` commits per write (like `TransactionRepository.add`), not
  service-owned (there is no multi-statement unit to wrap). `candidate_pairs`,
  `list_confirmed`, and `confirmed_txn_ids` are reads.

- **D7 ‚Äî "Confirm all" is greedy + consumption-safe (INV-4).** Iterate the current
  `candidates()` in order (deterministic ‚Äî `candidate_pairs` is `ORDER BY
  d.occurred_on, d.id, c.id`); for each, skip if its debit or credit was **already
  consumed** by an earlier confirm in the same pass (tracked in a local set), else
  confirm; report the count confirmed in the status label. So a debit matching two
  credits is confirmed against the **first** and its second candidate is dropped ‚Äî
  never a double-link. `confirm_all() -> int` returns the count committed (the
  status label shows it). It does **not** catch per-item errors: the consumed-set
  makes the INV-4 `ValueError` guard **unreachable within a pass**, so no error is
  expected; a hypothetical unexpected failure propagates and ‚Äî since each `confirm`
  is its own commit (D6) ‚Äî earlier confirms persist. The slot (D8) still catches
  `VaultLockedError`. This keeps D7 testable: a test asserts the returned count and
  that a debit matching two credits yields exactly **one** confirmed pair. The
  **Confirm all** button is enabled only when there is ‚â•1 pending candidate.

- **D8 ‚Äî No modal dialogs in the tab.** Confirm / Reject / Confirm-all / Unlink are
  **direct** button actions with status-label feedback ‚Äî no `QMessageBox`
  confirmation. Rationale: **Unlink** makes a confirm reversible (INV-6), and a
  **Reject** is low-harm (it merely keeps a pair counting as income+expenditure ‚Äî
  the safe direction of error, versus a mis-confirm which Unlink undoes). This
  keeps the auto-lock story trivial (no nested event loop to tear down ‚Äî the
  FIBR-0065 class of bug cannot arise here); each slot still catches
  `VaultLockedError` and returns, exactly like `RulesWidget._on_delete`.

- **D9 ‚Äî TransfersWidget (`ui/transfers.py`, `AuthService`) mirrors RulesWidget.**
  Two `QTableWidget`s in a `QVBoxLayout`:
  - **Suggested transfers** (`objectName "transfers_suggested"`) ‚Äî columns Date /
    Amount / From ‚Üí To / Description; buttons **Confirm**
    (`"transfers_confirm"`), **Reject** (`"transfers_reject"`) ‚Äî enabled on a
    single-row selection ‚Äî and **Confirm all** (`"transfers_confirm_all"`, enabled
    when ‚â•1 candidate).
  - **Confirmed transfers** (`objectName "transfers_confirmed"`) ‚Äî the same
    columns; an **Unlink** button (`"transfers_unlink"`, enabled on selection).

  **Column sourcing (single-valued cells, since a pair is two rows that may differ
  by up to `TRANSFER_WINDOW_DAYS` and by description):** **Date** and
  **Description** are the **debit** row's; **From** = the debit's account name,
  **To** = the credit's; **Amount** = the shared display magnitude. (Object names +
  fixed sourcing make the INV-10 `qtbot` cells deterministically assertable.)

  The widget sets `objectName("tab_transfers")` (INV-10, mirroring
  `RulesWidget.setObjectName("tab_rules")`). Parallel `self._candidates` /
  `self._confirmed` lists track the table rows (the `RulesWidget._rows` pattern);
  `_refresh()` re-reads both via the service; `_selected_row()` / selection-driven
  button enabling reused. A **status `QLabel`** reports outcomes ("Confirmed %n
  transfer(s).", "Rejected.", "Unlinked."). A new
  **`transfers.svg`** glyph in the icon set (loaded by `_make_action`; travels via
  the existing `*.svg` package-data glob).

- **D10 ‚Äî Shell wiring (`ui/main_window.py`).** Add `_TAB_TRANSFERS = 5`. In
  `_build_workspace`, build `self._transfers_tab = TransfersWidget(self._service)`
  and `addTab(..., self.tr("Transfers"))` as the **6th** tab; add the
  `_transfers_tab = None` line to `_clear_live`; add the `_refresh_tab` branch
  (`elif index == _TAB_TRANSFERS ... self._transfers_tab._refresh()`). Build
  `self._action_transfers = self._make_action("action_transfers",
  self.tr("Transfers"), "transfers", self._open_transfers)`, add it to
  `_menu_view` and the toolbar action list, and add `_open_transfers` (sets the
  index via `_ensure_workspace()`). **No** special post-import hook ‚Äî INV-7's
  live-on-view detection means the new candidates appear the moment the user opens
  the tab.

## Invariants

- **INV-1 ‚Äî Suggest, never auto-apply.** `candidates()` is the only producer of
  proposed pairs; a transaction enters `confirmed_transfer_txn_ids()` **only**
  through `confirm()`. No code path confirms without a user action. Test: seed a
  perfect debit/credit pair ‚Üí it appears in `candidates()` but
  `confirmed_transfer_txn_ids()` is empty until `confirm()`.
- **INV-2 ‚Äî What matches.** Exactly the D3 predicate. Tests (seeded vault):
  a matching pair at day offsets 0 and 3 **matches**; offset 4 **does not**;
  amounts off by one minor unit **do not**; two debits (same magnitude, both
  negative) **do not**; a same-account opposite pair **does not**;
  `TRANSFER_WINDOW_DAYS = 3` is the named constant.
- **INV-3 ‚Äî Decided pairs don't resurface.** After `reject(d, c)`, the pair is
  absent from `candidates()`; after `confirm(d, c)`, likewise. A *different*
  same-amount credit still surfaces for the same debit **only while the debit is
  unconfirmed** (INV-4).
- **INV-4 ‚Äî One transfer per transaction.** `confirm` on a txn already in a
  confirmed pair raises `ValueError`; after `confirm(d, c1)`, `d` (and `c1`) no
  longer appear in `candidates()`, so `(d, c2)` is gone. `confirm_all` on a debit
  with two candidate credits confirms exactly one.
- **INV-5 ‚Äî Exclusion primitive.** `confirmed_transfer_txn_ids()` == the union of
  both ids of every `'confirmed'` row (and **only** confirmed ‚Äî a rejection
  excludes nothing). Test: confirm one pair, reject another ‚Üí the set is exactly
  the two confirmed ids.
- **INV-6 ‚Äî Unlink reversible.** After `confirm` then `unlink`, the pair is back
  in `candidates()` and out of `confirmed_transfer_txn_ids()`.
- **INV-7 ‚Äî Live detection.** `candidates()` reads live vault state each call (no
  cache); a newly-inserted matching pair appears on the next call with no other
  action. (`_refresh()` on tab activation is the UI expression of this.)
- **INV-8 ‚Äî Statement-delete cascade.** Confirm a pair whose two txns belong to a
  statement, then delete the statement ‚Üí the `transfer_pairs` row is gone (no FK
  error, no dangling row). Test drives `TransactionRepository.delete_for_statement`
  (which runs under `PRAGMA foreign_keys = ON`, so the cascade fires) and asserts
  the count.
- **INV-9 ‚Äî Schema v7 ‚Üí v8.** A v7 vault migrates to v8 with `transfer_pairs`
  present and empty; `LATEST_SCHEMA_VERSION == 8`; a mid-step failure leaves a
  re-openable v7 (atomic). Test mirrors the existing migration tests (a
  `build_v7_vault` helper if needed).
- **INV-10 ‚Äî Transfers tab.** The workspace has **6** tabs; the Transfers tab
  (`objectName "tab_transfers"`) shows the two tables + buttons; Confirm/Reject/
  Confirm-all/Unlink drive the service and refresh; an auto-lock mid-click
  (`VaultLockedError`) is caught, no crash.
- **INV-11 ‚Äî No network; i18n.** No import under `src/` reaches the network
  (the existing `test_INV8_no_network_imports_under_src` AST scan still passes ‚Äî
  `transfers.py`/`ui/transfers.py` add none); every user string is `tr()`,
  every widget sits in a layout manager.
- **INV-12 ‚Äî Transactions untouched.** No transfer operation issues an
  `INSERT/UPDATE/DELETE` against `transactions`. Test: transaction rows (count,
  amounts, categories) are byte-identical before and after confirm/reject/unlink.

## Deliverables

1. **`models.py`** ‚Äî `TransferStatus(StrEnum)` (D2); `TransferPair` dataclass
   (`id, txn_a_id, txn_b_id, status, created_at`); the two read-model dataclasses
   the service returns, **field order pinned** (mirrors FIBR-0010's pin-every-field
   discipline so the widget + tests can't diverge):
   - `TransferCandidate(debit: Transaction, credit: Transaction, display_amount:
     Decimal, from_account: str, to_account: str)`.
   - `ConfirmedTransfer(pair_id: int, debit: Transaction, credit: Transaction,
     display_amount: Decimal, from_account: str, to_account: str)`.
   The two `Transaction` rows are named `debit` (the negative side) and `credit`
   (the positive side); `from_account`/`to_account` are the debit's / credit's
   account names; `display_amount` is the shared positive magnitude.
2. **`migrations.py`** ‚Äî `_migrate_to_v8` (D1); `LATEST_SCHEMA_VERSION = 8`;
   `_MIGRATIONS[8]`.
3. **`repositories/transfers.py`** ‚Äî `TransferRepository`:
   `candidate_pairs() -> list[tuple[int, int]]` (D3, `(debit_id, credit_id)`),
   `add_decision(txn_a_id, txn_b_id, status)` (canonical order, per-write commit,
   D4/D6), `delete_confirmed(pair_id)`, `list_confirmed() -> list[TransferPair]`,
   `confirmed_txn_ids() -> set[int]`, `is_confirmed(txn_id) -> bool` (the INV-4
   guard helper), and `pair_decided(txn_a_id, txn_b_id) -> bool` (the D5
   undecided-pair guard ‚Äî any-status canonical-pair existence).
4. **`services/transfer_detection.py`** ‚Äî `TransferDetectionService` (D5):
   `candidates`, `confirm`, `reject`, `unlink`, `confirmed_transfers`,
   `confirmed_transfer_txn_ids`, `confirm_all() -> int` (D7). The
   `TRANSFER_WINDOW_DAYS = 3` constant lives in **`repositories/transfers.py`**
   (alongside `candidate_pairs`, its **only** consumer, which binds `:window`
   against it ‚Äî one home, so the query and INV-2 can't drift); the service imports
   it from the repository (the normal service‚Üírepository direction), never the
   reverse. (`add_decision` is a plain `INSERT` and binds no window.)
5. **`ui/transfers.py`** ‚Äî `TransfersWidget(service: AuthService)` (D9).
6. **`ui/icons/transfers.svg`** ‚Äî the tab/action glyph (D9), matching the

   `--add-data`.
7. **`ui/main_window.py`** ‚Äî `_TAB_TRANSFERS = 5`; the 6th tab in

## Data model ‚Äî schema v7 ‚Üí v8

```sql
CREATE TABLE transfer_pairs(
    id INTEGER PRIMARY KEY,
    txn_a_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    txn_b_id INTEGER NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    status TEXT NOT NULL,               -- 'confirmed' | 'rejected'
    created_at TEXT NOT NULL,
    UNIQUE(txn_a_id, txn_b_id));
```

Canonical ordering `txn_a_id < txn_b_id` (D4). `ON DELETE CASCADE` on both FKs so
a statement delete (with `PRAGMA foreign_keys = ON`) auto-clears links (INV-8).
No backfill (a fresh table). `LATEST_SCHEMA_VERSION` ‚Üí 8.
