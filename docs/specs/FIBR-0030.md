## 1. Problem & intent

finbreak has **no backdoor** (`security-model.md` T11): a forgotten master
password is unrecoverable. The escape hatches are the hint (FIBR-0029, a
reminder) and an encrypted backup restore (FIBR-0014). When **all** of
those fail — the user has genuinely lost the password and has no usable
backup — the app is a permanent brick with their (undecryptable) data
locked inside. There is currently no way to get back to a working app
short of manually deleting files on disk.

FIBR-0030 adds the **last resort**: from the unlock screen, irreversibly
delete the vault and return to first-run setup so the user can start fresh.
This destroys the old data — which is **already unrecoverable** anyway
(without the key it cannot be decrypted, T11), so nothing of value is lost
that wasn't already lost. The entire risk is a user triggering it by
**accident**, so the design is dominated by **friction**: a clear
irreversible-warning **and** a second explicit "type DELETE" step before
anything is removed.

## 2. Scope

**In scope**
- A new `AuthService.reset_vault()` primitive that deletes the vault's
  **on-disk data footprint** (DB + KDF sidecar + both SQLite WAL sidecars)
  and wipes the in-memory key (§ 3.3).
- A **"Forgot password? Start over…"** flat button on the unlock screen
  (`UnlockDialog`), modelled on the existing restore button (§ 3.1).
- A **double confirmation** owned by the shell: Step 1 an
  irreversible-warning message box, Step 2 a purpose-built dialog whose OK
  is gated on the user typing the exact word `DELETE` (§ 3.2).
- Clearing the **vault-coupled** plaintext `window.ini` keys (unlock
  throttle counters + password hint) on reset, while leaving benign UI
  state (§ 3.4).
- Returning to first-run setup after deletion (§ 3.5).
- Recording the deletion primitive in `security-model.md` (§ 3.6).

**Out of scope**
- Any *recovery* of the old data (impossible by design — T11).
- A user-facing "change master password" flow (none exists — the
  `Vault.rekey` primitive at `vault.py:244` is used *only* by FIBR-0014
  restore, not exposed to the user). Reset is delete-and-recreate, not
  re-key.
- Exporting / archiving the old (undecryptable) vault before deletion —
  pointless, it can't be read.
- An undo / recycle-bin for the deleted vault (defeats the "irreversible"
  contract; the friction is the safety, not reversibility).

## 3. Design

### 3.1 The affordance (unlock screen, flat button + signal)

`ui/unlock.py` `UnlockDialog` already carries a flat, shell-driven
affordance — the restore button (recon-verified):
- signal `restore_requested = Signal()` (`unlock.py:46`),
- `self._restore_button = QPushButton(...)` + `setObjectName(...)` +
  `setFlat(True)` (`:67-71`), added to the layout at `:103`, and
  `self._restore_button.clicked.connect(self.restore_requested)` (`:108`);
  it is disabled during an in-progress derivation at `:180`
  (`setEnabled(not busy)`).

Mirror it exactly with a **second** flat button:
- a new signal `start_over_requested = Signal()`;
- `self._start_over_button = QPushButton(self.tr("Forgot password? Start over…"))`
  (label on one line — do not break the `tr()` literal across lines),
  then `setObjectName("unlock_start_over")`, `setFlat(True)`, added to the
  layout **after** the restore button (it is the more drastic option, so it
  sits below the gentler "restore" one);
- `self._start_over_button.clicked.connect(self.start_over_requested)`;
- disabled while a derivation runs — add its own `setEnabled(not busy)`
  beside the restore button's existing one (`unlock.py:180`) in the
  busy-handler (no reset while a derivation is running).

The button is **always present** on the unlock screen (unlike the hint's
show-button, whose presence is data-dependent) — "start over" must be
reachable precisely when the user has nothing (no hint, no password), so
it cannot be gated on any stored state.

The **shell owns the flow** — `UnlockDialog` only emits the signal, holds
no `AuthService` reference for it (like `restore_requested`). Add
`dialog.start_over_requested.connect(self._on_start_over)` beside the
existing restore wiring in `_open_unlock_dialog()` (`main_window.py:537`).
**Wire it there only.** `restore_requested` is wired at *two* sites —
`:537` (`UnlockDialog`) **and** `:522` (`FirstRunDialog`) — but start-over
must be wired at `:537` **only**: first-run has no vault to reset, so
`FirstRunDialog` neither has nor should emit `start_over_requested`. Do not
"mirror restore exactly" into the `:522` site.

### 3.2 Double confirmation (shell-owned, two explicit steps)

`main_window._on_start_over()` runs two gates; **either cancel aborts with
zero filesystem change** (nothing is deleted until both pass):

**Step 1 — irreversible warning (blocking message box).** Modelled on the
statements delete-confirm (`ui/statements.py:231-236`, `QMessageBox` →
check the return), but with a **destructive-safe default**: because this is
irreversible, pass explicit buttons and default to the *safe* one so a
stray Enter cannot proceed —

```python
choice = QMessageBox.warning(
    self,
    self.tr("Start over?"),
    self.tr(
        "This permanently erases your vault and everything in it. "
        "It cannot be undone, and the data cannot be recovered "
        "afterwards. Only do this if you have lost your password and "
        "have no backup."
    ),
    QMessageBox.StandardButton.Cancel | QMessageBox.StandardButton.Yes,
    QMessageBox.StandardButton.Cancel,  # default = the safe choice
)
if choice != QMessageBox.StandardButton.Yes:
    return
```

*(This deviates from `statements.py`'s no-explicit-buttons `.question`
call on purpose — a reversible single-row delete can default to Yes; an
irreversible whole-vault wipe must default to Cancel. Documented so it does
not read as accidental drift from the sibling pattern.)*

**Step 2 — type-DELETE gate (purpose-built modal dialog).** A new
`StartOverDialog(parent)` (`ui/start_over.py`), parented to the shell
(`self`, the `MainWindow`). Steps 1–2 deliberately **stack over the
still-open `UnlockDialog`** (which is torn down only after a *successful*
reset, § 3.5) — this is fine (Qt nests modals). Unlike every other dialog
in the app, `StartOverDialog` is **`exec()`-driven** (a nested modal loop
for the synchronous accept read-back) rather than the codebase's usual
`_open_dialog`+signal-read-back convention — an accepted, deliberate
departure for a one-shot confirm. It borrows **only the validation-gating
idiom** from `ui/backup_export.py` (`:57-62` for the `QDialogButtonBox` +
captured OK button; `:79-81`/`:91-106` for the textChanged→sync→setEnabled
gate):
- restates the warning prominently (a bold/emphasised label — the message
  box is transient, this one is read while typing);
- a `QLineEdit` labelled *"Type DELETE to confirm"*;
- a module constant `CONFIRM_WORD = "DELETE"`. **It is deliberately
  locale-independent** (a plain non-`tr()` literal). The field label may be
  `tr()`-translated, but the word the user types must stay the literal
  Latin `DELETE` — so the label must instruct typing `DELETE` verbatim
  (e.g. keep `DELETE` un-translated inside the `tr()` string). If the
  label's "DELETE" were localized while `CONFIRM_WORD` stayed `"DELETE"`,
  the OK gate could never enable — an explicit contract, not an accident.
- an OK button (captured as `self._ok`, text *"Erase everything"*) from a
  `QDialogButtonBox(Ok | Cancel)`, **disabled until the field text equals
  `CONFIRM_WORD` exactly** — case-sensitive, no surrounding whitespace
  tolerated (`self._field.text() == CONFIRM_WORD`). The gate re-checks on
  `textChanged` and once at construction (initial state disabled), exactly
  as `backup_export._sync_ok` does.
- **Accept/reject wiring (load-bearing — the accept differs from the
  model):** wire `buttons.accepted.connect(self.accept)` **and**
  `buttons.rejected.connect(self.reject)` (the latter as `backup_export.py:63`
  does, so Cancel actually aborts). The `accept` line is where this dialog
  departs from the model: `backup_export`'s OK (`:62`/`:109-111`) is wired
  to `_on_ok`, which *emits a signal* (`export_requested`) and never calls
  `self.accept()` (that dialog is signal-driven, its shell reads it back).
  `StartOverDialog` is `exec()`-driven, so it must `accept()` itself —
  mirroring `backup_export`'s OK wiring verbatim would make `exec()` never
  return `Accepted` and silently dead-end the delete.

Skeleton (the load-bearing lines pinned):

```python
CONFIRM_WORD = "DELETE"

class StartOverDialog(QDialog):
    def __init__(self, parent: QWidget | None = None):
        super().__init__(parent)
        self.setWindowTitle(self.tr("Start over — erase everything"))
        self._field = QLineEdit()
        buttons = QDialogButtonBox(
            QDialogButtonBox.StandardButton.Ok | QDialogButtonBox.StandardButton.Cancel
        )
        self._ok: QPushButton = buttons.button(QDialogButtonBox.StandardButton.Ok)
        self._ok.setText(self.tr("Erase everything"))
        # capture Cancel too, so INV-4's real-widget leg can click it directly
        self._cancel: QPushButton = buttons.button(QDialogButtonBox.StandardButton.Cancel)
        buttons.accepted.connect(self.accept)   # exec()-driven — differs from backup_export
        buttons.rejected.connect(self.reject)   # so Cancel actually aborts
        self._field.textChanged.connect(self._sync_ok)
        # ... a bold warning label + the field + buttons into a layout ...
        self._sync_ok()  # initial state: disabled

    @Slot()
    def _sync_ok(self) -> None:
        self._ok.setEnabled(self._field.text() == CONFIRM_WORD)
```

The shell `exec()`s it and **disposes it** on return
(`dialog.deleteLater()`), since it lives outside the managed `_dialog`
slot — otherwise repeated cancel/retry cycles leak hidden `MainWindow`
children.

Only when `StartOverDialog.exec()` returns `Accepted` does the shell call
`reset_vault()`. That call is wrapped so a failure surfaces gracefully —
see § 3.3 (failure-mode) and § 3.5.

### 3.3 `AuthService.reset_vault()` (new primitive — the complete footprint)

There is **no** existing reset method (recon-verified: grep empty across
`src/` + `tests/`). Add to `AuthService`:

```python
def reset_vault(self) -> None:
    """Irreversibly delete the vault and its on-disk data footprint
    (DB, KDF sidecar, and both SQLite WAL sidecars), then leave the
    service in a locked, first-run state. Safe to call while locked —
    the unlock screen is the only caller, so the vault is typically
    never-unlocked when this runs."""
    self.lock()  # idempotent when already locked (see below); releases any
                 # DB handle and wipes self._key if one is held
    vault_path = self._vault.vault_path
    sidecar_path = self._vault.sidecar_path
    for path in (
        vault_path,
        sidecar_path,
        # SQLite runs in WAL mode (finbreak/vault.py:104 create / :174 open),
        # so it writes `<db>-wal` and `<db>-shm` sidecars beside the DB. They
        # hold the vault's most recent (encrypted) transactions — part of the
        # user's data — and NO path helper names them, so nothing else unlinks
        # them today. "Start over" must remove them too: primarily so no
        # fragment of the old vault's data survives the delete, and
        # defensively so an orphaned -wal can't interact with the next,
        # differently keyed vault created at first-run.
        vault_path.with_name(vault_path.name + "-wal"),
        vault_path.with_name(vault_path.name + "-shm"),
    ):
        path.unlink(missing_ok=True)
    log.info("vault reset")
```

Design notes (each recon-verified):
- **`lock()` is safe when already locked** — invoked from the unlock
  screen the vault was never opened and `self._key is None`, yet
  `lock()` (`auth.py:300-305`) is fully idempotent: `_stop_timer` guards
  `if self._timer is not None`; `Vault.close` (`vault.py:190-194`) guards
  `if self._conn is not None`; `_wipe(None)` (`auth.py:146-150`) is a
  no-op (`if buffer:`). So calling it first both (a) releases the file
  handle so the unlink is Windows-safe and (b) wipes the key on the rare
  path where a caller resets while unlocked.
- **Paths come from the live `Vault`** (`self._vault.vault_path` /
  `.sidecar_path`), **not** the module-level `paths.vault_path()` — so a
  test constructing `AuthService(tmp/vault.db, tmp/vault.kdf.json)` deletes the
  test files, not the real user data dir. (`main_window` already sources
  the paths the same way at `:992-993`.)
- **`missing_ok=True`** makes reset idempotent and robust to a
  partial/crashed prior state *on entry* (e.g. sidecar already gone, or the
  -wal was already checkpointed away by a clean close) — deleting a
  complete or a partial footprint both converge to "gone". It does **not**
  cover a crash **between** reset's own two primary unlinks (`vault_path`
  then `sidecar_path`): that leaves exactly one of the pair present, a
  mixed state whose concrete consequence is severe — `MainWindow.__init__`
  reads `service.state()` at construction (`main_window.py:240`), which
  **raises `VaultStateError` on a mixed pair and propagates to `run()`'s
  guard so the window is never shown**, with no in-app recovery
  (`_reconcile_interrupted_restore` won't help — it only handles the
  distinct `*.old` interrupted-restore signature). The window is two
  adjacent `unlink` syscalls with no I/O between — an accepted negligible
  risk, but called out with its real outcome so "robust" isn't read as
  covering it.
- **No `paths.py` change.** The `-wal`/`-shm` suffixes are a SQLite naming
  convention derived inline from the DB path with an explanatory comment,
  used at exactly this one site — no module API is added for a single
  caller (§ 2 shortest-correct). The falsifiable guarantee is the
  behavioural INV-1 test, not a pure-helper unit test.
- **No `-journal` sidecar to delete.** WAL mode is always on
  (`vault.py:104`/`:174`), so SQLite uses the `-wal`/`-shm` pair, never a
  rollback `-journal` — the four-file footprint is complete as listed.
- **Layering:** `reset_vault` is crypto/filesystem only. Clearing the
  `window.ini` keys (§ 3.4) is a QSettings/ui concern and stays in the
  shell — `AuthService` (services layer) must not import Qt-`ui`.
- **Unlink-failure assumption.** `reset_vault` assumes no *other* process
  holds the vault open — finbreak is a single-user desktop app and `lock()`
  has already closed *this* process's handle, so the unlinks succeed on
  POSIX and Windows alike. A hypothetical second concurrent instance
  holding `vault.db` open would make `unlink` raise `PermissionError`
  (which `missing_ok` does **not** swallow — it tolerates absence only);
  that is out of scope (no second-instance guard exists today). It must
  **not** crash the handler: because Step 2 has already closed by the time
  `reset_vault()` runs, the shell wraps **only the `reset_vault()` call**
  in the `try` — on an `OSError` it shows an error via
  **`QMessageBox.critical(self, …)`** (named so INV-10's test can spy it,
  distinct from Step 1's `QMessageBox.warning`) and leaves the app on the
  unlock screen, rather than letting the exception propagate out of the
  `_on_start_over` slot. **Do not over-promise "intact" in the
  error copy:** the concurrent-instance case fails on the *first* unlink
  (`vault.db` is the held file) so both files do survive — but an `OSError`
  on the *second* unlink would leave the same mixed one-file state as the
  two-syscall window above. So the error message says the reset *failed*
  and to retry, **not** that the vault is guaranteed intact. **Every post-reset step (§ 3.4 key
  clearing, `_teardown_dialog`, `_route_pre_login`) runs only *after*
  `reset_vault()` returns successfully** — never in the failure branch. So
  a failed reset leaves the *old* vault **and** its coupled `window.ini`
  keys (throttle lockout, hint) intact and consistent; the keys are not
  wiped for a vault that still exists.

### 3.4 Clearing vault-coupled window.ini keys (shell, after reset)

The plaintext `window.ini` holds some keys **coupled to the now-deleted
vault** and some **benign** UI preferences. After `reset_vault()` returns
**successfully** (§ 3.3 — never in the failure branch), the shell clears
only the coupled ones (recon-verified adapters):
- **unlock throttle counters** — `UnlockThrottle().reset()`
  (`ui/_unlock_throttle.py:73-79`) removes `unlock/fail_count` and
  `unlock/last_fail`; the fail-lockout was for the *old* vault and must not
  carry into the fresh one.
- **password hint** — `clear_hint()` (`ui/_password_hint.py:42-45`) removes
  `hint/text`; the hint described the *old* password.

**Left untouched** (benign, not vault-secrets): window geometry / state,
last-open tab, theme, and the update-check opt-in. Wiping those would be
gratuitous UX loss — they describe the *app*, not the *vault*.

### 3.5 Return to first-run

Step 2 ran as a **child** modal (`StartOverDialog.exec()`), so the
originating `UnlockDialog` is still the shell's active `_dialog`.
`_on_start_over` must therefore **tear it down first** —
`self._teardown_dialog()` — before routing, exactly as `_open_restore`
(`main_window.py:972-981`) tears down its originating dialog before
swapping in the restore dialog ("one app-modal per `_dialog` slot;
`hide()`+`deleteLater` doesn't re-emit `rejected`"). Skipping this leaks
the `UnlockDialog`: `_show_first_run`'s `_open_dialog` overwrites
`self._dialog` (`:1424`) without disposing the prior modal.

Then the shell calls the existing `_route_pre_login()`
(`main_window.py:1025-1031`): with **both** primary files gone,
`self._service.state()` is now `"first_run"`, so `_route_pre_login`
dispatches to `_show_first_run()` (`:517-524`) and the user lands on the
create-vault flow. (Both files must go together — a
mixed state, one present one absent, makes `Vault.presence_state()` raise
`VaultStateError`; INV-1 deleting the complete footprint guarantees the
clean "both absent" → `"first_run"` state.)

### 3.6 security-model.md impact (a deletion primitive)

The deliverable is **two things** (both required, not either/or): (a) a
short prose note that FIBR-0030 introduces a **user-initiated destructive
reset** which deletes the vault + its KDF sidecar + its SQLite WAL sidecars
and returns to first-run, **and** (b) a **numbered invariant** stating the
deletion-completeness guarantee (falsifiable by INV-1 here).

The note introduces **no new attacker capability and no new plaintext
artefact** — it only *deletes* — so no new asset row or threat row is
required. The reasoning for "no new capability": local filesystem access
by a same-OS-user attacker or an evil-maid is outside the model's scope
(`security-model.md` § 4), and such an actor can delete `vault.db`
directly, so an in-app button adds nothing they lack. (Do **not** cite T1
here: T1 is a *confidentiality* threat — someone *reading* the vault — not
destruction/availability.) The type-DELETE friction targets **accidental**
self-inflicted loss, not an adversary. The honest disclosure is that the
reset is **irreversible by design** (consistent with T11 no-backdoor: the
data was already unrecoverable).

The required invariant (b) is a **deletion-completeness / data-remanence
hygiene** guarantee — deliberately *not* framed as confidentiality (the
deleted `-wal`/`-shm` fragments are old-key-encrypted and useless without
the key, which is why § 3.3 dropped the "or leak" wording). State it as:
*"The destructive reset removes the vault's complete on-disk data footprint
— DB, KDF sidecar, and both SQLite WAL sidecars — so no *file* of a deleted
vault remains to interfere with a subsequently created one."* This is
**logical deletion** (`unlink` removes the directory entry) — deliberately
**not** a secure media wipe; the residual sectors may hold old-key
ciphertext until overwritten, which is fine because it is old-key-encrypted
hygiene, not a confidentiality claim. Do not phrase the INV as an
anti-forensic / secure-erase guarantee `unlink` does not deliver. Number it the next free `security-model.md` INV — at time of writing
the highest is INV-11, so this is **INV-12** — but re-verify the number at
authoring time rather than trusting this note (a later spec may have taken
it first). If `security-model.md` § 5 carries an "enforcement arrives with
the phase" mapping, add the matching row for this INV too.

## 4. Invariants

- **INV-1 — Complete footprint deletion (incl. WAL sidecars).**
  `reset_vault` deletes all four files: `vault.db`, `vault.kdf.json`,
  `vault.db-wal`, `vault.db-shm`. None survives. *Test:* create a vault
  **and close it** (no open connection), then `touch` sidecar files at the
  **literal, hardcoded** paths `tmp/"vault.db-wal"` and `tmp/"vault.db-shm"`
  — hardcode these exact names in the test; do **not** derive them from the
  code's own `name + "-wal"` string. Call `reset_vault`, assert all four
  paths are absent. Hardcoding the literals is what pins the suffix: if
  `reset_vault` derived a wrong suffix (`"-wall"`), it would fail to unlink
  the literal `vault.db-wal` the test created and the assert-absent would
  fail — whereas a test that reused the code's string would mirror the bug
  and pass. *(Note: a **genuine** SQLite `-wal` can't be used to test the
  explicit unlink — `reset_vault`'s first act is `lock()` → `conn.close()`,
  the last connection, and SQLite checkpoints-and-deletes the real
  `-wal`/`-shm` on that clean close (§ 3.3), so the explicit unlink only
  ever runs against **orphaned** sidecars like the ones this test creates.)*
- **INV-2 — Clean slate for the next vault.** After a reset, first-run can
  create and open a brand-new vault under a new password. *Test:* reset,
  then `first_run(new_password, base_currency)` + open → succeeds and the
  new vault is empty. *(This is a **smoke test** that reset leaves a
  creatable slate — it does not by itself reproduce a stale-WAL fault
  (which would need a crafted non-empty old-key `-wal`, out of proportion
  to the risk); INV-1 is the real guarantee that the `-wal`/`-shm` are
  gone. Per § 3.3 the sidecar delete is justified primarily as
  complete-footprint deletion, not a specific corruption fix.)*
- **INV-3 — Safe while locked.** `reset_vault` is invoked from the unlock
  screen (`self._key is None`, connection never opened) and completes
  without error. *Test:* a never-unlocked `AuthService` → `reset_vault()`
  raises nothing and deletes the files.
- **INV-4 — Double confirmation gates the delete.** Deletion happens
  **only** after Step 1 (warning accepted) **and** Step 2
  (`StartOverDialog` accepted). Cancelling either leaves every file intact
  and does **not** call `reset_vault`. *Test:* two shell-level legs —
  cancel-at-Step-1 and cancel-at-Step-2 — both leave the vault present and
  `reset_vault` un-called (patch/spy the two dialogs' return values). *Plus
  a real-widget leg for the dialog's own Cancel* (so the `rejected → reject`
  wiring, § 3.2, is falsifiable and a dead Cancel button is caught):
  construct a real `StartOverDialog` (register it with
  `qtbot.addWidget(dialog)`), then **`qtbot.waitSignal(dialog.rejected)`
  around a `qtbot` click of `dialog._cancel`** — assert the `rejected`
  signal actually fires (equivalently, assert `dialog.isHidden()` after the
  click, per the project's qtbot `isHidden`-not-`isVisible` convention). Do
  **not** assert `result() == Rejected`: `QDialog.result()` is `Rejected`
  (0) from construction, so a *dead* Cancel (missing `rejected → reject`)
  passes that assertion vacuously — and `exec()` would hang headless. The
  fired-signal / `isHidden()` check is the discriminating one.
- **INV-5 — Step-2 OK gated on exact `CONFIRM_WORD`.** The
  `StartOverDialog` OK button is disabled until the field text equals
  `CONFIRM_WORD` (`"DELETE"`) exactly; `"delete"`, `"DELETE "`, `"DEL"`,
  `""` keep it disabled; `"DELETE"` enables it. *Test:* drive the field via
  `qtbot` and assert `ok.isEnabled()` across those inputs.
- **INV-6 — Vault-coupled window.ini keys cleared; benign state kept.**
  After reset, `unlock/fail_count`, `unlock/last_fail`, and `hint/text` are
  removed; a benign key (e.g. window geometry) written beforehand is
  retained. *Test:* seed the three coupled keys plus one benign key (e.g.
  window geometry), run the shell reset handler, assert the three coupled
  keys are absent and the benign key persists. (Like INV-4/7, running
  `_on_start_over` headless needs the Step-1 `QMessageBox.warning`
  staticmethod and `StartOverDialog.exec` monkeypatched — see § 6.)
- **INV-7 — Returns to first-run.** After reset the shell calls
  `_route_pre_login()`, `state() == "first_run"`, and the shown dialog is
  the `FirstRunDialog`. *Test:* shell flow (build `MainWindow`, invoke the
  reset handler with both dialogs auto-accepted) → `window._dialog` is a
  `FirstRunDialog` and both vault files are gone.
- **INV-8 — Key wiped, old data unrecoverable.** `reset_vault` wipes the
  in-memory key (via `lock()`) and retains nothing; with the files deleted
  and no backdoor (T11) the old data cannot be decrypted. *Test:* unlock a
  vault so `self._key` holds real key bytes, capture that `bytearray`, call
  `reset_vault`, then assert the captured buffer is zeroed **and**
  `self._key is None` — this exercises the `lock()`-wipe leg the INV
  claims. (Asserting only `self._key is None` on a *never-unlocked* service
  would prove nothing: it starts at `None` per INV-3.) Unrecoverability of
  the deleted files themselves is the T11 design guarantee, not a
  separately testable leg.
- **INV-9 — Affordance is unlock-screen-only and derivation-aware.** The
  "Start over" button lives on `UnlockDialog` (pre-login), emits
  `start_over_requested` on click, and is disabled while an unlock
  derivation is in progress (shares the `busy` line with the restore
  button). *Test:* `UnlockDialog` exposes the button; clicking it fires
  `start_over_requested`; `setEnabled(not busy)` toggles it.
- **INV-10 — A failed reset is contained (no crash, no partial state
  hygiene).** If `reset_vault()` raises `OSError` after both gates pass,
  `_on_start_over` catches it, shows an error, and runs **none** of the
  post-reset steps (§ 3.3/3.4): the app stays on the unlock screen, the
  coupled `window.ini` keys are left intact, and no routing to first-run
  occurs. *Test:* seed the coupled keys first (as INV-6 does), monkeypatch
  `reset_vault` to raise `OSError`, and drive the handler with all three
  modals patched (Step-1 `QMessageBox.warning`, `StartOverDialog.exec`, and
  the failure-path `QMessageBox.critical` — spy the last). Assert: no
  exception escapes the slot, `QMessageBox.critical` was called (the error
  box fired), `unlock/fail_count`/`hint/text` are still present, and
  `window._dialog` is still the `UnlockDialog` (not a `FirstRunDialog`).

## 5. Exit criteria (falsifiable)

1. **Footprint gone.** `reset_vault` removes DB + KDF sidecar + both WAL
   sidecars (INV-1); a fresh vault then creates cleanly (INV-2). Falsifier:
   any of the four files survives, or a post-reset first-run fails.
2. **Locked-safe.** A never-unlocked service resets without error (INV-3).
   Falsifier: `reset_vault` raises when `self._key is None`.
3. **Double-confirm holds.** Neither Step-1-cancel nor Step-2-cancel
   deletes anything; only both-accepted deletes (INV-4). The Step-2 OK is
   gated on exact `"DELETE"` (INV-5). Falsifier: a single cancel still
   deletes, or a non-`"DELETE"` string enables OK.
4. **State hygiene.** Coupled `window.ini` keys cleared, benign kept
   (INV-6); the app lands on first-run (INV-7). Falsifier: a stale throttle
   lockout or old hint survives, or benign UI prefs are wiped, or the app
   doesn't route to first-run.
5. **Unrecoverable + key wiped.** `self._key is None` post-reset (INV-8);
   the affordance is unlock-only and derivation-aware (INV-9). Falsifier: a
   retained key, or a reset button reachable while unlocked / during
   derivation.
6. **Failure is contained.** An `OSError` from `reset_vault` is caught, an
   error box is shown, and no post-reset step runs — coupled keys intact,
   still on the unlock screen (INV-10). Falsifier: the exception escapes the
   slot, the coupled keys are wiped, or the app routes to first-run on a
   failed reset.
7. **security-model.md updated + cold-eyes-clean.** Both the prose note
   **and** the numbered deletion-completeness invariant are added (§ 3.6);
   the doc edit passes its own `/cold-eyes`. Falsifier: the destructive
   primitive is undocumented, or the note lands without its numbered INV.
8. **Gate green.** `./scripts/ci-local.sh` passes (ruff, format, bandit,
   pip-audit, gitleaks, pytest, mypy), including the new
   `tests/features/vault_reset/` suite (spec.md + tests) and any unlock /
   main-window tests touched.
