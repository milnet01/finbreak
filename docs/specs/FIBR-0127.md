**Theme:** The app installs **no palette today** (`app.py` sets no palette — only
the app name, desktop file, window icon and layout direction), so it rides the
platform default (dark by convention).
This phase builds a real, app-wide **theme system**: a registry of **six named,
finance-flavoured themes** (three light — **Ledger · Parchment · Mint**; three
dark — **Midnight · Graphite · Emerald**), a **"Follow system"** mode that tracks
the OS light/dark setting **live**, a theme **picker in Settings** that applies
**instantly**, and a **sleek, modern polish** pass (gradient/glow accent borders;
row highlighting — hover + selection + stripes — in the grid views). Every theme
is defined by a small set of semantic colour **tokens** that expand into a full
Qt palette **and** the polish stylesheet, so all six get the same modern
treatment and adding one is cheap. The choice is stored **outside the encrypted
vault** (in the plaintext window INI) so it applies to the very first,
still-locked window — before unlock. See [ADR-0010](../decisions/0010-theme-system.md).

## Context

`app.py` today builds the `QApplication`, sets the app name, desktop file, window
icon, and layout direction, then constructs + shows `MainWindow(service)` — **no
palette, no style, no stylesheet**:

```python
app = QApplication(argv if argv is not None else sys.argv)
app.setApplicationName("finbreak")
QGuiApplication.setDesktopFileName("finbreak")
app.setWindowIcon(app_icon())
app.setLayoutDirection(QLocale().textDirection())
...
window = MainWindow(service)
window.show()
return app.exec()
```

Widgets that already read the live palette (so they adapt the moment a palette is
installed): `ui/icons.py` `_is_dark_theme()` (`QApplication.palette().color(Window)
.lightnessF() < 0.5`) drives `toolbar_icon`'s muted/vibrant glyph tints;
`ui/home.py` builds a `ChartTheme` from `palette().text()` (the dataclass is defined in `ui/charts.py`).

The **non-vault** store is `paths.window_settings_path()` — an unencrypted INI
"that must load *before* unlock", already read/written via
`QSettings(..., QSettings.Format.IniFormat)` in `ui/main_window.py`
(`_settings()`, geometry/last-tab) and `ui/_table_state.py`. This is the correct
home for the theme pref (D2).

The **Settings dialog** (`ui/settings.py`, FIBR-0055) is a modal `QDialog(service:
AuthService, base_currency, parent, *, update_enabled, update_supported)` opened
by `MainWindow._open_settings` (line 700), which already reads context and hands
it to the dialog. Its `_on_save` writes vault settings and emits `saved`; the
shell's `_on_settings_saved` (line 931) re-reads prefs and pushes them live to the
open tabs. `MainWindow.__init__` (line 171) is
`(service, parent=None, *, update_service=None, installer=None)` — keyword-only
extras, so a new optional keyword is non-breaking. `_make_action` (line 409) sets
`toolbar_icon(icon_name)` on each icon'd action; `_build_workspace` (line 541)
rebuilds the tabbed workspace on each unlock.

## Design decisions

- **D1 — Fusion + token-driven `QPalette` & stylesheet (ADR-0010).** The one
  mechanism that gives an **identical look on Windows/macOS/Linux** with full
  colour control (needed for six named themes + the "dark-theme polish"), while
  keeping each theme cheap (~8 semantic colours + an `is_dark` flag). Rejected: the native
  `styleHints().setColorScheme()` (light/dark only, OS-dependent look, no brand /
  polish) and per-theme hand-written QSS files (heavy, non-DRY). Full rationale in
  ADR-0010.

- **D2 — Theme pref in the non-vault window INI, not the vault.** The theme must
  apply to the **first, still-locked** window (before any key exists), and it is
  **not sensitive** — so it belongs in `paths.window_settings_path()` beside
  geometry/last-tab (FIBR-0052 INV-5), **not** the encrypted `settings` table.
  This is the deliberate inverse of FIBR-0055 D1 (the auto-lock timeout **is** a
  security control → vault); stated so the divergence reads as intentional.

- **D3 — A `ThemeController(QObject)` owns mode + live follow-system; free helpers
  own the pref.** `ThemeController(app)` holds the current **mode** (a theme id or
  `"system"`), applies themes, watches `styleHints().colorSchemeChanged`, and emits
  `themeChanged`. It **parents itself to `app`** (`super().__init__(app)`), so it
  survives for the application's lifetime and its `colorSchemeChanged` connection
  isn't GC'd if a future refactor stops threading it through `MainWindow` (the
  live follow-system behaviour depends on the controller staying alive).
  `load_theme_pref()` / `save_theme_pref(id)` are module-level functions over the
  INI. `app.py` builds the controller and applies the stored pref **before**
  `MainWindow`; the controller is passed into `MainWindow` as an **optional
  keyword** so the shell can hand it to the Settings dialog and re-tint the toolbar
  on `themeChanged`. **`set_theme(id)` with an unknown `id`** (not a known theme
  and not `"system"`) **sets `mode = "system"`** and resolves from there —
  mirroring INV-2's guard. Normalising the *mode* (not just the applied theme)
  keeps `current_mode()` and the `colorSchemeChanged` slot's `mode == "system"`
  live-follow guard consistent, so a bad id can't leave the app showing a
  system-resolved theme that silently stops tracking OS flips.

- **D4 — The theme picker is an *immediate* control, not Save/Cancel form data.**
  Selecting a theme applies **and** persists at once (via `set_theme`),
  independent of the dialog's `Save`/`Cancel`. Rationale: (a) it gives the instant
  "see it now" preview the user asked for; (b) it is the **simplest correct**
  design — no revert-on-cancel bookkeeping, and the auto-lock-teardown path
  (FIBR-0055 INV-7) has nothing to revert or commit for the theme; (c) the theme
  is non-vault and non-sensitive, so there is no reason to gate it behind `Save`.
  This is a deliberate, documented departure from FIBR-0055's uniform Save/Cancel
  model for the **one** immediate control. **Persist is best-effort:** `set_theme`
  applies the palette live **first**, then calls `save_theme_pref`. Note
  `QSettings.setValue` does **not** raise on an unwritable/read-only INI — it
  buffers, and a write failure surfaces only via `status()` after `sync()` — so the
  common unwritable case already non-persists silently (the theme still applies for
  the session and reverts to the last-saved value next launch). The `try/except`
  around the persist is therefore **defensive belt-and-braces** against any
  *unexpected* exception, so a look-preference that can't persist never crashes the
  `currentIndexChanged` slot. This is the presentation-only posture (INV-12), not a
  global-rule-§1 silent swallow: `save_theme_pref` does not itself `sync()`/inspect
  `status()` (a UI look-pref does not warrant surfacing a disk error to the user).

- **D5 — Six themes; `ledger`/`midnight` are the system-follow defaults;
  `Unknown` scheme → dark.** `DEFAULT_LIGHT="ledger"`, `DEFAULT_DARK="midnight"`.
  `styleHints().colorScheme()` returns `{Unknown, Light, Dark}`; `Unknown` (the OS
  did not report) maps to **Midnight** to preserve the app's established dark
  default (the pre-FIBR-0127 convention), documented so it is not read as a bug.

- **D6 — Gradient/glow via QSS accent gradients; no `QGraphicsDropShadowEffect`.**
  Qt has **no CSS `box-shadow`**, and a per-**row** blur is impossible (rows are
  not widgets). The gradient/glow aesthetic (user's choice) is delivered with
  `qlineargradient` accent fills/borders that **intensify on focus/selection**. A
  literal blurred halo is **out of scope** (Out of scope; a future
  `QGraphicsDropShadowEffect`-on-discrete-widgets follow-up).

- **D7 — Row highlighting = QSS `::item:hover`/`::item:selected` gradients +
  centralized striping.** Hover + selection highlighting is pure QSS (no per-view
  code). Alternating stripes need `setAlternatingRowColors(True)` per view, so a
  single `theme.polish_item_views(root)` pass enables it on all `QAbstractItemView`
  descendants of the freshly-built workspace (`_build_workspace`, only when a theme
  controller is present — D10) — one call, no ripple across the seven tab widgets.
  (The soft hover gradient is built from `accent_soft`, the stronger selected-row
  gradient from `accent` — INV-11; `build_stylesheet` reads **`is_dark`** to orient
  those accent gradients — lighten-toward on dark themes, darken-toward on light —
  which is `is_dark`'s stylesheet consumer, INV-3.) The stylesheet is **scoped to specific
  widget classes** (inputs, buttons, `QAbstractItemView`, `QHeaderView`,
  `QTabBar`, `QToolBar`, frames) so it does **not** wrap the chart `QGraphicsView`
  in an unwanted border (verify empirically — coding.md idiom check).

- **D8 — Live toolbar re-tint on theme change (delivers FIBR-0116).** Toolbar
  glyphs are built once (at chrome build) from `_is_dark_theme()`; a live
  light↔dark switch would leave them stale. `MainWindow` connects
  `theme_controller.themeChanged` → `_retint_toolbar_icons`, which re-runs
  `toolbar_icon(name)` for each icon'd action. This realises FIBR-0116's live
  re-tint as a natural consequence; the ROADMAP is reconciled on drafting this
  spec (FIBR-0116 annotated that its live re-tint lands in FIBR-0127; the
  FIBR-0127 bullet reworded from "Unblocks" to "Delivers" for the icon re-tint).

- **D9 — Keep `_amount.py`'s fixed mid-tones; per-theme amount re-tint deferred.**
  The red/green amount colours (`ui/_amount.py`'s `_POSITIVE_TEXT`/`_NEGATIVE_TEXT`
  mid-tone constants, also imported + used by `ui/home.py`) are chosen to read on
  both light and dark and are **left unchanged** this phase. `ThemeTokens` therefore carries **no**
  `positive`/`negative` field this cut (there is no consumer — YAGNI); making
  amounts fully palette-adaptive per theme is the FIBR-0116-adjacent follow-up the
  ROADMAP defers (Out of scope), and would add those tokens then.

- **D10 — `MainWindow` gains an optional `theme_controller` keyword (default
  `None`).** With `None` (every existing `MainWindow(service)` construction,
  including all current tests), the shell shows **no** theme picker, does no
  re-tint wiring, and **skips `polish_item_views`** — so existing suites see the
  unchanged default palette *and* unchanged `alternatingRowColors` state, and pass
  untouched. The theme GUI tests pass a real controller.

- **D11 — Fix the `ui/icons.py` ADR mis-citation.** `_is_dark_theme`'s docstring
  *"Defaults to dark (ADR-0002)"* is corrected to cite **ADR-0010** (the real
  theme decision). The headless default (dark when no `QApplication` exists) is
  unchanged — only the citation.

- **D12 — Test isolation: snapshot & restore the app palette/style/stylesheet.**
  GUI theme tests mutate the **shared** `qapp`'s palette/style/stylesheet; a
  `theme_isolation` fixture snapshots all three before and restores them after, so
  an applied theme cannot leak a light palette into a sibling test (e.g. the
  `ui/icons.py` `_is_dark_theme` suites, which assume the default dark palette).
  **Restore the style by *name*, not by object:** snapshot
  `app.style().objectName()` and restore with `app.setStyle(name)` — `style()`
  returns a **borrowed** `QStyle` that `setStyle("Fusion")` deletes, so holding and
  re-installing the old pointer is a use-after-free. Palette (a value copy) and
  stylesheet (a `str`) restore directly. **Note on `run()`/`ThemeController`
  leakage:** INV-1's `run()` test and the controller legs each build a
  `ThemeController` parented to the session `qapp` (never destroyed); this is
  benign because the tests force follow-system via the `_current_scheme()` seam and
  **never `emit` `colorSchemeChanged`**, so leaked controllers never fire — but a
  future test that actually emits the signal must first disconnect / delete prior
  controllers to avoid cross-talk.

## The six themes

Each theme is 8 semantic colour tokens (+ an `is_dark` flag); final hex values are tuned during TDD (the
table gives the intended character, verified for text-contrast on its own
background). All three dark themes keep `is_dark=True` with a **dark `window`
token** (lightness `< 0.5`), all three light `is_dark=False` with a light one — so
once `build_palette` installs the palette, `_is_dark_theme` (which reads the live
palette's *window* lightness, not the flag — `icons.py:66`) reports correctly and
the toolbar glyphs tint to suit (INV-3 pins the flag↔window-lightness agreement).

| id | Name | Light/Dark | Character (accent) |
|----|------|-----------|--------------------|
| `ledger` | Ledger | Light | Clean white "paper", navy text, **gold** accent — brand default light |
| `parchment` | Parchment | Light | Warm aged-paper cream, sepia ink, **muted amber** accent |
| `mint` | Mint | Light | Crisp near-white, **money-green** accent |
| `midnight` | Midnight | Dark | Deep navy base, **gold** accent — matches the app icon; brand default dark |
| `graphite` | Graphite | Dark | Neutral charcoal, understated **slate-blue** accent |
| `emerald` | Emerald | Dark | Dark base, rich **emerald-green** accent — "wealth" |

## Data model (non-vault key — no schema change)

No vault schema change. One key in the plaintext window INI
(`paths.window_settings_path()`):

| key | value | written by | read by |
|-----|-------|-----------|---------|
| `theme` | one of `system` \| `ledger` \| `parchment` \| `mint` \| `midnight` \| `graphite` \| `emerald` | `save_theme_pref` via `ThemeController.set_theme` | `load_theme_pref` at `app.py` startup + the Settings picker preselect |

Absent / unknown ⇒ `"system"` (INV-2).

## Out of scope (this cut)

- **A literal *blurred* glow** (`QGraphicsDropShadowEffect`) — Qt has no CSS
  `box-shadow`; the gradient/glow aesthetic is delivered with accent gradients
  (D6). A blurred-halo-on-focus follow-up can add the effect to discrete widgets.
- **Per-theme amount re-tinting** (`ui/_amount.py` / `ui/home.py` red/green from
  theme tokens) — deferred (D9), the FIBR-0116-adjacent follow-up the ROADMAP
  notes; the fixed mid-tones stay.
- **User-defined / custom themes** (a theme editor) — the six are a curated set.
- **A dedicated high-contrast / accessibility theme** — a separate, explicitly
  accessibility-scoped follow-up (the `features-accessibility` backlog).
- **Re-styling the QtCharts internals beyond palette adaptation** — the charts

  existing constructions stay valid (D10).
