# FIBR-0095 — Failed-unlock throttling (exponential backoff)

- **Roadmap:** FIBR-0095 (security-account-recovery); retires the
  duplicate FIBR-0031.
- **Kind:** security.
- **Depends on:** FIBR-0004 (the unlock path).

## Context — the gap

`AuthService.complete_unlock(raw) -> bool` (`services/auth.py` L235)
returns `False` on a wrong master password with **no delay, no counter,
no sleep** — verified 2026-07-11 and re-verified this session. Nothing
in `services/auth.py` or `ui/unlock.py` slows a rapid sequence of wrong
guesses driven through the unlock dialog.

Argon2id (INV-2) already makes each *derivation* slow, which is the real
protection against an attacker who copies `vault.db` + the KDF sidecar
and cracks it **offline** with their own tooling. This item does **not**
touch that path. It adds a second, softer layer: friction against
someone guessing **interactively through finbreak's own unlock window**
on an unattended/stolen machine — a growing delay after each wrong try,
so a one-off typo is barely noticeable but bulk guessing through the UI
becomes impractical.

### What this is NOT (honest scope)

- **Not a security boundary.** The backoff state lives in the plaintext
  `window.ini` (it must be readable *before* the vault is unlocked). Any
  attacker with filesystem access can delete/edit that file to reset it —
  and such an attacker can equally copy the vault and crack it offline,
  where INV-2 is the only real defence. The backoff raises the bar for the
  *casual* interactive-guesser (who would otherwise just relaunch the
  app), not for a determined attacker with disk access.
- **Not a lockout.** The delay is **capped** and a **correct password
  always clears the counter**, so the legitimate owner is never
  permanently denied access (FIBR-0031's explicit constraint: "no lockout
  that could deny the legitimate owner access, and no counter that weakens
  the crypto").
- **Not applied to the headless `AuthService.unlock()` path** — that is a
  test/programmatic entry, not the attack surface; anyone able to script
  against `AuthService` can crack offline anyway. Throttling lives with the
  interactive `UnlockDialog`.

## The schedule

The delay imposed **before the next attempt is accepted**, as a function
of the consecutive-failure count `n` (the count *including* the failure
just recorded), follows FIBR-0031's stated ramp `1s → 2s → 4s …, capped`:

```
delay(n) = 0                               for n <= 0
delay(n) = min(BASE * 2**(n - 1), CAP)     for n >= 1
BASE = 1.0 s     CAP = 30.0 s
```

| n (consecutive fails) | 1 | 2 | 3 | 4 | 5 | 6 | 7+ |
|-----------------------|---|---|---|---|---|---|----|
| delay before next try | 1s | 2s | 4s | 8s | 16s | 30s* | 30s |

`*` `2**5 = 32` clamped to the 30 s cap.

The **authoritative gate** is recomputed on every submit from persisted
wall-clock state — a cosmetic countdown timer drives the label, but it is
never trusted for the actual decision:

```
remaining(n, last_fail, now):
    if n <= 0: return 0.0
    if last_fail is None: return delay(n)         # partial/corrupt write → fail-safe (full wait)
    elapsed = (now - last_fail).total_seconds()   # negative if clock moved back → longer wait, fail-safe
    return max(0.0, delay(n) - elapsed)
```

## Persistence (plaintext `window.ini`)

State is stored via `QSettings(window_settings_path(), IniFormat)` — the
same plaintext, pre-unlock store `ui/_table_state.py` / `theme.py` already
use — under two keys:

- `unlock/fail_count` — int, consecutive failed attempts.
- `unlock/last_fail` — ISO-8601 UTC timestamp of the most recent failure.

Persisting (rather than holding the count in memory) is the deliberate
choice that stops the trivial **"just close and reopen the app"** bypass:
a relaunch constructs a fresh dialog that reads the same file and still
owes the remaining delay. `.sync()` is called after every write so a
same-process read-back sees it (matching the existing INV-5 QSettings
usage).

`reset()` removes both keys. It is called on a **successful** unlock only.

## Design decisions

- **D1 — Pure core, thin adapter, UI wiring (three layers).**
  `services/unlock_throttle.py` holds Qt-free pure functions
  (`backoff_delay_seconds`, `remaining_lockout_seconds`) + a frozen
  `ThrottleState(fail_count, last_fail)` dataclass — trivially unit-testable
  without Qt, mirroring the project's `detect_recurring` / `detect_date_format`
  pure-core pattern. `ui/_unlock_throttle.py` holds the `UnlockThrottle`
  QSettings adapter (`load`/`remaining`/`record_failure`/`reset`), mirroring
  the existing `ui/_table_state.py` / `ui/_datetime_prefs.py` `_`-prefixed
  QSettings adapters. `ui/unlock.py` wires it.
- **D2 — Injected clock.** All timing takes an explicit `now: datetime`
  (tz-aware UTC), defaulted at the boundary with `datetime.now(UTC)` —
  following the project's `today`-parameter convention so tests inject time.
  Persisted timestamps are **wall-clock** (survive a restart), not
  `monotonic()` (which resets per process and cannot be persisted).
- **D3 — Gate on entry, record on failure, reset on success.** In
  `UnlockDialog`: `_on_unlock` computes `remaining(now)` **before** spawning
  the derive worker and, if `> 0`, refuses the attempt (shows the countdown,
  starts the timer, returns without deriving); `_show_failure` calls
  `record_failure(now)` then starts the countdown to the new delay;
  the success path (`_on_derived` → before emitting `unlocked`) calls
  `reset()`.
- **D4 — Countdown is cosmetic; the file is authoritative.** A new 1-Hz
  `QTimer` on the dialog updates the "Try again in N s" label and
  re-enables submit at zero, but every real submit re-reads `remaining()`
  from `window.ini`, so an unreliable/backgrounded timer can never *grant*
  an attempt early.
- **D5 — Fail-safe corners.** Missing `last_fail` with a positive count →
  full delay owed; a `last_fail` in the future (clock moved back or a
  tampered stamp) → `elapsed < 0` → a *longer* wait. Both err toward more
  delay, never less. A tampered huge `fail_count` still clamps at `CAP`.
- **D6 — No new user-facing config.** The schedule constants are fixed (no
  Settings knob) — keeping the change surgical. (A configurable cap could be
  a later item, but the roadmap does not ask for one and a money-app default
  should be safe out of the box.)

## At a glance — invariants

| INV | Statement |
|-----|-----------|
| **INV-1** | `backoff_delay_seconds(n)` = `0` for `n<=0`, else `min(1.0 * 2**(n-1), 30.0)`; monotonic non-decreasing in `n`; never exceeds `CAP=30`. |
| **INV-2** | `remaining_lockout_seconds` returns `0` exactly when `n<=0` **or** `elapsed >= delay(n)`; otherwise `delay(n) - elapsed`, clamped `>= 0`. |
| **INV-3** | Missing `last_fail` with `n>0` → full `delay(n)` (fail-safe); `now < last_fail` → result `> delay(n)` (fail-safe). |
| **INV-4** | A recorded failure increments `fail_count` by exactly 1 and stamps `last_fail=now`, persisted to `window.ini`; a **new** `UnlockThrottle` reading the same file sees the same state (survives "restart"). |
| **INV-5** | A successful unlock calls `reset()`, after which `fail_count` reads `0` and `remaining(...)==0` (counter never weakens the crypto; owner never permanently locked out). |
| **INV-6** | `UnlockDialog` refuses to spawn the derive worker while `remaining(now) > 0` (the gate is on entry, not only on the disabled button). |
| **INV-7** | The backoff state is written **only** to the plaintext `window.ini`; no attempt-count/timestamp is written into the encrypted vault, and no password/key material is written to either store. |

### security-model.md INV-10 (verbatim to add)

> - **INV-10 — Interactive unlock is throttled (defence-in-depth, not a
>   boundary).** After a wrong master password in the unlock dialog, the next
>   attempt is delayed on a capped exponential schedule (1s, 2s, 4s, …, capped
>   at 30s), with the attempt count and last-failure time persisted in the
>   plaintext `window.ini` so an app restart does not reset the delay. This is
>   best-effort friction against guessing **through the app UI**; it is **not**
>   a security boundary — an attacker with filesystem access can reset it, and
>   the copied-vault offline-cracking path is defended only by INV-2 (Argon2id).
>   The delay is capped and a correct password always clears the counter, so
>   the legitimate owner is never permanently locked out.

## Out of scope

- Offline brute-force against a copied vault (INV-2 / Argon2id owns this).
- Throttling the headless `AuthService.unlock()` entry.
- Any hard lockout, account-disable, or destructive response.
- A user-configurable schedule/cap (fixed constants; possible later item).
- The "forgotten password → start over" reset (FIBR-0030) and hint
  (FIBR-0029) — separate items.
