## Context â€” the gap

`AuthService.complete_unlock(raw) -> bool` (`services/auth.py` L235)
returns `False` on a wrong master password with **no delay, no counter,
no sleep** â€” verified 2026-07-11 and re-verified this session. Nothing
in `services/auth.py` or `ui/unlock.py` slows a rapid sequence of wrong
guesses driven through the unlock dialog.

Argon2id (security-model INV-2) already makes each *derivation* slow, which is the real
protection against an attacker who copies `vault.db` + the KDF sidecar
and cracks it **offline** with their own tooling. This item does **not**
touch that path. It adds a second, softer layer: friction against
someone guessing **interactively through finbreak's own unlock window**
on an unattended/stolen machine â€” a growing delay after each wrong try,
so a one-off typo is barely noticeable but bulk guessing through the UI
becomes impractical.

### What this is NOT (honest scope)

- **Not a security boundary.** The backoff state lives in the plaintext
  `window.ini` (it must be readable *before* the vault is unlocked). Any
  attacker with filesystem access can delete/edit that file to reset it â€”
  and such an attacker can equally copy the vault and crack it offline,
  where security-model INV-2 (Argon2id) is the only real defence. The backoff raises the bar for the
  *casual* interactive-guesser (who would otherwise just relaunch the
  app), not for a determined attacker with disk access.
- **Not a lockout.** The delay is **capped** and a **correct password
  always clears the counter**, so the legitimate owner is never
  permanently denied access (FIBR-0031's explicit constraint: "no lockout
  that could deny the legitimate owner access, and no counter that weakens
  the crypto").
- **Not applied to the headless `AuthService.unlock()` path** â€” that is a
  test/programmatic entry, not the attack surface; anyone able to script
  against `AuthService` can crack offline anyway. Throttling lives with the
  interactive `UnlockDialog`.

## The schedule

The delay imposed **before the next attempt is accepted**, as a function
of the consecutive-failure count `n` (the count *including* the failure
just recorded), follows FIBR-0031's stated ramp `1s â†’ 2s â†’ 4s â€¦, capped`:

```
BASE = 1.0 s     CAP = 30.0 s     CAP_N = 6
delay(n) = 0                                         for n <= 0
delay(n) = min(BASE * 2**(min(n, CAP_N) - 1), CAP)   for n >= 1
```

`CAP_N` is the smallest `n` at which the cap is already reached
(`2**(6-1) = 32 â†’ 30`). Clamping the **exponent** with `min(n, CAP_N)`
*before* raising the power is deliberate: the count comes from the
plaintext `window.ini`, which the threat model already concedes is
attacker-writable, so an unguarded `2**(n - 1)` on a tampered
`fail_count` of, say, a billion would build a ~300-million-digit integer
and hang the app on launch. With the clamp, any `n >= CAP_N` costs the
same trivial `2**5` and returns `CAP`.

| n (consecutive fails) | 1 | 2 | 3 | 4 | 5 | 6+ |
|-----------------------|---|---|---|---|---|----|
| delay before next try | 1s | 2s | 4s | 8s | 16s | 30s |

(`2**5 = 32` clamped to the 30 s cap at `n = 6`, and every larger `n`
returns 30 s via the exponent clamp.)

The **authoritative gate** is recomputed on every submit from persisted
wall-clock state â€” a cosmetic countdown timer drives the label, but it is
never trusted for the actual decision (below, `delay` is the pure
`backoff_delay_seconds` and `remaining` is `remaining_lockout_seconds` â€”
see Deliverables 1):

```
remaining(n, last_fail, now):
    if n <= 0: return 0.0
    if last_fail is None: return delay(n)         # partial/corrupt write â†’ fail-safe (full wait)
    elapsed = (now - last_fail).total_seconds()   # negative if clock moved back â†’ longer wait, fail-safe
    return max(0.0, delay(n) - elapsed)
```

## Persistence (plaintext `window.ini`)

State is stored via `QSettings(window_settings_path(), IniFormat)` â€” the
same plaintext, pre-unlock store `ui/_table_state.py` / `theme.py` already
use â€” under two keys:

- `unlock/fail_count` â€” int, consecutive failed attempts.
- `unlock/last_fail` â€” ISO-8601 UTC timestamp of the most recent failure.

Persisting (rather than holding the count in memory) is the deliberate
choice that stops the trivial **"just close and reopen the app"** bypass:
a relaunch constructs a fresh dialog that reads the same file and still
owes the remaining delay. `.sync()` is called after every write so a
same-process read-back sees it (matching the existing
`ui/_table_state.py` QSettings usage, which persists window geometry
outside the vault per FIBR-0052's own INV-5).

`load()` reads both keys as raw strings and coerces them **defensively in
our own code** (not by trusting `QSettings`' type-coercion): `fail_count` via
`int(...)` inside a `try/except (TypeError, ValueError)` â†’ `0` on missing or
non-integer input; `last_fail` via `datetime.fromisoformat(...)` inside the
same guard â†’ `None` on missing **or malformed** input (e.g. an ISO string
truncated by an interrupted write). A defaulted `fail_count = 0` is fail-safe
(a wrong password simply re-increments) and no weaker than deleting the file,
which the threat model already concedes; a defaulted `last_fail = None` with a
positive `fail_count` yields the **full** delay (fail-safe, per this spec's INV-3).

`reset()` removes both keys, then `.sync()`s. It is called on a
**successful** unlock only. A crash *between* the two key removals is
self-healing: on the next `load()`, a lone surviving `last_fail` reads
`fail_count = 0 â†’ remaining() == 0`, and a lone surviving `fail_count`
(no `last_fail`) yields the full delay (fail-safe), which the next
successful unlock clears.

## Design decisions

- **D1 â€” Pure core, thin adapter, UI wiring (three layers).**
  `services/unlock_throttle.py` holds Qt-free pure functions
  (`backoff_delay_seconds`, `remaining_lockout_seconds`) + a frozen
  `ThrottleState(fail_count, last_fail)` dataclass â€” trivially unit-testable
  without Qt, mirroring the project's `detect_recurring` / `detect_date_format`
  pure-core pattern. `ui/_unlock_throttle.py` holds the `UnlockThrottle`
  QSettings adapter (`load`/`remaining`/`record_failure`/`reset`), mirroring
  the existing `ui/_table_state.py` / `ui/_datetime_prefs.py` `_`-prefixed
  QSettings adapters. `ui/unlock.py` wires it.
- **D2 â€” Injected clock.** All timing takes an explicit `now: datetime`
  (tz-aware UTC), defaulted at the boundary with `datetime.now(UTC)` â€”
  following the project's `today`-parameter convention so tests inject time.
  Persisted timestamps are **wall-clock** (survive a restart), not
  `monotonic()` (which resets per process and cannot be persisted).
- **D3 â€” Gate on entry, record on failure, reset on success.** In
  `UnlockDialog`: `_on_unlock` computes `remaining(now)` **before** spawning
  the derive worker and, if `> 0`, refuses the attempt (shows the countdown,
  starts the timer, returns without deriving); `_show_failure` calls
  `record_failure(now)` then starts the countdown to the new delay;
  the success path (`_on_derived` â†’ before emitting `unlocked`) calls
  `reset()`.
- **D4 â€” Countdown is cosmetic; the file is authoritative.** A new 1-Hz
  `QTimer` on the dialog updates the "Try again in N s" label and
  re-enables submit at zero, but every real submit re-reads `remaining()`
  from `window.ini`, so an unreliable/backgrounded timer can never *grant*
  an attempt early.
- **D5 â€” Fail-safe corners.** Missing **or malformed** `last_fail` (a
  truncated/garbage timestamp from a partial write) with a positive count â†’
  parsed as `None` â†’ full delay owed; a `last_fail` in the future (clock
  moved back or a tampered stamp) â†’ `elapsed < 0` â†’ a *longer* wait â€” all of
  these named corners err toward more delay, never less. A tampered huge
  `fail_count`
  clamps at `CAP` **without** computing a giant power (the exponent is
  clamped at `CAP_N` â€” see "The schedule"); a corrupt/non-integer
  `fail_count` defaults to `0`.
- **D6 â€” Concurrent instances (accepted best-effort).** finbreak ships no
  single-instance guard, so two running instances share one `window.ini`;
  a `record_failure()` racing a `reset()`/`record_failure()` in the other
  is a read-modify-write race that may drop an increment or clobber a
  reset. **Accepted, not locked:** this is soft friction, not a boundary,
  and the worst case (a lost increment) only slightly weakens an
  already-soft delay â€” it never locks out the owner. No file locking is
  added.
- **D7 â€” No new user-facing config.** The schedule constants are fixed (no
  Settings knob) â€” keeping the change surgical. (A configurable cap could be
  a later item, but the roadmap does not ask for one and a money-app default
  should be safe out of the box.)

## At a glance â€” invariants

_INV-1..7 below are **FIBR-0095's own local numbering**, distinct from the
global invariant list in `docs/security-model.md` (which this spec cites as
"security-model INV-N"). The one global invariant this spec adds to that list
is **INV-10**, in the verbatim block under Deliverables._

| INV | Statement |
|-----|-----------|
| **INV-1** | `backoff_delay_seconds(n)` = `0` for `n<=0`, else `min(1.0 * 2**(min(n, 6)-1), 30.0)`; the exponent is clamped at `n=6` (where the cap is already reached) so a tampered huge `fail_count` cannot trigger an unbounded-integer computation; monotonic non-decreasing in `n`; never exceeds `CAP=30`. |
| **INV-2** | `remaining_lockout_seconds` returns `0` exactly when `n<=0` **or** `elapsed >= delay(n)`; otherwise `delay(n) - elapsed`, clamped `>= 0` (the missing/malformed `last_fail` sub-case is INV-3). |
| **INV-3** | Missing `last_fail` with `n>0` â†’ full `delay(n)` (fail-safe); `now < last_fail` â†’ result `> delay(n)` (fail-safe). |
| **INV-4** | A recorded failure increments `fail_count` by exactly 1 and stamps `last_fail=now`, persisted to `window.ini`; a **new** `UnlockThrottle` reading the same file sees the same state (survives "restart"). |
| **INV-5** | A successful unlock calls `reset()`, after which `fail_count` reads `0` and `remaining(...)==0` (counter never weakens the crypto; owner never permanently locked out). |
| **INV-6** | `UnlockDialog` refuses to spawn the derive worker while `remaining(now) > 0` (the gate is on entry, not only on the disabled button). |
| **INV-7** | The backoff state is written **only** to the plaintext `window.ini`; no attempt-count/timestamp is written into the encrypted vault, and no password/key material is written to either store. |

### security-model.md INV-10 (verbatim to add)

> - **INV-10 â€” Interactive unlock is throttled.** After a wrong master
>   password in the unlock dialog, the next
>   attempt is delayed on a capped exponential schedule (1s, 2s, 4s, â€¦, capped
>   at 30s), with the attempt count and last-failure time persisted in the
>   plaintext `window.ini` so an app restart does not reset the delay. This is
>   best-effort friction against guessing **through the app UI**; it is **not**
>   a security boundary â€” an attacker with filesystem access can reset it, and

>   the legitimate owner is never permanently locked out.

## Out of scope

- Offline brute-force against a copied vault (security-model INV-2 /
  Argon2id owns this).
- Throttling the headless `AuthService.unlock()` entry.
- Any hard lockout, account-disable, or destructive response.
- A user-configurable schedule/cap (fixed constants; possible later item).
- The "forgotten password â†’ start over" reset (FIBR-0030) and hint
  (FIBR-0029) â€” separate items.
