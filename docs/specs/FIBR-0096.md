## 1. Problem & intent

finbreak's releases are already Ed25519-signed per artifact (FIBR-0054
INV-14): the AppImage gets a `.sig`, the Windows `.exe` gets a `.sig`, and
the in-app updater verifies the signature before installing. That covers
users who update **through the app**. It does **not** serve users who
download from the GitHub release page and want to verify **manually**, nor
does it disclose **what is bundled** inside the one-file artifacts.

FIBR-0096 adds two supply-chain-transparency artifacts per release:
1. **`SHA256SUMS`** — a plain checksum manifest of the downloadable
   artifacts, **itself Ed25519-signed** (`SHA256SUMS.sig`) with the same
   release key, so `sha256sum -c` gives a second, independently-verifiable
   integrity signal.
2. **A CycloneDX SBOM** per platform — a machine-readable parts-list of the
   **actually-bundled** dependency versions, for supply-chain auditing.

Neither changes the app; both are release-time artifacts attached to the
GitHub release.

## 2. Scope

**In scope**
- A `SHA256SUMS` manifest covering **both** release downloadables — the
  Linux `*.AppImage` and the Windows `*.exe` — in the standard
  `<hex>␠␠<basename>` format that `sha256sum -c` consumes.
- Signing `SHA256SUMS` → `SHA256SUMS.sig` via the existing artifact-agnostic
  `scripts/sign-release.py` (§ 3.2).
- The **two-phase** generation the split Linux/Windows publish forces
  (§ 3.3): phase 1 (`release-linux.sh`) writes+signs+uploads a manifest with
  the AppImage line; phase 2 (`release-windows.sh`) **adds** the exe line,
  re-signs, re-uploads.
- A **CycloneDX JSON SBOM per platform**, generated **inside each build**
  against the real bundled dependency set (§ 3.4), via
  `pip-audit --format cyclonedx-json`.
- Wiring all new artifacts into the `gh release` asset lists (§ 3.5).
- A `security-model.md` note + invariant for the signed manifest (§ 3.6).
- Tests via the script-scrape + helper-unit pattern (§ 4), no real release
  build required.

**Out of scope**
- Reproducible/deterministic builds (a separate, much larger concern).
- A secure-supply-chain attestation framework (SLSA, in-toto) — this is the
  cheap first rung: checksums + SBOM + an existing signature.
- Verifying the SBOM's completeness against the frozen binary byte-for-byte
  (the SBOM reflects the **build venv** pip resolves; see § 3.4's honesty
  note).
- Changing the in-app updater (it already verifies the per-artifact `.sig`;
  it does not consume `SHA256SUMS`).

## 3. Design

### 3.1 Where the pipeline stands (recon-verified)

- **`scripts/build-smoke.sh`** (driver, `--release` mode — reached via the thin
  `scripts/build-release-appimage.sh` wrapper that `release-linux.sh:57` calls,
  `exec …/build-smoke.sh --release`) emits artifacts to
  `$ROOT/dist/`: `finbreak-$VERSION-x86_64.AppImage` (+ `.sig` when a signing
  key is present), stamped off `__version__` (`src/finbreak/__init__.py:7`,
  the bump source of truth). The Linux freeze happens **inside a
  `python:3.12-slim-bookworm` container** via
  `scripts/_build-smoke-in-container.sh`, which pip-installs the runtime deps
  read from `pyproject.toml [project].dependencies` into a throwaway venv
  (`:52-53`) then PyInstaller-freezes.
- **`scripts/sign-release.py`** signs **any** file: `sig = key.sign(
  artifact.read_bytes())` → writes `<artifact>.sig` (raw 64-byte Ed25519).
  Positional `artifact` arg — so it signs `SHA256SUMS` unchanged.
- **Publishing is manual + host-side, in two runs:**
  `scripts/release-linux.sh` builds+signs+verifies then
  `gh release create <TAG> "$DIST/$APPIMAGE" "$DIST/$APPIMAGE.sig" …`
  (and an `upload --clobber` branch for re-runs). Later,
  `scripts/release-windows.sh` dispatches `windows-build.yml`, downloads the
  `.exe`, signs it, and `gh release upload <TAG> "$DIST/$EXE" "$DIST/$EXE.sig"
  --clobber` onto the **same** release.
- **No lockfile** exists; the full bundled transitive set only materialises
  in the container/CI build venv. The Windows freeze runs in
  `.github/workflows/windows-build.yml`.

### 3.2 The checksum helper (single source of the manifest format)

A new `scripts/gen-checksums.sh` — the **one** place that knows the manifest
format, so both release scripts call it rather than duplicating `sha256sum`
plumbing:

```
gen-checksums.sh <sumsfile> <artifact> [<artifact> …]
```
- For each `<artifact>` path, compute its SHA-256 and write/replace a line
  `<64-hex>␠␠<basename>` in `<sumsfile>` (basename only — so a user runs
  `sha256sum -c SHA256SUMS` from the folder holding the downloads).
- **Merge, don't clobber:** if `<sumsfile>` already lists other basenames,
  keep those lines and only add/replace the passed artifacts' lines. This is
  what lets phase 2 (§ 3.3) add the exe to a manifest phase 1 already wrote,
  without needing the AppImage present on the Windows-release machine.
- Deterministic line order (sort by basename) so re-generation is stable.
- Emits nothing else; signing is a separate explicit call (§ 3.3), keeping
  the helper pure/​testable.

### 3.3 Two-phase SHA256SUMS (the split-publish consequence)

Because the exe is attached in a **separate later run** from the AppImage,
`SHA256SUMS` is built in two phases. **Both phases follow the identical
`clean → fetch → merge → sign → verify → upload` shape** — the only difference
is which artifact's line each adds. The symmetry is deliberate: it makes
either script safe to re-run in any order without dropping the other
platform's line (the fetch below + § 3.2 merge), and it reuses the
sign-then-hard-gate-verify discipline the release scripts already enforce for
the binaries (`release-linux.sh:62-78`, `release-windows.sh:76-91`).

Each phase, in `release-<platform>.sh`, after its binary is
built/downloaded + signed + verified:
1. **Start clean:** `rm -f "$DIST/SHA256SUMS" "$DIST/SHA256SUMS.sig"`.
   `release-windows.sh:66` already `rm -f`s its dist targets; `release-linux.sh`
   must gain the same, so a stale `SHA256SUMS` from a **previous version**
   can't leak phantom basenames into this release's merge (INV-4).
2. **Fetch the current published manifest** if the release already exists:
   `gh release download "$TAG" -p SHA256SUMS -D "$DIST" 2>/dev/null || true`
   — so the **other** platform's already-published line survives even though
   that binary isn't on this machine. Absent (first phase to publish, or a
   single-platform release) → start fresh; the manifest then covers only this
   phase's artifact, which is correct, not partial.
3. **Merge this phase's line:** `gen-checksums.sh "$DIST/SHA256SUMS"
   "$DIST/<this-artifact>"` — adds/replaces only this artifact's basename,
   keeping the fetched line (§ 3.2).
4. **Sign the final bytes:** `sign-release.py … "$DIST/SHA256SUMS"` →
   `SHA256SUMS.sig`. The signature is always re-generated over the merged
   bytes, never reused — it must cover the **final** manifest (INV-3).
5. **HARD GATE — verify before publish:** verify `SHA256SUMS.sig` against the
   committed `RELEASE_PUBLIC_KEY_B64` and **refuse to upload on mismatch**,
   exactly as each script already gates its binary's `.sig`
   (`release-linux.sh:62-78`). A `Kind: security` item must not publish a
   signature its own committed key rejects.
6. **Upload:** add `SHA256SUMS` + `SHA256SUMS.sig` to the `gh release
   create`/`upload --clobber` asset list (§ 3.5); `--clobber` replaces any
   prior manifest with this merged, re-signed one.

**Invariant that falls out:** whatever `SHA256SUMS` is on the release at any
moment, its accompanying `.sig` verifies it against the committed key, and
every basename it lists is a real release asset (INV-1/2/3). Because **both**
phases clean-then-fetch-then-merge, re-running either script in any order
re-derives the union of whatever lines are already published plus its own — it
**never regresses** the manifest to a single platform (the failure a phase-1
that only wrote-its-own-line would cause). A brief window where only the
first-published platform is listed is correct, not partial — the manifest
always matches the assets present.

### 3.4 SBOM per platform (pip-audit CycloneDX, in-build)

The SBOM must reflect what is **actually bundled**, which only exists in the
build venv (no lockfile, § 3.1). So generate it **where the freeze happens**,
not on the host. Two recon-verified facts about the freeze environments drive
the mechanics:

- **pip-audit is NOT in the freeze venv.** Both freezes install *only* the
  runtime deps + PyInstaller (`_build-smoke-in-container.sh:53`,
  `build-windows-exe.py:75`); `pip-audit` lives in the **dev** group
  (`pyproject.toml:76`), which the gate installs on the **host**, never inside
  the freeze. So each SBOM step must **explicitly install the pinned tool**
  into the freeze venv first — a genuinely new in-build install, not a no-op.
  (No new *declared* project dependency — same tool the gate already pins —
  but it **is** fetched into the freeze venv, so the spec states it plainly
  rather than hand-waving "already installable.")
- **`$VERSION` is not in scope at either generation site.** `build-smoke.sh`
  passes `ONEFILE`/`APPIMAGE`/`APP_*` into the container but **not** `VERSION`
  (`build-smoke.sh:71-82`); `windows-build.yml` freezes to an **unversioned**
  `dist/finbreak.exe` and the version stamp is applied later by
  `release-windows.sh`. So each SBOM is version-stamped the same way its
  binary already is (below / § 3.5 / INV-6), not by assuming a `$VERSION`
  variable that isn't there.

The whole SBOM addition is **release-mode only** (gated on `VERSION` being
non-empty, step 5) so the default smoke build stays **byte-for-byte unchanged**
(`build-smoke.sh:31`).

**Linux** — in `_build-smoke-in-container.sh`:
1. **Capture the exact *installed* runtime closure, before any build tool
   enters the venv.** Two mistakes to avoid: auditing the live freeze venv
   pulls in PyInstaller + pip-audit + their trees (none bundled → over-report,
   § 1/§ 2); but `pip-audit -r <declared-deps>` — **or a parallel throwaway
   venv** — *re-resolves* the **ranged** (`beautifulsoup4>=4.9,<5`,
   `certifi>=2024.2.2`) and transitive versions away from what was actually
   installed and bundled. So in release mode **split** the current combined
   install (`_build-smoke-in-container.sh:53` installs `RUNTIME_DEPS` +
   PyInstaller in one line): install `RUNTIME_DEPS` (`:52`) first, then — with
   `FROZEN` defined once, e.g. `FROZEN="$(mktemp)"` — `pip freeze > "$FROZEN"`
   **immediately** — that snapshot is exactly what PyInstaller then collects
   from, with no build tools and no re-resolution — *then* install
   `pyinstaller==6.21.0` and proceed to freeze as today. (Smoke mode keeps the
   single combined install untouched.)
2. `python -m pip install --quiet pip-audit==2.10.0` (the same pin as
   `pyproject.toml:76`) — installed **after** `$FROZEN` is captured, so it
   never pollutes the snapshot; it only *reads* the frozen file.
3. Define the output path **once** — `OUT="/out/finbreak-$VERSION-linux.cdx.json"`
   — then `pip-audit -r "$FROZEN" --format cyclonedx-json --output "$OUT"
   || true`. **The `|| true` is deliberate and commented:** `pip-audit` exits
   **non-zero when it finds any advisory** (verified — it still **writes** the
   SBOM in that case), and the container runs under `set -euo pipefail`, so an
   un-tolerated call would **abort the release build** the moment a pinned dep
   ages into a CVE. SBOM generation is a *transparency* artifact, **not** the
   vuln **gate** — the gate still fails the build on the host (T7).
4. **Existence guard (a swallowed advisory is fine; a swallowed *crash* is
   not):** a genuine crash (resolver/network failure) writes **no file**, and
   `|| true` would swallow it — only to re-surface later as a confusing "file
   not found" at the §3.5 upload of a *mandatory* asset. So assert the output
   exists right after: `[ -f "$OUT" ] || { echo "SBOM: pip-audit produced no
   output — a real failure, not a tolerated advisory" >&2; exit 1; }`.
5. `$VERSION` reaches the container via a new **guarded**
   `-e "VERSION=${VERSION:-}"` on `build-smoke.sh`'s `run` invocation
   (`build-smoke.sh:76-82`) — guarded because that `run` block is
   **unconditional** yet `VERSION` is set only in `--release` mode
   (`build-smoke.sh:37-38`) under `set -u`; the SBOM steps run only when it is
   non-empty (the smoke-path gate above). Writes to `/out` (the same mount the
   AppImage lands in).

**Windows** — the same split-install + `pip freeze` + `pip-audit -r "$FROZEN"`
+ existence-guard, in `windows-build.yml`'s freeze job. `build-windows-exe.py`
also installs runtime deps + PyInstaller in one call (`build-windows-exe.py:75`)
— split it the same way so it can write the frozen runtime closure before
PyInstaller. Run pip-audit **after** the freeze produces `dist/`, writing an
**unversioned** `dist/finbreak-windows.cdx.json`. **The pip-audit step must set
`shell: bash`** — `windows-latest` `run:` steps default to **pwsh**
(`windows-build.yml:45,68` set pwsh explicitly), where `|| true` is not valid;
Git Bash is on the runner. The workflow currently uploads a single artifact
whose `path:` is exactly `dist/finbreak.exe` (`windows-build.yml:78-79`), so
widen that `path:` to also carry the `*.cdx.json`. `release-windows.sh` (which
`gh run download`s the artifact, `:67`) then **version-stamps it on arrival**
to `finbreak-$VERSION-windows.cdx.json`, exactly as it already renames
`finbreak.exe` → `finbreak-$VERSION-x86_64.exe`.

*Honest scope (both platforms):* the SBOM is a `pip freeze` of the runtime deps
**as actually installed** into the freeze venv (captured before any build tool)
audited via `pip-audit -r` — the exact set PyInstaller collects from. The one
residual is the § 2 out-of-scope reproducibility concern: it is **not** a
byte-level proof every listed package's files ended up inside the one-file.
State this in each SBOM step's comment so it isn't over-read.

Two SBOMs (one per platform) because the bundled wheels differ per OS
(PySide6/​pikepdf/​cryptography ship platform-specific binaries); a single SBOM
would misrepresent one platform. Reusing pip-audit's CycloneDX output adds no
new *project* dependency (user decision) and doubles the existing vuln tool as
the SBOM generator — same tool, extra output format.

### 3.5 Publish wiring

Add the new artifacts to the existing `gh release` asset lists:
- **`release-linux.sh`** — append `"$DIST/SHA256SUMS"`,
  `"$DIST/SHA256SUMS.sig"`, `"$DIST/finbreak-$VERSION-linux.cdx.json"` to
  both the `gh release create` and the `gh release upload --clobber` branches.
- **`release-windows.sh`** — append the re-generated `SHA256SUMS` +
  `.sig` (§ 3.3 phase 2) and `"$DIST/finbreak-$VERSION-windows.cdx.json"` to
  its `gh release upload --clobber`.
- The `build-smoke.sh` release-mode artifact-list echo (its "Release
  artifacts in $DIST" block) gains the new filenames so a human running it
  by hand sees them.

### 3.6 security-model.md impact

The published `SHA256SUMS` is a new **integrity** surface, so it earns a
note + an invariant (this is a `Kind: security` item on the supply-chain
path):
- A short note (near the trust-boundary / update-integrity discussion,
  `security-model.md:52-62`) that each release publishes a **signed**
  `SHA256SUMS` (a manual-verification integrity signal for users who download
  from the release page, complementary to the per-artifact `.sig` the in-app
  updater checks) and a per-platform CycloneDX SBOM (bundled-deps
  transparency).
- A **numbered invariant** (next free is **INV-13** — INV-12 at
  `security-model.md:264` is the current highest; re-verify at authoring
  time): *"The published `SHA256SUMS` manifest is Ed25519-signed with the
  release key over its final bytes; each hash line was computed by
  `gen-checksums.sh` from the actual artifact at its platform's release phase,
  and every basename it lists is a genuine release asset."* (Stated per-line
  **at generation time** — the invariant does not claim a global re-hash of the
  other platform's *carried* line, which is trusted from the prior **signed**
  manifest, § 3.3.) Falsifiable by the §4 checksum-helper + verify test. Frame
  it as an **integrity** guarantee (tamper-evidence), **not** confidentiality.
- **Add the INV-13 *definition* in § 5** after INV-12 (`security-model.md:264`).
  Do **not** blind-append INV-13 to the § 5 grouping sentence
  (`security-model.md:121`) or the § 6 tooling enumeration (`:288`): those are
  **curated per-phase-pytest subsets** tied to the vault / crypto / import /
  export / logging paths (`:121` omits INV-5a / 6 / 8 / 11; `:288` omits
  INV-5a / 6 / 8 but *carries* INV-11 — both are curated, neither is a running
  count), and INV-13 is a *release-pipeline* invariant asserted by the
  `release_integrity` suite, not one of those phases — filing it there would
  make the sentences factually wrong. If those sentences are touched at all,
  reword them to admit a release-time source-scrape invariant rather than
  slotting INV-13 into a phase clause it doesn't belong to.
- **Anchor, don't invent:** this is the **same threat class** as the existing
  download-integrity gate — a downloaded release artifact is installed/trusted
  only if its Ed25519 signature verifies over the exact bytes (**FIBR-0054
  INV-4**, `security-model.md:52-53`). The signed `SHA256SUMS` **extends that
  same gate to the manual-download path** the in-app updater doesn't cover.
  Reference **FIBR-0054 INV-4, not T9** — T9 (`security-model.md:90`) is
  *vault-file* tamper-evidence (per-page SQLCipher HMAC), a different surface
  entirely. No new threat *row* is required; if authoring judges the
  manual-download surface warrants its own row, that is a call to surface, not
  a mechanical add.

## 4. Invariants

- **INV-1 — Manifest format is `sha256sum -c`-valid.** `gen-checksums.sh`
  writes lines `<64-lowercase-hex>␠␠<basename>` (two spaces, basename only,
  sorted). *Test:* run the helper over two fixture files; assert exact line
  format, and that `sha256sum -c SHA256SUMS` **passes** in that dir and
  **fails** after a byte is flipped in a fixture.
- **INV-2 — Merge preserves prior lines.** Calling the helper with the exe,
  against a manifest already listing the AppImage, yields a manifest with
  **both** lines (AppImage line unchanged). *Test:* seed a manifest with an
  AppImage line, run the helper for an exe fixture, assert both basenames
  present and the AppImage hash byte-identical.
- **INV-3 — The manifest is signed over its final bytes, and publish is gated
  on verification.** *Production property:* after each phase,
  `release-<platform>.sh` verifies `SHA256SUMS.sig` against `SHA256SUMS` with
  the committed release **public** key (`update_key.RELEASE_PUBLIC_KEY_B64`)
  and refuses to upload on mismatch (§ 3.3 step 5), mirroring the binary
  `.sig` gate. *Test — two legs* (the crypto primitive is already proven by
  `test_INV14`; the FIBR-0096 value is in the manifest-specific properties):
  **(a) roundtrip** — with a **throwaway keypair** (the real private half isn't
  in-repo), mirroring
  `test_auto_update.py::test_INV14_signing_scripts_roundtrip`, generate → sign
  → verify a fixture manifest and assert a 1-byte edit fails verification;
  **(b) gate-present source-scrape** — assert each `release-<platform>.sh`
  contains the `SHA256SUMS.sig` verify-against-`RELEASE_PUBLIC_KEY_B64` step
  **before** its `gh release … SHA256SUMS` upload (parallel to **FIBR-0096**
  INV-4's scrape), so a future edit deleting the gate fails the suite. (The production property
  pins the *committed* key; leg (a) uses a *generated* key — mechanism
  identical, only the key differs.)
- **INV-4 — Both release scripts publish the new artifacts.** `release-linux.sh`
  uploads `SHA256SUMS` + `.sig` + the linux SBOM; `release-windows.sh`
  re-uploads the merged `SHA256SUMS` + `.sig` + the windows SBOM. *Test:*
  scrape each script's source; assert the `gh release` asset lists contain
  the new filenames (windows_build-style source assertion).
- **INV-5 — SBOM is generated in-build, per platform, over the installed
  runtime closure.** `_build-smoke-in-container.sh` and `windows-build.yml`
  each build the SBOM from a **`pip freeze` of the runtime deps as installed**
  (captured before PyInstaller enters the venv, § 3.4 — not the live freeze
  venv, not a re-resolution of the declared deps) and run `pip-audit -r
  <frozen> --format cyclonedx-json … || true` with an **output-existence
  guard**, writing a platform-stamped `*.cdx.json` to the artifact output.
  *Test:* scrape both build scripts; assert the pinned `pip-audit` install, the
  `pip-audit -r … cyclonedx-json` invocation, the existence guard, and the
  `-linux.cdx.json` / `-windows.cdx.json` output substrings exist.
- **INV-6 — Version-stamped off the single source.** Each SBOM name embeds
  `$VERSION` derived from `src/finbreak/__init__.py:__version__` (the bump
  source of truth) — **Linux** via a `VERSION` env passed into the build
  container (`build-smoke.sh` `-e "VERSION=${VERSION:-}"`, § 3.4), **Windows** via
  `release-windows.sh` renaming the unversioned in-workflow
  `finbreak-windows.cdx.json` to `finbreak-$VERSION-windows.cdx.json` on
  download — exactly as the exe is version-stamped on arrival today. *Test:*
  scrape the scripts — assert the Linux name is built from the `$VERSION` env
  (not a literal), and assert `release-windows.sh`'s rename maps the unversioned
  `finbreak-windows.cdx.json` to `finbreak-$VERSION-windows.cdx.json` using the
  `$VERSION` it derives — never a hardcoded version.
- **INV-7 — Signed manifest ⇒ security-model INV recorded.** The
  `security-model.md` trust-boundary note + the numbered **INV-13** definition
  (§ 3.6) are present. *Test:* a grep-style assertion that the doc contains the
  signed-`SHA256SUMS` note **and** an `INV-13` definition whose text names the
  signed manifest. (It does **not** assert INV-13 in the § 5/§ 6 curated
  per-phase enumerations — INV-13 is a release-pipeline invariant that doesn't
  belong there, § 3.6. The doc's `/cold-eyes` pass is an **exit gate** — § 5
  criterion 5 — not a pytest-expressible INV assertion.)

## 5. Exit criteria (falsifiable)

1. **Checksums verify.** The helper emits a `sha256sum -c`-valid manifest;
   merge preserves prior lines; a flipped byte fails `-c` (INV-1/2).
2. **Manifest is signed, verify-gated, + tamper-evident.** Each phase signs
   the final bytes and **hard-gates publish on verifying `SHA256SUMS.sig`
   against the committed key** (§ 3.3 step 5); a 1-byte edit fails
   verification (INV-3).
3. **Both platforms publish the artifacts.** Source assertions confirm
   `release-linux.sh` + `release-windows.sh` upload the manifest, its sig,
   and the per-platform SBOM; two-phase merge covers AppImage + exe
   (INV-4/2).
4. **SBOM is faithful + in-build.** Both build scripts generate the
   per-platform CycloneDX SBOM via pip-audit against the build venv (INV-5),
   version-stamped (INV-6).
5. **security-model.md updated + cold-eyes-clean.** The signed-manifest note
   + the INV-13 definition land (without mis-filing INV-13 into the § 5/§ 6
   per-phase enumerations, § 3.6) (INV-7), and the doc edit passes its own
   `/cold-eyes` (per global rule 14).
6. **Gate green.** `./scripts/ci-local.sh` passes, including the new
   `tests/features/release_integrity/` suite. (The full release build stays
   opt-in behind `FINBREAK_BUILD_SMOKE=1`, as today — these tests are
   source-scrape + helper-unit, no real build.)
