## 1. Problem & intent

finbreak's releases are already Ed25519-signed per artifact (FIBR-0054
INV-14): the AppImage gets a `.sig`, the Windows `.exe` gets a `.sig`, and
the in-app updater verifies the signature before installing. That covers
users who update **through the app**. It does **not** serve users who
download from the GitHub release page and want to verify **manually**, nor
does it disclose **what is bundled** inside the one-file artifacts.

FIBR-0096 adds two supply-chain-transparency artifacts per release:
1. **`SHA256SUMS`** — a plain checksum manifest of the downloadable
   artifacts, **itself Ed25519-signed** (`SHA256SUMS.sig`) with the same
   release key, so `sha256sum -c` gives a second, independently-verifiable
   integrity signal.
2. **A CycloneDX SBOM** per platform — a machine-readable parts-list of the
   **actually-bundled** dependency versions, for supply-chain auditing.

Neither changes the app; both are release-time artifacts attached to the
GitHub release.

## 2. Scope

**In scope**
- A `SHA256SUMS` manifest covering **both** release downloadables — the
  Linux `*.AppImage` and the Windows `*.exe` — in the standard
  `<hex>␠␠<basename>` format that `sha256sum -c` consumes.
- Signing `SHA256SUMS` → `SHA256SUMS.sig` via the existing artifact-agnostic
  `scripts/sign-release.py` (§ 3.2).
- The **two-phase** generation the split Linux/Windows publish forces
  (§ 3.3): phase 1 (`release-linux.sh`) writes+signs+uploads a manifest with
  the AppImage line; phase 2 (`release-windows.sh`) **adds** the exe line,
  re-signs, re-uploads.
- A **CycloneDX JSON SBOM per platform**, generated **inside each build**
  against the real bundled dependency set (§ 3.4), via
  `pip-audit --format cyclonedx-json`.
- Wiring all new artifacts into the `gh release` asset lists (§ 3.5).
- A `security-model.md` note + invariant for the signed manifest (§ 3.6).
- Tests via the script-scrape + helper-unit pattern (§ 4), no real release
  build required.

**Out of scope**
- Reproducible/deterministic builds (a separate, much larger concern).
- A secure-supply-chain attestation framework (SLSA, in-toto) — this is the
  cheap first rung: checksums + SBOM + an existing signature.
- Verifying the SBOM's completeness against the frozen binary byte-for-byte
  (the SBOM reflects the **build venv** pip resolves; see § 3.4's honesty
  note).
- Changing the in-app updater (it already verifies the per-artifact `.sig`;
  it does not consume `SHA256SUMS`).

## 3. Design

### 3.1 Where the pipeline stands (recon-verified)

- **`scripts/build-smoke.sh`** (driver, `--release` mode) emits artifacts to
  `$ROOT/dist/`: `finbreak-$VERSION-x86_64.AppImage` (+ `.sig` when a signing
  key is present), stamped off `__version__` (`src/finbreak/__init__.py:7`,
  the bump source of truth). The Linux freeze happens **inside a
  `python:3.12-slim-bookworm` container** via
  `scripts/_build-smoke-in-container.sh`, which pip-installs the runtime deps
  read from `pyproject.toml [project].dependencies` into a throwaway venv
  (`:52-53`) then PyInstaller-freezes.
- **`scripts/sign-release.py`** signs **any** file: `sig = key.sign(
  artifact.read_bytes())` → writes `<artifact>.sig` (raw 64-byte Ed25519).
  Positional `artifact` arg — so it signs `SHA256SUMS` unchanged.
- **Publishing is manual + host-side, in two runs:**
  `scripts/release-linux.sh` builds+signs+verifies then
  `gh release create <TAG> "$DIST/$APPIMAGE" "$DIST/$APPIMAGE.sig" …`
  (and an `upload --clobber` branch for re-runs). Later,
  `scripts/release-windows.sh` dispatches `windows-build.yml`, downloads the
  `.exe`, signs it, and `gh release upload <TAG> "$DIST/$EXE" "$DIST/$EXE.sig"
  --clobber` onto the **same** release.
- **No lockfile** exists; the full bundled transitive set only materialises
  in the container/CI build venv. The Windows freeze runs in
  `.github/workflows/windows-build.yml`.

### 3.2 The checksum helper (single source of the manifest format)

A new `scripts/gen-checksums.sh` — the **one** place that knows the manifest
format, so both release scripts call it rather than duplicating `sha256sum`
plumbing:

```
gen-checksums.sh <sumsfile> <artifact> [<artifact> …]
```
- For each `<artifact>` path, compute its SHA-256 and write/replace a line
  `<64-hex>␠␠<basename>` in `<sumsfile>` (basename only — so a user runs
  `sha256sum -c SHA256SUMS` from the folder holding the downloads).
- **Merge, don't clobber:** if `<sumsfile>` already lists other basenames,
  keep those lines and only add/replace the passed artifacts' lines. This is
  what lets phase 2 (§ 3.3) add the exe to a manifest phase 1 already wrote,
  without needing the AppImage present on the Windows-release machine.
- Deterministic line order (sort by basename) so re-generation is stable.
- Emits nothing else; signing is a separate explicit call (§ 3.3), keeping
  the helper pure/​testable.

### 3.3 Two-phase SHA256SUMS (the split-publish consequence)

Because the exe is attached in a **separate later run** from the AppImage,
`SHA256SUMS` is built in two phases, each self-contained:

**Phase 1 — `release-linux.sh`, after the AppImage is built+signed+verified,
before/with publish:**
1. `gen-checksums.sh "$DIST/SHA256SUMS" "$DIST/$APPIMAGE"` — manifest with the
   AppImage line.
2. `sign-release.py … "$DIST/SHA256SUMS"` → `SHA256SUMS.sig`.
3. Add `SHA256SUMS` + `SHA256SUMS.sig` to the `gh release create`/`upload`
   asset list (§ 3.5).

**Phase 2 — `release-windows.sh`, after the exe is downloaded+signed+verified:**
1. **Fetch the current manifest** the Linux phase published:
   `gh release download "$TAG" -p SHA256SUMS -D "$DIST"` (so the AppImage
   line survives even though the AppImage binary isn't on this machine). If
   absent (Windows-first / Linux-skipped release), start fresh — the manifest
   then simply covers the exe only.
2. `gen-checksums.sh "$DIST/SHA256SUMS" "$DIST/$EXE"` — **merges** the exe
   line into the fetched manifest (§ 3.2).
3. Re-sign → `SHA256SUMS.sig` (the signature must cover the **final** bytes,
   so it is re-generated, not reused).
4. `gh release upload "$TAG" "$DIST/SHA256SUMS" "$DIST/SHA256SUMS.sig"
   --clobber` — replaces the phase-1 manifest with the complete one.

**Invariant that falls out:** whatever `SHA256SUMS` is on the release at any
moment, its accompanying `.sig` verifies it, and every basename it lists is a
real release asset (INV-1/INV-2). A brief phase-1-only window (AppImage
listed, exe not yet) is correct, not partial — the manifest always matches
the assets present.

### 3.4 SBOM per platform (pip-audit CycloneDX, in-build)

The SBOM must reflect what is **actually bundled**, which only exists in the
build venv (no lockfile, § 3.1). So generate it **where the freeze happens**,
not on the host:

- **Linux:** in `_build-smoke-in-container.sh`, **after** the runtime deps
  are installed into the build venv (`:53`) and before/after the freeze, run
  `python -m pip install pip-audit==<pinned>` (or rely on it already being
  installable) then
  `pip-audit --format cyclonedx-json --output /out/finbreak-$VERSION-linux.cdx.json`
  against that venv. Writes to `/out` (the same mount the AppImage lands in).
  *Honest scope:* this inventories the **pip-resolved build venv**, which is
  what PyInstaller collects from — a faithful bundled-deps list, **not** a
  byte-level proof every listed package ended up in the one-file (that's the
  §2 out-of-scope reproducibility concern). State this in the SBOM step's
  comment so it isn't over-read.
- **Windows:** the same step in `windows-build.yml`'s freeze job, writing
  `finbreak-$VERSION-windows.cdx.json` as a workflow artifact that
  `release-windows.sh` downloads alongside the exe.
- Two SBOMs (one per platform) because the bundled wheels differ per OS
  (PySide6/​pikepdf/​cryptography ship platform-specific binaries); a single
  SBOM would misrepresent one platform.
- `pip-audit` is already `dev`-pinned (`pyproject.toml`) and already run in
  the gate; using its CycloneDX output adds **no new dependency** (user
  decision). It doubles as the existing vuln-audit — same tool, extra output
  format.

### 3.5 Publish wiring

Add the new artifacts to the existing `gh release` asset lists:
- **`release-linux.sh`** — append `"$DIST/SHA256SUMS"`,
  `"$DIST/SHA256SUMS.sig"`, `"$DIST/finbreak-$VERSION-linux.cdx.json"` to
  both the `gh release create` and the `gh release upload --clobber` branches.
- **`release-windows.sh`** — append the re-generated `SHA256SUMS` +
  `.sig` (§ 3.3 phase 2) and `"$DIST/finbreak-$VERSION-windows.cdx.json"` to
  its `gh release upload --clobber`.
- The `build-smoke.sh` release-mode artifact-list echo (its "Release
  artifacts in $DIST" block) gains the new filenames so a human running it
  by hand sees them.

### 3.6 security-model.md impact

The published `SHA256SUMS` is a new **integrity** surface, so it earns a
note + an invariant (this is a `Kind: security` item on the supply-chain
path):
- A short note (near the signing / release-integrity discussion) that each
  release publishes a **signed** `SHA256SUMS` (manual-verification integrity
  signal, complementary to the per-artifact `.sig` the updater checks) and
  a per-platform CycloneDX SBOM (bundled-deps transparency).
- A **numbered invariant** (next free — currently **INV-13**; re-verify at
  authoring time): *"The published `SHA256SUMS` manifest is Ed25519-signed
  with the release key over its final bytes, and every basename it lists is
  a genuine release artifact whose recorded hash matches the uploaded file."*
  Falsifiable by the §4 checksum-helper + verify test. Frame it as an
  **integrity** guarantee (tamper-evidence), **not** confidentiality.
- No new threat *row* is required — it strengthens the existing T9
  (tamper/downgrade) posture for the download path; reference T9 rather than
  inventing a threat.

## 4. Invariants

- **INV-1 — Manifest format is `sha256sum -c`-valid.** `gen-checksums.sh`
  writes lines `<64-lowercase-hex>␠␠<basename>` (two spaces, basename only,
  sorted). *Test:* run the helper over two fixture files; assert exact line
  format, and that `sha256sum -c SHA256SUMS` **passes** in that dir and
  **fails** after a byte is flipped in a fixture.
- **INV-2 — Merge preserves prior lines.** Calling the helper with the exe,
  against a manifest already listing the AppImage, yields a manifest with
  **both** lines (AppImage line unchanged). *Test:* seed a manifest with an
  AppImage line, run the helper for an exe fixture, assert both basenames
  present and the AppImage hash byte-identical.
- **INV-3 — The manifest is signed over its final bytes.** After each phase,
  `SHA256SUMS.sig` verifies against `SHA256SUMS` with the release public key
  (`update_key.RELEASE_PUBLIC_KEY_B64`). *Test:* generate → sign → verify a
  fixture manifest with a throwaway keypair (mirror the existing
  sign/verify test); assert a 1-byte manifest edit fails verification.
- **INV-4 — Both release scripts publish the new artifacts.** `release-linux.sh`
  uploads `SHA256SUMS` + `.sig` + the linux SBOM; `release-windows.sh`
  re-uploads the merged `SHA256SUMS` + `.sig` + the windows SBOM. *Test:*
  scrape each script's source; assert the `gh release` asset lists contain
  the new filenames (windows_build-style source assertion).
- **INV-5 — SBOM is generated in-build, per platform, via pip-audit CycloneDX.**
  `_build-smoke-in-container.sh` and `windows-build.yml` each run
  `pip-audit --format cyclonedx-json` against the build venv, writing a
  platform-stamped `*.cdx.json` to the artifact output. *Test:* scrape both
  build scripts; assert the pip-audit CycloneDX invocation + the
  `-linux.cdx.json` / `-windows.cdx.json` output names exist.
- **INV-6 — Version-stamped off the single source.** Every new artifact name
  embeds `$VERSION` derived from `src/finbreak/__init__.py:__version__` (the
  bump source of truth), like the AppImage/exe already do. *Test:* scrape the
  scripts; assert the SBOM/​manifest names are built from the same `$VERSION`
  variable, not a hardcoded version.
- **INV-7 — Signed manifest ⇒ security-model INV recorded.** The
  `security-model.md` note + numbered invariant (§ 3.6) are present. *Test:*
  assert the doc contains the signed-`SHA256SUMS` invariant (grep-style), and
  the doc passes its own `/cold-eyes`.

## 5. Exit criteria (falsifiable)

1. **Checksums verify.** The helper emits a `sha256sum -c`-valid manifest;
   merge preserves prior lines; a flipped byte fails `-c` (INV-1/2).
2. **Manifest is signed + tamper-evident.** `SHA256SUMS.sig` verifies over
   the final bytes; a 1-byte edit fails (INV-3).
3. **Both platforms publish the artifacts.** Source assertions confirm
   `release-linux.sh` + `release-windows.sh` upload the manifest, its sig,
   and the per-platform SBOM; two-phase merge covers AppImage + exe
   (INV-4/2).
4. **SBOM is faithful + in-build.** Both build scripts generate the
   per-platform CycloneDX SBOM via pip-audit against the build venv (INV-5),
   version-stamped (INV-6).
5. **security-model.md updated + cold-eyes-clean.** The signed-manifest note
   + invariant land and pass their own `/cold-eyes` (INV-7).
6. **Gate green.** `./scripts/ci-local.sh` passes, including the new
   `tests/features/release_integrity/` suite. (The full release build stays
   opt-in behind `FINBREAK_BUILD_SMOKE=1`, as today — these tests are
   source-scrape + helper-unit, no real build.)
