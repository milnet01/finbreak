**Theme:** Six content-widget pop-ups still open **blocking** via
`dialog.exec()`. When an idle **auto-lock** fires while one is open, the timer's
`MainWindow._lock()` runs **inside that dialog's nested `exec()` event loop** →
`_clear_live()` → `self._live.deleteLater()` destroys the pop-up's parent chain
mid-loop → when `exec()` returns, the very next line reads a **deleted C++
object** (`dialog.selected_category_id()`, `dialog.password()`, …) → `RuntimeError`,
which the `VaultLockedError` guards do **not** catch (wrong exception, and the
read sits *before* the guard). This item converts all six to the shell's
**non-blocking** pattern: `setModal(True)` + `show()`, read the result in an
`accepted` slot. With no nested loop, a lock simply **destroys the pop-up before
its slot can fire** — no dead-object read, and the slot (when it does run) is
guaranteed to see an **unlocked** vault. The two static `QMessageBox` calls are
**left blocking** — proven safe (they read no dialog object after returning).

## Context — the crash, verified

Read at spec time (2026-07-10), not recalled:

**The lock path** (`ui/main_window.py`):

```
def _lock(self):
    self._teardown_dialog()      # (1) tears down ONLY self._dialog (unlock/manual/settings)
    self._service.lock()         # (2) close vault + wipe key — the security invariant
    self._clear_live()           # (3) removeWidget + self._live.deleteLater()  <-- destroys the workspace
    ...
```

`_teardown_dialog` (main_window.py:728) only frees `self._dialog` — the
**shell-tracked** dialogs (unlock, manual-entry, settings), which are *already*
non-blocking (`_open_dialog`: `setModal(True)` + `show()`, main_window.py:714).
The **content-widget** pop-ups (CategoryPicker, RuleEdit, AccountPicker,
Password) are **local variables** inside content-widget methods, opened with
`exec()`, and are **not** tracked in `self._dialog`. So step (1) does nothing to
them; step (3)'s `self._live.deleteLater()` is what destroys them (the live
content widget — a workspace tab's `QTabWidget`, or the import wizard that
replaces it).

**Why `deleteLater` bites mid-`exec()`:** `QDialog.exec()` spins a **nested**
`QEventLoop`. `deleteLater()` posts a `DeferredDelete` event; Qt processes
`DeferredDelete` inside that nested loop, so the content widget **and its child
dialog** are deleted while `exec()` is still on the stack. When `exec()` returns,
the next line calls a method on the now-deleted dialog:

- `home.py:175` `chosen = dialog.selected_category_id()` (after `_on_set_category`'s `exec()` at :174)
- `home.py:204` `dialog.pattern(), dialog.selected_category_id()` (`_maybe_offer_rule`, :203)
- `rules.py:172` / `:199` `dialog.pattern(), dialog.selected_category_id()` (`_on_add` / `_on_edit`)
- `statements.py:134` `dialog.selected_account_id()` (`_on_reassign`)
- `import_wizard.py:479-484` `dialog.password()`, `dialog.remember()`, `dialog.deleteLater()` (`_decrypt_pdf`)

Each is a Qt method on a deleted C++ object → **`RuntimeError: Internal C++
object already deleted`**. The existing `try/except VaultLockedError` guards
(e.g. home.py:184-189) sit **after** these reads and catch the **wrong**
exception — they were written for "lock closes the dialog *before* the flow",
not "lock *during* `exec()`". The audit reproduced this empirically (H-B).

**Why a Python-attribute access on a dead widget does *not* crash** (the basis
for INV-7): after the C++ object is deleted, the PySide6 **Python wrapper**
survives; reading a plain Python attribute set in `__init__` (e.g.
`self._statements`) touches no C++ and does **not** raise. Only **calling a Qt
method** (or reading a Qt property) on the dead object raises. The six sites
above all call a Qt method on the **dialog**; the two `QMessageBox` sites call
**no** dialog method after returning and only touch guarded Python services — so
they are safe (INV-7).

**The existing non-blocking pattern to mirror** (`main_window.py`):
`_open_dialog(dialog)` sets `setModal(True)` then `show()`; the shell connects
`dialog.accepted`/`dialog.rejected` to slots; `_teardown_dialog` frees it. The
content-widget pop-ups (CategoryPickerDialog, RuleEditDialog, AccountPickerDialog,
PasswordDialog) already build a `QDialogButtonBox` wired
`accepted→accept`/`rejected→reject`, so they already emit the standard
`accepted`/`rejected`/`finished` signals — **no dialog-class change is needed**,
only the **callers** change.

## Design decisions

- **D1 — One shared helper: `ui/modal.py::show_modal(dialog, on_accept)`.** The
  five simple sites repeat one shape, and the PDF prompt (D5) makes a sixth call
  site; extract it (Rule of Three — six call sites, coding.md § 1.3):

  ```python
  def show_modal(dialog: QDialog, on_accept: Callable[[], None]) -> None:
      """Show `dialog` non-blocking + modal; run `on_accept` when the user
      confirms; free the dialog on close. Replaces dialog.exec() so an idle
      auto-lock that destroys the parent widget can never crash a post-exec read
      (FIBR-0065).

      The caller MUST construct `dialog` with a parent widget: the parent keeps
      it alive between show() and the user's click (a parented QObject is not
      Python-GC'd while its C++ parent lives), and destroys it on lock (INV-2).
      Every call site already passes `self` as the parent."""
      dialog.setModal(True)
      dialog.accepted.connect(on_accept)
      dialog.finished.connect(dialog.deleteLater)
      dialog.show()
  ```

  `on_accept` takes no args and reads the dialog via closure capture (each caller
  already holds the `dialog` reference it needs). `finished` fires on **both**
  accept and reject (Qt emits `finished(result)` from `done()`, then
  `accepted`/`rejected`), so `deleteLater` always runs on close; because
  `deleteLater` is **deferred** to the next loop turn, the `on_accept` slot
  (which runs synchronously right after `finished` in the same `done()` call)
  still sees a **live** dialog. On a **lock**, the parent workspace's
  `deleteLater` destroys the still-open dialog before any accept — `on_accept`
  never runs (INV-2). The dialog stays alive between `show()` and the user's
  click because it is **parented** to the content widget (a parented `QObject` is
  not Python-GC'd while its C++ parent lives). This helper does **not** track
  `self._dialog` (that is shell state for the *shell's* dialogs) — a content
  pop-up's lifetime is owned by its parent widget + `finished`.

- **D2 — `home.py`: the set → offer → refresh chain becomes async (two slots).**
  Today `_on_set_category` runs, synchronously: `exec()` the picker → set → (maybe
  `exec()` the offer) → refresh. Split it:
  - `_on_set_category` opens `CategoryPickerDialog` via `show_modal(dialog,
    lambda: self._apply_category(dialog, txn))`. It **returns immediately**.
  - `_apply_category(dialog, txn)` (the picker's accept slot) reads
    `chosen = dialog.selected_category_id()`, calls `set_manual_category(txn.id,
    chosen)`, **`refresh()`** (the manual change is now visible), then
    `_maybe_offer_rule(txn.description, chosen)`.
  - `_maybe_offer_rule` keeps its FIBR-0010 D11 guard (`chosen is None` → return; rules
    already agree → return), then opens `RuleEditDialog` via `show_modal(dialog,
    lambda: self._apply_learned_rule(dialog))`.
  - `_apply_learned_rule(dialog)` (the offer's accept slot) reads
    `pattern, category_id`, `add_rule(...)`, `apply_rules()`, then **`refresh()`**
    again (propagation to other auto rows is now visible).

  **Refresh sequencing (parity, INV-5):** the old code refreshed **once**, after
  the offer resolved. The new code refreshes **after the set** (immediate manual
  feedback) and **again after a learned rule is applied** (propagation). A
  *dismissed* offer therefore refreshes once (after the set) — same end state as
  before. Two repaints when a rule is learned is a benign, intentional change
  (each is tied to a real state change); any test asserting "refresh called
  exactly once" updates to this sequencing.

  **Guard boundaries (INV-3, defense-in-depth).** Each new slot keeps one
  `try/except VaultLockedError: return`: `_apply_category`'s wraps
  `set_manual_category` + `refresh()` + the `_maybe_offer_rule` call (the same
  span the old single `try` covered, minus the offer body, which now lives in its
  own slot); `_apply_learned_rule`'s wraps `add_rule` + `apply_rules` +
  `refresh()`. The comments are corrected: the **primary** guard is now
  structural (a lock destroys the open pop-up before the slot can run, INV-2), so
  these `try/except`s are belt-and-suspenders, not the mechanism that prevents
  the crash.

- **D3 — `rules.py`: `_on_add` / `_on_edit` → non-blocking.** Each opens
  `RuleEditDialog` via `show_modal(dialog, lambda: self._apply_add(dialog))` /
  `self._apply_edit(dialog, rule.id)` and returns. The accept slot reads
  `pattern, category_id = dialog.pattern(), dialog.selected_category_id()`, then
  runs the **existing** body verbatim: the `try: add_rule/update_rule except
  VaultLockedError: return except (ValueError, FinbreakError): self._error.setText`,
  then `self._refresh()`. No behavioural change beyond the loop-turn split.
  `self._error.clear()` stays at the **top of `_on_add`/`_on_edit`** (before the
  dialog opens), matching today.

- **D4 — `statements.py`: `_on_reassign` → non-blocking.** Opens
  `AccountPickerDialog` via `show_modal(dialog, lambda:
  self._apply_reassign(dialog, statement))` and returns. The accept slot reads
  `new_account_id = dialog.selected_account_id()`, then the **existing** body: the
  same-account no-op (INV-5 of FIBR-0059), the `try: reassign_account except
  VaultLockedError: return except ValueError: QMessageBox.warning(...)`, then
  `refresh()` + `reassigned.emit()`. The `QMessageBox.warning` **inside** the
  accept slot stays a static blocking call — it is reached only on a *successful
  (unlocked)* service branch that raised the span-collision `ValueError`, the
  widget is alive, and nothing reads a dialog object after it (INV-7 reasoning).

- **D5 — `import_wizard.py`: the PDF password retry loop → a signal-driven state
  machine (the one non-trivial conversion).** `_decrypt_pdf(data) -> bytes | None`
  is a `while True` loop that `exec()`s a `PasswordDialog` per attempt and returns
  the plaintext synchronously to `_select_pdf`, which then runs a long tail (SB
  parse → generic extract → table combo → preview/map). Blocking `exec()` is
  intrinsic to "loop until the password is right or the user cancels", so this
  splits into an explicit state machine. Replace `_decrypt_pdf` + the tail of
  `_select_pdf` with:

  - `_select_pdf(path)` — reads the size-capped bytes (unchanged), then calls
    `self._begin_decrypt(data)` and **returns** (no synchronous plaintext).
  - **`_try_decrypt(self, data, password) -> bytes | _NeedPassword | None`** — the
    **single shared attempt+dispatch helper** (a real code artifact, not just
    prose — it preserves the old `while` loop's ONE `except`,
    import_wizard.py:488-495, so the non-password-error catch is not duplicated
    across the callers). It runs `PdfImporter.decrypt_to_plaintext(data, password)`
    and returns **exactly one of three**: the **plaintext `bytes`** on success; the
    `_NEED_PASSWORD` **sentinel** on `PasswordError`; or **`None`** after
    `self._error.setText(str(exc))` on **any** non-password error `(PdfError,
    ValueError, OSError, FinbreakError)`. The sentinel is a **distinct type**, not
    `object()` — `class _NeedPassword: ...` with a module-level
    `_NEED_PASSWORD = _NeedPassword()` — so the `bytes | _NeedPassword | None`
    annotation resolves under mypy (the module carries `from __future__ import
    annotations`; a bare `_NEED_PASSWORD = object()` would fail mypy's "Variable is
    not valid as a type", breaking the INV-9 gate). Because the non-password-error
    catch lives **only here**, one test over `_try_decrypt` (INV-6f) genuinely
    exercises the branch for the initial, stored, **and** prompted attempts — none
    can silently drop it. The callers branch on the 3-way return (`isinstance(r,
    bytes)` / `r is _NEED_PASSWORD` / `r is None`) and hold **no `try/except` of
    their own** (this is what makes the "shared dispatch" real, closing the
    Rule-of-Three duplication of the old single `except` across three sites).
  - `_begin_decrypt(data)` — `r = _try_decrypt(data, None)`. `bytes` →
    `_after_decrypt(r, password=None, remember=False)`; `None` → stop (message
    already shown); `_NEED_PASSWORD` → try the **stored** password **once**
    (`get_pdf_password(self._target_account_id())`, FIBR-0009 INV-4): if present,
    `r2 = _try_decrypt(data, stored)` and branch the same three ways (`bytes` →
    `_after_decrypt(r2, stored, remember=False)`; `_NEED_PASSWORD` → prompt; `None`
    → stop); if absent → prompt. Trying the stored password **only inside
    `_begin_decrypt`** (entered once per import) structurally guarantees it is
    attempted **at most once** and never re-tried on a re-prompt — the old
    `tried_stored` latch, expressed by control flow rather than a flag.
  - `_prompt_pdf_password(data)` — builds `PasswordDialog(self._account_name(),
    self)` and `show_modal(dialog, lambda: self._on_pdf_password(dialog, data))`.
    A **Cancel** (reject) needs no slot — `finished→deleteLater` frees it and the
    wizard simply stays on the pick step (the old `return None` "cancel abandons
    the import cleanly"), storing nothing.
  - `_on_pdf_password(dialog, data)` (the accept slot) — reads
    `password, remember = dialog.password(), dialog.remember()`, then
    `r = _try_decrypt(data, password)`: `bytes` → `_after_decrypt(r, password,
    remember)`; `_NEED_PASSWORD` → `_prompt_pdf_password(data)` again (the
    **re-prompt**, FIBR-0009 INV-3); `None` → stop (message already shown).
  - `_after_decrypt(plaintext, password, remember)` — **iff `remember and
    password`** (only now — decrypt succeeded, so the password is verified), with
    `account_id = self._target_account_id()`, does **both** writes **inside that
    one branch** (matching import_wizard.py:497-502): `self._accounts.
    set_pdf_password(account_id, password)` **and** `self._stored_pw = (account_id,
    password)` (the FIBR-0057 carry). On the unencrypted and stored-success paths
    (`remember` False or `password` None) it stores **nothing** — neither write
    fires, so no `(account_id, None)` is ever stashed. Then calls
    `_continue_after_decrypt(plaintext)`. (`data` is **not** a parameter — nothing
    past decrypt needs the ciphertext.)
  - `_continue_after_decrypt(plaintext)` — the current `_select_pdf` tail verbatim
    (lines 418-456): the SB-parse fast path, `_extract_pdf_tables`, the table
    combo population, `_apply_pdf_table`, `_goto_step(_STEP_MAP)` or the
    single-candidate `_run_preview`.

  **Async-boundary safety.** The only event-loop turn the machine crosses is the
  password **prompt**; every other transition is synchronous. Two consequences:
  (1) `self._target_account_id()` is read on both sides of that boundary
  (`_begin_decrypt` for `get_pdf_password`, `_after_decrypt` for
  `set_pdf_password`) — safe because the wizard sits on the **pick step** while a
  password is prompted (the preview-step confirm-account combo that
  `_target_account_id` reads is not reachable yet), so the account cannot change
  mid-flow. (2) The D5 accept slots carry **no *dedicated*** `VaultLockedError`
  guard (unlike D2–D4, which *retain* a pre-existing `except VaultLockedError:
  return`). A lock error is not un-handled, though: `VaultLockedError` is a
  `FinbreakError` subclass (`errors.py`), so `_try_decrypt`'s `except (…,
  FinbreakError)` would surface it as a shown message + `None` rather than a silent
  return — but INV-2 makes even that moot: a lock during the prompt destroys the
  wizard before `_on_pdf_password` can fire, and the vault reads (`get_pdf_password`
  / `set_pdf_password`) only run inside a slot a lock would have pre-empted by
  destroying the wizard. Each attempt's dialog is freed by `show_modal`'s
  `finished→deleteLater` (parity with the old per-attempt `dialog.deleteLater()`,
  indie-review M1), so a wrong-password re-prompt never leaves a live dialog holding
  the typed password (INV-4).

- **D6 — Comment hygiene (in-scope orphan cleanup).** The `exec()`-era comments
  that describe the *old* hazard (home.py:179-183 "the learning offer's dialog
  pumps the event loop via `exec()`…"; the `# auto-lock fired while the dialog was

  fix — the pop-up is now non-blocking, so a lock **destroys it before the slot
  runs**; the retained `VaultLockedError` guard is defense-in-depth. This is
  cleanup the change itself creates (global rule § 11), not a drive-by. The
  `INV-14` citations in those comments (the FIBR-0010 auto-lock invariant) are
  updated to also cite FIBR-0065. **FIBR-0010 INV-14 reconciliation:** that
  invariant's stated *mechanism* ("auto-lock raises `VaultLockedError`, which is
  caught") is **superseded** by FIBR-0065 INV-2 — a lock during a blocking `exec()`
  raises `RuntimeError`, not `VaultLockedError`, so the guarantee now holds
  *structurally* (non-blocking pop-ups), not via the guard alone. `FIBR-0010.md`'s
  INV-14 row is annotated with this forward-pointer (a light note on the closed
  spec, not a rewrite). **Comment-token caution (INV-1):** the rewritten comments
  must not spell the literal `.exec(` token (write "exec()" / "the exec loop"), or
  the INV-1 grep would fire on the comment.

## Invariants

Restated with their enforcement point (the table above is the index).

- **INV-1 (no blocking pop-up)** — enforced by construction + a **grep test** in
  `tests/features/dialog_lifecycle/`. Operational definition (so the test is
  mechanically writable): **no line matching `\.exec\(` in `ui/home.py`,
  `ui/rules.py`, `ui/statements.py`, `ui/import_wizard.py`, EXCEPT the single
  `menu.exec(` in `home.py`** (the Home right-click "Set category…" context
  *menu* — a `QMenu`, not a modal dialog; it reads no dialog object after and is
  out of scope, per Out-of-scope). The token must be absent from **code AND
  comments** — so the D6 comment rewrites must not spell the literal
  `dialog.exec()` (write "exec()" / "the exec loop" without the `.exec(`), or the
  grep would fire on the corrected comment. `main_window.py` has **no** `.exec()`
  at all
