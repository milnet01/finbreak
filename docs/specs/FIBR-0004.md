**Theme:** prove the whole security model end-to-end with the
smallest real app. A first-run user sets a master password and a
base currency; the app derives a key with **Argon2id**, creates an
**AES-256 SQLCipher** vault, lets them enter **one** transaction
(written through a repository in a single DB transaction) and see
it in a table; **lock** wipes the key from memory and returns to
the unlock screen, and the on-disk file is unreadable without the
password. Everything else — accounts, import, dashboard, export —
is deliberately out of scope. This is the vertical slice that
turns the security model from a document into tested code.

## Context

P01 (Bootstrap) shipped the quality/security gate (FIBR-0001), the
data/build-output `.gitignore` guard (FIBR-0002), and the bundling
smoke-test that proved Qt + SQLCipher + qpdf freeze into a
Python-free artifact (FIBR-0003). No application logic exists yet:
`python -m finbreak` with no arguments prints a `FINBREAK_NOT_BUILT`
placeholder, and there is no UI, no vault, and no crypto.

This item builds the **spine** the rest of the app hangs off. Per
design.md the app is layered — **UI → Services → Repositories →
SQLCipher** — and the UI never touches storage directly. P02 stands
up the thinnest vertical cut through every layer:

- **CryptoService** — Argon2id key derivation (the one auditable
  crypto path, design.md "Components").
- **AuthService** — turn the master password into the vault key via
  CryptoService, open/close (unlock/lock) the vault, wipe the key on
  lock, and drop it on idle auto-lock.
- **Vault + one repository** — the single owned SQLCipher connection
  and a `TransactionRepository` that inserts and lists transactions.
- **Three real (but minimal) PySide6 screens** — first-run setup,
  unlock, and a main window with an add-transaction form and a
  table. Built translation-ready and RTL-safe **from this first
  screen** per design.md i18n and coding.md § 5.2 — it is far
  cheaper than retrofitting.

Getting this slice right de-risks every later feature: if the
key-derivation, vault-open, lock/wipe, and single-transaction-write
paths are correct and tested here, P03+ add tables and screens onto
a proven foundation.

## Design decisions baked into this spec

These are the non-obvious calls this spec makes; `/cold-eyes` and
the user sign-off gate them before code.

1. **D1 — Money is stored as an exact integer number of minor units,
   never a binary float.** User input is parsed with
   `decimal.Decimal` and scaled to a signed integer `amount_minor`
   (negative = money out). A binary `float` cannot represent `0.10`
   exactly; baking float money into the foundation would seed
   rounding drift through every later total. The base currency's
   **minor-unit exponent** (2 for the currencies this slice offers)
   is recorded in settings alongside the code, so a 0-decimal (JPY)
   or 3-decimal currency added later is a data migration, not a
   silent reinterpretation of existing rows.

2. **D2 — The Argon2id salt and KDF parameters live in a plaintext
   sidecar file beside the vault**, not inside it. This resolves a
   bootstrap chicken-and-egg: the app must derive the key *before*
   it can open the encrypted database, so the KDF inputs must be
   readable *without* the key. A salt and KDF cost parameters are
   **non-secret by design** (a salt's job is uniqueness, not
   secrecy), so a plaintext sidecar is the standard, auditable
   choice. This is the concrete form of security-model INV-2's
   requirement that the parameters and salt are recorded *with the
   vault* — see the **Note on INV-2 wording** below.

3. **D3 — The vault key is derived as SQLCipher's _raw_ key**, not a
   passphrase run through SQLCipher's own PBKDF2. The 32-byte
   Argon2id output is passed as `PRAGMA key = "x'<64-hex>'"`, so
   Argon2id — not SQLCipher's KDF — is the key-stretching function
   (security-model A3, INV-2). SQLCipher's `kdf_iter` is bypassed
   entirely in raw-key mode.

4. **D4 — Per-page HMAC integrity stays at the SQLCipher 4 default
   (`cipher_use_hmac = ON`, HMAC-SHA512).** AES gives confidentiality,
   not integrity; the HMAC makes a tampered page fail to open rather
   than return corrupt data (security-model T9, INV-1). This slice
   asserts the default is in force rather than re-configuring it.
   **(FIBR-0077 revisit, 2026-07-10.)** `_connect` now *also* issues
   `PRAGMA cipher_use_hmac = ON` explicitly, right after `PRAGMA key`.
   Pinning ON can never mismatch an existing file (every vault is created
   with the default ON), and it makes INV-1 correct-by-construction
   instead of resting on `sqlcipher3-binary`'s default — which a future
   dep bump could silently flip (global rule §5). The assert stays as the
   regression check; the pin is the defense-in-depth.

5. **D5 — Secrets are held in zeroable `bytearray` buffers, wiped in
   place on lock, idle auto-lock, and exit.** Python `str`/`bytes` are immutable and
   cannot be wiped; a `bytearray` can be overwritten. Argon2id
   returns immutable `bytes`, so the derived key is copied into a
   `bytearray` and the original reference dropped. This is
   best-effort defence-in-depth (the OS may still page memory to
   swap — explicitly out of scope per security-model § 4), and it is
   what makes INV-3's "wiped" **testable**.

### Note on INV-2 wording (resolved)

security-model INV-2 requires the parameters be "recorded **with the
vault**." Because the Argon2id inputs must be readable *before* the
key exists, this spec stores them in a plaintext sidecar *beside* the
vault file, not inside the encrypted database — that sidecar *is* the
concrete form of "with the vault." (A single loose "in the vault"
clause in the same INV was tightened to "with the vault" on
2026-07-01 alongside this spec's sign-off, so the two docs now
agree.)

## Invariants (testable)

> These `INV-N` are **local to this spec**. Where this spec upholds
> a project-wide security invariant it names it (e.g.
> "security-model INV-2"). Tests live in
> `tests/features/vault/` (spec.md + `test_vault.py`) unless noted,
> use `tmp_path` for every on-disk vault, and never touch the
> network or real financial data (testing.md § 6, security-model
> INV-6). GUI-touching invariants use the `pytest-qt` `qtbot`
> fixture (this item enables the plugin for the first time, by
> removing the `-p no:pytest-qt` line — see Deliverables).

- **INV-1 — Encrypted at rest, integrity-checked.** After the vault
  is created and closed, the on-disk file's first 16 bytes are
  **not** the SQLite plaintext magic (`b"SQLite format 3\x00"`) —
  the file is ciphertext. Re-opening with the correct derived key
  succeeds; opening with a wrong key (or after flipping a byte in
  the file body) raises `sqlcipher3.dbapi2.DatabaseError` on the
  first read (`SELECT count(*) FROM sqlite_master`), not silent
  corruption. The connection reports `PRAGMA cipher_use_hmac` on and
  `PRAGMA cipher_hmac_algorithm` = `HMAC_SHA512`. Source:
  security-model INV-1, T1, T9.

- **INV-2 — Strong, pinned, non-downgradeable KDF.** The key is a
  32-byte Argon2id output derived with the pinned parameters —
  **memory = 47104 KiB, time cost = 1, parallelism = 1**, from a
  unique **16-byte** random salt per vault (security-model INV-2).
  Enforced by three sub-invariants:
  - **INV-2a — Correct derivation.** For a fixed password + salt +
    params, `CryptoService` returns exactly 32 bytes, and the same
    inputs are deterministic (equal outputs); a different salt gives
    a different key. The 32-byte output is what is passed to
    `PRAGMA key` as the raw key (asserted by INV-1's successful
    re-open, making the "Argon2id is the KDF" claim testable).
  - **INV-2b — Strength floor on open.** Opening a vault whose
    recorded `memory_kib` is **below** the floor (47104 KiB) is
    refused — it raises a dedicated `KdfPolicyError` (a
    finbreak-defined exception, distinct from
    `sqlcipher3.dbapi2.DatabaseError` so the test asserts the *right*
    failure, not any exception); a recorded value **at or above** the
    floor opens. The floor is directional so the creation pin can be
    raised later without locking out existing vaults.
  - **INV-2c — Exact-format on open.** The **salt** exists on disk
    (decoded from `salt_hex`), so the check requires both
    `len(salt) == 16` **and** the recorded `salt_len == len(salt)` (a
    recorded field disagreeing with the real salt is itself a reject).
    The **key** is never on disk — it is derived fresh to
    `KEY_LEN = 32` each open — so its check is the recorded
    `key_len == 32` field vs the `KEY_LEN` constant. A *longer*
    recorded output or salt is rejected, not accepted. A
    malformed sidecar — non-JSON, truncated, **or a JSON object
    missing any of the seven required `KdfParams` fields (§ Data
    model)** — likewise raises `KdfPolicyError`, never an uncaught
    `JSONDecodeError` or `KeyError`. All refusals raise
    `KdfPolicyError`.
  Source: security-model INV-2, A3, T2, T9.

- **INV-3 — Key lifetime and wipe.** The derived key and the
  plaintext password exist in memory only while unlocked. After
  `AuthService.lock()` the key buffer is all-zero (a `bytearray`
  overwritten in place) and the vault connection is closed; a query
  through the (now-locked) service raises `VaultLockedError` (no key
  in memory) rather than returning data. The plaintext-password
  buffer is zeroed before it returns — for `unlock()` on **both** the
  success and wrong-password paths, and for `validate_first_run()` /
  `first_run()` on return (first-run has no password *check* — a
  mismatch is caught in `validate_first_run`, INV-5). The key is wiped
  on **all three** of
  security-model INV-3's triggers — **lock**, **idle auto-lock**, and
  **application exit**:
  - *idle auto-lock:* the idle-timeout handler performs the same
    wipe+close as a manual lock (asserted by invoking the handler
    directly — no wall-clock wait, testing.md § 6).
  - *exit:* a `QApplication.aboutToQuit` handler wipes the key buffer
    on shutdown (asserted by invoking the handler and checking the
    buffer is all-zero); it is a **no-op when already locked** (no key
    held), never a crash on an absent buffer.
  Source: security-model INV-3 ("wiped on lock/exit … dropped by
  auto-lock"), T3 ("wiped on lock and on exit"), A2, A3.

- **INV-4 — One transaction, one DB transaction, exact money.**
  - **INV-4a — Atomicity + exact round-trip.**
    `TransactionRepository.add()` writes a single row inside **one**
    SQLCipher transaction. The connection opens in **manual-commit
    mode** (`sqlcipher3.dbapi2`'s DBAPI default `isolation_level=""`,
    *not* `None`/autocommit) and the write is delimited by an
    **explicit `conn.commit()`** — so an INSERT with no following
    commit rolls back, which is what gives the test a wedge point. The
    test forces a
    mid-transaction failure by monkeypatching the repository to raise
    *after* the INSERT statement executes but *before* `commit()`, then
    asserts the row count is **zero when read from a freshly re-opened
    vault** (a rolled-back INSERT is invisible to a new connection) —
    atomicity, design.md "Persistence". A
    `Decimal("-12.34")` entered against a 2-decimal base currency is
    stored as `amount_minor = -1234` and reads back as exactly
    `Decimal("-12.34")` — no float round-trip. The **repository
    stores and returns the raw integer** `amount_minor`; the
    **service layer** reconstructs the display `Decimal` as
    `Decimal(amount_minor).scaleb(-minor_unit_exponent)`, reading
    `minor_unit_exponent` from `settings` (stored `TEXT` like every
    `settings` value; the service **coerces it to `int`** before
    scaling) — so the scale lives in one place, not re-inferred per
    call-site.
  - **INV-4b — Money input is validated, never silently mutated.**
    The parse/validate step raises `ValueError` (caught at the form
    boundary and shown as a field error — testable by asserting the
    `ValueError` per rejected input) when the input: has **more
    fractional digits than `minor_unit_exponent`** (e.g. `1.005`
    against exponent 2 — never silently rounded, since rounding
    mutates money), is **not a finite decimal**, is **zero**, the
    **description is empty**, or the **`occurred_on` date is not a
    valid ISO-8601 date** (the form provides a date picker defaulting
    to today; a future date is allowed). No row is stored on any of
    these. A
    well-formed **non-zero** value at or within the exponent is
    accepted — **either sign** (negative = money out, positive = money
    in, per D1); a very large magnitude is accepted as-is within
    SQLite's signed 64-bit `INTEGER` range (±2⁶³−1) — bounding beyond
    that is out of scope (no realistic personal-finance amount
    approaches it).
  Source: this spec (decision D1); design.md "Persistence".

- **INV-5 — First-run vs unlock routing (with mixed-state guard).**
  With **neither** vault nor sidecar present the app enters
  **first-run** (set password + base currency); with **both** present
  it enters **unlock**. A **mixed** state — sidecar present but vault
  missing, or vault present but sidecar missing — is treated as a
  corrupt/incomplete install and surfaced by raising `VaultStateError`
  **before any first-run write** (never a silent re-first-run, which
  would overwrite a real vault or derive a key with no matching
  data). First-run requires the password to be entered twice; a
  **mismatch is a form-boundary validation error that creates no
  vault and derives no key** (testable via the first-run validator,
  headless — no `qtbot` needed). The base currency is chosen from the
  offered 2-decimal set. First-run creates the **vault (with its
  settings + `schema_version` rows) first, then writes the sidecar
  last** — so a crash mid-first-run leaves at most a vault without a
  sidecar, which the next launch catches as the same `VaultStateError`
  mixed state above (the user resolves it by deleting the partial
  install; no automatic recovery in this slice). After successful
  first-run the vault, its sidecar, and the settings rows
  (`base_currency`, `minor_unit_exponent`, `schema_version`) exist.
  Source: design.md
  "Components" (Unlock / first-run), roadmap FIBR-0004.

- **INV-6 — Unlock/lock round-trip through the UI.** Entering the
  correct password on the unlock screen opens the main window and
  the previously-saved transaction appears in the table; entering a
  wrong password is refused with a clear message and no crash and no
  key retained (re-asserts INV-3 from the UI). The service contract
  behind the UI: `AuthService.unlock()` runs a fixed **open
  sequence** — (1) validate the sidecar KDF record (raises
  `KdfPolicyError` on a below-floor / malformed record, INV-2b/2c),
  (2) derive the key, (3) open the vault. A wrong password surfaces
  at step 3 as the `DatabaseError` the wrong raw key raises on first
  read, which `unlock()` catches and turns into **`return False`**
  (an expected outcome, not an exception) with no key retained; a
  correct password returns `True`. Because the policy check (step 1)
  precedes the password check (step 3), a below-floor-sidecar vault
  and a wrong password are **deterministic and independently
  testable**, and the UI's "clear message" has a headless source
  (assert `unlock(wrong)` is `False`, no key held). A **tampered
  vault body** (INV-1's flipped byte) also raises `DatabaseError` at
  step 3, so it too maps to `return False` and is reported to the
  user as a failed unlock — a wrong password and a tampered vault are
  **deliberately indistinguishable** at unlock, because without the
  correct key the app cannot tell them apart (the HMAC that would
  prove tamper is itself keyed). This is an accepted stance, not an
  oversight. Pressing **Lock** returns to the unlock screen and wipes
  the key (INV-3). Source: roadmap FIBR-0004; design.md "Data flow"
  (the "Feedback loop" paragraph).

- **INV-7 — No plaintext secret on disk or in logs.** No code path
  writes the master password, the derived key, or the raw-key hex to
  disk, to the sidecar, or to the log file; the sidecar contains
  only the salt (hex) + non-secret KDF parameters + format version
  (security-model INV-9, A2, A3). Asserted by scanning the sidecar
  contents and — for the log leg — capturing this slice's own logging
  (a `caplog`/`propagate` capture over the `finbreak` logger during a
  first-run → unlock → lock cycle) and asserting the password/key
  bytes never appear. To make this leg **non-vacuous**, the slice
  emits a few **non-secret lifecycle lines** via the `finbreak`
  logger at `INFO` — e.g. "first-run: vault created", "unlock
  attempted", "unlock failed", "locked" — carrying no password, key,
  or amount. The full rotating **log file** is a later observability
  phase (design.md "Observability"); this leg proves the lifecycle
  lines are clean and becomes fuller when that lands.
  Additionally, the vault file **and** the sidecar are created with
  restrictive **`0o600`** (owner-only) permissions per coding.md § 7
  (which names the vault explicitly), asserted on POSIX via
  `path.stat().st_mode & 0o777 == 0o600` (skipped on Windows, where
  POSIX mode bits don't apply). Source: security-model INV-9, A2, A3
  (the master password A2 and derived key A3 are "never stored
  anywhere" / "in memory only"); coding.md § 7 (perms). *Note:
  security-model INV-4 is scoped to decrypted statement/PDF bytes, an
  import concern with no code path in this slice — it is deliberately
  not cited here.*

- **INV-8 — No network dependency introduced.** No `socket`,
  `http`, `https`, `urllib`, `urllib.request`, `requests`, or
  `ftplib` network import appears anywhere under `src/finbreak/`, and
  no networking package is added to `pyproject.toml` runtime deps.
  (`asyncio` is deliberately **not** on this list — it is an event
  loop, not a network transport, so scanning for it would
  over-reject.) This
  lands the runtime-code leg of security-model INV-8 that INV-8
  itself says "lands with the first runtime code." Source:
  security-model INV-8, § 5.

## Deliverables

**New application modules** (`src/finbreak/`; shortest correct
implementation per coding.md § 1.1 — no scaffolding for later
phases):

- `paths.py` — resolve the per-OS-user data directory via
  `QStandardPaths.AppDataLocation` (design.md "Persistence"); expose
  the vault path and the KDF-sidecar path. Set the app name to
  `finbreak` via `QCoreApplication.setApplicationName` so the path
  resolves consistently.
- `crypto.py` — `CryptoService` with the pinned Argon2id constants
  (`ARGON2_MEMORY_KIB = 47104`, `ARGON2_TIME_COST = 1`,
  `ARGON2_PARALLELISM = 1`, `KEY_LEN = 32`, `SALT_LEN = 16`) and
  `derive_key(password: bytearray, salt: bytes, params: KdfParams)
  -> bytearray` over `argon2.low_level.hash_secret_raw(...,
  type=Type.ID)`. Pass `memory_cost=47104` — the API argument is in
  **KiB**, so use `47104`, not `46` (security-model INV-2's "47104
  KiB (46 MiB)" is one value; "46 MiB" is its human-readable gloss,
  not the API argument). **Always pass these by keyword** — the
  low-level signature puts `time_cost` before `memory_cost`, so
  positional args silently transpose them. Also the on-open floor + exact-format +
  malformed-sidecar validation that raises `KdfPolicyError`
  (INV-2b/2c). **Timing disposition: correctness-only, no perf INV**
  — the pinned params (security-model INV-2) are the sole governor;
  derivation latency is deliberately **unmeasured and unbudgeted**,
  and derivation runs off the GUI thread (below) so a slow machine
  never freezes the UI. Do not tune the cost params.
- `models.py` — `@dataclass` records: `Transaction` (id,
  `occurred_on`, `amount_minor`, `description`, `created_at`) and
  `KdfParams` (`format_version`, `memory_kib`, `time_cost`,
  `parallelism`, `key_len`, `salt_len`, `salt: bytes`). The sidecar
  JSON is the same fields **flat, name-for-name — except `salt`
  (bytes) serialises as the hex string `salt_hex`** (JSON has no
  bytes type) — see Data model. No nested objects, so the on-disk ↔
  dataclass mapping is that one rename and nothing else.
- `errors.py` — the finbreak exception taxonomy (coding.md § 2,
  "specific exceptions over generic"): `KdfPolicyError` (a bad
  recorded KDF record — below floor, wrong lengths, or a malformed /
  missing-field sidecar; INV-2b/2c), `VaultStateError` (an on-disk
  vault/sidecar **presence** mismatch; INV-5), and `VaultLockedError`
  (an operation attempted while locked; INV-3). Named types so every
  INV failure-leg asserts the *right* exception, not a bare `raises`.
- `vault.py` — the single owned SQLCipher connection: read/write the
  plaintext sidecar (`json`), derive-and-open with the raw-key
  pragma **as the first statement**, assert HMAC defaults, create
  schema on first-run (`schema_version` = 1), and close. The sidecar
  is written **atomically** (temp file + `os.replace`, coding.md § 7)
  so a crash mid-write cannot leave a truncated sidecar; the vault
  and sidecar are created **`0o600`** (INV-7, coding.md § 7).
  Wrong-key / tamper detection per INV-1; the vault/sidecar presence
  guard (`VaultStateError`, INV-5); and refusing operations after
  lock (`VaultLockedError`, INV-3).
- `repositories/transactions.py` — `TransactionRepository.add()` /
  `.list_all()`, each write wrapped in one explicit transaction.
- `services/auth.py` — `AuthService`:
  `validate_first_run(password: bytearray, confirm: bytearray,
  base_currency: str) -> None` (raises `ValueError` on a password
  mismatch, an empty password, or a base currency outside the offered
  set — the headless entry point INV-5's first-run-mismatch leg
  asserts, before any key is derived or file written; **non-empty is
  the only enforced password rule** — length/complexity is
  advisory-only, deferred to FIBR-0029, per security-model T2);
  `first_run(password: bytearray, base_currency: str) -> None`
  (assumes a validated password); `unlock(password: bytearray) ->
  bool` (`False` on wrong password, no key retained — INV-6);
  `lock()` (wipe key + close); and the idle auto-lock timer.
  Passwords cross the service boundary as **`bytearray`** (matching
  `derive_key`) so they can be wiped in place; the originating
  `QLineEdit.text()` `str` is unavoidably unwipeable (best-effort per
  D5 — convert to `bytearray` early and drop the `str`).
  **Thread model (keeps the wipe single-threaded):** the **key buffer
  is owned and wiped only on the GUI/main thread**. The worker thread
  (below) merely *computes* the raw key and returns the 32 bytes via
  a signal; the main thread copies them into its zeroable buffer. So
  every wipe (lock, idle, exit) runs on the owning thread with no
  cross-thread race, and the INV-3 tests invoke each handler on that
  same thread — the real path, not a bypass. The auto-lock default is
  a **fixed `AUTO_LOCK_MINUTES = 10`** idle, a placeholder until
  FIBR-0014's Settings screen makes it user-configurable (out of
  scope here); the INV-3 test invokes the timeout handler directly,
  so the number is not what the test asserts. UI responsiveness while
  the worker derives (the unlock control shows a busy state) is a
  design property, not a separately-asserted INV.
- `ui/first_run.py`, `ui/unlock.py`, `ui/main_window.py` — the three
  minimal screens, all strings via `self.tr(...)`, layouts via Qt
  layout managers (RTL-safe), amounts formatted via `QLocale` per
  coding.md § 5.2. Blocking key derivation runs off the GUI thread
  (a `QThread`/worker) so the UI stays responsive (design.md
  "Concurrency"). The first-run base-currency picker offers a
  minimal **2-decimal** set — `ZAR`, `USD`, `EUR`, `GBP`, `AUD`,
  `CAD` (a named module-level constant, each recorded with
  `minor_unit_exponent = 2`); the list (and 0-/3-decimal currencies)
  is extended in a later phase (D1).
- `app.py` (or the routing added to `__main__.py`) — build the
  `QApplication`, set the app name, and route first-run vs unlock
  (INV-5).

## Data model (minimal; forward-compatible)

Single SQLCipher database, `schema_version = 1`. Only what the slice
needs; later phases migrate forward (design.md "Persistence",
forward-only, each migration in a transaction):

- `schema_version(version INTEGER NOT NULL)` — exactly one row,
  upheld by first-run-only creation (guarded by INV-5's mixed-state
  check), not a schema constraint.
- `settings(key TEXT PRIMARY KEY, value TEXT NOT NULL)` — holds
  `base_currency` (ISO code) and `minor_unit_exponent` (e.g. `2`).
- `transactions(id INTEGER PRIMARY KEY, occurred_on TEXT NOT NULL,
  amount_minor INTEGER NOT NULL, description TEXT NOT NULL,
  created_at TEXT NOT NULL)` — signed minor units; ISO-8601 dates.

The plaintext **KDF sidecar** (`vault.kdf.json` beside the vault),
readable before unlock — a **flat** object matching `KdfParams`
field-for-field **except `salt` (bytes) → `salt_hex` (hex string)**:
`{format_version, memory_kib, time_cost, parallelism, key_len,
salt_len, salt_hex}`. Written atomically and `0o600` (INV-7). Contains
**no secret** (the salt and cost parameters are non-secret; the
password, derived key, and raw-key hex never appear here).

## Out of scope (deferred to named phases)

- **Accounts model.** `transactions` has no `account_id` yet; the
  Accounts phase (P03) adds it via a forward migration that backfills
  a default account. This slice's transaction is intentionally
  account-less.
- **Import / categorisation / transfers / dashboard / export** —
  their own phases (FIBR-0007+). Only *manual* single-transaction
  entry is in scope.
- **User-configurable auto-lock timeout, stored PDF passwords, theme,
  backup export** — the Settings screen (FIBR-0014). P02 ships the
  auto-lock *mechanism* with the fixed `AUTO_LOCK_MINUTES = 10`
  default only (not user-configurable until FIBR-0014).
- **Password strength meter beyond a minimal advisory hint, password
  hint, and destructive reset** — FIBR-0029 / FIBR-0030. First-run
  shows at most a non-blocking advisory; it does **not** enforce a
  strength policy (security-model T2: advisory, not an INV).
- **macOS / Windows packaging of the new native dep.** Note: the
  pinned `sqlcipher3-binary==0.6.0` currently publishes **Linux
  x86-64 wheels only**; the cross-platform freeze is P13
  (FIBR-0015/0016) and must revisit the wheel source there. P02
  develops and tests on Linux; this is a flagged P13 risk, not a P02
  blocker.
- **Schema migration framework.** P02 creates `schema_version = 1`
  directly; the forward-migration runner arrives with the first
  phase that needs a version bump (P03).

## Verified API basis

Pins the idioms the INVs assert against, so the contract cites real
APIs (global rule § 13). Verified 2026-07-01 against the upstream
argon2-cffi docs (readthedocs), the SQLCipher API reference
(Zetetic), and PyPI; re-confirm at implementation time per FIBR-0001
INV-4:

- **argon2-cffi 25.1.0** —
  `argon2.low_level.hash_secret_raw(secret, salt, time_cost,
  memory_cost, parallelism, hash_len, type, version=19)`;
  `memory_cost` in **KiB**; `type=Type.ID` for Argon2id; returns
  `bytes`. Native lib vendored in the wheel.
- **sqlcipher3-binary 0.6.0** — `from sqlcipher3 import dbapi2`;
  raw-key open `PRAGMA key = "x'<64 hex>'"` **as the first
  statement**; raw key bypasses SQLCipher's PBKDF2 (`kdf_iter`
  irrelevant); SQLCipher-4 defaults `cipher_use_hmac = ON`,
  `cipher_hmac_algorithm = HMAC_SHA512`, `cipher_page_size = 4096`;
  wrong key / tamper → `DatabaseError` ("file is not a database") on
  first read. **Caveat:** 0.6.0 publishes Linux x86-64 wheels only —
  a P13 cross-platform packaging concern (see Out of scope).
- **Zeroing** — `bytearray` is overwritable in place; `str`/`bytes`
  are not. `hash_secret_raw` returns immutable `bytes`, copied into a
  `bytearray` for wipeability. Best-effort only (swap out of scope,
  security-model § 4).
