## The problem (ADR-0007 + the stale-pin trap)

ADR-0007 requires every release to be **fully self-contained**: download one
file, run it, it works, on a clean machine with **no Python installed**. The
Linux AppImage delivers this today. Windows does not — there is no `.exe`.

The 2026-07-13 readiness scan recorded Windows as blocked because
`sqlcipher3-binary==0.6.0` (the pinned SQLCipher engine, ADR-0003) ships
**Linux/macOS wheels only** — no Windows wheel — and PyInstaller cannot
cross-compile, so a Windows `.exe` seemed to need a hand-compiled SQLCipher
(Wine + MSVC, or a vendored DLL) before anything else.

That premise was **stale**. It was true for the *exact package we pinned*, but a
sister distribution — **`sqlcipher3-wheels`** — publishes the **same project**
(a fork, `github.com/laggykiller/sqlcipher3`, of `coleifer/sqlcipher3`) with a CI
matrix that builds **Windows, macOS, and Linux** wheels, including
`cp312-cp312-win_amd64`. It exposes the **identical `sqlcipher3` import package**
(a drop-in — zero application-code change) and bundles the **identical SQLCipher
4.12.0 community** engine our current package uses. So the Windows build reduces
to a **dependency swap + a PyInstaller job on a Windows runner** — no crypto
compilation.

Note ADR-0007's exit criterion is "no Python *installed*, gates every *release*."
This slice ships a **CI artifact for testing, not a published release** (D6), and
its required gate is the weaker "no Python on `PATH`" runner check (D4); it does
**not** claim to meet ADR-0007's literal release bar, which the eventual signed
Windows release carries (the optional servercore container becomes required
then). So the ADR-0007 framing above motivates the work; it is not a promise this
slice keeps.

## Verified current state (2026-07-13)

Empirically checked on this machine (not recalled):

- **Import surface (drop-in).** Every DB-touching module imports through the
  **`sqlcipher3` package name**, which is identical for both packages, so the
  swap changes **no application code** and no mypy override
  (`[[tool.mypy.overrides]]` targets `sqlcipher3` / `sqlcipher3.*`). Concretely
  (grep-verified): **16** modules use `from sqlcipher3 import dbapi2`
  (`db.py`, `migrations.py`, `vault.py`, all **9** `repositories/*.py`, and
  `services/{categorization,reporting,transactions,transfer_detection}.py`);
  **3** use `from sqlcipher3.dbapi2 import DatabaseError`
  (`services/auth.py`, `services/backup.py`, `ui/main_window.py`); **1** uses
  `import sqlcipher3` (`_selftest.py`). All resolve to the same package.
- **Engine parity.** `sqlcipher3-binary==0.6.0` → `PRAGMA cipher_version` =
  **`4.12.0 community`**. `sqlcipher3-wheels==0.5.7` (installed into a scratch
  venv) → **`4.12.0 community`** — byte-identical. The two packages carry
  **independent version lineages** (0.6.0 for `-binary`, 0.5.7 for `-wheels`);
  the lower `-wheels` number is **not** a downgrade of a shared package, and
  **0.5.7 is `sqlcipher3-wheels`' latest stable release** (PyPI, verified
  2026-07-13 — satisfies `dependencies.md` §3's "pin must be latest stable";
  re-confirm on the `dependencies.md` §5 sweep — global rule §5c — at
  implementation). The security-relevant part —
  the SQLCipher *engine* — is the same build.
- **Binding-surface parity (the FIBR-0014 backup path).** Under
  `sqlcipher3-wheels==0.5.7`, both `PRAGMA rekey` (restore re-key) and
  `SELECT sqlcipher_export('bak')` into an `ATTACH`-ed, separately-keyed,
  HMAC-on target (export) round-trip correctly — proven in the scratch venv. So
  the older *binding* number does not drop any method the app calls; the app uses
  only standard DB-API plus SQL-level PRAGMAs/functions, which live in the
  4.12.0 engine, not the Python wrapper.
- **Cross-package vault round-trip (both directions), HMAC-on.** A vault created
  with `-binary` (raw-hex key, `cipher_compatibility=4`, `cipher_use_hmac=ON`,
  mirroring `vault.py`) **opens and reads correctly under `-wheels`**; a vault
  created with `-wheels` **opens under `-binary`**. HMAC-on means a wrong
  key/page/KDF/HMAC config fails at first read — both directions passing proves
  the key derivation, page format, and HMAC all match. This is the FIBR-0015
  analogue of FIBR-0014's D2 spike.
- **Windows wheel availability.** PyPI JSON for `sqlcipher3-wheels==0.5.7` lists
  `cp312-cp312-win_amd64` (plus `win32`, `win_arm64`) among 128 files spanning
  cp38–cp314 across Linux/macOS/Windows. `sqlcipher3-binary`'s latest (0.6.0) has
  **no** Windows wheel on any version (PyPI JSON confirmed). The macOS wheels the
  fork also ships mean the readiness-scan's "no Windows *or macOS* wheel" blocker
  is cleared for macOS too (its deferral below is packaging-only, Deliverable 7).
- **The `--self-test` smoke is a 9-stack check (FIBR-0003).**
  `_selftest.py:275-296` runs, in order, **nine** native-stack legs —
  `qt`, `qtcharts`, `icons`, `sqlcipher`, `pikepdf`, `pdf_encrypt`, `argon2`,
  `ofxparse`, `pdfplumber` — and prints the single sentinel line
  **`FINBREAK_SELFTEST_OK`** (or `FINBREAK_SELFTEST_FAIL: <name>`). So the smoke
  requires the Windows wheels for **Qt, QtCharts, SQLCipher, qpdf (incl. the
  `pdf_encrypt` `QPdfWriter`+pikepdf-AES path), Argon2, ofxparse(lxml),
  pdfplumber(Pillow/PDFium/cryptography)** to *all* travel, and the `icons` leg
  (`_check_icons`) exercises the packaged-icon path at runtime — it is not "Qt +
  SQLCipher + qpdf" alone.
- **Build machinery to reuse (FIBR-0003).** `scripts/_build-smoke-in-container.sh`
  installs `pyinstaller==6.21.0` then freezes `__main__.py` with `pyinstaller
  --onefile`, reading `[project].dependencies` from `pyproject.toml`
  (manifest-driven, not hand-listed — the fix for the FIBR-0004 argon2 /
  FIBR-0008 ofxparse bundle-drops), a single-Qt-binding guard, a directory-level
  `--add-data` for `ui/icons`, and a long `--hidden-import`/`--collect-*` flag
  set. **This one script is the *only* freeze**: `scripts/build-release-appimage.sh`
  is a thin wrapper that `exec`s `scripts/build-smoke.sh --release`, which freezes
  with the *same* `_build-smoke-in-container.sh` — so the **released** AppImage
  users run and the smoke share one flag set. `.github/workflows/build-smoke.yml`
  is the opt-in (`workflow_dispatch` + weekly) build workflow that keeps the fast
  gate fast.
- **Repo visibility.** PUBLIC → GitHub Actions Linux **and Windows** runner
  minutes are free; a `windows-latest` job costs nothing.

## At a glance — invariants

- **INV-1 (vault portability across the swap).** A vault and a `.fbk` backup
  created under `sqlcipher3-binary==0.6.0` open unchanged under
  `sqlcipher3-wheels==0.5.7`, and vice-versa — same SQLCipher 4.12.0,
  `cipher_compatibility=4`, HMAC on. Proven bidirectionally by the round-trip +
  binding-surface spikes. Two regression gates keep it true after the swap: (a)
  the **full test suite green under the new dep** — which includes the FIBR-0014
  backup suite exercising `sqlcipher_export` + `rekey` — proves the
  *same-package* (`-wheels`-creates → `-wheels`-reads) path; (b) a **committed
  `-binary`-0.6.0-created fixture vault + `.fbk`** (generated while `-binary` is
  still installed, before Deliverable 1 removes it) that a test opens under
  `-wheels` locks the real *upgrade* path an existing user hits — the direction
  CI can no longer reproduce once `-binary` is gone (Deliverable 4). The reverse
  (`-wheels`→`-binary`) direction is **spike-proven only**, not regression-gated
  — and needn't be: no user downgrades the shipped package.
- **INV-2 (single Qt binding at freeze).** Exactly one Qt binding is present
  before PyInstaller runs on Windows (carries the Linux
  `_build-smoke-in-container.sh` guard — PyInstaller 6.x refuses >1). *(Realised
  in the Deliverable-2 driver `build-windows-exe.py`, reusing the FIBR-0003
  guard.)*
- **INV-3 (no native-stack drift between OS builds).** The set of PyInstaller
  collection flags (`--hidden-import` / `--collect-binaries` / `--collect-all`)
  and the `--add-data` **target** (`finbreak/ui/icons`; the OS path *separator*
  differs per INV-5) used by the Windows freeze are **identical** to the Linux
  freeze's, enforced by an automated parity guard (D3) — a dependency collected
  for one OS cannot be silently dropped from the other's bundle (the FIBR-0004 /
  FIBR-0008 failure class). Because the Linux freeze is the single freeze behind
  both smoke and release, this also guards the released AppImage.
- **INV-4 (Python-independence of the artifact).** `finbreak.exe --self-test`
  runs the full 9-stack check (the `_selftest.py` leg identifiers, lowercase:
  `qt, qtcharts, icons, sqlcipher, pikepdf, pdf_encrypt, argon2, ofxparse,
  pdfplumber` — `pikepdf` wraps the qpdf engine) and prints
  **`FINBREAK_SELFTEST_OK`** when
  launched on the Windows runner with **no Python on `PATH`** (the D4 *required*
  gate). A PyInstaller onefile runs its own bundled CPython and never touches
  system Python — the independence is structural. The stronger ADR-0007 "no
  Python *installed*" bar is the *optional* servercore-container stretch (D4) and
  the gate a signed public Windows release would carry; this build-enablement
  slice ships a CI artifact, not a published release (D6), so the runner gate is
  the required bar and the residual is owned (D4).
- **INV-5 (`--add-data` separator correctness).** The `ui/icons` directory is
  placed at package-relative `finbreak/ui/icons` using the **OS path separator**
  (`;` on Windows, `:` on POSIX), so `ui/icons.py`'s
  `_ICON_DIR = Path(__file__).parent / "icons"` resolves inside the frozen
  bundle. Gated at runtime by the `--self-test` `icons` leg (`_check_icons`),
  which is part of the 9-stack sentinel (INV-4) — this leg, not the parity guard,
  owns `--add-data` *source*-path correctness.
- **INV-6 (manifest-driven deps).** The Windows freeze installs exactly
  `[project].dependencies` read from `pyproject.toml` (the same manifest read as
  Linux), so a future dependency bump reaches the Windows bundle without editing
  the build script. *(Realised in the Deliverable-2 driver `build-windows-exe.py`,
  reusing the FIBR-0003 manifest read.)*
- **INV-7 (the gate stays fast).** The Windows build is a **dedicated** workflow
  kept off the per-push `ci.yml` gate (like `build-smoke.yml`) but
  `workflow_dispatch`-only — no weekly schedule. Gated by Exit criterion 6.

## Design decisions

- **D1 — swap to `sqlcipher3-wheels==0.5.7` project-wide** (not a per-OS marker;
  user-approved "same fork everywhere"). One package on every OS ⇒ guaranteed
  identical crypto engine (proven, INV-1), one pin to bump, simplest to reason
  about. Rationale over the "official on Linux, fork on Windows" alternative: two
  packages providing the same `sqlcipher3` import add surface for no benefit once
  round-trip parity is proven.
  - **Fork-maintenance / advisory-watch (supersedes FIBR-0003's framing;
    recorded in ADR-0009, Deliverable 8).** FIBR-0003 recorded `sqlcipher3-binary`
    as *"the maintained package"* and a *"provisional"* choice whose formal ADR
    (anticipated via FIBR-0004) was never written; this item makes the choice
    final, moves to the **community fork** `laggykiller/sqlcipher3`, and writes
    that ADR now. Per `dependencies.md` §2 ("Vendored native libraries"),
    `pip-audit` **cannot** see a CVE inside a vendored native lib — the advisory
    only clears when the *wheel maintainer* re-releases. After the swap that
    maintainer is the fork author, so the vendored-native (SQLCipher + SQLite +
    **OpenSSL**) advisory watch is now on the fork's release cadence. D1/ADR-0009
    record this ownership and add `sqlcipher3-wheels` to the §2 vendored-native
    watch list; if the fork lags a SQLCipher/OpenSSL CVE, the fallback is to build
    the wheel from the fork's source (its README documents the conan-based OpenSSL
    build) or revert Windows to a compile path — logged in
    `docs/known-issues.md` per §5 until it ships.
  - **Touch:** `pyproject.toml` dependency + the in-file comment (`:16`) that
    names `sqlcipher3-binary`; the provenance comments in `vault.py:28`
    (names the package) and `migrations.py:8` (among the prose comments, the only
    one that restates the version "0.6.0"); **and the two live build-script
    comments that name it** — `scripts/ci-setup.sh:54` and
    `scripts/_build-smoke-in-container.sh:47` (global rule §5b: bump the dep *and*
    the calling context together). `docs/standards/dependencies.md` records the
    swap + rationale (Deliverable 5). No application code changes (import name
    unchanged). *(Frozen history — prior FIBR specs and their
    `tests/features/*/spec.md` test-contracts (e.g. `tests/features/backup/spec.md:11`'s
    past-tense spike note), journals, and CHANGELOG — is left as-is; those record
    what actually ran. Live forward-looking invariants in closed specs that name
    the old package — `FIBR-0014` INV-13's "a `sqlcipher3-binary` bump" wording —
    keep their package-agnostic intent (guard the SQLCipher cipher-profile across
    an engine bump); the swap preserves the 4.12.0 profile, so INV-13 still holds
    and is not rewritten.)*
- **D2 — build on GitHub Actions `windows-latest`** (not Wine, not
  cross-compile). PyInstaller cannot cross-compile; the `.exe` must be produced on
  Windows. `windows-latest` is free on this public repo, native, and reproducible.
  A new `.github/workflows/windows-build.yml`, `workflow_dispatch`-triggered
  (INV-7), pins `setup-python` to **3.12** against the verified `cp312`
  `sqlcipher3-wheels` wheel, reuses the FIBR-0003 single-Qt-binding guard (INV-2)
  and manifest read (INV-6) in the Deliverable-2 driver, and uploads
  `finbreak.exe` as a build artifact. The
  user-noted local Wine+MSVC path is dropped as unnecessary now a wheel exists
  (Out of scope). *(A future Python-runtime bump past the fork's wheel matrix
  (currently cp38–cp314) would need a wheel refresh first — a non-issue for this
  cp312-pinned slice, noted so a later bump doesn't trip it.)*
- **D3 — single canonical flag list, guarded across OS freezes.** The
  drop-prevention requirement (INV-3) is met by making the Windows freeze derive
  its PyInstaller collection flags from **one canonical list**
  (`scripts/windows_freeze_flags.py`, a plain Python list of `--hidden-import` /
  `--collect-binaries` / `--collect-all` names + the `--add-data` source→target
  pair), and a **parity guard test** that scrapes the existing Linux freeze and
  asserts equality. The extraction contract is explicit and writable: the test
  reads `_build-smoke-in-container.sh`, **scopes to the `pyinstaller … \`
  invocation block only** — from the (unindented) line beginning with
  `pyinstaller` through the first line that is **not** `\`-continued (its
  terminating entry-point argument, `__main__.py`) — and **strips `#`-comment
  lines** (belt-and-suspenders: the block scope is the load-bearing defense — a
  bash comment can't sit inside a `\`-continuation, so the strip only matters if
  the block boundary is ever loosened; the script's *prose before the block*
  mentions `--collect-all`, which a whole-file scrape would wrongly capture).
  Within that block it matches every
  `--(hidden-import|collect-binaries|collect-all)\s+(\S+)` token **anywhere on a
  line** (not only line-leading — so a future reformat that moves a flag onto the
  `pyinstaller` line, or packs two flags on one line, is still caught; a
  line-leading-only filter would leave that blind spot) plus the `--add-data`
  target — the token **after the final `:`, with surrounding quotes stripped**
  (the bash value is `"…/ui/icons:finbreak/ui/icons"`, so a naive split leaves a
  trailing `"` that must be removed before comparing to the canonical
  `finbreak/ui/icons`) — and asserts that set equals the canonical list's. For the guard to stay sound the freeze must keep every collection flag
  inside this **one** `\`-continued invocation — a second `pyinstaller`
  invocation or a `$VAR`-indirected flag would evade the scrape and is disallowed
  (stated here so soundness rests on a written rule, not an unstated formatting
  convention). The canonical `--add-data` **source** path is *not*
  parity-checked (it is OS-specific); its correctness is owned by the EC3 `icons`
  self-test leg, which fails if icons don't travel. This deliberately does **not**
  edit the released bash freeze (it stays the source the guard reads); rewiring
  that bash script to *import* the canonical list is a noted follow-up (Out of
  scope). Because `_build-smoke-in-container.sh` is the single freeze behind both
  smoke and release, guarding against it covers the AppImage users run.
- **D4 — clean-room the `.exe` with a required runner gate and an optional
  container stretch.** ADR-0007's exit criterion is "no Python installed." The
  **required** gate (Exit criterion 3): the workflow launches
  `finbreak.exe --self-test` on the `windows-latest` runner **with `python` and
  `py` removed from `PATH`** and asserts the `FINBREAK_SELFTEST_OK` sentinel —
  proving the `.exe` launches and loads all 9 native stacks without invoking any
  interpreter. A PyInstaller onefile bundles its own CPython **and** the MSVC C
  runtime, so system-Python-independence holds by construction and the bundle
  carries its own runtime. **Optional stretch:** additionally launch the same
  `.exe` inside a Python-free Windows container
  (`mcr.microsoft.com/windows/servercore:ltsc2022`) for parity with the Linux
  `debian:13-slim` clean-room; this is not required to pass, because Windows
  containers on hosted runners are heavy and the servercore/nanoserver base-DLL
  surface is unproven for Qt. **Intentional asymmetry, owned residual:** this
  required Windows bar (Python off `PATH`) is deliberately **weaker** than the
  Linux precedent — FIBR-0003 INV-3 runs the Linux bundle in a truly Python-free
  `debian:13-slim` container — and the runner-only gate does not prove the `.exe`
  is free of *every* borrowed stock-Windows DLL beyond what `windows-latest`
  carries. Both the asymmetry and the residual are **accepted by the project owner
  for this testing-milestone CI artifact** (friends run normal Windows boxes with
  the MSVC redistributable present) and are revisited when a signed public Windows
  release is cut (then the servercore container becomes the required gate).
- **D5 — unsigned single `finbreak.exe`, manual update, no installer.** Deliver
  one onefile `.exe` (the "download one file, run it" ADR-0007 UX). **No Windows
  code-signing** this slice (a paid cert; unsigned means testers see a SmartScreen
  "unknown publisher" prompt — documented in the deliverable note). **No
  auto-update on Windows** (roadmap-confirmed; the FIBR-0054 self-update is
  AppImage/Linux-only and stays so — a Windows tester replaces the `.exe`
  manually). **No MSI/installer.**
- **D6 — distribution for testing is the CI artifact.** The `windows-build.yml`
  run uploads `finbreak.exe` as a downloadable artifact; getting it to friends
  today = download the artifact and copy to the user's Windows test machine / the
  `/mnt/Games/Scripts/Apps/finbreak/` drop share. Attaching the `.exe` to a
  *published* GitHub release follows the FIBR-0054 AppImage release flow and is
  deferred to a real Windows release (Out of scope).

## Deliverables

1. **`pyproject.toml`** — dependency `sqlcipher3-binary==0.6.0` →
   `sqlcipher3-wheels==0.5.7`; refresh the in-file comment (`:16`) that names the
   old package. **Comment refresh across every live site** naming the old package
   (D1): `vault.py:28`, `migrations.py:8` (names the version),
   **`scripts/ci-setup.sh:54`**, **`scripts/_build-smoke-in-container.sh:47`**. No
   mypy-override change. **No glibc-floor comment change** (`pyproject.toml:18-23`):
   `sqlcipher3-wheels`' Linux wheel is `manylinux_2_28`, *below* PySide6's
   governing `manylinux_2_34`, which is itself below the 2.36 bookworm build image
   — so the effective-floor note still holds (verified 2026-07-13).
2. **`scripts/windows_freeze_flags.py`** — the single canonical PyInstaller
   collection-flag list (D3), plus **`scripts/build-windows-exe.py`** (the Windows
   freeze driver; Python over PowerShell so the manifest read + `os.pathsep` logic
   is shared-language with the codebase and unit-testable). The driver reads
   `[project].dependencies` (INV-6), asserts exactly one Qt binding (INV-2), and
   invokes `pyinstaller==6.21.0 --onefile --name finbreak` with the canonical
   flags and an `os.pathsep`-joined `--add-data` for `ui/icons` (INV-5). Produces
   `dist/finbreak.exe`.
3. **`.github/workflows/windows-build.yml`** — `workflow_dispatch` (INV-7);
   `runs-on: windows-latest`; `actions/setup-python@v6` pinned `3.12`;
   `pip install .` + `pyinstaller==6.21.0`; run `build-windows-exe.py`; the D4
   **required** clean-room step (`--self-test` with `python`/`py` off `PATH`,
   assert `FINBREAK_SELFTEST_OK`); the optional servercore container stretch;
   `actions/upload-artifact@v4` of `finbreak.exe`. (Pin `upload-artifact` /
   `setup-python` / `checkout` to the versions current at implementation, §5a.)
4. **`tests/features/windows_build/`** (`spec.md` + tests) —
   - **INV-1 regression (same-package):** `sqlcipher3` imports; `cipher_version`
     is `4.12.0 community`; a raw-hex-keyed, `cipher_compatibility=4`, HMAC-on
     vault round-trips a row; a `sqlcipher_export` + `rekey` round-trip (the
     FIBR-0014 backup binding surface). *The whole existing suite passing under
     the new dep — including the FIBR-0014 backup suite — is the broader gate;
     these are the focused, always-run assertions.*
   - **INV-1 regression (cross-package upgrade path):** commit a small fixture —
     a vault `.db` **and** a `.fbk` backup **created with `sqlcipher3-binary`
     0.6.0** (generated once, before Deliverable 1 removes `-binary`; synthetic
     data + a documented test key, no real financial data) under
     `tests/fixtures/windows_build/`. **Commit the generation script alongside
     the fixture** (a skipped-by-default helper that recreates the `.db`/`.fbk`
     from `-binary`), so the opaque encrypted bytes have provenance and can be
     regenerated (by temporarily reinstalling `-binary`) rather than being an
     unauditable blob. A test opens both under the
     installed `-wheels` and asserts the row/backup read — regression-locking the
     direction CI can no longer reproduce once `-binary` is gone (INV-1).
   - **INV-3 parity guard:** import the canonical flag list from
     `windows_freeze_flags.py`; scrape `_build-smoke-in-container.sh` per the D3
     invocation-scoped, comment-ignoring regex contract; assert the
     `--hidden-import` / `--collect-binaries` / `--collect-all` sets and the
     `--add-data` package-relative **target** are equal.
   - **Windows-freeze shape (INV-2, INV-6):** `build-windows-exe.py` uses
     `os.pathsep` for `--add-data` (not a hard-coded `:`), reads deps from the
     manifest (INV-6), and carries the single-Qt guard (INV-2). *(These are static
     presence assertions; INV-2's behavioural proof is EC2 — a green freeze can't
