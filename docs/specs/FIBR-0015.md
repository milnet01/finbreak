# FIBR-0015 — Windows build (self-contained `.exe`)

> **Status:** ✍️ DRAFT — pending `/cold-eyes` (project cap 7). Not yet cleared for
> code.
> **Phase:** P13, narrowed to the **Windows slice only** (user-approved
> 2026-07-13). P13's original scope — Windows `.exe`, macOS `.app`/`.dmg`,
> AppImage, Flatpak/Flathub — is split: the **Linux AppImage** already shipped
> under FIBR-0054 (`scripts/build-release-appimage.sh`, released `v0.1.0`+); this
> item delivers the **Windows `.exe`**; **macOS** and **Flatpak/Flathub** stay in
> P13 for a later slice (Out of scope, below).
> **Kind:** chore (build / ci / packaging).
> **Security:** MEDIUM — no new app logic, but it swaps the **package that
> supplies the SQLCipher encryption engine** for the money vault, so vault
> portability across the swap is an invariant proven empirically before the swap
> lands (INV-1).

## The problem (ADR-0007 + the stale-pin trap)

ADR-0007 requires every release to be **fully self-contained**: download one
file, run it, it works, on a clean machine with **no Python installed**. The
Linux AppImage delivers this today. Windows does not — there is no `.exe`.

The 2026-07-13 readiness scan recorded Windows as blocked because
`sqlcipher3-binary==0.6.0` (the pinned SQLCipher engine, ADR-0003) ships
**Linux/macOS wheels only** — no Windows wheel — and PyInstaller cannot
cross-compile, so a Windows `.exe` seemed to need a hand-compiled SQLCipher
(Wine + MSVC, or a vendored DLL) before anything else.

That premise was **stale**. It was true for the *exact package we pinned*, but a
sister distribution — **`sqlcipher3-wheels`** — publishes the **same project**
(a fork, `github.com/laggykiller/sqlcipher3`, of `coleifer/sqlcipher3`) with a CI
matrix that builds **Windows, macOS, and Linux** wheels, including
`cp312-cp312-win_amd64`. It exposes the **identical `from sqlcipher3 import
dbapi2`** import (a drop-in — zero application-code change) and bundles the
**identical SQLCipher 4.12.0 community** engine our current package uses. So the
Windows build reduces to a **dependency swap + a PyInstaller job on a Windows
runner** — no crypto compilation.

## Verified current state (2026-07-13)

Empirically checked on this machine (not recalled):

- **Import surface.** `from sqlcipher3 import dbapi2` is used in 10 modules
  (`vault.py`, `db.py`, `migrations.py`, `services/{transactions,reporting,
  categorization}.py`, `repositories/{settings,transfers}.py`,
  `services/backup.py`, `_selftest.py`). The import name is `sqlcipher3` for both
  packages, so the swap changes **no application code** and no mypy override
  (`[[tool.mypy.overrides]]` targets `sqlcipher3` / `sqlcipher3.*`).
- **Engine parity.** `sqlcipher3-binary==0.6.0` → `PRAGMA cipher_version` =
  **`4.12.0 community`**. `sqlcipher3-wheels==0.5.7` (installed into a scratch
  venv) → **`4.12.0 community`** — byte-identical. (The *binding* version differs,
  0.6.0 vs 0.5.7; the *SQLCipher engine* — the security-relevant part — is the
  same build.)
- **Cross-package vault round-trip (both directions), HMAC-on.** A vault created
  with `-binary` (raw-hex key, `cipher_compatibility=4`, `cipher_use_hmac=ON`,
  mirroring `vault.py`) **opens and reads correctly under `-wheels`**; a vault
  created with `-wheels` **opens under `-binary`**. HMAC-on means a wrong
  key/page/KDF/HMAC config fails at first read — both directions passing proves
  the key derivation, page format, and HMAC all match. This is the FIBR-0015
  analogue of FIBR-0014's D2 spike.
- **Windows wheel availability.** PyPI JSON for `sqlcipher3-wheels==0.5.7` lists
  `cp312-cp312-win_amd64` (plus `win32`, `win_arm64`) among 128 files spanning
  cp38–cp314 across Linux/macOS/Windows. `sqlcipher3-binary`'s latest (0.6.0) has
  **no** Windows wheel on any version (PyPI JSON confirmed) — the web-summary
  claim of a "0.6.2 with Windows wheels" was false.
- **Build machinery to reuse (FIBR-0003).** `scripts/_build-smoke-in-container.sh`
  freezes `python -m finbreak` with `pyinstaller==6.21.0 --onefile`, reading
  `[project].dependencies` from `pyproject.toml` (manifest-driven, not
  hand-listed — the fix for the FIBR-0004 argon2 / FIBR-0008 ofxparse
  bundle-drops), a single-Qt-binding guard, a directory-level `--add-data` for
  `ui/icons`, and a long `--hidden-import`/`--collect-*` flag set. `--self-test`
  (FIBR-0003) loads Qt + SQLCipher + qpdf and prints a sentinel — the ready-made
  smoke. `.github/workflows/build-smoke.yml` is the opt-in (`workflow_dispatch` +
  weekly) build workflow that keeps the fast gate fast.
- **Repo visibility.** PUBLIC → GitHub Actions Linux **and Windows** runner
  minutes are free; a `windows-latest` job costs nothing.

## At a glance — invariants

- **INV-1 (vault portability across the swap).** A vault and a `.fbk` backup
  created under `sqlcipher3-binary==0.6.0` open unchanged under
  `sqlcipher3-wheels==0.5.7`, and vice-versa — same SQLCipher 4.12.0,
  `cipher_compatibility=4`, HMAC on. Proven by the round-trip spike; the **full
  test suite green under the new dep** is the regression gate that keeps it true.
- **INV-2 (single Qt binding at freeze).** Exactly one Qt binding is present
  before PyInstaller runs on Windows (carries the Linux
  `_build-smoke-in-container.sh` guard — PyInstaller 6.x refuses >1).
- **INV-3 (no native-stack drift between OS builds).** The set of PyInstaller
  collection flags (`--hidden-import` / `--collect-binaries` / `--collect-all` /
  the `--add-data` mapping) used by the Windows freeze is **identical** to the
  Linux freeze's, enforced by an automated parity check — a dependency collected
  for one OS cannot be silently dropped from the other's bundle (the FIBR-0004 /
  FIBR-0008 failure class).
- **INV-4 (Python-independence of the artifact).** `finbreak.exe --self-test`
  loads Qt + SQLCipher + qpdf and prints the FIBR-0003 sentinel when launched in a
  Windows environment with **no Python installed** (ADR-0007 exit criterion). A
  PyInstaller onefile runs its own bundled CPython and never touches system
  Python — the independence is structural; the check proves no *other* system
  component is silently borrowed.
- **INV-5 (`--add-data` separator correctness).** The `ui/icons` directory is
  placed at package-relative `finbreak/ui/icons` using the **OS path separator**
  (`;` on Windows, `:` on POSIX), so `ui/icons.py`'s
  `Path(__file__).parent / "icons"` resolves inside the frozen bundle (the DoD #2
  non-null-pixmap self-test legs).
- **INV-6 (manifest-driven deps).** The Windows freeze installs exactly
  `[project].dependencies` read from `pyproject.toml` (the same manifest read as
  Linux), so a future dependency bump reaches the Windows bundle without editing
  the build script.
- **INV-7 (the gate stays fast).** The Windows build is a **dedicated, opt-in**
  workflow (`workflow_dispatch`), not a step on the per-push `ci.yml` gate — the
  same posture as `build-smoke.yml`.

## Design decisions

- **D1 — swap to `sqlcipher3-wheels==0.5.7` project-wide** (not a per-OS marker;
  user-approved "same fork everywhere"). One package on every OS ⇒ guaranteed
  identical crypto engine (proven, INV-1), one pin to bump, simplest to reason
  about. The `-binary` package stays *available* upstream, but the project no
  longer depends on it. Rationale over the "official on Linux, fork on Windows"
  alternative: two packages providing the same `sqlcipher3` import add surface for
  no benefit once round-trip parity is proven. Touch: `pyproject.toml`
  dependency + the provenance comments that name "sqlcipher3-binary 0.6.0"
  (`pyproject.toml` dep block, `vault.py`, `migrations.py`) — refreshed to name
  the new package and the retained 4.12.0 engine (global rule §5b: bump the dep
  *and* the calling context together). `docs/standards/dependencies.md` records
  the swap + rationale. No application code changes (import name unchanged).
- **D2 — build on GitHub Actions `windows-latest`** (not Wine, not
  cross-compile). PyInstaller cannot cross-compile; the `.exe` must be produced on
  Windows. `windows-latest` is free on this public repo, native, and reproducible.
  A new `.github/workflows/windows-build.yml`, `workflow_dispatch`-triggered
  (INV-7), uploads `finbreak.exe` as a build artifact. The user-noted local
  Wine+MSVC path is dropped as unnecessary now a wheel exists (recorded in Out of
  scope).
- **D3 — enforce flag parity with an automated guard, do not refactor the
  released Linux build.** The anti-drift requirement (INV-3) is met by a **parity
  test** (`tests/features/windows_build/`) that extracts the PyInstaller
  collection flags from *both* the Linux freeze (`_build-smoke-in-container.sh`)
  and the Windows freeze script and asserts set-equality — red the moment they
  diverge. This deliberately does **not** rewire the proven, *released* Linux /
  AppImage freeze path (which builds the signed AppImages users run) to source
  flags from a new shared module: destabilising a load-bearing release path for
  DRY is the wrong trade on a correctness-critical app under a "ship today"
  constraint (global rules §2, §11). Full single-sourcing of the flag list is a
  noted follow-up (Out of scope), not this slice.
- **D4 — clean-room the `.exe` in a Python-free Windows container, with a
  documented runner fallback.** To satisfy ADR-0007 ("no Python installed"), the
  workflow launches `finbreak.exe --self-test` inside a Python-free Windows
  container (`mcr.microsoft.com/windows/servercore:ltsc2022` — the parity of the
  Linux `debian:13-slim` clean-room). **To verify empirically:** Windows
  containers on hosted runners can be heavy and Qt may need specific system
  runtime DLLs; PyInstaller bundles the VC++ runtime, so the `.exe` is expected to
  be self-sufficient, but this is unproven on servercore/nanoserver. **Fallback if
  the container proves impractical:** run the `.exe` on the runner with `python` /
  `py` removed from `PATH` (proving it launches without invoking any interpreter),
  and document the residual (onefile structurally bundles its own CPython, so
  system-Python-independence holds by construction; only the "no other borrowed
  system DLL" claim is weaker under the fallback). The exit criterion commits to
  the *fallback* at minimum, the container ideally.
- **D5 — unsigned single `finbreak.exe`, manual update, no installer.** Deliver
  one onefile `.exe` (the "download one file, run it" ADR-0007 UX). **No Windows
  code-signing** this slice (a paid cert; unsigned means testers see a SmartScreen
  "unknown publisher" prompt — documented in the deliverable note). **No
  auto-update on Windows** (roadmap-confirmed; the FIBR-0054 self-update is
  AppImage/Linux-only and stays so — a Windows tester replaces the `.exe`
  manually). **No MSI/installer.**
- **D6 — distribution for testing is the CI artifact.** The `windows-build.yml`
  run uploads `finbreak.exe` as a downloadable artifact; getting it to friends
  today = download the artifact and copy to the user's Windows test machine / the
  `/mnt/Games/Scripts/Apps/finbreak/` drop share. Attaching the `.exe` to a
  *published* GitHub release follows the FIBR-0054 AppImage release flow and is
  deferred to a real Windows release (Out of scope).

## Deliverables

1. **`pyproject.toml`** — dependency `sqlcipher3-binary==0.6.0` →
   `sqlcipher3-wheels==0.5.7`; refresh the in-file comments that name the old
   package; provenance-comment refresh in `vault.py` and `migrations.py` (name the
   new package; note the retained SQLCipher 4.12.0). No mypy-override change.
2. **`scripts/build-windows-exe.py`** (Python, runs on the Windows runner; Python
   over PowerShell so the manifest read + `os.pathsep` logic is shared-language
   with the codebase and unit-testable). Reads `[project].dependencies` (INV-6),
   asserts exactly one Qt binding (INV-2), and invokes `pyinstaller==6.21.0
   --onefile --name finbreak` with the **same** collection flag set as the Linux
   freeze and an `os.pathsep`-joined `--add-data` for `ui/icons` (INV-5). Produces
   `dist/finbreak.exe`.
3. **`.github/workflows/windows-build.yml`** — `workflow_dispatch` (INV-7);
   `runs-on: windows-latest`; `actions/setup-python@v6` pinned `3.12`;
   `pip install .` + `pyinstaller==6.21.0`; run `build-windows-exe.py`; launch
   `finbreak.exe --self-test` on the runner (fast fail); the D4 clean-room
   container launch; `actions/upload-artifact@v4` of `finbreak.exe`.
4. **`tests/features/windows_build/`** (`spec.md` + tests) —
   - **INV-1 regression:** `sqlcipher3` imports; `cipher_version` is `4.12.0
     community`; a raw-hex-keyed, `cipher_compatibility=4`, HMAC-on vault
     round-trips a row (the swap is safe on the running platform). *The whole
     existing suite passing under the new dep is the broader gate; this test is
     the focused, always-run assertion.*
   - **INV-3 parity guard:** parse the collection flags from
     `_build-smoke-in-container.sh` and from `build-windows-exe.py`; assert the
     `--hidden-import` / `--collect-binaries` / `--collect-all` sets and the
     `--add-data` package-relative target are equal.
   - **Windows-freeze shape:** `build-windows-exe.py` uses `os.pathsep` for
     `--add-data` (not a hard-coded `:`), reads deps from the manifest (INV-6),
     and carries the single-Qt guard (INV-2).
5. **`docs/standards/dependencies.md`** — record the `sqlcipher3-binary` →
   `sqlcipher3-wheels` swap, the parity rationale, and the retained 4.12.0 engine.
6. **A short note** (README "Windows (testing)" subsection or the journal) —
   unsigned `.exe` + SmartScreen expectation + manual-update, so a tester knows
   what to expect. `--self-test` (FIBR-0003) is reused unchanged as the smoke.

## Out of scope

- **macOS `.app`/`.dmg`** and **Flatpak/Flathub** — remain in P13 for a later
  slice.
- **Windows code-signing** (paid cert), **MSI/installer**, **Windows
  auto-update** — deferred; testers use the unsigned onefile and update manually.
- **The Wine + MSVC local build path** — dropped; the `sqlcipher3-wheels`
  Windows wheel removes the need to compile SQLCipher.
- **Single-sourcing the PyInstaller flag list** across the Linux and Windows
  freezes — a follow-up refactor; this slice guards drift with a parity test
  (D3) rather than rewiring the released Linux path.
- **Attaching `finbreak.exe` to a published GitHub release** — follows a real
  Windows release, not this build-enablement slice.

## Exit criteria

1. **Dep swap safe:** the full `pytest` suite is green with `sqlcipher3-wheels`
   installed (INV-1) on the Linux gate; `ci-local.sh` (ruff / format / mypy /
   bandit / pip-audit / gitleaks / tests) stays green — **including pip-audit
   raising no new advisory** on `sqlcipher3-wheels` or its bundled OpenSSL (if it
   does, the finding is triaged, not suppressed).
2. **`windows-build.yml` runs green** on `windows-latest`, producing
   `finbreak.exe`.
3. **`finbreak.exe --self-test` prints the FIBR-0003 sentinel** in a Windows
   environment with no Python installed (INV-4 — the D4 container ideally, the
   scrubbed-runner fallback at minimum).
4. **Parity guard green** (INV-3) and the new focused tests pass.
5. No regression to the Linux AppImage build (`build-smoke.yml` still green).

## Dependencies

- **FIBR-0037** (app icon travels as `ui/icons` package data) — the `--add-data`
  mapping carries it (INV-5).

- **ADR-0007** (self-contained bundled releases; the exit criterion) and
  **ADR-0003** (SQLCipher storage; the engine the swap preserves).
