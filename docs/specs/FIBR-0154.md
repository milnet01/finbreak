# FIBR-0154 — Category UI: a 3rd tier (Type → Category → Sub-category)

> **Status:** Draft — pending `/cold-eyes` (§ 8, project cadence
> `--max-loops 7`). Contract for FIBR-0154 (Kind: feature; lanes: ui,
> services). Source: user-request-2026-07-21. **User decision (2026-07-21):**
> the UI exposes **exactly three levels** — a hard UI cap (the data model &
> service already allow arbitrary depth; only the UI is bounded).

## 1. Problem & intent

A user asked to categorise a transaction three levels deep — e.g.
**Expenditure › Groceries › Spar**. Today the app UI stops at **two** levels
(Type → Category): the `CategoriesWidget` tree renders only root → one hop of
children (`ui/categories.py:212-229`), and its Add form always parents a new
category onto a **root Type** (`ui/categories.py:116`). This was a deliberate
deferral — the module docstring names it "D9" (`ui/categories.py:9-10`).

The **data model and service already support arbitrary depth** — `parent_id`
is a self-referencing FK with no depth column or cap
(`migrations.py:117-120`, `models.py:230-242`), and `add_category` /
`_require_parent` only require the parent to **exist**, not to be a root
(`services/categories.py:106-116`), so a grandchild is already creatable
through the service (proven today by `tests/features/categories/
test_categories.py:287-304`, which builds a 4-deep chain via the service). So
FIBR-0154 is **UI work**, plus one **label-disambiguation** change to the flat
category pickers that a 3rd tier newly requires.

## 2. Level terminology (fixed for this spec)

- **Level 1 — Type:** a root (`parent_id IS NULL`), `kind` ∈ {Income,
  Expenditure} (`migrations.py:122-133`). Not user-renamable/re-parentable
  (`services/categories.py:57-60`).
- **Level 2 — Category:** a direct child of a Type (e.g. *Groceries*).
- **Level 3 — Sub-category:** a child of a Level-2 Category (e.g. *Spar*).
- **Depth of a node** = its ancestor count + 1. **Max UI depth = 3.** "A node
  is at Level 3" ⇔ it has a grandparent that is a root.

## 3. Scope

**In scope**
- `CategoriesWidget`: render a **3-deep** tree and let a new category be
  parented onto a **Type (→ Level 2)** *or* an existing **Level-2 Category
  (→ Level 3)** — never onto a Level-3 node (the cap, § 4.2).
- A **UI-only depth cap of 3**: the Add/Update parent choices exclude Level-3
  nodes; the service and schema are **unchanged** (§ 4.4).
- The three flat category pickers (Set-category dialog, Rules editor,
  Transactions category filter) must **render and resolve** Level-3
  sub-categories, disambiguated by a **parent breadcrumb** where the
  Type-tag alone (FIBR-0123) is insufficient (§ 4.3).
- Rename/delete continue to work at all three levels (delete still blocked
  when the node has children — `services/categories.py:81-84`, already
  depth-agnostic).
- Spec → `/cold-eyes` → reproduce-first TDD (§ 5).

**Out of scope**
- Any schema migration or `parent_id`/model change — none is needed
  (`LATEST_SCHEMA_VERSION = 10`, `migrations.py:24`; a 3rd tier adds no
  column).
- Relaxing the UI cap to arbitrary depth (the model already allows it; a
  future item can lift the UI cap without data work).
- Re-solving the two-**root** same-name ambiguity — **FIBR-0123** shipped
  that (the `Name (Type)` tag, `docs/specs/FIBR-0123.md`); this item
  **extends** that baseline for the deeper level.
- The reporting/drill-down math — **FIBR-0138** already climbs `parent_id`
  chains and sums descendants (depth-safe); no change here.

## 4. Design

### 4.1 Where the code stands (recon-verified)

- **Model/service already allow depth 3.** `add_category → _require_parent`
  checks only that the parent exists and is non-None
  (`services/categories.py:106-116`); `_reject_cycle` guards acyclicity
  (`:118-139`); `update_category` blocks editing a root and re-uses the cycle
  guard (`:50-65`); `delete_category` blocks deleting a node **with children**
  (`:81-84`). Sibling-name uniqueness is **per-parent**
  (`_validate` → `repo.children_of(parent_id)`, `:141-159`) — so the same
  sub-category name can legitimately exist under two different parents. **No
  depth check exists anywhere.**
- **`CategoriesWidget` is hard-wired to two levels.** It uses a `QTreeWidget`
  (`ui/categories.py:61`). `_refresh` loops roots (`children_of(None)`,
  `:215`) then one hop of children (`children_of(root.id)`, `:223`) — **no
  recursion**, so a grandchild never renders. The Add parent is
  `self._type.currentData()` (`:116`), and `self._type` is populated with
  **roots only** (`:222`). Selection-load maps a node's parent back into
  `self._type` via `select_combo_data` (`:195`) — a no-op for a non-root
  parent.
- **All three flat pickers funnel through one helper.**
  `add_grouped_categories` (`ui/_widgets.py:48-77`) renders a flat combo: a
  disabled bold **Type** header per section, then one row per category tagged
  `"{name} ({type})"` (`:73-75`), carrying `category.id` as userData. It is
  fed by `CategorizationService.leaf_categories_grouped()`
  (`services/categorization.py:157-195`), which groups each non-root category
  under its **kind-bearing root** (depth-safe already) and emits **only the
  Type token + the leaf's own name — no parent-category path**. Consumers:
  Set-category `ui/category_picker.py:37-39,56-58`; Rules
  `ui/rules.py:64-67,101-105`; Transactions filter
  `ui/transactions.py:219-256`.

### 4.2 CategoriesWidget — pick a parent, render 3 deep

**Render (`_refresh`).** Replace the two-loop render with a **recursive**
build so root → Level-2 → Level-3 all appear as nested `QTreeWidgetItem`s.
Recursion is **unbounded** — it renders whatever depth exists, so a
pre-existing deeper node (§ 4.4) shows in full; the cap governs **authoring**,
not rendering.

**Add — a child of the currently *selected* node (no shared parent-combo).**
Today Add reads a roots-only combo (`categories.py:116`), which cannot express
"a child of *Groceries*". Re-anchor Add to the **tree selection**: Add creates
`<name>` as a child of the selected node — select *Groceries* (Level-2) then
Add "Spar" → *Groceries › Spar*; select a **Type** then Add → a Level-2
Category. This is unambiguous (the parent is the visible, selected row) and
avoids the **wrong-parent trap** a shared parent-combo creates — a combo
preselected to the selected node's *parent* would silently put the new child
one level too high. `add_category(selected.id, name)` — the service call is
unchanged (it accepts any existing parent). **The cap is enforced at the
anchor:** Add is **disabled** when the selection is a **Level-3** (or deeper)
node — its child would be Level 4 — and when **nothing** is selected (no parent
to add under). A short inline hint states why when disabled.

**Rename — the Update button, name field only.** `update_category(selected.id,
new_name, selected.parent_id)` keeps the node under its existing parent
(`services/categories.py:50-65`) — rename touches no parent control, so it
carries no wrong-parent or depth risk. (Renaming a Type stays blocked by the
service, `:57-60`.)

**Re-parent — a dedicated, subject-aware "Move under…" selector, used by
Update only (never by Add).** Moving a category to a different parent stays
available (today's Update does it via the roots combo). Because Add no longer
reads any combo, this **subject-aware** selector is unambiguous. It offers the
Types + Level-2 Categories (breadcrumbed, translatable per § 4.3), **minus**:
- the **subject itself** and its **descendants** (a cycle — `_reject_cycle`
  rejects them anyway, `services/categories.py:118-139`, but they must not even
  be offered);
- any parent that would push the subject's **deepest descendant** past Level 3.
  `update_category` has **no** depth check (`:50-65`), so this is the only place
  the re-parent cap lives: a **childed** Level-2 (e.g. *Groceries* with child
  *Spar*) offers **Types only** (under a Type its children sit at Level 3; under
  another Level-2 they would be Level 4); a **childless** node offers {Types,
  Level-2 Categories}.

(For a UI-authored tree the subtree height is ≤ 1, so childed-vs-childless
matches the "deepest descendant ≤ 3" principle exactly. For a **pre-existing**
subtree already deeper than the cap — § 4.4 — "Types only" still never
**deepens** it; the guard's job is to not *author* new depth, not to retro-fix
tolerated deep data.)

The "Move under…" list is rebuilt in `_on_selection_changed`
(`categories.py:186-195`) from the newly-selected subject (its subtree height +
descendant set) before any preselect; when nothing is selected it is
empty/disabled (re-parent needs a subject). INV-3 tests the Add-anchor cap
(disabled on a Level-3 selection) **and** this re-parent cap (a childed Level-2
offers Types only).

### 4.3 Flat pickers — disambiguate the deeper level with a breadcrumb

FIBR-0123's `Name (Type)` tag disambiguates categories across the two
**roots**. It is **not** enough at depth 3: because sibling uniqueness is
per-parent, *Groceries › Spar* and *Fuel › Spar* both render today as
`"Spar (Expenditure)"` — indistinguishable. So the pickers gain a **parent
breadcrumb for Level-3 entries only**:

- **Level-2 Category** row: unchanged — `"Groceries (Expenditure)"`.
- **Level-3 Sub-category** row: `"Groceries › Spar (Expenditure)"` — the
  immediate parent's name prepended (one hop; not the full chain, since the
  Type tag already carries the root).

Mechanically — **without changing `leaf_categories_grouped()`'s element
shape** (its entries stay `Category` objects, because `ui/transactions.py:251-254`
destructures the grouped list and indexes `c.id`, and existing tests unpack
`c.name`/`c.id` — `test_categorisation.py:494,506`; a tuple element would crash
both):
- Add a service accessor `CategorizationService.sub_category_parent_names() →
  dict[int, str]` mapping any category whose parent is **non-root** (a
  grandchild-or-deeper node) to its immediate parent's name; a Level-2 id is
  absent. (It therefore also breadcrumbs any pre-existing Level-4+ node —
  harmless, § 4.4. Computable from one `list_all()` + a `by_id` dict, the same
  pattern `leaf_categories_grouped` already uses.)
- Give `add_grouped_categories` (`_widgets.py:48-77`) an optional
  `parent_names: dict[int, str] | None`; when a row's `category.id` is in it,
  render a **translatable** composite via the same
  `QCoreApplication.translate(_LABEL_CONTEXT, …)` path the `"{name} ({type})"`
  tag already uses (`_widgets.py:62`) — e.g.
  `translate(_LABEL_CONTEXT, "{parent} › {row}")`, **not** a hardcoded
  separator. Default `None` → today's behaviour verbatim.
- **Thread the map at every site — this is NOT one line.** The two picker
  *dialogs* take the already-grouped list, not a service
  (`category_picker.py:6-8`), so each gains a new `parent_names` constructor
  arg it forwards to its internal `add_grouped_categories`:
  `CategoryPickerDialog.__init__` (`category_picker.py:28-39`) and
  `RuleEditDialog.__init__` (`rules.py:52-65`). Pass the map at **every
  construction site** — `transactions.py:403` (Set-category),
  `transactions.py:420` (learning-offer rule), `rules.py:196` + `rules.py:221`
  (Rules add/edit) — each already holds `self._categorization` to call
  `sub_category_parent_names()`. The Transactions **filter** combo
  (`transactions.py:255`) keeps its `c.id` intersection unchanged and passes
  the map directly.

The Type grouping/headers, the sort, and the `category.id` userData resolution
are unchanged — the pickers still resolve a selection via `currentData()`
(`category_picker.py:56-58`, `rules.py:101-105`; the Transactions filter
resolves in `_matches`, `transactions.py:324`). *(The one-hop breadcrumb + the
Type tag is unique at the UI-authored depth ≤ 3. For any pre-existing **deeper**
node — § 4.4 — the flat-picker label may not be fully unique; the picker still
renders and resolves it by id, consistent with § 4.4's "show deeper data, don't
author it.")*

### 4.4 The depth cap is UI-only — service & schema unchanged

The cap of 3 lives **only in the UI** (the Add/Update parent choices exclude
Level-3 nodes; a re-parent that would push a subtree past Level 3 is
disallowed in the widget). The **service is deliberately not changed**:
`add_category` still accepts any existing parent, because existing service
tests build 4-deep chains on purpose (`test_categories.py:287-304`) and the
model/service allow arbitrary depth by design (FIBR-0006; the UI two-level limit
was the deferral, `ui/categories.py:9-10`). Enforcing the cap in the
service would break those tests and over-constrain a model designed to grow.
The cap is a **UI/UX legibility choice**, reversible without data work.

*(Consequence, stated so cold-eyes doesn't flag it as a gap: a vault could in
principle already contain a Level-4 node created by a prior direct-service
call or a future relaxed UI. The recursive render (§ 4.2) shows whatever depth
exists; the cap only governs what the UI lets a user **create**. This is
acceptable — the UI never crashes on deeper data, it just won't author it.)*

### 4.5 Assignability — a mid-tier Category stays a valid tag target

`leaf_categories()` returns **every non-root** category
(`services/categorization.py:148-155`), and `_require_leaf` only checks
`parent_id is not None` (`:297-303`) — so a Level-2 Category is assignable
today. FIBR-0154 **keeps** that: when *Groceries* gains a *Spar* child,
*Groceries* remains a valid pick target (both appear in the pickers, § 4.3).

Rationale: the alternative — "only true leaves assignable" — would
**retroactively invalidate** every transaction already tagged to a Level-2
category the moment it gains a child, a breaking change to existing user data.
Keeping "any non-root assignable" is backward-compatible and needs no
re-tagging. Reporting is unaffected: FIBR-0138 already sums a category's own
directly-tagged transactions **plus** its descendants' (depth-safe climb of
`parent_id`), so a parent that is both assignable and has children is counted
correctly.

- **INV-1 — The tree renders three levels.** `CategoriesWidget._refresh`
  renders root (Type) → Level-2 (Category) → Level-3 (Sub-category) as nested
  items. *Test (qtbot):* seed *Expenditure › Groceries › Spar* (via the
  service), build the widget, assert the tree has a 3-deep item whose text
  path is Type/Category/Sub-category.
- **INV-2 — Add creates a Level-3 sub-category under the selected node.** With
  *Groceries* selected in the tree, Add "Spar" creates a child of *Groceries*
  (§ 4.2, Add is anchored to the selection). *Test (qtbot):* select *Groceries*
  in the tree, set the name to "Spar", click Add, assert
  `children_of(groceries.id)` now contains a category named "Spar".
- **INV-3 — The UI caps depth at 3 (creation AND re-parent).** *(a)* Add is
  **disabled** when the selected node is Level-3 (or deeper) — a child would be
  Level 4 — and when nothing is selected (§ 4.2). *(b)* The "Move under…"
  re-parent selector for a **childed** Level-2 (a Level-2 with sub-categories)
  offers **only Types** — so re-parenting it under another Level-2 (which would
  push its children to Level 4) is impossible from the UI. *Test:* (a) select a
  seeded Level-3 "Spar", assert the Add control is disabled; select a Level-2,
  assert Add is enabled; (b) select a childed Level-2 (*Groceries* with child
  *Spar*), assert its "Move under…" choices = Types only (no other Level-2), and
  that the subject/its descendants are absent from the list.
- **INV-4 — Pickers disambiguate same-named Level-3 sub-categories, at the
  real call sites.** With *Groceries › Spar* and *Fuel › Spar* both seeded, each
  picker renders them as **distinct** rows (`"Groceries › Spar (Expenditure)"`
  vs `"Fuel › Spar (Expenditure)"`), each carrying its own `category.id`. *Test
  — dialog RENDER proven directly, per-site THREADING proven with a capture-spy*
  (the handlers build a local dialog + `show_modal` with no live-instance
  accessor, `transactions.py:403`, so read a real combo only where there is no
  dialog, and prove the dialog sites via the map they pass):
  - **accessor contract:** `sub_category_parent_names()` maps `spar.id` →
    "Groceries"/"Fuel" and **omits** `groceries.id`/`fuel.id` (Level-2) — an
    over-inclusive bug (breadcrumbing a Level-2 as "Expenditure › Groceries") is
    caught.
  - **dialog render:** construct `CategoryPickerDialog(…, parent_names=map)`
    (and `RuleEditDialog` likewise) directly, assert two distinct labels and
    that `currentData()` resolves each to the right id.
  - **per-site threading (capture-spy):** extend the existing
    `stub_picker`/`spy_learning` conftest factories to **capture** the
    `parent_names` kwarg; drive `_on_set_category` (`transactions.py:403`), the
    learning-offer (`:420`), and the Rules handlers (`rules.py:196`/`:221`), and
    assert each captured `parent_names == sub_category_parent_names()` — a site
    that forgets to pass it fails, no live combo needed.
  - **Transactions filter (real widget, no dialog):** with transactions tagged
    to both *Spar* nodes, build a `TransactionsView` and read the filter combo
    (`transactions.py:255`) → the two distinct breadcrumbed rows.

  All three consumers, and every site that threads the map, are covered — not
  just the shared helper.
- **INV-5 — A mid-tier Category stays assignable.** After *Groceries* gains a
  *Spar* child, *Groceries* still appears in the pickers and is still a valid
  manual/rule target; a transaction previously tagged to *Groceries* remains
  valid. *Test:* seed the child, assert *Groceries*'s id is still among the
  picker entries and `set_manual_category(txn.id, groceries.id)`
  (`categorization.py:271`) still succeeds.
- **INV-6 — Service & schema unchanged; existing depth tests still pass.** No

  the existing `test_categories.py` depth/cycle/delete tests (esp. the 4-deep

- **INV-7 — Rename & delete work at all three levels.** Renaming a Level-3
  sub-category updates it; deleting a Level-2 Category that **has** children is
  still refused (`CategoryHasChildrenError`), and deleting a childless Level-3

## 6. Exit criteria (falsifiable)

1. **Three levels visible & authorable.** The widget renders 3 deep and Add
   creates a Level-3 sub-category under a chosen Level-2 Category (INV-1/2).
2. **Cap holds — creation and re-parent.** No UI path creates a Level-4 node:
   the Add picker excludes Level-3 parents, and a childed Level-2 can only be
   re-parented under a Type (INV-3a/b); the service/schema are untouched and
   the existing arbitrary-depth service tests stay green (INV-6).
3. **Pickers unambiguous & resolving.** Same-named Level-3 sub-categories
   render as distinct breadcrumbed rows across all three pickers and resolve
   to the right id (INV-4); mid-tier categories remain assignable (INV-5).
4. **Rename/delete intact at depth 3** (INV-7).
5. **Gate green.** `./scripts/ci-local.sh` passes, including the new/extended
   `tests/features/categories/` (+ picker) tests.
