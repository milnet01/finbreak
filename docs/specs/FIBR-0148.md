## The bug

Import de-duplication (`services/import_.py._dedup`, FIBR-0052 D6/INV-5) is a
**multiset delta**: a transaction that already exists is **dropped at import**,
never stored twice. The single stored copy is stamped
(`transactions.statement_period_id`, a **single-valued** FK) to whichever
statement imported it **first**.

`StatementService.delete_statement(period_id)` today deletes **every** row
stamped with `period_id`, then the period row — with **no** check for other
statements. When two statements' periods overlap, the shared transactions live
under only **one** of them (the first importer), so deleting that one destroys
rows the **other, still-present** statement legitimately covers.

**Concrete data loss.** Account has statement **A** (period 2026-01-01…01-31),
imported first, and statement **B** (period 2026-01-01…02-28), imported second.
B's January rows are deduped away at import — only February is inserted — so
every January row is stamped to **A**. Delete **A** → the January transactions
vanish, even though **B** is still present and its period covers all of January.
B is not re-imported, so nothing restores them: B now shows a coverage span with
a silent January hole. Totals, the dashboard, and recurring-detection all
under-count from that moment, with no error and no undo.

## The fix — ownership hand-off on delete (direction (a))

Before removing a statement's rows, **hand off** each row whose date another
**remaining** statement of the **same account** also covers: re-stamp its
`statement_period_id` to that remaining statement instead of deleting it. Only
rows that **no** remaining statement covers are deleted. Delete becomes
"remove this statement, and remove only the transactions nothing else still
accounts for."

This mirrors, in reverse, the v5→v6 backfill (`_migrate_to_v6`): that query
stamps an unattributed row to the one period whose span covers it; this query
re-stamps a row **away** from the deleted period to a **remaining** period whose
span covers it.

**Why span-overlap is the right test.** A bank statement lists **every**
transaction in its period; import only dropped B's January rows because A had
already stored them. So "B's period covers this date" is a faithful proxy for
"B's statement listed this transaction" — B is a correct owner. (This is the
one modelling assumption, carried from the v6 backfill's own D9 assumption, and
called out in the roadmap bullet.)

### Ordering (one atomic unit)

`delete_statement` runs **one** `owned_transaction`, in this order:

1. **Hand off** — re-stamp rows of `period_id` that a remaining same-account
   statement covers, to that statement.
2. **Delete orphans** — delete rows **still** stamped `period_id` (nothing else
   covers them).
3. **Delete the period row.**

Hand-off **must** precede the delete (else the covered rows are already gone),
and both precede the period-row delete so the plain (non-cascade) FK never
trips: after step 1 the covered rows point at a **remaining** period (valid FK);
after step 2 no row points at `period_id`; step 3 then removes it. Any failure
`ROLLBACK`s all three to a re-openable vault (FIBR-0052 INV-9, unchanged).

### Choosing the owner when several remain

When **≥2** remaining statements cover a row's date, any of them is a valid
owner (each legitimately lists that transaction). The pick is **deterministic**
— `ORDER BY period_start, id LIMIT 1` (earliest-starting, then lowest id) —
purely so the outcome is stable and testable; it carries no semantic weight.

### The NULL trap (the money-critical corner)

The correlated sub-query that supplies the new owner returns `NULL` for a row no
remaining statement covers. If that `NULL` were written, the row would silently
become a **manual** (un-stamped) row instead of being deleted — a different but
equally wrong outcome. The `UPDATE` is therefore guarded by an `EXISTS` on the
**same** predicate as the sub-query, so **only** rows that genuinely have a
covering remaining statement are re-stamped; every other row keeps `period_id`
and is deleted in step 2. **Invariant: this operation never turns a row into a
manual (`NULL`-stamped) row.**

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| **INV-1** | **No silent loss.** After `delete_statement(A)`, every transaction whose date falls in a **remaining** same-account statement's period still exists, re-stamped to that statement. A test builds the A(Jan)+B(Jan–Feb) overlap, deletes A, and asserts January rows survive **and** now report `statement_period_id == B`. | The bug; money-correctness |
| **INV-2** | **Orphans still go.** A transaction that **no** remaining statement covers is deleted (unchanged behaviour). A test with a single statement, or two non-overlapping ones, asserts the target's rows are removed and the returned count equals the number actually deleted. | FIBR-0052 INV-9 |
| **INV-3** | **Never manual-ised.** No row is left `statement_period_id IS NULL` by a delete. A test asserts zero rows flip to `NULL` (the covered ones move to B, the uncovered ones are deleted — none become manual). | The NULL trap |
| **INV-4** | **Atomic + re-openable.** Hand-off, orphan-delete, and period-delete are one `owned_transaction`; a forced failure mid-operation `ROLLBACK`s **all three** and the vault re-opens with A, B, and every row intact. A test wedges a failure and asserts full rollback. | FIBR-0052 INV-9 |
| **INV-5** | **Deterministic owner.** With ≥2 remaining statements covering a date, the row is handed to the one ordered first by `(period_start, id)`. A test with two remaining covering statements asserts the stable pick. | Testability |
| **INV-6** | **Account-scoped.** Hand-off only ever re-stamps to a statement of the **same account** (`q.account_id = transactions.account_id`); an overlapping period on a *different* account never adopts a row. A test with a same-span statement on another account asserts no cross-account hand-off. | Data-model integrity |
| **INV-7** | **Provenance only.** Hand-off changes `statement_period_id` alone — never `account_id`, `category_id`, `amount_minor`, `occurred_on`, or `description` — so dashboard totals, transfers, recurring-detection, and categories are unaffected by the row's change of owner. A test asserts the moved row's other columns are byte-identical. | No ripple |

## Design decisions

- **D1 — Repository seam.** Add `TransactionRepository.hand_off_covered(
  statement_period_id: int) -> int`, commit-free (like `delete_for_statement` /
  `reassign_account`), returning the rowcount re-stamped. Keeps the SQL beside
  the other statement-scoped transaction mutations and out of the service.
- **D2 — Service change is additive + ordered.** `delete_statement` gains one
  line — `handed_off = tx_repo.hand_off_covered(period_id)` as the **first**
  statement inside the existing `with owned_transaction(conn):`, before
  `delete_for_statement`. The return value stays `int` = **rows deleted**
  (orphans), preserving the INV-9 contract's meaning ("number of transactions
  deleted"). The log line names both counts.
- **D3 — Return value semantics.** `delete_statement` returns the count of rows
  **actually deleted**, not the statement's linked-row count. For a
  non-overlapping delete this equals the old value (all rows orphaned); for an
  overlapping delete it is smaller (some handed off). The UI does **not** use
  the return value (the confirm dialog names `statement.transaction_count`, and
  the refresh is signal-driven), so no UI change is required or made.
- **D4 — SQL reuses the v6 idiom verbatim in shape.** The `WHERE` +
  correlated-`EXISTS` predicate is the v6 backfill's overlap test with the
  polarity flipped (`q.id <> :pid` = "a *different*, i.e. remaining, period").
  The `SET` sub-query adds `ORDER BY period_start, id LIMIT 1` for the
  deterministic single-valued pick (INV-5).
- **D5 — No confirm-dialog wording change.** The dialog says "Delete this
  statement and its %n transaction(s)?" — still true (those rows *are* this
  statement's; some simply survive under another statement rather than being
  destroyed). Re-wording to explain hand-off adds cognitive load for the common
  (non-overlapping) case where nothing is handed off. Left unchanged; revisit
  only if users find the count confusing (a follow-up, not this fix).
- **D6 — No schema change, no migration.** The column, FK, and indexes already
  exist (`statement_period_id`, `idx_transactions_statement_period_id` from
  FIBR-0148's sibling v10 work). `LATEST_SCHEMA_VERSION` is untouched.

## Out of scope

- The **many-to-many** transaction↔statement model (roadmap direction (b)) — the
  most complete fix but a data-model change; not needed once hand-off closes the
  loss path. Left on the roadmap as a possible future.
- **Warn-on-delete** (direction (c)) — unnecessary once (a) makes delete safe.
- **Re-import restoration** of already-lost rows from a prior delete — this fix
  is forward-looking; it does not reconstruct data a *previous* buggy delete
  already destroyed (there is nothing on disk to reconstruct from).
- Any change to import, dedup, or the FIBR-0147 transfer-pairs cascade index.
