## The bug

Import de-duplication (`services/import_.py._dedup`, FIBR-0052 D6/INV-5) is a
**multiset delta**: a transaction that already exists is **dropped at import**,
never stored twice. The single stored copy is stamped
(`transactions.statement_period_id`, a **single-valued** FK) to whichever
statement imported it **first**.

`StatementService.delete_statement(period_id)` today deletes **every** row
stamped with `period_id`, then the period row ŌĆö with **no** check for other
statements. When two statements' periods overlap, the shared transactions live
under only **one** of them (the first importer), so deleting that one destroys
rows the **other, still-present** statement legitimately covers.

**Concrete data loss.** Account has statement **A** (period 2026-01-01ŌĆ”01-31),
imported first, and statement **B** (period 2026-01-01ŌĆ”02-28), imported second.
B's January rows are deduped away at import ŌĆö only February is inserted ŌĆö so
every January row is stamped to **A**. Delete **A** ŌåÆ the January transactions
vanish, even though **B** is still present and its period covers all of January.
B is not re-imported, so nothing restores them: B now shows a coverage span with
a silent January hole. Totals, the dashboard, and recurring-detection all
under-count from that moment, with no error and no undo.

## The fix ŌĆö ownership hand-off on delete (direction (a))

Before removing a statement's rows, **hand off** each row whose date another
**remaining** statement of the **same account** also covers: re-stamp its
`statement_period_id` to that remaining statement instead of deleting it. Only
rows that **no** remaining statement covers are deleted. Delete becomes
"remove this statement, and remove only the transactions nothing else still
accounts for."

This mirrors, in reverse, the v5ŌåÆv6 backfill (`_migrate_to_v6`): that query
stamps an unattributed row to the one period whose span covers it; this query
re-stamps a row **away** from the deleted period to a **remaining** period whose
span covers it.

**Why span-overlap is the right test.** A bank statement lists **every**
transaction in its period; import only dropped B's January rows because A had
already stored them. So "B's period covers this date" is a faithful proxy for
"B's statement listed this transaction" ŌĆö B is a correct owner. (This is the
one modelling assumption, carried from the v6 backfill's own D9 assumption, and
called out in the roadmap bullet.)

### Ordering (one atomic unit)

`delete_statement` runs **one** `owned_transaction`, in this order:

1. **Hand off** ŌĆö re-stamp rows of `period_id` that a remaining same-account
   statement covers, to that statement.
2. **Delete orphans** ŌĆö delete rows **still** stamped `period_id` (nothing else
   covers them).
3. **Delete the period row.**

Hand-off **must** precede the delete (else the covered rows are already gone),
and both precede the period-row delete so the plain (non-cascade) FK never
trips: after step 1 the covered rows point at a **remaining** period (valid FK);
after step 2 no row points at `period_id`; step 3 then removes it. Any failure
`ROLLBACK`s all three to a re-openable vault (FIBR-0052 INV-9, unchanged).

### The hand-off query (literal)

Step 1 is exactly this one statement (`:pid` = the period being deleted). The
`SET` sub-query and the `EXISTS` guard share a **byte-identical** predicate (the
three indented lines) ŌĆö this is the money-critical invariant, so it is written
out in full rather than left to prose (as FIBR-0052 D9 does for its own backfill):

```sql
UPDATE transactions
   SET statement_period_id = (
           SELECT q.id FROM statement_periods q
            WHERE q.account_id = transactions.account_id
              AND q.id <> :pid
              AND transactions.occurred_on BETWEEN q.period_start AND q.period_end
            ORDER BY q.period_start, q.id
            LIMIT 1)
 WHERE statement_period_id = :pid
   AND EXISTS (
           SELECT 1 FROM statement_periods q
            WHERE q.account_id = transactions.account_id
              AND q.id <> :pid
              AND transactions.occurred_on BETWEEN q.period_start AND q.period_end);
```

The `SET` sub-query adds only `ORDER BY q.period_start, q.id LIMIT 1` on top of
that shared predicate ŌĆö the deterministic single-owner pick (INV-5). Because the
`EXISTS` guard uses the same predicate, the sub-query can only run (and so can
only return non-`NULL`) for rows the guard already admitted: no row is ever
re-stamped to `NULL`.

### Choosing the owner when several remain

When **Ōēź2** remaining statements cover a row's date, any of them is a valid
owner (each legitimately lists that transaction). The pick is **deterministic**
ŌĆö `ORDER BY period_start, id LIMIT 1` (earliest-starting, then lowest id) ŌĆö
purely so the outcome is stable and testable; it carries no semantic weight.

### The NULL trap (the money-critical corner)

The correlated sub-query that supplies the new owner returns `NULL` for a row no
remaining statement covers. If that `NULL` were written, the row would silently
become a **manual** (un-stamped) row instead of being deleted ŌĆö a different but
equally wrong outcome. The `UPDATE` is therefore guarded by an `EXISTS` on the
**same** predicate as the sub-query, so **only** rows that genuinely have a
covering remaining statement are re-stamped; every other row keeps `period_id`
and is deleted in step 2. **Invariant: this operation never turns a row into a
manual (`NULL`-stamped) row.**

### Relationship to FIBR-0052 INV-9

This fix **amends** FIBR-0052 INV-9's *deletion* clause. INV-9 as written
guarantees `delete_statement` "removes only the target's stamped rows + its
record" ŌĆö i.e. it deletes **every** stamped row. FIBR-0148 supersedes that one
clause: covered rows are now **handed off, not deleted**. INV-9's other
guarantees are **retained unchanged** ŌĆö the operation is still one atomic
`owned_transaction` that leaves a re-openable vault on failure, still never
touches manual (`NULL`-stamped) rows, and its return value still means "rows
deleted". The "Upholds: FIBR-0052 INV-9" cells above refer to these retained
guarantees. FIBR-0052's own INV-9 row carries a forward-pointer to here, and the
now-inaccurate line in `tests/features/statements/spec.md` is corrected (D-c).

## Design decisions

- **D1 ŌĆö Repository seam.** Add `TransactionRepository.hand_off_covered(
  statement_period_id: int) -> int`, commit-free (like `delete_for_statement` /
  `reassign_account`), returning the rowcount re-stamped. Keeps the SQL beside
  the other statement-scoped transaction mutations and out of the service.
- **D2 ŌĆö Service change is additive + ordered.** `delete_statement` gains one
  line ŌĆö `handed_off = tx_repo.hand_off_covered(period_id)` as the **first**
  statement inside the existing `with owned_transaction(conn):`, before
  `delete_for_statement`. The return value stays `int` = **rows deleted**
  (orphans), preserving the INV-9 contract's meaning ("number of transactions
  deleted"). The log line names both counts.
- **D3 ŌĆö Return value semantics.** `delete_statement` returns the count of rows
  **actually deleted**, not the statement's linked-row count. For a
  non-overlapping delete this equals the old value (all rows orphaned); for an
  overlapping delete it is smaller (some handed off). The UI does **not** use
  the return value (the confirm dialog names `statement.transaction_count`, and
  the refresh is signal-driven), so no UI change is required or made.
- **D4 ŌĆö SQL reuses the v6 idiom, with three concrete changes.** The overlap
  test (same-account `q.id <> :pid` ŌĆ” `BETWEEN q.period_start AND q.period_end`)
  is copied verbatim from `_migrate_to_v6`; what changes is: (1) the guard flips
  `NOT EXISTS` ŌåÆ `EXISTS` (skip-if-covered becomes hand-off-if-covered); (2) the
  outer `WHERE` keys on `statement_period_id = :pid` instead of `IS NULL`; and
  (3) the `SET` sub-query adds `ORDER BY q.period_start, q.id LIMIT 1` for the
  deterministic single-valued pick (INV-5). See the literal query above.
- **D5 ŌĆö No confirm-dialog wording change.** The dialog says "Delete this
  statement and its %n transaction(s)?" ŌĆö still true (those rows *are* this
  statement's; some simply survive under another statement rather than being
  destroyed). Re-wording to explain hand-off adds cognitive load for the common
  (non-overlapping) case where nothing is handed off. Left unchanged; revisit
  only if users find the count confusing (a follow-up, not this fix).
- **D6 ŌĆö No schema change, no migration.** The column, FK, and index already
  exist: `statement_period_id` (the FIBR-0052 v5ŌåÆv6 provenance stamp) and
  `idx_transactions_statement_period_id` (the v9ŌåÆv10 hot-path index step,
  `_migrate_to_v6`'s sibling `_migrate_to_v10`, FIBR-0098/0071/0026).
  `LATEST_SCHEMA_VERSION` (10) is untouched.
