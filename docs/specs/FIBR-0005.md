**Theme:** money lives in accounts. A user can own several accounts
(a current account, a savings account, a credit card, a home loan…),
each tagged with a **type**, and **every transaction belongs to one
account**. This phase adds the account model, its CRUD, an Accounts
manager screen, and — because it is the first phase to change the
database shape — the **forward-only schema-migration runner** that
FIBR-0004 deliberately deferred. The FIBR-0004 vault was created at
schema version 1 with account-less transactions; this phase's
migration takes any such vault to version 2: it creates the
`accounts` table, seeds one **"Default"** account, links every
existing transaction to it, and makes `transactions.account_id` a
required column. Import, transfer detection, and the dashboard all
depend on this account link, so it must exist before any of them —
which is why P03 sits immediately after the security spine.

## Context

FIBR-0004 (P02) shipped the vertical security slice: Argon2id →
SQLCipher vault → one account-less transaction → table → lock. Its
`transactions` table has **no `account_id`**, and the vault is
created at `schema_version = 1` **directly** — FIBR-0004 created the
baseline schema in `Vault.create()` and explicitly deferred the
"forward-migration runner" to "the first phase that needs a version
bump (P03)" (FIBR-0004 spec, "Out of scope"). This is that phase.

Per design.md the app is layered — **UI → Services → Repositories →
SQLCipher** — and the "Persistence" concern already states the
migration policy this phase implements: *"a `schema_version` table;
migrations run on unlock, forward-only, each in a transaction."* The
`accounts` table and the "Accounts manager" screen are both named in
design.md ("Repository layer", "Components"). So this phase adds no
new architecture — it fills in the account aggregate that was always
planned, plus the migration machinery every later schema change will
reuse.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` and the user sign-off
gate them before code. **D1–D5 are engineering calls made in this
spec; D6, D8, and D9 are the user's product decisions of 2026-07-02
(D7 follows from INV-5 + the always-seeded Default account).**

1. **D1 — First-run lays down the v1 baseline, then migrates to
   latest — one schema-evolution path, no duplication.**
   `Vault.create()` keeps writing the FIBR-0004 **v1 baseline**
   (`schema_version = 1`, `settings`, account-less `transactions`),
   then runs the migration runner to bring the fresh vault to the
   latest version — exactly the path an *existing* v1 vault takes on
   unlock. So the `accounts` schema and the `account_id` link are
   defined in **one place** (the v1→v2 migration), never duplicated
   between a "create latest directly" path and a "migrate old vaults"
   path (coding.md § 1.3, reuse-before-rewriting). A brand-new vault and
   an upgraded vault converge on the identical v2 shape, and the
   migration path is therefore exercised on every first run (it is
   continuously tested, not a rarely-run branch).

2. **D2 — The migration runs on both `create()` and `open()`, is
   idempotent, and is atomic.** (Running on `open()` *is* design.md's
   "migrations run on unlock"; running on `create()` too is a
   deliberate **extension** — it is what lets first-run and upgrade
   converge on one v2 shape, D1 — not a contradiction of that rule.)
   `run_migrations(conn)` reads the
   current `schema_version` and applies each pending step in order,
   updating the version row as it goes; on an already-current vault
   it is a no-op. The **whole v1→v2 step is one all-or-nothing unit**
   — a failure partway (e.g. after the old `transactions` table is
   dropped but before the rebuilt one is in place) must **roll back**
   to a re-openable v1 vault, never a half-migrated one. SQLite DDL
   (`CREATE` / `DROP` / `ALTER`) is transactional *at the engine
   level*, but the DBAPI driver decides whether statements are wrapped
   in a transaction — and with the vault's `isolation_level=""`,
   `sqlcipher3`'s implicit `BEGIN` fires before **DML** (INSERT/UPDATE/
   DELETE) but **not** before **DDL**, so an un-wrapped `DROP`/`RENAME`
   would each autocommit and the rollback would silently not occur. So
   **the runner itself owns the transaction**:
   `try: BEGIN … COMMIT except: ROLLBACK; raise` around the whole v1→v2
   step. The explicit `BEGIN` is the step's **first statement** (before
   the seed `INSERT` and the DDL), so `sqlcipher3`'s
   implicit-`BEGIN`-before-DML can't pre-open a transaction that would
   make the explicit `BEGIN` raise "cannot start a transaction within a
   transaction." The rollback is therefore the runner's own — on
   **both** the `open()` and `create()` paths — never the driver's
   implicit `BEGIN` nor a side effect of the caller closing the
   connection. The implementation-time check (global rule § 13) confirms
   only *how* to own it under `isolation_level=""` (explicit `BEGIN`, or
   briefly setting `conn.isolation_level=None`), not *whether*.

3. **D3 — Adding the `NOT NULL account_id` column is a SQLite table
   rebuild, not an `ALTER … ADD COLUMN`.** SQLite cannot `ALTER TABLE
   … ADD COLUMN` a column that is both `NOT NULL` **and** carries a
   `REFERENCES` clause: a `NOT NULL` add needs a non-NULL literal
   default, while — with foreign keys enabled (which D4 turns on) — a
   `REFERENCES` add requires the default be `NULL`, so the two cannot
   be combined in one `ADD COLUMN`. The idiomatic fix (SQLite docs,
   "Making Other Kinds Of Table Schema Changes") is to build a new
   `transactions` table with the desired shape, `INSERT … SELECT` the
   old rows (stamping each with the Default account's id), `DROP` the
   old, and `RENAME` the new into place — all inside the D2
   transaction. The rebuilt column is
   `account_id INTEGER NOT NULL REFERENCES accounts(id)` with **no
   baked-in column default** (the backfill supplies the value
   explicitly).

4. **D4 — Referential integrity is enforced two ways: an explicit,
   tested app-level guard is the primary contract; `PRAGMA
   foreign_keys = ON` is defence-in-depth.** SQLite leaves foreign
   keys **off** by default, so this phase turns them on once per
   connection (in `Vault._connect`, after the raw-key pragma and
   before the connection's first statement — a *change* to
   `foreign_keys` is a no-op inside a transaction, so it must be set
   pre-transaction; once ON it **stays enforced** across the D2
   migration transaction, which is why the rebuild is written to leave
   no dangling reference — and it is safe precisely because nothing
   references `transactions` in the v2 schema). `_connect` is the
   **single** connect seam shared by `create()` and `open()`, so the
   FK pragma governs the first-run baseline write too (intended per
   D1) — harmlessly, since the v1 baseline schema has **no**
   `REFERENCES` clauses at all (FK-ON is inert until the migration's
   rebuild introduces the `account_id` foreign key).
   That makes the
   `REFERENCES accounts(id)` clause real: inserting a transaction
   against a non-existent account raises `IntegrityError`, and (with
   no `ON DELETE CASCADE`) deleting a referenced account raises it
   too. But the **enforced, user-facing contract** for "you can't
   delete an in-use account" (D6) is an **explicit app-level count
   check** in `AccountService` that raises `AccountInUseError` with a
   clear message *before* any `DELETE` runs — the FK is the backstop,
   not the mechanism the user sees. Belt-and-suspenders, each with
   its own test leg (INV-5, INV-6).

5. **D5 — Account `type` is a closed enum stored as a stable token;
   the salt-vs-secrecy analogue for i18n is token-vs-label.**
   `AccountType` is a `str`-valued `Enum` of exactly seven members —
   `current`, `savings`, `credit_card`, `personal_loan`,
   `home_loan`, `investment`, `other`. (The roadmap spells the
   multi-word ones with spaces — "credit card" → `credit_card`,
   "personal loan" → `personal_loan`, "home loan" → `home_loan` — as
   prose: those are the display labels; the underscored forms are the
   stored tokens.) The **token** (the enum value)
   is what is stored in `accounts.type` and what any later code
   branches on; it is **never translated** (design.md i18n:
   "Non-display strings — … enum values — are not translated"). The
   human-readable **label** shown in the type picker ("Credit card",
   "Home loan", …) is produced separately via `tr()` in the UI layer.
   So adding a locale never risks rewriting stored data, and the DB
   stays language-neutral.

6. **D6 — Deleting an account that still holds transactions is
   *blocked*, not cascaded or auto-reassigned** (user decision
   2026-07-02). The safest default for financial data: the app
   refuses the delete and tells the user to clear or move the
   transactions first — no silent data loss. Cascade-delete and
   delete-time reassignment were both considered and rejected for
   this phase.

7. **D7 — There is always at least one account; the last one cannot
   be deleted.** Because a transaction must belong to an account
   (INV-5) and first-run/migration always seeds the Default account,
   the invariant "≥ 1 account exists" holds from creation. Deleting
   the *only* remaining account is refused (a distinct, clear message
   from the in-use guard), so the add-transaction screen can always
   offer at least one account and never reaches a dead-end with no
   account to attach a transaction to.

8. **D8 — The seeded account is named "Default", type `current`, and
   is fully editable** (user decision 2026-07-02: auto-create, don't
   prompt at first-run). First-run stays a two-field screen (password
   + base currency, unchanged from FIBR-0004); the account is seeded
   by the migration for **both** new and upgraded vaults, and the
   user renames / retypes / adds more in the Accounts manager. The
   name "Default" is **seed data, not a UI string** — it is written
   to the DB by the migration and is therefore **not** run through
   `tr()` (translating it would make it locale-dependent stored
   data, violating D5); the user renaming it is the localisation
   path.

9. **D9 — Changing which account an *existing* transaction belongs to
   is out of scope** (user decision 2026-07-02). The account is
   chosen when a transaction is added; editing an existing
   transaction's account arrives with the broader transaction-editing
   work in a later phase. Combined with D6 (block-on-delete), this
   means an account that has ever accrued transactions cannot be
   emptied *within this phase* — an accepted, documented limitation
   for the MVP, not an oversight.

## Invariants (testable)

> These `INV-N` are **local to this spec**. Tests live in
> `tests/features/accounts/` (spec.md + `test_accounts.py`) unless
> noted, use `tmp_path` for every on-disk vault, and never touch the
> network or real financial data (testing.md § 6). GUI-touching
> invariants use the `pytest-qt` `qtbot` fixture (enabled since
> FIBR-0004). Schema-shape changes that ripple into the FIBR-0004
> vault tests are updated in place in `tests/features/vault/`
> (see Deliverables).

- **INV-1 — Account model & CRUD round-trip.**
  `AccountRepository.add(name, type)` inserts one row and returns its
  new integer id; `.list_all()` returns every account as an
  `Account(id, name, type, created_at)` record ordered
  deterministically by **name (case-insensitive), then id**;
  `.get(id)` returns the matching `Account` or `None`;
  `.update(id, name, type)` overwrites name + type in place;
  `.delete(id)` removes exactly that row. Deleting or updating a
  **missing** id is an idempotent no-op (zero rows affected, no raise).
  A round-trip (add → get)
  returns the same `name`, `type`, and a well-formed ISO-8601
  `created_at`. Source: design.md "Repository layer".

- **INV-2 — Account type is a closed, non-translated set.**
  `AccountType` has exactly the seven members `current`, `savings`,
  `credit_card`, `personal_loan`, `home_loan`, `investment`,
  `other`; their stored values are those exact lowercase tokens.
  `AccountService.add_account` / `.update_account` **reject** a type
  outside this set with `ValueError` (the token is validated against
  `AccountType`, not against a display label), and store the token
  **verbatim** (a value read back from the DB equals the token that
  was written — no case-folding, no translation). INV-2 asserts only
  token storage + validation (headless); the seven display labels'
  one-to-one mapping onto the tokens is owned and asserted by **INV-7a**
  (it needs the UI). A bad type (INV-2) and a bad name (INV-3) both
  raise plain `ValueError` **on purpose** — they are input-validation
  errors, not lifecycle errors, so unlike the delete-guard's
  distinct-typed `AccountInUseError`/`LastAccountError` they need no
  type distinction. Source: D5 (the closed-set decision); roadmap
  FIBR-0005 (the enumeration); design.md i18n.

- **INV-3 — Name validation & uniqueness.** `add_account` /
  `update_account` raise `ValueError` when the name is empty or
  whitespace-only (after `str.strip()`), or when it duplicates an
  existing account's name **case-insensitively** (trimmed) — on
  `update`, the account's *own* current name is excluded from the
  duplicate check so re-saving an unchanged name is allowed. A valid,
  unique, trimmed name is accepted and stored trimmed. Source:
  design.md "Error handling".

- **INV-4 — v1→v2 migration: forward-only, atomic, idempotent, with
  backfill.** Opening a **v1** vault (the FIBR-0004 shape:
  `schema_version = 1`, account-less `transactions`, possibly holding
  rows) runs `run_migrations`, which:
  - creates the `accounts` table (§ Data model),
  - inserts exactly one seed account — name `"Default"`, type
    `current`, ISO-8601 `created_at` — and captures its id,
  - rebuilds `transactions` with a `NOT NULL account_id REFERENCES
    accounts(id)` column (D3), copying every existing row and
    **stamping each with the Default account's id** (every backfilled
    row's `account_id` equals the seeded Default account's id; no row
    lost, no amount/description/date altered — asserted by comparing
    the full pre/post row set),
  - sets `schema_version` to 2.

  Four guarantees hold, each a test leg:
  - **Atomic.** The whole v1→v2 step runs inside the runner's own
    `try: BEGIN … COMMIT except: ROLLBACK; raise` (D2). A forced
    failure midway — the test wraps `conn.execute` so it raises **only
    when the SQL text contains `RENAME`** (letting `BEGIN`, `DROP`, the
    `INSERT … SELECT` backfill, and the runner's own `ROLLBACK` pass
    through), i.e. after the old `transactions` table is dropped, before
    the rebuilt one is renamed into place — leaves the vault
    **re-openable at v1 with its original account-less rows intact**
    (the `ROLLBACK` undoes the DROP), never half-migrated. (This
    SQL-text discriminator is the migration's wedge point, the analogue
    of FIBR-0004 INV-4a's `_commit` seam.)
  - **Idempotent.** Re-running `run_migrations` on a **v2** vault makes
    no change (version stays 2; the Default account is not duplicated).
  - **Baseline-complete (INV-4 owns this claim).** A brand-new
    first-run vault ends at exactly **v2** with the Default account
    present (D1 — same path), because the baseline `SCHEMA_VERSION`
    (= 1, what `create()` writes) equals the first migration step's
    *from*-version — so create-then-migrate leaves no gap and a fresh
    vault is never stuck at v1.
  - **Forward-only.** No downgrade path; a vault whose recorded version
    exceeds `LATEST_SCHEMA_VERSION` (= 2) is refused by raising the
    dedicated `SchemaVersionError` (a new `errors.py` type — written by
    a newer build; assertable by type, kept distinct from the
    presence-mismatch `VaultStateError`), not silently downgraded.

  Source: design.md "Persistence"; FIBR-0004 "Out of scope".

- **INV-5 — Every transaction belongs to an account.** After
  migration, `transactions.account_id` is `NOT NULL`.
  `TransactionRepository.add(account_id, occurred_on, amount_minor,
  description)` requires the account id; `.list_all()` returns it on
  each `Transaction`; `TransactionService.add_transaction` takes the
  chosen account and `.list_transactions()` carries each
  transaction's **account name** for display. With `PRAGMA
  foreign_keys = ON` (D4), inserting a transaction whose `account_id`
  does **not** exist raises `sqlcipher3.dbapi2.IntegrityError` (the
  FK backstop), and the row is not written. Two **headless** legs lock
  the link below the UI: (a) a transaction inserted under account *A*
  reads back with `account_id == A.id` **and** `id != account_id`
  (distinct values — catches an `id`/`account_id` column transposition;
  the test arranges the two to differ, e.g. a second account or a
  second transaction, since the *first* row under the Default account
  would have `id == account_id == 1`); (b) `list_transactions()`
  returns *A*'s **name** for that row (the id→name join is verified
  without a `qtbot` test). The
  FIBR-0004 atomicity
  and exact-money invariants (one INSERT + explicit commit; minor
  units; no float) are **preserved** across the schema change.
  Source: roadmap FIBR-0005; design.md "Persistence".

- **INV-6 — Delete guard (block-in-use + keep ≥ 1).**
  `AccountService.delete_account(id)`:
  - raises `AccountInUseError` (nothing removed) when the account has
    **≥ 1 transaction** — the count is checked in the service via
    `TransactionRepository.count_for_account(id)` **before** any
    `DELETE` runs (D6), so the user-facing failure is the app guard,
    not a raw FK `IntegrityError`;
  - raises `LastAccountError` — a type **distinct** from
    `AccountInUseError` — when the target **exists and is the only
    row** (`get(id) is not None` **and** total accounts == 1) (D7),
    even if it is empty; the "exists" clause is what makes a *missing*
    id fall through instead of being mistaken for the last account;
  - **succeeds**, removing exactly that row, for an **empty,
    non-last** account.
  - on a **missing** id, **falls through both guards** to the
    repository's zero-row no-op (the count is 0 so not in-use, and a
    missing id is not the last account) — no early "exists?" guard, no
    raise; consistent with INV-1's missing-id no-op.

  (finbreak is single-user and single-threaded, so nothing writes
  between the count check and the `DELETE`; the `PRAGMA foreign_keys`
  backstop only bites if a *future* code path forgets the count guard,
  and a raw `IntegrityError` on that unreachable path is acceptable —
  the guard, not the backstop, is what INV-6 asserts.) Asserted
  headless (create two accounts; give one a transaction; confirm the
  in-use one is refused, the empty one deletes, and the now-only
  account is refused). Source: user decisions D6/D7.

- **INV-7 — Accounts-manager UI round-trip** (`qtbot`; sub-cases per
  testing.md § 4):
  - **INV-7a** — the **type picker** offers exactly the seven types
    with translated labels, and each label maps back to its stored
    token (the label↔token assertion INV-2 delegates here).
  - **INV-7b** — entering a name + type and pressing **Add** makes the
    account appear in the on-screen list.
  - **INV-7c** — the added account is selectable in the main window's
    add-transaction **account picker**, and a transaction added under
    it shows **that account's name** in the table's Account column (the
    id→name resolution, INV-5).
  - **INV-7d** — attempting to **delete** an account that has a
    transaction shows a clear message and removes nothing (re-asserts
    INV-6 from the UI).
  - **INV-7e** — an empty, non-last account deletes and disappears
    from the list.
  - **INV-7f** — selecting an account loads its name + type into the
    form, and **Update selected** renames / retypes it in place (the
    add/**edit** form of the Deliverable; the UI path onto the
    already-tested `AccountService.update_account`, INV-1/INV-3). This
    is the editing route that keeps a mistyped name — or the seeded
    "Default" name (D8) — correctable, given deletion is block-only
    (D6).

  All strings via `self.tr(...)`; layout via Qt layout managers
  (RTL-safe, coding.md § 5.2). Source: design.md "Accounts manager".

- **INV-8 — No secret/network regression in the new modules.** The
  files this phase adds introduce **no** network import (`socket`,
  `http`, `urllib`, `requests`, `ftplib`, …) anywhere under
  `src/finbreak/`, and log **no** account data beyond non-secret
  lifecycle lines (an account *name* is user data, not a secret, but
  transaction **amounts** are never logged). This **folds FIBR-0004's
  INV-7 log-cleanliness leg into this INV** — P03 adds no new
  secret-bearing module, only user-data-bearing ones — with two legs:
  - **No-network (automatic).** The vault-suite's existing **static
    import scan** already walks **all** of `src/finbreak/`, so it
    covers the new modules with no new test.
  - **No-secret-log (new leg).** FIBR-0004's INV-7 `caplog` capture ran
    only over the auth lifecycle, not account CRUD, so this phase adds
    a `caplog` capture over an **account add → delete cycle** asserting
    no password/key bytes appear (account names and non-secret
    lifecycle lines are allowed; transaction amounts must not appear).

  Source: security-model INV-8, INV-9.

## Deliverables

**New application modules** (`src/finbreak/`; shortest correct
implementation, coding.md § 1.1):

- `migrations.py` — the forward-only runner: `LATEST_SCHEMA_VERSION =
  2`, an ordered registry mapping a target version to its migration
  function, and `run_migrations(conn) -> None` (read current version
  → apply each pending step inside a runner-owned
  `try: BEGIN … COMMIT except: ROLLBACK; raise`, the explicit `BEGIN`
  the step's first statement (D2 — the rollback is the runner's, not
  the caller's) → raise `SchemaVersionError` on a version newer than
  `LATEST_SCHEMA_VERSION`, INV-4). Holds the single v1→v2 step (create `accounts`,
  seed Default, rebuild `transactions` with `account_id`, backfill,
  bump version). Seed constants (`DEFAULT_ACCOUNT_NAME = "Default"`,
  `DEFAULT_ACCOUNT_TYPE = "current"`) live here as **data** (D8 — not
  `tr()`-wrapped).
- `repositories/accounts.py` — `AccountRepository` with `add`,
  `list_all`, `get`, `update`, `delete` (each write one explicit
  transaction, mirroring `TransactionRepository`'s commit seam).
- `services/accounts.py` — `AccountService(vault)` (constructed like
  `TransactionService(vault)`) with `list_accounts`, `add_account`,
  `update_account`, `delete_account`; owns name/type validation
  (INV-2/INV-3) and the delete guard (INV-6, using
  `TransactionRepository.count_for_account`).
- `ui/accounts.py` — `AccountsWidget`: a list of the user's accounts
  (name + translated type), an add/edit form (name field + a type
  `QComboBox` whose items carry the token as `itemData` and show a
  `tr()` label), a Delete button (surfacing `AccountInUseError` and
  the last-account refusal as clear messages), and a **Done** signal
  to return to the main window. Strings via `self.tr`, layouts via Qt
  layout managers (RTL-safe).

## Data model (v2)

The v1→v2 migration produces:

- `accounts(id INTEGER PRIMARY KEY, name TEXT NOT NULL, type TEXT NOT
  NULL, created_at TEXT NOT NULL)` — one row per account; `type` is
  one of the seven `AccountType` tokens; uniqueness of `name` is
  enforced at the **service** layer (case-insensitive, INV-3), not by
  a DB `UNIQUE` constraint (a single enforcement point with a
  friendly message; coding.md § 2).
- `transactions(id INTEGER PRIMARY KEY, account_id INTEGER NOT NULL
  REFERENCES accounts(id), occurred_on TEXT NOT NULL, amount_minor
  INTEGER NOT NULL, description TEXT NOT NULL, created_at TEXT NOT
  NULL)` — the FIBR-0004 columns **plus** the required `account_id`
  foreign key (rebuilt per D3). The repository `SELECT` and the
  `Transaction` dataclass fields share this exact column order —
  `id, account_id, occurred_on, amount_minor, description, created_at`
  — so `Transaction(*row)` stays aligned (`id` and `account_id` are
  adjacent `int` columns, so swapping their order would compile but
  silently corrupt data — hence the fixed order).
- `schema_version` — the single row now holds `2`.
- `settings` — unchanged (`base_currency`, `minor_unit_exponent`).

Forward-only: a later phase adds v3+ as a new migration step; this
one is never reversed.

## Out of scope (deferred to named phases)

- **Editing an existing transaction's account** (D9) — arrives with
  transaction-editing in a later phase. The account is set at add
  time only.
- **Reassigning / bulk-moving transactions between accounts, and
  delete-time reassignment** — not built; deletion is block-only
  (D6). An account that has ever held transactions cannot be emptied
  this phase.
- **Per-account opening balance, currency, or institution metadata,
  account archiving/hiding, and account ordering by the user** — the
  model is name + type only; richer account attributes are a later
  enhancement (not currently on the roadmap; capture if requested).
- **The category tree (FIBR-0006), import (FIBR-0007), and the later

  *consume* the account link this phase establishes.
- **Multi-currency per account** — every account shares the vault's
  single `base_currency` (FIBR-0004 D1); per-account currency is not
  modelled here.
- **A general migration *framework* beyond forward-only stepping** —
  no down-migrations, no out-of-process migration tool; just the
  in-process forward runner design.md specifies.

## Verified API basis

Pins the idioms the INVs assert against, so the contract cites real
APIs (global rule § 13). To be **re-verified empirically at
implementation time** against the pinned `sqlcipher3-binary` 0.6.0
and PySide6 6.11.1. Two legs especially need the empirical check: (i)
the transaction-control mechanism — that an explicit `BEGIN` can be the
step's first statement under `isolation_level=""` (D2); and (ii) that
the **FK-ON table rebuild** (`DROP`+`RENAME` with nothing referencing
`transactions`) does not trip `foreign_key` enforcement under 0.6.0
(D4 keeps FK ON through the rebuild rather than toggling it off).

- **SQLite schema change** — a `NOT NULL` + `REFERENCES` column is
  added by **table rebuild** (`CREATE new` → `INSERT … SELECT` →
  `DROP old` → `ALTER … RENAME`), per the SQLite docs "Making Other
  Kinds Of Table Schema Changes"; `ALTER TABLE … ADD COLUMN` cannot
  express `NOT NULL` + `REFERENCES` together. SQLite DDL is
  rollback-able, so the rebuild is atomic **when wrapped in the
  runner's explicit `BEGIN … COMMIT`/`ROLLBACK`** (D2 — not from any
  implicit driver transaction, which does not fire for DDL under
  `isolation_level=""`) (INV-4).
- **`PRAGMA foreign_keys`** — off by default in SQLite; set `ON`
  **per connection, outside a transaction** (a *change* is a no-op
  mid-transaction, but an already-ON value stays enforced), to enforce
  `REFERENCES` (D4).
- **`sqlcipher3-binary` 0.6.0** — DBAPI over SQLite; raw-key open
  unchanged from FIBR-0004; `IntegrityError` is the FK-violation
  exception (subclass of `DatabaseError`); with `isolation_level=""`
  the driver does **not** implicitly `BEGIN` around DDL, so the
  migration wraps its step in an **explicit `BEGIN … COMMIT`** (D2) —
  confirmed empirically at implementation time, not left open.
- **`enum.StrEnum`** (stdlib, 3.11+) — `class AccountType(StrEnum)`
  gives string-valued members whose `.value` is the stored token;
  `AccountType(token)` validates membership (raises `ValueError` on an
  unknown token — the INV-2 mechanism). The modern idiom over the older
  `class AccountType(str, Enum)` (coding.md § 1.5).
- **PySide6 6.11.1** — `QComboBox.addItem(label, userData=token)` +
  `currentData()` to carry the stored token behind a `tr()` label;
  standard `self.tr(...)` + layout-manager idioms (coding.md § 5.2).
