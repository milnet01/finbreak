**Theme:** FIBR-0004 shipped a **fixed** idle auto-lock — `AUTO_LOCK_MINUTES = 10`,
a module constant in `services/auth.py`, whose own comment says *"user
configurability land[s] with … the Settings screen."* This phase delivers that
Settings screen. Its **priority control** is a **user-configurable auto-lock
timeout**: the user picks how many minutes of being open the vault waits before
it auto-locks, the change is **applied live** (the running timer re-arms) and
**persisted** (survives a restart). The screen also surfaces the vault's **base
currency read-only** as honest "core preferences" context. Everything else the
roadmap bullet floated (theme toggle, stored-PDF-password management, a
confirm-before-delete toggle, a startup-tab preference) is **explicitly out of
scope** for this cut (Out of scope) — each is its own mini-feature and the
priority is the timeout.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **The auto-lock timeout is read from the vault, with a safe default (guarded parse).** `AuthService.auto_lock_minutes() -> int` reads the value under the `settings` key `auto_lock_minutes` (via `SettingsRepository.get`, which returns `None` for an absent key) and returns it as an int **only if** the value is present **and** `int()` parses it **and** the result is in `ALLOWED_AUTO_LOCK_MINUTES`; on an **absent** key (`get` → `None`, a fresh or pre-FIBR-0055 vault — the `None` is treated as absent, never passed to `int()`), a **non-integer** stored value (`int()` raises `ValueError`), or an **out-of-set** value, it returns `DEFAULT_AUTO_LOCK_MINUTES` (= 10, the FIBR-0004 value). The guarded parse is a **deliberate defensive fallback to the secure default** for a corrupt / hand-edited persisted value — not a global-rule-§1 silent swallow: the named constraint is "a stored timeout that isn't one of the offered choices must not weaken or crash the lock; fall back to the 10-minute default." A test asserts: absent → 10; `"5"` → 5; `"abc"` → 10; `"7"` (a valid int, not in the set) → 10. The malformed / out-of-set legs **seed the value via `SettingsRepository.set` (the generic key→value writer, which does no validation) or raw SQL** — bypassing `set_auto_lock_minutes`, whose own validation (INV-2) would refuse them | FIBR-0004 (fixed 10-min lock) |
| INV-2 | **Setting the timeout validates, persists, then re-arms — in that order.** `AuthService.set_auto_lock_minutes(n)` runs three steps in order: (1) **validate** — reject `n ∉ ALLOWED_AUTO_LOCK_MINUTES` with `ValueError` **before any write** (nothing persisted, timer untouched); (2) **persist** — UPSERT `settings['auto_lock_minutes'] = str(n)` and `commit()`; (3) **re-arm** — `self._arm_timer()`. Re-arm is deliberately **last** and best-effort: if it no-ops (headless, `QCoreApplication.instance() is None`, INV-3) the value is already durably persisted and takes effect on the next arm (next unlock). The `commit()` is **not** wrapped in a swallow, so a failure (e.g. disk error) propagates rather than silently leaving a stale value — a **code-posture note** per global rule § 1, not a separately-named test leg. Tests: an invalid `n` raises `ValueError` and leaves both the stored value and the timer unchanged; a valid `n` persists (read back == n) and calls `_arm_timer` — asserted via a **monkeypatch spy** on `_arm_timer` (called / not-called), decoupled from an event loop, since a headless service test has no live `QTimer` to inspect (INV-3's live-`qapp` interval leg is the one exception) | FIBR-0004 atomic-commit posture (by analogy; INV-4a itself is money-transaction-scoped); security-model (auto-lock bound) |
| INV-3 | **The persisted timeout drives the live timer on every arm.** `AuthService._arm_timer` starts the `QTimer` for `auto_lock_minutes() * 60_000` ms (not the old constant), so the configured value takes effect on **first-run, unlock, and every re-arm** (a `set_` call). A test drives `_arm_timer` after `set_auto_lock_minutes(1)` and asserts the timer interval is `60_000` ms | FIBR-0004 (idle timer) |
| INV-4 | **The setting persists across a lock/unlock cycle (and a real restart).** Written inside the encrypted vault `settings` table, it survives lock → unlock — `Vault.close()` then `open()` re-reads it on a **fresh SQLite connection** — which is the test's durability proxy; the same files re-opened by a **fresh `AuthService`** (a true restart) read identically. A test sets a value, locks, unlocks on the same `AuthService`, and asserts `auto_lock_minutes()` still returns it; an optional second leg re-opens the same paths with a new `AuthService` | FIBR-0004 INV-3; ADR-0003 |
| INV-5 | **No schema change; existing vaults keep working.** The `settings` table exists since v1 (FIBR-0004); this phase adds **no** migration and **no** schema-version bump. A vault created before FIBR-0055 (no `auto_lock_minutes` row) reads the 10-minute default (INV-1) and gains the row only when the user changes the timeout. A test uses a **fresh `first_run` vault** (which has no `auto_lock_minutes` row — no special fixture needed, since D2 adds no migration) and asserts it reports 10 | FIBR-0005 (migration runner, untouched); FIBR-0004 settings table |
| INV-6 | **(Security.) The Settings screen is vault-dependent chrome.** The `Settings…` action (`action_settings`) lives in the **File** menu, which `_set_vault_chrome_enabled(False)` **disables on lock** (alongside the View menu + toolbar) and re-enables on unlock — so Settings is unreachable while locked, and its writes never hit a closed connection. The enforcement is **menu-level**: `setEnabled(False)` on the **`_menu_file` `QMenu`** makes it unopenable (its child `action_settings` is therefore unreachable) — note Qt does **not** propagate a menu's disabled state to a child `QAction.isEnabled()`, so the test asserts at the menu, not the action. A `qtbot` test asserts `window._menu_file.isEnabled()` is **False** on the locked shell and **True** after unlock (mirroring the house `test_INV6c` menu-level idiom), and that `action_settings` is **in `_menu_file.actions()`** (Qt's `addAction` lists the action on the menu without reparenting it — its QObject parent stays the window, so `_menu_file.findChild(QAction, …)` would wrongly return `None`) | FIBR-0051 INV-4 (lock disables the vault-dependent chrome: toolbar + File/View; carried unchanged into FIBR-0052 INV-4) |
| INV-7 | **(Security.) An idle auto-lock while Settings is open destroys the dialog, so no Save reaches a closed vault.** `SettingsDialog` is opened as the shell's **tracked application-modal** (`_open_dialog`, `setModal(True)`, `self._dialog`). Because it is modal, the **manual** `Lock` action is **unreachable while it is open** (`self._dialog is None` never holds there, so the lock path's step-1 teardown is a no-op on that path — exactly as FIBR-0051 INV-4 establishes for `ManualEntryDialog`). The reachable path is the **idle auto-lock**: the timer fires regardless of dialog modality, so `AuthService._on_idle_timeout` closes the vault and **then** calls `on_auto_lock` → `MainWindow._lock`, whose step-1 `_teardown_dialog()` (FIBR-0051 INV-4b) hides + `deleteLater`s the open dialog — the user can no longer issue a Save (the dialog is gone), so no Settings write reaches the (now closed) connection. **Defence in depth:** `set_auto_lock_minutes` on a locked vault raises `VaultLockedError` from `Vault.connection`. Two tests: a `qtbot` test opens Settings, **captures the dialog reference**, then drives the auto-lock path (`service._on_idle_timeout()`, which invokes `on_auto_lock` = `_lock`) and asserts the **captured `SettingsDialog` is destroyed** (`shiboken6.isValid(dialog)` is `False` after a `DeferredDelete` flush) — note `_lock` step 5 **re-opens the `UnlockDialog` into `self._dialog`**, so the assertion is on the captured ref, **not** `self._dialog is None`; a service-level test locks the vault then asserts `set_auto_lock_minutes(<a VALID n>)` raises `VaultLockedError` (a valid `n` so validation passes and the closed connection is actually reached) | FIBR-0051 INV-4/INV-4b (modal ⇒ manual Lock unreachable; dialog closed in the auto-lock sequence) |
| INV-8 | **The dialog reflects the current values and offers only allowed choices.** On open, `SettingsDialog` **preselects** the current `auto_lock_minutes()` in a `QComboBox` populated **only** from `ALLOWED_AUTO_LOCK_MINUTES` (each item's `userData` is the **int** minutes, so `currentData()` returns an int), and shows the **base currency read-only** (a non-editable `QLabel`, value **passed in by the shell** as a `str` — the dialog takes **no** vault / `TransactionService` dependency; the shell reads `TransactionService.base_currency()` and passes it, D8/Deliverable 4). A `qtbot` test asserts the combo's `currentData()` equals the stored timeout, and that the currency widget is a read-only `QLabel` (`isinstance(widget, QLabel)`; the dialog exposes no editable field — `findChild(QLineEdit) is None`) showing the vault's currency | user-request-2026-07-09 |
| INV-9 | **Save applies + closes via the service; Cancel changes nothing.** On **Save**, the dialog calls `service.set_auto_lock_minutes(chosen)` and emits `saved`; the shell tears the dialog down and reports `tr("Settings saved")`. On **Cancel** (`rejected`) the dialog closes and **nothing** is written or re-armed. A `qtbot` test: Save with a new value → stored value changes + `saved` emitted; Cancel after changing the combo → stored value unchanged | user-request-2026-07-09; FIBR-0051 (tracked dialog) |
| INV-10 | **(i18n.) Every visible string is translatable; the new widget is RTL-safe.** All `SettingsDialog` labels/buttons/window-title use `self.tr("literal")`; the minute-choice labels use a **fixed per-value `tr()` literal** (`{1: self.tr("1 minute"), 5: self.tr("5 minutes"), …}`), never `self.tr(variable)` (which `lupdate` cannot extract — FIBR-0051 D8/INV-10 note); the shell's new `Settings…` action text is a `tr()` literal too. **Two enforcement legs, stated honestly (mirroring FIBR-0051 INV-10):** (a) a **runnable** check — `ui/settings.py` is covered by the existing `test_INV10_no_fixed_geometry_in_new_ui` source-scan (it globs **all** `ui/*.py` for `\.(setGeometry|move|resize)\(\s*\d`), so use layout managers with **no fixed geometry** and the shell mirrors for RTL from `layoutDirection`; (b) a **review-checklist item, not a unit test** — the `tr()`-wrapping of the literals is enforced by `lupdate` catalog coverage + review per `coding.md § 5.2`, **not** by a grep (a runtime build cannot distinguish `QLabel("Foo")` from `self.tr("Foo")` with no catalog loaded) | `coding.md § 5.2`; FIBR-0051 D8/INV-10 |

## Context

`services/auth.py` today:

```python
# Fixed lock-out timeout measured from unlock — NOT reset on user activity.
# True idle-detection (activity-reset) + user configurability land with
# FIBR-0014's Settings screen.
AUTO_LOCK_MINUTES = 10
...
def _arm_timer(self) -> None:
    if QCoreApplication.instance() is None:
        return
    if self._timer is None:
        self._timer = QTimer()
        self._timer.setSingleShot(True)
        self._timer.timeout.connect(self._on_idle_timeout)
    self._timer.start(AUTO_LOCK_MINUTES * 60 * 1000)
```

`_arm_timer` is called from `complete_first_run` and `complete_unlock`. The timer
is **single-shot, measured from unlock, NOT reset on activity** — FIBR-0055 keeps
that semantics unchanged (D5); it only makes the *duration* configurable. The
vault `settings` table (`key TEXT PRIMARY KEY, value TEXT NOT NULL`) is created at
v1 and already holds `base_currency` + `minor_unit_exponent`; reads elsewhere use
`SELECT value FROM settings WHERE key = '<literal>'` (a literal key, not a bound
`?` — `services/transactions.py`). The shell
opens its **tracked content** dialogs (e.g. `ManualEntryDialog`) via
`MainWindow._open_dialog(dialog, defer=False)` (sets `self._dialog`,
`setModal(True)`, `show()`; the parameter's default is `defer=True` — a deferred
first-event-loop-turn show used by the unlock dialog — but the content dialogs pass
`defer=False`); `_lock` step 1 tears the tracked
dialog down before the vault closes; `_set_vault_chrome_enabled(False)` disables
the File menu + View menu + toolbar while locked. `ManualEntryDialog(service: AuthService)` is
the template modal this dialog mirrors.

## Design decisions

- **D1 — Persist in the vault `settings` table, not the plaintext window INI.**
  The timeout is a security control and is only *needed* while unlocked (the timer
  only runs post-unlock), so it belongs with the encrypted vault, read on unlock —
  matching the roadmap bullet's lean. The plaintext `window.ini` (FIBR-0052 INV-5)
  is reserved for non-sensitive layout state that must be restorable *before*
  unlock; the timeout has no pre-unlock use.

- **D2 — No schema migration.** The `settings` table already exists (v1). Rather
  than seed `auto_lock_minutes` at `create()` + backfill existing vaults with a
  migration, treat **absence as the default** (INV-1) and **UPSERT on change**
  (INV-2). This is the surgical choice: no schema-version bump, no ripple across
  the ~six suites' `LATEST_SCHEMA_VERSION` asserts, and pre-FIBR-0055 vaults keep
  the current 10-minute behaviour until the user changes it.

- **D3 — A generic `SettingsRepository` (key→value) over the settings table.**
  With `base_currency`, `minor_unit_exponent`, and now `auto_lock_minutes`, the
  table has three call-sites (Rule of Three). Add `repositories/settings.py`
  `SettingsRepository(connection)` with `get(key) -> str | None` and
  `set(key, value) -> None` (UPSERT + commit) for the **new** setting.
  **Do not refactor** the two existing raw reads (`read_minor_unit_exponent`,
  `TransactionService.base_currency`) onto it in this phase — that is adjacent to
  the request, out of lane (global rule § 11); the new repo is the seam a later
  consolidation would use.

- **D4 — The timeout API lives on `AuthService`.** `AuthService` already owns both
  the vault (to read/write settings via `SettingsRepository(self._vault.connection)`)
  and the timer (to re-arm), so it is the natural home for `auto_lock_minutes()` /
  `set_auto_lock_minutes(n)`. `_arm_timer` reads through `auto_lock_minutes()`.
  Rename the module constant `AUTO_LOCK_MINUTES` → `DEFAULT_AUTO_LOCK_MINUTES`
  (an internal rename; **not referenced by any test** — verified) and add
  `ALLOWED_AUTO_LOCK_MINUTES = (1, 5, 10, 15, 30)`.

- **D5 — Keep the fixed-from-unlock semantics; re-arm resets the countdown.**
  This phase does **not** add activity-reset idle detection (that stays deferred to
  FIBR-0014). The timer remains single-shot measured from the last arm. A
  `set_auto_lock_minutes` therefore **restarts** the countdown from the moment of
  change — acceptable and sensible (the user just interacted with Settings), and
  documented so it is not read as a bug.

- **D6 — No "Never" option in this cut.** The bullet floated a "never auto-lock"
  choice behind a warning. It defeats the app's core security invariant (a
  decrypted financial vault must not sit open indefinitely), so it is **excluded**
  here: `ALLOWED_AUTO_LOCK_MINUTES` is a bounded set with a floor of **1 minute**.
  A warning-gated "Never" can be a later, explicitly-scoped follow-up. Surfaced,
  not silently dropped. **`DEFAULT_AUTO_LOCK_MINUTES` is itself a member of
  `ALLOWED_AUTO_LOCK_MINUTES`** (10 ∈ the set) so the default always resolves to a
  valid combo item to preselect (INV-8) — a `tests`-side assertion pins this.

- **D7 — Settings surface is a modal dialog, opened from the File menu.** A modal
  `QDialog` (not a workspace tab): Settings is a configure-and-dismiss interaction,
  not a persistent content view, and the modal path already has the auto-lock
  teardown wiring (INV-7). It is reached from **File → Settings…** (not a new
  top-level menu for one item, not the toolbar — no new SVG asset / PyInstaller
  add-data / self-test icon leg needed this phase). File-menu placement also makes
  it **vault-dependent chrome for free** (INV-6).

- **D8 — Base currency is shown read-only.** The value already exists
  (`TransactionService.base_currency()`); displaying it is free context. Making it
  *editable* is out of scope — changing the base currency changes the minor-unit
  exponent and would need a data-rescale migration (a separate, larger feature).

## Data model (no schema change)

The `settings` table is unchanged (`key TEXT PRIMARY KEY, value TEXT NOT NULL`,
v1). One new **row** is written on demand:

| key | value | written by | read by |
|-----|-------|-----------|---------|
| `auto_lock_minutes` | the chosen minutes as a decimal string (e.g. `"5"`) | `SettingsRepository.set` via `AuthService.set_auto_lock_minutes` | `AuthService.auto_lock_minutes` |

Absent ⇒ `DEFAULT_AUTO_LOCK_MINUTES` (10). No `schema_version` bump; no migration.

## Verified API basis

- `services/auth.py` — `AUTO_LOCK_MINUTES = 10` constant; `_arm_timer` starts the

- `services/transactions.py` — `TransactionService(vault).base_currency()` reads
