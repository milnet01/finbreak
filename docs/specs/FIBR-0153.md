# FIBR-0153 — Currency symbol in the Amount column + a dedicated Currency column

> **Status:** Draft — contract for FIBR-0153 (Kind: fix, lanes: ui).
> Source: user-request-2026-07-19. Depends on the shared money
> formatter `_format_amount` (`src/finbreak/ui/_amount.py`) and the
> supported-currency set `CURRENCY_EXPONENTS` (`src/finbreak/services/auth.py:48`).
> Cold-eyes: see § 6 (project cadence `--max-loops 7`).

## 1. Problem

Money renders as **`ZAR1,234.49`** — hard to read. Two causes, one
site: `_format_amount` passes the caller's `symbol` argument straight
into `QLocale().toCurrencyString(value, symbol, decimals)`
(`ui/_amount.py:38`), and every caller passes `base_currency()`, which
returns the **ISO code** (`"ZAR"`), not a symbol
(`services/transactions.py:97-101`). Qt jams the code onto the digits
with **no space**. The user asked for two things (2026-07-19):

1. **Show the amount with the proper symbol + a space** — `R 1,234.49`.
2. **Move the ISO code into its own dedicated "Currency" column** on the
   Transactions table.

The base currency is a **single fixed value per vault**, chosen at
first-run and not editable afterwards (`vault.py:113-114`,
`ui/settings.py:146`) — so every transaction row carries the *same*
code. A per-row Currency column is therefore uniform today; the user
requested it explicitly for tidy separation and forward-compatibility
with future multi-currency. **Product note (not a blocker):** because
the column repeats one value, an equally-valid lighter design is a
currency-in-header (`Amount (ZAR)`); we implement the **explicit
column** as requested and record this observation only.

## 2. Scope

**In scope**
- Resolve the base-currency ISO code → a display **symbol** via a single
  small map; render the Amount as `«symbol»␣«grouped-magnitude»` with
  exactly one U+0020 space.
- Add a **Currency** column to the Transactions table holding the ISO
  code (`ZAR`).
- The one-line `_format_amount` change fixes **every** money site at
  once (Transactions, Home tiles, PDF export) because all pass the ISO
  code through the same formatter — audited in § 3.4.

**Out of scope**
- Charts render **no** currency symbol (`ui/charts.py:183-185` — a bare
  `QValueAxis`), so nothing changes there.
- The Settings dialog's read-only base-currency **label**
  (`ui/settings.py:146`) intentionally shows the raw code — unchanged.
- Making the base currency editable, or any multi-currency data model —
  unchanged. The Currency column is display-only.
- The Statements table (also 5-column, shares layout keys) renders no
  amounts (`ui/statements.py:10`) — untouched.

## 3. Design

### 3.1 Single-homed symbol map

Add, **beside** `CURRENCY_EXPONENTS` in `services/auth.py` (one home for
"which currencies we support and their properties"):

```python
CURRENCY_SYMBOLS: dict[str, str] = {
    "ZAR": "R", "USD": "$", "EUR": "€",
    "GBP": "£", "AUD": "A$", "CAD": "C$",
}
```

Keys are **exactly** the `CURRENCY_EXPONENTS` keys (the validated set,
`auth.py:179-180`) — the two dicts describe the same 6 currencies.
`USD`/`AUD`/`CAD` all use `$`; the dollar variants take `A$`/`C$` so
they stay unambiguous even without reading the Currency column.

### 3.2 `_format_amount` composes symbol + space explicitly

`_format_amount(display, symbol, negative_style)` keeps its signature —
callers still pass the ISO code as `symbol`. Internally:

1. `sym = CURRENCY_SYMBOLS.get(symbol, symbol)` — **unknown code falls
   back to the code itself**, so an unmapped currency degrades to
   today's behavior (`R`→`R`, `XYZ`→`XYZ`), never crashes.
2. `magnitude = QLocale().toString(float(abs(display)), "f", decimals)`
   — the locale-grouped number with **no** currency symbol at all,
   keeping grouping / decimal separator / decimal places locale-correct.
   **Do NOT use `toCurrencyString(v, "", decimals)`** to get a bare
   number: an empty-but-non-null symbol does **not** suppress the symbol —
   Qt substitutes the **ISO currency code**, so under any non-C locale it
   returns e.g. `ZAR1 234,50` (en_ZA), `USD1,234.50` (en_US),
   `1.234,50 EUR` (de_DE); the composed body would then be
   `"$ USD1,234.50"` — a double currency indicator (verified empirically
   against PySide6 6.11.1; the C locale is the *only* one where the ISO
   code is empty, which is why it deceptively looks correct there).
   `QLocale().toString(x, "f", d)` returns a symbol-free grouped number in
   **every** locale (`1234.50` / `1 234,50` / `1,234.50` / `1.234,50`) —
   confirmed. INV-3 is its proof and MUST run under a **non-C** locale so
   it can actually catch an ISO-code leak.
3. `body = f"{sym} {magnitude}"` — symbol, one plain space, magnitude.
   The symbol is **always a prefix** with one space, by our composition —
   we deliberately override QLocale's per-locale symbol *placement* (some
   locales suffix, e.g. EUR as `1 234,56 €`) to honour the user's
   `R 1,234.49` request. Grouping/decimals stay from QLocale; only symbol
   placement is fixed by us. This is intentional, not a locale bug.
4. Sign notation wraps the **whole** body (unchanged FIBR-0105 rule):
   `f"({body})"` for brackets, `f"-{body}"` for minus, else `body`.

The `float()` stays a display-only bounded conversion — storage /
computation remain exact `Decimal` (D1). **Home of the map (decided, not
deferred):** `CURRENCY_SYMBOLS` lives in `services/auth.py` beside
`CURRENCY_EXPONENTS` (§ 3.1, single home — INV-8), and `ui/_amount.py`
imports it. This adds a `ui → services.auth` import; it is a **clean
one-way edge** (`services/auth.py` imports no `ui` module, so there is no
circular risk) and consistent with the layered architecture — other `ui`
modules already depend on `services`. Do **not** inline a second copy of
the map in `_amount.py` (that would violate INV-8's single-map claim).

### 3.3 Currency column on the Transactions table

`ui/transactions.py`:
- Column constants become `_COL_DATE=0`, `_COL_AMOUNT=1`,
  **`_COL_CURRENCY=2`**, `_COL_DESCRIPTION=3`, `_COL_ACCOUNT=4`,
  `_COL_CATEGORY=5` (was 0..4). Currency sits **adjacent to Amount**
  (natural reading: number, then its currency).
- `QTableWidget(0, 5)` → `QTableWidget(0, 6)` (`:163`); header list
  (`setHorizontalHeaderLabels`, `:169`) inserts `self.tr("Currency")`
  after `self.tr("Amount")`.
- Populate the currency cell with the ISO code (`symbol` from
  `base_currency()`, `:270`) as a plain non-editable `QTableWidgetItem`,
  same item flags as the Account cell.
- `_on_copy_amount` still reads `_COL_AMOUNT` (value unchanged at `1`),
  so "Copy amount" now copies `R 1,234.49` with no rework (FIBR-0032
  reads rendered cell text).

**Persisted-layout reset on upgrade (one-time, safe).** The table's
column widths / order / sort are persisted via `remember_columns` →
`QHeaderView.restoreState()`, keyed by `objectName` in the window INI
(`ui/_table_state.py:122-127`). Qt's `restoreState` **rejects** a saved
blob whose section count differs from the live header, so an existing
user's saved **5-column** layout is discarded (a no-op) the first time
the **6-column** table loads, and the table falls back to default widths
/ order. This is a deliberate, acceptable one-time reset — **not** a
crash or data loss (INV-9). We do not migrate the old blob.

### 3.4 App-wide consistency (via 3.2, no per-site edits)

All confirmed to route through `_format_amount` with the ISO code —
each passes a local `symbol` (or `home.py`'s cached `self._drill_symbol`,
`:551`) derived from `base_currency()`, not a literal call — so the
§ 3.2 change fixes each with **zero** extra edits:
- Transactions Amount cell (`transactions.py:285`).
- Home tiles: net strip, column totals, recurring, drill-tree
  (`home.py:408,426,497,549-551`).
- PDF export: summary, per-account, per-transaction
  (`pdf_export.py:295-297,317-319,388`).

The Currency **column** is Transactions-only (the user's ask); Home and
PDF gain only the symbol fix.

## 4. Invariants

- **INV-1 — Symbol, not code, in the amount.** A formatted amount begins
  with the base currency's display symbol (`R` for `ZAR`), never the ISO
  code, for every currency in `CURRENCY_SYMBOLS`. *Test:*
  `_format_amount(Decimal("1234.49"), "ZAR")` starts with `"R"`, and
  `"ZAR"` does not appear in it.
- **INV-2 — Exactly one space between symbol and magnitude.** The string
  contains `"«sym» «digit»"` — one U+0020, no more, no fewer. This space
  is **composed by us** (`f"{sym} {magnitude}"`), so it is present in
  every locale regardless of QLocale's own symbol spacing. *Test:* regex
  `^R 1` matches; `"R  "`/`"R1"` do not.
- **INV-3 — Magnitude is locale-grouped, Decimal-exact, and carries NO
  ISO code.** Grouping and decimal places match the amount's own exponent
  under the runtime locale; storage/compute never see the `float`; and no
  currency code/symbol other than the mapped one appears — the whole
  point of using `toString`, not `toCurrencyString`, for the magnitude.
  **The test MUST pin a non-C locale** (the C locale is the one place the
  broken `toCurrencyString(v, "")` path also looks clean, so a C-only
  test cannot falsify an ISO-code leak). *Test:* under a pinned **en_US**
  locale, `_format_amount(Decimal("1234.49"), "ZAR")` == `"R 1,234.49"`
  **exactly**, and neither `"ZAR"` nor `"USD"` appears in it;
  `Decimal("1234")` at 2 dp → `"R 1,234.00"`.
- **INV-4 — Sign notation wraps the whole body.** Negative minus →
  `"-R 1,234.49"`; brackets → `"(R 1,234.49)"` (symbol inside the sign
  wrap, as today). *Test:* both styles, both asserted.
- **INV-5 — Unknown code degrades, never crashes.** An ISO code absent
  from `CURRENCY_SYMBOLS` falls back to the code itself:
  `_format_amount(Decimal("1"), "XYZ")` → starts `"XYZ "`. *Test:* asserts
  the fallback string, no exception.
- **INV-6 — Currency column shows the code.** The Transactions table has
  a `Currency` column at `_COL_CURRENCY` whose every populated cell holds
  the base-currency ISO code. *Test:* seed 2 txns; both currency cells
  read `base_currency()`.
- **INV-7 — Column shape is 6 and indices are consistent.** The table
  reports `columnCount() == 6`; the Amount cell is at `_COL_AMOUNT`
  (`R …`) and the Category cell at `_COL_CATEGORY` (=5). *Test:* asserts
  count + that a known category lands in column 5, not 4.
- **INV-8 — Single map home.** `CURRENCY_SYMBOLS` keys equal
  `CURRENCY_EXPONENTS` keys (no currency supported by one and not the
  other). *Test:* `set(CURRENCY_SYMBOLS) == set(CURRENCY_EXPONENTS)`.
- **INV-9 — Stale saved layout is safe on the column-count change.**
  Loading the 6-column table with a **5-column** saved header blob in the
  window INI does not crash — `restoreState` rejects the mismatch and the
  table falls back to defaults (§ 3.3). *Test:* write a 5-section header
  `saveState()` blob under the table's INI key, construct the (6-column)
  view, assert no exception and `columnCount() == 6`.

## 5. Exit criteria (falsifiable)

1. **Symbol + space, app-wide, no ISO code.** Under a pinned **en_US**
   locale (non-C, so the test can catch an ISO-code leak),
   `_format_amount(Decimal("1234.49"), "ZAR")` returns `"R 1,234.49"`
   exactly (INV-1/2/3). Falsifier: any `"ZAR"`/`"USD"` substring, a
   missing space, or a wrong-grouped magnitude.
2. **Sign styles preserved.** Minus and brackets each wrap `R …` (INV-4).
   Falsifier: symbol outside the brackets, or a lost sign.
3. **Currency column present + correct.** Transactions table is 6
   columns; the Currency column at index 2 shows `ZAR` on every row; the
   Category cell is at index 5 (INV-6/7). Falsifier: 5 columns, blank
   currency cell, or category read from the old index 4.
4. **Copy-amount follows the new format.** With FIBR-0032, "Copy amount"
   copies `R 1,234.49` (reads `_COL_AMOUNT` text). Falsifier: copies
   `ZAR1,234.49` or a raw number.
5. **Stale saved layout is safe.** A 5-section saved header blob in the
   window INI loads into the 6-column table without crashing; the table
   ends up 6 columns (INV-9). Falsifier: an exception, or the old blob
   forcing a 5-column render.
6. **Map is single-homed and complete.** `set(CURRENCY_SYMBOLS) ==
   set(CURRENCY_EXPONENTS)` (INV-8); unknown code degrades (INV-5).
   Falsifier: a supported currency with no symbol, or a crash on an
   unknown code.
7. **Gate green.** `./scripts/ci-local.sh` passes (ruff, format, bandit,
   pip-audit, gitleaks, pytest, mypy), including the updated formatter /
   `_COL_*` / column-count tests. Test files that genuinely break on the
   column shift and MUST be updated: `app_shell` (formatter contract),
   `transactions_tab`, `clipboard`, `category_library` (hard-coded
   `_COL_*`), and `pdf_export` / `dashboard_focus` (formatter-output
   asserts). `transfers` and `accounts` index a *different* table /
   iterate `range(columnCount())`, so they are column-shift-agnostic —
   re-run them as a precaution, no edit expected.
