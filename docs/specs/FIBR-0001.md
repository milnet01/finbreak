# FIBR-0001 — P01 Bootstrap: lint/format/test/security harness

> **Status:** Draft — awaiting sign-off, then the Phase D
> `/cold-eyes` loop.
> **Phase:** P01 — Bootstrap. **Kind:** chore. **Source:** planned.
> **Roadmap:** [ROADMAP.md → FIBR-0001](../../ROADMAP.md).
> **Governed by:** `docs/standards/{coding,testing,commits}.md`,
> `docs/security-model.md`.

**Theme:** stand up the build/dev plumbing and the **quality +
security gate** before any feature code exists, so the harness is
proven working and every later phase runs against it.

## Context

This is the first code phase. It ships **zero user-facing
features** on purpose. Its value is that the tooling that judges
all later work — linter, formatter, tests, and the
security scanners the user has made paramount — is known-good and
runs identically in CI and locally. Sibling P01 bullets
(`FIBR-0002` `.gitignore`/secret-leak guard, `FIBR-0003` bundling
smoke-test) are specced just-in-time when their turn comes.

## Invariants (testable)

> These `INV-N` are **local to this spec** (bootstrap-harness
> invariants). They are a separate numbering from the project-wide
> security invariants in `docs/security-model.md § 5`; where this
> spec cites a security invariant it names the doc explicitly
> (e.g. "security-model.md INV-6").

- **INV-1 — One command, all gates, locally.** Running
  `scripts/ci-local.sh` from a clean checkout runs, in order
  (cheapest-first; any order is fine provided all stages run and
  the script exits non-zero on any failure), `ruff check`, `ruff
  format --check`, `bandit`, `pip-audit`, `gitleaks`, then
  `pytest -m "not perf"` (testing.md § 3.4 permits excluding perf
  tests when noisy; this gate excludes them unconditionally.
  Integration tests **do** run in the gate). The script assumes
  the dev dependency group is already installed — the one-time
  `pip install` of the `dev` group is a documented setup step (CLAUDE.md
  "Build and test"), not a gate stage; CI performs that install before
  invoking the script. This is the P01 stage list; `FIBR-0003` later appends a
  build smoke-test stage to the same script. Source: discovery.md
  "Local CI emulation" (the lint/test spine) + security-model.md
  § 6 (the `bandit`/`pip-audit`/`gitleaks` security stages, which
  originate there, not in discovery's three-stage gate).
- **INV-2 — CI runs the identical gate list.**
  `.github/workflows/ci.yml` runs the same stages in the same
  order as `scripts/ci-local.sh`. The gate list has **one source
  of truth**: `ci.yml`'s only project-specific steps are to set up
  Python and install the dev dependency group; it then *invokes*
  `scripts/ci-local.sh` rather than re-listing the stages, so the
  two cannot drift. At P01 neither file contains a build stage
  (added to both via the script by `FIBR-0003`). Source:
  discovery.md.
- **INV-3 — Green on an empty project.** On the bootstrap commit
  (placeholder `src/` + empty test suite) every stage exits 0:
  `ruff` clean, `ruff format --check` clean, `bandit` 0 issues,
  `pip-audit` 0 known-vulnerable deps, `gitleaks` 0 leaks,
  `pytest` 0 collected/0 failed. (A later upstream CVE can
  legitimately flip the `pip-audit` stage red without any project
  change — that is the gate working, not a regression.) Source:
  app-workflow P01 definition.
- **INV-4 — Reproducible, pinned dev env.** `pyproject.toml`
  declares Python `>=3.12` and pins dev-tool versions **exactly**
  (`==`) for ruff, bandit, pip-audit, pytest, and pytest-qt — in a
  PEP 735 `[dependency-groups]` table (`dev` group) — so two
  machines resolve the same toolchain. (`pytest-qt` is pinned now
  but first used at P02 when GUI tests appear.) Refreshing pins to
  latest-stable is a periodic-sweep process rule (global rule
  § 5c), not a P01 exit check. The exact pins serve INV-4's
  reproducibility and do **not** conflict with the latest-stable
  posture (coding.md § 1.5 / global rule § 5) — the sweep refreshes
  the pins to current-stable. Source: INV-4 reproducibility (exact
  pins) + global rule § 5c (refresh cadence).
- **INV-5 — Security stage is non-bypassable in the gate.** A
  deliberately-planted insecure pattern (e.g. a hardcoded secret
  fixture) makes `bandit`/`gitleaks` fail the gate; removing it
  makes the gate pass. This proves the security stage is wired,
  not stubbed. Source: security-model.md T8 (the `bandit`
  insecure-pattern half) and INV-6 (no secret in the repo — the
  `gitleaks` half) + § 6 (the tooling table).
- **INV-6 — Single-test ergonomics.** A named single test can be
  run in isolation (`pytest -k <name>` / `pytest <path>::<test>`)
  and the command is documented in `CLAUDE.md` "Build and test".
  Source: testing.md § 3.2.

## Deliverables

- `pyproject.toml` — project metadata, `>=3.12`, dev-dependency
  group, `[tool.ruff]`, `[tool.pytest.ini_options]` (markers:
  `features`, `integration`, `perf`), `[tool.bandit]`.
- `src/finbreak/__init__.py` — placeholder package so the linter
  and importer have a target.
- `tests/` — empty but collectable suite (a trivial
  `test_smoke.py` asserting the package imports).
- `scripts/ci-local.sh` — the one-command local gate (INV-1).
- `.github/workflows/ci.yml` — CI mirror (INV-2); pinned
  `ubuntu-24.04` (not floating `ubuntu-latest`, per global rule
  § 5a) for the lint/test/security gate (the build matrix arrives
  with packaging in P13).
- `CLAUDE.md` "Build and test" + "Module map" sections filled in.

## Out of scope

- Any application feature (that's P02 onward).
- The **build smoke-test** stage of `ci-local.sh`. discovery.md's
  "Local CI emulation" lists a build smoke-test as part of the
  gate; that stage is added to the script in `FIBR-0003` (bundling
  smoke-test), not here. This bullet ships the
  lint/format/security/test stages only.
- The multi-OS build matrix and packaging (P13).
- Pre-commit hooks — the gate is the script + CI; a pre-commit
  hook can be added later if desired but is not required here.

## Exit criteria (Definition of Done)

1. `scripts/ci-local.sh` exits 0 on a clean checkout (INV-1,
   INV-3).
2. CI is green on the bootstrap commit and runs the same stages
   (INV-2).
3. The INV-5 planted-secret check demonstrably flips the gate
   red, then green when removed.
4. `/audit` and `/indie-review` return zero **actionable**
   findings — non-allowlisted, severity ≥ LOW per
   `docs/audit-allowlist.md` — on the same closing pass. At P01 the
   audit scope is all P01-authored files: `scripts/ci-local.sh`,
   `.github/workflows/ci.yml`, `pyproject.toml`, the placeholder
   `src/finbreak/` package, and `tests/` — since no application
   feature code exists yet.
5. User approves closing P01's first item.

## Dependencies

None — this is the root of the build DAG.
