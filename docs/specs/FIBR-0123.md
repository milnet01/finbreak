# FIBR-0123 тАФ Group category pickers by Income/Expenditure type (disambiguate same-named categories)

> **Status:** тЬЕ CLEARED FOR CODE тАФ `/cold-eyes` **converged at loop 7 (polish)**
> (project `--max-loops 7`). Loops 1тАУ6 each caught a distinct **mechanical** defect
> (inverted INV-3 test; two missing dialog construction sites; a `.arg()`/`%1` tag
> idiom that doesn't run in PySide6; a `.model().item()` idiom that fails the mypy
> gate; an order-brittle id-compare); loop 7 surfaced no mechanical defect (only
> path-prefix nits + two additive test legs) = polish convergence. See the loop log
> at the end. Ready for TDD (global rule ┬з 14).
> **Phase:** near-term dogfooding fix (surfaced 2026-07-13). The category tree has
> two Type roots тАФ **Income** and **Expenditure** (FIBR-0006) тАФ but the three
> *flat* pickers collapse both roots into one undifferentiated combo. So (1) you
> can't tell an income category from an expenditure one at pick time, and (2) two
> categories that share a name under different roots are indistinguishable тАФ the
> real case that surfaced this: a seeded income **Lottery** (winnings) and a
> user-added expenditure **Lottery** (tickets) render as two identical rows. The
> category *manager* already shows the tree grouped; this fixes only the flat
> picker combos. Known-deferred at FIBR-0010 (D13: the category
> selector is a flat combo; grouping the leaves visually under their root was
> deferred as UX polish).
> **Security:** none тАФ presentation-only over already-fetched categories; no
> network, no new stored data, no schema change.
> **Kind:** ux.

## Verified current state (2026-07-13)

- **Three flat category combos** (two picker dialogs тАФ the Set-category dialog and
  the Rules editor тАФ plus the transactions **filter**; "picker" is used below as the
  umbrella, "filter" when the transactions combo's present-only behaviour matters)
  flatten the tree today:
  - `ui/category_picker.py:37-39` тАФ `CategoryPickerDialog` adds `Uncategorised`
    (data `None`) then one flat row per `leaf.name` (data `leaf.id`).
  - `ui/rules.py:64-68` тАФ `RuleEditDialog` adds one flat row per `leaf.name`; no
    `Uncategorised` row. OK is gated by `_sync_ok` on
    `bool(self._pattern.text().strip()) and self._category.count() > 0`
    (`rules.py:92`) тАФ the leaf-existence half (`count() > 0`) is the part this
    feature touches; the non-empty-pattern half is untouched.
  - `ui/transactions.py:191-195` тАФ the filter adds `All categories` (`_FILTER_ALL`)
    + `Uncategorised` (`None`) then one flat row per present category from
    `_unique_categories()` (built from the transaction rows in `self._master`,
    which carry `category_id` + `name` but **not** the Type).
- **Type lives on the root only.** `Category` = `id, parent_id, name, kind,
  created_at` (`models.py:225-238`); `kind` (a `CategoryKind` token тАФ
  `income`/`expenditure`, `models.py:103-109`) is set on the two roots
  (`parent_id IS NULL`) and `None` on every descendant. So a category's Type is
  its **ancestor root's** `kind`, not a field on the category itself.
- **`CategorizationService.leaf_categories()`** (`services/categorization.py:76`)
  is the single definition of the assignable (non-root) set (FIBR-0010 INV-9 leaves-only; D13 names `leaf_categories()` the single definition):
  `[c for c in CategoryRepository(self._conn).list_all() if c.parent_id is not
  None]`. It is flat and carries no Type.
- **The tree is 2-level in practice, but not enforced at the data layer.**
  `CategoryService._require_parent` (`services/categories.py:105-115`) rejects a
  `None` parent (no third root) but accepts **any existing** parent тАФ it does
  **not** require the parent to be a root. The category add/edit paths
  (`ui/categories.py:218-225,116-119,129-143`) fill the Type combo with **root ids
  only**, so every user-added category is a direct child of a root. The grouping
  therefore resolves Type by **ascending `parent_id` to the root that carries a
  `kind`** (depth-safe), so a category is never dropped even if a deeper node ever
  exists (INV-2 / D3).
- **The Income/Expenditure label map is duplicated.** `ui/categories.py:59-62`
  builds `{INCOME.value: tr("Income"), EXPENDITURE.value: tr("Expenditure")}`
  inline. This feature lifts it into one shared helper so the pickers and the
  manager can't drift (INV-4).
- **`select_combo_data`** (`ui/_widgets.py:17`) preselects by `userData` and
  leaves the selection unchanged when nothing matches тАФ unaffected by header rows
  (headers carry no data), so prefill keeps working (INV-7).
- **QComboBox uses a `QStandardItemModel` by default** (no `setModel` on any of
  the three), so a header row is disabled via
  `cast(QStandardItemModel, combo.model()).item(i).setFlags(Qt.ItemFlag.NoItemFlags)`
  тАФ the standard Qt "non-selectable combo item" idiom. The `cast` is required for
  **mypy 0**: `QComboBox.model()` is typed `QAbstractItemModel`, which has no
  `.item()`. (If the pinned PySide6 stub types `.item()` as `QStandardItem | None`,
  add an `assert item is not None` before `.setFlags` so the chain stays mypy-clean
  тАФ confirm against the installed stub at implementation.) (D1; asserted by test,
  INV-3).

## Design decisions

- **D1 тАФ Grouped combo via disabled `QStandardItem` headers, not a tree popup.**
  Keep the pickers as `QComboBox`es and insert greyed, non-selectable section
  headers (`cast(QStandardItemModel, combo.model()).item(i).setFlags(
  Qt.ItemFlag.NoItemFlags)`, **bold** font тАФ the `cast` keeps mypy green) with
  the categories as ordinary rows beneath. This is the minimal change тАФ no new
  widget, the smallest caller/test churn тАФ and was the brainstorm choice over a
  `QTreeView` popup (heavier, larger surface). *(The manager keeps its own
  `QTreeWidget`; only the flat combos change.)*
- **D2 тАФ Type shown on the section header **and** on every row.** Per the
  brainstorm: the grey, disabled, bold **`Income`** / **`Expenditure`** header
  (its text is the plain `category_type_labels()` label тАФ the `тФАтФА тАж тФАтФА` dashes in
  the brainstorm mock were illustrative, not the shipped string) gives the
  open-dropdown grouping, and an inline `Name (Type)` tag on every row makes the
  Type visible in the **collapsed** box too (a combo shows only the row text when
  closed) тАФ so the same-named `Lottery` case is unambiguous in every state. The
  tag uses translate-then-`.format()` (the `.format()`-not-`.arg()` shape is
  exemplified by `ui/export_dialog.py:192-194`'s `self.tr("тАж{n}тАж").format(n=тАж)`;
  the free-function form here necessarily uses `QCoreApplication.translate`, as at
  `services/pdf_export.py:56`): `QCoreApplication.translate("CategoryTypeLabels", "{name}
  ({type})").format(name=<category name>, type=category_type_labels()[token])` тАФ
  **not** Qt's `%1`/`.arg()` placeholders (a PySide6 `translate()` returns a plain
  Python `str`, which has no `.arg()`). Both the labels and this tag use the one
  fixed `"CategoryTypeLabels"` context.
- **D3 тАФ Resolve Type by ascending `parent_id` to the root, not one hop.** The
  service layer allows a category under any existing parent (`_require_parent`),
  so the 2-level shape is a UI convention, not a data guarantee. Walking up to the
  root that carries a `kind` groups correctly at any depth and **never hides** a
  category (a one-hop `parent_id тЖТ root.kind` lookup would drop a deeper node by
  mapping to `None`). Cost is trivial (a `{id: Category}` map + a short ascent).
  The ascent always terminates at a `kind`-bearing root: the `parent_id` FK holds
  (`vault.py` opens with `foreign_keys = ON`) and only roots have `parent_id IS
  NULL`, so no category dangles below a missing parent.
- **D4 тАФ The dialogs take pre-grouped data, keeping the "dumb dialog" contract.**
  `CategoryPickerDialog` and `RuleEditDialog` already take a fetched
  `list[Category]`, not a service (FIBR-0010 D10 for `CategoryPickerDialog`, D13 for
  `RuleEditDialog`). They change to take
  `grouped: list[tuple[str, list[Category]]]` (the `leaf_categories_grouped()`
  shape); their call sites fetch that instead of `leaf_categories()`. The dialog
  stays presentation-only тАФ it can't re-derive Type from flat leaves (no `kind`
  on a leaf), so the grouping is done once, in the service.
- **D5 тАФ `RuleEditDialog` OK-gate moves to "a leaf is selected".** Only the
  category clause of `_sync_ok` changes тАФ from `count() > 0` to `currentData() is
  not None` тАФ so the **full predicate becomes** `bool(self._pattern.text().strip())
  and self._category.currentData() is not None`. The non-empty-**pattern** guard
  (the `_sync_ok` predicate, `rules.py:92`) is preserved. Header-safe (a header's
  `currentData()` is `None`) and preserves FIBR-0079: with zero leaves there are
  zero section headers, the combo is empty, `currentData()` is `None`, and OK
  stays disabled. The initial selection is set to the first **enabled** row so the
  dialog never opens resting on a header. Init order: populate via
  `add_grouped_categories` тЖТ prefill via `select_combo_data` (both dialogs) тЖТ in
  `RuleEditDialog` only, the existing `_sync_ok()` call, so the OK-gate reads a
  settled selection (`CategoryPickerDialog` has no `_sync_ok`).

## Deliverables

1. **`src/finbreak/services/categorization.py`** тАФ `leaf_categories_grouped(self)
   -> list[tuple[str, list[Category]]]`: from `CategoryRepository(self._conn).list_all()`,
   ascend each non-root category to its root's `kind` (D3), group into
   `income`-then-`expenditure` order, name-sort (casefold) within each, and omit a
   section with no categories. The tuple's first element is the **untranslated
   `CategoryKind.value` token** (`"income"` / `"expenditure"`), never a display
   label тАФ the service does **no** `tr()` (labels belong to `_widgets`, INV-4).
   Same underlying rows as `leaf_categories()` (INV-2/INV-9). Every ascent lands in
   one of exactly two buckets: FIBR-0006 fixes the tree at **two roots**
   (FIBR-0006 INV-5) seeded with `kind` `income` / `expenditure` (FIBR-0006 INV-2)
   (a third root is
   impossible тАФ `_require_parent` rejects a `None` parent), so there is **no
   third-kind / `kind`-less-root case** to drop; INV-2's "no category hidden" holds
   without a fallback branch.
