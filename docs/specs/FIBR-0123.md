# FIBR-0123 — Group category pickers by Income/Expenditure type

> **Status:** ✏️ DRAFT — pending `/cold-eyes` (project `--max-loops 7`). Not yet
> cleared for TDD.
> **Phase:** near-term dogfooding fix (surfaced 2026-07-13). The category tree has
> two Type roots — **Income** and **Expenditure** (FIBR-0006) — but the three
> *flat* pickers collapse both roots into one undifferentiated combo. So (1) you
> can't tell an income category from an expenditure one at pick time, and (2) two
> categories that share a name under different roots are indistinguishable — the
> real case that surfaced this: a seeded income **Lottery** (winnings) and a
> user-added expenditure **Lottery** (tickets) render as two identical rows. The
> category *manager* already shows the tree grouped; this fixes only the flat
> picker combos. Known-deferred at the FIBR-0010 close ("category selectors are
> flat combos, grouping deferred").
> **Security:** none — presentation-only over already-fetched categories; no
> network, no new stored data, no schema change.
> **Kind:** ux.

## Verified current state (2026-07-13)

- **Three flat pickers** flatten the tree today:
  - `ui/category_picker.py:37-39` — `CategoryPickerDialog` adds `Uncategorised`
    (data `None`) then one flat row per `leaf.name` (data `leaf.id`).
  - `ui/rules.py:64-68` — `RuleEditDialog` adds one flat row per `leaf.name`; no
    `Uncategorised` row. OK is gated by `_sync_ok` on
    `bool(self._pattern.text().strip()) and self._category.count() > 0`
    (`rules.py:92`) — the leaf-existence half (`count() > 0`) is the part this
    feature touches; the non-empty-pattern half is untouched.
  - `ui/transactions.py:191-195` — the filter adds `All categories` (`_FILTER_ALL`)
    + `Uncategorised` (`None`) then one flat row per present category from
    `_unique_categories()` (built from the transaction rows in `self._master`,
    which carry `category_id` + `name` but **not** the Type).
- **Type lives on the root only.** `Category` = `id, parent_id, name, kind,
  created_at` (`models.py:225-238`); `kind` (a `CategoryKind` token —
  `income`/`expenditure`, `models.py:103-109`) is set on the two roots
  (`parent_id IS NULL`) and `None` on every descendant. So a category's Type is
  its **ancestor root's** `kind`, not a field on the category itself.
- **`CategorizationService.leaf_categories()`** (`services/categorization.py:76`)
  is the single definition of the assignable (non-root) set (FIBR-0010 INV-9):
  `[c for c in CategoryRepository.list_all() if c.parent_id is not None]`. It is
  flat and carries no Type.
- **The tree is 2-level in practice, but not enforced at the data layer.**
  `CategoryService._require_parent` (`services/categories.py:105-115`) rejects a
  `None` parent (no third root) but accepts **any existing** parent — it does
  **not** require the parent to be a root. The only creation path
  (`ui/categories.py:218-225,116-119`) fills the Type combo with **root ids
  only**, so every user-added category is a direct child of a root. The grouping
  therefore resolves Type by **ascending `parent_id` to the root that carries a
  `kind`** (depth-safe), so a category is never dropped even if a deeper node ever
  exists (INV-2 / D3).
- **The Income/Expenditure label map is duplicated.** `ui/categories.py:59-62`
  builds `{INCOME.value: tr("Income"), EXPENDITURE.value: tr("Expenditure")}`
  inline. This feature lifts it into one shared helper so the pickers and the
  manager can't drift (INV-4).
- **`select_combo_data`** (`ui/_widgets.py:17`) preselects by `userData` and
  leaves the selection unchanged when nothing matches — unaffected by header rows
  (headers carry no data), so prefill keeps working (INV-7).
- **QComboBox uses a `QStandardItemModel` by default** (no `setModel` on any of
  the three), so a header row is disabled via
  `combo.model().item(i).setFlags(Qt.ItemFlag.NoItemFlags)` — the standard Qt
  "non-selectable combo item" idiom (D1; asserted by test, INV-3).

## Design decisions

- **D1 — Grouped combo via disabled `QStandardItem` headers, not a tree popup.**
  Keep the pickers as `QComboBox`es and insert greyed, non-selectable section
  headers (`model().item(i).setFlags(Qt.ItemFlag.NoItemFlags)`, bold font) with
  the categories as ordinary rows beneath. This is the minimal change — no new
  widget, the smallest caller/test churn — and was the brainstorm choice over a
  `QTreeView` popup (heavier, larger surface). *(The manager keeps its own
  `QTreeWidget`; only the flat combos change.)*
- **D2 — Type shown on the section header **and** on every row.** Per the
  brainstorm: the grey **── Income ──** / **── Expenditure ──** header gives the
  open-dropdown grouping, and an inline `Name (Type)` tag on every row makes the
  Type visible in the **collapsed** box too (a combo shows only the row text when
  closed) — so the same-named `Lottery` case is unambiguous in every state. The
  tag is one translatable format (`translate("CategoryPicker", "%1 (%2)")`) filled
  with the category name + the shared Type label.
- **D3 — Resolve Type by ascending `parent_id` to the root, not one hop.** The
  service layer allows a category under any existing parent (`_require_parent`),
  so the 2-level shape is a UI convention, not a data guarantee. Walking up to the
  root that carries a `kind` groups correctly at any depth and **never hides** a
  category (a one-hop `parent_id → root.kind` lookup would drop a deeper node by
  mapping to `None`). Cost is trivial (a `{id: Category}` map + a short ascent).
- **D4 — The dialogs take pre-grouped data, keeping the "dumb dialog" contract.**
  `CategoryPickerDialog` and `RuleEditDialog` already take a fetched
  `list[Category]`, not a service (FIBR-0010 D10). They change to take
  `grouped: list[tuple[str, list[Category]]]` (the `leaf_categories_grouped()`
  shape); their call sites fetch that instead of `leaf_categories()`. The dialog
  stays presentation-only — it can't re-derive Type from flat leaves (no `kind`
  on a leaf), so the grouping is done once, in the service.
- **D5 — `RuleEditDialog` OK-gate moves to "a leaf is selected".** Only the
  category clause of `_sync_ok` changes — from `count() > 0` to `currentData() is
  not None` — so the **full predicate becomes** `bool(self._pattern.text().strip())
  and self._category.currentData() is not None`. The non-empty-**pattern** guard
  (the dialog's whole purpose, `rules.py:8`) is preserved. Header-safe (a header's
  `currentData()` is `None`) and preserves FIBR-0079: with zero leaves there are
  zero section headers, the combo is empty, `currentData()` is `None`, and OK
  stays disabled. The initial selection is set to the first **enabled** row so the
  dialog never opens resting on a header.

## Deliverables

1. **`src/finbreak/services/categorization.py`** — `leaf_categories_grouped(self)
   -> list[tuple[str, list[Category]]]`: from `CategoryRepository.list_all()`,
   ascend each non-root category to its root's `kind` (D3), group into
   `income`-then-`expenditure` order, name-sort (casefold) within each, and omit a
   section with no categories. Same underlying rows as `leaf_categories()`
   (INV-2/INV-9).
2. **`src/finbreak/ui/_widgets.py`** — `category_type_labels() -> dict[str, str]`
   (the shared `{income: tr("Income"), expenditure: tr("Expenditure")}` map via
   `QCoreApplication.translate`) and `add_grouped_categories(combo: QComboBox,
   grouped: list[tuple[str, list[Category]]]) -> None` (append a disabled header
   per non-empty section + a `Name (Type)` row per category with the id as
   `userData`. **Selection algorithm:** a combo auto-rests on index 0, which may be
   a disabled header — so after populating, if the current index is not selectable
   (`flags()` lacks `ItemIsEnabled`), scan for and select the first `ItemIsEnabled`
   row; if a selectable row (e.g. the picker's `Uncategorised`) is already current,
   leave it — INV-1/INV-3).
3. **`src/finbreak/ui/category_picker.py`** — `CategoryPickerDialog` takes
   `grouped` (D4): keep `Uncategorised` (data `None`) at index 0 (its selectable
   index-0 presence is why the helper leaves the resting selection on it when
   `current_category_id` is `None`), then `add_grouped_categories(self._combo,
   grouped)`, then prefill via `select_combo_data`. `selected_category_id()`
   unchanged.
4. **`src/finbreak/ui/rules.py`** — `RuleEditDialog` takes `grouped` (D4):
   `add_grouped_categories(self._category, grouped)`, then prefill via
   `select_combo_data` (matching the picker); `_sync_ok`'s full predicate becomes
   `bool(pattern.strip()) and currentData() is not None` (D5). **Both** rule-dialog
   construction sites switch their `leaf_categories()` arg to
   `leaf_categories_grouped()`: `RulesWidget._on_add` (`rules.py:191`) and
   `RulesWidget._on_edit` (`rules.py:216`).
5. **`src/finbreak/ui/transactions.py`** — **two** construction sites pass
   `leaf_categories_grouped()`: `_on_set_category` (`transactions.py:289`) to
   `CategoryPickerDialog`, and `_maybe_offer_rule` (`transactions.py:306`) to
   `RuleEditDialog` (the learn-a-rule offer — easy to miss, breaks at runtime if
   left flat). The filter combo keeps `All categories` + `Uncategorised`, then
   renders the **present** categories grouped — intersect `leaf_categories_grouped()`
   with the present category ids (from `self._master`) and feed the filtered
   grouped list to `add_grouped_categories`, preserving the
   `_select_data(self._category, held_category)` restore (INV-5).
   `_unique_categories()` is replaced by the present-id filter. *(The intersection
   never drops a present category: a categorised row's `category_id` is always a
   current leaf — picks/rules target leaves only (INV-9), and `delete_category`
   clears its transactions via `clear_category_for` + re-file — so every present id
   is in the grouped set.)*
6. **`src/finbreak/ui/categories.py`** — replace the inline `_type_labels` dict
   with `category_type_labels()` (INV-4; no behaviour change to the manager).
7. **Tests** — `tests/features/categories/` (the `leaf_categories_grouped`
   service legs — INV-2, order/sort/empty-section — INV-1); a `_widgets` helper
   leg (header non-selectable + row data + tag text — INV-1/INV-3);
   `tests/features/transactions/` (the filter grouping + special rows + still
   filters — INV-5, and the `CategoryPickerDialog` prefill — INV-7);
   `tests/features/rules/` (`RuleEditDialog` grouped + OK-gate + prefill — INV-3/5/7).
   The FIBR-0051 geometry glob already covers INV-6.
8. **Docs** — a CHANGELOG **Changed** entry (picker grouping) in the current
   unreleased block; the journal at close.

  stored on the category.

  to a root; this feature only *reads* the tree for grouping.
