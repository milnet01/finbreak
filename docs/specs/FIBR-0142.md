**Theme:** finbreak learns to recognise your **regular** money. Today every
transaction is a one-off in the ledger. This adds a detector that notices when
the *same payee* is charged (or pays you) *the same sort of amount* on a
*regular rhythm* — a subscription, a debit order, insurance, a salary, a
standing deposit — and gathers those into one **recurring item** you can
**Confirm** (yes, that's a real standing arrangement) or **Dismiss** (no, ignore
it). A new **Recurring** tab lists what it found — merchant, in/out, how often,
the typical amount, when the next one is due, and how many times it has been
seen. Confirmed items later roll up into a Home dashboard card showing your
total monthly recurring in/out (that card is deferred to FIBR-0143). Nothing the
detector decides ever changes a stored amount — grouping only decides *which
recurring item a transaction belongs to*.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| **INV-1** | **Money-safety.** Every recurring item's representative `amount`, `monthly_equivalent`, and the dashboard `RecurringSummary` totals are derived from integer `amount_minor` (the representative is `statistics.median_low` of the real member magnitudes — an **actual member value**, integer, never an averaged/float midpoint; the monthly-equivalent by the per-cadence factor in D8); grouping (D3) and cadence classification (D6) only ever decide **which recurring item a transaction is placed under** — they can **never** change a stored amount. `to_display_decimal` is **exact** (`scaleb`, no rounding); the **one** new lossy step is the monthly-equivalent's `×52/12` (etc.) division quantized to the minor unit with `ROUND_HALF_EVEN` (D8) — FIBR-0012's tile totals are exact-integer, so there is no rounding precedent to inherit. `RecurringSummary` totals sum those already-quantized per-item `monthly_equivalent`s **on purpose**, so the per-item rows a user sees add up to the shown total (the FIBR-0138 drill "children sum to parent" property); the rounding magnitude is negligible far below finbreak's ~10^20-minor-unit range. | FIBR-0007; FIBR-0012 INV-13 |
| **INV-2** | **Pure, deterministic, clock-injected detection.** `detect_recurring(rows, today, exponent, excluded_ids)` is a pure function: no I/O, no `date.today()` inside (`today` is a parameter), no randomness. It is **decision-agnostic** — it returns *every* active, qualifying group (both directions); applying the user's Confirm/Dismiss decisions is the **service's** job (D9), not the detector's. The same `(rows, today, exponent, excluded_ids)` always yields the same list in the same **order** (D10 explicit sort key, uniform types — no mixed-type sort). | testing.md (deterministic tests); FIBR-0138 INV-7 |
| **INV-3** | **Confirmed transfers are excluded.** The detector's `excluded_ids` parameter carries every transaction id in `TransferDetectionService.confirmed_transfer_txn_ids()` (passed by `RecurringService`, D3); those rows are dropped before grouping, so a self-transfer between your own accounts is never mistaken for a subscription or standing income. An **un**confirmed pair is still ordinary money and remains eligible (the exclusion keys on *confirmed*, matching the dashboard's `_excluded()`). | FIBR-0011; FIBR-0012 INV-1 |
| **INV-4** | **Direction from sign; zero excluded.** A transaction with `amount_minor < 0` is **out**; `amount_minor > 0` is **in**; `amount_minor == 0` (no money moved, no determinable direction) is excluded from detection entirely — a defensive guard for the pure detector's synthetic inputs, since a stored row never has a zero amount (`parse_transaction` rejects it, `services/transactions.py:74`; can't-happen, coding.md § 2). A payee that both bills and refunds you forms **two** independent groups (one per direction). | FIBR-0007 |
| **INV-5** | **Grouping key reuses the FIBR-0138 cleanup, unchanged.** Transactions are grouped by `(direction, merchant_key)` where `merchant_key = normalise_text(merchant_name(description))` — the **same** pure, total primitives the drill-down uses. The cleanup is fuzzy by design and refined per release; because of INV-1 a mis-grouped payee is only cosmetic. A description that cleans to empty falls back to its trimmed raw text (never a blank key — stored descriptions are non-empty). | FIBR-0138 INV-5; coding.md § 1.3 |
| **INV-6** | **"Balanced" qualification is a fixed, tested rule.** A group qualifies as recurring iff **all** of: (a) **≥ 3** member transactions (a cheap early guard — logically subsumed by (c)'s ≥ 3 distinct dates, but it also counts same-day duplicates toward `occurrences`); (b) **amount stability** — every member magnitude within **±10%** of the group representative, tested **integer-exact** as `100 × max(abs(m − med)) ≤ 10 × med` where `med = statistics.median_low(magnitudes)` is an integer member value (no float ratio — D4); (c) **cadence regularity** — sorting members by date and discarding same-day/zero gaps (D6), there are **≥ 2** remaining (non-zero) gaps — i.e. **≥ 3 distinct charge dates** — and **all** fall in the **same** cadence band, which becomes the group's cadence (two data points aren't a cadence; a group left with fewer than two non-zero gaps does not qualify). A group failing any clause is not surfaced. The thresholds (`_MIN_OCCURRENCES = 3`, the ±10% band, the D6 bands) are named module constants, changed in **one** place. | roadmap B3 (Balanced) |
| **INV-7** | **Activeness filter — stale arrangements age out.** A qualifying group is surfaced only if it is **active**: `(today − last_seen).days ≤ _ACTIVE_GRACE × nominal_interval_days(cadence)` (`_ACTIVE_GRACE = 2` — one missed period tolerated; `last_seen` may be up to 2× nominal old, and since `next_expected = last_seen + 1 interval` an overdue item shows a next-due date at most ~1× the nominal interval in the past). A subscription cancelled long ago (no charge within 2× its cadence) drops off rather than lingering indefinitely. Applies to confirmed items too (detection is the single source of truth, D9). | roadmap B3; D7 |
| **INV-8** | **Decisions persist by key, not by transaction.** A user's Confirm/Dismiss is stored in `recurring_decisions` keyed on `(direction, merchant_key)` — **not** on transaction ids — because a recurring item's members grow with every import. A **dismissed** key never appears in Suggested or the dashboard, even as new charges arrive; a **confirmed** key appears in Confirmed (and the dashboard) for as long as it still detects as active; a key with no decision is **suggested**. `reset` removes the decision (back to suggested). | FIBR-0011 (transfer_pairs analogue) |
| **INV-9** | **The three lists are a clean partition.** For a given `(rows, today)`, every **active, qualifying** detected item is in exactly one of Suggested / Confirmed / (hidden) Dismissed, by its stored decision. An item that stops qualifying or goes inactive leaves all three lists (it simply isn't detected) — a confirmed-then-cancelled item disappears rather than showing stale; if it reappears it returns to Confirmed (its key decision survived). | INV-7; INV-8 |

## Context

finbreak already has two precedents this feature composes:

1. **FIBR-0011 transfer detection** — a `TransferDetectionService` produces
   candidate pairs, the user confirms/rejects, and each decision is one row in a
   `transfer_pairs` table (`status` = `'confirmed'`/`'rejected'`, both FKs
   `ON DELETE CASCADE`, `UNIQUE` on the ordered pair). The 7th **Transfers** tab
   shows Suggested + Confirmed tables. This spec mirrors that shape: a
   detection service + a decisions table + a Suggested/Confirmed tab.

2. **FIBR-0138 merchant cleanup** — `merchant_name(description)` is a **pure,
   total** best-guess payee name (strip noise prefixes, drop digit-heavy
   reference tokens, title-case, non-empty fallback), and the grouping key
   callers compare is `normalise_text(merchant_name(desc))`. Recurring detection
   groups on exactly that key — reuse, not re-invention.

The difference from transfers: a transfer decision is about **two specific
transactions**; a recurring decision is about a **payee relationship that
persists across imports**, so it must key on the merchant, not on ids (D5/INV-8).

## Design decisions

- **D1 — A new subsystem, mirroring transfer detection.** New
  `services/recurring.py` (`RecurringService`), `repositories/recurring.py`
  (`RecurringRepository` — the `recurring_decisions` CRUD **and** the
  full-history transaction read the detector consumes), model records in
  `models.py`, migration `_migrate_to_v9`, UI `ui/recurring.py`
  (`RecurringWidget`), and a new tab in `main_window.py`. The **pure detector**
  is a module-level `detect_recurring(...)` function (no `self`, fully
  unit-testable without a vault), the way FIBR-0138's `_sorted_nodes` /
  `merchant_name` are pure.

- **D2 — Detection reads full history, all accounts.** Unlike the dashboard
  (period + account scoped), recurrence is a property of the **whole** ledger —
  a monthly sub is only visible across months. `RecurringRepository.recurring_rows()`
  returns every transaction as a lean `_RecurRow(id, occurred_on, amount_minor,
  description)` (no window, no account filter, no category — none are needed).
  Account scoping is deliberately **not** offered in v1 (a subscription is the
  same arrangement whichever account it debits).

- **D3 — Grouping key + the exclusion seam.** Before grouping, `detect_recurring`
  drops every id in its `excluded_ids` argument (INV-3). `RecurringService`
  sources that set by building a `TransferDetectionService(vault)` (the
  `ReportingService._excluded` precedent) and wrapping its
  `confirmed_transfer_txn_ids()` — a `set[int]` — in `frozenset(...)`; a
  pure-detector test passes a synthetic `frozenset` (so the exclusion is testable
  without a vault). Then `merchant_key =
  normalise_text(merchant_name(row.description))` and group by `(direction,
  merchant_key)`. `direction` is `"out"` when `amount_minor < 0`, `"in"` when
  `> 0` (INV-4). `occurrences` is `len(members)`; `txn_ids` is the members' ids
  **sorted by that same `(date.fromisoformat(occurred_on), id)` key**. Each item's
  display `merchant` is `merchant_name(description)` of its **earliest** member —
  min `(date.fromisoformat(occurred_on), id)`, the **same parsed-date basis** as
  D6/D7 (never the raw string, so it can't disagree with `first_seen`) — a
  deterministic choice, since members sharing a `merchant_key` can still differ
  in raw case/spacing (INV-2). Reuses FIBR-0138 verbatim (INV-5).

- **D4 — Amount stability, integer-exact (INV-6b).** Work in **magnitudes**
  (`abs(amount_minor)`, integers). `med = statistics.median_low(magnitudes)` —
  the lower-middle **actual member value**, so `med` is always an integer
  minor-unit amount (no even-count averaging, no float, no `round()` — the
  money-safety fix over plain `statistics.median`, which returns a **float** on
  an even count). The group is amount-stable iff
  `100 × max(abs(m − med) for m in magnitudes) ≤ 10 × med` — a pure-integer
  cross-multiplication of the ±10% band, so the accept/reject decision is exact
  and float-free at the boundary. `med` is never 0 (zero-amount rows are
  excluded, INV-4; magnitudes ≥ 1), so the test is well-defined. `med` is the
  **representative amount** carried forward; the display `Decimal` is
  `to_display_decimal(med, exponent)`. `median_low` is a **lower-biased** centre
  (an actual low-middle member, not the mean), so a series oscillating within
  ±10% of its *mean* but not its *low-median* is rejected — a deliberate
  simplicity choice, money-safe (a false negative, never a wrong total).

- **D5 — Decisions key on `(direction, merchant_key)`, not ids (INV-8).** The
  `recurring_decisions` table stores one row per decided key. This is the core
  divergence from `transfer_pairs` and the reason a confirmed subscription keeps
  its status as new statements import.

- **D6 — Cadence bands (INV-6c).** Parse each member's `occurred_on` to a `date`
  via `date.fromisoformat`, sort by that `date` (never by the raw string — the
  stored value isn't guaranteed zero-padded-canonical), and compute consecutive
  day-gaps. **Discard any zero gap** (same-day duplicate / retry / split charges
  for one payee) before classifying, so a legitimate series with one same-day
  double-charge isn't rejected; a group left with **no** non-zero gap (all members
  on a single day) does not qualify. Non-overlapping bands classify each
  remaining gap:

  | Cadence | Nominal | Accept band (days) |
  |---------|---------|--------------------|
  | `weekly` | 7 | 5 ≤ g ≤ 10 |
  | `fortnightly` | 14 | 11 ≤ g ≤ 18 |
  | `monthly` | 30 | 25 ≤ g ≤ 35 |
  | `yearly` | 365 | 330 ≤ g ≤ 400 |

  Gaps in the dead zones (19–24, 36–329, > 400, < 5) match no band. A group is
  cadence-regular iff there are **≥ 2** remaining (non-zero) gaps (so ≥ 3 distinct
  charge dates — two points aren't a cadence) and **all** fall in the **same**
  band; that band is the group's cadence. (One skipped period would produce a ~2×
  gap that matches no band → the group is rejected; tolerating skips is a
  documented v1 non-goal, see Out of scope.) The bands are the single source of
  both the qualification test and the `Cadence` enum. Note the two senses of
  "monthly": the `nominal_interval_days = 30` used by the activeness grace (INV-7)
  vs the true **calendar month** `_add_cadence` uses for `next_expected` (D7).

- **D7 — First/last seen + nominal interval + next-expected.** `first_seen` /
  `last_seen` are the **min / max** member `occurred_on` (as `date`s).
  `nominal_interval_days` per cadence is the Nominal column of D6.
  `next_expected = _add_cadence(last_seen, cadence)`: a small **pure
  calendar-aware** helper — `weekly`/`fortnightly` add
  7/14 days; `monthly` adds one calendar month (clamping day-of-month, e.g.
  Jan 31 → Feb 28/29); `yearly` adds one calendar year (Feb 29 → Feb 28 on a
  non-leap year). No `dateutil` dependency — a dozen lines of stdlib
  `datetime`. Activeness (INV-7) uses `nominal_interval_days`.

- **D8 — Monthly-equivalent (dashboard summary).** For the deferred card and
  each item's `monthly_equivalent` field: `weekly → amount × 52 / 12`,
  `fortnightly → amount × 26 / 12`, `monthly → amount × 1`, `yearly → amount /
  12`. Computed on the item's display `amount` (`Decimal`, from D4) × the factor
  at the standard 28-significant-figure context precision, then **quantized** to
  the currency's minor unit with **`ROUND_HALF_EVEN`** (banker's rounding —
  Python `Decimal`'s default; the codebase has no prior `quantize`, so the mode
  is pinned **here** and the summary test locks a half-way case). (`52/12` and
  `26/12` are repeating decimals, so this quantize is the **one** new lossy step
  in the feature — FIBR-0012's tile totals are exact-integer with no rounding, so
  there is no helper to inherit; its magnitude is negligible far below the
  ~10^20-minor-unit range finbreak handles.) `RecurringSummary.monthly_out` sums the
  confirmed **out** items' `monthly_equivalent`; `monthly_in` the **in** items';
  `net = monthly_in − monthly_out`.

- **D9 — Detection is the single source of truth for what's shown.** The service
  never stores a snapshot of a group's amount/cadence/members — it re-runs
  `detect_recurring` on each read and joins the result to the stored decisions.
  Consequence (INV-9): a confirmed item that stops detecting (cancelled, or its
  merchant cleanup changed across a release) simply disappears from Confirmed;
  the decision row survives, so if it reappears it returns to Confirmed. This
  keeps the store tiny and avoids stale snapshots. Documented as intended.
  **Perf (dim 9):** `detect_recurring` is **O(N log N)** in the transaction count
  N (a sort — whole-list or per-group, both O(N log N) — plus linear group /
  qualify passes; `merchant_name`'s per-row regex is
  O(1) in the description length). The tab's `refresh()` calls
  `RecurringService.snapshot(today)` — a **single** detection pass that returns
  `(suggested, confirmed, summary)` together — never three separate full-history
  scans. Design target: comfortably interactive at a large personal ledger
  (~50 000 transactions); a `@pytest.mark.perf` detector benchmark guards against
  pathological blow-up against a generous ceiling (perf tests are excluded from
  the default gate — `pytest -m "not perf"`, so a machine-dependent number never
  reddens CI). The `candidates()` / `confirmed()` / `summary()` methods are thin
  conveniences (targeted use + tests) and may each run their own detection.

- **D10 — Deterministic ordering (uniform-typed sort key).** `detect_recurring`
  returns its list **already sorted** by `(-item.monthly_equivalent,
  item.merchant_key, item.direction)` — biggest recurring spend first, then a
  **stable string** tiebreak, then direction; the service's suggested/confirmed
  partition preserves that order (Suggested and Confirmed are each already in it). All three key elements are uniform-typed (`Decimal`,
  `str`, `str`), heeding the FIBR-0138 mixed-type sort-key bug. The `(direction,
  merchant_key)` pair is unique per item, so the tiebreak is total — the order is
  fully deterministic.

- **D11 — UI: a new "Recurring" tab (the deferred card is FIBR-0143).** A
  `RecurringWidget` with two `QTableWidget`s mirroring the Transfers tab:
  **Suggested** (columns: Merchant · Direction · Cadence · Amount · Next due ·
  Seen ×N; per-row **Confirm** / **Dismiss**) and **Confirmed** (same columns;
  per-row **Un-confirm** = `reset`). A `refresh()` re-reads on view (INV-2/D9),
  a status line and an empty-state message ("No recurring items detected yet —
  import a few months of statements."). All strings `tr()`-wrapped (INV-11). The
  **Next due** column may show a **past** date for an active-but-overdue item
  (INV-7) — expected, not a bug. The
  tab is added after Transfers in `main_window.py`. The **Home dashboard card
  widget** is **not** built here — it is deferred to FIBR-0143 so it slots into
  the reworked dashboard rather than the current one. The `RecurringSummary`
  dataclass and `RecurringService.summary()` that a card *consumes* **are** built
  and tested in this item — only the UI widget defers, so FIBR-0143 is a thin
  consumer.

- **D12 — Direction & cadence as stable enums.** `Direction` and `Cadence` are
  `StrEnum`s (matching `TransferStatus`) with ASCII token values
  (`"in"`/`"out"`; `"weekly"`/`"fortnightly"`/`"monthly"`/`"yearly"`). The UI
  maps each to a `tr()`-ed label at render time (INV-11); the stored/compared
  values are the tokens.

## Data model — schema v9

```sql
CREATE TABLE recurring_decisions(
  id           INTEGER PRIMARY KEY,
  direction    TEXT NOT NULL,      -- 'in' | 'out'
  merchant_key TEXT NOT NULL,      -- normalise_text(merchant_name(description))
  status       TEXT NOT NULL,      -- 'confirmed' | 'dismissed'
  created_at   TEXT NOT NULL,
  UNIQUE(direction, merchant_key)
);
```

No FK to `transactions` (the decision outlives any single transaction — D5); no
cascade needed. `set_decision` is an upsert on the `UNIQUE(direction,
merchant_key)` constraint (`INSERT … ON CONFLICT(direction, merchant_key) DO
UPDATE SET status = excluded.status`); it stamps `created_at` (ISO `today`) on
insert and leaves it unchanged on the conflict-update. `created_at` is a
provenance/audit column (matching `transfer_pairs`) — written, not read by any
`RecurringItem` field. No backfill.

## Test plan

**Pure detector (`detect_recurring`) — no vault, synthetic `_RecurRow`s:**
- Qualifies a clean 5× monthly out-charge (same merchant, ±3% amounts, ~30d
  gaps) → one `monthly`/`out` item; `amount` = `median_low`; `occurrences` = 5;
  `next_expected` = last + 1 month.
- **INV-1 money-safety:** the item's `amount` equals
  `to_display_decimal(statistics.median_low(magnitudes), exponent)` and
  `monthly_equivalent` equals the D8 factor applied to it — asserted against
  hand-computed expectations (no float drift); a member set whose true sum is
  known confirms grouping never altered a total.
- **INV-5 grouping key:** two rows with noisy descriptions that both clean to the
  same payee (differing only in a trailing digit-heavy reference token) land in
  **one** group whose key equals `normalise_text(merchant_name(desc))`; a
  distinct payee stays separate; an all-digits description (cleans to empty) falls
  back to a **non-blank** `merchant_key` (its trimmed raw text). The item's
  display `merchant` is the earliest member's `merchant_name` even when a later
  member has different casing (D3 determinism).
- **Empty / degenerate inputs:** an empty `rows` list → `[]`; a single-row
  merchant → `[]` (below `_MIN_OCCURRENCES`); **all-same-day** members (every gap
  = 0, all discarded → no non-zero gap left) → rejected.
- **Same-day duplicate tolerated (D6):** a clean monthly series with **one**
  extra same-day charge (a retry/split) still qualifies as `monthly` — the zero
  gap is discarded, the ~30d gaps remain in-band; `occurrences` counts all
  members including the duplicate.
- **Two distinct dates only → rejected (INV-6c ≥ 2 non-zero gaps):** 3 members on
  2 distinct dates (`[Jan 1, Jan 1, Feb 1]` → one non-zero gap after the discard)
  is **not** surfaced — one interval isn't a cadence.
- Rejects a 2× group (below `_MIN_OCCURRENCES`).
- Rejects amount drift > 10% (one member 15% off the median).
- Median robustness: 4 members at 100 + 1 at 500; `median_low` 100, the 500
  breaks the ±10% test → rejected (not silently averaged in).
- **±10% boundary (INV-6b), regression-locked like the cadence edge:** `med = 100`,
  a member at **110** → accepted (`100 × 10 ≤ 10 × 100`); at **111** → rejected —
  the exact integer accept/reject edge.
- Rejects irregular gaps (30, 30, 90 days — the 90 matches no band).
- Cadence bands: weekly (7d), fortnightly (14d), yearly (365d) each classified;
  boundary gaps 25 & 35 → monthly, 24 & 36 → rejected (dead zone).
- Direction split: a merchant with both +charges and −charges → two items.
- `amount_minor == 0` rows excluded (INV-4).
- Confirmed-transfer ids excluded before grouping (INV-3).
- Activeness (INV-7): a monthly group whose `last_seen` is 100 days before
  `today` (> 2×30) is dropped; 55 days (≤ 60) is kept.
- `_add_cadence` calendar edges: Jan 31 +monthly → Feb 28 (non-leap) / Feb 29
  (leap); Feb 29 +yearly → Feb 28; ordinary +weekly/+fortnightly.
- Determinism/order (D10): a fixed input yields a stable, biggest-first order;
  the mixed-type-safe key never raises on same-amount different-merchant items.
  Earliest-member `merchant` and `txn_ids` order are stable when a group mixes a
  dashed date (`"2026-01-05"`) and a basic-format one (`"20260105"`) — both parse
  to the same `date`, so the parsed-date sort (D3/D6) is order-correct where a raw
  lexical sort would not be.
- **Perf (`@pytest.mark.perf`, D9):** `detect_recurring` over 50 000 synthetic
  rows completes under a generous ceiling (guards O(N log N), not a tight
  machine-dependent ms number); excluded from the default `-m "not perf"` gate.

**Service (`RecurringService`) — real vault:**
- `candidates` excludes confirmed & dismissed keys; `confirmed` shows only
  confirmed-and-still-detected; a dismissed key never appears in either.
- `confirm`/`dismiss`/`reset` persist and round-trip (upsert semantics: a
  dismiss-then-confirm flips status, one row).
- A confirmed item that stops detecting (drop its recent rows) vanishes from
  `confirmed`; re-add rows → it returns (decision survived — INV-9/D9).
- `summary` monthly-equivalent: a confirmed weekly item and a monthly item sum
  to the expected (minor-unit-rounded, D8) `monthly_out`; an `in` item feeds
  `monthly_in`; `net = monthly_in − monthly_out`.
- **`snapshot(today)` single-pass:** its returned `(suggested, confirmed,
  summary)` triple equals `candidates(today)` / `confirmed(today)` /
  `summary(today)` computed separately, and it runs detection **once** (assert via
  a spy / call-count on the repository read or `detect_recurring`).
- **D2 all-accounts / full-history:** members of one merchant spread across **≥ 2
  accounts** group into a **single** item (no account or date-window filter on the
  detection read).
- Migration: a v8 vault opens, migrates to v9, the table exists, old data intact.
  The single-transaction atomicity ("mid-step failure leaves a re-openable v8",
  INV-10) is **inherited** from the shared `owned_transaction` pattern (FIBR-0011
  migration coverage), not re-tested here; a `BackupService` test asserts a vault
  whose `schema_version` exceeds this build's `LATEST_SCHEMA_VERSION` is refused
  by the version-ceiling gate.

**UI (`RecurringWidget`, `qtbot`):**
- Both tables populate from the service; a Confirm moves a row Suggested →
  Confirmed and a Dismiss removes it from Suggested; Un-confirm returns it.
- Empty-state message when nothing is detected.
- Column headers / labels are `tr()`-wrapped (grep-style INV-11 check as
  FIBR-0127 did for theme strings).
- **INV-11 enum tokens (model test):** `Direction.OUT.value == "out"` /
  `Direction.IN.value == "in"` and each `Cadence.*.value` is its stable ASCII
  token — the stored/compared values are the tokens, decoupled from the
  `tr()`-ed display labels.
- `main_window` shell-shape test sees the new tab after Transfers; the
  transaction-count wiring is unaffected.

## Out of scope (v1 — YAGNI)

- **No missed-charge alerts / notifications / predictions** beyond the displayed
  `next_expected`.
- **No skip tolerance** — a group with one skipped period (a ~2× gap) is not
  detected (D6). Revisit only if real statements show it matters.
- **Only weekly / fortnightly / monthly / yearly cadences** — other rhythms
  (quarterly, bi-monthly, semi-annual) fall in the D6 dead zones and are not
  detected in v1.
- **No manual "add a subscription"** and **no editing** a detected amount /
  cadence — detection-only (D9).
- **No per-account scoping** of detection (D2).
- **No Home dashboard card** — deferred to FIBR-0143 (B4); `RecurringSummary`
  and `summary()` are built and tested here so the card is a thin consumer later.
- **No budgets** — that is the FIBR-0022 follow-up.
