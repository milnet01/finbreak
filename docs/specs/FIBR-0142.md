**Theme:** finbreak learns to recognise your **regular** money. Today every
transaction is a one-off in the ledger. This adds a detector that notices when
the *same payee* is charged (or pays you) *the same sort of amount* on a
*regular rhythm* — a subscription, a debit order, insurance, a salary, a
standing deposit — and gathers those into one **recurring item** you can
**Confirm** (yes, that's a real standing arrangement) or **Dismiss** (no, ignore
it). A new **Recurring** tab lists what it found — merchant, in/out, how often,
the typical amount, when the next one is due, and how many times it has been
seen. Confirmed items later roll up into a Home dashboard card showing your
total monthly recurring in/out (that card is deferred to FIBR-0143). Nothing the
detector decides ever changes a stored amount — grouping only decides *which
recurring item a transaction belongs to*.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| **INV-1** | **Money-safety.** Every recurring item's representative `amount`, `monthly_equivalent`, and the dashboard `RecurringSummary` totals are computed from integer `amount_minor` (median of the real member magnitudes; monthly-equivalent by the exact per-cadence factor in D8); grouping (D3) and cadence classification (D6) only ever decide **which recurring item a transaction is placed under** — they can **never** change a stored amount. The only lossy step is the final `to_display_decimal` at the display boundary, exactly as FIBR-0012/0138 do it. | FIBR-0007; FIBR-0012 INV-13 |
| **INV-2** | **Pure, deterministic, clock-injected detection.** `detect_recurring(rows, today, dismissed_keys, confirmed_keys)` is a pure function: no I/O, no `date.today()` inside (`today` is a parameter), no randomness. The same `(rows, today, decisions)` always yields the same list in the same **order** (D10 explicit sort key, uniform types — no mixed-type sort). | coding.md § 1.2; FIBR-0138 INV-9 |
| **INV-3** | **Confirmed transfers are excluded.** Every transaction id in `TransferDetectionService.confirmed_transfer_txn_ids()` is dropped before grouping, so a self-transfer between your own accounts is never mistaken for a subscription or standing income. An **un**confirmed pair is still ordinary money and remains eligible (the exclusion keys on *confirmed*, matching the dashboard's `_excluded()`). | FIBR-0011; FIBR-0012 INV-1 |
| **INV-4** | **Direction from sign; zero excluded.** A transaction with `amount_minor < 0` is **out**; `amount_minor > 0` is **in**; `amount_minor == 0` (no money moved, no determinable direction) is excluded from detection entirely. A payee that both bills and refunds you forms **two** independent groups (one per direction). | FIBR-0007 |
| **INV-5** | **Grouping key reuses the FIBR-0138 cleanup, unchanged.** Transactions are grouped by `(direction, merchant_key)` where `merchant_key = normalise_text(merchant_name(description))` — the **same** pure, total primitives the drill-down uses. The cleanup is fuzzy by design and refined per release; because of INV-1 a mis-grouped payee is only cosmetic. A description that cleans to empty falls back to its trimmed raw text (never a blank key — stored descriptions are non-empty). | FIBR-0138 INV-5; coding.md § 1.3 |
| **INV-6** | **"Balanced" qualification is a fixed, tested rule.** A group qualifies as recurring iff **all** of: (a) **≥ 3** member transactions; (b) **amount stability** — every member magnitude within **±10%** of the group **median** magnitude (`median` is robust to one outlier); (c) **cadence regularity** — sorting members by `occurred_on`, **every** consecutive gap (in days) falls inside exactly one cadence band (D6), and that band is the group's cadence. A group failing any clause is not surfaced. The three thresholds (`_MIN_OCCURRENCES = 3`, `_AMOUNT_TOLERANCE = 0.10`, the D6 bands) are named module constants, changed in **one** place. | roadmap B3 (Balanced) |
| **INV-7** | **Activeness filter — stale arrangements age out.** A qualifying group is surfaced only if it is **active**: `(today − last_seen).days ≤ _ACTIVE_GRACE × nominal_interval_days(cadence)` (`_ACTIVE_GRACE = 2`, i.e. one missed period tolerated). A subscription cancelled long ago (no recent charges) drops off the suggestions rather than lingering with a next-due date in the past. Applies to confirmed items too (detection is the single source of truth, D9). | roadmap B3; D7 |
| **INV-8** | **Decisions persist by key, not by transaction.** A user's Confirm/Dismiss is stored in `recurring_decisions` keyed on `(direction, merchant_key)` — **not** on transaction ids — because a recurring item's members grow with every import. A **dismissed** key never appears in Suggested or the dashboard, even as new charges arrive; a **confirmed** key appears in Confirmed (and the dashboard) for as long as it still detects as active; a key with no decision is **suggested**. `reset` removes the decision (back to suggested). | FIBR-0011 (transfer_pairs analogue) |
| **INV-9** | **The three lists are a clean partition.** For a given `(rows, today)`, every **active, qualifying** group is in exactly one of Suggested / Confirmed / (hidden) Dismissed, by its stored decision. A group that stops qualifying or goes inactive leaves all three lists (it simply isn't detected) — a confirmed-then-cancelled item disappears rather than showing stale; if it reappears it returns to Confirmed (its key decision survived). | INV-7; INV-8 |
| **INV-10** | **Schema v9 is additive + atomic.** The `recurring_decisions` table is added by `_migrate_to_v9` as **one** `owned_transaction` unit (explicit `BEGIN` first, `UPDATE schema_version` last), no backfill, no change to any existing table — a mid-step failure leaves a re-openable v8. `LATEST_SCHEMA_VERSION` becomes 9; `BackupService`'s floor check and the migration map both pick it up. | FIBR-0011 `_migrate_to_v8` |
| **INV-11** | **i18n-clean UI.** Every user-facing string in the Recurring tab (column headers, the direction/cadence labels, buttons, empty-state text) is wrapped in `tr()`; the cadence/direction **enum values** stored/compared in code are stable ASCII tokens, never the translated display text (the FIBR-0138 `DrillLabels` lesson — labels injected, tokens fixed). | FIBR-0017 readiness; coding.md § 5.2 |

## Context

finbreak already has two precedents this feature composes:

1. **FIBR-0011 transfer detection** — a `TransferDetectionService` produces
   candidate pairs, the user confirms/rejects, and each decision is one row in a
   `transfer_pairs` table (`status` = `'confirmed'`/`'rejected'`, both FKs
   `ON DELETE CASCADE`, `UNIQUE` on the ordered pair). The 7th **Transfers** tab
   shows Suggested + Confirmed tables. This spec mirrors that shape: a
   detection service + a decisions table + a Suggested/Confirmed tab.

2. **FIBR-0138 merchant cleanup** — `merchant_name(description)` is a **pure,
   total** best-guess payee name (strip noise prefixes, drop digit-heavy
   reference tokens, title-case, non-empty fallback), and the grouping key
   callers compare is `normalise_text(merchant_name(desc))`. Recurring detection
   groups on exactly that key — reuse, not re-invention.

The difference from transfers: a transfer decision is about **two specific
transactions**; a recurring decision is about a **payee relationship that
persists across imports**, so it must key on the merchant, not on ids (D5/INV-8).

## Design decisions

- **D1 — A new subsystem, mirroring transfer detection.** New
  `services/recurring.py` (`RecurringService`), `repositories/recurring.py`
  (`RecurringRepository` — the `recurring_decisions` CRUD **and** the
  full-history transaction read the detector consumes), model records in
  `models.py`, migration `_migrate_to_v9`, UI `ui/recurring.py`
  (`RecurringWidget`), and a new tab in `main_window.py`. The **pure detector**
  is a module-level `detect_recurring(...)` function (no `self`, fully
  unit-testable without a vault), the way FIBR-0138's `_sorted_nodes` /
  `merchant_name` are pure.

- **D2 — Detection reads full history, all accounts.** Unlike the dashboard
  (period + account scoped), recurrence is a property of the **whole** ledger —
  a monthly sub is only visible across months. `RecurringRepository.recurring_rows()`
  returns every transaction as a lean `_RecurRow(id, occurred_on, amount_minor,
  description)` (no window, no account filter, no category — none are needed).
  Account scoping is deliberately **not** offered in v1 (a subscription is the
  same arrangement whichever account it debits).

- **D3 — Grouping key.** `merchant_key = normalise_text(merchant_name(row.description))`.
  Group by `(direction, merchant_key)`. `direction` is `"out"` when
  `amount_minor < 0`, `"in"` when `> 0` (INV-4). Reuses FIBR-0138 verbatim
  (INV-5).

- **D4 — Amount stability on the median (INV-6b).** Work in **magnitudes**
  (`abs(amount_minor)`, integers). `med = statistics.median(magnitudes)`. The
  group is amount-stable iff `max(abs(m − med) for m in magnitudes) ≤
  _AMOUNT_TOLERANCE × med` (`_AMOUNT_TOLERANCE = 0.10`). The **representative
  amount** carried forward is `med` (an integer minor-unit value; median of an
  even count averages the two middles — rounded to the nearest minor unit with
  `round()`, kept integer). `med` is never 0 (zero-amount rows are excluded,
  INV-4), so the ratio is well-defined.

- **D5 — Decisions key on `(direction, merchant_key)`, not ids (INV-8).** The
  `recurring_decisions` table stores one row per decided key. This is the core
  divergence from `transfer_pairs` and the reason a confirmed subscription keeps
  its status as new statements import.

- **D6 — Cadence bands (INV-6c).** Sort members by `occurred_on`; compute
  consecutive day-gaps. Non-overlapping bands classify a gap:

  | Cadence | Nominal | Accept band (days) |
  |---------|---------|--------------------|
  | `weekly` | 7 | 5 ≤ g ≤ 10 |
  | `fortnightly` | 14 | 11 ≤ g ≤ 18 |
  | `monthly` | 30 | 25 ≤ g ≤ 35 |
  | `yearly` | 365 | 330 ≤ g ≤ 400 |

  Gaps in the dead zones (19–24, 36–329, > 400, < 5) match no band. A group is
  cadence-regular iff **every** consecutive gap falls in the **same** band; that
  band is the group's cadence. (One skipped period would produce a ~2× gap that
  matches no band → the group is rejected; tolerating skips is a documented v1
  non-goal, see Out of scope.) The bands are the single source of both the
  qualification test and the `Cadence` enum.

- **D7 — Nominal interval + next-expected.** `nominal_interval_days` per cadence
  is the Nominal column of D6. `next_expected = _add_cadence(last_seen,
  cadence)`: a small **pure calendar-aware** helper — `weekly`/`fortnightly` add
  7/14 days; `monthly` adds one calendar month (clamping day-of-month, e.g.
  Jan 31 → Feb 28/29); `yearly` adds one calendar year (Feb 29 → Feb 28 on a
  non-leap year). No `dateutil` dependency — a dozen lines of stdlib
  `datetime`. Activeness (INV-7) uses `nominal_interval_days`.

- **D8 — Monthly-equivalent (dashboard summary, exact).** For the deferred card
  and any per-item "≈ per month" display: `weekly → amount × 52 / 12`,
  `fortnightly → amount × 26 / 12`, `monthly → amount × 1`, `yearly → amount /
  12`. Computed as `Decimal(representative_minor) × factor` at the display
  boundary (the multiply/divide on `Decimal` is exact for these rationals except
  the final round to the currency's minor unit — the same rounding the tiles
  already accept). `RecurringSummary.monthly_out` sums the confirmed **out**
  items' monthly-equivalents; `monthly_in` the **in** items'; `net = in − out`.

- **D9 — Detection is the single source of truth for what's shown.** The service
  never stores a snapshot of a group's amount/cadence/members — it re-runs
  `detect_recurring` on each read and joins the result to the stored decisions.
  Consequence (INV-9): a confirmed item that stops detecting (cancelled, or its
  merchant cleanup changed across a release) simply disappears from Confirmed;
  the decision row survives, so if it reappears it returns to Confirmed. This
  keeps the store tiny and avoids stale snapshots. Documented as intended.

- **D10 — Deterministic ordering (uniform-typed sort key).** Suggested and
  Confirmed are each sorted by `(-monthly_equivalent_minor, merchant_key,
  direction)` — biggest recurring spend first, then a **stable string** tiebreak,
  then direction. All three key elements are uniform-typed (`int`, `str`,
  `str`), heeding the FIBR-0138 mixed-type sort-key bug. `monthly_equivalent_minor`
  is an integer intermediate (representative × cadence factor, kept in minor
  units) purely for ordering.

- **D11 — UI: a new "Recurring" tab (the deferred card is FIBR-0143).** A
  `RecurringWidget` with two `QTableWidget`s mirroring the Transfers tab:
  **Suggested** (columns: Merchant · Direction · Cadence · Amount · Next due ·
  Seen ×N; per-row **Confirm** / **Dismiss**) and **Confirmed** (same columns;
  per-row **Un-confirm** = `reset`). A `refresh()` re-reads on view (INV-2/D9),
  a status line and an empty-state message ("No recurring items detected yet —
  import a few months of statements."). All strings `tr()`-wrapped (INV-11). The
  tab is added after Transfers in `main_window.py`. The **Home dashboard card**
  (`RecurringSummary`) is **not** built here — it is deferred to FIBR-0143 so it
  slots into the reworked dashboard rather than the current one.

- **D12 — Direction & cadence as stable enums.** `Direction` and `Cadence` are
  `StrEnum`s (matching `TransferStatus`) with ASCII token values
  (`"in"`/`"out"`; `"weekly"`/`"fortnightly"`/`"monthly"`/`"yearly"`). The UI
  maps each to a `tr()`-ed label at render time (INV-11); the stored/compared
  values are the tokens.

## Data model — schema v9

```sql
CREATE TABLE recurring_decisions(
  id           INTEGER PRIMARY KEY,
  direction    TEXT NOT NULL,      -- 'in' | 'out'
  merchant_key TEXT NOT NULL,      -- normalise_text(merchant_name(description))
  status       TEXT NOT NULL,      -- 'confirmed' | 'dismissed'
  created_at   TEXT NOT NULL,
  UNIQUE(direction, merchant_key)
);
```

No FK to `transactions` (the decision outlives any single transaction — D5); no
cascade needed. `set_decision` is an upsert on the `UNIQUE(direction,
merchant_key)` constraint (`INSERT … ON CONFLICT(direction, merchant_key) DO
UPDATE SET status = excluded.status`). No backfill.

## Test plan

**Pure detector (`detect_recurring`) — no vault, synthetic `_RecurRow`s:**
- Qualifies a clean 5× monthly out-charge (same merchant, ±3% amounts, ~30d
  gaps) → one `monthly`/`out` item; `amount` = median; `occurrences` = 5;
  `next_expected` = last + 1 month.
- Rejects a 2× group (below `_MIN_OCCURRENCES`).
- Rejects amount drift > 10% (one member 15% off the median).
- Median robustness: 4 members at 100 + 1 at 500; median 100, the 500 breaks the
  ±10% test → rejected (not silently averaged in).
- Rejects irregular gaps (30, 30, 90 days — the 90 matches no band).
- Cadence bands: weekly (7d), fortnightly (14d), yearly (365d) each classified;
  boundary gaps 25 & 35 → monthly, 24 & 36 → rejected (dead zone).
- Direction split: a merchant with both +charges and −charges → two items.
- `amount_minor == 0` rows excluded (INV-4).
- Confirmed-transfer ids excluded before grouping (INV-3).
- Activeness (INV-7): a monthly group whose `last_seen` is 100 days before
  `today` (> 2×30) is dropped; 55 days (≤ 60) is kept.
- `_add_cadence` calendar edges: Jan 31 +monthly → Feb 28 (non-leap) / Feb 29
  (leap); Feb 29 +yearly → Feb 28; ordinary +weekly/+fortnightly.
- Determinism/order (D10): a fixed input yields a stable, biggest-first order;
  the mixed-type-safe key never raises on same-amount different-merchant items.

**Service (`RecurringService`) — real vault:**
- `candidates` excludes confirmed & dismissed keys; `confirmed` shows only
  confirmed-and-still-detected; a dismissed key never appears in either.
- `confirm`/`dismiss`/`reset` persist and round-trip (upsert semantics: a
  dismiss-then-confirm flips status, one row).
- A confirmed item that stops detecting (drop its recent rows) vanishes from
  `confirmed`; re-add rows → it returns (decision survived — INV-9/D9).
- `summary` monthly-equivalent: a confirmed weekly item and a monthly item sum
  to the exact expected `monthly_out`; an `in` item feeds `monthly_in`; `net`.
- Migration: a v8 vault opens, migrates to v9, the table exists, old data intact.

**UI (`RecurringWidget`, `qtbot`):**
- Both tables populate from the service; a Confirm moves a row Suggested →
  Confirmed and a Dismiss removes it from Suggested; Un-confirm returns it.
- Empty-state message when nothing is detected.
- Column headers / labels are `tr()`-wrapped (grep-style INV-11 check as
  FIBR-0127 did for theme strings).
- `main_window` shell-shape test sees the new tab after Transfers; the
  transaction-count wiring is unaffected.

## Out of scope (v1 — YAGNI)

- **No missed-charge alerts / notifications / predictions** beyond the displayed
  `next_expected`.
- **No skip tolerance** — a group with one skipped period (a ~2× gap) is not
  detected (D6). Revisit only if real statements show it matters.
- **No manual "add a subscription"** and **no editing** a detected amount /
  cadence — detection-only (D9).
- **No per-account scoping** of detection (D2).
- **No Home dashboard card** — deferred to FIBR-0143 (B4); `RecurringSummary`
  and `summary()` are built and tested here so the card is a thin consumer later.
- **No budgets** — that is the FIBR-0022 follow-up.
