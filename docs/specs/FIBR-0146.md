**Theme:** finbreak stops making a person type programmer date codes. Today, when
a bank's date layout doesn't match the box's default, the whole import fails with
a wall of raw error text (the mechanism is in Context; the tester hit exactly
this). This change reads the actual dates, **guesses** the layout, and offers a
**plain-English picker** ("20/07/2026", "2026-07-20", "20 Jul 2026", …) with the
guess pre-selected and a **live preview** of how the dates read — so the common
case just works, and a wrong guess is visible and fixable *before* import rather
than a mystery afterwards. Nothing about the money math changes; only the date
the importer reads, and only after the user can see it.

## At a glance — invariants

|-----|-----------|---------|
| **INV-1** | **Never a silent wrong-day.** Auto-detection only ever *pre-selects* a format in a **visible** picker; the resulting dates are shown to the user — the map-step live preview (D6) **and** the existing preview-step Date column (`_fill_preview_table`) — before any row is written. ## Context

The import wizard (`ui/import_wizard.py`) has three format paths (`_select_file`):

1. **OFX** — self-describing; parsed and jumped straight to preview, **skipping
   the map step**. No date-format box is ever shown. **Unaffected.**
2. **PDF** — a **recognised Standard Bank** statement is parsed by its own reader
   and skips the map step (**unaffected**); **any other PDF** is extracted to a
   CSV-text table (FIBR-0009 D1) and shown the map step — **in scope**.
3. **CSV** — an exact header-signature match auto-applies its saved profile and
   jumps to preview; **no match** shows the map step — **in scope**.

So the fix lives entirely at **the map step**, reached by an **unmatched CSV or
a generic (non-SB) PDF** — exactly the tester's case (an unrecognised bank, no
saved profile). At that step today, `date_format` is a free-text `QLineEdit`
pre-filled with `%Y-%m-%d` (`import_wizard.py:163`), and `_mapping_from_form`
reads `self._date_format.text().strip()`. The parse itself is
`datetime.strptime(row[mapping.date_column].strip(), mapping.date_format)`
(`csv_importer.py:74`); a failure becomes `RowError(row_number, str(exc))`
(`:88-89`), whose `str(exc)` is the raw `time data '…' does not match format '…'`.

The pipeline downstream of the mapping is **correct and stays untouched** — this
change only produces a better `date_format` guess and a friendlier failure.

## Design decisions

- **D1 — A pure detector module, mirroring `csv_importer`.** New
  `src/finbreak/importers/date_detect.py` — **no Qt, no vault**, unit-testable on
  plain strings the way `CsvImporter` is. It exports the ordered constant
  `KNOWN_DATE_FORMATS: list[tuple[str, str]]` (each `(example, fmt)` — the
  language-neutral example rendering and its `strptime` pattern) and the pure
  `detect_date_format(samples: Sequence[str]) -> DateFormatGuess` (a `Sequence`,
  not a one-shot iterable — D8/New symbols). The example
  strings render one **fixed unambiguous reference date** — day **20**, month
  **07**, year **2026** — so day-first ("20/07/2026") and month-first
  ("07/20/2026") are visually distinct in the picker.

- **D2 — The candidate list + the ambiguity + year-width rules.**
  `KNOWN_DATE_FORMATS` (ordered — ISO first, then day-first, then month-first,
  then named-month, then the 2-digit-year slash variants; the order is the
  ambiguity tiebreak, INV-2):

  | fmt | example (ref 2026-07-20) |
  |-----|--------------------------|
  | `%Y-%m-%d` | 2026-07-20 |
  | `%Y/%m/%d` | 2026/07/20 |
  | `%d/%m/%Y` | 20/07/2026 |
  | `%d-%m-%Y` | 20-07-2026 |
  | `%d.%m.%Y` | 20.07.2026 |
  | `%m/%d/%Y` | 07/20/2026 |
  | `%m-%d-%Y` | 07-20-2026 |
  | `%m.%d.%Y` | 07.20.2026 |
  | `%d %b %Y` | 20 Jul 2026 |
  | `%d %B %Y` | 20 July 2026 |
  | `%d-%b-%Y` | 20-Jul-2026 |
  | `%b %d, %Y` | Jul 20, 2026 |
  | `%B %d, %Y` | July 20, 2026 |
  | `%d/%m/%y` | 20/07/26 |
  | `%m/%d/%y` | 07/20/26 |

  **Detection:** strip each sample, drop blanks. For each candidate, count the
  non-blank samples it parses, where a sample **parses** iff `strptime` succeeds
  (no year window, no clock — INV-2). **2-digit and 4-digit year layouts are
  separated by `strptime` itself, not by any guard:** `%Y` matches **exactly four
  digits** and `%y` **exactly two**, and a leftover digit fails with "unconverted
  data remains" — so `strptime("20-07-26", "%d-%m-%Y")` **raises** (the 2-digit
  "26" is not four digits) and `strptime("20/07/2026", "%d/%m/%y")` **raises**
  ("unconverted data remains: 26"). A `%y`/`%Y` pattern therefore never claims a
  field of the wrong width, so including both the 4-digit and 2-digit variants
  adds no spurious cross-width match (both were confirmed against CPython 3.13,
  the 3.12+ target's stable `_strptime`). `DateFormatGuess.fmt` is the candidate
  with the **highest** parse-count, ties broken by the fixed list order above;
  `fmt` is `None` when **no** candidate parses **any** sample.
  `DateFormatGuess.ambiguous` is `True` when **two or more** candidates share the
  maximum parse-count (> 0). For a **uniform-format** column (the in-scope case) the only
  tie is **numeric day-first vs month-first** (ISO/named-month/separator/width
  variants are mutually exclusive on the same string), and it survives only when
  **every** sampled day-number is ≤ 12 (a day-number > 12 makes the month-first
  `%m` fail those rows → a lower count → no tie; `%m` matches only 1–12,
  confirmed). A **mixed-format** column (out of scope) could tie on other axes
  (e.g. 2- vs 4-digit width), where the D6 "day and month" nudge wording is
  approximate — an accepted limitation for an out-of-scope input. Detection is best-effort *pre-selection* only; the picker + previews
  are always shown (INV-1), so a wrong guess is never acted on silently.

- **D3 — Friendly per-row date error (`csv_importer`).** Split the date parse out
  of the shared per-row `try`: bind `raw = row[mapping.date_column].strip()` (the
  stripped cell) and parse it in its own `try`, and on `ValueError` append a
  friendly `RowError` then `continue`: `f'could not read the date "{raw}"'` for a
  non-empty cell, or the clearer `"the date cell is empty"` when `raw == ""` (a
  blank date cell — common in PDF-serialised tables — which under a normal format
  raises `strptime` just like any other unparseable value; the empty-format case
  can't reach here, D4). Either message keeps the failure visible and drops the raw
  `%`-format. The **amount** parse + `parse_transaction`
  stay in their existing `try` with their existing `str(exc)` (those messages are
  already human — e.g. "amount is not a valid number", "occurred_on must be a valid
  ISO-8601 date" — and are **not** changed; `parse_transaction`'s own date
  re-validation cannot leak raw `%`-format text either, since the `occurred_on`
  passed to it is the freshly-built `strptime(...).date().isoformat()` string, so a
  date *failure* there can't occur). No change to `RowError`'s shape (still
  `row_number` + `reason`), so OFX and the shared `ImportService` are untouched
  (INV-6). The ragged-row guard above is unchanged.

- **D4 — The friendly picker (`import_wizard`).** Replace the `QLineEdit("%Y-%m-%d")`
  with `self._date_format` = a `QComboBox` (objectName `import_date_format`) on the
  form row still labelled `tr("Date format")` (the label the D7 banner tells the
  user to "check", so the two must match), populated from `KNOWN_DATE_FORMATS` —
  each entry's **text** is the example string, its **data** the `fmt` — followed by a final **"Custom…"** entry (data
  a sentinel). Selecting "Custom…" reveals a raw-pattern `QLineEdit`
  (`import_date_format_custom`, hidden otherwise), preserving the power-user /
  exotic-bank capability (INV-4). `_mapping_from_form` reads the format via a new
  `_selected_date_format()` (the combo's `currentData()`, or the custom field's
  text when "Custom…" is active). An **empty/blank** Custom pattern is **rejected**
  at the mapping-validation boundary — `ImportService._validate_mapping` gains a
  non-empty `date_format.strip()` check that raises the friendly `ValueError`
  `"choose a date format"`, surfaced on the map step (via `_on_map_next`) so the
  user can't proceed with no format. This is deliberate: an empty format string is
  a **trap**, not merely unparseable — `strptime("", "")` *succeeds* and returns
  **1900-01-01**, so a blank date cell (which PDF-serialised tables produce
  routinely) under an empty format would silently import as a phantom 1900 date
  rather than error. Rejecting the empty format forecloses that; the format is
  never silently defaulted to ISO either. `_apply_profile_to_combos` selects the entry
  whose data equals `mapping.date_format`, or — when the stored format is not in
  the known list — selects "Custom…" and fills the raw field with it verbatim
  (INV-4 round-trip). The combo's **initial/default** selection is the first entry
  (ISO `%Y-%m-%d` — today's default); it is overridden by auto-detect (D5) or a
  matched profile, and **left at that default** when auto-detect returns `None`
  (an unrecognised date column), so the live-preview "couldn't read" nudge (D6)
  then guides the user to pick. Programmatic combo changes are
  `QSignalBlocker`-wrapped so they fire no re-detect (D5).

- **D5 — Auto-detect wiring (unmatched case + manual date-column change).**
  `_autodetect_date_format()` reads the **currently-selected** date column's
  samples via `_date_samples` (D8 — the `csv.DictReader` read over `self._text`,
  the same text a PDF is serialised into), calls `detect_date_format`, stores
  `self._date_ambiguous = guess.ambiguous`, and — only when `guess.fmt` is
  non-`None` — sets the picker to that format. It does **not** itself refresh the
  preview — **the caller** calls `_update_date_preview` right after (one owner, no
  double refresh), so a `None` result still shows the "couldn't read" nudge. It is
  called: (a) in `_select_file`'s **no-match** CSV branch
  (before `_goto_step(_STEP_MAP)`); (b) in `_continue_after_decrypt`'s **no-match**
  generic-PDF branch (before showing the map step), **and** in
  `_on_pdf_table_changed`'s no-match branch — so a user switching to a *different
  unmatched* PDF table re-detects the format and refreshes the live preview for
  the new table (never a stale format/preview from the previous table); and (c)
  from a new slot on the **date-column** combo's `currentIndexChanged` when the
  **user** changes it — intentionally re-detecting for the newly-chosen column
  (which may hold a different layout) and **overriding any prior manual format
  pick**; that override is deliberate (the column changed, so the old pick no
  longer applies), and the preview immediately shows the re-detected reading.
  Each of (a)–(c) calls `_autodetect_date_format` then `_update_date_preview`. Programmatic fills (`_populate_mapping_combos`,
  `_apply_profile_to_combos`) are signal-blocked so only a real user change
  re-detects. A **matched profile short-circuits** auto-detect (its stored format
  wins, INV-4) — the matched branches already skip to preview or apply the
  profile's format. `_apply_profile_to_combos` **clears** `self._date_ambiguous`
  (a matched profile is authoritative, never "ambiguous") and its caller refreshes
  `_update_date_preview`, so switching a PDF from an ambiguous *unmatched* table to
  a *matched* one clears the stale nudge + preview rather than leaving the previous
  table's reading on screen.

- **D6 — Live "how the dates read" preview (the *confirm* affordance).** A read-only
  `QLabel` (`import_date_preview`) under the picker, refreshed by
  `_update_date_preview()` on any format change (the `import_date_format` combo's
  `currentIndexChanged` **and** the `import_date_format_custom` field's
  `textChanged`) **and** date-column change. It takes the first `_PREVIEW_SAMPLES`
  (**up to 3**) values from `_date_samples` (the same non-blank values, D8 — a
  small table may yield only 1–2) and, under the currently-selected format,
  renders them as ISO dates — `tr("Dates read as: {samples}")` — **only when the
  selected format parses all of the shown samples** (a clean, trustworthy preview;
  "all of the shown" so a valid 1- or 2-sample column still gets the clean
  branch, never the fallback). Two distinct fallbacks cover the other
  cases, so the label is never left showing an empty or half-parsed "Dates read
  as:" tail: (i) **no samples** — `_date_samples` returns `[]` (a header-only
  extracted table or an all-blank date column; an empty CSV raises earlier in
  `read_header` and never reaches the map step) → `tr("No dates found in this
  column.")`; (ii) **samples present but the selected format fails any of the shown
  ones** (partial *or* total failure — including a non-date "junk" column, whose
  non-blank cells *are* sampled and simply don't parse) → `tr("These dates couldn't
  be read with this format — pick another.")`. The ambiguity nudge —
  `tr("Check these are right — the day and month might be the other way around.")`
  — is prepended **iff** `self._date_ambiguous` is set (D5). A **manual** format
  change (`_on_date_format_changed`) clears `self._date_ambiguous` first, so the
  nudge never goes stale against a format the user chose by hand. This is the
  immediate "eyeball before importing" the user asked for; the preview **step**
  remains the full row-by-row confirmation.

- **D7 — Whole-import banner (preview step).** In `_apply_preview_counts`, when
  `preview.new_count == 0 and preview.duplicate_count == 0 and preview.errors`
  (nothing landed, at least one error — the tester's exact 0·0·165 state), show a
  prominent banner (`import_preview_banner`, hidden otherwise):
  `tr("None of the rows could be imported. The usual cause is the wrong date
  format — go back and check the Date format matches your statement.")`. The
  message names the **usual** cause without over-claiming it is *always* dates (a
  full amount mis-map can also zero the import). Any successful or duplicate row
  hides the banner (partial failures are already flagged per-row); the
  zero-rows/zero-errors case (a header-only extracted table — nothing to import
  and nothing failed) is also banner-less, and Import stays disabled as today. The
  banner is separate from `_summary_label` (the counts line stays).

- **D8 — Sampling bound + reuse.** `_date_samples(column)` returns at most
  `_MAX_DATE_SAMPLES` (50) **non-blank** values of `column` (blank cells skipped,
  not counted toward the 50) — a statement's date column is uniform, so 50 is
  ample and the read stops once 50 are collected. Detection is **format-outer** (loop the 15 candidate formats, each re-scanning
  the ≤ 50 samples — the order `detect_date_format`'s `Sequence` requires, and it
  keeps CPython's small `_strptime` regex cache warm rather than thrashing it),
  bounded at ≤ 50 × 15 = 750 `strptime` calls (a few milliseconds — well under a
  single 16 ms UI frame), so the map step stays responsive on a large file. The read is a
  small private `_date_samples` in the wizard using `csv.DictReader` over
  `self._text` (no new importer surface; `read_header` already establishes the
  column names). `_autodetect_date_format` (D5) and `_update_date_preview` (D6)
  both consume it.

## New symbols (signatures)

```python
# importers/date_detect.py  (pure — no Qt, no vault)
# (example rendering of ref date 2026-07-20, strptime pattern) — ORDERED (D2);
# the order is the ambiguity tiebreak (INV-2).
KNOWN_DATE_FORMATS: list[tuple[str, str]] = [ ("2026-07-20", "%Y-%m-%d"), ... ]

@dataclass(frozen=True)
class DateFormatGuess:
    fmt: str | None      # best-guess strptime pattern, or None if nothing parsed any sample
    ambiguous: bool      # >=2 formats tie for the max parse-count (> 0)

def detect_date_format(samples: Sequence[str]) -> DateFormatGuess:
    """Pure best-effort guess of the strptime format for a column of date strings.
    `samples` is a Sequence (NOT a one-shot Iterable): the scan is format-outer —
    each candidate re-scans all samples (D8) — so a generator would exhaust after
    the first format and corrupt the count. A sample counts for a format iff
    strptime succeeds (no year window, no clock). fmt = the highest parse-count
    format; ties break by KNOWN_DATE_FORMATS order (INV-2); None if nothing parses
    any sample. Deterministic."""

# ui/import_wizard.py (new privates; self._date_format becomes a QComboBox)
_MAX_DATE_SAMPLES = 50      # detector sampling bound (D8)
_PREVIEW_SAMPLES = 3        # how many parsed dates the live preview shows (D6)
_CUSTOM_FORMAT = object()   # sentinel data for the "Custom…" combo entry

self._date_ambiguous: bool  # initialised False in __init__; set by _autodetect
                            # (True => show the day/month nudge); cleared by any
                            # manual format change (D6)

def _selected_date_format(self) -> str: ...      # combo data, or the custom field text
def _date_samples(self, column: str) -> list[str]: ...   # up to _MAX_DATE_SAMPLES non-blank values of `column`
def _autodetect_date_format(self) -> None: ...   # detect on the selected date column, seed the picker + _date_ambiguous (D5)
def _update_date_preview(self) -> None: ...      # refresh the "Dates read as: …" label (D6)
@Slot(int)
def _on_date_column_changed(self, index: int) -> None: ...   # user date-column change -> re-detect (D5)
@Slot(int)
def _on_date_format_changed(self, index: int) -> None: ...   # user picker change -> clear _date_ambiguous, show/hide custom + preview (D6)
```

## Test plan

**Pure detector (`detect_date_format`) — no Qt, plain strings (the heart):**
- Day-first numeric with a **disambiguating** day > 12 (`["20/07/2026",
  "21/07/2026", "31/07/2026", "02/08/2026"]`, the tester's shape) → `fmt =
  "%d/%m/%Y"`, `ambiguous = False` (month-first fails "20"/"21"/"31").
- ISO (`"2026-07-20"`) → `%Y-%m-%d`; slashes-ISO → `%Y/%m/%d`.
- Month-first US with a disambiguating **second** field > 12 (`"07/20/2026"`) →
  `%m/%d/%Y`, not ambiguous.
- **Dotted, both directions:** `"20.07.2026"` → `%d.%m.%Y`; `"07.20.2026"` →
  `%m.%d.%Y`; a dotted all-days-≤-12 column (`["05.06.2026", "07.08.2026"]`) →
  `ambiguous = True` (the `%m.%d.%Y` counterpart exists, so the tie — and the
  nudge — fire, matching the slash/dash behaviour; no silent day-first).
- **2-digit-year ambiguity:** an all-days-≤-12 2-digit column (`["05/06/26",
  "07/08/26"]`) → `ambiguous = True` (`%d/%m/%y` vs `%m/%d/%y` tie), so the `%y`
  variants get the same nudge as their 4-digit counterparts.
- **Ambiguous** all-days-≤-12 (`["05/06/2026", "07/08/2026"]`) → `ambiguous =
  True`; `fmt` is the **fixed-order** winner (`%d/%m/%Y`, ahead of `%m/%d/%Y`),
  deterministic across runs and input order.
- Named-month: `"20 Jul 2026"` → `%d %b %Y`; `"20 July 2026"` → `%d %B %Y`;
  `"Jul 20, 2026"` → `%b %d, %Y`; `"20-Jul-2026"` → `%d-%b-%Y`.
- **2-digit vs 4-digit year cleanly separated (no guard):** `["20/07/26"]` → `fmt
  = "%d/%m/%y"`, `ambiguous = False` — the 4-digit `%d/%m/%Y` **rejects** the
  2-digit year (`%Y` needs four digits), and conversely a 4-digit `["20/07/2026"]`
  is rejected by `%d/%m/%y` ("unconverted data remains"), so the two never
  cross-claim. (A dashed/dotted 2-digit string like `"20-07-26"` matches **no**
  candidate → `fmt is None`, since the known 2-digit formats are slash-only,
  out-of-scope note below.)
- **No match:** `["not a date", "banana"]` → `fmt is None`, `ambiguous = False`.
- **Empty / blank:** `[]` and `["", "  "]` → `fmt is None` (no samples).
- **Mixed / one junk row:** a clean day-first column with **one** garbage cell →
  the correct format still wins on parse-count (guards against a single bad row
  vetoing the guess); `fmt` correct.
- **Determinism:** the same `samples` yield an identical `DateFormatGuess` across
  calls and regardless of sample order (INV-2).
- **Clock-free (INV-2 falsifier):** a far-past and a far-future day-first column
  (`["20/07/1998", "21/07/1998"]` and `["20/07/2099", "21/07/2099"]`) each yield
  `fmt = "%d/%m/%Y"` — a hidden `date.today()`/year-window dependency would flip a
  decades-out year and fail this, so the "no clock, no year window" claim is
  falsifiable, not just asserted. (The sibling "no `locale.setlocale`, no
  randomness" clauses of INV-2 are code-review-verified — the detector calls
  neither — rather than test-gated.)

**Importer (`CsvImporter.parse`) — friendly date error (D3/INV-3):**
- A row whose date doesn't match `mapping.date_format` yields
  `RowError(reason == 'could not read the date "<raw>"')` — the reason contains the
  raw value and **not** the raw `strptime` internals: assert absence of
  `"does not match format"` and of a format token (`"%Y"`, `"%d"`, …), **not** a
  bare `"%"` (a junk cell like `"50%"` embeds a literal `%` in `<raw>` without any
  parser leak). A blank date cell (`raw == ""`) yields the `"the date cell is
  empty"` message (D3).
- An **amount** failure (non-numeric, or both debit+credit populated) keeps its
  existing human `str(exc)` message unchanged — the split didn't regress
  amount-error text.
- A valid row still parses (no behaviour change for the happy path).

**Wizard (`ImportWizardWidget`, `qtbot`):**
- **Auto-detect seeds the picker:** load an unmatched CSV whose dates are
  day-first (day > 12) → the map step's `import_date_format` combo is pre-selected
  to the `%d/%m/%Y` entry (not the old `%Y-%m-%d` default), and the preview step
  then shows the dates parsed (no error rows). The **regression** for the tester's
  bug: the same fixture imported end-to-end lands rows instead of 165 errors.
- **Live preview (D6):** the `import_date_preview` label shows `Dates read as: …`
  with correctly-parsed ISO samples; switching the picker to a wrong format
  flips it to the "couldn't be read" message; an all-days-≤-12 fixture shows the
  ambiguity nudge, and **manually** switching the picker to another format
  **clears** that nudge (it is not stale against a hand-chosen format).
- **Two preview fallbacks (D6):** an **all-blank** date column (`_date_samples`
  → `[]`) shows `tr("No dates found in this column.")`; a **non-date "junk"**
  column (non-blank cells that don't parse) shows `tr("These dates couldn't be
  read with this format — pick another.")` — **never** an empty or half-parsed
  "Dates read as:" tail; in both, auto-detect leaves the picker at the ISO default
  (D4).
- **Short column clean preview (D6, <3 samples):** a valid column with only **2**
  date rows shows the clean `Dates read as: …` branch (both parsed), **not** a
  fallback — the "all of the shown samples" gate, not a literal count of 3.
- **Preview refreshed once (D5 ownership):** a spy/call-count asserts
  `_update_date_preview` fires **exactly once** per auto-detect fire point (the
  caller owns it; `_autodetect_date_format` does not also refresh — no double
  refresh regresses the "one owner" contract).
- **Custom round-trip (INV-4):** selecting "Custom…" reveals `import_date_format_custom`
  and `_mapping_from_form` reads its text; a **saved profile** whose `date_format`
  is an exotic pattern not in `KNOWN_DATE_FORMATS` (e.g. `"%Y.%m.%d"`) applied via
  `_apply_profile_to_combos` selects "Custom…" and fills the field verbatim — the
  mapping the form produces equals the stored format (no silent rewrite).
- **Empty Custom rejected (D4 / the 1900 trap):** selecting "Custom…" and leaving
  the field **blank** → `_on_map_next` surfaces the friendly `"choose a date
  format"` error and does **not** advance to preview; a `CsvImporter.parse` /
  `_validate_mapping` unit assertion locks that an empty `date_format` is rejected
  (so `strptime("", "")` → 1900-01-01 can never reach a committed row).
- **Matched profile wins (INV-4/D5):** applying a profile whose format **is** a
  known entry selects that entry and does **not** re-run auto-detect over the data.
- **Whole-import banner (D7):** a preview with `new_count == 0`,
  `duplicate_count == 0`, and ≥ 1 error shows `import_preview_banner`; a preview
  with ≥ 1 new **or** ≥ 1 duplicate hides it.
- **i18n (INV-5):** the added strings ("Custom…", the preview label templates, the
  banner) are `tr()`-wrapped (grep-style check as prior UI phases did); the combo
  entry **data** values are the fixed `%`-patterns.
- **PDF path (INV-6):** a generic (non-SB) PDF fixture whose extracted table is
  day-first reaches the map step with auto-detect applied (the PDF-serialised-to-
  CSV text is the same `self._text` the detector reads).
- **Date-column change re-detects (D5c):** on an unmatched multi-column file,
  switching `_column_combos["date"]` to a *different* date-bearing column re-seeds
  `import_date_format` for that column and refreshes the preview.
- **PDF table-switch re-detects (D5b):** a multi-table generic PDF whose two
  tables use different date layouts — switching `_pdf_table_combo` re-detects the
  format and refreshes the preview for the new table (no stale reading); switching
  from an ambiguous unmatched table to a *matched* table clears the nudge + refreshes.
- **INV-6 pipeline-untouched:** the happy-path importer test (a valid row still
  parses unchanged) is tagged `(INV-6)` as the direct no-regression anchor.
- **OFX / Standard-Bank PDF skip the picker (INV-6 suppression):** an OFX import
  and a recognised-SB-PDF import reach the preview step **without** showing
  `import_date_format` and **without** calling `_autodetect_date_format` (asserted
  as covered by / added alongside the existing FIBR-0007/0009 path tests).
- **Auto-detect writes nothing (INV-1):** after loading an unmatched file and
  auto-detect running (picker seeded, preview shown), the vault holds **zero**
  transactions for the target account — the irreversible write happens only on the
  explicit Import press — falsifying "detection commits nothing" directly.

## Out of scope (v1 — YAGNI)

- **English month names only** — `%b`/`%B` parse English (`strptime` is
  locale-sensitive and the detector deliberately does **not** mutate locale,
  INV-2). Localised month names are deferred to FIBR-0017 (i18n).
- **No per-row mixed formats** — one `date_format` per import (a statement's date
  column is uniform); a file mixing layouts row-to-row is not auto-handled.
- **No auto-resolution of a genuinely ambiguous column** (all day-numbers ≤ 12) —
  by design the user confirms via the live preview (INV-1); finbreak must never
  silently pick day-first vs month-first when the data can't tell them apart.
- **No time-of-day / datetime parsing** — dates only (the ledger stores a date).
- **No detection of the amount / debit-credit columns** — this fix is date-format
  only; column-role guessing is a separate future idea.
- **No change to OFX or the recognised Standard-Bank PDF path** — both skip the
  map step (self-describing) and never show a date-format box.
- **2-digit-year layouts are slash-only in the known list** (`%d/%m/%y`,
  `%m/%d/%y`) — a dash- or dot-separated 2-digit year (`20-07-26`, `20.07.26`) is
  reachable only via "Custom…", not auto-detected. The two slash variants cover
  the common case; the long tail is the Custom escape hatch's job (INV-4). A
  2-digit year uses Python's fixed 1969–2068 century pivot (`…/69` → 1969, `…/68`
  → 2068) — deterministic and clock-free (INV-2 holds), and shown in the live
  preview (INV-1) so a surprising century is caught before import.
