# FIBR-0105 — User-configurable amount display (negative sign + colour)

> **Status:** 📝 DRAFT — pending `/cold-eyes` (project `--max-loops 7`), then TDD.
> **Phase:** near-term user request (dogfooding v0.1.4). The Home **Amount**
> column renders negatives in accounting **parentheses** (`(ZAR25,000.00)`) — the
> QLocale default — with no colour. The user asked why some values are bracketed
> and requested a **Settings choice** over the style. **Both styles are offered on
> purpose:** brackets stay available so an accountant using the app keeps the
> notation they're familiar with, while minus is the friendlier default for a
> non-accountant. Ships in **v0.1.5** with the inline update notes.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Display-only; stored data is never mutated.** Sign/colour are applied at the display layer; `transactions.amount_minor` keeps its stored value. Test: render `HomeView`, then re-read the row through `TransactionService` and assert the stored amount is unchanged. | ADR-0003 (data integrity) |
| INV-2 | **Two independent prefs, persisted in the vault.** `amount_negative_style` (`"minus"` / `"brackets"`) and `amount_colour` (`"true"` / `"false"`) are `SettingsRepository` keys in the vault `settings` table (mirroring `auto_lock_minutes` / the FIBR-0083 datetime keys). **No schema migration.** Read on unlock; display is post-unlock only. Default (absent) = **`"minus"` + colour ON**. Test: round-trip via `AuthService.set_amount_prefs` → `AuthService.amount_prefs`. | FIBR-0055 / FIBR-0083 pattern; ADR-0003 |
| INV-3 | **Negative-sign style is a display choice; positives are unchanged.** `"brackets"` → the QLocale accounting format (`(ZAR25,000.00)`, today's behaviour); `"minus"` → the magnitude with a single leading `"-"` (`-ZAR25,000.00`, no parentheses). A positive amount renders identically under both. Test: `format_amount(Decimal("-25000"), "ZAR", "minus")` starts `-`, has no `(`/`)`; `…, "brackets")` has `(`+`)`, no leading `-`; a positive has neither under both. | user request |
| INV-4 | **Colour marks direction when on; nothing when off.** Colour ON → a **negative** amount cell is red (`_NEGATIVE_TEXT`), a **positive** cell green (`_POSITIVE_TEXT`), **zero** left at the default text colour; colour OFF → every amount cell is the default colour (no per-cell foreground set). Colour applies to the **Amount cell only**, never other columns. Test: render `HomeView` with colour ON — a negative row's Amount item foreground is `_NEGATIVE_TEXT`, a positive row's is `_POSITIVE_TEXT`; with colour OFF the item has no explicit foreground override. | user request |
| INV-5 | **Fail-safe on a bad/unknown stored pref.** An `amount_negative_style` not in `{"minus","brackets"}` resolves to `"minus"`; an `amount_colour` not in `{"true","false"}` resolves to `True` (ON) — the defaults, never a crash. Test: a bogus stored value yields exactly the default resolution. | robustness |
| INV-6 | **(i18n / geometry.)** Every new fixed-word label / combo item / checkbox uses `self.tr("literal")` (never `self.tr(variable)`); currency samples are data shown verbatim. The controls have **no fixed-geometry calls** — covered by the FIBR-0051 `test_INV10_no_fixed_geometry_in_new_ui` glob over `ui/*.py`. | `coding.md § 5.2`; FIBR-0051 INV-10 |
| INV-7 | **First-run collects the choices and persists them only when the vault is created** (mirrors FIBR-0083 INV-8 / base-currency timing): the wizard pre-fills the two controls with the defaults; a successful vault creation writes them via `AuthService.set_amount_prefs`; a cancelled first-run persists nothing. Test: a first-run happy path asserts the prefs are stored; a cancel asserts `set_amount_prefs` was **never called** (spy). | FIBR-0004 INV-5 |

## Design decisions

- **D1 — Mirror the FIBR-0083 display-prefs shape exactly.** A frozen
  `AmountPrefs` dataclass + `AuthService.amount_prefs()` / `set_amount_prefs()` over
  `SettingsRepository`, the shell reads it once post-unlock and passes it as a value
  to `HomeView`, and re-pushes on Settings Save. This is the established pattern
  (FIBR-0083 D3/D7) — reuse, not re-invent. *(No shared "DisplayPrefs" umbrella
  refactor: `AmountPrefs` and `DateTimePrefs` are separate frozen dataclasses,
  separate keys, separate controls — the Rule-of-Three isn't met by a second prefs
  group.)*
- **D2 — `_format_amount` gains a `negative_style` arg; colour lives at the call
  site.** The sign shaping is a pure string transform (`_format_amount(display,
  symbol, negative_style) -> str`): `"brackets"` returns
  `QLocale().toCurrencyString(float(display), symbol, decimals)` verbatim (today's
  path — the accounting parentheses); `"minus"` formats the **magnitude**
  (`abs(display)`) and prepends `"-"` for a negative (`display < 0`), so the locale's
  own negative pattern is bypassed. Colour is **not** in `_format_amount` (a string
  has no colour) — `HomeView.refresh` sets the Amount `QTableWidgetItem`'s foreground
  from `amount_prefs.colour` + the sign of `display`.
- **D3 — Colour constants readable on the app's dark theme (and acceptable on
  light).** `_NEGATIVE_TEXT = QColor(224, 108, 117)` (a soft red) and
  `_POSITIVE_TEXT = QColor(152, 195, 121)` (a soft green) — module constants in
  `home.py`, mid-tone so they read on both palettes. *(Palette-adaptive theming is
  FIBR-0014; these fixed tones are the pragmatic choice for a dark-default app,
  ADR-0002.)*
- **D4 — Settings + first-run controls follow the existing idioms.** A
  **combo** for the negative style (`settings_amount_negative`, two items:
  *Minus (−)* → `"minus"`, *Brackets ( )* → `"brackets"`; `userData` = the stored
  token, preselected via `select_combo_data`) and a **checkbox** for colour
  (`settings_amount_colour`), matching the auto-lock combo + update checkbox already
  in the dialog. Persisted **inside `_on_save`** under the same `VaultLockedError`
  guard as auto-lock / the datetime write. First-run writes them at the post-create
  `_on_derived` site (D6-style, FIBR-0083). Radio buttons are **not** used (the
  dialog has no radio idiom; a 2-item combo is consistent).

## Deliverables

1. **`src/finbreak/services/auth.py`** — `AmountPrefs` (frozen dataclass:
   `negative_style: str`, `colour: bool`); `AuthService.amount_prefs() ->
   AmountPrefs` (reads `amount_negative_style` default `"minus"`, `amount_colour`
   default `True`, with the INV-5 fail-safe) and `set_amount_prefs(prefs)` (writes
   both keys — `colour` as `"true"`/`"false"`). Mirrors `datetime_prefs` /
   `set_datetime_prefs`.
2. **`src/finbreak/ui/home.py`** — `_format_amount(display, symbol, negative_style)`
   (D2 sign logic); the `_NEGATIVE_TEXT` / `_POSITIVE_TEXT` `QColor` constants;
   `HomeView` gains an `amount_prefs: AmountPrefs` constructor param **after `prefs`,
   before `parent`** (defaults to `AmountPrefs("minus", True)` when omitted, so the
   ~12 direct-construction call sites stay valid — same low-churn default as
   FIBR-0083's `prefs`), consumed by `refresh()` (sign + per-cell colour), plus a
   `set_amount_prefs(prefs)` method that stores + re-renders.
3. **`src/finbreak/ui/settings.py`** — the negative-style combo + colour checkbox
   (D4), preselected from `service.amount_prefs()`, persisted in `_on_save` under the
   existing `VaultLockedError` guard.
4. **`src/finbreak/ui/first_run.py`** — the same two controls, pre-filled with the
   defaults, persisted at `_on_derived` after `complete_first_run` (INV-7).
5. **`src/finbreak/ui/main_window.py`** — read `self._amount_prefs =
   self._service.amount_prefs()` once post-unlock in `_build_workspace` (beside the
   FIBR-0083 `self._prefs` read), pass it to `HomeView`, and on Settings Save re-read
   + push via `HomeView.set_amount_prefs` (beside the existing datetime push).
   `StatementsWidget` is **not** touched (no amount column).
6. **Tests** — `tests/features/app_shell/` (the `_format_amount` sign legs + a
   `HomeView` colour render leg — INV-3/4, and the INV-1 no-mutation leg);
   `tests/features/settings/` (the prefs round-trip INV-2/5 + the Settings combo +
   checkbox qtbot legs); `tests/features/first_run/` (INV-7 happy-path persist +
   cancel spy, via the synchronous `DeriveWorker` stand-in); the geometry glob
   (INV-6).
7. **Docs** — CHANGELOG entry (Added/Changed); this ships in the already-open
   `[0.1.5]` block.

## Data model (vault `settings` keys; no schema change)

| key | value | default (absent) |
|-----|-------|------------------|
| `amount_negative_style` | `"minus"` or `"brackets"` | `"minus"` |
| `amount_colour` | `"true"` or `"false"` | `"true"` (ON) |

## Out of scope

- **Per-account / per-currency amount rules** — a single base currency today
  (FIBR-0087 multi-currency later).
- **Palette-adaptive colour** (re-tinting red/green to the OS light/dark theme) —
  FIBR-0014; fixed mid-tones here (D3).
- **Colouring other columns / a running-balance column** — Amount cell only.
- **Number-locale formatting** (grouping/decimal separators) — already via QLocale
  (FIBR-0017 owns any change).

## Exit criteria

- `./scripts/ci-local.sh` green (ruff/format/bandit/pip-audit/gitleaks + mypy 0 +
  `pytest -m "not perf"`).
- Home Amount shows the chosen sign style and, when colour is on, money-out red /
  money-in green; changing either pref in Settings updates the open Home tab live;
  the stored amounts are untouched.

## Dependencies

- FIBR-0083 (the display-prefs pattern this mirrors; `HomeView` already carries a
  `prefs` value + a live-push on Save) — **shipped**.
- No new runtime dependency (QLocale / QColor are Qt-core/-gui, already used).
