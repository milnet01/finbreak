**Theme:** today the Home dashboard leads with the charts — a spending donut and a
12-month income-vs-spending bar chart — and the plain-language **breakdown** (the
FIBR-0138 openable Income / Spending / Transfers tree) sits at the very bottom. The user
wants that inverted: **the breakdown is the star.** This rework lays the dashboard out as
**three side-by-side columns** — **Expenditure**, **Income**, **Transfers** — each a
self-contained card: a small pie on top, then a bold coloured heading with the big total,
then the expandable breakdown for *that* branch (category → shop → transactions). A slim
**Net** line sits above the columns; a full-width **Recurring money** card (the deferred
FIBR-0142 card) and the demoted **monthly-trend** bar strip sit below. **No money figure
changes** — every total, slice, and row still comes from the same integer-exact
aggregations; this is purely how they are arranged on screen.

## Design decisions

- **D1 — Three-column layout replaces the single stack (`ui/home.py`).** `_build_dashboard`
  keeps its `QScrollArea` wrap and the selectors row, then lays out **top to bottom**:
  1. the **selectors** row (`_build_selectors`, unchanged);
  2. a slim **Net strip** (D4) — one full-width line;
  3. a **three-column** row: a `QHBoxLayout` holding three equal-stretch column widgets
     (`objectName` `dashboard_col_expenditure` / `dashboard_col_income` /
     `dashboard_col_transfers`), each a `QVBoxLayout` of **pie → header → breakdown tree**
     (D2/D3/D7);
  4. the full-width **Recurring** card (D5);
  5. the full-width **monthly-trend** strip (D6).

  The three-tile row (`_build_tiles`) is **removed** — the Income/Spending totals move
  into their column headers; the `dashboard_income` / `dashboard_expenditure` label
  objectNames are retired, while **`dashboard_net` is re-homed** to the Net strip (D4 — the
  one retained tile objectName). Their tests move to the new surfaces (Test-plan ripple).
  The single `dashboard_category_chart` /
  `dashboard_category_empty` / `dashboard_drilldown` are likewise retired in favour of
  the three per-column pies + trees. Per the user directive (INV-9) the columns carry
  **no** borders and uniform row heights. Each column widget gets a stable `objectName` so
  qtbot can find it, and so does every child the tests locate: the column widgets
  `dashboard_col_{expenditure,income,transfers}`, the pies
  `dashboard_pie_{expenditure,income,transfers}` (a `QChartView`, kept **present but
  hidden** on an empty branch — the `_render_donut` pattern, D8), the per-column empty
  placeholders `dashboard_pie_empty_{…}` (the `tr("No data for this period")` label, shown
  iff the pie is hidden), the header **name** labels `dashboard_heading_{…}` and header
  **total** labels `dashboard_total_{…}`, and the trees `dashboard_breakdown_{…}`.

- **D2 — One `drill_down()` feeds all three columns; `summary()` feeds only the Net strip;
  no new service code.** `refresh()` makes exactly these reads (all already existing):
  `drill_down(prefs, account_ids, labels=labels)` (the three branch nodes → each column's
  **header total** `node.amount`, **pie** from `node.children`, and **breakdown tree** from
  `node`), `summary(prefs, account_ids)` (**only** the Net strip's `Summary.net`, D4 —
  the Income/Spending headers come from the drill nodes, which equal `Summary` by FIBR-0138
  INV-1, so the strip reconciles), `monthly_trend(prefs, account_ids)` (the trend strip),
  and `self._recurring.summary(date.today())` (the card).
  **Node → column map (load-bearing — `drill_down` order ≠ display order).** `drill_down`
  returns `[Income, Spending, Transfers]` = `nodes[0], nodes[1], nodes[2]`, but the columns
  render in the mockup's order **Expenditure, Income, Transfers**. So the mapping is
  explicit, **not** a positional `zip`: the **Expenditure** column ← `nodes[1]` (Spending),
  the **Income** column ← `nodes[0]` (Income), the **Transfers** column ← `nodes[2]`
  (Transfers). Each column's branch colour (Spending `_NEGATIVE_TEXT`,
  Income `_POSITIVE_TEXT`, Transfers `None` = the default text colour, all gated on
  `amount_prefs.colour`) travels with its node and applies to the **header + tree only**
  (D7), so the green/positive Income node can never render inside the red Expenditure
  column. Each pie is fed `[(c.label, c.amount) for c in node.children]`; its **wedges take
  `_DONUT_PALETTE` colours, never the branch sign colour** (D3 — `build_breakdown_donut`
  has no colour parameter). `spending_by_category` is **no longer called from Home** (only
  `pdf_export` uses it now — it stays on the service). **New** service methods:
  **zero** — every number already exists (coding.md § 1.1).

- **D3 — `build_breakdown_donut` (`ui/charts.py`), a sibling of the spending donut.** A
  new builder draws a column's pie from generic `(label, amount)` slices:

  ```python
  def build_breakdown_donut(
      slices: list[tuple[str, Decimal]],   # a branch node's children, pre-sorted (FIBR-0138 INV-7)
      other_label: str,                    # tr("Other") — the collapsed-tail wedge
      theme: ChartTheme,
  ) -> QChart:
  ```

  It reuses the **same** module constants the spending donut uses — `_MAX_WEDGES` (8),
  `_OTHER_COLOUR`, `_DONUT_PALETTE` — and the `setHoleSize(0.4)` donut hole (so the column
  charts read as the same "pie" the dashboard already shows — "pie" throughout this spec
  means the donut `build_breakdown_donut` draws; there is no separate flat-pie builder),
  applied through the shared `_themed_chart`. It does its **own** cap-and-collapse loop:
  when `len(slices) ≤ _MAX_WEDGES` (8) keep them **all** (no Other wedge); otherwise keep
  the top `_MAX_WEDGES − 1` slices and sum the tail into one **Other** wedge, colouring the
  kept slices from the palette. **No** helper is extracted from `_donut_wedges`, so
  `_donut_wedges` / `build_donut_chart` / `spending_by_category` stay **byte-for-byte
  unchanged** (D10) and the PDF-export donut is provably untouched. This is the deliberate
  `coding.md § 1.3` call — Rule of Three extracts a helper at the **third** call-site, not
  the second; the two donut shapes also genuinely differ (see next), so a small,
  self-contained cap loop in `build_breakdown_donut` is the smaller, clearer change than a
  shared helper carrying a "reserve-a-wedge-for-Uncategorised" parameter. The difference:
  `build_breakdown_donut` does **not** carry the FIBR-0012 `Uncategorised`-by-`category_id`
  neutral-grey special case — a `DrillNode` child has no category id (it is a display
  model), and identifying Uncategorised by its display **name** is exactly what FIBR-0138
  **INV-4** forbids — so the Uncategorised slice competes for the 7 kept slots and takes
  an ordinary palette colour like any category. This is a deliberate, minor cosmetic
  change to the Expenditure pie (a grey Uncategorised wedge becomes a palette-coloured
  one), the price of the coherent "pie mirrors the list" design (INV-4); the authoritative
  spending donut in the **PDF export** keeps its `build_donut_chart` neutral-grey
  treatment, unchanged. No new dependency (QtCharts already imported). `HomeView` passes the
  `other_label` as `self.tr("Other")` — the **existing** donut tail label it already passes
  into `build_donut_chart` (`home.py:377`), not a newly-minted string, which is why INV-7's
  new-strings list omits it.

- **D4 — The Net strip (`ui/home.py`).** A slim full-width `QLabel`
  (`objectName "dashboard_net"` — the one retained tile objectName, re-homed) above the
  columns, showing `tr("Net")` + `_format_amount(summary.net, symbol,
  amount_prefs.negative_style)`, coloured by the FIBR-0105 rule (positive → `_POSITIVE_TEXT`,
  negative → `_NEGATIVE_TEXT`) when `amount_prefs.colour` is on, else uncoloured — the
  identical logic `_render_tiles` uses for the Net tile today (moved, not rewritten).

- **D5 — The recurring card (`ui/home.py` + `ui/main_window.py`).** `HomeView.__init__`
  gains a **required** `recurring: RecurringService` parameter (the card always renders, so
  it is not defaulted), inserted **after `auth`, before `amount_prefs=None`**. **Caller
  hazard:** because it lands before the already-defaulted `amount_prefs`, any caller passing
  `amount_prefs` **positionally** (the `main_window` call site at `:588` does) must move it —
  otherwise `amount_prefs` silently binds to `recurring`. `main_window` constructs
  `RecurringService(self._service.vault)` at the `HomeView(...)` call site (mirroring the
  `ReportingService` line above it); the shared test fixture passes `recurring=` **by
  keyword** to sidestep the shift. A full-width card (`objectName "dashboard_recurring"`)
  below the columns renders `self._recurring.summary(date.today())`: a `tr("Recurring")`
  heading and a `tr("Per month")` row of three `_format_amount` figure labels with pinned
  objectNames — `dashboard_recurring_in` / `dashboard_recurring_out` /
  `dashboard_recurring_net` (so the qtbot INV-5 test can grab each against
  `summary(today)`), labelled `tr("In")` / `tr("Out")` / `tr("Net")`. **Unscoped (INV-3):**
  `summary(today)` takes **no** prefs/account_ids, so the card shows **all** confirmed
  recurring money vault-wide regardless of the Home period/account selectors — a deliberate
  choice (recurring money is a standing-arrangement view, not period-bound); scoping it to
  the account selector would need a service change and is Out of scope.
  `monthly_in` / `monthly_out` are both **positive magnitudes** (`_summarise` sums positive
  `monthly_equivalent`s, `recurring.py:261`; `net = monthly_in − monthly_out` may be
  negative). So the colours are **forced by role, not derived from sign** (when
  `amount_prefs.colour` is on): In → `_POSITIVE_TEXT`, Out → `_NEGATIVE_TEXT` (an outflow,
  though its value is positive), Net → the FIBR-0105 sign rule (`_POSITIVE_TEXT` if
  `net >= 0` else `_NEGATIVE_TEXT`); each amount uses `amount_prefs.negative_style` via
  `_format_amount`.
  When **`monthly_in == 0 and monthly_out == 0`** (no confirmed item contributes) the card
  shows a single `tr()`-ed hint line ("Confirm recurring items on the Recurring tab to see
  them here.") instead of a row of zeros. The gate is the two **directional** totals, not
  `net`: equal confirmed in and out cancel to a zero `net` while still being real recurring
  money, so `net == 0` must **not** trigger the hint (INV-5). The card is **read-only** —
  confirm/dismiss stays on the Recurring tab (INV-5; Out of scope). The widget passes
  `date.today()` explicitly because `RecurringService.summary(today)` has **no** default
  (`recurring.py:246`) — unlike the `summary` / `drill_down` / `monthly_trend` reads, which
  default `today=None` internally; the service takes `today` for testability and already
  has its own `today`-injected unit tests from FIBR-0142, so the widget is not that seam.

- **D6 — The trend strip (`ui/home.py`).** The 12-month `build_trend_chart` bar chart
  (unchanged builder, unchanged `monthly_trend` feed) is the last, full-width element —
  demoted from its FIBR-0012 mid-page position to the bottom strip the mockup shows
  ("Monthly Trend"). The `dashboard_trend_chart` `QChartView` objectName is **retained**
  (the chart is the same, only its position changes), so existing trend-render tests
  survive with only a position-agnostic lookup.

- **D7 — Three per-column breakdown trees (`ui/home.py`).** Each column holds its own
  `QTreeWidget` (`dashboard_breakdown_{expenditure,income,transfers}`) rendering **one**
  branch node's **children** (the branch's own header row is not repeated — the column
  header shows the branch name + total). Each tree reuses the FIBR-0138 rendering: a
  single-column (or Name+Amount) read-only tree, `setSortingEnabled(False)` (the service
  pre-sorts, FIBR-0138 INV-7), populated by the **same** recursive `_add_node(parent_item,
  node, branch_colour=None)` (its existing signature — the per-column call passes the
  branch colour positionally) — the merchant/pair `×{count}` template, the `_format_amount`
  amount cell, and the branch colour (Expenditure `_NEGATIVE_TEXT`, Income `_POSITIVE_TEXT`,
  Transfers default) threaded down (FIBR-0138 D7). Both header labels (name **and**
  total) and the tree's branch colour carry that column's branch colour, **gated on
  `amount_prefs.colour`** exactly as FIBR-0138's `_render_drilldown` gates it
  (`_POSITIVE_TEXT` for Income / `_NEGATIVE_TEXT` for Spending / the default text colour for
  Transfers when the pref is on, else the default text colour throughout) — the colour is a
  display choice, never the data. This mirrors the retired tiles, which sign-coloured their
  value labels (`_render_tiles`, `home.py:337`); here the whole header (name + big total)
  takes the branch colour, matching the mockup's coloured column headings. `_render_drilldown` becomes
  `_render_column_breakdown(tree, node, branch_colour)` called once per column with that
  column's node; the old single-tree three-top-row structure is gone (the three tops are
  now the three column headers). Trees are independently expandable (INV-9). Each column
  header is **two stacked `QLabel`s** (pinned, so the tests have locators): a **name** label
  `dashboard_heading_{…}` above a **total** label `dashboard_total_{…}`. The header **text**
  is the branch `DrillNode.label` — i.e. the passed-in `DrillLabels` string
  (`tr("Spending")` / `tr("Income")` / `tr("Transfers")`), so the heading uses the app's
  **established** words and INV-7 stays true (headings travel via `DrillLabels`, no new
  string minted). The mockup's word "Expenditure" is the internal concept only — it names
  the objectName (`dashboard_col_expenditure`) and this spec's prose, but the **rendered**
  heading is the existing `tr("Spending")` (consistent with the tile / drill / trend
  legend the app already ships). The header **total** is the branch `DrillNode.amount`
  (the authoritative figure, D2), formatted by `_format_amount` with
  `amount_prefs.negative_style` — the same formatter the tree amounts and Net strip use.

- **D8 — Empty states (INV-4/INV-5).** A branch with no children → the column shows its
  header (zero-formatted total) + an empty tree, and its pie `QChartView`
  (`dashboard_pie_{…}`) is **hidden but still present** while its `dashboard_pie_empty_{…}`
  placeholder label (the shared `tr("No data for this period")` string, reused across all
  three columns — a **new** string, not the FIBR-0012 donut's `tr("No spending in this period")`) is **shown** — the exact `_render_donut` toggle the FIBR-0012 dashboard
  already uses (hide the chart, show the placeholder), so the `dashboard_pie_{…}` objectName
  always resolves for the INV-9 test whether or not the branch is empty. The recurring card
  with `monthly_in == 0 and monthly_out == 0` → the hint line (D5).
  The three columns are **always** present (the three headings always render), even when
  one direction is empty — matching FIBR-0138 INV-7's always-three-tops guarantee.

- **D9 — No new dependency; no schema; bundle-neutral.** `build_breakdown_donut` uses the
  already-imported `QtCharts`; the column widgets are `QWidget`/`QLabel`/`QTreeWidget`/
