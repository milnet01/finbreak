> **Kind:** feature.
> **Source:** user-request-2026-07-11 — the Statements **Imported** column showed
> a raw UTC ISO timestamp (`2026-07-11T06:49:15.506928+00:00`); the user asked to

> **Security: LOW.** Display-only. No new stored financial data, no network, no
> vault-schema change. The three prefs are a timezone id + two format tokens.
> **Roadmap:** [ROADMAP.md → FIBR-0083](../../ROADMAP.md). **Subsumes
> [FIBR-0048]** (user-configurable date-display format) — delivered here.
> **Governed by:** `docs/standards/{coding,naming,testing}.md` (`coding.md § 5.2`
> `tr()` / i18n for the new controls; `naming.md § 4` for the new module
> `datetime_format.py`).
> **Builds on:** [FIBR-0004](FIBR-0004.md) (the first-run wizard + the vault
> `settings` table + `SettingsRepository`), [FIBR-0055](FIBR-0055.md) (the
> Settings dialog + the `auto_lock_minutes` vault-settings pattern this mirrors),
> [FIBR-0052](FIBR-0052.md) (the Statements **Imported**/**Period** columns),
> [FIBR-0051](FIBR-0051.md) (the Home transaction-date column),
> [FIBR-0047](FIBR-0047.md) (the ISO date **input** pickers, whose Qt format
> tokens this reuses for **display**).

**Theme:** every date and timestamp finbreak shows renders in the user's chosen
**date format**, **time format**, and (for real timestamps) **time zone** —
auto-detected from the system by default, overridable in the first-run wizard and
in Settings. It is **display-only**: stored data stays UTC/ISO, so the change is
reversible and cannot corrupt anything.

**The core distinction (D2):** finbreak stores two kinds of "when":

- **Calendar dates** — `transactions.occurred_on`, `statement_periods` start/end
  (e.g. `2025-06-19`, date-only). These are **labels the bank already assigned in
  its own context**; a timezone does **not** apply. They are **reformatted only**,
  **never** timezone-converted (converting a date-only value risks shifting it to
  the wrong day and breaking the match to the printed statement).
- **Timestamps** — `statements.imported_at` (stored as `datetime.now(UTC)
  .isoformat()`, a real instant). These are **converted from UTC to the user's
  timezone** and then formatted (date + time). "When did I import this" is
  genuinely local-time-meaningful.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Display-only; stored data is never mutated.** Formatting is applied at the display layer; `occurred_on`, the `statement_periods` rows, and `imported_at` keep their stored UTC/ISO values. A test reads a row before and after a formatted display and asserts the DB value is unchanged. | ADR-0003 (data integrity) |
| INV-2 | **Calendar dates are reformatted, NEVER timezone-converted.** `format_date(iso, date_pref)` takes **no** timezone and returns the same calendar day in the chosen format. A test: `2025-06-19` under any timezone pref stays `2025-06-19`'s day (only the format changes); the function signature has no tz parameter. | statement fidelity (D2) |
| INV-3 | **Timestamps are converted UTC → the user's timezone, then formatted.** `format_timestamp(iso_utc, tz_pref, date_pref, time_pref)` shifts the instant into the resolved zone before formatting. A test: `2026-07-11T06:49:15+00:00` in `Africa/Johannesburg` (UTC+2) with `yyyy/MM/dd`+24h → `2026/07/11 08:49`. | user request |
| INV-4 | **Defaults are auto-detected from the system; zero-config works.** An absent pref (or the explicit sentinel `"system"`) resolves at display time to the current system timezone (`QTimeZone.systemTimeZoneId()`) and the system locale format (`QLocale.system()`), so a fresh vault shows sensible values with no choice made — and **follows** the OS if the user later changes it. An **explicit** selection is pinned. | user request ("auto-detect, allow override") |
| INV-5 | **The three prefs persist in the vault, hold no sensitive data.** `timezone`, `date_format`, `time_format` are `SettingsRepository` keys in the vault `settings` table (mirroring `auto_lock_minutes`), each a short opaque string (a zone id or a format token / `"system"`). **No schema migration** — the `settings` table is v1 key-value. Read on unlock; display is post-unlock only. A test round-trips each via `set_datetime_prefs` → `datetime_prefs`. | FIBR-0055 (auto-lock pattern); ADR-0003 |
| INV-6 | **Fail-safe on a bad/unknown stored pref.** An unrecognised timezone id or format token resolves to the system default rather than crashing or showing a garbled value; an unparseable stored date/timestamp string renders as its raw stored text (never an exception to the UI). A test feeds a bogus zone id and a bogus token and asserts a sane fallback string, no raise. | robustness |
| INV-7 | **(i18n / geometry.) The new controls are translatable and layout-managed.** Every new label/combo item that is a fixed word uses `self.tr("literal")` (never `self.tr(variable)`; timezone ids and format samples are data, shown verbatim); the controls use layout managers, no fixed geometry — covered by the FIBR-0051 `test_INV10_no_fixed_geometry_in_new_ui` glob and the `coding.md § 5.2` review item. | `coding.md § 5.2`; FIBR-0051 INV-10 |

A test drives a first-run happy path and asserts the prefs are stored; a cancel asserts nothing is. | FIBR-0004 INV-5 |

## Context

**Verified current state (2026-07-11):**

- `statements.py::StatementsWidget.refresh` sets the **Imported** cell to
  `statement.imported_at` **verbatim** (the raw UTC ISO string) and the **Period**
  cell to `f"{statement.period_start} – {statement.period_end}"` (two raw ISO
  dates).
- `home.py` sets the **Date** cell to `transaction.occurred_on` **verbatim** (a
  raw ISO date).
- Stored values are all UTC/ISO: repositories write `datetime.now(UTC)
  .isoformat()` (timestamps) and `date...isoformat()` (dates).
- Persistence pattern: `SettingsRepository(connection).get/set(key)` (vault
  `settings` table, v1); `AuthService.auto_lock_minutes()` /
  `set_auto_lock_minutes()` read/write via it. `base_currency` is set at vault
  creation through `complete_first_run(...)`.
- The first-run wizard (`ui/first_run.py`) collects password + `base_currency`
  (a `QComboBox`) and, on submit, derives the key on a `DeriveWorker`, then
  `complete_first_run`. The Settings dialog (`ui/settings.py`, FIBR-0055) hosts an
  auto-lock combo + a read-only currency label + the FIBR-0054 update checkbox.
- **Qt API confirmed** (PySide6 6.11, headless): `QTimeZone.systemTimeZoneId()` →
  `Africa/Johannesburg`; `QTimeZone.availableTimeZoneIds()` → 643 ids;
  `QDateTime.fromString(iso, Qt.DateFormat.ISODateWithMs)` parses the stored
  microsecond+offset timestamp; `.toTimeZone(QTimeZone(b"…"))` converts;
  `.toString("yyyy/MM/dd HH:mm")` / `"…h:mm AP"` formats; `QDate.fromString(iso,
  Qt.DateFormat.ISODate).toString(token)` reformats a date; `QLocale.system()
  .toString(qdate, ShortFormat)` gives the system-locale format.

## Design decisions

- **D1 — Qt's date engine (`QDateTime`/`QDate`/`QTimeZone`/`QLocale`), not Python
  stdlib.** The format tokens (`yyyy/MM/dd`) are the **same** ones the FIBR-0047
  date **pickers** already use, so input and display stay consistent; and the
  calls are testable headless (no event loop). *Rejected:* `zoneinfo` + `strftime`
  (a second, `%Y`-style token vocabulary to map and keep in sync); a user-typed
  custom format string (error-prone, poor UX — curated presets instead, D5).

- **D2 — Dates reformatted only; timestamps timezone-converted.** The semantic
  split above. `format_date` has **no** timezone parameter by design.

- **D3 — Prefs live in the vault via `SettingsRepository`, not the plaintext
  `window.ini`.** Display happens **only post-unlock** (the Statements/Home tabs
  are vault-dependent), so the prefs never need pre-unlock access; keeping them in
  the vault mirrors `auto_lock_minutes` and `base_currency`. No schema migration
  (key-value `settings` table).

- **D4 — A `"system"` sentinel means "resolve dynamically at display time."** The
  stored pref is either the sentinel `"system"` (→ current `systemTimeZoneId()` /
  `QLocale.system()` at each render, so it **follows** the OS) or a pinned
  specific value (a zone id / a format token). Absent ⇒ treated as `"system"`.
  This is exactly "auto-detect by default, override to pin" (the user's ask). The
  first-run/Settings combos show **System default (<detected>)** as the first,
  pre-selected item.

- **D5 — Curated preset lists (wide variety), plus "System default".** Dates:
  System · `2026-07-11` (`yyyy-MM-dd`) · `2026/07/11` (`yyyy/MM/dd`) · `11/07/2026`
  (`dd/MM/yyyy`) · `07/11/2026` (`MM/dd/yyyy`) · `11 Jul 2026` (`dd MMM yyyy`) ·
  `11 July 2026` (`dd MMMM yyyy`) · `Jul 11, 2026` (`MMM dd, yyyy`). Times: System ·
  `14:30` (`HH:mm`, 24h) · `14:30:00` (`HH:mm:ss`) · `2:30 PM` (`h:mm AP`, 12h). A
  timestamp = the date format + `" "` + the time format, in the resolved zone.

- **D6 — First-run persists after the vault exists.** Settings can only be written
  once the vault is created, so the wizard writes the three prefs in the
  DeriveWorker-success path (right after `complete_first_run`), not before.
  `base_currency` stays on its existing `complete_first_run` path — untouched.

- **D7 — Fail-safe resolution (INV-6).** `QTimeZone(bad_id).isValid()` is `False`
  → fall back to system; an unknown format token is only ever chosen from our own
  preset list, but a stored value not in the list → fall back to `"system"`; an
  unparseable stored date/timestamp → render its raw text (defensive; our own
  writers always produce ISO).

## Deliverables

1. **`src/finbreak/datetime_format.py`** *(new, pure, Qt-core only)* — the
   formatter: `format_date(iso: str, date_pref: str) -> str`;
   `format_timestamp(iso_utc: str, tz_pref: str, date_pref: str, time_pref: str)
   -> str`; the preset tables (`DATE_PRESETS`, `TIME_PRESETS` — ordered
   `(stored_token, sample_label)`); `system_timezone_id() -> str`; and the
   `"system"`-sentinel resolution + INV-6 fallbacks. No Qt **Widgets** import (core
   only), so it is unit-testable without `qtbot`.
2. **`src/finbreak/services/auth.py`** — `DateTimePrefs` (frozen dataclass:
   `timezone`, `date_format`, `time_format`, each a stored string); `datetime_prefs()
   -> DateTimePrefs` (reads the three `SettingsRepository` keys, each defaulting to
   `"system"` when absent) and `set_datetime_prefs(prefs: DateTimePrefs)` (writes
   the three keys). Mirrors `auto_lock_minutes` / `set_auto_lock_minutes`.
3. **`src/finbreak/ui/settings.py`** — three controls added to `SettingsDialog`: a
   **timezone** combo (`settings_timezone`; first item *System default (<id>)* →
   `"system"`, then the sorted `availableTimeZoneIds()`; editable+completer for
   search), a **date-format** combo (`settings_date_format`), a **time-format**
   combo (`settings_time_format`) — each pre-selected from `datetime_prefs()`; on
   Save the shell persists them via `set_datetime_prefs`.
4. **`src/finbreak/ui/first_run.py`** — the **same** three controls, pre-filled
   with detected defaults (the `"system"` items selected); on a successful vault
   creation, persisted via `set_datetime_prefs` (D6).
5. **`src/finbreak/ui/statements.py`** — **Period** → `format_date` each endpoint;
   **Imported** → `format_timestamp`, using the shell's current `datetime_prefs()`.
6. **`src/finbreak/ui/home.py`** — **Date** → `format_date`.
7. **`src/finbreak/ui/main_window.py`** — pass the active `DateTimePrefs` to the
   Statements/Home widgets on build/refresh (they take the prefs, hold no
   `AuthService`), and to `SettingsDialog`/`FirstRunDialog`; persist on Settings
   Save (mirroring the FIBR-0055 currency/auto-lock wiring).
8. **Tests** — `tests/features/datetime_format/{spec.md,test_datetime_format.py}`:
   the helper legs (INV-2/3/4/6, hermetic + offscreen: date reformat with no
   tz-shift; UTC→+2 timestamp; `"system"` resolution; bad-zone/bad-token
   fallback); the prefs round-trip (INV-5); the display-only DB-unchanged leg
   (INV-1); and `qtbot` legs for the Settings + first-run controls (defaults
   detected, selection persisted — INV-8) and the geometry glob (INV-7).
9. **Docs** — mark **FIBR-0048 → ✅** (subsumed here) at close; annotate FIBR-0047
   that display now reuses its input tokens; CHANGELOG entry.

## Data model (vault `settings` keys; no schema change)

| key | value | default (absent) |
|-----|-------|------------------|
| `timezone` | `"system"` or an IANA id (e.g. `"Africa/Johannesburg"`) | `"system"` |
| `date_format` | `"system"` or a Qt date token (e.g. `"yyyy/MM/dd"`) | `"system"` |
| `time_format` | `"system"` / `"HH:mm"` / `"HH:mm:ss"` / `"h:mm AP"` | `"system"` |

## Out of scope

- **Per-transaction time-of-day** — transactions store a date only (`occurred_on`),
  no time; nothing to zone-convert there.
- **Multi-currency / per-account currency** — [FIBR-0087].
- **Amount/number locale formatting** — [FIBR-0017] (currency already via QLocale).
- **Custom user-typed format strings** — curated presets only (D5).
- **Changing the date-*input* pickers** — FIBR-0047 fixed those to `yyyy/MM/dd`;
  input parsing is untouched (this is display-only).
- **Correcting a timezone-bearing OFX `<DTPOSTED>` at import** — [FIBR-0042], an
  import-time calendar-date concern, not display.
