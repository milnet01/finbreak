**Theme:** every date and timestamp finbreak shows renders in the user's chosen
**date format**, **time format**, and (for real timestamps) **timezone** —
auto-detected from the system by default, overridable in the first-run wizard and
in Settings. It is **display-only**: stored data stays UTC/ISO, so the change is
reversible and cannot corrupt anything.

**The core distinction (D2):** finbreak stores two kinds of "when":

- **Calendar dates** — `transactions.occurred_on`, `statement_periods` start/end
  (e.g. `2025-06-19`, date-only). These are **labels the bank already assigned in
  its own context**; a timezone does **not** apply. They are **reformatted only**,
  **never** timezone-converted (converting a date-only value risks shifting it to
  the wrong day and breaking the match to the printed statement).
- **Timestamps** — `statement_periods.imported_at` (stored as `datetime.now(UTC)
  .isoformat()`, a real instant; surfaced on the Statements tab as
  `StatementRow.imported_at`). These are **converted from UTC to the user's
  timezone** and then formatted (date + time). "When did I import this" is
  genuinely local-time-meaningful.

## Design decisions

- **D1 — Qt's date engine (`QDateTime`/`QDate`/`QTimeZone`/`QLocale`), not Python
  stdlib.** The format tokens (`yyyy/MM/dd`) are the **same** ones the FIBR-0047
  date **pickers** already use (`ui/import_wizard.py:234`, `ui/manual_entry.py:50`),
  so input and display stay consistent; and the calls are testable headless (no
  event loop). *Rejected:* `zoneinfo` + `strftime` (a second, `%Y`-style token
  vocabulary to map and keep in sync); a user-typed custom format string
  (error-prone, poor UX — curated presets instead, D5).

- **D2 — Dates reformatted only; timestamps timezone-converted.** The semantic
  split above. `format_date` has **no** timezone parameter by design.

- **D3 — Prefs live in the vault via `SettingsRepository`, not the plaintext
  `window.ini`.** Display happens **only post-unlock** (the Statements/Home tabs
  are vault-dependent), so the prefs never need pre-unlock access; keeping them in
  the vault mirrors `auto_lock_minutes` and `base_currency`. No schema migration
  (key-value `settings` table). **Written dialog-side** — exactly like auto-lock:
  `SettingsDialog._on_save` calls `service.set_datetime_prefs(...)` (the dialog
  already holds `service`), and the first-run wizard calls it after
  `complete_first_run` (D6). *(This is deliberately the auto-lock pattern, NOT the
  FIBR-0054 update-flag pattern where the shell persists a network-free checkbox.)*

- **D4 — A `"system"` sentinel means "resolve dynamically at display time."** The
  stored pref is either the sentinel `"system"` (→ resolved at each render:
  timezone via `systemTimeZoneId()`, date/time via `QLocale.system()`, so it
  **follows** the OS) or a pinned specific value (a zone id / a Qt format token).
  Absent ⇒ treated as `"system"`. This is exactly "auto-detect by default,
  override to pin" (the user's ask). The first-run/Settings combos show **System
  default (&lt;detected&gt;)** as the first, pre-selected item, which stores
  `"system"`.

- **D5 — Curated preset lists (wide variety), plus a prepended "System default".**
  `DATE_PRESETS` / `TIME_PRESETS` (Deliverable 1) hold **only concrete `(token,
  sample_label)` rows**; the UI **prepends** a dynamic *System default
  (&lt;detected&gt;)* item that maps to the stored `"system"` sentinel (its label
  is dynamic, so it is not a static table row). Dates: System · `2026-07-11`
  (`yyyy-MM-dd`) · `2026/07/11` (`yyyy/MM/dd`) · `11/07/2026` (`dd/MM/yyyy`) ·
  `07/11/2026` (`MM/dd/yyyy`) · `11 Jul 2026` (`dd MMM yyyy`) · `11 July 2026`
  (`dd MMMM yyyy`) · `Jul 11, 2026` (`MMM dd, yyyy`). Times: System · `14:30`
  (`HH:mm`, 24h) · `14:30:00` (`HH:mm:ss`) · `2:30 PM` (`h:mm AP`, 12h).
  **Timestamp composition (the `"system"`-aware rule):** `format_timestamp` first
  resolves the **timezone** — `tz_pref=="system"` → the system zone
  (`QTimeZone(systemTimeZoneId())`, the same call the delegation test names); else
  `QTimeZone(tz_pref.encode())` (the stored `str` id is encoded to the
  `QByteArray` the constructor takes; `.isValid()` `False` → system, INV-6) —
  parses the UTC instant (`QDateTime.fromString(iso, ISODateWithMs)`) and
  `.toTimeZone(...)`s it into that zone; then resolves the two format halves
  **independently** and joins with a single space. Each half is **three-way**
  (this is what upholds INV-6): `pref=="system"` →
  `QLocale.system().toString(dt.date()|dt.time(), ShortFormat)`; `pref` is a
  **known preset token** (a token in the **first column** of
  `DATE_PRESETS`/`TIME_PRESETS`) → `dt.date().toString(token)` /
  `dt.time().toString(token)`; **any other value** (a stored token no longer in the
  preset set) → **system default** — never passed raw to `toString`, which would
  render unknown format chars as literal garble. So a `"system"` or unknown half is
  never concatenated as a raw token — each half is formatted, then joined.
  `format_date` parses via `QDate.fromString(iso, ISODate)` and applies the same
  **three-way** date-part rule (no time, no tz). **Unparseable input:** if
  `QDateTime.fromString` / `QDate.fromString` yields an **invalid** value (a
  malformed stored string — our own writers never produce one), the raw `iso`
  string is returned unchanged (INV-6), never `""`.

- **D6 — First-run persists after the vault exists.** Settings can only be written
  once the vault is created, so the wizard writes the three prefs at its
  `DeriveWorker`-success site (`ui/first_run.py:139`, `_on_derived`, right after
  `complete_first_run` — defined at `auth.py:127` — has opened the vault), via
  `service.set_datetime_prefs`.
  `base_currency` stays on its existing `complete_first_run` path — untouched.

- **D7 — One data-flow: the shell reads prefs once and passes them as a
  value.** The shell (`MainWindow`) **reads** `DateTimePrefs` once via
  `self._service.datetime_prefs()` (the `"system"` sentinels kept verbatim —
  "resolve" is reserved for the display-time expansion in D4/D5) and passes that
  **plain value** (not the
  `AuthService`) to the display widgets. `HomeView` and `StatementsWidget` each
  gain a `prefs: DateTimePrefs` constructor parameter **positioned before
  `parent`** — `HomeView(transactions, categorization, prefs, parent=None)` and
  `StatementsWidget(service, prefs, parent=None)` — with their positional call
  sites in `main_window.py` updated in lockstep. The param is consumed by their
  existing `refresh()` render loops, **and** each gains a `set_datetime_prefs(prefs)`
  method that
  stores the new value and re-renders. On **Settings Save**, the shell updates its
  held prefs and calls `set_datetime_prefs` on the open Home/Statements tabs, so a
  change takes effect immediately without a relaunch. The widgets keep their
  existing **data-service objects** (`StatementsWidget`'s
  `StatementService`/`AccountService`, built from `service.vault`; `HomeView`'s
  `TransactionService`/`CategorizationService`) for data — this feature only adds
  the formatting *input*, computed once by the shell and passed as a value. (INV-1
  forbids formatting *mutating* data, not the widgets reading it.)

## Performance note

`format_date` / `format_timestamp` are called **per cell** inside the
`refresh()` render loops (`ui/home.py`, `ui/statements.py`), each constructing a
`QDateTime`/`QTimeZone`. This is **deliberately un-cached**: the tables are
user-scrolled and hold tens–hundreds of rows, so the per-row cost is negligible
(no hot loop, no large N). No perf invariant is pinned; if a future dataset makes
this measurable, a resolved-pref cache is the obvious lever.
