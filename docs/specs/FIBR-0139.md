**Theme:** file transactions into sensible categories **the first time you import**,
before you have written a single rule. Today auto-categorise only matches
**your** rules (FIBR-0010), so a brand-new vault imports everything as
**Uncategorised** (the cold-start problem). This ships a bundled, per-release
**category library** — a plain list of *"if the description contains ‹merchant›,
file it as ‹category›"* entries, South-Africa-first (Pick n Pay, Shoprite, Shell,
Dis-Chem, Vodacom, Eskom…) plus universal (Netflix, Spotify, Uber, Steam…) — that
runs **after** your own rules and **only** on rows nothing else has claimed. Every
library guess is **marked as a guess** and is **always overridable**; a category
**you set by hand still wins** and is never touched. There is **no new button** and
**no schema change** — the library rides the existing import + "Apply rules now"
paths and the existing free-text `category_source` column.

**Contents:** At a glance (invariants) · Context · Design decisions (D1–D8) ·
Deliverables · Data model (no migration) · Test ripple · Out of scope · Exit
criteria · Dependencies · Verified code basis · Cold-eyes loop log.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| **INV-1** | **Money-safety (the headline).** Categorisation only ever writes `(category_id, category_source)`; it **never reads or alters** `amount_minor`, a row's existence, or its account. Every total, per-category sum, and net figure is **identical** before and after any library apply/import. | FIBR-0010 INV-12; design.md Atomicity |
| **INV-2** | **Your rules always win; the library is the fallback.** For any description, a matching **user rule** (FIBR-0010) takes precedence — the library is consulted **only** when no user rule matches. `categorize(desc, rules)` (user rules) is tried first; the library second. A user rule created to override a library guess therefore beats it on the next run (the FIBR-0140 bridge). | FIBR-0010 INV-2 |
| **INV-3** | **Manual always wins; the golden rule is untouched.** The library runs through the **same** `auto_rows` read set that excludes every **manual** row, so a hand-set category (or a manual **clear**) is never touched by a library guess. FIBR-0010 INV-1/INV-3 hold verbatim. | FIBR-0010 INV-1/INV-3 |
| **INV-4** | **Library rows are stamped `'library'` — no migration.** A row filed by the library carries `category_source = 'library'` (new `CategorySource.LIBRARY`); a rule row `'rule'`, a hand row `'manual'`, an untouched row `NULL`. `category_source` is already a **free-text `TEXT`** column (v7, no `CHECK`), and `auto_rows`'s `IS NULL OR <> 'manual'` predicate already recomputes a `'library'` row — so **no schema change**. | FIBR-0010 D1 (free-text column) |
| **INV-5** | **Runs on the existing paths only.** The library is consulted inside the **same** `recategorize_auto_rows(conn)` that import (`commit_import`), the Rules-tab **"Apply rules now"**, and the delete-category cascade already call — no new trigger, no new button. It is likewise folded into `would_categorize` so confirming a library guess raises **no** learning nag (a spurious "make a rule?" prompt for something the library already handles) and overriding one **does** offer the rule. | FIBR-0010 INV-4/INV-5 |
| **INV-6** | **Bind by name; rename-safe, never mis-filed.** Each library entry names its category by **display name**, resolved to an id at apply time against the current **leaf** categories. An entry whose category name **no longer exists** (the user renamed or deleted that default) is **skipped**, so the row falls through to **Uncategorised** — it is **never** mis-filed into some other category. | FIBR-0010 INV-9 (leaves only) |
| **INV-7** | **Off switch (Settings toggle, default ON).** A vault setting `auto_category_library` (default **ON** when absent) gates the library. **Off** → the library is not consulted, and the **next** apply/import reverts every `'library'` row to Uncategorised (they are non-manual, so in the auto read set); a user's own rules and manual picks are unaffected. Stored via `SettingsRepository` — **no schema change**. Toggling does **not** itself re-file (the change lands on the next run — matches the approved design; immediate re-apply is out of scope). | FIBR-0055 D3 (settings seam) |
| **INV-8** | **Fail-safe load.** A **missing, unreadable, or structurally-invalid** library file yields an **empty** library and the app runs normally (just no built-in guesses) — it never raises into import/apply. A single malformed **entry** (missing/blank `pattern` or `category`) is skipped, not fatal to the rest. | coding.md § 1.1 (fail-safe default) |
| **INV-9** | **Every guess is marked, and overridable.** In the **Transactions** tab, a `'library'` row's Category cell shows a small **"~ guess"** tag + an explanatory tooltip; a rule/manual row shows the plain name. The marker is display-only — the category **filter** (keyed on `category_id`) is unaffected — and **Set category…** clears it (the row becomes `'manual'`). *(The dashboard `HomeView` has no per-row cell; a library row carries a real `category_id`, so it aggregates into the spending donut normally — see D7 on the roadmap-D6 wording correction.)* | FIBR-0010 D10 (overridable); FIBR-0012 (dashboard) |
| **INV-10** | **Idempotent; no re-dedup.** A second apply with unchanged rules + library changes **nothing** (`set_category` writes only rows whose value actually changes). The engine issues only `UPDATE`s of the two category columns — never `INSERT`/`DELETE` — so the transaction **row count** is identical before and after (this is the mechanism behind INV-1). | FIBR-0010 INV-12/INV-13 |
| **INV-11** | **No network; i18n.** The library ships **in the bundle**; nothing is fetched. No new `urllib` importer (FIBR-0054 INV-12 scan stays green). The one new user string is `tr()`-wrapped; every widget stays in a layout manager. | coding.md § 5.2; FIBR-0054 INV-12 |

## Context

- **The engine (FIBR-0010).** `categorize(description, rules) -> int | None`
  (`services/categorization.py:36`) is the pure first-match matcher: normalise both
  sides (`normalise_text`), return the first rule whose pattern is a **substring** of
  the description — no wildcards, no regex. `recategorize_auto_rows(conn) -> int`
  (`:47`) reads the rules once, iterates `TransactionRepository.auto_rows()`, and
  `set_category`s each row to `RULE`/`None`, returning the changed count. It is
  **commit-free**: `apply_rules()` (`:172`), `commit_import` (`import_.py:288`), and
  the delete cascade each own their transaction. `would_categorize(description)`
  (`:125`) is the learning "differs" check.
- **The source model.** `category_source` is a **free-text `TEXT`** column (v7, no
  `CHECK`); `auto_rows()` (`transactions.py:52`) reads `WHERE category_source IS NULL
  OR category_source <> 'manual'` — so any non-`'manual'` value (including a new
  `'library'`) is in the recompute set. `CategorySource(StrEnum)` (`models.py:62`)
  has `RULE`/`MANUAL`. `set_category(txn_id, category_id, source)` (`transactions.py:65`)
  writes **only if changed** and returns the rowcount (INV-10).
- **Default categories (v3 seed, `migrations.py`).** Income = Salary, Sales,
  Interest, Gifts, Lottery, Other income. Expenditure = Groceries, Fast food,
  Bills & utilities, Rent / Mortgage, Transport, Medical, Entertainment, Clothing,
  Insurance, Other expenditure. These are the names the library binds to (INV-6). The
  names are **data**, not `tr()` strings — a user may rename them, which INV-6
  handles by falling through.
- **The settings seam (FIBR-0055).** `SettingsRepository(conn).get/set(key, value)`
  over the v1 `settings(key TEXT PRIMARY KEY, value TEXT NOT NULL)` table; `set`
  owns its commit. `AuthService.auto_lock_minutes()`/`set_auto_lock_minutes()`
  (`auth.py:304/322`) and `amount_prefs` (`auth.py:364/380`, the `"true"/"false"`
  boolean idiom) are the precedent the toggle copies.
- **The Settings dialog + shell.** `SettingsDialog` (`ui/settings.py`) is
  constructed by the shell with flags (e.g. `update_enabled` / `update_supported`)
  and exposes read-backs (`update_enabled() -> bool`); the shell persists on
  `saved` (`main_window.py:734` construct, `:962` `_on_settings_saved` →
  `set_enabled(dialog.update_enabled())`). A `CategorizationService(self._service.vault)`
  is already in shell scope (`main_window.py:597`).
- **The tables.** `home.py` is the **dashboard** (FIBR-0012/0138 — summary tiles,
  category donut, trend chart; **no per-row category cell**). The per-transaction
  table with the **Category** column (`_COL_CATEGORY = 4`) and the *Set category… →
  learn-a-rule* chain is `ui/transactions.py`; its render loop
  (`transactions.py:230`) already binds `transaction` (so `transaction.category_source`
  is in hand) and the filter keys on `transaction.category_id` (`:275`).
- **Bundling (FIBR-0037/0051).** `ui/icons` ships as package-data
  (`pyproject.toml:93` `finbreak = ["ui/icons/*.svg", "ui/icons/*.png"]`), resolves
  at runtime via `Path(__file__).parent / "icons"` (`ui/icons.py:26`), and is pulled
  into the freeze with `--add-data "<src>:finbreak/ui/icons"`
  (`windows_freeze_flags.py:56` `ADD_DATA_TARGET`, `build-windows-exe.py:80/102`,
  `_build-smoke-in-container.sh:87`). The library file copies this exactly.

## Design decisions

- **D1 — The bundled, per-release library file (INV-8, INV-11).**
  `src/finbreak/data/category_library.json` — a JSON **array** of objects
  `{"pattern": "<substring>", "category": "<default category name>"}`, e.g.
  `[{"pattern": "pick n pay", "category": "Groceries"}, {"pattern": "netflix",
  "category": "Entertainment"}, …]`. **No `version` wrapper** and no per-entry extras
  (coding.md § 1.1 — the shape is exactly what the matcher reads; a schema-version
  field is added only when a real second shape appears). Patterns are stored
  lower-case (they are `normalise_text`-folded at match time anyway; lower-case keeps
  the file readable). It travels **like `ui/icons`**: a **second** package-data glob
  `"data/*.json"` and a **second** `--add-data` pair `<src>/finbreak/data →
  finbreak/data` in all three freeze paths (D-Deliverables 8–9). At runtime a loader
  resolves it via `Path(__file__).parent / "data" / "category_library.json"`
  (mirroring `ui/icons.py:26` — works from source **and** frozen). **Fail-safe:** any
  read/parse error, or a JSON value that is not an array, → empty library (INV-8),
  logged at `WARNING` (no raise into import/apply). The initial seed is SA-first +
  universal; the file is expected to **grow per release** (its whole point — we ship
  a better library over time with zero code change).

- **D2 — Two-layer match, user rules first (INV-2), reusing `categorize`
  (coding.md § 1.3).** New **pure** helpers:
  - `category_library.match_library(description, entries, name_to_id) -> int | None`
    — normalise the description once; return the id of the **first** entry whose
    `normalise_text(pattern)` is a substring **and** whose `category` name resolves
    in `name_to_id`; else `None`. (Same substring/`normalise_text` primitive as
    `categorize`; a separate 4-line function rather than a forced shared abstraction —
    the two call shapes differ, so Rule-of-Three does not bite.)
  - `categorization.categorize_with_library(description, rules, entries, name_to_id)
    -> tuple[int | None, str | None]` — **reuse** `categorize(description, rules)`
    for the rule layer: if it matches, return `(id, CategorySource.RULE.value)`;
    else `match_library(...)` → `(id, CategorySource.LIBRARY.value)`; else
    `(None, None)`. This single function is the one place the two layers compose, and
    both the apply path (D4) and `would_categorize` (D4) call it, so they can never
    diverge.

- **D3 — `CategorySource.LIBRARY = "library"` (INV-4).** One enum member added to
  `models.py:62`. **No migration** — the column is free-text and `auto_rows` already
  recomputes non-`'manual'` rows. (The FIBR-0010 verified-basis note that the column
  has no `CHECK` is the load-bearing fact here.)

- **D4 — Extend `recategorize_auto_rows` + `would_categorize` (INV-5).**
  - `recategorize_auto_rows(conn)`: read the rules (as today); **iff** the toggle is
    on (`library_enabled(conn)`), also `load_library()` and build the leaf
    `name_to_id` map (D5); per auto row call `categorize_with_library(...)` and
    `set_category(txn_id, cat_id, source)`. When the toggle is **off**, the library
    list is empty → every previously-`'library'` row recomputes to `(None, None)`
    (reverted to Uncategorised, INV-7). The commit-free contract, the changed-row
    return, and the import/delete-cascade callers are **unchanged** (they still pass
    only `conn`).
  - `would_categorize(description)`: build the **same** inputs (rules + library +
    map, toggle-gated) and return `categorize_with_library(...)[0]`. This is what
    keeps the FIBR-0010 learning offer **honest** under a library: confirming a
    library guess (`chosen_id == would`) → **no** offer; overriding it → offer, and
    the accepted rule outranks the library next run (INV-2). *(This is the one
    deliberate change to a FIBR-0010 behaviour, and it is the correct one — without
    it, confirming any library guess would nag.)*

- **D5 — Leaf `name_to_id` map; rename-safe (INV-6).**
  `categorization._leaf_name_to_id(conn) -> dict[str, int]` over the **leaf**
  categories (`CategoryRepository.list_all()` filtered `parent_id is not None`,
  matching `leaf_categories()`), keyed by the **exact stored name**. On a duplicate
  name (a user created a second leaf with the same name) the **first** by
  `list_all()` order (ascending id) wins — deterministic; a rare user edge, not the
  library's concern. `match_library` only returns an id when the entry's `category`
  is a **key** of this map, so a renamed/deleted default silently drops that entry
  (INV-6) — the row falls to Uncategorised, never mis-filed.

- **D6 — The toggle (INV-7), reusing the settings seam.**
  - `categorization.LIBRARY_ENABLED_KEY = "auto_category_library"` and
    `library_enabled(conn) -> bool` — a module helper (so `recategorize_auto_rows`,
    which takes only `conn`, reads it directly): `SettingsRepository(conn).get(key)
    != "false"` (absent/anything-but-`"false"` → **True**, default ON; the
    `"true"/"false"` idiom from `amount_prefs`).
  - `CategorizationService.library_enabled() -> bool` (delegates to the helper) and
    `set_library_enabled(enabled: bool)` (`SettingsRepository(conn).set(key, "true"
    if enabled else "false")`, which owns its commit).
  - `SettingsDialog`: a **new `QCheckBox`** `settings_category_library`
    (*"Guess categories for new transactions"* — `tr()`), constructed from a
    `library_enabled: bool` ctor arg (default `True`), added to the form, and exposed
    as `library_enabled() -> bool` — **mirroring the update checkbox exactly** (the
    dialog holds no service ref for it; it only reports state on Save).
  - Shell (`main_window.py`): `_open_settings` passes
    `library_enabled=CategorizationService(self._service.vault).library_enabled()`;
    `_on_settings_saved` persists via
    `CategorizationService(self._service.vault).set_library_enabled(dialog.library_enabled())`
    (constructed on demand — cheap, stateless — no new shell member). Toggling does
    not re-file; the change lands on the next import/Apply (INV-7).

- **D7 — The "~ guess" marker, Transactions tab only (INV-9); roadmap-D6 wording
  correction.** In `transactions.py`'s render loop, the Category cell for a row with
  `transaction.category_source == CategorySource.LIBRARY.value` **and** a non-empty
  `category_name` is built as `self.tr("{category} ~ guess").format(category=
  category_name)` (a single `tr()` template with a data placeholder — the FIBR-0083
  `.format(detected=…)` idiom, extractable by `lupdate`) with a tooltip
  `self.tr("Guessed from the built-in library — right-click to confirm or change.")`;
  a rule/manual/uncategorised row keeps the plain `QTableWidgetItem(category_name)`.
  The marker is display text only; the category filter keys on `category_id`
  (`transactions.py:275`), so it is unaffected. **Roadmap D6 said "Home + Transactions
  tables"** — but `home.py` is now the **dashboard** (FIBR-0012/0138), which has no
  per-row category cell, so the marker is **Transactions-tab-only**. A library row
  still carries a real `category_id`, so it aggregates into the dashboard's spending
  donut normally (no marker needed there — it is an aggregate view). This corrects the
  brainstorm wording against the current code; the intent (guesses are visibly
  distinguished) is fully met.

- **D8 — Fail-safe loading, split pure/impure for testability (INV-8).**
  `category_library.parse_library(text: str) -> list[LibraryEntry]` is **pure** (JSON
  text → entries; whole-text-invalid → `[]`; per-entry non-string/blank `pattern` or
  `category` → skipped) and is the unit-tested core. `load_library() -> list[LibraryEntry]`
  reads the bundled path, hands the bytes to `parse_library`, and **caches**
  (`functools.lru_cache(maxsize=1)` — the file is read-only for the process lifetime).
  `LibraryEntry` is a `@dataclass(frozen=True)` `(pattern: str, category: str)`.
  Callers reach the loader **through the module** (`category_library.load_library()`,
  not a `from … import load_library` binding) so tests can monkeypatch it (Test
  ripple).

## Data model — no migration

**No schema change.** `category_source` is already free-text `TEXT` (v7), so
`'library'` needs no `ALTER`; the toggle is a row in the existing v1 `settings`
table. `LATEST_SCHEMA_VERSION` is **unchanged** — this feature adds no `_migrate_to_*`
and no ripple to the `== 7` "lands-at-latest" assertions. The library **data file**
is the only new on-disk artifact, and it ships in the bundle (never in the vault).

## Test ripple

**The one real ripple — protect existing suites from the shipped JSON.** With the
library **default-ON**, `recategorize_auto_rows` / `would_categorize` now consult
`load_library()` in **every** test, so any pre-existing suite (notably FIBR-0010's
`tests/features/categorisation/`) whose synthetic descriptions happen to contain a
real library substring would see its category assertions perturbed. Resolution: an
**autouse fixture in the top-level `tests/conftest.py`** —
`_neutralise_category_library` — that monkeypatches
`finbreak.category_library.load_library` to return `[]` for the whole suite by
default, so **no** test is coupled to the shipped file's contents. The new
`category_library` suite **opts in** by monkeypatching `load_library` to its own
small fixture list. *(Verify the exact `conftest.py` location during implementation;
if per-feature conftests are the norm, place the autouse fixture at the widest scope
that covers the pre-existing suites.)*

**New legs (all runnable headless — pure logic + monkeypatched library):**
- `parse_library`: valid array → entries; non-array JSON / malformed text → `[]`;
  an entry missing/blank `pattern` or `category` is skipped, the rest survive (INV-8).
- `match_library`: first substring match whose `category` resolves in the map wins;
  an entry whose `category` is absent from the map is skipped (rename fall-through,
  INV-6); no match / empty library → `None`.
- `categorize_with_library`: a matching **user rule** beats a matching **library**
  entry (INV-2); library-only match → `(id, 'library')`; neither → `(None, None)`.
- `recategorize_auto_rows` end-to-end (with a fixture library injected): a fresh
  import stamps `'library'` on guessed rows and leaves un-guessable rows Uncategorised;
  a **manual** row is untouched (INV-1/INV-3); totals identical before/after (INV-1);
  a second apply changes 0 rows (INV-10).
- Toggle (INV-7): default (absent key) consults the library; after
  `set_library_enabled(False)` the next `apply_rules()` reverts every `'library'`
  row to `(None, None)` while manual + user-rule rows are unaffected;
  `library_enabled()` round-trips the setting.
- `would_categorize` (INV-5): confirming a library guess → `would == chosen` (no
  offer); overriding it → differ (offer), and a rule created for the override then
  wins over the library on the next apply (INV-2).

## Out of scope

- **Learn-from-your-own-history** (statistical, no hand-written rule) → **FIBR-0140**
  (the approved "later" half). This spec ships the static library only.
- **Structural (id-based) binding of library entries** — INV-6 binds by **name** and
  falls through on rename; wiring the library to survive renames is a future
  enhancement (roadmap D7), not MVP.
- **Immediate re-file on toggle** — toggling the setting takes effect on the next
  import/Apply (INV-7); an auto-`apply_rules()` on Save is possible polish.
- **Regex / wildcards / per-account / amount-direction library entries** — substring
  only, exactly as the FIBR-0010 rule engine (user decision 2026-07-09 carried over).
- **A user-editable / importable library, or in-app library updates** — the library
  updates **per release** (bundled file); an in-app editor or fetch is not built
  (and a fetch would breach INV-11's no-network posture).
- **A guess marker on the dashboard** — the dashboard is an aggregate view with no
  per-row cell (D7).
