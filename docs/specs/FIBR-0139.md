**Theme:** file transactions into sensible categories **the first time you import**,
before you have written a single rule. Today auto-categorise only matches
**your** rules (FIBR-0010), so a brand-new vault imports everything as
**Uncategorised** (the cold-start problem). This ships a bundled, per-release
**category library** â€” a plain list of *"if the description contains â€ąmerchantâ€ş,

runs **after** your own rules and **only** on rows nothing else has claimed. Every
library guess is **marked as a guess** and is **always overridable**; a category
**you set by hand still wins** and is never touched. There is **no new button** and
**no schema change** â€” the library rides the existing import + "Apply rules now"
paths and the existing free-text `category_source` column.

## Context

- **The engine (FIBR-0010).** `categorize(description, rules) -> int | None`
  (`services/categorization.py:36`) is the pure first-match matcher: normalise both
  sides (`normalise_text`), return the first rule whose pattern is a **substring** of
  the description â€” no wildcards, no regex. `recategorize_auto_rows(conn) -> int`
  (`:47`) reads the rules once, iterates `TransactionRepository.auto_rows()`, and
  `set_category`s each row to `RULE`/`None`, returning the changed count. It is
  **commit-free**: `apply_rules()` (`:172`), `commit_import` (`import_.py:236`, its
  `recategorize_auto_rows` call at `:288`), and the delete cascade each own their
  transaction. `would_categorize(description)`
  (`:125`) is the learning "differs" check.
- **The source model.** `category_source` is a **free-text `TEXT`** column (v7, no
  `CHECK`); `auto_rows()` (`transactions.py:52`) reads `WHERE category_source IS NULL
  OR category_source <> 'manual'` â€” so any non-`'manual'` value (including a new
  `'library'`) is in the recompute set. `CategorySource(StrEnum)` (`models.py:62`)
  has `RULE`/`MANUAL`. `set_category(txn_id, category_id, source)` (`transactions.py:65`)

## Design decisions

- **D1 â€” The bundled, per-release library file (INV-8, INV-11).**
  `src/finbreak/data/category_library.json` â€” a JSON **array** of objects
  `{"pattern": "<substring>", "category": "<default category name>"}`, e.g.
  `[{"pattern": "pick n pay", "category": "Groceries"}, {"pattern": "netflix",
  "category": "Entertainment"}, â€¦]`. **No `version` wrapper** and no per-entry extras
  (coding.md Â§ 1.1 â€” the shape is exactly what the matcher reads; a schema-version
  field is added only when a real second shape appears). Patterns are stored
  lower-case (they are `normalise_text`-folded at match time anyway; lower-case keeps
  the file readable). It travels **like `ui/icons`**: a **second** package-data glob
  `"data/*.json"` and a **second** `--add-data` pair `<src>/finbreak/data â†’
  finbreak/data` in all three bundling sites â€” the two real freeze invocations plus the
  shared constants module (D-Deliverables 8â€“9). At runtime a loader
  resolves it via `Path(__file__).parent / "data" / "category_library.json"`
  (mirroring `ui/icons.py:26` â€” works from source **and** frozen). **Fail-safe:** any
  read/parse error, or a JSON value that is not an array, â†’ empty library (INV-8),
  logged at `WARNING` (no raise into import/apply). The initial seed is SA-first +
  universal; the file is expected to **grow per release** (its whole point â€” we ship
  a better library over time with zero code change).

- **D2 â€” Two-layer match, user rules first (INV-2), reusing `categorize`
  (coding.md Â§ 1.3).** New **pure** helpers:
  - `category_library.match_library(description, entries, name_to_id) -> int | None`
    â€” normalise the description once; return the id of the **first** entry whose
    `normalise_text(pattern)` is a substring **and** whose `category` name resolves
    in `name_to_id`; else `None`. (Same substring/`normalise_text` primitive as
    `categorize`; a separate 4-line function rather than a forced shared abstraction â€”
    the two call shapes differ, so Rule-of-Three does not bite.) **Cost:**
    `recategorize_auto_rows` runs O(rows Ă— (rules + entries)) substring tests â€” the
    same shape as the FIBR-0010 rule scan, just with a second small list; `entries` is
    curated in the **low hundreds** and `normalise_text`-folding is cheap, so no perf
    budget beyond FIBR-0010's applies and **no index / trie** is built (coding.md Â§ 1.1
    â€” YAGNI at this scale). (Patterns are re-folded per row for simplicity; if a future
    library grows past a few thousand entries, fold each pattern **once at parse time**
    onto `LibraryEntry` â€” noted, not built.)
  - `categorization.categorize_with_library(description, rules, entries, name_to_id)
    -> tuple[int | None, str | None]` â€” **reuse** `categorize(description, rules)`
    for the rule layer: if it matches, return `(id, CategorySource.RULE.value)`;
    else `match_library(...)` â†’ `(id, CategorySource.LIBRARY.value)`; else
    `(None, None)`. This single function is the one place the two layers compose, and
    both the apply path (D4) and `would_categorize` (D4) call it, so they can never
    diverge.
  - `categorization._match_inputs(conn) -> tuple[list[CategorizationRule],
    list[LibraryEntry], dict[str, int]]` â€” the **single source** of the toggle-gated
    inputs: always read the rules (slot 1); **iff** `library_enabled(conn)`,
    `load_library()` + build the D5 `name_to_id` map, else the library pair is
    `[], {}` â€” i.e. the return is `(rules, [], {})` when off. Both
    `recategorize_auto_rows` and
    `would_categorize` build their inputs **only** through this helper, so the
    *gating* (off â†’ empty library) can't drift between them â€” `categorize_with_library`
    guarantees match consistency, `_match_inputs` guarantees toggle consistency.

- **D3 â€” `CategorySource.LIBRARY = "library"` (INV-4).** One enum member added to
  `models.py:62`. **No migration** â€” the column is free-text and `auto_rows` already
  recomputes non-`'manual'` rows. (The FIBR-0010 verified-basis note that the column
  has no `CHECK` is the load-bearing fact here.)

- **D4 â€” Extend `recategorize_auto_rows` + `would_categorize` (INV-5).**
  - `recategorize_auto_rows(conn)`: `rules, entries, name_to_id = _match_inputs(conn)`;
    per auto row call `categorize_with_library(desc, rules, entries, name_to_id)` and
    `set_category(txn_id, cat_id, source)`. When the toggle is **off**, `entries` is
    empty â†’ every previously-`'library'` row recomputes to `(None, None)` (reverted to
    Uncategorised, INV-7). The commit-free contract, the changed-row return, and the
    import/delete-cascade callers are **unchanged** (they still pass only `conn`).
  - `would_categorize(description)`: build the inputs through the **same**
    `_match_inputs(self._conn)` helper and return `categorize_with_library(...)[0]`.
    This is what
    keeps the FIBR-0010 learning offer **honest** under a library: confirming a
    library guess (`chosen_id == would`) â†’ **no** offer; overriding it â†’ offer, and
    the accepted rule outranks the library next run (INV-2). *(This is the one
    deliberate change to a FIBR-0010 behaviour, and it is the correct one â€” without
    it, confirming any library guess would nag.)*

- **D5 â€” Leaf `name_to_id` map; rename-safe (INV-6).**
  `categorization._leaf_name_to_id(conn) -> dict[str, int]` over the **leaf**
  categories (`CategoryRepository.list_all()` filtered `parent_id is not None` â€” the
  same predicate as the `self`-bound `leaf_categories()` at `categorization.py:76`,
  re-expressed here because this is a **module** function with no `self`; a noted
  3-line duplication, coding.md Â§ 1.3), keyed by the **exact stored name,
  case-sensitive**
  (category names are **not** folded â€” only the merchant *patterns* are
  `normalise_text`-folded, D2; so a user who renames "Groceries" to "groceries" makes
  the "Groceries" library entries fall through, which INV-6 permits). On a duplicate
  name (a user created a second leaf with the same name) the **first** by
  `list_all()` order wins â€” build the map with `dict.setdefault(name, id)` (a plain
  `{c.name: c.id â€¦}` comprehension would be *last*-wins), giving a deterministic
  first-wins over `list_all`'s `ORDER BY parent_id, name COLLATE NOCASE, id`
  (`repositories/categories.py:43` â€” `parent_id` is the **leading sort column**, so
  "lowest id" holds only among leaves sharing a parent; the guarantee is
  *deterministic*, not literally lowest-id). A rare user edge, not the library's
  concern. `match_library`
  only returns an id when the entry's `category`
  is a **key** of this map, so a renamed/deleted default silently drops that entry
  (INV-6) â€” the row falls to Uncategorised, never mis-filed.

  - `SettingsDialog`: a **new `QCheckBox`** `settings_category_library`

- **D7 â€” The "~ guess" marker, Transactions tab only (INV-9); roadmap-D6 wording
  correction.** In `transactions.py`'s render loop, **every** Category cell becomes a
  **`SortableItem`** (`ui/_table_state.py:42`; ctor `__init__(text, sort_key)` at
  `:48`) whose **sort key is always the bare `category_name`** and whose **display**
  text is the plain `category_name` **except** for a row with
  `transaction.category_source == CategorySource.LIBRARY.value` and a non-empty
  `category_name`, where the display is `self.tr("{category} ~ guess").format(
  category=category_name)` (a single `tr()` template with a data placeholder â€” the
  FIBR-0083 `.format(detected=â€¦)` idiom, extractable by `lupdate`) plus a tooltip
  `self.tr("Guessed from the built-in library â€” right-click to confirm or change.")`.
  **Why every cell, not just the guess cells:** `SortableItem.__lt__`
  (`_table_state.py:52`) uses its `sort_key` **only when both compared items carry
  one** â€” against a plain `QTableWidgetItem` it falls back to a *display-text* compare.
  So if only guess cells were `SortableItem`, a "Groceries ~ guess" vs a plain
  "Groceries" comparison would still sort by display text and split the group. Making
  **all** Category cells `SortableItem(display, category_name)` means every comparison
  uses the bare-name key, so a guessed row sorts **with** its plain-named siblings.
  The `enable_sorting(self._table)` call (`transactions.py:159`) makes the column
  genuinely sortable, so this is load-bearing, not hypothetical. The marker is thus
  display-only â€” **both** the column **sort** (uniform bare-name key) **and** the
  category **filter** (keyed on `category_id`, `transactions.py:272/275`) are
  unaffected. **Roadmap D6 said "Home + Transactions
  tables"** â€” but `home.py` is now the **dashboard** (FIBR-0012/0138), which has no
  per-row category cell, so the marker is **Transactions-tab-only**. A library row
  still carries a real `category_id`, so it aggregates into the dashboard's spending
  donut normally (no marker needed there â€” it is an aggregate view). This corrects the
  brainstorm wording against the current code; the intent (guesses are visibly
  distinguished) is fully met.

- **D8 â€” Fail-safe loading, split pure/impure for testability (INV-8).**
  `category_library.parse_library(text: str) -> list[LibraryEntry]` is **pure and
  total for ANY `str`** â€” it **never raises**: it wraps `json.loads(text)` in
  `try/except ValueError` (`json.JSONDecodeError` **is** a `ValueError`), so a
  non-JSON / unparseable file â†’ `[]`; a parsed top-level value that is **not a list**
  â†’ `[]`; each element that is **not a dict** carrying a non-blank string `pattern`
  **and** `category` is **skipped** (so `[1, 2, 3]`, `["netflix"]`, and `{}` elements
  drop cleanly, never raising `AttributeError`/`TypeError`). A `pattern` that is blank
  / whitespace-only (i.e. `normalise_text(pattern) == ""`, which as an empty substring
  would match **every** description) is also skipped. Totality is load-bearing:
  `load_library`'s `try/except` below covers only the *file read*, **not**
  `parse_library`, so every content-shape failure must be swallowed **here**.
  `parse_library` is the unit-tested core. `load_library() -> list[LibraryEntry]`
  reads a **module-level `_LIBRARY_PATH`** constant (`= Path(__file__).parent /
  "data" / "category_library.json"`) **inside a `try/except (OSError,
  UnicodeDecodeError)`** (a missing/unreadable file â†’ `OSError`; a corrupt
  **non-UTF-8** file â†’ `UnicodeDecodeError` from `Path.read_text()`, which is a
  `ValueError`, **not** an `OSError` â€” so the narrower `except OSError` alone would
  leak it; both â†’ `[]`, logged `WARNING` â€” the INV-8 fail-safe for the *file* layer,
  distinct from `parse_library`'s *content* layer), hands the text to `parse_library`,
  and **caches** (`functools.lru_cache(maxsize=1)` â€” the file is read-only for the
  process lifetime). `LibraryEntry` is a `@dataclass(frozen=True)`
  `(pattern: str, category: str)`. Callers reach the loader/matcher **through the
  module** (`category_library.load_library()`, not a `from â€¦ import load_library`
  binding) so tests can monkeypatch it. **The `_LIBRARY_PATH` module constant is the
  test seam:** the file-layer fail-safe leg monkeypatches `_LIBRARY_PATH` to point at
  nonexistent / garbage / non-UTF-8 files and calls
  `category_library.load_library.cache_clear()` before each (the cache is
  process-wide, so a prior call's result would otherwise leak). *(Tests that inject a
  fixture library monkeypatch `load_library` **itself**, bypassing both the path and
  the cache â€” no `cache_clear` needed there.)*

## Test ripple

**The one real ripple â€” protect existing suites from the shipped JSON.** With the
library **default-ON**, `recategorize_auto_rows` / `would_categorize` now consult
`load_library()` in **every** test, so any pre-existing suite (notably FIBR-0010's
`tests/features/categorisation/`) whose synthetic descriptions happen to contain a
real library substring would see its category assertions perturbed. Resolution: an
**autouse fixture in the top-level `tests/conftest.py`** (verified the single
suite-root conftest â€” the widest scope, so it covers **every** feature suite, not
just a subset: categorisation, transactions_tab, import, ofx_import, pdf_import,
standard_bank_pdf, statements, dashboard â€” all of which call
`recategorize_auto_rows` / `apply_rules` / `commit_import` / `would_categorize`) â€”
`_neutralise_category_library` â€” that monkeypatches
`finbreak.category_library.load_library` to return `[]` for the whole suite by
default, so **no** test is coupled to the shipped file's contents. **Why suite-wide
matters:** `tests/features/transactions_tab/test_transactions_tab.py` uses
`"pick n pay"` â€” a **seed library entry** (D1) â€” in several `would_categorize`
assertions (e.g. `:214`). Those particular lines also install a user rule for
`"pick n pay"`, so they'd survive on the rule layer (INV-2) even with the library on;
but a suite that asserted a *library-eligible* description with **no** covering rule
would flip the moment the shipped file loads. The suite-wide fixture removes that
whole class of coupling â€” do not scope it to a subset of suites. The new
`category_library` suite **opts in** by monkeypatching `load_library` to its own
small fixture list. **Ordering (load-bearing):** the opt-in patch MUST run **after**
the autouse neutralise, or `[]` clobbers the fixture library and the opt-in
assertions silently test an empty library (a false green). Since an autouse fixture
with no declared dependency runs *before* other fixtures, the opt-in patch is applied
**in the test body** (or in a fixture that explicitly lists `_neutralise_category_library`
as a dependency) so last-write-wins guarantees the injected list survives.

**New legs (all runnable headless â€” pure logic + monkeypatched library):**
- `parse_library` totality (INV-8): valid array â†’ entries; non-array JSON /
  unparseable text â†’ `[]`; a mixed array `[1, "netflix", {â€¦validâ€¦}]` drops the
  non-dict elements and keeps only the valid entry (no raise); an entry with a

  the rest survive.
