# FIBR-0128 — Forget remembered statement passwords (on the Accounts screen)

> **Status:** 🚧 DRAFT — `/cold-eyes` pending (project `--max-loops 7`). Not yet
> cleared for code.
> **Phase:** P12 follow-up; split from FIBR-0014. The store already exists
> (FIBR-0009, schema v5): `accounts.statement_pdf_password` is written implicitly
> when a locked PDF statement is imported and auto-tried next time, but there is
> **no way to see or clear it**. This adds a per-account **Forget** control so a
> user can wipe a remembered statement password. (Distinct from the FIBR-0013
> export password, which is ephemeral and never stored.)
> **Security:** MED — manages a **stored credential** (clears it), but never
> **displays** or **transmits** it, adds **no new stored secret**, and touches no
> network. The plaintext never crosses into the UI layer (INV-1).
> **On upgrade:** no schema change and no data migration — the column has existed
> since v5; this only adds a way to null it. An account with no remembered password
> (the default) shows no marker and a disabled button — the existing screen is
> byte-for-byte unchanged for it.

## Design decisions

- **D1 — The controls live on the Accounts screen, not Settings** (user directive
  2026-07-14) — a deliberate divergence from the roadmap FIBR-0128 body's original
  "Settings-hosted" wording. Rationale: the remembered password is a **per-account**
  attribute, and different accounts legitimately have different statement passwords;
  the Accounts screen is the existing per-account surface (add / rename / retype /
  delete), and its **selection-driven** Update/Delete idiom fits "Forget the selected
  account's password" exactly. The ROADMAP FIBR-0128 body is updated to record the
  new placement.
- **D2 — Never display the secret; presence is an ID-set, not the plaintext.** Add
  `AccountRepository.ids_with_pdf_password() -> set[int]`
  (`SELECT id FROM accounts WHERE statement_pdf_password IS NOT NULL`), wrapped by
  `AccountService.account_ids_with_pdf_password()`, so the secret column is **never
  selected merely to list**. Clearing **reuses** the existing
  `AccountService.set_pdf_password(id, None)` — no new write method (§3
  reuse-before-rewriting; the UI passes only `None`, never a value). This keeps the
  plaintext out of the UI **by construction** (INV-1), extending FIBR-0009's decision
  to keep the password out of the broadly-passed `Account` dataclass.
- **D3 — A selection-driven Forget button with a confirm, mirroring "Delete
  selected."** The button joins the existing actions row; its enabled state is
  recomputed in `_on_selection_changed` (the `None`/no-selection case → disabled)
  from a presence flag stashed on each row as item data (no per-toggle DB hit), and
  it starts disabled (nothing is selected at build). **Ordering trap:** the current
  `_on_selection_changed` (`ui/accounts.py:148-153`) does `if item is None: return`
  **before** its body, so the enable/disable branch must run **before** that early
  return (or the `None` branch must explicitly disable) — otherwise the `_refresh` a
  Forget triggers clears the list, fires `currentItemChanged(None)`, and the button
  is left **stale-enabled** on an empty selection (INV-3's final leg falsifies this).
  `_on_forget_password` confirms
  via `QMessageBox.question` (the exact idiom `_on_delete` uses — indie-review H-E
  parity), clears on Yes, re-renders, and swallows `VaultLockedError` silently like
  every other handler.
- **D4 — The presence marker is translatable suffix *text*, not an icon-only glyph,
  with one pinned literal.** `_refresh` appends the **exact literal** `"  ·  🔑
  statement password saved"` — inline as `self.tr("  ·  🔑 statement password
  saved")` (an **inline literal**, *not* a module constant fed to `tr()`, because
  `lupdate` extracts only literal `tr()` arguments — INV-6) — to the row label, so a
  screen reader announces the words "statement password saved" (WCAG 1.3.1); an
  emoji-only marker would be silent to assistive tech. Pinning the literal here is
  what lets INV-2's test assert against it without the test and `_refresh` drifting.
  The row text stays the single source of the marker (QListWidget rows are plain
  strings; no extra column).
- **D5 — Forget-only; no manual set/change here** (user directive 2026-07-14). A
  password is (re-)remembered by importing a locked statement, exactly as today; this
  screen only clears. This keeps a password-entry field off the Accounts screen and
  keeps credential **capture** on a single path (the import flow, FIBR-0009).

## Data model (no schema change)

`accounts.statement_pdf_password` (nullable, vault-encrypted at rest) has existed
since **schema v5** (FIBR-0009). This feature adds **no column and no migration** —
it only exposes a presence query and reuses the existing null-write to clear.

## Out of scope

- **Revealing / displaying** the stored password — the user chose never-show; the
  screen only reports presence and clears.
- **Manual set / change** of a password on this screen (D5) — the import flow
  (FIBR-0009) owns credential capture.
- **Settings-screen placement** — superseded by D1.
- **Business / Personal account grouping** — separate item **FIBR-0137**.
- **The auto-try-on-import behaviour** — owned by FIBR-0009; unchanged here.
- **Concurrency / races** — the Accounts screen is a single-threaded Qt widget; there
  is no concurrent-caller path. Two edge cases don't need handling: the **no-accounts**
  state is structurally unreachable (first-run seeds a Default account and
  `AccountService.delete_account` raises `LastAccountError` at the last one, so ≥1
  account always exists), and **forget-twice** can't occur (clearing a password
  disables the button, so there is no second forget to make).
