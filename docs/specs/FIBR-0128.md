## Design decisions

- **D1 â€” The controls live on the Accounts screen, not Settings** (user directive
  2026-07-14) â€” a deliberate divergence from the roadmap FIBR-0128 body's original
  "Settings-hosted" wording. Rationale: the remembered password is a **per-account**
  attribute, and different accounts legitimately have different statement passwords;
  the Accounts screen is the existing per-account surface (add / rename / retype /
  delete), and its **selection-driven** Update/Delete idiom fits "Forget the selected
  account's password" exactly. The ROADMAP FIBR-0128 body is updated to record the
  new placement.
- **D2 â€” Never display the secret; presence is an ID-set, not the plaintext.** Add
  `AccountRepository.ids_with_pdf_password() -> set[int]`
  (`SELECT id FROM accounts WHERE statement_pdf_password IS NOT NULL`), wrapped by
  `AccountService.account_ids_with_pdf_password()`, so the secret column is **never
  selected merely to list**. Clearing **reuses** the existing
  `AccountService.set_pdf_password(id, None)` â€” no new write method (Â§3
  reuse-before-rewriting; the UI passes only `None`, never a value). This keeps the
  plaintext out of the UI **by construction** (INV-1), extending FIBR-0009's decision
  to keep the password out of the broadly-passed `Account` dataclass.
- **D3 â€” A selection-driven Forget button with a confirm, mirroring "Delete
  selected."** The button joins the existing actions row; its enabled state is
  recomputed in `_on_selection_changed` (the `None`/no-selection case â†’ disabled)
  from a presence flag stashed on each row as item data (no per-toggle DB hit), and
  it starts disabled (nothing is selected at build). **Ordering trap:** the current
  `_on_selection_changed` (`ui/accounts.py:148-153`) does `if item is None: return`
  **before** its body, so the enable/disable branch must run **before** that early
  return (or the `None` branch must explicitly disable) â€” otherwise the `_refresh` a
  Forget triggers clears the list, fires `currentItemChanged(None)`, and the button
  is left **stale-enabled** on an empty selection (INV-3's final leg falsifies this).
  `_on_forget_password` confirms
  via `QMessageBox.question` (the exact idiom `_on_delete` uses â€” indie-review H-E
  parity), clears on Yes, re-renders, and swallows `VaultLockedError` silently like
  every other handler.
- **D4 â€” The presence marker is translatable suffix *text*, not an icon-only glyph,
  with one pinned literal.** `_refresh` appends the **exact literal** `"  Â·  ðŸ”‘
  statement password saved"` â€” inline as `self.tr("  Â·  ðŸ”‘ statement password
  saved")` (an **inline literal**, *not* a module constant fed to `tr()`, because
  `lupdate` extracts only literal `tr()` arguments â€” INV-6) â€” to the row label, so a
  screen reader announces the words "statement password saved" (WCAG 1.3.1); an
  emoji-only marker would be silent to assistive tech. Pinning the literal here is
  what lets INV-2's test assert against it without the test and `_refresh` drifting.
  The row text stays the single source of the marker (QListWidget rows are plain
  strings; no extra column).
- **D5 â€” Forget-only; no manual set/change here** (user directive 2026-07-14). A
  password is (re-)remembered by importing a locked statement, exactly as today; this
  screen only clears. This keeps a password-entry field off the Accounts screen and
  keeps credential **capture** on a single path (the import flow, FIBR-0009).

## Data model (no schema change)

`accounts.statement_pdf_password` (nullable, vault-encrypted at rest) has existed
since **schema v5** (FIBR-0009). This feature adds **no column and no migration** â€”
it only exposes a presence query and reuses the existing null-write to clear.

## Out of scope

- **Revealing / displaying** the stored password â€” the user chose never-show; the
  screen only reports presence and clears.
- **Manual set / change** of a password on this screen (D5) â€” the import flow
  (FIBR-0009) owns credential capture.
- **Settings-screen placement** â€” superseded by D1.
- **Business / Personal account grouping** â€” separate item **FIBR-0137**.
- **The auto-try-on-import behaviour** â€” owned by FIBR-0009; unchanged here.
- **Concurrency / races** â€” the Accounts screen is a single-threaded Qt widget; there
  is no concurrent-caller path. The no-accounts and forget-twice edge cases are handled
  by construction (an empty `_refresh` loop draws nothing; once a marker clears the
  button disables, so there is no second forget to make).
