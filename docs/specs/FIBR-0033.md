## Context

FIBR-0014 ships an encrypted `.fbk` backup + a destructive restore. But a
backup a user has *never test-restored* is a guess, not a safety net ‚Äî a
corrupt file, a mistyped backup password, or a truncated copy is only
discovered at the worst possible moment (an actual restore, after the live
vault is already lost). FIBR-0033 lets a user **check a backup proactively ‚Äî
from inside the running app, while they still have access ‚Äî instead of finding
out during a real restore.** It adds a **one-click, read-only "does this backup
actually open?" check**: point at a `.fbk`, enter its backup password, and
confirm it decrypts and its schema + row counts are intact ‚Äî **without touching
the live vault** and without installing anything. (Verifying a backup when you
are *already* locked out ‚Äî the recovery case ‚Äî needs a pre-login affordance and
is a separate deferred enhancement; see Out of scope.)

The verify path is *already* the first three steps of `restore_backup`
(`services/backup.py:126`): read + guard the zip ‚Üí validate KDF params ‚Üí open
the backup's `vault.db` into a temp with the backup key. Restore then rekeys
and installs; **verify simply stops after the open** and reports what it found.

## The mechanism

`BackupService.verify_backup(src: Path, backup_password: str, *, on_key: OnKey |
None = None) -> VerifyResult`:

1. `_read_fbk(src)` + `_guard_manifest(manifest)` ‚Äî the FIBR-0014 pre-login zip
   guards (fixed entry names, per-entry size caps checked *before* inflating,
   traversal rejection, `format_version`/`sqlcipher_compat`/`schema_version`
   gates). INV-11/INV-12/INV-13/INV-6a.
2. Materialise `params.json` to a `0o600` temp and the backup's ciphertext
   `vault.db` bytes to a `tmp_db` temp; `load_and_validate_params` (KDF floor,
   INV-11); `derive_key(backup_password, salt, params)`.
3. `Vault(tmp_db, tmp_sidecar).open(backup_key, in_memory_temp=True,
   cipher_compat=SQLCIPHER_COMPAT)` ‚Äî the page-1 guard read forces a
   decrypt+HMAC, so a wrong password or a tampered file raises `DatabaseError`
   here; `run_migrations` inside `open` **migrates an older backup's temp copy
   up to `LATEST_SCHEMA_VERSION`** (a write to the *temp* copy only ‚Äî never the
   live vault) and raises `SchemaVersionError` if the backup is newer than this
   build supports.
4. On a clean open (which has just migrated the temp copy, step 3):
   - **4a ‚Äî Integrity.** Run **`PRAGMA cipher_integrity_check`** ‚Äî SQLCipher's
     full-database pass that re-verifies the HMAC of **every** page, including the
     overflow pages holding large cell payloads that `SELECT count(*)` never reads
     (introduced in SQLCipher 4.3.0; finbreak bundles **4.12.0** via
     `sqlcipher3-wheels==0.5.7`, and the backup is opened at
     `cipher_compatibility = 4`). Zero rows ‚Üí fully intact; **any row ‚Üí `ok=False`
     (`corrupt`), returning before 4b/4c**. This is what makes the "intact"
     promise honest ‚Äî the page-1 guard read alone proves only that page 1
     decrypts. (On an *older* backup this validates the **migrated** temp; a page
     migration rewrote is re-HMACed validly, so a body/overflow corruption in an
     older backup instead surfaces as `wrong_password` during migration ‚Äî verify's
     clean `corrupt` guarantee is strongest for a current-schema backup, the
     common case, since export always writes the current schema.)
   - **4b ‚Äî Schema.** Read `schema_version`; it equals `LATEST_SCHEMA_VERSION`
     after any clean open+migrate, so verify reports the **as-migrated** version,
     not the backup's as-stored one (INV-2).
   - **4c ‚Äî Counts (display only).** A per-table row count by dynamic enumeration
     (`SELECT name FROM sqlite_master WHERE type='table' AND name NOT LIKE
     'sqlite_%'`, then `SELECT count(*)` per table ‚Äî the same *enumeration* as the
     FIBR-0014 tests' `_snapshot_tables`, but counting rows into `dict[str, int]`
     rather than snapshotting full row-sets; read after the migration ran).
   - **4d ‚Äî Teardown.** Close the connection and discard the temp dir; the derived
     key and password buffer are wiped on every path (INV-7).

**Verify owns a system `tempfile.TemporaryDirectory()`** (not under the vault
dir ‚Äî verify never renames anything into place) and passes it to
`_open_backup_vault` as the work dir (restore instead passes an
`AppDataLocation`-local dir for its same-filesystem install rename ‚Äî see D1). The
only files written are the backup's own **ciphertext** `vault.db` plus its
`0o600` params temp; `in_memory_temp=True` keeps SQLCipher's scratch in RAM. The
temp dir is removed on **every** path ‚Äî success, wrong password, corrupt file, or
an unexpected error (a `finally`/context manager) ‚Äî and the backup-password
buffer plus the derived backup key are **zeroed on every path** too (INV-7),
never left live after verify returns.

`VerifyResult` (frozen dataclass): `ok: bool`; `schema_version: int | None` (the
as-migrated version `== LATEST_SCHEMA_VERSION` on success, `None` on failure);
`table_counts: dict[str, int] | None` (populated on success, `None` on failure);
`reason: str | None` ‚Äî a **stable code** (an enum / key) on failure, `None` on
success, **never** raw exception text. The service stays UI-free (mirroring
`settings.py` "keeps no BackupService reference", and Export's failure path which
shows a *fixed* translated `QMessageBox` rather than `str(exc)`):
`BackupVerifyDialog` maps each code to a translated (`self.tr(...)`) message.

`verify_backup` maps each failure **trigger** ‚Äî a caught exception *type* (never
its message text, since the shipped guards raise one opaque `BackupError` and
message-sniffing would be fragile) or a non-empty `cipher_integrity_check` result
‚Äî to `ok=False` + a code:

| Trigger | Code | Covers |
|---------|------|--------|
| `DatabaseError` | `wrong_password` | wrong backup password **or** a damaged page-1 header ‚Äî SQLCipher's page-1 HMAC can't tell them apart. It also catches a body/overflow corruption in an *older-schema* backup, because `open`'s in-line `run_migrations` reads (HMAC-checks) body pages **before** step 4a runs ‚Äî so clean `corrupt` detection (4a) is guaranteed only for a **current-schema** backup (the common case, since export always writes the current schema). Dialog copy names all three. |
| non-empty `cipher_integrity_check` | `corrupt` | a damaged body / overflow page (which `count(*)` can't reach) on a current-schema backup |
| `KdfPolicyError` | `bad_kdf_params` | KDF params below the pinned floor **or** a malformed / unreadable `params.json` ‚Äî `load_and_validate_params` normalises both to `KdfPolicyError` |
| `SchemaVersionError` | `too_new` | embedded schema newer than this build ‚Äî only the rare *manifest-under-states* case (a crafted `.fbk` whose `manifest.json` declares a lower schema than the embedded DB, so it slips the manifest gate and trips `run_migrations`) |
| `BackupError` | `invalid` | **every** guard/read failure: not-a-zip / missing / unexpected-or-oversized entry / traversal / bad `format_version`/`sqlcipher_compat`, **and** the common "made by a newer finbreak" case (its manifest-schema gate raises `BackupError` *before* `open`, so it reports `invalid`, not `too_new`) |
| `OSError` | `io_error` | a temp-write failure (disk-full / read-only temp; the helper raises it **raw** since the normaliser stays in restore, D1, so verify must catch it or it crashes the synchronous GUI handler, D7) |

The `BackupError` family collapses to one `invalid` code **by design** ‚Äî the
shipped FIBR-0014 guards raise a single opaque `BackupError` for all of these, and
splitting them finer would need either message-sniffing (fragile) or adding a
`.code` to `BackupError` (a FIBR-0014 change outside this feature's scope): a
deliberate non-goal. Truly unexpected exceptions propagate (a bug, not a "bad
backup" answer).

## Design decisions

- **D1 ‚Äî Extract, don't duplicate (rule 3b).** Factor the shared read ‚Üí guard ‚Üí
  materialise-params ‚Üí derive ‚Üí open sequence out of `restore_backup` into a
  private `BackupService._open_backup_vault(src, backup_password, work_dir, *,
  on_key) -> Vault`. Contract points the extraction must honour:
  - **The caller creates and owns `work_dir`** (a `TemporaryDirectory`) and passes
    it in. The returned `Vault` holds an open SQLCipher connection to a `vault.db`
    **inside** `work_dir` that must outlive the call ‚Äî restore keeps using it
    (rekey ‚Üí `_write_sidecar` ‚Üí `_install`, whose `os.replace` renames it into
    place, which is *why* restore's dir is `AppDataLocation`-local for a
    same-filesystem move), and verify reads counts from it before closing. A
    helper that owned a `with TemporaryDirectory()` would delete the dir on
    return and orphan both the connection and the files restore still needs ‚Äî so
    the location cannot be hard-coded in the helper.
  - **The helper owns the key/password hygiene it creates:** it builds the
    password `bytearray`, derives the backup key, and **wipes both in its own
    `finally` on every path** (INV-7). `on_key` is threaded through for the
    FIBR-0014 wipe-assertion test seam; `verify_backup` exposes an optional
    `on_key=None` (defaulting to `_noop_on_key`) and forwards it to the helper, so
    an INV-7 test can inject a capturing callback exactly as the existing
    export/restore wipe tests do (both expose `on_key: OnKey | None = None`). The
    helper returns the opened `Vault` **only** ‚Äî the manifest is fully consumed by
    `_guard_manifest` inside the helper, so no caller needs it back.
  - **Exception boundary:** the helper leaves the KDF/open/migration failures
    **raw** (`KdfPolicyError`, `DatabaseError`, `SchemaVersionError`) and lets
    `_read_fbk`/`_guard_manifest` raise `BackupError` directly, exactly as today.
    The normalising `try/except (KdfPolicyError, SchemaVersionError,
    DatabaseError, BadZipFile, OSError) ‚Üí BackupError` stays in **`restore_backup`**,
    NOT in the helper ‚Äî otherwise every failure collapses to one `BackupError`
    and verify could not map each class to a distinct reason (INV-4).
    `verify_backup` therefore catches `BackupError` **and** the three raw classes
    (see the mechanism above).

  `restore_backup`'s externally-observable behaviour is unchanged ‚Äî the existing
  FIBR-0014 restore tests are the regression guard, and must stay green untouched.
- **D2 ‚Äî Verify derives only the backup key.** No `new_master_password`, no
  rekey, no sidecar write. Verify answers "can this open?", nothing more ‚Äî and
  the derived backup key + the password buffer are wiped on every path (INV-7),
  the same key-hygiene restore already honours. (The wipe lives in
  `_open_backup_vault`, which owns the derive ‚Äî see D1.)
- **D3 ‚Äî Read-only w.r.t. the live vault.** Verify never writes `vault_path` or
  its sidecar, never calls `_install`, never unlocks the live vault. The temp
  dir is a fresh system tempdir, so even a same-name `vault.db` can't collide
  with the live file.
- **D4 ‚Äî Structured result, not exceptions, for the expected cases.** The four
  known "bad backup" outcomes are *answers* the user asked for ("no, and here's
  why"), so they return `ok=False` with a friendly reason, not a stack trace.
  This keeps the UI a simple ok/not-ok render.
- **D5 ‚Äî Post-login placement, mirroring Export.** The affordance is a "Verify
  backup‚Ä¶" button in the Settings dialog next to "Export backup‚Ä¶"
  (`settings.py:189`), emitting a `verify_backup_requested` signal the shell
  handles. This matches the feature's motivation (Context): verify is a
  **proactive, peace-of-mind check run while the owner still has access** ‚Äî the
  natural home is inside the unlocked app, alongside Export. Verify only needs
  the `.fbk` + its backup password, so it *does not technically require* the live
  vault unlocked; a pre-login placement is therefore possible. The distinct
  *locked-out recovery* case (verifying a backup when you can no longer unlock
  the live vault) is a separate deferred enhancement, left out of scope here to
  keep the surface minimal ‚Äî see Out of scope.
- **D6 ‚Äî No new plaintext-at-rest.** The temp `vault.db` is the backup's own
  ciphertext (backup-keyed, HMAC on); `in_memory_temp=True` keeps scratch in
  RAM; cleanup runs on all paths. Verify introduces no new decrypted-to-disk
  surface beyond what restore already does (security-model T4/T5).
- **D7 ‚Äî Synchronous, bounded by the existing restore cost.** Verify runs
  **synchronously on the GUI thread under a wait cursor**, mirroring Export
  (`main_window.py:843`). Unlike Export it needs **no** auto-lock protection ‚Äî
  verify works on a *separate* temp `Vault` and never touches the live vault
  (D3), so an auto-lock firing mid-verify is harmless; the synchronous / wait-
  cursor choice is purely for the simple one-shot UX (kick off, block briefly,
  render ok / not-ok). Resource budget: verify may hold up to
  `MAX_BACKUP_DB_BYTES` (512 MiB) of backup ciphertext in RAM (`_read_fbk` reads
  the `vault.db` entry whole, never streamed) and block the loop for the Argon2id
  derive + open + **a full-database `cipher_integrity_check` HMAC pass over every
  page** (up to ~128k pages for a 512 MiB DB) + the per-table `count(*)`. The
  memory bound matches restore's, but the integrity pass is **verify's one
  genuinely new synchronous cost** ‚Äî restore never issues it. Acceptable for a
  one-shot, user-initiated action (a healthy multi-year vault is a few MiB, not
  512), and 512 MiB is the hard ceiling restore already accepts.

## At a glance ‚Äî invariants

_INV-1..7 are FIBR-0033's own local numbering (distinct from
`docs/security-model.md`'s global list)._

| INV | Statement |
|-----|-----------|
| **INV-1** | Verify never mutates the live vault: `vault_path` and its sidecar are byte-identical (a content hash matches) after `verify_backup`, and **no new files appear in the vault dir**, on **every** outcome (valid, wrong password, corrupt, unexpected error). Verify never opens or writes the live vault at all (D3). |
| **INV-2** | A valid `.fbk` + correct backup password, **with `PRAGMA cipher_integrity_check` returning zero rows**, ‚Üí `VerifyResult(ok=True)` with `schema_version == LATEST_SCHEMA_VERSION` (the **as-migrated** version ‚Äî `open` upgrades the temp copy, step 3, so verify does not report the backup's as-stored version) and `table_counts` equal to the opened-and-migrated temp's actual per-table counts. A vault with zero *transactions* (only the seed + `schema_version` rows every migrated vault carries ‚Äî an all-zero `table_counts` is unreachable) is still `ok=True`. |
| **INV-3** | A wrong backup password ‚Üí `VerifyResult(ok=False)` with the `wrong_password` code (from the caught `DatabaseError`), no exception escapes, live vault untouched. SQLCipher's page-1 HMAC cannot distinguish a wrong key from a damaged page-1 header ‚Äî and an *older-schema* backup's body corruption also surfaces here (migration reads the bad page before 4a) ‚Äî so `wrong_password` deliberately covers all three, and the dialog copy names them. |
| **INV-4** | Each FIBR-0014 tamper class (missing/renamed/extra entry, oversized entry, traversal, bad `format_version`/`sqlcipher_compat`, KDF below floor **or malformed `params.json`**, manifest `schema_version` > supported), a **body/overflow-page corruption** caught by `cipher_integrity_check` that page-1 + `count(*)` would pass, and a missing / non-zip picked file ‚Üí `VerifyResult(ok=False)` ‚Äî reusing the existing guards, not a second implementation. The code is by exception **type** (see the mapping table), so the whole `BackupError` guard family shares the `invalid` code: this INV asserts `ok=False` + a friendly reason, **not** a unique code per class. The overflow-corruption leg needs a fixture that actually has a `count(*)`-invisible page ‚Äî a **latest-schema** backup (migration is then a no-op) seeded with a **long transaction note** to force an overflow page, then a byte flipped in that page. |
| **INV-5** | Verify leaves **no** temp files behind (temp dir removed on success and on every failure path) and writes no decrypted plaintext to disk (only backup-keyed ciphertext + the `0o600` params temp, both inside the removed temp dir). Test seam: monkeypatch `tempfile.TemporaryDirectory` (or pass a known `work_dir`) to a path the test controls, then assert it is gone after every outcome. |
| **INV-6** | `restore_backup`'s externally-observable behaviour is unchanged by the D1 extraction ‚Äî the FIBR-0014 restore test suite passes untouched. |
| **INV-7** | Verify leaves no live key material behind: the backup-password buffer and the derived backup key are zeroed on every path (success, wrong password, corrupt, unexpected error) ‚Äî the project-wide key-hygiene rule restore honours. The wipe is owned by `_open_backup_vault` (D1) and is observable through the `on_key` test seam. |

   backup_password, work_dir, *, on_key) -> Vault` (shared with `restore_backup`,
   D1; the caller owns `work_dir`, the helper wipes the backup key + password and
   raises the failure classes raw); add `verify_backup(src, backup_password, *,
   on_key=None) -> VerifyResult` (opens under a system `TemporaryDirectory`, runs
   `cipher_integrity_check`, reads schema/counts, maps each failure class to a
   `reason` code, closes); a frozen `VerifyResult` dataclass in `backup.py`

## Out of scope

- Pre-login (locked) placement of Verify ‚Äî post-login Settings only for now; the
  locked-out recovery case (verify when you can no longer unlock the live vault)
  is a **separate deferred enhancement**, tracked as a roadmap follow-up.
- Auto-verifying a backup right after export (a possible later nicety).
- Verifying anything about the *live* vault (this is about a `.fbk` file).
- Repairing a bad backup ‚Äî verify only reports.

## Exit criteria

- `./scripts/ci-local.sh` green (ruff, format, bandit, pip-audit, gitleaks,
  pytest, mypy).
- INV-1..7 each have a falsifiable test; INV-1 asserts the live vault (content
  hash + no new files in the vault dir), INV-5 the temp dir (emptiness, via a
  monkeypatched `tempfile.TemporaryDirectory` path), INV-7 the wiped key/password
  (via the `on_key` seam).
- The FIBR-0014 restore suite passes unmodified (INV-6).
