## Context

FIBR-0014 ships an encrypted `.fbk` backup + a destructive restore. But a
backup a user has *never test-restored* is a guess, not a safety net — a
corrupt file, a mistyped backup password, or a truncated copy is only
discovered at the worst possible moment (an actual restore, after the live
vault is already lost). FIBR-0033 lets a user **check a backup proactively —
from inside the running app, while they still have access — instead of finding
out during a real restore.** It adds a **one-click, read-only "does this backup
actually open?" check**: point at a `.fbk`, enter its backup password, and
confirm it decrypts and its schema + row counts are intact — **without touching
the live vault** and without installing anything. (Verifying a backup when you
are *already* locked out — the recovery case — needs a pre-login affordance and
is a separate deferred enhancement; see Out of scope.)

The verify path is *already* the first three steps of `restore_backup`
(`services/backup.py:126`): read + guard the zip → validate KDF params → open
the backup's `vault.db` into a temp with the backup key. Restore then rekeys
and installs; **verify simply stops after the open** and reports what it found.

## The mechanism

`BackupService.verify_backup(src: Path, backup_password: str, *, on_key: OnKey |
None = None) -> VerifyResult`:

1. `_read_fbk(src)` + `_guard_manifest(manifest)` — the FIBR-0014 pre-login zip
   guards (fixed entry names, per-entry size caps checked *before* inflating,
   traversal rejection, `format_version`/`sqlcipher_compat`/`schema_version`
   gates). INV-11/INV-12/INV-13/INV-6a.
2. Materialise `params.json` to a `0o600` temp; `load_and_validate_params` (KDF
   floor, INV-11); `derive_key(backup_password, salt, params)`.
3. `Vault(tmp_db, tmp_sidecar).open(backup_key, in_memory_temp=True,
   cipher_compat=SQLCIPHER_COMPAT)` — the page-1 guard read forces a
   decrypt+HMAC, so a wrong password or a tampered file raises `DatabaseError`
   here; `run_migrations` inside `open` **migrates an older backup's temp copy
   up to `LATEST_SCHEMA_VERSION`** (a write to the *temp* copy only — never the
   live vault) and raises `SchemaVersionError` if the backup is newer than this
   build supports.
4. On a clean open (which has just migrated the temp copy, step 3): run
   **`PRAGMA cipher_integrity_check`** — SQLCipher's full-database pass that
   re-verifies the HMAC of **every** page, including the overflow pages holding
   large cell payloads that `SELECT count(*)` never reads (a SQLCipher-4 feature;
   the backup is opened at `cipher_compatibility = 4`). It returns **zero rows**
   when the ciphertext is fully intact; any row means a corrupt / tampered /
   truncated page → `ok=False`. This is what makes the "intact" promise honest —
   the page-1 guard read alone proves only that page decrypts. Then read the
   now-current `schema_version` — it equals `LATEST_SCHEMA_VERSION` after any
   clean open+migrate, so verify reports the **as-migrated** version, not the
   backup's as-stored one (INV-2) — and, **for the result display only**, a
   per-table row count by dynamic enumeration (`SELECT name FROM sqlite_master
   WHERE type='table' AND name NOT LIKE 'sqlite_%'`, then `SELECT count(*)` per
   table — the same *enumeration* as the FIBR-0014 tests' `_snapshot_tables`, but
   counting rows into `dict[str, int]` rather than snapshotting full row-sets;
   counts are read after the migration ran). Close the connection and discard the
   temp dir (the derived key and password buffer are wiped on every path —
   INV-7).

**Verify owns a system `tempfile.TemporaryDirectory()`** (not under the vault
dir — verify never renames anything into place) and passes it to
`_open_backup_vault` as the work dir (restore instead passes an
`AppDataLocation`-local dir for its same-filesystem install rename — see D1). The
only files written are the backup's own **ciphertext** `vault.db` plus its
`0o600` params temp; `in_memory_temp=True` keeps SQLCipher's scratch in RAM. The
temp dir is removed on **every** path — success, wrong password, corrupt file, or
an unexpected error (a `finally`/context manager) — and the backup-password
buffer plus the derived backup key are **zeroed on every path** too (INV-7),
never left live after verify returns.

`VerifyResult` (frozen dataclass): `ok: bool`; on success
`schema_version: int` (the as-migrated version, `== LATEST_SCHEMA_VERSION`) +
`table_counts: dict[str, int]`; on failure a **stable `reason` code** — an enum /
key (`wrong_password`, `corrupt`, `too_new`, `kdf_too_weak`, `unreadable`), **not**
raw exception text. The service stays UI-free (mirroring `settings.py` "keeps no
BackupService reference", and Export's failure path which shows a *fixed*
translated `QMessageBox` rather than `str(exc)`): `BackupVerifyDialog` maps each
code to a translated (`self.tr(...)`) message.

`verify_backup` maps every known "bad backup" outcome to `ok=False` + a code:
`BackupError` (guard / zip-read failures) → `corrupt` / `unreadable`;
`DatabaseError` (wrong password / tamper) → `wrong_password`; a non-empty
`PRAGMA cipher_integrity_check` (corrupt page) → `corrupt`; `KdfPolicyError` (KDF
below floor) → `kdf_too_weak`; `SchemaVersionError` (too new) → `too_new`; and
`OSError` (a temp-write failure — disk-full / read-only temp; the helper raises
it **raw** since the normaliser stays in restore, D1, so verify must catch it or
it crashes the synchronous GUI handler, D7) → `unreadable`. Truly unexpected
exceptions propagate (a bug, not a "bad backup" answer).

## Design decisions

- **D1 — Extract, don't duplicate (rule 3b).** Factor the shared read → guard →
  materialise-params → derive → open sequence out of `restore_backup` into a
  private `BackupService._open_backup_vault(src, backup_password, work_dir, *,
  on_key) -> Vault`. Contract points the extraction must honour:
  - **The caller creates and owns `work_dir`** (a `TemporaryDirectory`) and passes
    it in. The returned `Vault` holds an open SQLCipher connection to a `vault.db`
    **inside** `work_dir` that must outlive the call — restore keeps using it
    (rekey → `_write_sidecar` → `_install`, whose `os.replace` renames it into
    place, which is *why* restore's dir is `AppDataLocation`-local for a
    same-filesystem move), and verify reads counts from it before closing. A
    helper that owned a `with TemporaryDirectory()` would delete the dir on
    return and orphan both the connection and the files restore still needs — so
    the location cannot be hard-coded in the helper.
  - **The helper owns the key/password hygiene it creates:** it builds the
    password `bytearray`, derives the backup key, and **wipes both in its own
    `finally` on every path** (INV-7). `on_key` is threaded through for the
    FIBR-0014 wipe-assertion test seam; `verify_backup` exposes an optional
    `on_key=None` (defaulting to `_noop_on_key`) and forwards it to the helper, so
    an INV-7 test can inject a capturing callback exactly as the existing
    export/restore wipe tests do (both expose `on_key: OnKey | None = None`). The
    helper returns the opened `Vault` **only** — the manifest is fully consumed by
    `_guard_manifest` inside the helper, so no caller needs it back.
  - **Exception boundary:** the helper leaves the KDF/open/migration failures
    **raw** (`KdfPolicyError`, `DatabaseError`, `SchemaVersionError`) and lets
    `_read_fbk`/`_guard_manifest` raise `BackupError` directly, exactly as today.
    The normalising `try/except (KdfPolicyError, SchemaVersionError,
    DatabaseError, BadZipFile, OSError) → BackupError` stays in **`restore_backup`**,
    NOT in the helper — otherwise every failure collapses to one `BackupError`
    and verify could not map each class to a distinct reason (INV-4).
    `verify_backup` therefore catches `BackupError` **and** the three raw classes
    (see the mechanism above).

  `restore_backup`'s externally-observable behaviour is unchanged — the existing
  FIBR-0014 restore tests are the regression guard, and must stay green untouched.
- **D2 — Verify derives only the backup key.** No `new_master_password`, no
  rekey, no sidecar write. Verify answers "can this open?", nothing more — and
  the derived backup key + the password buffer are wiped on every path (INV-7),
  the same key-hygiene restore already honours. (The wipe lives in
  `_open_backup_vault`, which owns the derive — see D1.)
- **D3 — Read-only w.r.t. the live vault.** Verify never writes `vault_path` or
  its sidecar, never calls `_install`, never unlocks the live vault. The temp
  dir is a fresh system tempdir, so even a same-name `vault.db` can't collide
  with the live file.
- **D4 — Structured result, not exceptions, for the expected cases.** The four
  known "bad backup" outcomes are *answers* the user asked for ("no, and here's
  why"), so they return `ok=False` with a friendly reason, not a stack trace.
  This keeps the UI a simple ok/not-ok render.
- **D5 — Post-login placement, mirroring Export.** The affordance is a "Verify
  backup…" button in the Settings dialog next to "Export backup…"
  (`settings.py:189`), emitting a `verify_backup_requested` signal the shell
  handles. This matches the feature's motivation (Context): verify is a
  **proactive, peace-of-mind check run while the owner still has access** — the
  natural home is inside the unlocked app, alongside Export. Verify only needs
  the `.fbk` + its backup password, so it *does not technically require* the live
  vault unlocked; a pre-login placement is therefore possible. The distinct
  *locked-out recovery* case (verifying a backup when you can no longer unlock
  the live vault) is a separate deferred enhancement, left out of scope here to
  keep the surface minimal — see Out of scope.
- **D6 — No new plaintext-at-rest.** The temp `vault.db` is the backup's own
  ciphertext (backup-keyed, HMAC on); `in_memory_temp=True` keeps scratch in
  RAM; cleanup runs on all paths. Verify introduces no new decrypted-to-disk
  surface beyond what restore already does (security-model T4/T5).
- **D7 — Synchronous, bounded by the existing restore cost.** Verify runs
  **synchronously on the GUI thread under a wait cursor**, mirroring Export
  (`main_window.py:843`). Unlike Export it needs **no** auto-lock protection —
  verify works on a *separate* temp `Vault` and never touches the live vault
  (D3), so an auto-lock firing mid-verify is harmless; the synchronous / wait-
  cursor choice is purely for the simple one-shot UX (kick off, block briefly,
  render ok / not-ok). Resource budget: verify may
  hold up to `MAX_BACKUP_DB_BYTES` (512 MiB) of backup ciphertext in RAM
  (`_read_fbk` reads the `vault.db` entry whole, never streamed) and block the
  loop for the Argon2id derive + open + per-table `count(*)` scan. This is
  acceptable for a one-shot, user-initiated action and is **the same 512 MiB /
  memory-hard-KDF cost restore already accepts** — verify adds no new hot path
  and pins no tighter budget than restore.

## At a glance — invariants

_INV-1..7 are FIBR-0033's own local numbering (distinct from
`docs/security-model.md`'s global list)._

| INV | Statement |
|-----|-----------|
| **INV-1** | Verify never mutates the live vault: `vault_path` and its sidecar are byte-identical (a content hash matches) after `verify_backup`, and **no new files appear in the vault dir**, on **every** outcome (valid, wrong password, corrupt, unexpected error). Verify never opens or writes the live vault at all (D3). |
| **INV-2** | A valid `.fbk` + correct backup password, **with `PRAGMA cipher_integrity_check` returning zero rows**, → `VerifyResult(ok=True)` with `schema_version == LATEST_SCHEMA_VERSION` (the **as-migrated** version — `open` upgrades the temp copy, step 3, so verify does not report the backup's as-stored version) and `table_counts` equal to the opened-and-migrated temp's actual per-table counts. A vault with zero *transactions* (only the seed + `schema_version` rows every migrated vault carries — an all-zero `table_counts` is unreachable) is still `ok=True`. |
| **INV-3** | A wrong backup password → `VerifyResult(ok=False, reason=…)` (from the caught `DatabaseError`), no exception escapes, live vault untouched. |
| **INV-4** | Each FIBR-0014 tamper class (missing/renamed/extra entry, oversized entry, traversal, bad `format_version`/`sqlcipher_compat`, KDF below floor, `schema_version` > supported), **plus a body/overflow-page corruption caught by `cipher_integrity_check` that page-1 + `count(*)` would pass, and a missing / non-`.fbk` / non-zip picked file**, → `VerifyResult(ok=False)` with a reason naming the class — reusing the existing guards, not a second implementation. |
| **INV-5** | Verify leaves **no** temp files behind (temp dir removed on success and on every failure path) and writes no decrypted plaintext to disk (only backup-keyed ciphertext + the `0o600` params temp, both inside the removed temp dir). |
| **INV-6** | `restore_backup`'s externally-observable behaviour is unchanged by the D1 extraction — the FIBR-0014 restore test suite passes untouched. |
| **INV-7** | Verify leaves no live key material behind: the backup-password buffer and the derived backup key are zeroed on every path (success, wrong password, corrupt, unexpected error) — the project-wide key-hygiene rule restore honours. The wipe is owned by `_open_backup_vault` (D1) and is observable through the `on_key` test seam. |

## Deliverables

1. **`src/finbreak/services/backup.py`** — extract `_open_backup_vault(src,
   backup_password, work_dir, *, on_key) -> Vault` (shared with `restore_backup`,
   D1; the caller owns `work_dir`, the helper wipes the backup key + password and
   raises the failure classes raw); add `verify_backup(src, backup_password, *,
   on_key=None) -> VerifyResult` (opens under a system `TemporaryDirectory`, runs
   `cipher_integrity_check`, reads schema/counts, maps each failure class to a
   `reason` code, closes); a frozen `VerifyResult` dataclass in `backup.py`
   (alongside the existing Export/restore types).
2. **`src/finbreak/ui/backup_verify.py`** — a small `BackupVerifyDialog(QDialog)`
   mirroring `BackupExportDialog`: a `.fbk` file picker + backup-password field
   → `verify_requested` signal; a result area that renders ok ("Backup is
   readable — schema v10, N transactions, …") or the failure reason. The result
   area renders the `table_counts` dict — it may surface a couple of known,
   friendly table names (e.g. `transactions`) and otherwise show the raw
   `table: count` list; it must not assume a fixed schema.
3. **`src/finbreak/ui/settings.py`** — a "Verify backup…" button
   (`objectName="settings_verify_backup"`) next to Export, emitting
   `verify_backup_requested`.
4. **`src/finbreak/ui/main_window.py`** — wire `verify_backup_requested` → open
   the verify dialog → run `BackupService.verify_backup` **synchronously under a
   wait cursor** (mirroring Export, so the blocked event loop keeps auto-lock
   from firing mid-verify — D7) → show the `VerifyResult` in the dialog.
5. **`docs/security-model.md`** — a one-line note on **T5** (crafted `.fbk`
   parsed pre-login) that a backup can now be *verified* read-only through the
   same guards; no new threat surface (verify parses a `.fbk` exactly as restore
   already does). No new §5 INV (verify's guarantees are FIBR-0033-spec-local).
6. **`tests/features/backup/`** — extend with verify legs reusing
   `_export_from_seed` + `_rebuild_fbk`: valid → ok + counts + a passing
   `cipher_integrity_check` (INV-2); an older-schema backup → ok=True with
   `schema_version == LATEST_SCHEMA_VERSION` (INV-2 as-migrated); a
   zero-*transaction* backup → ok=True (INV-2); wrong password → ok=False (INV-3);
   each tamper class, a **body/overflow-page corruption** (a flipped byte in the
   `vault.db` entry that page-1 + `count(*)` pass but `cipher_integrity_check`
   catches), and a missing/non-zip file → ok=False naming it (INV-4);
   live-vault-untouched + no-new-files-in-vault-dir before/after (INV-1);
   temp-dir-empty after every path (INV-5); backup key + password wiped after
   verify via the `on_key` seam (INV-7); a restore-still-works regression (INV-6).
7. **Roadmap** flip FIBR-0033 ✅; CHANGELOG `[Unreleased]/Added`.

## Out of scope

- Pre-login (locked) placement of Verify — post-login Settings only for now; the
  locked-out recovery case (verify when you can no longer unlock the live vault)
  is a **separate deferred enhancement**, tracked as a roadmap follow-up.
- Auto-verifying a backup right after export (a possible later nicety).
- Verifying anything about the *live* vault (this is about a `.fbk` file).
- Repairing a bad backup — verify only reports.

## Exit criteria

- `./scripts/ci-local.sh` green (ruff, format, bandit, pip-audit, gitleaks,
  pytest, mypy).
- INV-1..7 each have a falsifiable test; INV-1 asserts the live vault (content
  hash + no new files in the vault dir), INV-5 the temp dir (emptiness), INV-7
  the wiped key/password (via the `on_key` seam).
- The FIBR-0014 restore suite passes unmodified (INV-6).
