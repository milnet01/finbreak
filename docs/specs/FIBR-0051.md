**Theme:** every phase to date grew the app **inside a bare full-screen
panel-swapper** (`AppShell` = a `QStackedWidget` that replaces the whole window
with whichever screen is current — first-run form, unlock box, main screen,
accounts, categories, import). There is **no window chrome**: no menubar, no
toolbar, no status bar, and the first thing a new user meets is a tiny
password box. This phase turns that into a **real desktop app window** — a
`QMainWindow` with a menubar, an icon toolbar of shortcuts, a status bar that
narrates activity, and a central content area — and moves first-run and unlock

from the first launch. It is the **frame** the remaining feature phases (P08
rules, P09 transfers, P10 dashboard) drop into: this ships **first**, on purpose,
so those features are built against the finished shell with no rewiring
(build-order decision, 2026-07-05). **No new persisted data, no schema change, no
new pip dependency** — this is a UI reshape that reuses every existing screen and
service.

## Context

The app is layered **UI → Services → Repositories → SQLCipher** (design.md). Today
the **entire UI** is `app.py`'s **`AppShell(QStackedWidget)`**: it holds one
full-screen `QWidget` at a time and swaps it (`_swap` removes + deletes the old,
adds + shows the new). Its routing:

- **`AppShell.__init__`** reads `service.state()` — `"first_run"` → `FirstRunWidget`,
  else → `UnlockWidget` — and wires `service.on_auto_lock = self._show_unlock`.
- **`FirstRunWidget`** (`completed` signal) and **`UnlockWidget`** (`unlocked` /
  `unlock_failed` signals) are **full-screen `QWidget`s**; on success the shell
  swaps in **`MainWindow`**.
- **`MainWindow(QWidget)`** *is* the main screen: a manual-entry `QFormLayout`
  (Account / Date / Amount / Description) + a row of **plain `QPushButton`s**
  (`Add`, `Manage accounts…`, `Manage categories…`, `Import transactions…`,
  `Lock`) + an error `QLabel` + the transaction `QTableWidget`. It emits `locked`
  / `manage_accounts` / `manage_categories` / `import_transactions`; the shell
  swaps in `AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget` (each a
  `QWidget` with a **`done`** signal that returns to a fresh `MainWindow`).
- **`run()`** builds the `QApplication`, wires `app.aboutToQuit →
  service.on_about_to_quit`, constructs `AppShell` (catching `VaultStateError` →
  `QMessageBox.critical` + `return 1`), and `shell.show()`.

This phase **reuses, unchanged**, the machinery already shipped:

- **`AuthService`** — `state()` (`→ presence_state()`: returns `"first_run"` /
  `"unlock"` — the token for the both-files-present locked state — or raises
  `VaultStateError`), `complete_first_run` / `complete_unlock` (called by
  the dialogs' existing worker flow), `lock()`, `on_auto_lock`,
  `on_about_to_quit`, `vault`. **No change.**
- **`FirstRunWidget` / `UnlockWidget` internals** — the two-field password form +
  currency picker (first-run), the masked field (unlock), the background
  **`DeriveWorker`** KDF flow, the `validate_first_run` / `complete_unlock`
  wiring, the error messages. This phase **re-homes** these into `QDialog`
  subclasses (D2) but keeps their derivation logic verbatim.
- **`AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget`** — the content
  screens, each with a **`done`** signal. Reused as content-stack pages **with no
  change to the widgets themselves** (D4).
- **`TransactionService`** (`add_transaction` / `list_transactions` /
  `base_currency`) and **`_format_amount`** — the manual-entry + table logic
  lifted out of `MainWindow` into `ManualEntryDialog` + `HomeView` (D3), calling
  the same service.
- **`PasswordDialog`** (`ui/password_dialog.py`) — the **template** for the new
  dialogs' shape (`QDialog` + `QDialogButtonBox` + accessor methods + `exec()` →
  `Accepted`/`Rejected`).

So the genuinely new code is: the `QMainWindow` shell + its state machine
(`ui/main_window.py`, rewritten), a `HomeView` (`ui/home.py`), a
`ManualEntryDialog` (`ui/manual_entry.py`), the `QWidget → QDialog` re-home of
first-run/unlock, a small bundled SVG icon set + loader, and the Donate wiring.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` gates them before code. **D5
carries the load-bearing security decision** (locked-shell data hygiene); the
rest are UI-structure/idiom calls.

- **D1 — `QMainWindow` is the top-level; the content stack is its central widget.**
  Replace `AppShell(QStackedWidget)` with a `QMainWindow` subclass whose
  `setCentralWidget` is a `QStackedWidget` content area. `QMainWindow` is Qt's
  purpose-built app-window (native menubar/toolbar/status-bar docking regions);
  bolting chrome onto a bare `QStackedWidget` would reinvent it (rejected — more
  code, less idiomatic; coding.md § 1.1). `run()` builds the `QMainWindow`
  directly and keeps the `aboutToQuit` + `VaultStateError` handling verbatim
  (INV-5). The window title stays `"finbreak"`.
- **D2 — First-run & unlock become application-modal `QDialog`s shown
  *non-blocking* over the window.** The approved UX is *window first, popup over
  it* — for **both** first-run and the returning-user unlock. So
  `FirstRunWidget(QWidget)` → **`FirstRunDialog(QDialog)`** and
  `UnlockWidget(QWidget)` → **`UnlockDialog(QDialog)`**. The shell shows each with
  **`setModal(True)` + `show()`** — application-modal *appearance* (input to the
  main window is blocked) **without** a nested event loop — **not** `exec()`. This
  is deliberate: the internal **`DeriveWorker`** (a background `QThread`) keeps
  running on the **normal** event loop, so its queued `done`/`failed` signals are
  delivered exactly as today and there is **no** nested-loop / worker-teardown race
  (an `exec()` nested loop plus a transient dialog dropped right after `accept()`
  risks "QThread destroyed while thread is still running"; `show()` avoids it). The
  dialogs therefore **keep their existing signals** — `FirstRunDialog.completed`,
  `UnlockDialog.unlocked` / `unlock_failed` — and the shell connects to them
  (`completed` / `unlocked` → `_enter_unlocked`: closes the dialog, enables chrome,
  shows Home; `unlock_failed` → the error stays in the still-open dialog for a
  retry, as `UnlockWidget` does today). **The shell holds a reference to the live
  dialog** (`self._dialog`) until success or dismissal, then closes +
  `deleteLater()`s it **and sets `self._dialog = None`**. Every teardown path
  (success, `reject`, `committed`, `_lock`) clears it, and `_lock`'s close is a
  **no-op when `self._dialog is None`** — so a second lock (or a lock after a normal
  close) never calls `.close()` on a `deleteLater`'d object (no `RuntimeError:
  Internal C++ object already deleted`). **The dialog always outlives its *running* worker** — teardown is blocked until the worker's `done`/`failed` clears `self._worker` (see below), so the parented `QThread` is only ever destroyed (with its parent dialog) *after* it has finished, never mid-run. **Test seam (preserved):** because the
  dialogs stay signal-driven and non-blocking, tests drive them with
  **`qtbot.waitSignal(dialog.completed / .unlocked)`** exactly as the FIBR-0004
  vault tests drive the current `UnlockWidget` (`test_vault.py:479-486`) — no
  monkeypatch of a blocking `exec()` is needed. **Cancel is disabled while a
  derivation is in flight.** The `DeriveWorker` `QThread` is **parented to the
  dialog** (`first_run.py:88` / `unlock.py:63`, unchanged), so dismissing the dialog
  *during* the ~1–2 s Argon2id derivation would `deleteLater()` the dialog and with
  it the **still-running child thread** — the very "QThread destroyed while thread
  is still running" crash this design avoids. So each dialog **disables its Cancel
  button (and blocks window-close / `reject()`) while `self._worker is not None`**
  (the dialog already tracks the worker, and today's widgets already disable the
  *submit* button the same way), re-enabling Cancel on **every** path that clears
  `self._worker` — the worker's `done`/`failed` **and** any early-return branch
  (e.g. `first_run.py`'s `if params is None: return`) — so no exit leaves Cancel
  stuck disabled. No teardown ever races a live worker (INV-2f). **Dismissal (only when idle):** the
  dialog's `QDialogButtonBox` Cancel / window-close fires `rejected` — for
  `FirstRunDialog` the shell **quits** the app (no vault can be created); for
  `UnlockDialog` the shell leaves the **locked shell** on screen (chrome disabled,
  🔒 Locked placeholder + an **Unlock** button, INV-2) so a stray Escape neither
  kills the app nor unlocks it. Rejected alt: `exec()` modal (the nested-loop +
  worker-teardown + test-seam costs above); stacked full-screen panels
  (contradicts the window-then-popup UX). **Entry trigger (two-phase, so the
  window paints first):** the shell's `__init__` reads `state()` (a `VaultStateError`
  here propagates to `run()`'s guard, INV-2c), builds the chrome + the state's
  placeholder, **creates** the dialog (not yet shown), and schedules its `show()`
  via **`QTimer.singleShot(0, self._show_pending_dialog)`**. `run()` then calls
  `window.show()`; the single-shot fires on the **first event-loop turn** — after the
  window is up — so the dialog opens *over* an already-painted window (no blank
  flash), with no blocking `exec()`.
- **D3 — `MainWindow` splits into the shell + `HomeView` + `ManualEntryDialog`.**
  Today's `MainWindow(QWidget)` conflates the *frame* with the *content* (the
  manual-entry form and the table). Split it: the new `MainWindow(QMainWindow)` is
  the **frame + state machine**; **`HomeView(QWidget)`** (`ui/home.py`) holds the
  getting-started panel (shown when there are **zero transactions**) **and** the
  transaction `QTableWidget` (shown once there is data),
  with the `_format_amount` `QLocale` helper moved verbatim beside it;
  **`ManualEntryDialog(QDialog)`** (`ui/manual_entry.py`) holds the
  Account/Date/Amount/Description form, opened by the `Manual entry` action.
  `HomeView.refresh()` re-reads via `TransactionService` (unchanged) and reports
  the row count for the status bar (INV-7). This keeps each file focused (the
  brainstorming isolation principle) and is the split the "add-transaction becomes
  a popup, content area becomes Home/dashboard" UX requires. The old
  `manage_accounts` / `manage_categories` / `import_transactions` / `locked`
  signals are replaced by direct `QAction` handlers on the shell.
- **D4 — Content screens reused unchanged, in a *one-widget-at-a-time* content
  stack.** `AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget` are already
  self-contained `QWidget`s with a `done` signal. **The content `QStackedWidget`
  holds exactly ONE content widget at a time:** `_set_live(widget)` (and
  `_show_home`) **first remove + `deleteLater()` the current widget, then add the
  new one** — the exact `AppShell._swap` pattern in use today (`app.py:78-85`, which
  removes+deletes every existing widget before adding). Consequences: there is
  **never a hidden `HomeView` behind another page** (so a lock destroys the *sole*
  content widget — no decrypted rows survive, INV-3/D5), and **"return to Home"
  builds a *fresh* `HomeView`** (not a re-shown one). `done` → `_show_home`; auto-lock
  → the 🔒 Locked placeholder (INV-4). **No change to the three widgets.** (Their
  internal modality/behaviour — e.g. the import wizard's own dialogs — is out of
  scope for this phase.)
- **D5 — Locked-shell data hygiene: the content widget is *destroyed* on lock, not
  hidden.** (Security, INV-3.) In the `"first_run"` and `"unlock"` states the
  content area shows only a **placeholder** (`Welcome` / `🔒 Locked`); no
  `HomeView` (or other content page) exists yet, so nothing reads the vault.
  `HomeView.refresh()` (the only method that reads transactions) runs **only** on
  an unlock/first-run success and after a data-changing action. On **lock /
  auto-lock** the current content widget is **removed from the stack and
  `deleteLater()`d** (the build-then-discard pattern the old `AppShell._swap`
  already uses) and the content reverts to the 🔒 Locked placeholder — so a
  `HomeView` holding decrypted `QTableWidgetItem`s **cannot survive a lock**; the
  next unlock builds a **fresh** `HomeView`. Rejected alt: keep one persistent
  `HomeView` and merely `hide()` it while locked (or add a `clear()`) — risks a
  data widget retaining decrypted rows across a lock; destroying it is simpler and
  leak-free by construction. This preserves FIBR-0004's "nothing visible until
  unlocked" posture through the reshaped routing. (INV-4a asserts the **same
  `MainWindow`/shell instance** persists across the lock — it is the *content
  widget*, not the window, that is rebuilt.)
- **D6 — Donate = `.github/FUNDING.yml` links via `QDesktopServices`.** A `Donate`
  menu (last on the menubar) with three items → `QDesktopServices.openUrl(QUrl(u))`
  for `u` in module-level constants derived from `.github/FUNDING.yml`
  (`github: milnet01` → `https://github.com/sponsors/milnet01`;
  `patreon: AntsProjectsHub` → `https://www.patreon.com/AntsProjectsHub`;
  `custom` → `https://paybru.co.za/tip/ants-projects-hub`, verbatim). `openUrl`
  hands the URL to the OS browser — a **user-initiated** action, **not** an app
  network call (the app opens no socket; security-model INV-8 holds, INV-8). The
  URLs are **constants in the source**, not read from `FUNDING.yml` at runtime
  (no file read on the happy path; the two are kept in sync by a comment
  cross-referencing `FUNDING.yml`). Tested by monkeypatching `openUrl` and
  asserting the `QUrl` argument (no real browser launches in the test).
- **D7 — Icons: hand-authored minimal monochrome SVG glyphs, bundled as package
  data.** The toolbar actions carry small **hand-drawn SVG glyphs** (a `＋` for
  Manual entry, a down-arrow for Import, etc.) shipped in **`src/finbreak/ui/icons/`**
  and loaded via `QIcon(str(path))`, with `ToolButtonTextUnderIcon` so a **text
  label always shows** under each icon (unambiguous + translatable, INV-10 — the
  **label**, not the glyph, carries the meaning, so an imperfectly-tinted icon is
  never ambiguous). Hand-authored (not a third-party icon set) so there is **zero
  icon-library licensing** to track and no new dependency. The app currently ships
  **no custom theme** (verified — no `setStyleSheet` / `QPalette` / `setStyle` in
  `src/finbreak`; it inherits the OS palette, light **or** dark), so the glyphs use
  a **neutral mid-tone** that reads acceptably on both; **palette-adaptive icon
  re-tinting** is deferred to FIBR-0014 (P12 — the ROADMAP bullet's "dark-theme
  polish"). Rejected alts: `QIcon.fromTheme` (unreliable in a frozen cross-platform
  bundle — no icon theme on Windows/macOS); Qt `SP_*` standard pixmaps (a limited,
  mismatched set). **Bundle obligation:** the `icons/` SVGs are **package data**

  travelled) (DoD #2), plus a source-tree unit test that every
  named icon renders. A **branded app/window icon** is deferred (§ Out of scope —
  FIBR-0037).
- **D8 — Status bar: transient `showMessage` + one permanent count widget.** A
  helper on the shell (`_status(text)`) calls `statusBar().showMessage(text,
  timeout)` — **callers pass an already-`tr()`-wrapped string *literal*** (e.g.
  `self._status(self.tr("Vault locked"))`), and the helper does **not** re-wrap.
  This matters: `self.tr(variable)` cannot be extracted by `lupdate` (it scans for
  `tr("literal")`), so wrapping inside `_status` would silently leave every
  status string out of the translation catalog — defeating INV-10 for exactly the
  new strings this phase adds. A permanent `QLabel` added via `addPermanentWidget`
  holds the transaction count, updated from `HomeView.refresh()`'s row count and
  hidden while locked (INV-7). At rest the transient area reads a `tr("Ready")`
  literal. Simple + idiomatic; no activity-queue over-engineering (coding.md § 1.1).

  re-enabled and the dialog stands.
- **INV-4a** (auto-lock) — driving the idle timeout through
  **`AuthService._on_idle_timeout()`** (production's entry point — it calls `lock()`

  `AccountsWidget` in the content stack (the prior content widget destroyed, D4);
  emitting its `done` returns to Home (a fresh `HomeView`, refreshed); likewise
  Categories/Import.
- **INV-8a** (donate) — each of the three Donate actions calls a monkeypatched
  `QDesktopServices.openUrl` exactly once with a `QUrl` equal to the pinned

  sole `openUrl` callers" — a stray fourth call fails it). **Plus a sync check:** the test

  parse of the three keys — **not** `import yaml`, since PyYAML is only a
  *transitive* bandit dep, not a declared one) and asserts each pinned constant
  matches the derived URL, so a silent drift between the source constants and

  `patreon: AntsProjectsHub`, `custom: ["https://paybru.co.za/tip/ants-projects-hub"]`
