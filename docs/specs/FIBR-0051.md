## Context

The app is layered **UI ‚Üí Services ‚Üí Repositories ‚Üí SQLCipher** (design.md). Today
the **entire UI** is `app.py`'s **`AppShell(QStackedWidget)`**: it holds one
full-screen `QWidget` at a time and swaps it (`_swap` removes + deletes the old,
adds + shows the new). Its routing:

- **`AppShell.__init__`** reads `service.state()` ‚Äî `"first_run"` ‚Üí `FirstRunWidget`,
  else ‚Üí `UnlockWidget` ‚Äî and wires `service.on_auto_lock = self._show_unlock`.
- **`FirstRunWidget`** (`completed` signal) and **`UnlockWidget`** (`unlocked` /
  `unlock_failed` signals) are **full-screen `QWidget`s**; on success the shell
  swaps in **`MainWindow`**.
- **`MainWindow(QWidget)`** *is* the main screen: a manual-entry `QFormLayout`
  (Account / Date / Amount / Description) + a row of **plain `QPushButton`s**
  (`Add`, `Manage accounts‚Ä¶`, `Manage categories‚Ä¶`, `Import transactions‚Ä¶`,
  `Lock`) + an error `QLabel` + the transaction `QTableWidget`. It emits `locked`
  / `manage_accounts` / `manage_categories` / `import_transactions`; the shell
  swaps in `AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget` (each a
  `QWidget` with a **`done`** signal that returns to a fresh `MainWindow`).
- **`run()`** builds the `QApplication`, wires `app.aboutToQuit ‚Üí
  service.on_about_to_quit`, constructs `AppShell` (catching `VaultStateError` ‚Üí
  `QMessageBox.critical` + `return 1`), and `shell.show()`.

This phase **reuses, unchanged**, the machinery already shipped:

- **`AuthService`** ‚Äî `state()` (`‚Üí presence_state()`: `"first_run"` / `"locked"`
  / raises `VaultStateError`), `complete_first_run` / `complete_unlock` (called by
  the dialogs' existing worker flow), `lock()`, `on_auto_lock`,
  `on_about_to_quit`, `vault`. **No change.**
- **`FirstRunWidget` / `UnlockWidget` internals** ‚Äî the two-field password form +
  currency picker (first-run), the masked field (unlock), the background
  **`DeriveWorker`** KDF flow, the `validate_first_run` / `complete_unlock`
  wiring, the error messages. This phase **re-homes** these into `QDialog`
  subclasses (D2) but keeps their derivation logic verbatim.
- **`AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget`** ‚Äî the content
  screens, each with a **`done`** signal. Reused as content-stack pages **with no
  change to the widgets themselves** (D4).
- **`TransactionService`** (`add_transaction` / `list_transactions` /
  `base_currency`) and **`_format_amount`** ‚Äî the manual-entry + table logic
  lifted out of `MainWindow` into `ManualEntryDialog` + `HomeView` (D3), calling
  the same service.
- **`PasswordDialog`** (`ui/password_dialog.py`) ‚Äî the **template** for the new
  dialogs' shape (`QDialog` + `QDialogButtonBox` + accessor methods + `exec()` ‚Üí
  `Accepted`/`Rejected`).

So the genuinely new code is: the `QMainWindow` shell + its state machine
(`ui/main_window.py`, rewritten), a `HomeView` (`ui/home.py`), a
`ManualEntryDialog` (`ui/manual_entry.py`), the `QWidget ‚Üí QDialog` re-home of
first-run/unlock, a small bundled SVG icon set + loader, and the Donate wiring.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` gates them before code. **D5
carries the load-bearing security decision** (locked-shell data hygiene); the
rest are UI-structure/idiom calls.

- **D1 ‚Äî `QMainWindow` is the top-level; the content stack is its central widget.**
  Replace `AppShell(QStackedWidget)` with a `QMainWindow` subclass whose
  `setCentralWidget` is a `QStackedWidget` content area. `QMainWindow` is Qt's
  purpose-built app-window (native menubar/toolbar/status-bar docking regions);
  bolting chrome onto a bare `QStackedWidget` would reinvent it (rejected ‚Äî more
  code, less idiomatic; coding.md ¬ß 1.1). `run()` builds the `QMainWindow`
  directly and keeps the `aboutToQuit` + `VaultStateError` handling verbatim
  (INV-5). The window title stays `"finbreak"`.
- **D2 ‚Äî First-run & unlock become modal `QDialog`s shown over the window.** The
  approved UX is *window first, popup over it* ‚Äî for **both** first-run and the
  returning-user unlock. So `FirstRunWidget(QWidget)` ‚Üí **`FirstRunDialog(QDialog)`**
  and `UnlockWidget(QWidget)` ‚Üí **`UnlockDialog(QDialog)`**, mirroring
  `PasswordDialog`: the shell calls `dialog.exec()` (application-modal); on
  derivation success the dialog `accept()`s (the shell then enables chrome + shows
  Home), on failure the error stays **in the dialog** (the existing error
  `QLabel`) and the user retries. The internal `DeriveWorker` async flow is
  preserved ‚Äî `exec()`'s nested event loop still delivers the worker's queued
  `done`/`failed` signals (verified pattern: Qt delivers queued cross-thread
  signals inside a modal `exec()` loop). **Dialog dismissal:** closing/cancelling
  `FirstRunDialog` (no vault can be created) **quits** the app; closing/cancelling
  `UnlockDialog` leaves the **locked shell** on screen (chrome disabled, üîí Locked
  placeholder) with the placeholder's **Unlock** button (INV-2) to re-open the
  dialog ‚Äî so a stray Escape doesn't kill the app, and the vault stays locked.
  Rejected alt: keep them as stacked full-screen panels ‚Äî contradicts the approved
  window-then-popup UX. **Signals** `completed` / `unlocked` / `unlock_failed` are
  **removed** in favour of the `QDialog` accept/reject contract (test ripple:
  Deliverable 10).
- **D3 ‚Äî `MainWindow` splits into the shell + `HomeView` + `ManualEntryDialog`.**
  Today's `MainWindow(QWidget)` conflates the *frame* with the *content* (the
  manual-entry form and the table). Split it: the new `MainWindow(QMainWindow)` is
  the **frame + state machine**; **`HomeView(QWidget)`** (`ui/home.py`) holds the
  getting-started panel (shown when there are zero accounts **or** zero
  transactions) **and** the transaction `QTableWidget` (shown once there is data),
  with the `_format_amount` `QLocale` helper moved verbatim beside it;
  **`ManualEntryDialog(QDialog)`** (`ui/manual_entry.py`) holds the
  Account/Date/Amount/Description form, opened by the `Manual entry` action.
  `HomeView.refresh()` re-reads via `TransactionService` (unchanged) and reports
  the row count for the status bar (INV-7). This keeps each file focused (the
  brainstorming isolation principle) and is the split the "add-transaction becomes
  a popup, content area becomes Home/dashboard" UX requires. The old
  `manage_accounts` / `manage_categories` / `import_transactions` / `locked`
  signals are replaced by direct `QAction` handlers on the shell.
- **D4 ‚Äî Content screens reused unchanged as content-stack pages.**
  `AccountsWidget` / `CategoriesWidget` / `ImportWizardWidget` are already
  self-contained `QWidget`s with a `done` signal. They are added to the central
  `QStackedWidget` on demand and removed on `done` (returning to Home) ‚Äî the same
  build-then-discard pattern the current `AppShell._show_accounts` etc. use, so
  auto-lock still swaps a content page away (INV-4). **No change to the three
  widgets.** (Their internal modality/behaviour ‚Äî e.g. the import wizard's own
  dialogs ‚Äî is out of scope for this phase.)
- **D5 ‚Äî Locked-shell data hygiene: the table is never built/populated while
  locked.** (Security, INV-3.) In the first_run and locked states the content
  area shows a **placeholder** (`Welcome` / `üîí Locked`), and `HomeView` is either
  not shown or shown **un-refreshed** (no `refresh()` call, so it never touches
  the ‚Äî closed ‚Äî vault). `HomeView.refresh()` is called **only** on an
  unlock/first-run success and after a data-changing action. Rejected alt: build +
  populate Home always and merely hide it while locked ‚Äî risks a data widget
  holding decrypted rows across a lock; instead the table is empty until unlock
  and cleared on lock. This preserves FIBR-0004's "nothing visible until
  unlocked" posture through the reshaped routing.
- **D6 ‚Äî Donate = `.github/FUNDING.yml` links via `QDesktopServices`.** A `Donate`
  menu (last on the menubar) with three items ‚Üí `QDesktopServices.openUrl(QUrl(u))`
  for `u` in module-level constants derived from `.github/FUNDING.yml`
  (`github: milnet01` ‚Üí `https://github.com/sponsors/milnet01`;
  `patreon: AntsProjectsHub` ‚Üí `https://www.patreon.com/AntsProjectsHub`;
  `custom` ‚Üí `https://paybru.co.za/tip/ants-projects-hub`, verbatim). `openUrl`
  hands the URL to the OS browser ‚Äî a **user-initiated** action, **not** an app
  network call (the app opens no socket; security-model INV-8 holds, INV-8). The
  URLs are **constants in the source**, not read from `FUNDING.yml` at runtime
  (no file read on the happy path; the two are kept in sync by a comment
  cross-referencing `FUNDING.yml`). Tested by monkeypatching `openUrl` and
  asserting the `QUrl` argument (no real browser launches in the test).
- **D7 ‚Äî Icons: hand-authored minimal monochrome SVG glyphs, bundled as package
  data.** The toolbar actions carry small **hand-drawn SVG glyphs** (a `Ôºã` for
  Manual entry, a down-arrow for Import, etc.) shipped in **`src/finbreak/ui/icons/`**
  and loaded via `QIcon(str(path))`, with `ToolButtonTextUnderIcon` so a **text
  label always shows** under each icon (unambiguous + translatable, INV-10).
  Hand-authored (not a third-party icon set) so there is **zero icon-library
  licensing** to track and no new dependency; they are coloured for the **dark
  theme** (ADR-0002 ‚Äî the only theme today). Rejected alts: `QIcon.fromTheme`
  (unreliable in a frozen cross-platform bundle ‚Äî no icon theme on Windows/macOS);
  Qt `SP_*` standard pixmaps (a limited, mismatched set). **Bundle obligation:**
  the `icons/` SVGs are **package data** and MUST travel into the FIBR-0003 bundle
  (Deliverable 9); a unit test asserts each named icon file exists and loads to a
  **non-null** `QIcon`. **Theme-adaptive re-tinting** (light theme) and a
  **branded app/window icon** are deferred (¬ß Out of scope ‚Äî FIBR-0014 theme
  polish / FIBR-0037 branded icon).
- **D8 ‚Äî Status bar: transient `showMessage` + one permanent count widget.** A
  helper on the shell (`_status(msg)`) calls `statusBar().showMessage(self.tr(msg),
  timeout)` for action feedback (auto-clears to `"Ready"`); a permanent
  `QLabel` added via `addPermanentWidget` holds the transaction count, updated from
  `HomeView.refresh()`'s row count (INV-7). At rest the transient area reads
  `"Ready"`. Simple + idiomatic; no activity-queue over-engineering (coding.md
  ¬ß 1.1).
- **D9 ‚Äî No schema change, no new pip dependency, no new native lib.** This phase
  persists nothing new (`LATEST_SCHEMA_VERSION` **unchanged** ‚Äî no migration) and
  adds no runtime dependency (the SVGs are static assets, not a package). So there
  is **no** `== N` schema-version test ripple (unlike FIBR-0007/0009) and **no new
  native-bundle surface** ‚Äî the FIBR-0003 build smoke's role here is only to prove
  the **icon package-data** travels (Deliverable 9 / DoD #2), not a native stack.
  Stated explicitly so `/cold-eyes` doesn't hunt for a migration that isn't there.
- **D10 ‚Äî The routing/`MainWindow` test ripple is named, not discovered late.**
  Converting `AppShell(QStackedWidget)` ‚Üí `MainWindow(QMainWindow)` and
  first-run/unlock ‚Üí dialogs **breaks** the FIBR-0004 routing tests that assert
  `AppShell(...).currentWidget()` `isinstance` `MainWindow` / `UnlockWidget`
  (`tests/features/vault/test_vault.py:539-551`) and the accounts/vault tests that
  construct `MainWindow(service)` as a plain `QWidget`
  (`test_vault.py:509-529`, `test_accounts.py:373-396`). These are **edited**
  (not created): the routing test now asserts the **`QMainWindow` + which
  dialog/placeholder is shown** per state; the `MainWindow(service)` constructions
  that drove the old manual-entry form are re-pointed at `ManualEntryDialog` /
  `HomeView` as appropriate. The edits are enumerated in Deliverable 10 so the
  "all existing suites green" DoD (#4) is concrete.

## Invariants (testable)

The table under *At a glance* is the canonical list (INV-1..INV-10). Sub-invariants:

- **INV-2a** (first-run happy path, `qtbot`) ‚Äî from a `presence_state() ==
  "first_run"` vault dir, the shell shows with chrome disabled + Welcome
  placeholder and a `FirstRunDialog`; completing it (monkeypatched/driven
  derivation, as the FIBR-0004 vault tests already do) creates the vault, enables
  the chrome, and shows a populated-or-getting-started Home.
- **INV-2b** (unlock happy path, `qtbot`) ‚Äî from a `"locked"` vault dir, the shell
  shows chrome-disabled + üîí Locked placeholder + an `UnlockDialog`; a correct
  password enables chrome + shows Home; a wrong password keeps the dialog (error
  shown), chrome still disabled (reuses the FIBR-0004 wrong-password path).
- **INV-2c** (corrupt install) ‚Äî a mixed vault/sidecar pair still yields the
  `QMessageBox.critical` + non-zero exit at startup (`VaultStateError`, unchanged).
- **INV-4a** (auto-lock) ‚Äî invoking the wired `on_auto_lock` callback (as the
  FIBR-0004 tests invoke the timeout handler directly) closes the vault, clears +
  hides the Home table, disables chrome, and shows the üîí Locked placeholder +
  `UnlockDialog` ‚Äî **the same `MainWindow` instance** (window not destroyed).
- **INV-6a** (content routing) ‚Äî triggering the `Accounts` action shows
  `AccountsWidget` in the content stack; emitting its `done` returns to Home and
  re-refreshes; likewise Categories/Import.
- **INV-8a** (donate) ‚Äî each of the three Donate actions calls a monkeypatched
  `QDesktopServices.openUrl` exactly once with a `QUrl` equal to the pinned
  constant; **no** other action triggers `openUrl`.

The security invariants INV-3 (no data while locked), INV-4 (lock returns to
locked shell + key wiped), INV-5 (key lifetime untouched), and INV-8 (no app
network call) each carry a dedicated `qtbot`/monkeypatch test.
