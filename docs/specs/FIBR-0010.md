**Theme:** Teach finbreak to file transactions into categories **by itself**,
from a user-editable rule set, while a category **you set by hand always wins**
and is never overwritten. Each rule is `‹text to look for› → ‹category›`. Rules
run **on import** and on an explicit **"Apply rules now"**. When you correct a
category the app had auto-assigned, it **offers to make a rule** so future
transactions follow the correction. Deleting a category cleanly unwinds the rules
and transactions that pointed at it and lets the remaining rules reclaim them.

## Context

- **`categories`** (FIBR-0006, v3) is a self-referential tree: two seeded Type
  **roots** (`parent_id IS NULL`, `kind` = `income`/`expenditure`) and their
  **children** (`kind IS NULL`). `CategoryService` creates/edits/deletes only
  children; `delete_category` today refuses a root (`ProtectedCategoryError`) or a
  category with children (`CategoryHasChildrenError`) and otherwise deletes the
  leaf. `PRAGMA foreign_keys = ON` on every vault connection (`vault.py:142`).
- **`transactions`** (v2 rebuild; v6 stamp) has `id, account_id, occurred_on,
  amount_minor, description, created_at, statement_period_id`. **No category link
  today** — this phase adds it. `TransactionRepository.list_all` `SELECT`s the six
  original columns into `Transaction(*row)` (the v6 `statement_period_id` is a
  table column but is **not** in the dataclass/SELECT).
- **Import** (`ImportService.commit_import`, FIBR-0007) inserts deduped rows inside
  one owned `BEGIN…COMMIT`. Its `_normalise` (casefold + collapse whitespace) is
  the text-matching primitive the rule engine reuses. Re-import only **adds**
  genuinely-new rows (multiset-delta dedup), so existing rows — and their
  categories — already survive re-import (INV-3's re-import half is free).
- **Shell** (`main_window.py`): a 4-tab `QTabWidget` workspace (Home/Statements/
  Accounts/Categories); `_build_workspace` builds each tab once,
  `_refresh_tab(index)` refreshes on navigation. `HomeView` (`home.py`) is a
  read-only 4-column table (Date/Amount/Description/Account) built from
  `TransactionService.list_transactions()` → `[(Transaction, Decimal display,
  account_name)]`. The View menu + toolbar hold per-tab actions built by
  `_make_action`; Settings is the one management surface that is a **dialog**
  (under File), while Accounts/Categories are **tabs**.

## The category-source model

Every transaction carries two new nullable fields:

- `category_id` — the assigned leaf category, or `NULL` (uncategorised).
- `category_source` — **`'manual'`** (you set it — frozen) or **`'rule'`**
  (a rule set it), or `NULL` (never touched → auto, uncategorised).

"**Auto**" = `category_source` is **not** `'manual'` (i.e. `'rule'` **or**
`NULL`). The engine's whole contract (INV-1): *recompute every auto row from the
current rules; leave every manual row alone.* A manual **clear** is
`category_id = NULL, category_source = 'manual'` — deliberately distinct from a
never-touched `NULL/NULL`, so a cleared row is not re-filled by a rule (INV-3).

`CategorySource` is a `StrEnum` (`RULE = "rule"`, `MANUAL = "manual"`) alongside
`AccountType`/`CategoryKind` in `models.py`; the stored `.value` is the token.

## Design decisions

- **D1 — Schema v7 (INV-15).** `_migrate_to_v7`, one atomic unit (explicit
  `BEGIN` first statement, `UPDATE schema_version` last before `COMMIT`,
  `ROLLBACK` on exception — the established pattern). Two nullable `ADD COLUMN`s on
  `transactions` (`category_id INTEGER REFERENCES categories(id)` — plain, default
  `NULL`, so an in-place add per the v6 precedent; `category_source TEXT`) and one
  `CREATE TABLE categorization_rules(id INTEGER PRIMARY KEY, pattern TEXT NOT NULL,
  category_id INTEGER NOT NULL REFERENCES categories(id), priority INTEGER NOT
  NULL, created_at TEXT NOT NULL)`. **No `ON DELETE` clause** on either FK — with
  `foreign_keys = ON` the delete cascade (INV-7) clears references explicitly
  *before* deleting the category, inside its own transaction, which also lets it
  reset `category_source` and re-apply (things `ON DELETE SET NULL` cannot do). No
  backfill (pre-v7 rows are all auto/uncategorised — `NULL/NULL` — which is exactly
  correct; the first rule run categorises them). Register `7: _migrate_to_v7`.

- **D2 — Shared text normaliser (coding.md § 1.3).** The rule matcher needs the same
  casefold-+-collapse-whitespace normalisation `ImportService._normalise` uses.
  This is the **second** call-site of that concept — per reuse-before-rewrite,
  **extract** it to a module function `normalise_text(s: str) -> str` (new
  `finbreak/text.py`, one tiny pure helper) and have **both** `ImportService`
  (its `_normalise` becomes a one-line delegator, so its dedup behaviour is
  byte-identical) and the rule engine call it. (Rule of Three is satisfied: import
  dedup + rule match + the learning "differs" check are three uses.)

- **D3 — Pure matcher (INV-2, INV-9).** `categorize(description: str, rules:
  Sequence[CategorizationRule]) -> int | None` (module function in
  `services/categorization.py`): normalise the description once; return the
  `category_id` of the **first** rule (ascending `priority`) whose
  `normalise_text(pattern)` is a substring of it; else `None`. Pure — no DB, takes
  the rules list — so it is trivially unit-tested and reused by both the apply
  routine and the learning "differs" check. Rules only ever target leaves (D6
  validates on write), so a returned id is always an assignable child. **Empty
  rule set** (`categorize(desc, [])`) or no rule matching → `None` — the natural
  base case (a vault with no rules leaves every auto row uncategorised).

- **D4 — Apply routine, commit-free + wrapped (INV-1, INV-4, INV-13).** A module
  function `recategorize_auto_rows(conn) -> int` (commit-free): read the rules once
  (`list_all`, priority order) and every **auto** row's `(id, description)`
  (`category_source IS NULL OR category_source <> 'manual'`); for each, compute
  `categorize(description, rules)`; write the new `(category_id, source)` — `source
  = 'rule'` when a rule matched, else `NULL` (auto/uncategorised) — **only for rows
  whose value actually changes** (so INV-13 idempotence holds and the row-write
  count is the honest "re-filed" number returned). Manual rows are excluded by the
  `WHERE`, never read, never written (INV-1). `CategorizationService.apply_rules()`
  wraps it in one owned `BEGIN…COMMIT`/`ROLLBACK` and returns the count;
  **`ImportService.commit_import`** (D9) and **`CategoryService.delete_category`**
  (D8) both call the **commit-free** `recategorize_auto_rows(conn)` inside their
  *own* transactions, so categorisation is part of the same atomic import / delete
  (no uncategorised flash, no second transaction). **What NOT to do:** the read
  predicate must be the full `category_source IS NULL OR category_source <>
  'manual'` — do **not** collapse it to a bare `category_source <> 'manual'`,
  because in SQL three-valued logic `NULL <> 'manual'` evaluates to `NULL`
  (unknown), *not* true, so a bare `<> 'manual'` silently drops every
  never-touched `NULL`-source row — the **dominant** auto state right after an
  import (D9 inserts `NULL/NULL`), which would leave fresh imports uncategorised.
  The `IS NULL OR …` disjunct is load-bearing.

- **D5 — `CategorizationRuleRepository`** (`repositories/categorization_rules.py`,
  mirroring `CategoryRepository`): `list_all() -> list[CategorizationRule]` (`ORDER
  BY priority, id`); `add(pattern, category_id, priority) -> int`; `update(id,
  pattern, category_id)`; `delete(id)`; `get(id) -> CategorizationRule | None`;
  `min_priority() -> int | None` (for D6 top-insert); `set_priority(id, priority)`
  (for D7 reorder); `delete_for_category(category_id) -> int` (for INV-7);
  `count_for_category(category_id) -> int` (for the INV-8 blast radius; shape-mirror
  of `TransactionRepository.count_for_account`). Standalone
  manager writes commit per-write (like `CategoryRepository`); the cascade path
  uses `delete_for_category` **commit-free** inside `CategoryService`'s transaction
  (the repo methods that both paths share follow the FIBR-0059 commit-free
  convention — the owning service commits).

- **D6 — New rules insert at the top (INV-6, INV-9).**
  `CategorizationService.add_rule(pattern, category_id)`: validate `pattern` is
  non-empty after strip (`ValueError` otherwise) and `category_id` is an existing
  **child** category (not a root — reuse a `CategoryRepository.get` + `parent_id
  is not None` check; `ValueError` otherwise); insert with `priority =
  (min_priority() or 0) - 1` so it sorts first (a genuine `0` minimum is treated
  like `None` here — harmless: `-1` still sorts before `0`). `add_rule` does **not** auto-apply
  (INV-4) — the manager's explicit "Apply rules now" and the learning flow's own
  re-apply own that. Uniform rule: **newest is highest priority**; reorder if you
  disagree. (Learning depends on this — a correction must beat the rule it
  corrects, D11.) **`update_rule(rule_id, pattern, category_id)`** applies the
  **same** validation (non-empty stripped `pattern`; `category_id` an existing
  child, not a root) and, like `add_rule`, does **not** auto-apply (INV-4 — the
  next run re-files) and leaves the rule's `priority` unchanged (an edit is not a
  re-prioritise; use Move up/down for that, D7). This is the second half of INV-9.

- **D7 — Reorder by adjacent swap (INV-11).**
  `CategorizationService.move_rule(rule_id, direction)`: swap the `priority` of the
  target rule and its neighbour in the current order (no-op at the ends). Priorities
  need not be contiguous (D3/D5 only read their **order**), so a swap is a
  two-row `set_priority`. First-match is `ORDER BY priority, id`, so the `id`
  tiebreak keeps two equal priorities deterministic.

- **D8 — Delete-category cascade (INV-7, INV-8), in `CategoryService`.**
  `delete_category` keeps its two guards (root → `ProtectedCategoryError`; has
  children → `CategoryHasChildrenError`) then, for a leaf, becomes one owned
  `BEGIN…COMMIT`: (1) `UPDATE transactions SET category_id = NULL, category_source
  = NULL WHERE category_id = ?` (clears the ref **and** resets those rows to auto,
  so the re-apply can reclaim them — this includes rows that were *manual* to that
  category: the category is gone, so freezing to it is meaningless, per the user's
  2026-07-09 decision); (2) `rules.delete_for_category(id)`; (3) `repo.delete(id)`
  (now unreferenced, so `foreign_keys = ON` permits it); (4)
  `recategorize_auto_rows(conn)`; `commit`; `rollback` + re-raise on any failure
  (a re-openable vault). `CategoryService` gains a `CategorizationRuleRepository`
  accessor and imports `recategorize_auto_rows`. **Order matters:** clear refs and
  delete rules *before* the category delete (else the FK blocks it).

- **D9 — Import runs the rules (INV-4a).** `ImportService.commit_import`, as its
  **last** step before `conn.commit()` (after the `add_batch`), calls
  `recategorize_auto_rows(conn)`. Newly-inserted rows are auto (`NULL/NULL` — the
  batch insert sets no category), so they are categorised in the same transaction;
  existing manual rows are untouched (INV-3). `ImportResult` is unchanged (the
  count of *inserted* rows is still the import's headline number; categorisation is
  a side effect, not reported there). One new import: `from finbreak.services.
  categorization import recategorize_auto_rows`.

- **D10 — `HomeView` gains the Category column + manual-set (INV-10).** The table
  becomes 5 columns (Date/Amount/Description/Account/**Category**). `HomeView` is
  constructed with a `CategorizationService` in addition to its `TransactionService`
  (the shell passes both, `_build_workspace`); it no longer holds *only*
  `TransactionService` (the FIBR-0051 "Home holds only TransactionService" note is
  superseded here — category assignment genuinely needs category data). The table
  gets a right-click **context menu** (`customContextMenuRequested`) with **"Set
  category…"**, enabled on a row: it opens `CategoryPickerDialog` (leaves +
  Uncategorised, preselecting the row's current category); on accept it calls
  `CategorizationService.set_manual_category(txn_id, chosen_id)` (sets
  `category_id` + `'manual'`; the service **owns its `BEGIN…COMMIT`** — the repo
  `set_category` is commit-free), then runs the D11 learning check, then `refresh()`.
  `list_transactions` (D12) supplies the `txn.id` + current category per row. All
  strings `tr()`; the row→transaction map is the table's row order (same order as
  `list_transactions`). `VaultLockedError` around the service calls is caught
  (INV-14).

- **D11 — Learning offer (INV-5, INV-6).** Immediately after a successful manual
  set to a **non-None** leaf `chosen_id`, `HomeView` computes `would =
  CategorizationService.would_categorize(description)`; **iff `chosen_id != would`**
  it shows `RuleEditDialog` **pre-filled** with the transaction's raw description
  (editable) and the chosen category, framed *"Always file transactions like this
  as ‹Category›?"*. **Accept** → `CategorizationService.add_rule(pattern, chosen_id)`
  (top priority, D6) **then** `apply_rules()` (propagate to other auto rows) then
  `refresh()`. **Dismiss** → nothing more (the one row is already manual). The
  corrected row itself is **manual**, so the subsequent `apply_rules` leaves it
  exactly as set (INV-1) — learning changes *other* rows and the *future*, not the
  row you corrected. No offer when `chosen_id == would` (the rules already agree) or
  when `chosen_id is None` (a manual clear is not a rule to learn). **`would` is
  computed against the rules as they stand *before* the new rule is added** (the
  offer's whole point is that the current rules disagree with the pick).
  **Non-atomicity is deliberate:** the flow is three *separate* commits
  (`set_manual_category`, then `add_rule`, then `apply_rules`), not one. If
  `apply_rules` fails after `add_rule` commits, the rule simply persists un-applied
  and INV-4's next run (import or the manager's Apply) applies it — there is no
  lost-rule or half-applied state to recover, and the corrected row is already
  safely `manual`.

- **D12 — `TransactionService.list_transactions` carries the category (INV-10).**
  Extend its return to `[(Transaction, Decimal display, account_name, category_name)]`
  where `category_name` is the leaf's name or `""` (uncategorised); resolve
  id→name once (like account names) from a `{id: name}` map over
  `CategoryRepository.list_all()`. `Transaction` gains `category_id: int | None`
  and `category_source: str | None` (appended **after** `created_at`), and
  `TransactionRepository.list_all`'s `SELECT` adds the two columns in that order so
  `Transaction(*row)` stays aligned. `add()` (manual entry) inserts no category →
  the two columns default `NULL` (auto/uncategorised), correct.

- **D13 — Rules tab + dialogs (INV-11).** `RulesWidget` (`ui/rules.py`, mirroring
  `CategoriesWidget`): a `QTableWidget` (Pattern · Category, priority order) + Add /
  Edit / Delete / Move up / Move down buttons gated on selection + an **"Apply
  rules now"** button that calls `apply_rules()` and `_status`-style reports
  *"Re-filed N transactions."* — **N is the D4 changed-row count** (rows whose
  category actually changed, which may include rows that went blank), not the total
  categorised; a second Apply with unchanged rules reports 0 (INV-13).
  `RuleEditDialog` (same file, one dialog for **both** manager
  Add/Edit **and** the D11 learning offer): a `QLineEdit` for the pattern + a
  **category selector** (leaves grouped by root) + OK/Cancel; constructed with an
  optional pre-filled pattern + preselected category. The category selector and the
  D10 `CategoryPickerDialog`'s leaf list share one small helper that builds the
  leaf list from `CategoryService` (avoid two divergent tree-walks) — a
  `leaf_categories(service) -> list[Category]` module function in
  `ui/category_picker.py`. Wired as a **5th tab** (View menu `action_rules` +
  toolbar, `_open_rules`), consistent with Accounts/Categories being tabs — this
  refines the brainstormed "dialog" to a tab because every other management surface
  here is a tab (Settings is the lone dialog, and it has no list/reorder needs).
  (The manager grows **Delete / Move / Apply** beyond design.md's + the ROADMAP
  bullet's "view/add/edit" summary — a deliberate elaboration, not a contradiction.)

- **D14 — Shell wiring (INV-4, INV-10, INV-11).** `_build_workspace` builds
  `RulesWidget` as the 5th tab and passes a `CategorizationService` into both it and
  `HomeView`. `action_rules` ("Rules", `_open_rules` → switch to the tab) joins the
  View menu + toolbar. After an import (`_on_import_done`) the Home tab already
  refreshes; since D9 categorises inside the import, the refreshed Home shows the
  new categories with no extra call. When the Rules tab re-files (Apply now) or the
  Categories tab deletes a category, the Home tab is refreshed on next navigation
  (`_refresh_tab`) — the existing per-tab refresh already covers it; no new
  cross-tab signal is required for MVP (the user sees updated categories when they
  return to Home). *(A live cross-tab refresh is a possible polish, out of scope.)*

## Invariants

Restated with their enforcement point (the table above is the index).

- **INV-1 (golden rule)** — `recategorize_auto_rows` `SELECT`s only
  `category_source IS NULL OR category_source <> 'manual'` rows (the NULL-safe
  predicate, D4) and writes only those; a manual row is never in the read set. Tested: a manual row keeps its category across an
  `apply_rules()` that would otherwise match it to a different one.
- **INV-2 (first-match)** — `categorize` returns the first matching rule in
  `priority, id` order. Tested with two rules both matching one description →
  the higher-priority (smaller `priority`) one wins; no match → `None`.
- **INV-3 (manual survives)** — the re-import half is FIBR-0007 dedup (existing
  rows kept); the rule-run half is INV-1. Tested: import a statement, manually
  categorise a row, re-import the same file **and** press Apply → the manual row is
  unchanged; a manual **clear** (`NULL/'manual'`) stays clear after Apply.
- **INV-4 (when rules run)** — D9 (import) + the Apply button (D13). Tested: import
  categorises new rows in one go; adding a rule does **not** change existing rows
  until `apply_rules()`.
- **INV-5 (learning offer)** — the D11 `chosen_id != would` gate. Tested four
  ways: (a) correcting a rule-assigned row to a different leaf → offer shown, accept
  creates a rule; (b) categorising a blank row whose description no rule matches →
  offer shown; (c) setting the category the rules already produce → **no** offer;
  (d) a manual clear → **no** offer.

## Data model — schema v6 → v7

```sql
ALTER TABLE transactions ADD COLUMN category_id INTEGER REFERENCES categories(id);
ALTER TABLE transactions ADD COLUMN category_source TEXT;   -- 'rule' | 'manual' | NULL
CREATE TABLE categorization_rules(
    id INTEGER PRIMARY KEY,
    pattern TEXT NOT NULL,
    category_id INTEGER NOT NULL REFERENCES categories(id),
    priority INTEGER NOT NULL,
    created_at TEXT NOT NULL);
```

No `ON DELETE` clause on either FK (INV-7 cascade is service-owned). No backfill
(pre-v7 rows are all auto/uncategorised — `NULL/NULL` — which the first rule run
fills). `LATEST_SCHEMA_VERSION` → 7. (design.md's Persistence list names this table
`categorization_rules`; its exact columns are fixed here.)
