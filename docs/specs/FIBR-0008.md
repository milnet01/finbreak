**Theme:** P05 built the first importer (CSV) and, with it, the whole write
pipeline ‚Äî dedup, the atomic transaction+period commit, the wizard. OFX is the
**second** importer, and the design was built for exactly this: a new format is
"adding one importer; nothing else changes" (design.md). OFX (Open Financial
Exchange) is a **worldwide bank-export standard** ‚Äî self-describing, so unlike
CSV it needs **no per-bank mapping profile**: the columns, the date format, the
amount sign, and the statement's coverage period are all defined by the format.
So this phase is deliberately small: a **pure `OfxImporter`** that turns an OFX
file into the same `ParseResult` a `CsvImporter` produces, a tiny refactor so
`ImportService` accepts a `ParseResult` from either importer, and an OFX branch
in the existing wizard that **skips the column-mapping step**. **No schema
change, no migration, no new table** ‚Äî the v4 `transactions` + `statement_periods`
tables (FIBR-0007) already hold everything OFX needs.

## Context

The app is layered ‚Äî **UI ‚Üí Services ‚Üí Repositories ‚Üí SQLCipher** (design.md) ‚Äî
and its headline flow is **import ‚Üí categorise ‚Üí insight**. FIBR-0007 built the
first importer (CSV) and, with it, the entire write path. This phase adds the
**second** importer, which design.md explicitly planned for:

- **The pluggable importer interface** (design.md "Importers (pluggable)"):
  every importer "exposes the same interface: `parse(source) -> ParseResult` ‚Äî
  the transaction drafts plus any errors ‚Ä¶ a new format = adding one importer;
  nothing else changes." OFX is that new format.
- **`ImportService` selects the importer by extension / content** (design.md
  "Dominant path ‚Äî import to insight", step 3). This phase adds the OFX branch.

This phase **reuses, unchanged**, the machinery FIBR-0007 shipped:

- **`parse_transaction(occurred_on, raw_amount, description, exponent)`**
  (`services/transactions.py`) ‚Äî the single money-and-fields validator, reused
  per transaction (so OFX and CSV amounts obey the identical contract).
- **`ImportService.commit_import(preview, period_start, period_end,
  source_filename)`** ‚Äî the atomic dedup-delta insert + `statement_periods`
  record. **Not touched** by this phase; the OFX path produces the same
  `ImportPreview` and calls the same `commit_import`.
- **The multiset-delta dedup** (`ImportService._dedup`) and the
  `statement_periods` span-dedup (`exists`) ‚Äî reused verbatim.
- **`TransactionDraft`**, the **`ParseResult`** / **`RowError`** value objects,
  and the `read_minor_unit_exponent(conn)` currency read.
- **`AppShell` stacked-view routing + the import wizard** (FIBR-0007) ‚Äî the OFX
  path is a branch inside the existing `ImportWizardWidget`, not a new screen.

So this phase adds **no** new architecture and **no** persistence change ‚Äî a
pure `OfxImporter`, a small `ImportService` refactor to accept a pre-built
`ParseResult`, an OFX branch in the wizard, and one pinned dependency.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` gates them before code. **D1, D2,
D3, D4, D5, D6, D7, D9, D10, D12, D14, D15 are engineering calls; D8
(multi-account, no silent drop) and D11 (single-currency, consistent with the
app) carry a product/scope requirement; D13 (resource bounds) carries a security
requirement (security-model INV-5b).**

1. **D1 ‚Äî A pure `OfxImporter` produces the same `ParseResult` as the CSV
   importer.** Mirroring `CsvImporter` (no DB, no Qt), `OfxImporter` parses
   already-read OFX **bytes** into `ParseResult`(s) via `ofxparse` 0.21
   (`OfxParser.parse(io.BytesIO(data))`). This realises design.md's uniform
   `parse(source) -> ParseResult` importer interface ‚Äî OFX's `source` is the
   file bytes. Keeping it pure makes INV-1/2/3 unit-testable with no vault, and
   keeps the money rule in exactly one place (`parse_transaction`, reused).

2. **D2 ‚Äî `ImportService` gains a shared `ParseResult`‚Üí`ImportPreview` core;
   `commit_import` is untouched.** The CSV `preview(text, mapping, account_id)`
   currently parses **and** dedup-analyses in one method. The dedup-analysis
   tail (dedup delta + carry the `ParseResult`'s period into an `ImportPreview`)
   is **extracted** to `ImportService._preview_from_result(result, account_id)`;
   `preview(...)` becomes "read header + validate mapping + `CsvImporter().parse`
   + `_preview_from_result`", and a new `preview_ofx(result, account_id)` is
   "`_preview_from_result`". This is a refactor **to satisfy** the second
   importer's need (global rule ¬ß 3(b)/¬ß 11 ‚Äî the second call-site, extracted now
   rather than after a third; coding.md ¬ß 1.3's Rule-of-Three is *not* invoked
   here, this is the two-call-site "refactor to cover the new case" clause) ‚Äî not
   an adjacent rewrite. `commit_import` is **byte-for-byte unchanged**: both
   importers produce the same `ImportPreview` and call it.

3. **D3 ‚Äî The OFX amount is already signed; pass it straight to
   `parse_transaction`.** Unlike CSV (single signed column, or a debit/credit
   pair the importer signs), each `ofxparse` transaction's `.amount` is already
   a **signed `Decimal`** ‚Äî money-out negative, money-in positive, **taken from
   `<TRNAMT>` verbatim** (empirically: a debit line `<TRNAMT>-10.00` ‚Üí
   `Decimal('-10.00')`, a credit line `<TRNAMT>1000.00` ‚Üí `Decimal('1000.00')`;
   ofxparse does **not** invert by `<TRNTYPE>`, so a `DEBIT` with `<TRNAMT>10.00`
   stays `+10.00`), exactly finbreak's convention. So the importer passes
   `.amount` straight to
   `parse_transaction` ‚Äî no debit/credit logic, no `invert_amount`. The OFX
   `.type` token (`debit`/`credit`/`atm`/‚Ä¶) is **informational only** and does
   not drive the sign (the signed `.amount` is authoritative).

4. **D4 ‚Äî The coverage period is the embedded DTSTART‚ÜíDTEND, not the drafts'
   min/max.** OFX statements carry an authoritative
   `<BANKTRANLIST><DTSTART>‚Ä¶<DTEND>` ‚Äî `ofxparse` exposes it as
   `statement.start_date` / `statement.end_date` (datetimes when present). The
   importer sets `ParseResult.period_start/period_end` to those
   (`start_date.date().isoformat()`), so the wizard defaults the confirm-period
   fields to the **real** statement span even when the transactions don't span
   the full period (e.g. a quiet month ‚Äî see D14 for how a zero-transaction but
   period-bearing statement still records its coverage). **Missing-span
   handling:** ofxparse 0.21 sets `start_date`/`end_date` to the **empty string
   `''`** (not `None`) when `<DTSTART>`/`<DTEND>` are absent (verified, ¬ß Verified
   API basis), so the importer detects "missing" by **falsiness** on **both**
   endpoints (`if not st.start_date or not st.end_date`), **never `is None`** ‚Äî
   ofxparse sets the pair together, but the symmetric guard means a half-present
   span can't slip a `''` end through; else `''.date()` raises and the D7
   boundary catch would turn a valid statement into a total failure. A missing
   span falls back to the drafts' min/max (the CSV rule), or `None` when zero
   drafts (INV-2). This is the design-ahead FIBR-0007's `statement_periods` table
   was built for (D8 there): OFX populates it from **embedded** metadata,
   exactly as that spec anticipated.

5. **D5 ‚Äî Description is payee, falling back to memo.** An OFX transaction has a
   `NAME` (payee ‚Äî usually the merchant) and an optional `MEMO`. The draft
   `description` is `payee` when non-empty (after strip), else `memo`; if
   **both** are empty the row is a `RowError` (blank description ‚Äî via
   `parse_transaction`, INV-3). An empty `payee` arises from an **absent or
   empty** `<NAME>` tag (both yield `payee == ''` in natural per-line SGML ‚Äî
   D15; the tag content is the newline, stripped to `''`); `<MEMO>` is likewise
   optional and may be absent or empty, both yielding `memo == ""`. Combining
   payee **and** memo into one richer
   description is a deferred enhancement (¬ß Out of scope) ‚Äî a single, predictable
   field keeps the dedup key stable and matches the CSV importer's one-column
   description.

6. **D6 ‚Äî Dedup reuses the content multiset-delta; FITID is not used as the
   key.** OFX transactions carry a unique `FITID`, which *could* be a more
   robust dedup key. But the roadmap directs OFX to feed "the same
   ImportService pipeline (dedup, ‚Ä¶)", and a FITID-based key would work only for
   OFX and diverge from CSV/PDF. So OFX reuses the **same** account-scoped content
   key `(occurred_on, amount_minor, normalised description)` ‚Äî `account_id` scopes
   the existing-row lookup rather than sitting in the key tuple, matching
   `ImportService._key` (FIBR-0007 D6) ‚Äî consistent across every importer, and
   already correct (genuine repeats kept on first import, zero on re-import). FITID-based dedup (robust against
   two content-identical but genuinely-distinct OFX transactions across
   re-imports) is a named future enhancement (¬ß Out of scope).

7. **D7 ‚Äî One documented boundary catch for ofxparse's unpredictable error
   surface.** `ofxparse` raises an **inconsistent** variety on malformed input ‚Äî
   empirically an `OfxParserException` ("The ofx file is empty!") on empty/HTML,
   but a bare `ValueError` ("not enough values to unpack") on other garbage.
   Because this is the **untrusted-file boundary**, the OFX read is wrapped in a
   single `try/except Exception` that re-raises a friendly
   `ValueError("this file could not be read as OFX")` (INV-4), with a comment
   naming the constraint (global rule ¬ß 1 ‚Äî a documented boundary catch, not a
   silent swallow: the original is chained via `from exc`). This is the **only**
   broad catch; per-transaction `parse_transaction` failures are the narrow
   `RowError` path (INV-3), unchanged. A structurally-valid but **statement-less**
   OFX (empty `ofx.accounts`) is not an ofxparse exception ‚Äî the importer guards
   it explicitly and raises the same friendly `ValueError` **type** (with a
   *distinct* "no statements found" message, INV-4) **after** the boundary catch
   returns (not inside it, so its message isn't collapsed into the generic one),
   before indexing ‚Äî so the wizard never hits an uncaught `IndexError`.

8. **D8 ‚Äî A multi-account OFX surfaces all statements; the user picks one (no
   silent drop).** An OFX file may contain more than one `<STMTRS>` (bank +
   credit-card, or several accounts). `ofxparse` exposes them as
   `ofx.accounts`. Silently importing only `ofx.account` (the first) would drop
   the others' transactions. So the `OfxImporter` returns **one `ParseResult`
   per statement**, each tagged with an `OfxAccountInfo(account_id,
   account_type)`; the wizard lists them (a label `"<account_id> ¬∑ <type>"`,
   falling back to just the id when `account_type` is empty, e.g. a credit card)
   and the user chooses which statement to import into the selected finbreak
   account (one at a time). A single-statement file auto-selects its sole
   statement (no extra click). Importing several OFX
   accounts in one pass (a multi-select) is a deferred enhancement.

9. **D9 ‚Äî No schema change, no migration, no new table.** OFX needs no mapping
   profile (self-describing) and no new columns ‚Äî a transaction's account,
   amount, date, description, and the statement period all map onto the existing
   v4 `transactions` + `statement_periods` tables. So `migrations.py`,
   `LATEST_SCHEMA_VERSION`, and `_MIGRATIONS` are **untouched**, and there is
   **no** schema-version ripple in the test suites. This is the phase's biggest
   simplification over FIBR-0007.

10. **D10 ‚Äî The wizard detects OFX and skips the column-mapping step.** The
    existing `ImportWizardWidget` (a non-modal `QStackedLayout`, FIBR-0007 D9)
    gains an OFX branch: on file select, it detects OFX by **extension**
    (`.ofx`/`.qfx`) with a **content sniff** fallback (the file begins with an
    `OFXHEADER` line or an `<OFX>` tag), then parses via `OfxImporter`, offers
    the statement chooser when there is more than one account (D8), and jumps
    straight to the **preview** step ‚Äî the mapping step is bypassed exactly as it
    is for an auto-matched CSV profile. The preview / confirm-period / Import
    step is **reused unchanged** (it already operates on an `ImportPreview`). CSV
    files take the existing path.

11. **D11 ‚Äî Single base currency, consistent with the whole app.** finbreak is a
    single-base-currency app (one `minor_unit_exponent` in `settings`; manual
    entry and CSV import both assume it). OFX statements carry a `CURDEF`, but
    the importer does **not** convert or reject on a currency mismatch ‚Äî it
    scales `.amount` by the base-currency exponent exactly as CSV does, so all
    three entry paths behave identically. Multi-currency handling (detect
    `CURDEF`, convert or warn) is an app-wide concern deferred to a future phase,
    not an OFX-specific one (¬ß Out of scope).

12. **D12 ‚Äî `ParseResult` + `RowError` move to a shared importer module.** These
    two value objects currently live in `importers/csv_importer.py`; OFX needs
    the same types. They move to **`importers/base.py`** (imported by both
    `csv_importer` and `ofx_importer`, and by `services/import_`), a pure
    behaviour-preserving relocation (`csv_importer` keeps importing them so its
    public surface is unchanged). `TransactionDraft` stays in `models.py`
    (already shared). No test that imports `CsvImporter`/`read_header` from
    `csv_importer` is affected.

13. **D13 ‚Äî Untrusted-input resource bounds (security-model INV-5b).** An OFX
    file is untrusted input, and `security-model.md` INV-5b names **FIBR-0008**
    as a pin site for the import resource budget. Two enforceable caps
    (INV-10): (a) a **max file size** ‚Äî the OFX read path stat-checks the file
    and refuses one larger than `_MAX_OFX_BYTES` (**16 MiB** ‚Äî orders of
    magnitude above any real personal statement, which run KB‚Äìlow-MB) with a
    friendly `ValueError` **before** reading the bytes into memory (so an
    oversized file can't exhaust RAM, T5); (b) a **max transaction count** ‚Äî
    after parsing, a file whose **total** transaction count across all its
    statements exceeds `_MAX_OFX_TRANSACTIONS` (**100 000**) is refused
    whole-file with a `ValueError` (a bounded file, not a fabricated bomb; the
    whole-file refusal is a clear safety error, not a silent per-account drop ‚Äî
    cf. INV-5/INV-10). **Parse time** (the third dimension INV-5b lists) is
    bounded **transitively** by the 16 MiB size cap ‚Äî not by a separate
    wall-clock cap: ofxparse is synchronous on the GUI thread, so a
    16-MiB-bounded input bounds its parse work; a hard wall-clock timeout would
    need a watchdog thread (YAGNI until a real slow-parse is measured ‚Äî matching
    FIBR-0007's GUI-thread posture). So all three INV-5b dimensions are covered
    (size + count directly, parse-time transitively). The
    caps live as module constants so they are one-line-tunable. (The sibling CSV
    path (FIBR-0007) predates this pin and does **not** yet enforce a size cap;
    back-filling it is tracked separately ‚Äî ¬ß Out of scope ‚Äî not silently folded
    in here.)

14. **D14 ‚Äî A period-bearing statement with zero transactions still records its
    coverage (predicate relaxed, CSV-equivalent).** OFX's embedded span makes a
    **quiet month** real: a statement with a genuine `DTSTART`/`DTEND` but **zero**
    transactions is exactly what statement-gap detection (FIBR-0038) wants
    recorded (D4). FIBR-0007's wizard disabled **Import** on `len(drafts) == 0`
    because a CSV with no rows has no period. That predicate is **relaxed** to
    `len(preview.drafts) > 0 **or** preview.period_start is not None` ‚Äî so a
    quiet-month OFX (zero drafts, embedded span) keeps Import **enabled** and
    records its period via the reused `commit_import` (whose fresh-span /
    zero-delta leg, FIBR-0007 INV-6, already writes a period row with zero
    inserts). This is **equivalent for CSV**: a CSV `ParseResult` has
    `period_start == None` whenever `drafts` is empty (period = min/max of
    drafts), so the added disjunct is never true for CSV and the FIBR-0007
    wizard tests stay green (INV-9). No change to `commit_import` itself.

15. **D15 ‚Äî OFX parses all-or-nothing per statement; per-row `RowError`
    collection covers only *post-parse* validation.** Unlike the CSV importer
    (which reads rows itself and tolerates a bad one), `ofxparse` 0.21 parses a
    whole statement eagerly (default `fail_fast=True`) and **raises
    `OfxParserException` during `OfxParser.parse` on a structurally-malformed
    transaction** ‚Äî verified: an unparseable `<DTPOSTED>` ("time data ‚Ä¶ does not
    match format") or an empty/garbage `<TRNAMT>` ("Invalid Transaction Amount")
    each abort the **entire** statement (even a single good sibling row is lost).
    (An **empty** `<NAME>` does **not** abort in natural one-tag-per-line SGML ‚Äî
    ofxparse reads the newline as the tag's content and yields `payee == ''`, i.e.
    it behaves like an *absent* tag ‚Üí memo-fallback / blank-description
    `RowError`; only the pathological zero-whitespace `<NAME><MEMO>` layout, which
    no real export or our fixtures use, trips the "Empty transaction name" path ‚Äî
    so the **canonical structural-abort fixture is a bad `<DTPOSTED>` / empty
    `<TRNAMT>`**, not an empty `<NAME>`.) ofxparse exposes **no** per-transaction
    skip. So finbreak's error model splits in two:
    - **Structural malformation ‚Üí whole-statement failure.** Anything ofxparse
      itself rejects is caught by the D7 boundary and surfaced as a friendly
      **`ValueError`** (INV-4) ‚Äî the statement is not importable; the importer
      does not silently drop the good rows around the bad one (it can't isolate
      them), and does not silently succeed. This is the honest MVP behaviour;
      pre-scrubbing the SGML to recover a sloppy row is deferred (¬ß Out of scope).
    - **Post-parse validation ‚Üí per-row `RowError`.** A transaction ofxparse
      **successfully** parsed but that `parse_transaction` then rejects ‚Äî a
      **zero** / non-finite / **over-precise** amount, or a **blank description**
      (reachable only when **both** `<NAME>` and `<MEMO>` tags are **absent**, so
      `payee == '' and memo == ''`) ‚Äî becomes a collected `RowError`, and the
      other valid transactions in that statement **do** import (INV-3). This is
      the only per-row-isolation guarantee OFX can make.
    (The memo-fallback of D5 is reachable via an **absent** *or empty* `<NAME>`
    tag ‚Äî both yield `payee == ''` in natural layout.)

## Invariants (testable)

> These `INV-N` are **local to this spec**. Tests live in
> `tests/features/ofx_import/` (`spec.md` + `test_ofx_import.py`) unless noted,
> use `tmp_path` for every on-disk vault, and never touch the network
> (testing.md ¬ß 6). GUI-touching invariants use the `pytest-qt` `qtbot` fixture.
> OFX fixtures are tiny in-repo strings (no real statements ‚Äî never commit real
> financial data, security-model.md). The vault fixture is the FIBR-0007
> first-run vault (already at v4 ‚Äî no migration in this phase).

- **INV-1 ‚Äî OFX parsing to transaction drafts.** Given the currency exponent,
  `OfxImporter.parse(data, exponent)` returns a list of `(OfxAccountInfo,
  ParseResult)` ‚Äî one per statement. For a single-account fixture, one
  `ParseResult` whose `.drafts` carry, per OFX transaction, a
  `TransactionDraft(row_number, occurred_on, amount_minor, description)`:
  - **INV-1a (signed amount passthrough).** A transaction whose `<TRNAMT>` is
    `-10.00` ‚Üí `amount_minor == -1000` (exponent 2); a `<TRNAMT>` of `1000.00` ‚Üí
    `+100000`. The sign is the **`TRNAMT` field verbatim**, not the DEBIT/CREDIT
    `<TRNTYPE>` token ‚Äî ofxparse never inverts by type (verified), and real OFX
    exports already carry the sign in `TRNAMT` (debits negative). So the fixture
    pins `<TRNAMT>-10.00`, not `10.00`.
  - **INV-1b (date).** `DTPOSTED = 20260105` ‚Üí `occurred_on == "2026-01-05"`.
  - **INV-1c (description).** `NAME = "Coffee Shop"` ‚Üí `description ==
    "Coffee Shop"`; a transaction with the `<NAME>` tag **absent** (`payee ==
    ""`) and `MEMO = "ATM"` ‚Üí `description == "ATM"` (memo fallback, D5). A
    present-but-**empty** `<NAME>` is a different case ‚Äî it aborts the whole
    statement at parse (D15), so the fallback fixture omits the tag entirely.
  - **INV-1d (money never floats).** The **testable** leg: an OFX amount with
    more fractional digits than the currency allows is a `RowError`, **not**
    silently rounded (`parse_transaction` over-precision) ‚Äî this is the asserted
    guarantee. That the path constructs no `float` is a **design property**
    (`.amount` is a `Decimal`, routed through `parse_transaction`, which builds
    no float), upheld by review, not a separate automated test (mirroring
    FIBR-0007 INV-3d).
  Source: design.md importer interface; FIBR-0007 D5; glossary "Draft".

- **INV-2 ‚Äî Coverage period is the embedded span.** For a statement with
  `DTSTART = 20260101`, `DTEND = 20260131`, and transactions only on the 5th and
  20th, `ParseResult.period_start == "2026-01-01"` and `period_end ==
  "2026-01-31"` (the embedded span, **not** `2026-01-05`/`2026-01-20`). A
  statement whose embedded span is absent falls back to the drafts' min/max;
  zero drafts ‚Üí `None`/`None`. Source: FIBR-0007 D8 (design-ahead); FIBR-0007 INV-6.

- **INV-3 ‚Äî Post-parse per-row errors collected, valid rows still import
  (D15).** Because ofxparse hands the importer **already-parsed** transaction
  objects, per-row isolation covers only failures `parse_transaction` raises on a
  successfully-parsed transaction. A statement mixing a valid transaction, a
  **zero-amount** transaction, and a **blank-description** transaction (both
  `<NAME>` and `<MEMO>` tags **absent** ‚Üí `payee == "" and memo == ""`) yields
  drafts for the valid one(s) and `RowError`s for the two rejected ones
  (collected, not raised); `row_number` is 1-based over the statement's
  transactions in file order. **What this does NOT cover (D15):** a
  structurally-malformed transaction (an unparseable `<DTPOSTED>` or an empty
  `<TRNAMT>`) makes ofxparse abort the **whole statement** at parse ‚Äî that is
  INV-4's whole-statement `ValueError`, **not** a `RowError`, and the good
  sibling rows are **not** recoverable (ofxparse offers no per-transaction skip).
  Source: ADR-0005; FIBR-0007 INV-4; D15.

- **INV-4 ‚Äî Malformed / statement-less / bad-transaction input ‚Üí one friendly
  `ValueError`.** `OfxImporter.parse` raises a single `ValueError` (not
  `OfxParserException`, not a bare unpack `ValueError` leaking through, not an
  unhandled crash), with the original chained, on three classes of unusable
  input: (a) **empty bytes / HTML / arbitrary garbage** (the D7 boundary catch);
  (b) a **structurally-valid statement whose transaction ofxparse cannot parse**
  (a bad `<DTPOSTED>` or empty `<TRNAMT>` ‚Äî verified to
  raise `OfxParserException` mid-parse; the whole statement is unimportable,
  D15); and (c) a **statement-less** but otherwise valid envelope (empty
  `ofx.accounts` ‚Äî e.g. sign-on-only; verified: `ofx.account` then raises
  `AttributeError`), refused with a distinct "no statements found" message. The
  empty-`ofx.accounts` guard runs **after** the D7 `try/except` returns (not
  inside it), so its distinct message is not collapsed into the generic
  "could not read as OFX". Source: coding.md ¬ß 2; D7; D15.

- **INV-5 ‚Äî Multi-account OFX surfaces all statements.** A two-statement OFX
  fixture (two `<STMTRS>` with distinct `ACCTID`s) yields **two**
  `(OfxAccountInfo, ParseResult)` pairs, with the correct `account_id`/
  `account_type` on each and each statement's own transactions ‚Äî none dropped.
  Source: design.md; D8.

- **INV-6 ‚Äî OFX feeds the same write pipeline (dedup + period + atomic).**
  Headless, on a `Vault`-opened connection: `preview = ImportService.preview_ofx(
  result, account_id)` then `commit_import(preview, preview.period_start,
  preview.period_end, source)` (the period is carried **into** the
  `ImportPreview` by `_preview_from_result`, so the preview's period is used ‚Äî
  the same object the CSV path commits) inserts the new rows and one
  `statement_periods` row; re-running the same import adds **zero** transactions
  and **no** second period row (dedup + span-dedup, reused from FIBR-0007). An
  OFX transaction that duplicates a manually-entered one is deduped. (The test
  reads the exponent via `read_minor_unit_exponent(conn)` ‚Äî the ZAR fixture's
  is 2 ‚Äî and unwraps the single-statement result:
  `result = OfxImporter().parse(data, exponent)[0][1]`.) Source: roadmap;
  FIBR-0007 INV-5/6/7.

- **INV-7 ‚Äî Import-wizard OFX round-trip** (`qtbot`):
  - **INV-7a** ‚Äî a helper `_select_file(path)` on an `.ofx` file detects OFX,
    parses, and (single-account) lands on the **preview** step (`_stack`
    index 2), **skipping** the mapping step (index 1).
  - **INV-7b** ‚Äî the preview shows every transaction, the `N new / M duplicate /
    K error` summary, and the period fields defaulted to the embedded span; the
    amount column renders the **decimal** (`-10.00`), not raw minor units
    (reusing the FIBR-0007 review-fix).
  - **INV-7c** ‚Äî **Import** inserts exactly the new rows, records a period row,
    and emits `done`.
  - **INV-7d** ‚Äî importing the **same** file again previews as all-duplicate
    (`new_count == 0`) and Import adds zero (still enabled, to record a fresh
    coverage span ‚Äî the relaxed disable predicate, D14).
  - **INV-7e (multi-account)** ‚Äî a two-statement OFX file populates the
    `_ofx_statement_combo` with **both** statements (a chooser shown only when
    `> 1` account); the wizard previews statement 0 by default, and selecting
    the **second** entry (which fires the combo's `currentIndexChanged` ‚Üí
    re-runs `preview_ofx` for that statement, no separate "Next" button) shows
    the **second** statement's rows; **Import** then inserts exactly that
    statement's new rows into the chosen finbreak account. A single-statement
    file leaves `_ofx_statement_combo` hidden ‚Äî the test asserts the **explicit**
    hidden state (`_ofx_statement_combo.isHidden()`, which reflects an explicit
    `hide()` regardless of whether the parent was shown), **not** raw
    `isVisible()` (always `False` for a never-shown widget under offscreen qtbot,
    which would pass vacuously).
  - **INV-7f (quiet month)** ‚Äî an `.ofx` file with a real embedded span but
    **zero** transactions previews with zero drafts, the period fields defaulted
    to the embedded span, and **Import enabled** (the D14 predicate:
    `drafts > 0 or period_start is not None`); pressing Import records the
    `statement_periods` row with **zero** inserted transactions.
  Source: FIBR-0007 INV-10; coding.md ¬ß 5.2; D8/D10/D14.

- **INV-8 ‚Äî No secret / network / schema regression; input treated as data.**
  No network import (`socket`, `http`, `urllib`, `requests`, `ftplib`, ‚Ä¶)
  anywhere under `src/finbreak/` (the existing vault-suite static scan covers the
  new module); a parse ‚Üí preview ‚Üí import cycle logs no secret (`caplog`, the
  fixture password never appears); `LATEST_SCHEMA_VERSION` is **unchanged** and no
  new table is created (asserted: opening a first-run vault is still v4 with
  exactly the FIBR-0007 tables). **Data-inertness leg (security-model INV-5a):**
  an OFX transaction whose `<NAME>`/`<MEMO>` looks like a spreadsheet formula or
  a filesystem path (e.g. `=cmd|'/c calc'`, `../../etc/passwd`) round-trips as
  **literal** stored text ‚Äî the importer never evaluates it or derives a path
  from it (the importer only opens the user-picked file path). Source:
  security-model INV-5a/INV-8/INV-9; FIBR-0007.

- **INV-9 ‚Äî CSV suite stays green; reuse verified.** After the D2 refactor and
  the D12 relocation, the full FIBR-0007 import suite
  (`tests/features/import_/`) passes unchanged, proving `commit_import`, the
  dedup, and the `ParseResult`/`RowError` contracts were reused, not forked.
  Source: coding.md ¬ß 1.3.

- **INV-10 ‚Äî OFX is resource-bounded (D13, security-model INV-5b).** Headless:
  reading an OFX file whose on-disk size exceeds `_MAX_OFX_BYTES` (in
  `services/import_.py`) raises a friendly `ValueError` **without** loading it
  (the check stat-s the path before `read_bytes`); a file whose **total**
  transaction count across all its statements exceeds `_MAX_OFX_TRANSACTIONS`
  (in `importers/ofx_importer.py`) is refused **whole-file** with a `ValueError`
  (a safety refusal with a clear message ‚Äî distinct from INV-5's no-silent-drop,
  which governs normal imports, not an over-budget file). Tests monkeypatch each
  constant **down** in its own module (import_ / ofx_importer respectively) so a
  fixture need not be 16 MiB / 100 000 rows. Source: security-model INV-5b (T5);
  coding.md "Error handling".

## Verified API basis

Pins the `ofxparse` 0.21 idioms the INVs assert against (verified empirically
against the installed package, global rule ¬ß 13; re-verify at implementation).

- **`OfxParser.parse(fileobj)`** ‚Äî accepts an `io.BytesIO(data)` of the raw file
  bytes (verified). Returns an `ofx` object with `ofx.account` (the first
  statement's account) and `ofx.accounts` (list ‚Äî one per `<STMTRS>`/`<CCSTMTRS>`).
  Its **`fail_fast` parameter defaults to `True`** (verified) ‚Äî the whole
  D15 all-or-nothing-per-statement model **rests on this default**; the importer
  relies on the default (does not pass `fail_fast=False`, which would silently
  swallow bad statements).
- **`account.account_id`** (str), **`account.account_type`** (e.g. `"CHECKING"`),
  **`account.routing_number`** (str) ‚Äî the statement's account identity
  (verified).
- **`account.statement`** ‚Üí **`.start_date`** / **`.end_date`** (`datetime`, the
  embedded DTSTART/DTEND), **`.currency`** (str, lower-cased e.g. `"zar"`),
  **`.transactions`** (list) (verified).
- **`transaction.amount`** ‚Äî a **signed `Decimal`** (debit `Decimal('-10.00')`,
  credit `Decimal('1000.00')`) (verified), taken from `<TRNAMT>` verbatim (no
  `<TRNTYPE>` inversion). Edge: a `null` `<TRNAMT>` maps to the **`int` `0`**,
  which `parse_transaction` (accepting `str | Decimal`) rejects as zero-amount ‚Üí
  a post-parse `RowError` (INV-3), harmless. **`.date`** (`datetime`, DTPOSTED);
  **`.payee`** (NAME), **`.memo`** (MEMO), **`.id`** (FITID), **`.type`**
  (`"debit"`/`"credit"`/‚Ä¶). `transaction.date.date().isoformat()` (the DTPOSTED
  date ‚Äî **not** `.amount.date()`, since `.amount` is a `Decimal`) and `.amount`
  feed `parse_transaction`.
- **Missing embedded span** ‚Äî when a statement has no `<DTSTART>`/`<DTEND>`,
  ofxparse 0.21 sets `statement.start_date` / `.end_date` to the **empty string
  `''`** (not `None`, not a `datetime`) ‚Äî verified. The importer detects a
  missing span by **falsiness** (`if not st.start_date`), never `is None`, or
  `''.date()` raises (D4).
- **Credit-card statements** (`<CCSTMTRS>`) parse into `ofx.accounts` with their
  transactions and embedded span intact, but with an **empty `account_type ==
  ''`** (verified; a bank account yields e.g. `'CHECKING'`). The wizard's
  statement chooser falls back to a label when `account_type` is empty (D8).
- **`account.routing_number`** exists (str) but is **not consumed** this phase
  (the chooser shows account id + type only) ‚Äî listed for completeness.
- **Malformed input** ‚Äî `OfxParser.parse` raises `OfxParserException` ("The ofx
  file is empty!") on empty/HTML input but a bare `ValueError` ("not enough
  values to unpack") on other garbage (verified) ‚Äî hence the single documented
  boundary catch (D7).
- **`parse_transaction`** (`services/transactions.py`, verified in FIBR-0007) ‚Äî
  `(occurred_on ISO, raw_amount: str|Decimal, description, exponent) -> (str,
  int, str)`; the OFX importer passes the signed `Decimal` amount and the
  ISO date.
