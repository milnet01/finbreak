**Theme:** this is the first way real money gets *into* the app from a
file — the import path P06 (OFX) and P07 (PDF) then reuse. A statement
CSV has **no universal shape** (ADR-0005): column order, names, date
format, and how the amount is written (one signed column, or separate
debit/credit columns) all vary by bank. So the user maps a given bank's
columns **once**, in a wizard with a live preview that shows parsed rows
and per-row errors *before anything is written*; the mapping is saved as
a **profile** and matched automatically on the next import of that same
layout. Re-importing an overlapping statement adds **zero** duplicate
transactions (success criterion 2). And every import records the
statement's **coverage period** (start/end date) per account as
first-class data — the reliable input a later "you're missing a
statement" feature (FIBR-0038) needs, captured now because it is cheap
at import time and expensive to retrofit.

|-----|-----------|---------|

| INV-2 | The profile signature is the **exact** header fingerprint — the tuple of `csv.DictReader` fieldnames joined in file order; case, spacing, and order are all significant (any header change → an unmatched layout). ## Context

The app is layered — **UI → Services → Repositories → SQLCipher**
(design.md) — and its headline data flow is **import → categorise →
insight**. Everything before this phase built the *store*: the encrypted
vault + one manual transaction (FIBR-0004), multiple accounts + a
forward-only migration runner (FIBR-0005), and the category tree
(FIBR-0006). FIBR-0007 builds the first **importer**, so the store
starts filling from real statements instead of one-at-a-time typing.

This phase reuses, unchanged, the machinery those phases shipped:

- **`parse_transaction(occurred_on, raw_amount, description, exponent)`**
  (`services/transactions.py:22`) — the single money-and-fields
  validator (raises `ValueError` on a blank description, non-ISO date,
  non-numeric/non-finite/zero amount, or an amount with more fractional
  digits than the currency allows). Money is an exact integer number of
  minor units, never a binary float (FIBR-0004 D1). The CSV importer
  **reuses** this per row rather than re-deriving the money rules
  (coding.md § 1.3 reuse-before-rewrite) — so CSV-imported and
  manually-typed amounts obey the identical contract.
- **The forward-only migration runner** (`migrations.py`):
  `LATEST_SCHEMA_VERSION` (currently **3**), the ordered `_MIGRATIONS`
  registry, and `run_migrations(conn)` applying each pending step inside
  a **runner-owned** `try: BEGIN … COMMIT except: ROLLBACK; raise` (the
  explicit `BEGIN` is the step's first statement, because with the
  vault's `isolation_level=""` the driver does not implicitly `BEGIN`
  around DDL). It runs on both `Vault.create()` and `Vault.open()`, so a
  brand-new and an upgraded vault converge on the same latest shape.
- **`PRAGMA foreign_keys = ON`** on every connection (FIBR-0005 D4) —
  which makes `statement_periods.account_id REFERENCES accounts(id)`
  real.
- **`SchemaVersionError`** for a vault newer than this build — reused
  unchanged (a v4 build refuses a v5+ vault; already covered
  symbolically by the accounts suite's `LATEST_SCHEMA_VERSION + 1`
  refusal legs).
- **The service pattern** — a service takes a `Vault` and builds a fresh
  repository per operation from `self._vault.connection`
  (`AccountService`/`CategoryService`/`TransactionService`). `ImportService`
  mirrors it.
- **`AppShell` stacked-view routing + `qtbot`** (FIBR-0004) — the import
  screen is one more stacked `QWidget`, reached from `MainWindow`
  exactly like the Accounts and Categories managers.

So this phase adds **no** new architecture — a pure `CsvImporter`, an
`ImportService` orchestrator, two small repositories, two new tables via
one more forward migration step, and one wizard screen — all following
patterns already shipped three times.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` gates them before code.
**D1, D2, D3, D4, D5, D7, D9, D10, D12 are engineering calls made in this
spec; D6, D8, D11 carry a product/scope requirement from the roadmap +
ADR-0005 (D6 the SC2 zero-duplicate goal, D8 the coverage-period
design-ahead, D11 the MVP input bounds), though each still makes an
engineering choice about how to meet it.**

1. **D1 — One new forward migration step (v3→v4) adds two empty tables;
   first-run reuses the create-then-migrate path unchanged.** Adding
   `_migrate_to_v4` to `_MIGRATIONS` and bumping
   `LATEST_SCHEMA_VERSION` **3 → 4** means a fresh `Vault.create()`
   vault now lands at **v4** with the two import tables present-but-empty
   — with **no change to `create()`** (it already calls
   `run_migrations` after the v1 baseline). A brand-new vault and an
   upgraded (v3) vault converge on the identical v4 shape, so the schema
   is defined in **one place** (the migration), and the migration path
   is exercised on every first run (D1 rationale from FIBR-0005/0006,
   applied again). Unlike v2→v3, the step has **no seed** (no starter
   profiles — D11) — it is two `CREATE TABLE`s only — but is still
   wrapped in the runner's own `BEGIN … COMMIT`/`ROLLBACK` so a failure
   between the two `CREATE`s cannot leave a half-built v3.5 (INV-8).

2. **D2 — Layered, reuse-first decomposition.** A **pure** `CsvImporter`
   (no DB, no Qt) turns `(file text, ColumnMapping, exponent)` into a
   `ParseResult` (`TransactionDraft` rows + per-row `RowError`s + the
   min/max date span — matching design.md's `parse(source) -> ParseResult`
   importer interface); an `ImportService(vault)` orchestrator owns the DB
   side (match/save profiles, dedup, the atomic write, the period record);
   two thin repositories (`ImportProfileRepository`,
   `StatementPeriodRepository`) own their SQL; the existing
   `TransactionRepository` is **extended** (not duplicated) with a
   bucket-read (feeding the dedup delta) and a commit-free batch insert.
   Keeping parsing pure makes INV-3/4/5 unit-testable with no vault at
   all, and keeps the money rule in exactly one place (`parse_transaction`,
   reused).

3. **D3 — A mapping profile is discrete typed columns, not a JSON blob;
   its signature is `UNIQUE`.** Mirroring `accounts`/`categories` (no
   opaque blobs in this codebase), a profile stores its mapping as named
   columns: `date_column`, `description_column`, and **either**
   `amount_column` **or** (`debit_column` + `credit_column`), plus a
   `date_format` string and an `invert_amount` flag (§ Data model). The
   "exactly one amount style" rule is validated at the **service** layer
   (raising `ValueError`, the form-boundary convention), not by a DB
   `CHECK` — consistent with FIBR-0006 D7's service-layer validation.
   The `signature` column, however, **is** `UNIQUE` at the DB, because it
   is the *match key*: two profiles with the same signature would make
   auto-matching ambiguous, so ambiguity must be impossible at the store
   (this is identity, not a friendly-name uniqueness — hence the
   deliberate divergence from FIBR-0006's no-DB-UNIQUE stance).

4. **D4 — The signature is the exact header row; save is
   upsert-by-signature.** The signature is the tuple of the file's header
   fieldnames (as `csv.DictReader` reads them), joined in **file order** by
   a delimiter (`\x1f`, a control char that cannot occur in a header) — an
   **exact** fingerprint (`signature_for(header)`, reused by
   `match_profile`/`save_profile`/tests). No case-folding or whitespace
   normalisation: a bank's CSV export header is deterministic, so exact
   matching is the simplest stable "header row fingerprint" ADR-0005 names,
   and it keeps column resolution trivial — because a profile is applied
   **only** on an exact match, the file's header equals the profile's, so
   each mapped column resolves by a direct `row[col]` (`csv.DictReader`),
   with no normalised index. Any header change (case, spacing, order, or a
   renamed/added/removed column) yields a different signature → the layout
   is treated as unmapped and the wizard shows the mapping step (INV-9),
   never a mis-parse. (Forgiving case/space variants is deferred with the
   other dialect-robustness items, D11; the **dedup** normaliser (D6) is a
   separate, description-only transform.) Saving a profile whose signature
   already exists **updates** the existing row in place — overwriting both
   its `name` and its mapping (the latest save wins) — so re-mapping a bank
   refines its one recipe rather than accumulating duplicates (INV-1). Two
   columns with the **same** name make `signature_for(header)` raise
   `ValueError` (a name-based mapping is ambiguous — `csv.DictReader` would
   silently collapse them); since `match_profile` and `save_profile` both
   compute the signature, they **propagate** that `ValueError` (they do
   **not** return `None`), and the wizard's step-1 catches it — see D11.

5. **D5 — Money reuses `parse_transaction` + the settings exponent;
   debit→negative / credit→positive.** Per row the importer resolves the
   mapped cells (each `.strip()`ed — the date cell too, so a padded date
   parses like a padded amount), forms a **signed `Decimal`**, and hands
   the date + amount + description to `parse_transaction(iso_date, amount,
   description, exponent)` — which enforces every money rule (finite,
   non-zero, fractional-digit limit, integer minor units, no float) and
   returns `(occurred_on, amount_minor, description)` or raises
   `ValueError`. **Ragged-row guard:** a short row (fewer cells than the
   header — `csv.DictReader` pads the missing mapped cells with `None`) is
   turned into a `RowError` up-front, before any conversion, so
   `Decimal(None)` / `None.strip()` never fire (those raise `TypeError` /
   `AttributeError`, outside the catch). A **long** row (more cells than
   the header) is not an error — `csv.DictReader` shunts the extras under
   its `restkey`, which the importer ignores, and the mapped columns still
   resolve by name (a genuinely misaligned row from an unquoted delimiter
   is a data problem out of scope, D11). **The rest of the per-row body
   runs inside `try: … except (ValueError, decimal.InvalidOperation): → a
   RowError`** — because the importer's own `Decimal(cell)` conversion
   raises `decimal.InvalidOperation` (an `ArithmeticError`, **not** a
   `ValueError`) on a non-numeric cell, so catching only `ValueError` would
   let a bad amount crash the whole parse instead of becoming one
   `RowError` (INV-4). **Amount style** is chosen by which columns the
   `ColumnMapping` populates — `amount_column` set ⇒ single-column;
   `debit_column` + `credit_column` set ⇒ the debit/credit pair — and the
   **service** validates "exactly one style, every mapped column present in
   the header" (raising `ValueError`) at the start of **both** `preview`
   and `save_profile`, so the pure importer parses a mapping already known
   good. Sign convention: for a **single** amount column the cell's own
   sign is used, negated iff `invert_amount` (some banks print money-out as
   a positive "debit" figure); for the **debit/credit** pair, a value in
   the debit column is money **out** (negative) and a value in the credit
   column is money **in** (positive), with **exactly one** of the pair
   populated per row (both empty or both populated → `RowError`, INV-4).
   "Populated" means non-empty after `str.strip()`; the debit/credit cells
   are **unsigned magnitudes** — the importer forms the signed value as
   `-Decimal(cell)` for a debit, `Decimal(cell)` for a credit, and
   `-Decimal(cell) if invert_amount else Decimal(cell)` for a single amount
   column. A debit/credit cell that parses to a **negative** `Decimal`
   (`Decimal(cell) < 0`) is a **`RowError`**, **not** a flipped sign —
   money-out booked as money-in would be a silent mis-mapping of a
   financial figure, which ADR-0005 forbids; a signed debit/credit cell is
   a data/mapping problem to surface, not to reinterpret. `invert_amount`
   applies only to the single-amount style (a debit/credit profile stores
   `invert_amount = 0`, and the importer ignores it for that style). An
   amount cell carrying a thousands separator or currency symbol
   (`"1,234.56"`, `"R10.00"`) is **not** pre-cleaned — it fails
   `parse_transaction` and becomes a `RowError` this phase (stripping is a
   follow-up, § Out of scope). The currency exponent
   is read from `settings` (`minor_unit_exponent`) — the **same** value
   `TransactionService` uses. To avoid a second copy of the settings-key
   string (a typo there would silently read the wrong scale), the read is
   **extracted** to a module-level `read_minor_unit_exponent(conn)` in
   `services/transactions.py` and reused by `TransactionService._exponent`
   and `ImportService` (Rule-of-Three: this is the **second** call-site,
   and the value is money-critical — the extraction is justified now
   rather than after a third; coding.md § 1.3). This is a refactor **to
   satisfy** the import's need (global rule § 3(b)/§ 11), not an adjacent
   rewrite.

6. **D6 — Dedup is multiset-delta on the content key (description
   normalised); no new column.** The dedup key is `(account_id,
   occurred_on, amount_minor, _normalise(description))`, where
   `_normalise(s)` (a **description-only** transform, distinct from the
   exact `signature_for` of D4) folds whitespace and case:
   `" ".join(s.split())` → `str.casefold()` — so it matches design.md's
   data-flow step 5 and the glossary "Dedup" entry ("normalised
   description"). The **raw** description is stored; only the dedup
   *comparison* normalises (a re-export that re-cases or re-spaces a
   description still dedups). For the batch, incoming rows are grouped by
   that key; for each group the service reads the existing rows in the
   same `(account_id, occurred_on, amount_minor)` bucket
   (`TransactionRepository.existing_for(...)`), `_normalise`s **their**
   descriptions in Python (SQLite cannot `casefold`), counts those equal
   to the group's normalised description, and inserts `max(0, incoming −
   existing)` copies — the **first** `delta` incoming rows in file order
   (their raw descriptions preserved). This is the correct dedup for a
   **finance** app: a
   naive `UNIQUE` constraint would silently *drop* two legitimately-
   identical same-day transactions (e.g. two identical coffees), which is
   unacceptable — the multiset delta keeps genuine repeats on first import
   yet adds **zero** on re-import of an overlapping statement (success
   criterion 2, INV-5). Because the count is against *all* existing rows,
   an imported row that duplicates a **manually-entered** one is also
   deduped (no double-count). **No** hash/index column is added this
   phase; the indexed normalised-description hash (a lookup optimisation
   for large accounts) is **FIBR-0026's** scope ("Index the import
   de-duplication lookup") — personal-statement volumes (thousands of
   rows) don't need it for MVP.

7. **D7 — The write phase is one atomic transaction owned by the
   service.** `ImportService` wraps the delta inserts **and** the
   `statement_periods` row in a single `try: BEGIN … COMMIT except:
   ROLLBACK; raise` — the migration-runner's ownership pattern (Context)
   applied to a service. Either the whole statement (its new transactions
   and its coverage record) commits, or nothing does (INV-7). The dedup
   **existing counts** (D6) and the span `exists(...)` check (D8/INV-6)
   are both read **inside** this transaction (after `BEGIN`, before the
   inserts), so nothing is counted then races an insert (single-connection,
   single-threaded). If the span already exists, its period insert is
   **skipped** while the deduped transaction rows still commit — the two
   are one unit, so a skipped period never orphans committed rows. Within
   the boundary the repositories expose **commit-free** helpers
   (`TransactionRepository.add_batch(rows)` and
   `StatementPeriodRepository.add(...)` execute but do **not** commit — the
   service owns the commit); the existing per-row-committing
   `TransactionRepository.add()` used by the **manual** add-transaction
   form is **unchanged**. This is also a performance win — one commit for
   a whole statement instead of one per row (coding.md perf discipline).

8. **D8 — Statement coverage period is first-class, captured at import.**
   A new `statement_periods` table records, per import,
   `(account_id, period_start, period_end, source_filename,
   imported_at)`. CSV carries **no** embedded period, so the wizard's
   preview defaults `period_start`/`period_end` to the min/max of the
   parsed rows' `occurred_on` and lets the user confirm/adjust before the
   write (INV-6). This is the design-ahead the roadmap calls for: it is
   the reliable input for statement-gap detection (FIBR-0038), cheap to
   capture now and expensive to retrofit (would need a re-import to learn
   periods). OFX (FIBR-0008) and PDF (FIBR-0009) populate the **same**
   table from their embedded period metadata, so the data-model hook is
   established **here**, at the first importer. To keep the coverage data
   clean, an identical `(account_id, period_start, period_end)` span is
   not duplicated on re-import — a service-layer `exists(...)` check (the
   skip-but-commit mechanics are D7/INV-7); no DB `UNIQUE`.

9. **D9 — The wizard is a stacked `QWidget`, not a modal dialog.**
   `AppShell`'s `_show_accounts`/`_show_categories` comments (app.py) note
   managers are stacked views — **not** modal — precisely so the idle
   auto-lock can swap them away.
   A modal `QWizard`/`QDialog` would block that. So the import screen is
   a normal stacked `QWidget` (`ImportWizardWidget`) reached from
   `MainWindow` (a new `import_transactions` signal → `AppShell.
   _show_import`), using an **internal** `QStackedLayout` for its steps:
   (1) pick file (`QFileDialog`) + choose the target account; (2) map
   columns — **shown only when no saved profile matches** the file's
   header, and offering to save the new mapping as a named profile;
   (3) preview — a table of parsed rows with per-row errors highlighted,
   the `N new / M duplicate / K error` summary, and the confirm-period
   fields, then an **Import** button. Auto-lock routing is unchanged and
   works uniformly because the screen is a normal stacked view.

10. **D10 — Row errors are collected values, not exceptions.** A per-row
    parse failure is a **collected value** (`RowError(row_number,
    reason)`), not an exception — so one bad row never aborts a whole
    import and every failure is surfaced (INV-4). Structural / config
    failures raise plain `ValueError` at the form boundary (like
    `parse_transaction`): a duplicate-named header (D4), or saving a
    mapping with neither amount style / a mapped column absent from the
    header. This phase adds **no** new `FinbreakError` subclass — the
    exact-signature design (D4) makes a "profile applied to a header it
    doesn't fit" state unreachable, so no `ProfileMismatchError` (or any
    `ImportError`-named exception, which would shadow the Python builtin)
    is needed (coding.md § 2 — no error path for an impossible scenario,
    INV-9).

11. **D11 — MVP input scope is bounded and the limits are documented.**
    For this first importer: **comma**-delimited, **UTF-8** (BOM
    tolerated), **header-row-present** CSVs with **unique** column names.
    Semicolon/tab delimiters, headerless files, and duplicate-named
    columns are **deferred** with a clear message (§ Out of scope) rather
    than mis-parsed — ADR-0005's "any bank, any country" is served by the
    profile *mechanism*; broadening the dialect — and pre-cleaning amount
    cells (thousands separators, currency symbols) — is a follow-up, not a
    silent guess. Parsing runs on the GUI thread (personal statements
    parse in well under a frame-budget's worth of perceptible delay; a
    background worker is YAGNI until a real large-file jank is measured —
    no generic worker exists, only the Argon2 `DeriveWorker`). This
    **refines** design.md § Concurrency, updated to match: worker-thread
    parsing is the deferred enhancement, not the v1 model. **Starter
    profiles** (ADR-0005's "ships with a couple") are **deferred** until
    real Standard Bank / Absa sample exports are on hand: fabricating a
    bank's exact header format would violate the no-guessing rule (global
    rule § 13), and ADR-0005 itself calls starter profiles "data, not
    code paths" — so they can be seeded later with **no** code change.

12. **D12 — Test scaffolding reuses conftest; the v3 fixture and the
    schema-version ripple are pinned.** The v3→v4 migration test needs a
    **v3** vault, built by the established chain `build_v1_vault`
    (already in `tests/conftest.py`) → `_migrate_to_v2` → `_migrate_to_v3`
    (both importable from `finbreak.migrations`). The categories suite's
    `_build_v2_vault` already does the first two steps privately (used at
    three call-sites there); rather than a fragile cross-module import of
    that private helper into the import suite, the v2/v3
    builders are **lifted to `tests/conftest.py`** (`build_v2_vault`,
    `build_v3_vault`) and the categories suite imports them from there —
    well past the Rule-of-Three threshold (coding.md § 1.3), and the
    natural home since
    `build_v1_vault` + `keyed_connection` already live in conftest.
    The `LATEST_SCHEMA_VERSION` **3 → 4** bump ripples the "lands-at-
    latest" version assertions (§ Deliverables → Tests enumerates the
    seven); these stay **literal** (`== 4`, not `== LATEST_SCHEMA_VERSION`)
    on purpose — a hard-coded literal is an **independent oracle** (a bug
    that mis-set `LATEST` would still be caught), which is why FIBR-0006
    used literals for "lands-at" and symbolic only for the `+ 1` refusal
    legs; that split is preserved, not refactored away.

## Invariants (testable)

> These `INV-N` are **local to this spec**. Tests live in
> `tests/features/import_/` (`spec.md` + `test_import.py`; the trailing
> underscore avoids shadowing the `import` keyword) unless noted, use
> `tmp_path` for every on-disk vault, and never touch the network
> (testing.md § 6). GUI-touching invariants use the `pytest-qt` `qtbot`
> fixture (enabled since FIBR-0004). CSV fixtures are tiny in-repo strings
> written to `tmp_path` (no real statements — never commit real financial
> data, security-model.md).

- **INV-1 — Mapping-profile CRUD + signature round-trip + upsert.**
  `ImportProfileRepository.add(profile fields)` inserts one row and
  returns its new integer id; `.get(id)` / `.list_all()` read it back
  field-for-field (with a well-formed ISO-8601 `created_at`);
  `.get_by_signature(signature)` returns the matching `ImportProfile` or
  `None`. `ImportService.save_profile(name, header, mapping)` computes
  the signature (D4) and **upserts**: a first save inserts; a second save
  under the **same** signature updates the existing row in place (one row
  total, not two — the upsert overwrites the name **and** mapping, D4).
  `ImportService.match_profile(header)` returns the saved profile for a
  header with the same **exact** signature, `None` for an unknown (but
  valid) header, and **raises** `ValueError` for a header with duplicate
  column names (it computes the signature, which rejects the ambiguity —
  D4; the wizard's step-1 catches it). Source: ADR-0005; design.md
  "Repository layer".

- **INV-2 — Signature is the exact header fingerprint.** The signature of
  `["Date", "Amount", "Details"]` equals that of `["Date", "Amount",
  "Details"]` (identical) but **differs** from `["date", "amount",
  "details"]` (case), `[" Date ", "Amount", "Details"]` (spacing), and
  `["Amount", "Date", "Details"]` (reorder) — case, spacing, and order are
  all significant, so any header change is treated as an unmatched layout
  (INV-9). A header containing two equal names is refused by
  `signature_for` with `ValueError` (a name-based mapping is ambiguous —
  `csv.DictReader` would silently collapse the columns), which both
  `match_profile` and `save_profile` propagate (D4). Source: ADR-0005
  (header fingerprint).

- **INV-3 — CSV parsing to transaction drafts.** Given a `ColumnMapping`
  and the currency exponent, `CsvImporter.parse(...)` yields, per valid
  data row,
  a `TransactionDraft(row_number, occurred_on, amount_minor, description)`
  (the glossary's "Draft" — a normalised, not-yet-saved row; `row_number`
  is its 1-based source data row) where
  `occurred_on` is the mapped date cell parsed with the profile's
  `date_format` and re-emitted ISO-8601, `description` is the **stripped**
  mapped cell (`parse_transaction` trims it), and `amount_minor` is the
  signed integer minor units from
  `parse_transaction` (the sign carries the debit/credit or single-column
  direction, D5). Sub-cases:
  - **INV-3a (signed column).** A `+123.45` cell → `+12345` (exponent 2);
    a `-10.00` cell → `-1000`; with `invert_amount = 1`, `+123.45` →
    `-12345`.
  - **INV-3b (debit/credit).** A row with `debit = 10.00`, `credit`
    empty → `-1000`; `credit = 200.00`, `debit` empty → `+20000`.
  - **INV-3c (date format).** `"31/01/2026"` with `date_format
    = "%d/%m/%Y"` → `occurred_on == "2026-01-31"`.
  - **INV-3d (money never floats).** The **testable** leg: an amount with
    more fractional digits than the currency exponent allows is a
    `RowError`, **not** silently rounded (parse_transaction's
    over-precision rule) — this is the asserted guarantee. That the import
    path uses only `Decimal`/`int` and never `float` is a **design
    property** (the money path routes every amount through
    `parse_transaction`, which constructs no `float`), upheld by review,
    not a separate automated test.
  Source: ADR-0005; FIBR-0004 D1; glossary "Draft".

- **INV-4 — Per-row errors collected, never silent; valid rows still
  import.** A data row is a `RowError(row_number, reason)` (not raised,
  not written) when: the date cell fails `date_format` parsing
  (`ValueError`); the amount cell is non-numeric (the importer's
  `Decimal(cell)` → `decimal.InvalidOperation`) or non-finite / zero /
  over-precise (`parse_transaction` → `ValueError`); the description is
  blank; the row is **short** (fewer cells than the header — a `None`
  mapped cell, guarded up-front, D5); or (debit/credit profile) **both**
  debit and credit are empty, **both** populated, or the populated one
  parses to a **negative** `Decimal` (a signed value in an unsigned
  magnitude column — D5). The per-row body catches **both** `ValueError`
  **and** `decimal.InvalidOperation` (D5), and the short-row/`None`-cell
  case is caught by the up-front guard, so any of these becomes a
  `RowError` rather than aborting the parse.
  `CsvImporter.parse` returns the valid drafts **and** the `RowError`
  list; `preview`/the wizard show the count and the offending rows; a file
  with a mix imports the valid rows and reports the errors (nothing
  silently dropped). `row_number` is **1-based over the data rows
  `csv.DictReader` yields** — the first data row after the header is
  `row_number` 1 (the header is not counted; blank lines `DictReader`
  skips do not consume a number). A file with a header but **zero** data
  rows, or where **every** row errors, yields zero drafts: **Import** is
  disabled and no `statement_periods` row is written (the INV-6 zero-rows
  leg). The button's **sole** disable predicate is `len(drafts) == 0` —
  **not** `new_count == 0`: an all-duplicate file (drafts > 0 but
  `new_count == 0`) keeps Import **enabled**, so the user can still record
  its coverage period (INV-6 fresh-span leg; INV-10e). A truly **empty**
  file (no header line — `DictReader.fieldnames` is `None`) has no columns
  to map, so `read_header` raises `ValueError` at file-pick (D9 — the
  file-pick step; the raise has no dedicated INV leg). Source: ADR-0005;
  coding.md "Error handling".

- **INV-5 — Dedup is multiset-delta (description normalised).** For the
  batch grouped by `(account_id, occurred_on, amount_minor,
  _normalise(description))` (the D6 transform, **not** D4's exact
  signature; the **raw** description is stored), the service inserts
  `max(0, incoming_count − existing_count)` rows per key. Legs (headless,
  on a `Vault`-opened connection):
  - **Zero on re-import.** Importing a statement, then importing the
    **same** statement again, adds the rows the first time and **zero**
    the second (per-tuple `incoming == existing` → delta 0). Total row
    count after the second import equals after the first.
  - **Overlap.** Importing `{A,B,C}` then `{B,C,D}` yields `{A,B,C,D}`
    (only `D` added on the second).
  - **Genuine repeats kept.** A single statement with **two** identical
    rows (same tuple) imports **both** the first time (existing 0,
    incoming 2 → insert 2); re-importing it adds 0.
  - **Counts vs manual rows.** A row equal to a **manually-entered**
    transaction (same key, same account) is deduped (delta 0 against
    the manual row).
  - **Normalisation.** Two rows whose descriptions differ only by
    case/whitespace (e.g. `"POS  Coffee"` vs `"pos coffee"`) share one
    dedup key — re-importing a bank's re-cased/re-spaced re-export of the
    same row adds zero.
  Source: roadmap FIBR-0007 (success criterion 2); design.md / glossary
  "Dedup".

- **INV-6 — Coverage period recorded + span-deduped.** After a
  successful import, `StatementPeriodRepository.list_for_account(id)`
  returns a row whose `period_start`/`period_end` equal the values the
  wizard confirmed (defaulting to the min/max of the parsed drafts'
  `occurred_on`), with the `source_filename` and an ISO-8601
  `imported_at`. Legs (headless):
  - **Recorded.** A first import of a span writes exactly one period row.
  - **Span-deduped.** Re-importing the **same** `(account_id,
    period_start, period_end)` span adds no second period row (a
    service-layer `exists(...)` check run **inside** the import
    transaction, D7); a **different** span (or a different account) does
    add one.
  - **Skip-but-import.** If the span already exists but the batch has
    genuinely new (delta) transactions, those transactions still commit
    while the period insert is skipped — one atomic unit (D7/INV-7).
  - **Zero rows.** If zero drafts parse, `ParseResult.period_start` /
    `period_end` are `None`, no period row is written, and Import is
    unavailable (INV-4).
  - **Ordering + well-formedness.** `commit_import` rejects an inverted
    span (`period_start > period_end`) **and** a malformed endpoint (one
    that fails `date.fromisoformat`) with `ValueError` before writing — so
    a hand-edited period is held to the same ISO-date rigor as a
    transaction's date.
  - **Fresh span, zero delta.** An all-duplicate import (`new_count == 0`)
    whose confirmed span does **not** yet exist still records the period
    row (`period_recorded = True`, `inserted_count = 0`) — the user did
    cover that span with a statement, which is exactly what gap-detection
    (FIBR-0038) wants to know, even when it added no new transactions.
  Source: roadmap design-ahead; FIBR-0038.

- **INV-7 — Atomic write (all-or-nothing across both tables).** The
  service's write phase runs inside its own `try: BEGIN … COMMIT except:
  ROLLBACK; raise`. **(FIBR-0052 addendum — insert order reversed.)**
  FIBR-0052 added a `transactions.statement_period_id` FK, so
  `commit_import` now inserts the `statement_periods` row **first** and
  stamps every transaction with the resulting period id — the reverse of
  this spec's original transactions-first order. A forced failure midway —
  the test wraps the connection to raise on the **first** `INSERT INTO
  transactions` batch (letting `BEGIN` and the earlier `INSERT INTO
  statement_periods` row through) — leaves the
  vault, **on the same connection before any reopen**, with **neither**
  the batch's transactions **nor** a period row (the `ROLLBACK` undoes
  the inserts inside the one transaction), and re-openable so a retry
  imports cleanly. (Choosing the transactions batch as the wedge proves
  the already-inserted period row is inside the rolled-back unit.) Because
  `commit_import` reads `self._vault.connection` (no connection argument),
  the wedge injects the raising proxy via a **stand-in vault** whose
  `.connection` returns it — mirroring the categories suite's
  `_FailAtFirstCategoryInsert` proxy. The wedge test
  imports a **fresh** span, so a period `INSERT` exists to intercept; the
  span-already-exists path — where the period insert is skipped but the
  transactions still commit — is a separate INV-6 leg. Mirrors the
  FIBR-0005/0006 same-connection rollback framing. Source: design.md
  "Persistence"; FIBR-0005.

- **INV-8 — v3→v4 migration: forward-only, atomic, idempotent, existing
  tables untouched.** Opening a **v3** vault runs `run_migrations`,
  which creates `import_profiles` and `statement_periods` (§ Data model),
  sets `schema_version = 4`, and leaves `transactions`, `accounts`,
  `categories`, and `settings` unchanged (asserted by pre/post row-set +
  column-shape comparison of each). Four legs:
  - **Atomic.** The whole step runs inside the runner's `try: BEGIN …
    COMMIT except: ROLLBACK; raise`. A forced failure on the **second**
    `CREATE TABLE` (`statement_periods`) — the test wraps `conn.execute`
    to raise on the statement whose text contains `CREATE TABLE
    statement_periods`, forwarding `BEGIN`, the first `CREATE`, and
    `ROLLBACK` — leaves the vault, **on the same connection before
    reopen**, still at **v3** with **neither** import table (the
    `ROLLBACK` undoes the first `CREATE`), never half-built.
  - **Idempotent.** Re-running `run_migrations` on a **v4** vault is a
    no-op (version stays 4; no table re-created) — inherent in the
    version loop.
  - **Baseline-complete.** A brand-new first-run vault ends at exactly
    **v4** with both import tables present and empty (D1).
  - **Forward-only.** A vault whose recorded version exceeds
    `LATEST_SCHEMA_VERSION` (now = 4) is refused with the existing
    `SchemaVersionError` — already covered by the accounts suite's
    `LATEST_SCHEMA_VERSION + 1` refusal legs (symbolic, auto-tracking);
    the import suite does **not** duplicate it.
  Source: design.md "Persistence"; FIBR-0005/0006 (the runner).

- **INV-9 — A changed header is unmapped, never mis-parsed.** Because a
  profile is applied **only** on an exact header-signature match (INV-1/
  INV-2), its mapped columns are present in the file by construction and
  are resolved by exact `csv.DictReader` fieldname (a direct `row[col]`),
  so no mapped column is ever missing at parse time. A file whose header
  differs in **any** column — renamed, added, removed, re-cased, re-spaced,
  or reordered — has a different signature, so `match_profile` returns
  `None` and the wizard shows the mapping step; a stale profile never
  silently mis-maps a changed export. Leg: save a profile against a header,
  then present a file with a one-column-renamed header — `match_profile`
  returns `None` (the wizard maps fresh), and no parse uses the old
  profile. (This is why no `ProfileMismatchError` exists — an exact match
  guarantees column presence, so guarding the missing-column case would be
  an error path for an impossible scenario, coding.md § 2.) Source:
  ADR-0005 (bank changed its export); coding.md § 2.

- **INV-10 — Import-wizard UI round-trip** (`qtbot`; sub-cases per
  testing.md § 4):
  - **INV-10a** — picking a file whose header **matches** a saved profile
    skips the mapping step and lands on the preview with the profile
    auto-applied.
  - **INV-10b** — picking a file with **no** matching profile shows the
    mapping step; completing it (and opting to save) previews the parsed
    rows and persists a new profile (INV-1 through the UI).
  - **INV-10c** — the preview shows the parsed rows, the
    `N new / M duplicate / K error` summary, per-row error highlighting,
    and confirm-period fields (`_period_start`/`_period_end`) defaulted to
    the parsed min/max.
  - **INV-10d** — pressing **Import** inserts exactly the `N` new rows
    and returns to the main window with the table showing them; a period
    row is recorded (INV-6).
  - **INV-10e** — importing the **same** file a second time previews as
    all-duplicate (`N = 0`) and the **Import** adds zero (INV-5 through
    the UI).
  All strings via `self.tr(...)`; layout via Qt layout managers +
  `QStackedLayout` (RTL-safe, coding.md § 5.2). The `qtbot` legs poke
  named widget attributes (as FIBR-0006's `_select_category` did): the
  test hooks `_select_file(path)` (bypassing `QFileDialog`) and
  `_goto_step(index)`; the step-1 `_account_combo` (populated from
  `AccountService(service.vault).list_accounts()`); the mapping-form
  `_column_combos` (a `dict` keyed by role — `"date"`, `"description"`,
  `"amount"`, `"debit"`, `"credit"` — each a `QComboBox` of header
  fieldnames), `_amount_style` (single-vs-debit/credit chooser),
  `_invert_amount` (checkbox), `_date_format`, and `_profile_name`; the
  preview `_preview_table` (**every** data row in file order, error rows
  flagged/highlighted by their `RowError.row_number`), the `_summary_label`
  (the `N new / M duplicate / K error` counts), `_period_start` /
  `_period_end` fields, and the `_import_button` (disabled when zero
  drafts, INV-4). Source: glossary "mapping profile"; design.md § Data flow
  (import to insight); coding.md § 5.2.

- **INV-11 — No secret/network regression in the new modules.** The
  files this phase adds introduce **no** network import (`socket`,
  `http`, `urllib`, `requests`, `ftplib`, …) anywhere under
  `src/finbreak/`, and log **no** secret. Two legs (mirroring
  FIBR-0005/0006 INV-8):
  - **No-network (automatic).** The vault-suite's existing static import
    scan already walks all of `src/finbreak/`, covering the new modules
    with no new test.
  - **No-secret-log (new leg).** A `caplog` capture over a
    match-profile → preview → import cycle asserts the fixture's known
    vault password — `conftest._PW` is `bytes`, so the assertion compares
    its `.decode()` against the joined captured log messages (as the vault
    suite's `test_INV7_…_logs_carry_no_secret` leg does) — never appears in
    the captured log. `_PW` is the one concrete secret in play, since
    `ImportService` only ever sees the *unlocked* vault. Transaction
    descriptions/amounts and the source filename are user data, not
    secrets, and may appear in lifecycle lines.
  Source: security-model INV-8, INV-9.

## Deliverables

**New application modules** (`src/finbreak/`; shortest correct
implementation, coding.md § 1.1):

- `importers/csv_importer.py` — a **pure** `CsvImporter` (no DB, no Qt):
  `read_header(text) -> list[str]` (the `csv.DictReader` fieldnames, for
  signature + mapping UI; raises `ValueError` on an **empty** file — no
  header line, `DictReader.fieldnames` is `None`; a genuinely headerless

  `parse(text, mapping: ColumnMapping, exponent) -> ParseResult` — a value

## Data model (v4)

The v3→v4 migration adds two tables and leaves the rest untouched:

- `import_profiles(id INTEGER PRIMARY KEY, name TEXT NOT NULL, signature
  TEXT NOT NULL UNIQUE, date_column TEXT NOT NULL, description_column
  TEXT NOT NULL, amount_column TEXT, debit_column TEXT, credit_column
  TEXT, date_format TEXT NOT NULL, invert_amount INTEGER NOT NULL DEFAULT
  0, created_at TEXT NOT NULL)` — one row per saved bank layout.
  `signature` is `UNIQUE` (the match key, D3). Exactly one amount style
  is populated (`amount_column`, **or** `debit_column` + `credit_column`)
  — enforced at the **service** layer (D3), not by a DB `CHECK`.
  `invert_amount` is 0/1 (SQLite has no bool). The repository `SELECT`
  and the `ImportProfile` dataclass share this exact column order so
  `ImportProfile(*row)` stays aligned (positional-unpack pin).
- `statement_periods(id INTEGER PRIMARY KEY, account_id INTEGER NOT NULL
  REFERENCES accounts(id), period_start TEXT NOT NULL, period_end TEXT
  NOT NULL, source_filename TEXT, imported_at TEXT NOT NULL)` — one row
  per import, the coverage-period record (D8). The `account_id` FK is
  enforced by `PRAGMA foreign_keys = ON` (FIBR-0005 D4). Span-uniqueness
  per account is enforced at the **service** layer (INV-6), not a DB
  `UNIQUE`. `period_start`/`period_end`/`imported_at` are ISO-8601
  strings (dates for the period, a UTC timestamp for `imported_at`),
  matching the app's string-date convention (`Transaction.occurred_on`);
  the service validates `period_start <= period_end` (INV-6).
  `source_filename` is nullable — the CSV path always records the picked
  file's basename, but it is left `TEXT` (not `NOT NULL`) because a future
  OFX/PDF importer may not have a filename to record.
- `transactions`, `accounts`, `categories`, `settings`,
  `schema_version` — **unchanged** (no new column; dedup reads existing
  descriptions and normalises them in the service, D6). `schema_version`'s
  single row now holds `4`.

**No seed** — the v3→v4 step is two `CREATE TABLE`s only (starter
profiles deferred, D11). Forward-only: a later phase (P08/FIBR-0010) adds
`transactions.category_id` via its **own** migration step when it first
assigns categories; this one is never reversed.

## Verified API basis

Pins the idioms the INVs assert against, so the contract cites real APIs
(global rule § 13). To be **re-verified empirically at implementation
time** against the pinned `sqlcipher3-binary` 0.6.0 and PySide6 6.11.1.
The migration mechanics (the runner-owned transaction — see Context) are
already proven by FIBR-0005's `_migrate_to_v2` and FIBR-0006's
`_migrate_to_v3` — this phase adds one identical step.

- **`parse_transaction`** (`services/transactions.py:22`, verified) —
  `(occurred_on, raw_amount, description, exponent) -> (str, int, str)`,
  raising `ValueError` on blank description / non-ISO date /
  non-numeric·non-finite·zero·over-precise amount. Accepts `raw_amount`
  as `str | Decimal`; the importer passes a signed `Decimal`/string. The
  exponent is read via the new `read_minor_unit_exponent(conn)`
  extraction (D5), reusing the `settings.minor_unit_exponent` row the
  vault seeds.
- **stdlib `csv.DictReader`** — name-based row access keyed on the header
  (`reader.fieldnames`), so a profile maps each field by exact column
  **name** (`row[col]`). Duplicate fieldnames collapse in `DictReader`,
  which is why a header with two equal names is refused up-front (a
  `ValueError` from `signature_for`, D4). Comma dialect, `utf-8-sig`
  decode (BOM tolerated) for MVP (D11).
- **`datetime.strptime(cell, date_format).date().isoformat()`** — the
  profile's `date_format` (e.g. `"%d/%m/%Y"`) parses the mapped date cell;
  a mismatch raises `ValueError` → a `RowError` (INV-4). The stored
  `occurred_on` is the re-emitted ISO-8601 date, matching
  `Transaction.occurred_on`.
- **Multiset dedup (normalised description)** — the service fetches
  existing descriptions in the `(account_id, occurred_on, amount_minor)`
  bucket (`SELECT description FROM transactions WHERE account_id = ? AND
  occurred_on = ? AND amount_minor = ?`), `_normalise`s them in Python
  (the D6 transform — SQLite cannot `casefold`), buckets by the normalised
  value, and inserts the positive delta per incoming normalised key. No
  schema change this phase; the indexed hash is FIBR-0026 (D6).
- **SQLite `UNIQUE` + FK** — `import_profiles.signature ... UNIQUE`
  (the match key, D3) and `statement_periods.account_id ... REFERENCES
  accounts(id)` (enforced by `PRAGMA foreign_keys = ON`, FIBR-0005 D4).
- **v3→v4 migration** — two plain `CREATE TABLE`s (no seed, no rebuild),
  wrapped in the runner's explicit `BEGIN … COMMIT`/`ROLLBACK`; written
  as **discrete** `conn.execute()` statements so INV-8's per-statement
  proxy can intercept the second `CREATE`.
- **PySide6 6.11.1 UI** — `QStackedLayout` for the wizard steps (a
  non-modal stacked `QWidget`, D9); `QFileDialog.getOpenFileName` for the
  file pick; `QTableWidget` for the preview with per-row error
  highlighting; `QComboBox.addItem(label, userData)` + `currentData()`
  for the per-field column pickers (mirroring the accounts/categories
  managers). Standard `self.tr(...)` + layout-manager idioms
  (coding.md § 5.2).
