**Theme:** The Home screen becomes a proper **dashboard** — a plain
income-vs-spending summary for a chosen period (defaulting to **last month**), a
**donut** of where the money went by category, and a **month-by-month** bar chart
of income against spending. Money you moved **between your own accounts** (a
confirmed transfer) never counts as income or spending. The raw transaction list
moves off Home into its own **Transactions** tab, where you can **search** by
description and filter by **date range**, **account**, and **category** — any or
all at once.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Transfers never counted.** Every transaction id in `TransferDetectionService.confirmed_transfer_txn_ids()` is excluded from **every** dashboard number — income, expenditure, net, the category donut, and the trend — whether one account or all-consolidated is selected. | ADR-0006; SC3 |
| INV-2 | **Default previous month; persisted.** With no stored preference the period is the **previous calendar month**. | INV-3 | **Period resolution is pure + total.** `resolve_period(prefs, today)` maps each of the five modes (previous-month, current-month, specific-month, year-to-date, specific-year) to an **inclusive** `[start, end]` date range, deterministically from an injected `today`. | testing.md § 2.1 (hermetic) |
| INV-4 | **Income / expenditure / net.** Over the period's **non-transfer** rows for the selected account (or all): income = Σ of positive `amount_minor`; expenditure = Σ of the magnitudes of negative `amount_minor`; net = income − expenditure. ## Context

Verified against source 2026-07-12 (see **Verified code basis**):

- **`transactions`** (v8) has `id, account_id, occurred_on, amount_minor,
  description, created_at, statement_period_id, category_id, category_source`.
  `occurred_on` is a **`TEXT` ISO-8601 date** (`YYYY-MM-DD`,
  `date.fromisoformat`-validated at the form boundary), so a date range is a plain
  **lexicographic** `occurred_on BETWEEN ? AND ?` (ISO dates sort as text).
- **Sign convention (FIBR-0007).** `amount_minor` is a **signed** integer: a
  **debit is negative** (money out → expenditure), a **credit is positive**
  (money in → income); `parse_transaction` forbids a **zero** amount. Confirmed by
  `HomeView.refresh`'s colouring (`display < 0` → "money out") and FIBR-0011 INV-2.
- **Exclusion primitive (FIBR-0011).** `TransferDetectionService(vault)
  .confirmed_transfer_txn_ids() -> set[int]` returns the txn ids in every
  **confirmed** transfer pair (rejections exclude nothing). This is the single
  primitive INV-1 subtracts.
- **Money scaling.** `services/transactions.py` `read_minor_unit_exponent(conn)`
  (the base-currency minor-unit exponent from `settings`) + `to_display_decimal(
  amount_minor, exponent)` turn minor units into a display `Decimal`;
  `TransactionService.base_currency()` reads the symbol. All reused — a **single
  base currency**, not per-account.
- **`settings` table (v1).** `SettingsRepository(conn).get(key)/set(key, value)`
  is the generic key→value store auto-lock (FIBR-0055), date-format (FIBR-0083)
  and amount (FIBR-0105) prefs already use — **no schema change** to persist a
  period. `AuthService.datetime_prefs()/amount_prefs()` are the read/parse-with-
  fallback + `set_*` write pattern `report_prefs` mirrors: each stored key is
  parsed defensively and falls back to its default (INV-2).
- **`list_transactions()`** returns `list[tuple[Transaction, Decimal, str, str]]`
  — each row + its display `Decimal` + account name + **leaf** category name
  (`""` for uncategorised); the id→name maps are resolved **once** per call. This
  is the exact feed the relocated Transactions table needs (INV-8/9); the
  **dashboard** aggregation instead reads raw `(id, occurred_on, amount_minor,
  category_id)` rows in a date range (D4).
- **Home (`ui/home.py`).** `HomeView(transactions, categorization, prefs,
  amount_prefs)` is a `QStackedWidget` toggling `home_page_empty` (getting-started)
  / `home_page_table` (the table). It owns the table, the right-click
  `_show_context_menu` → `_on_set_category` → `_apply_category` →
  `_maybe_offer_rule` → `_apply_learned_rule` chain, `_selected_txn`/`_select_txn`,
  `set_datetime_prefs`/`set_amount_prefs` live-update, `transaction_count()` (feeds
  the status bar), and the module-level `_format_amount(display, symbol,
  negative_style)`. The getting-started page emits `add_account_requested` /
  `import_requested` / `add_transaction_requested`.
- **Shell (`ui/main_window.py`).** A **6-tab** `QTabWidget` (`_TAB_HOME = 0` …
  `_TAB_TRANSFERS = 5`). `_build_workspace` builds each tab once — Home from
  `TransactionService` + `CategorizationService` over `self._service.vault` (+ the
  `_prefs`/`_amount_prefs` read once at the top); the other tabs from
  `self._service` (an `AuthService`). `_refresh_tab(index)` refreshes the activated
  tab; `_on_tab_changed` also `_save_geometry()` (persists the last tab).
  `_clear_live` nulls every `_*_tab` on lock. Tab actions are built once via
  `_make_action("action_rules", self.tr("Rules"), "rules", self._open_rules)` and
  added to `_menu_view` + the toolbar; `_open_rules` sets the index via
  `_ensure_workspace()`. `_enter_unlocked` reads `self._home_tab.transaction_count()`
  for the status bar; `_on_manual_committed` calls `_show_home()`.
- **`_table_state.py` (FIBR-0117).** `SortableItem`, `enable_sorting`,
  `fill_guard`, `remember_columns`, `tag_row(table, row, index)` /
  `selected_index` / `select_by_index` — the sort-safe row-tag scheme the relocated
  table already uses on Home and carries with it.

## Design decisions

- **D1 — QtCharts (ADR-0008).** Both charts use `PySide6.QtCharts` — the donut a
  `QChart` + `QPieSeries` (a non-zero `setHoleSize` makes it a donut); the trend a
  `QChart` + `QBarSeries` with two `QBarSet`s (Income, Expenditure) over a
  `QBarCategoryAxis` of 12 month labels. Each sits in a `QChartView`. No new
  dependency (verified `PySide6.QtCharts` 6.11.1 imports); PDF-capable for
  FIBR-0013 (same `QPainter` engine). Full rationale + trade-off in ADR-0008.

- **D2 — `ReportPrefs` + persistence (no schema change).** A frozen
  `ReportPrefs(mode: str, year: int | None, month: int | None)` in
  **`services/reporting.py`** — co-located with the period resolver that consumes
  it (a deliberate, cohesion-driven deviation from the flat `AmountPrefs` /
  `DateTimePrefs` living in `auth.py`: unlike those, this prefs value drives a
  non-trivial period model that belongs with `resolve_period`). The five `mode`
  tokens are module constants (`MODE_PREVIOUS_MONTH = "previous_month"`,
  `MODE_CURRENT_MONTH`, `MODE_SPECIFIC_MONTH`, `MODE_YEAR_TO_DATE`,
  `MODE_SPECIFIC_YEAR`). `AuthService.report_prefs() -> ReportPrefs` /
  `set_report_prefs(prefs)` read/write three `settings` keys
  (`report_period_mode` / `report_period_year` / `report_period_month`), **each
  parsed defensively** with a fallback (an unknown mode, or a non-int/out-of-range
  year/month, → the previous-month default), mirroring `amount_prefs` /
  `datetime_prefs`. `AuthService` imports `ReportPrefs` from `services.reporting`
  (acyclic: `reporting.py` never imports `auth.py`).

- **D3 — Pure period model (`services/reporting.py`).** Two pure functions,
  `today` injected (default `date.today()`) so tests are hermetic (the
  `datetime_format.py` precedent):
  - `resolve_period(prefs, today) -> tuple[date, date]` — inclusive `[start,
    end]`:
    - **previous-month:** the calendar month before `today`'s (first day … last
      day). Crosses the year boundary in January (→ the prior December).
    - **current-month:** `today`'s month, first day … last day.
    - **specific-month(`year`, `month`):** that month's first … last day.
    - **year-to-date:** Jan 1 of `today`'s year … **`today`** (inclusive).
    - **specific-year(`year`):** Jan 1 … Dec 31 of `year`.
  - `resolve_trend_months(prefs, today) -> list[tuple[int, int]]` — the **12**
    `(year, month)` pairs ending at the resolved period's **end month**
    (inclusive), oldest first (INV-6). (End month = `resolve_period(...)[1]`'s
    year/month.) So specific-year(2025) → Jan..Dec 2025; year-to-date in July →
    Aug (prev year)..July; previous-month in January → Feb (prev year)..Jan.

  Month arithmetic uses stdlib `datetime` / `calendar.monthrange` only (no new
  dep). Both functions are covered by hermetic unit tests with a fixed `today`,
  including the January year-boundary and the Feb-29 leap edge for month-length.

- **D4 — `ReportingService` + `ReportingRepository`.**
  `repositories/reporting.py` `ReportingRepository(conn).rows_in_range(start_iso,
  end_iso, account_id: int | None) -> list[tuple[int, str, int, int | None]]`
  returns `(id, occurred_on, amount_minor, category_id)` for rows whose
  `occurred_on` is in `[start_iso, end_iso]` and (`account_id is None` **or**
  `account_id = ?`) — a read, no commit. `services/reporting.py`
  `ReportingService(vault)` (vault-scoped, `_conn` property mirroring
  `CategorizationService`) exposes, each taking `(prefs, account_id, today)` (or a
  pre-resolved range) and building the exclusion set **once** per call from
  `TransferDetectionService(self._vault).confirmed_transfer_txn_ids()`:
  - `summary(prefs, account_id, today) -> Summary` — resolve period → rows →
    drop transfer ids → sum (INV-4).
  - `spending_by_category(prefs, account_id, today) -> list[CategorySpend]` —
    expenditure rows only → drop transfers → group by leaf category **name**
    (an id→name map from `CategoryRepository.list_all()`, resolved once; `None` →
    "Uncategorised") → sort descending (INV-5).
  - `monthly_trend(prefs, account_id, today) -> list[MonthlyTotal]` — one
    `rows_in_range` over the 12-month span (`resolve_trend_months` first→last),
    drop transfers, bucket by `occurred_on[:7]` into the 12 months (zero-filled),
    income/expenditure per month (INV-6).

  The exclusion, the sign split, and the minor→Decimal display scaling (reusing
  `read_minor_unit_exponent` + `to_display_decimal`) all live in this **one**
  service — no duplicate aggregation predicate (coding.md § 1.3). The self-joined
  transfer set + the range scan are un-indexed reads recomputed live per view
  (INV-10); **no index needed at personal-finance scale** (coding.md § 1.1).

- **D5 — Read-model DTOs (`models.py`).** Field order pinned (mirrors FIBR-0011's
  discipline so the widget + tests can't diverge):
  - `Summary(income: Decimal, expenditure: Decimal, net: Decimal)` — display
    Decimals (positive magnitudes; `net` may be negative).
  - `CategorySpend(name: str, amount: Decimal)` — one donut slice (positive
    magnitude).
  - `MonthlyTotal(label: str, income: Decimal, expenditure: Decimal)` — one trend
    month; `label` is the display month tag (e.g. `"2026-06"`; the axis renders
    it, formatting is a UI concern, not stored).

- **D6 — `HomeView` becomes the dashboard.** `HomeView` keeps its
  `QStackedWidget`: page 0 the **unchanged** getting-started page (`home_page_empty`
  + its three signals), page 1 now the **dashboard** (`home_page_dashboard`, was
  `home_page_table`). The dashboard page: a top row with a **period selector**
  (`objectName "period_selector"` — a `QComboBox` of the five modes; specific-month
  / specific-year reveal a secondary month/year picker) and an **account selector**
  (`"account_selector"` — "All accounts" first (id `None`) then each account); a
  row of **three tiles** (`"dashboard_income"` / `"dashboard_expenditure"` /
  `"dashboard_net"`, income green / expenditure red per D9); the **donut**
  `QChartView` (`"dashboard_category_chart"`); the **trend** `QChartView`
  (`"dashboard_trend_chart"`). `HomeView` now holds a `ReportingService` + an
  `AccountService` (for the account list) instead of the table; it drops the table,
  the context-menu chain, and `_selected_txn`/`_select_txn` (moved to D7).
  `refresh()` re-reads: builds the account list, resolves the held `ReportPrefs`,
  and repaints tiles + both charts (getting-started iff zero transactions, INV-7).
  Changing either selector persists the new `ReportPrefs` (period) via
  `AuthService.set_report_prefs` and re-renders; the account selection is **not**
  persisted (D11 out-of-scope). `set_datetime_prefs` is dropped (Home shows no
  dates); `set_amount_prefs` stays (the tiles use `_format_amount`).

- **D7 — `TransactionsView` (`ui/transactions.py`).** A new `QWidget`
  (`objectName "tab_transactions"`) holding the **relocated** table + a **filter
  bar**. The table, its 5 columns (Date / Amount / Description / Account /
  Category), the `_table_state` sort + remembered columns, the right-click
  `_show_context_menu` → set-category → `_maybe_offer_rule` → learn chain, and
  `_selected_txn`/`_select_txn` move **verbatim** from `HomeView` (it holds a
  `TransactionService` + `CategorizationService`, as Home did). `refresh()` reloads
  `list_transactions()` into an **unfiltered** master list, then applies the active
  filters to populate the table (INV-9). `transaction_count()` returns the
  **unfiltered** total (the status-bar figure, INV-14). `set_amount_prefs` /
  `set_datetime_prefs` live-update (the table shows both amounts and dates).

- **D8 — Filter bar (in-memory, AND-combined).** Four independently-active filters
  over the master `list_transactions()` rows (personal-finance scale → filter in
  Python, no new query; `rows_in_range` is dashboard-only):
  - **Search** (`"txn_search"`, `QLineEdit`) — case-insensitive substring of
    `description`; empty → inactive.
  - **Date range** (`"txn_date_enable"` `QCheckBox` gating two `QDateEdit`s
    `"txn_date_from"` / `"txn_date_to"`) — unchecked → inactive; checked → keep
    rows with `from <= occurred_on <= to` (ISO string compare). Explicit enable so
    "no date filter" is unambiguous (a bare `QDateEdit` has no empty state).
  - **Account** (`"txn_account"`, `QComboBox`, "All accounts" = `None` first) — All
    → inactive; else `account_id` match.
  - **Category** (`"txn_category"`, `QComboBox`, "All categories" first, then
    "Uncategorised" (`category_id is None`), then each leaf) — All → inactive.

  Any change re-applies all active filters and repopulates under `fill_guard`
  (re-tagging rows over the filtered list so `selected_index` stays correct after a
  re-sort, INV-9). The master list is reloaded only by `refresh()` (tab
  activation) — filtering never re-hits the DB.

- **D9 — Chart theming + palette.** Charts are **theme-aware**: the `QChart`
  background is transparent (the app panel colour shows), title/legend/axis text
  from the current `QPalette` (ADR-0002 dark default). A fixed **accessible
  categorical palette** colours the donut slices (the app-icon family —
  green / blue / teal / orange … — extended to a documented N-colour set, dataviz
  contrast-checked); the "Uncategorised" slice is a muted neutral. The trend's two
  bar sets reuse the tile direction colours: Income green (`_POSITIVE_TEXT` family),
  Expenditure red (`_NEGATIVE_TEXT` family). The tile amount colours and the shared
  `_format_amount` (D10) keep screen + (future) PDF consistent.

- **D10 — Shared `_format_amount` (`ui/_amount.py`).** Relocate the existing
  module-level `_format_amount(display, symbol, negative_style)` out of `home.py`
  into a new `ui/_amount.py`; import it in the dashboard tiles (`home.py`) **and**
  the relocated table (`transactions.py`). It already exists — this is a
  **relocation** so both new call sites share the one implementation
  (coding.md § 1.3), not a new abstraction. Move the direction-colour constants
  (`_NEGATIVE_TEXT` / `_POSITIVE_TEXT`) alongside it (both files + the charts use
  them).

- **D11 — Shell wiring (`ui/main_window.py`).** `_TAB_TRANSACTIONS = 1`; the
  others renumber (`_TAB_STATEMENTS = 2` … `_TAB_TRANSFERS = 6`). In
  `_build_workspace`, build `self._transactions_tab = TransactionsView(
  TransactionService(self._service.vault), CategorizationService(
  self._service.vault), self._prefs, self._amount_prefs)` and `addTab(...,
  self.tr("Transactions"))` **second** (after Home); Home is built as the dashboard
  (`HomeView(ReportingService(...), AccountService(...), self._service, ...)` — the
  exact constructor pinned at implementation). Add `_transactions_tab = None` to
  `_clear_live`; add the `_refresh_tab` branch (`elif index == _TAB_TRANSACTIONS`);
  build `self._action_transactions = self._make_action("action_transactions",
  self.tr("Transactions"), "transactions", self._open_transactions)` + View-menu +
  toolbar entry + `_open_transactions`. **Count sourcing** moves to the Transactions
  tab: `_enter_unlocked` and `_refresh_tab`'s Home branch read
  `self._transactions_tab.transaction_count()` (Home no longer owns the table);
  `_on_manual_committed` navigates to **Transactions** (so the just-added row is
  visible) and refreshes it. New tab/action glyph **`ui/icons/transactions.svg`**
  (travels via the existing `*.svg` package-data glob + PyInstaller `--add-data`).

- **D12 — Bundle-travel proof (`_selftest.py`).** Add a `import PySide6.QtCharts`
  leg to the `--self-test` native-stack check (alongside Qt / SQLCipher / qpdf /
  pdfplumber), so PyInstaller's PySide6 hook bundles `Qt6Charts` and a missing lib
  fails the FIBR-0003 build smoke, not a user launch (INV-11).

## Invariants

- **INV-1 — Transfers never counted.** Confirm a transfer pair, then assert its
  two txn ids contribute to **none** of `summary` / `spending_by_category` /
  `monthly_trend`, in both the consolidated (`account_id=None`) and single-account
  views. Test: seed two accounts with a matched debit/credit + ordinary income +
  expense; the confirmed pair drops out of every figure; an **un**confirmed same
  pair is still counted (proves the exclusion keys on *confirmed*).
- **INV-2 — Default previous month; persisted.** A fresh vault → `report_prefs()`
  is `MODE_PREVIOUS_MONTH`. `set_report_prefs(specific_year(2025))` then a fresh
  `AuthService` over the same vault reads it back. A garbage `report_period_mode`
  value → previous-month, no exception. (Round-trip + fallback tests in the
  settings suite, mirroring `datetime_prefs` / `amount_prefs`.)
- **INV-3 — Period resolution.** Table-driven hermetic tests with a fixed `today`
  (e.g. 2026-07-12): each mode → its exact `[start, end]`; the **January**
  previous-month boundary (→ prior December) and a **leap-February** specific-month
  (2024-02 → 29 days) covered.
- **INV-4 — Income / expenditure / net.** Seeded vault, fixed period: income = Σ
  positive, expenditure = Σ |negative|, net = income − expenditure, transfers
  excluded, over the selected account (and consolidated). A period with no rows →
  all-zero. Computed on `amount_minor` (INV-13).
- **INV-5 — Category donut.** Expenditure rows grouped by leaf name; an
  uncategorised expense lands in exactly one "Uncategorised" slice; an **income**
  row is absent; slices sorted descending. Transfers excluded.
- **INV-6 — Trend span.** `monthly_trend` returns **12** points, oldest first,
  ending at the period's end month; an empty month is a zero point (present, not
  dropped); specific-year → that year's Jan..Dec; transfers excluded. A row on the
  last day of the end month is included; the day after the span start of 12 months
  ago is the first bucket.
- **INV-7 — Home is the dashboard.** `HomeView` with zero transactions shows
  `home_page_empty`; with data shows `home_page_dashboard` carrying
  `period_selector` / `account_selector` / the three tiles / both `QChartView`s and
  **no** `QTableWidget`.
- **INV-8 — Transactions tab.** The tab (`tab_transactions`) shows the table;
  right-click **Set category…** sets a row manual and offers to learn a rule
  exactly as the old Home path did (the moved chain), auto-lock mid-dialog caught.
- **INV-9 — Filters combine.** With all four active at once, the visible rows are
  exactly the AND of the predicates; clearing one widens the set; "Uncategorised"
  selects `category_id is None`; after a header re-sort the right-click still acts
  on the correct transaction (the `tag_row` index is over the filtered list). Test
  drives each filter alone and all-combined on a seeded set.
- **INV-10 — Live on view.** `ReportingService` reads live vault state each call
  (no cache); `TransactionsView.refresh()` reloads the master list. A newly-inserted
  row / a newly-confirmed transfer changes the next render with no other action.
- **INV-11 — QtCharts bundled, no new dep.** `pyproject.toml` gains **no** runtime
  dependency; `--self-test` imports `PySide6.QtCharts`; the FIBR-0003 build smoke
  still passes (a bundle missing `Qt6Charts` fails it).
- **INV-12 — No network; i18n.** `test_INV8_no_network_imports_under_src` still
  passes (the new modules import no network); every user string `tr()`, every
  widget in a layout manager (ruff/convention).
- **INV-13 — Money exact.** No aggregation touches `float`; the only float
  conversion is inside `_format_amount` for the on-screen string. Test: a large
  set sums identically via the service and via a `Decimal` reference.
- **INV-14 — Tab order + count.** The workspace has **7** tabs in the D11 order;
  `_TAB_*` constants match; the status bar shows the unfiltered transaction count;
  a `VaultLockedError` on any new slot is caught (no crash).

## Data model — no schema change

**No migration.** `LATEST_SCHEMA_VERSION` is unchanged (stays 8). The one
persisted preference is a **period selection** stored in the existing **v1
`settings`** table under `report_period_mode` / `report_period_year` /
`report_period_month` (D2) — the same key→value store auto-lock, date-format, and
amount prefs already use. All reporting is **read-only** aggregation over the
existing `transactions` / `categories` / `transfer_pairs` tables; nothing new is
written to the schema.

## Out of scope

- **Rendering these charts into the locked PDF export** — **FIBR-0013** (P11).
  QtCharts is chosen (ADR-0008) *so that* FIBR-0013 can paint the same chart
  objects onto the `QPdfWriter` device; this phase renders on screen only.
- **Persisting the selected account** — the user asked only for the **period** to
  persist; the account selector resets to "All accounts" each session (a stored
  account id is also fragile across account deletes). A clean future follow-up.
- **Budgets, forecasting, savings goals, alerts** — not in the dashboard brief.
- **Drill-down from a donut slice / trend bar to its transactions** — a nice
  future affordance; not built now (coding.md § 1.1).
- **Marking transfer rows in the Transactions table** (a transfer badge) — display
  is unchanged; carried from FIBR-0011's out-of-scope, a possible later polish.
- **Near-amount / fee-tolerant transfer matching, multi-currency dashboards,
  per-account currency** — single base currency; exact-magnitude transfers only
  (ADR-0006 / FIBR-0011). Unchanged here.
- **A custom day-granular range on the dashboard** — the period model is
  month/year-granular by the user's choice; the **Transactions tab** covers
  arbitrary day ranges (D8).
