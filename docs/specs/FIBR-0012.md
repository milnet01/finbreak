income-vs-spending summary for a chosen period (defaulting to **last month**), a

## Context

Verified against source 2026-07-12 (see **Verified code basis**):

- **`transactions`** (v8) has `id, account_id, occurred_on, amount_minor,
  description, created_at, statement_period_id, category_id, category_source`.
  `occurred_on` is a **`TEXT` ISO-8601 date** (`YYYY-MM-DD`,
  `date.fromisoformat`-validated at the form boundary), so a date range is a plain
  **lexicographic** `occurred_on BETWEEN ? AND ?` (ISO dates sort as text).
- **Sign convention (FIBR-0007).** `amount_minor` is a **signed** integer: a
  **debit is negative** (money out в†’ expenditure), a **credit is positive**
  (money in в†’ income); `parse_transaction` forbids a **zero** amount. Confirmed by
  `HomeView.refresh`'s colouring (`display < 0` в†’ "money out") and FIBR-0011 INV-2.
- **Exclusion primitive (FIBR-0011).** `TransferDetectionService(vault)
  .confirmed_transfer_txn_ids() -> set[int]` returns the txn ids in every
  **confirmed** transfer pair (rejections exclude nothing). This is the single
  primitive INV-1 subtracts.
- **Money scaling.** `services/transactions.py` `read_minor_unit_exponent(conn)`
  (the base-currency minor-unit exponent from `settings`) + `to_display_decimal(
  amount_minor, exponent)` turn minor units into a display `Decimal`;
  `TransactionService.base_currency()` reads the symbol. All reused вЂ” a **single
  base currency**, not per-account.
- **`settings` table (v1).** `SettingsRepository(conn).get(key)/set(key, value)`
  is the generic keyв†’value store auto-lock (FIBR-0055), date-format (FIBR-0083)
  and amount (FIBR-0105) prefs already use вЂ” **no schema change** to persist a
  period. `AuthService.datetime_prefs()/amount_prefs()` are the read/parse-with-
  fallback + `set_*` write pattern `report_prefs` mirrors: each stored key is
  parsed defensively and falls back to its default (INV-2).
- **`list_transactions()`** returns `list[tuple[Transaction, Decimal, str, str]]`
  вЂ” each row + its display `Decimal` + account name + **leaf** category name
  (`""` for uncategorised); the idв†’name maps are resolved **once** per call. This
  is the exact feed the relocated Transactions table needs (INV-8/9); the
  **dashboard** aggregation instead reads raw `(id, occurred_on, amount_minor,
  category_id)` rows in a date range (D4).
- **Home (`ui/home.py`).** `HomeView(transactions, categorization, prefs,
  amount_prefs)` is a `QWidget` holding an internal `QStackedWidget` toggling `home_page_empty` (getting-started)
  / `home_page_table` (the table). It owns the table, the right-click
  `_show_context_menu` в†’ `_on_set_category` в†’ `_apply_category` в†’
  `_maybe_offer_rule` в†’ `_apply_learned_rule` chain, `_selected_txn`/`_select_txn`,
  `set_datetime_prefs`/`set_amount_prefs` live-update, `transaction_count()` (feeds
  the status bar), and the module-level `_format_amount(display, symbol,
  negative_style)`. The getting-started page emits `add_account_requested` /
  `import_requested` / `add_transaction_requested`.
- **Shell (`ui/main_window.py`).** A **6-tab** `QTabWidget` (`_TAB_HOME = 0` вЂ¦
  `_TAB_TRANSFERS = 5`). `_build_workspace` builds each tab once вЂ” Home from
  `TransactionService` + `CategorizationService` over `self._service.vault` (+ the
  `_prefs`/`_amount_prefs` read once at the top); the other tabs from
  `self._service` (an `AuthService`). `_refresh_tab(index)` refreshes the activated
  tab; `_on_tab_changed` also `_save_geometry()` (persists the last tab).
  `_clear_live` nulls every `_*_tab` on lock. Tab actions are built once via
  `_make_action("action_rules", self.tr("Rules"), "rules", self._open_rules)` and
  added to `_menu_view` + the toolbar; `_open_rules` sets the index via
  `_ensure_workspace()`. `_enter_unlocked` reads `self._home_tab.transaction_count()`
  for the status bar; `_on_manual_committed` calls `_show_home()`.
- **`_table_state.py` (FIBR-0117).** `SortableItem`, `enable_sorting`,
  `fill_guard`, `remember_columns`, `tag_row(table, row, index)` /
  `selected_index` / `select_by_index` вЂ” the sort-safe row-tag scheme the relocated
  table already uses on Home and carries with it.

## Design decisions

- **D1 вЂ” QtCharts (ADR-0008).** Both charts use `PySide6.QtCharts` вЂ” the donut a
  `QChart` + `QPieSeries` (a non-zero `setHoleSize` makes it a donut); the trend a
  `QChart` + `QBarSeries` with two `QBarSet`s (Income, Expenditure) over a
  `QBarCategoryAxis` of 12 month labels. Each sits in a `QChartView`. No new
  dependency (verified `PySide6.QtCharts` 6.11.1 imports); PDF-capable for
  FIBR-0013 (same `QPainter` engine). Full rationale + trade-off in ADR-0008.

- **D2 вЂ” `ReportPrefs` + persistence (no schema change).** A frozen
  `ReportPrefs(mode: str, year: int | None, month: int | None)` in
  **`services/reporting.py`** вЂ” co-located with the period resolver that consumes
  it (a deliberate, cohesion-driven deviation from the flat `AmountPrefs` /
  `DateTimePrefs` living in `auth.py`: unlike those, this prefs value drives a
  non-trivial period model that belongs with `resolve_period`). The five `mode`
  tokens are module constants (`MODE_PREVIOUS_MONTH = "previous_month"`,
  `MODE_CURRENT_MONTH`, `MODE_SPECIFIC_MONTH`, `MODE_YEAR_TO_DATE`,
  `MODE_SPECIFIC_YEAR`). `AuthService.report_prefs() -> ReportPrefs` /
  `set_report_prefs(prefs)` read/write three `settings` keys
  (`report_period_mode` / `report_period_year` / `report_period_month`), **each
  parsed defensively** with a fallback (an unknown mode, or a non-int/out-of-range
  year/month, в†’ the previous-month default), mirroring `amount_prefs` /
  `datetime_prefs`. `AuthService` imports `ReportPrefs` from `services.reporting`
  (acyclic: `reporting.py` never imports `auth.py`).

- **D3 вЂ” Pure period model (`services/reporting.py`).** Two pure functions,
  `today` injected (default `date.today()`) so tests are hermetic (the
  `datetime_format.py` precedent):
  - `resolve_period(prefs, today) -> tuple[date, date]` вЂ” inclusive `[start,
    end]`:
    - **previous-month:** the calendar month before `today`'s (first day вЂ¦ last
      day). Crosses the year boundary in January (в†’ the prior December).
    - **current-month:** `today`'s month, first day вЂ¦ last day.
    - **specific-month(`year`, `month`):** that month's first вЂ¦ last day.
    - **year-to-date:** Jan 1 of `today`'s year вЂ¦ **`today`** (inclusive).
    - **specific-year(`year`):** Jan 1 вЂ¦ Dec 31 of `year`.
  - `resolve_trend_months(prefs, today) -> list[tuple[int, int]]` вЂ” the **12**
    `(year, month)` pairs ending at the resolved period's **end month**
    (inclusive), oldest first (INV-6). (End month = `resolve_period(...)[1]`'s
    year/month.) So specific-year(2025) в†’ Jan..Dec 2025; year-to-date in July в†’
    Aug (prev year)..July; previous-month in January в†’ **Jan..Dec (prev year)**
    (its period end month is the prior December, so the 12-month window is exactly
    that prior calendar year).

  Month arithmetic uses stdlib `datetime` / `calendar.monthrange` only (no new
  dep). Each `(year, month)` pair converts to its ISO first/last day via
  `calendar.monthrange`; a month's `rows_in_range` bucket key **and** the
  `MonthlyTotal.label` are the zero-padded `f"{year:04d}-{month:02d}"` (so keys and
  labels can't drift). Both functions are covered by hermetic unit tests with a
  fixed `today`, including the January year-boundary and the Feb-29 leap edge for
  month-length.

- **D4 вЂ” `ReportingService` + `ReportingRepository`.**
  `repositories/reporting.py` `ReportingRepository(conn).rows_in_range(start_iso,
  end_iso, account_id: int | None) -> list[tuple[int, str, int, int | None]]`
  returns `(id, occurred_on, amount_minor, category_id)` for rows whose
  `occurred_on` is in `[start_iso, end_iso]` and (`account_id is None` **or**
  `account_id = ?`) вЂ” a read, no commit. `services/reporting.py`
  `ReportingService(vault)` (vault-scoped, `_conn` property mirroring
  `CategorizationService`) exposes, each taking `(prefs, account_id, today)` (or a
  pre-resolved range) and building the exclusion set **once** per call from
  `TransferDetectionService(self._vault).confirmed_transfer_txn_ids()`:
  - `summary(prefs, account_id, today) -> Summary` вЂ” resolve period в†’ rows в†’
    drop transfer ids в†’ sum (INV-4).
  - `spending_by_category(prefs, account_id, today) -> list[CategorySpend]` вЂ”
    expenditure rows only в†’ drop transfers в†’ group by **`category_id`** (resolve
    each id to its leaf **name** once via a `{id: name}` map from
    `CategoryRepository.list_all()` вЂ” grouping by **id**, so two same-named leaves
    under different parents stay distinct, FIBR-0006 sibling-unique names; `None` в†’
    a synthetic "Uncategorised" bucket) в†’ sort descending (INV-5).
  - `monthly_trend(prefs, account_id, today) -> list[MonthlyTotal]` вЂ” one
    `rows_in_range` over the 12-month span (`resolve_trend_months` firstв†’last),
    drop transfers, bucket by `occurred_on[:7]` into the 12 months (zero-filled),
    income/expenditure per month (INV-6). The trend buckets **whole calendar
    months** by design вЂ” so for a year-to-date period ending mid-month the trend's
    final month spans the full month even though the summary tiles stop at `today`
    (harmless with no post-dated rows; documented, not a discrepancy).

  The exclusion, the sign split, and the minorв†’Decimal display scaling (reusing
  `read_minor_unit_exponent` + `to_display_decimal`) all live in this **one**
  service вЂ” no duplicate aggregation predicate (coding.md В§ 1.3). The self-joined
  transfer set + the range scan are un-indexed reads recomputed live per view
  (INV-10); **no index needed at personal-finance scale** (coding.md В§ 1.1).

- **D5 вЂ” Read-model DTOs (`models.py`).** Field order pinned (mirrors FIBR-0011's
  discipline so the widget + tests can't diverge):
  - `Summary(income: Decimal, expenditure: Decimal, net: Decimal)` вЂ” display
    Decimals (positive magnitudes; `net` may be negative).
  - `CategorySpend(name: str, amount: Decimal)` вЂ” one donut slice (positive
    magnitude).
  - `MonthlyTotal(label: str, income: Decimal, expenditure: Decimal)` вЂ” one trend
    month; `label` is the display month tag (e.g. `"2026-06"`; the axis renders
    it, formatting is a UI concern, not stored).

- **D6 вЂ” `HomeView` becomes the dashboard.** `HomeView` keeps its
  `QStackedWidget`: page 0 the **unchanged** getting-started page (`home_page_empty`
  + its three signals), page 1 now the **dashboard** (`home_page_dashboard`, was
  `home_page_table`). The dashboard page: a top row with a **period selector**
  (`objectName "period_selector"` вЂ” a `QComboBox` of the five modes; specific-month
  / specific-year reveal a secondary month/year picker **defaulting to the current
  calendar month / year** until the user changes it) and an **account selector**
  (`"account_selector"` вЂ” "All accounts" first (id `None`) then each account); a
  row of **three tiles** (`"dashboard_income"` / `"dashboard_expenditure"` /
  `"dashboard_net"`, income green / expenditure red per D9); the **donut**
  `QChartView` (`"dashboard_category_chart"`); the **trend** `QChartView`
  (`"dashboard_trend_chart"`). `HomeView` now holds a `ReportingService` + an
  `AccountService` (the account list via `AccountService.list_accounts() ->
  list[Account]`, keyed on `Account.id` / `Account.name`) instead of the table; it drops the table,
  the context-menu chain, and `_selected_txn`/`_select_txn` (moved to D7).
  `refresh()` re-reads: rebuilds the account list **preserving the current
  in-session account selection** (re-select the held account id if it still exists,
  else fall back to "All accounts"), resolves the held `ReportPrefs`,
  and repaints tiles + both charts (getting-started iff zero transactions, INV-7).
  Changing either selector persists the new `ReportPrefs` (period) via
  `AuthService.set_report_prefs` and re-renders; the account selection is **not**
  persisted (D11 out-of-scope). `set_datetime_prefs` is dropped (Home shows no
  dates); `set_amount_prefs` stays (the tiles use `_format_amount`).

- **D7 вЂ” `TransactionsView` (`ui/transactions.py`).** A new `QWidget`
  (`objectName "tab_transactions"`) holding the **relocated** table + a **filter
  bar**. The table, its 5 columns (Date / Amount / Description / Account /
  Category), the `_table_state` sort + remembered columns, the right-click
  `_show_context_menu` в†’ set-category в†’ `_maybe_offer_rule` в†’ learn chain, and
  `_selected_txn`/`_select_txn` move **verbatim** from `HomeView` (it holds a
  `TransactionService` + `CategorizationService`, as Home did). `refresh()` reloads
  `list_transactions()` into an **unfiltered** master list, then applies the active
  filters to populate the table (INV-9). `transaction_count()` returns the
  **unfiltered** total (the status-bar figure, INV-14). `set_amount_prefs` /
  `set_datetime_prefs` live-update (the table shows both amounts and dates).

- **D8 вЂ” Filter bar (in-memory, AND-combined).** Four independently-active filters
  over the master `list_transactions()` rows (personal-finance scale в†’ filter in
  Python, no new query; `rows_in_range` is dashboard-only):
  - **Search** (`"txn_search"`, `QLineEdit`) вЂ” case-insensitive substring of
    `description`; empty в†’ inactive.
  - **Date range** (`"txn_date_enable"` `QCheckBox` gating two `QDateEdit`s
    `"txn_date_from"` / `"txn_date_to"`) вЂ” unchecked в†’ inactive; checked в†’ keep
    rows with `from <= occurred_on <= to` (ISO string compare). Explicit enable so
    "no date filter" is unambiguous (a bare `QDateEdit` has no empty state).
  - **Account** (`"txn_account"`, `QComboBox`, "All accounts" = `None` first) вЂ” All
    в†’ inactive; else `account_id` match.
  - **Category** (`"txn_category"`, `QComboBox`, "All categories" first, then
    "Uncategorised" (matches `category_id is None`), then each leaf вЂ” each item
    carries its `category_id` as `itemData` so the match is by **id**, not the
    display name, FIBR-0006 sibling-unique names) вЂ” All в†’ inactive.

  Any change re-applies all active filters and repopulates under `fill_guard`
  (re-tagging rows over the filtered list so `selected_index` stays correct after a
  re-sort, INV-9). The master list is reloaded only by `refresh()` (tab
  activation) вЂ” filtering never re-hits the DB.

- **D9 вЂ” Chart theming + palette (pinned).** Charts are **theme-aware**: the
  `QChart` background is transparent (the app panel colour shows), title / legend /
  axis text from the current `QPalette` (ADR-0002 dark default). The donut slices
  use a **fixed ordered categorical palette** (accessible on the dark default; the
  app-icon family extended), applied in descending-magnitude order вЂ” **N = 8**:
  `#4E9F3D` (green), `#3D7EA6` (blue), `#2FA4A0` (teal), `#D98A29` (orange),
  `#8E6FBE` (purple), `#C0504D` (red), `#C9A227` (gold), `#7FB069` (light green).
  When a period has **more than 8** spending categories, the top **7** by magnitude
  keep their colour and the remainder collapse into a single **"Other"** slice
  (muted slate `#6B7A8F`) so the ring never exceeds 8 wedges; the synthetic
  **"Uncategorised"** bucket (INV-5) is always the muted slate `#9AA6B2` regardless
  of rank. An **empty** donut (a period with no spending вЂ” e.g. income only) shows a
  centred **"No spending in this period"** placeholder
  (`objectName "dashboard_category_empty"`), not a blank ring. The trend's two bar
  sets reuse the tile direction colours: Income = `_POSITIVE_TEXT`, Expenditure =
  `_NEGATIVE_TEXT` (the existing `home.py` constants relocated per D10). The three
  summary tiles: Income `_POSITIVE_TEXT`, Expenditure `_NEGATIVE_TEXT`, **Net
  `_POSITIVE_TEXT` when в‰Ґ 0 else `_NEGATIVE_TEXT`**. The tile amount colours and the
  shared `_format_amount` (D10) keep screen + (future) PDF consistent.

- **D10 вЂ” Shared `_format_amount` (`ui/_amount.py`).** Relocate the existing

## Invariants

- **INV-1 вЂ” Transfers never counted.** Confirm a transfer pair, then assert its
  two txn ids contribute to **none** of `summary` / `spending_by_category` /
  `monthly_trend`, in both the consolidated (`account_id=None`) and single-account
  views. Test: seed two accounts with a matched debit/credit + ordinary income +
  expense; the confirmed pair drops out of every figure; an **un**confirmed same
  pair is still counted (proves the exclusion keys on *confirmed*).
- **INV-2 вЂ” Default previous month; persisted.** A fresh vault в†’ `report_prefs()`
  is `MODE_PREVIOUS_MONTH`. `set_report_prefs(specific_year(2025))` then a fresh
  `AuthService` over the same vault reads it back. A garbage `report_period_mode`
  value в†’ previous-month, no exception. (Round-trip + fallback tests in the
  settings suite, mirroring `datetime_prefs` / `amount_prefs`.)
- **INV-3 вЂ” Period resolution.** Table-driven hermetic tests with a fixed `today`
  (e.g. 2026-07-12): each mode в†’ its exact `[start, end]`; the **January**
  previous-month boundary (в†’ prior December) and a **leap-February** specific-month
  (2024-02 в†’ 29 days) covered.
- **INV-4 вЂ” Income / expenditure / net.** Seeded vault, fixed period: income = ОЈ
  positive, expenditure = ОЈ |negative|, net = income в€’ expenditure, transfers
  excluded, over the selected account (and consolidated). A period with no rows в†’
  all-zero. Computed on `amount_minor` (INV-13).
- **INV-5 вЂ” Category donut.** Expenditure rows grouped by leaf name; an
  uncategorised expense lands in exactly one "Uncategorised" slice; an **income**
  row is absent; slices sorted descending. Transfers excluded.
- **INV-6 вЂ” Trend span.** `monthly_trend` returns **12** points, oldest first,
  ending at the period's end month; an empty month is a zero point (present, not
  dropped); specific-year в†’ that year's Jan..Dec; transfers excluded. The first
  bucket is `resolve_trend_months(...)[0]` and the last is the period's end month; a
  row on the **last day** of the end month is included.
- **INV-7 вЂ” Home is the dashboard.** `HomeView` with zero transactions shows
  `home_page_empty`; with data shows `home_page_dashboard` carrying
  `period_selector` / `account_selector` / the three tiles / both `QChartView`s and
  **no** `QTableWidget`.
- **INV-8 вЂ” Transactions tab.** The tab (`tab_transactions`) shows the table;
  right-click **Set categoryвЂ¦** sets a row manual and offers to learn a rule
  exactly as the old Home path did (the moved chain), auto-lock mid-dialog caught.
- **INV-9 вЂ” Filters combine.** With all four active at once, the visible rows are
  exactly the AND of the predicates; clearing one widens the set; "Uncategorised"
  selects `category_id is None`; after a header re-sort the right-click still acts
  on the correct transaction (the `tag_row` index is over the filtered list). Test
  drives each filter alone and all-combined on a seeded set.
- **INV-10 вЂ” Live on view.** `ReportingService` reads live vault state each call
  (no cache); `TransactionsView.refresh()` reloads the master list. A newly-inserted
  row / a newly-confirmed transfer changes the next render with no other action.
- **INV-11 вЂ” QtCharts bundled, no new dep.** `pyproject.toml` gains **no** runtime
  dependency; `--self-test` imports `PySide6.QtCharts`; the FIBR-0003 build smoke
  still passes (a bundle missing `Qt6Charts` fails it).
- **INV-12 вЂ” No network; i18n.** `test_INV8_no_network_imports_under_src` still
  passes (the new modules import no network); every user string `tr()`, every
  widget in a layout manager (ruff/convention).
- **INV-13 вЂ” Money exact.** No aggregation touches `float`; the only float
  conversion is inside `_format_amount` for the on-screen string. Test: a large
  set sums identically via the service and via a `Decimal` reference.
- **INV-14 вЂ” Tab order + count.** The workspace has **7** tabs in the D11 order;
  `_TAB_*` constants match; the status bar shows the unfiltered transaction count;
  a `VaultLockedError` on any new slot is caught (no crash).
