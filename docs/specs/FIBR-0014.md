## The problem (ADR-0003)

ADR-0003 (SQLCipher + Argon2id) accepts, as a deliberate consequence, that **a
forgotten master password means unrecoverable data** — and names *"an explicit
encrypted backup-export feature"* as the mitigation. For a backup to actually
mitigate a **forgotten** master password, it cannot be locked by that same
password. So the backup is keyed by a **separate backup password** the user keeps
safe; restoring needs only the backup password plus a **new** master password —
never the forgotten one.

## Verified current state (2026-07-13)

- **No backup code exists** — no export/import of the vault, no `.fbk`, no
  SQLCipher backup/rekey call anywhere. This is a from-scratch feature.
- **The crypto/vault path to reuse** (Explore-verified):
  - `crypto.py:18-24` — pinned Argon2id params (`ARGON2_MEMORY_KIB=47104`,
    `time_cost=1`, `parallelism=1`, `KEY_LEN=32`, `SALT_LEN=16`); `derive_key`
    (`crypto.py:41-61`) → 32-byte raw key; `new_params` mints a fresh random salt.
  - **KDF-params sidecar** — a **plaintext JSON** record
    (`format_version, memory_kib, time_cost, parallelism, key_len, salt_len,
    salt_hex`; `crypto.py:28-38`), written by `Vault._write_sidecar`
    (`vault.py:172-186`), read+validated by `load_and_validate_params`
    (`crypto.py:94-123`). The salt is **public by design** (a KDF salt is not a
    secret), so a plaintext params record beside an AES-256 DB is the app's
    existing, accepted shape — the backup reuses it verbatim.
  - `Vault._connect` (`vault.py:141-170`) — `PRAGMA key = "x'<64-hex>'"` raw key +
    `cipher_use_hmac=ON`; `Vault.open` (`:113-134`) forces a page-1 decrypt/HMAC
    read (wrong key → `DatabaseError`) then runs migrations.
  - `AuthService.complete_first_run` (`auth.py:194-217`) creates the vault via
    `Vault.create(key, params, base_currency, exponent)`; keys live in wipeable
    `bytearray`s, `_wipe`d on **every** path incl. failure (`auth.py:252-257`).
- **Password floors:** the PDF-export password floor is `MIN_EXPORT_PASSWORD_LEN
  = 8` (`ui/export_dialog.py:50`); the **master** password has **no minimum** —
  `AuthService` rejects only an empty password (`auth.py:169`, `len == 0`). So the
  **backup password** mirrors the export floor (**8**), and the **new master
  password** set during restore follows the **existing first-run policy**
  (non-empty) — this spec does **not** change master-password policy (stay in lane).
- **Restore is a pre-login surface:** the shell shows an `UnlockDialog` when a
  vault exists (re-opened by `MainWindow._lock`, `main_window.py:504-514`) and the
  first-run wizard when `Vault.presence_state()` is `"first_run"`. Restore must
  hang off **both** — a forgotten-password user cannot reach Settings.
- **Version/schema:** `Vault.open` runs migrations to the app's current
  `user_version`; the backup records that number so an older app refuses a newer
  backup (INV-6). *(The exact `user_version` constant is wired at implementation —
  cited, not hard-coded here.)*

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **The `.fbk` is a portable, self-contained, encrypted file with no plaintext financial data.** It is a **zip** of three entries: `manifest.json` (`format_version`, `app_version`, `schema_version` — plaintext), `params.json` (the KDF sidecar — plaintext salt+params, reused verbatim from `crypto.py`), and `vault.db` (a **SQLCipher AES-256** database re-keyed to `Argon2id(backup password)`). The only plaintext is the public salt/params + version integers — the same exposure the live vault's sidecar already accepts. Test: export → the three entries exist; the raw `.fbk` bytes do **not** contain a known seeded plaintext value (e.g. an account name). | ADR-0003 |
| INV-2 | **Round-trip fidelity.** Restoring an exported backup reproduces every table's rows exactly. Test: seed a vault, export, restore into an empty location, assert every application table's rows equal the source (accounts, transactions, categories, rules, settings, statement passwords, …). | data integrity |
| INV-3 | **Separate backup password; recovery without the old master password.** The `.fbk` is keyed by `Argon2id(backup password)`, independent of the master key. Restore consumes only the **backup password** + a **new** master password and **never** the old master password. Test: export under backup password `B`; restore with `B` + a new master `M2` (old master never supplied) → the vault opens under `M2` and the old master fails. | ADR-0003 (the mitigation) |
| INV-4 | **Fail-closed on wrong password / corrupt / truncated / incomplete file.** A wrong backup password (page-1 HMAC fail), a non-zip or truncated `.fbk`, or a missing entry aborts restore with a clear message and **changes nothing on disk** — no vault is created, replaced, or moved. Test: wrong password → error, on-disk vault state unchanged; a truncated `.fbk` → error, unchanged. | robustness |
| INV-5 | **Overwrite safety — never destroy an existing vault in place.** When a vault already exists, restore moves the current vault **and its sidecar** aside to a timestamped `*.old` copy **before** installing the restored vault; if a later step fails, the old vault remains recoverable. Test: with an existing vault, restore → the old DB+sidecar exist as `*.old` and the new vault is active; an injected mid-restore failure leaves the old vault in place. | data integrity |
| INV-6 | **Version guard — no newer-into-older.** `manifest.schema_version` above the running app's latest `user_version` is **refused** before any disk change ("backup from a newer version"); an older-or-equal backup is migrated up on open. Test: `schema_version = current+1` → refused, no disk change; an older backup → migrated + restored. | migration safety |
| INV-7 | **Atomic writes + key hygiene.** Export writes the `.fbk` to a temp path then `os.replace`; restore assembles the new vault in temp then atomically installs it. Both the backup-derived key and the new-master key live in wipeable `bytearray`s and are `_wipe`d on **every** path (success and failure), matching `AuthService`. Test: an injected failure mid-export leaves no partial `.fbk`; a key buffer is asserted wiped after export and after a failed restore. | ADR-0003; `AuthService` INV-3 |
| INV-8 | **Restore is reachable pre-login; export requires an unlocked vault.** A "Restore from backup" entry point exists on **both** the first-run welcome (no vault) and the `UnlockDialog` (vault exists but locked); "Export backup" exists only in Settings (unlocked). Test: the first-run and unlock surfaces expose a restore trigger; Settings exposes export. | ADR-0003 |
| INV-9 | **i18n / geometry.** Every new label/message goes through `tr()` / `translate` with a fixed context; no fixed-geometry call (covered by the FIBR-0051 `test_INV10_no_fixed_geometry_in_new_ui` glob). | `coding.md §5.2`; FIBR-0051 INV-10 |

## Design decisions

- **D1 — `.fbk` = a stdlib `zip` of `manifest.json` + `params.json` + `vault.db`.**
  Reuse the vault's existing shapes rather than invent a crypto container: the
  encrypted DB is a SQLCipher file (verbatim), the KDF params are the existing
  sidecar JSON (verbatim), and `zipfile` (stdlib) frames them into one portable,
  inspectable file. The salt lives in plaintext `params.json` because a KDF salt is
  not secret — exactly the vault's own sidecar model. No bespoke framing, no new
  runtime dependency.
- **D2 — Re-key via SQLCipher `sqlcipher_export()`.** Produce the backup DB by
  `ATTACH DATABASE '<tmp>' AS backup KEY "x'<backup raw hex>'"; SELECT
  sqlcipher_export('backup'); DETACH` against the already-open, master-keyed
  connection — reusing SQLCipher's own copy path over a hand-rolled schema+row
  walk. **Verify-empirically (rule 13):** a throwaway spike proves
  `sqlcipher_export` **and** `PRAGMA rekey` behave on this build's `sqlcipher3`
  binding **before** any production code (mirrors the FIBR-0013 Qt spike). Fallback
  if the binding lacks `sqlcipher_export`: create a fresh backup-keyed DB and copy
  schema + rows explicitly. The spike's outcome is recorded in the journal.
- **D3 — Separate backup password, same KDF, fresh salt.** `Argon2id` with the
  pinned params and a **fresh random salt per backup** (`new_params`); backup
  password floor **8** (mirrors `MIN_EXPORT_PASSWORD_LEN` — define
  `MIN_BACKUP_PASSWORD_LEN = 8`, or share the constant). The **new master
  password** set at restore follows the unchanged first-run policy (non-empty).
- **D4 — Restore rekeys to the new master, then installs atomically.** Open the
  backup DB with the backup key → migrate up → `PRAGMA rekey` to the new master key
  → write a fresh master sidecar (new salt) → move any existing vault aside (INV-5)
  → atomically place the restored DB+sidecar at the vault path → the shell unlocks
  under the new master password. A failure before the final move leaves the old
  vault intact.
- **D5 — A new `services/backup.py` `BackupService` owns export/restore.** It
  collaborates with the existing `crypto.py` + `Vault` (adding a small
  `Vault.export_to(path, key)` / rekey helper as reuse) and keeps `AuthService`
  focused. UI is thin: an Export control in Settings and a Restore
  dialog reachable from the two pre-login surfaces.
- **D6 — Version guard from the manifest.** Compare `manifest.schema_version`
  against the app's latest `user_version` before touching disk (INV-6);
  `format_version` guards the container shape itself.

## Deliverables

1. **`src/finbreak/services/backup.py`** — `BackupService` with
   `export_backup(dest: Path, backup_password: str)` and
   `restore_backup(src: Path, backup_password: str, new_master_password: str)`;
   the `.fbk` zip read/write, the version guard, the overwrite-aside, and key
   hygiene (wipeable buffers on every path).
2. **`src/finbreak/crypto.py` / `vault.py`** — a re-key / export helper reused by
   `BackupService` (`sqlcipher_export` + `PRAGMA rekey`), plus a
   `MIN_BACKUP_PASSWORD_LEN` (or reuse of the export constant).
3. **`src/finbreak/ui/settings.py`** — an "Export backup…" button opening a small
   dialog (backup password + confirm ≥ 8, then a native save dialog), under the
   existing `VaultLockedError` guard.
4. **`src/finbreak/ui/backup_restore.py`** (new dialog) — file pick + backup
   password + new master password ×2; surfaced from the first-run welcome and the
   `UnlockDialog` (a "Forgot password? Restore from backup" affordance that warns
   it replaces the current vault).
5. **`src/finbreak/ui/main_window.py` / first-run / unlock wiring** — reach the
   restore dialog pre-login (INV-8); on success, enter the unlocked shell under the
   new master password.
6. **A throwaway spike** proving `sqlcipher_export` + `PRAGMA rekey` on this build
   (D2) — run and recorded before production code, not shipped.
7. **Tests** — round-trip (INV-2), separate-password recovery (INV-3), wrong
   password / corrupt / truncated fail-closed (INV-4), overwrite-aside + mid-fail
   (INV-5), version guard (INV-6), atomic export + key-wipe (INV-7), pre-login
   reachability (INV-8), no-plaintext-in-file (INV-1). The FIBR-0051 geometry glob
   covers INV-9.
8. **Docs** — a CHANGELOG **Added** entry ("Encrypted backup export/restore"); the
   journal at close (incl. the D2 spike outcome). *(An ADR for the `.fbk` format —
   container + separate-passphrase decision — may be warranted; flagged for the
   cold-eyes/implementation call, not on this spec's critical path.)*

## Out of scope

- **Backup-staleness nudge** (FIBR-0018), **scheduled/automatic backups**, and
  **restore-integrity verification beyond open-and-HMAC-check** (FIBR-0033).
- **Cloud sync / off-device transport** — the user places the `.fbk` themselves.
- **Non-encrypted / portable-data export** (CSV is FIBR-0093).
- **Changing master-password policy** (min length, strength meter) — unchanged.
- **Theme, PDF-password management, language switcher** — the split-out
  FIBR-0127 / 0128 / 0129.

## Exit criteria

- `./scripts/ci-local.sh` green (ruff/format/bandit/pip-audit/gitleaks + mypy 0 +
  `pytest -m "not perf"`).
- Settings exports a portable encrypted `.fbk`; the unlock **and** first-run
  screens restore it with the backup password + a new master password, recovering
  all data without the old master password; a wrong password, a corrupt/truncated
  file, and a newer-version backup are each refused cleanly with the on-disk vault
  unchanged; an existing vault is moved aside (not destroyed) on restore.
