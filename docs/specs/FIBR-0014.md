## The problem (ADR-0003 + security-model.md T11)

ADR-0003 accepts, as a deliberate consequence, that **a forgotten master password
means unrecoverable data** — and names *"an explicit encrypted backup-export
feature"* as the mitigation. `security-model.md` **T11** currently states the
backup mitigates *loss/corruption, not a forgotten password* — true **only** when
the backup is locked by that same master password. This feature keys the backup by
a **separate backup password** the user keeps safe; restoring then needs only the
backup password plus a **new** master password — **never** the forgotten one — so
the backup *does* recover a forgotten master password. **T11 is updated to say so**
(Deliverable 9); this spec and T11 must not contradict.

## Threat model note — the `.fbk` is an off-device asset

The live vault never leaves the trusted machine (ADR-0003 / security-model.md §2:
"the machine is trusted"). The `.fbk` is **designed to leave** it — USB, cloud,
email — so it faces a **different** exposure: an attacker who *has the file* and
attacks the password offline. Consequences carried through this spec: (a) the
backup DB is AES-256 with the **HMAC on** (tamper-evident, INV-4); (b) the backup's
KDF params are **re-validated against the pinned Argon2 floor on restore** so a
tampered `params.json` can't force a weak KDF (INV-6b, T9); (c) export shows a
**strong-password warning**; (d) **no plaintext ever touches disk** during export
(`temp_store=MEMORY`, INV-1b). The backup-password floor stays **8** (matching the
PDF-export floor, the user's chosen policy); raising the per-backup Argon2 cost
above the vault's is a considered future hardening, **deferred** (it would fork the
KDF params from the reused vault set — noted, not done here).

## Verified current state (2026-07-13)

- **No backup code exists** — no export/import of the vault, no `.fbk`, no
  SQLCipher backup/rekey call. From-scratch feature. `sqlcipher_export` /
  `PRAGMA rekey` appear **nowhere** in `src/` (grep-confirmed) — hence the D2 spike.
- **Schema versioning is a table, not `PRAGMA user_version`:**
  `run_migrations` reads `SELECT version FROM schema_version` (`migrations.py:56`)
  and migrates up to the constant `LATEST_SCHEMA_VERSION = 8` (`migrations.py:24`);
  a `current > LATEST_SCHEMA_VERSION` vault is **refused** (`migrations.py:57-60`,
  surfaced as `SchemaVersionError` on open). The SQLite `PRAGMA user_version` is
  **never** used. The version guard here uses **that table + constant** (INV-6).
- **The crypto/vault path to reuse** (Explore-verified):
  - `crypto.py:18-24` — pinned Argon2id params (`ARGON2_MEMORY_KIB=47104`,
    `time_cost=1`, `parallelism=1`, `KEY_LEN=32`, `SALT_LEN=16`); `derive_key(
    password: bytearray, salt: bytes, params) -> bytearray` (`crypto.py:41`);
    `validate_params` (`crypto.py:64`) enforces the memory **floor** + exact
    lengths; `load_and_validate_params(sidecar_path) -> KdfParams`
    (`crypto.py:94-123`) reads+validates the plaintext sidecar (refusing
    downgrade/tamper — the guard the backup reuses on `params.json`, INV-6b).
  - **KDF-params sidecar** — a plaintext JSON record (`format_version, memory_kib,
    time_cost, parallelism, key_len, salt_len, salt_hex`; `crypto.py:28-38`),
    written by `Vault._write_sidecar` (`vault.py:172-186`). The salt is **public by
    design** (a KDF salt is not secret), so a plaintext params record beside an
    AES-256 DB is the app's existing, accepted shape — the backup reuses it verbatim.
  - `Vault._connect` (`vault.py:141-170`) — `PRAGMA key = "x'<64-hex>'"` raw key
    (`:152`) + `cipher_use_hmac=ON` (`:161`); `Vault.open` (`:113-134`) forces a
    page-1 decrypt/HMAC read (wrong key → `DatabaseError`) then runs migrations.
  - `AuthService.complete_first_run` (`auth.py:194-217`) creates the vault via
    `Vault.create(key, params, base_currency, exponent)`; keys live in wipeable
    `bytearray`s, `_wipe`d (`auth.py:120`) on **every** path incl. the create
    failure branch (`auth.py:210-212`, `except Exception: _wipe(key)`). Argon2 runs
    **off the main thread** (`AuthService.derive_raw`, `auth.py:126`). `new_params()`
    mints a fresh random salt (`AuthService.new_params`, `auth.py:177-186`).
- **Password floors:** the PDF-export floor is `MIN_EXPORT_PASSWORD_LEN = 8`
  (`ui/export_dialog.py:50`, a **UI** module); the **master** password has **no
  minimum** — `AuthService` rejects only empty (`auth.py:169`, `len == 0`). The
  backup floor mirrors export (**8**) via a **new `MIN_BACKUP_PASSWORD_LEN = 8` in a
  non-UI module** (`services/backup.py`), **not** an `import` from `ui/` (which
  would be an upward layer dependency). The **new master** password set at restore
  follows the existing first-run policy (non-empty) — this spec does **not** change
  master-password policy.
- **Restore is a pre-login surface:** the shell shows `UnlockDialog`
  (`ui/unlock.py:36`) when a vault exists (re-opened by `MainWindow._lock`,
  `main_window.py:500-514`) and the first-run wizard when `Vault.presence_state()`
  is `"first_run"` (`vault.py:39-46`). The vault + sidecar file names are
  `vault.db` / `vault.kdf.json` in `QStandardPaths.AppDataLocation`
  (`paths.py:17-18,26`). Restore must hang off **both** pre-login surfaces.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **The `.fbk` is a portable, self-contained, encrypted file with no plaintext financial data.** A **zip** of three entries: `manifest.json` (`format_version`, `app_version`, `schema_version` — plaintext), `params.json` (the KDF sidecar — plaintext salt+params, reused verbatim), and `vault.db` (a **SQLCipher AES-256** DB re-keyed to `Argon2id(backup password)`, HMAC on). The only plaintext is the public salt/params + version integers — the vault's existing sidecar exposure. Test: export → the three entries exist; the raw `.fbk` bytes do **not** contain a distinctive seeded sentinel (a UUID-like account name written before export). | ADR-0003; sec-model INV-1 |
| INV-1b | **No plaintext spills to disk during export.** The export connection sets `PRAGMA temp_store=MEMORY` so `sqlcipher_export`/sorting never writes vault contents to a plaintext temp/journal file. Test (in the D2 spike + a unit leg): during an export, no plaintext temp/journal containing the sentinel appears in the temp dir. | sec-model INV-1 |
| INV-2 | **Round-trip fidelity — every table, exactly.** Restoring an exported backup reproduces every application table's rows. Test: seed a vault, export, restore into an empty location, and for **every** table in `sqlite_master WHERE type='table'` (enumerated dynamically, not a hand list) assert row-set equality with the source (including `schema_version`). | data integrity |
| INV-3 | **Separate backup password; recovery without the old master password.** The `.fbk` is keyed by `Argon2id(backup password, fresh salt)`, independent of the master key even if the two passwords coincide (distinct salts). Restore consumes only the **backup password** + a **new** master password and **never** the old master. Test: export under backup password `B`; restore with `B` + new master `M2` (old master never supplied) → the vault opens under `M2`; the old master fails. | ADR-0003 (the mitigation); sec-model T11 |
| INV-4 | **Fail-closed on wrong password / corrupt / truncated / incomplete / bad-`format_version` file.** A wrong backup password (page-1 HMAC fail), a non-zip / truncated / entry-missing `.fbk`, or an unknown/newer `format_version` aborts restore with a clear message and **changes nothing on disk**. Test: each of {wrong password, truncated file, `format_version` = current+1} → error, on-disk vault state byte-identical. | robustness |
| INV-5 | **Overwrite safety — never destroy an existing vault in place.** When a vault already exists, restore moves the current `vault.db` **and** `vault.kdf.json` aside to timestamped `*.old` copies **before** installing the restored pair; the old vault is always recoverable from the `*.old` copies. Test: with an existing vault, restore → the old DB+sidecar exist as `*.old` and the new vault is active; inject a failure **after** the move-aside → the vault is recoverable from the `*.old` pair (not silently lost). | data integrity |
| INV-6 | **Version guard — no newer-into-older.** Two checks: (a) an **early UX gate** on the plaintext `manifest.schema_version` — above `migrations.LATEST_SCHEMA_VERSION` → refuse **before any disk change** with a clear message; (b) the **authoritative** guard is `Vault.open`'s post-HMAC read of the embedded DB's `schema_version` table (`SchemaVersionError`), which an attacker-edited manifest cannot bypass. An older-or-equal backup is migrated up on open. Test: `manifest.schema_version = LATEST+1` → refused, no disk change; a DB whose `schema_version` table exceeds `LATEST` → refused on open even if the manifest lies. | migration safety |
| INV-6b | **Restore re-validates the backup's KDF params against the pinned floor.** Before deriving the backup key, restore runs `load_and_validate_params` on the backup's `params.json`; a params record below the Argon2 memory floor or with wrong lengths is **refused** (a tampered `.fbk` can't force a weak KDF). Test: an `.fbk` whose `params.json` sets `memory_kib` below the floor → refused, no key derived, no disk change. | sec-model INV-2; T9 |
| INV-7 | **Atomic writes + key hygiene.** Export writes the `.fbk` to a temp path then `os.replace` (temp beside the destination so the replace is same-filesystem; the temp is unlinked on any failure — no partial `.fbk`, no leftover temp). Restore assembles the new vault in temp then installs it. `BackupService` accepts passwords as `str` at the UI boundary, converts each to a `bytearray` at entry, and `_wipe`s **every** derived key **and** the password buffer in a `finally` on all paths (success and failure), matching `AuthService`. Test: an injected failure mid-export leaves no partial `.fbk` and no temp; a wipe seam (the buffer is returned/injected or a patch point) asserts the backup-key buffer is zeroed after export and after a failed restore. | ADR-0003; `AuthService` key hygiene |
| INV-8 | **Restore is reachable pre-login; export requires an unlocked vault.** A "Restore from backup" entry exists on **both** the first-run welcome (no vault) and the `UnlockDialog` (vault exists but locked); "Export backup" exists only in Settings (unlocked). Test: the first-run and unlock surfaces expose a restore trigger; Settings exposes export. | ADR-0003 |
| INV-9 | **Auto-lock is suspended during a backup op; the connection is single-threaded.** Export/restore suspend the inactivity auto-lock (`AuthService` timer) for their duration so a mid-op `lock()`/`close()` cannot race the owned connection or the re-key; the expensive Argon2 derivation runs off the main thread (as `derive_raw` does) but the owned SQLCipher connection is touched from one thread only. Test: an auto-lock timeout fired mid-export does not close the connection until the export completes; the vault is intact afterward. | data integrity; FIBR-0114 |
| INV-10 | **i18n / geometry.** Every new label/message goes through `tr()` / `translate` (fixed context); no fixed-geometry call (FIBR-0051 `test_INV10_no_fixed_geometry_in_new_ui` glob). New-`BackupService` log lines never record passwords, keys, or row contents (sec-model INV-9). | `coding.md §5.2`; FIBR-0051; sec-model INV-9 |

## Design decisions

- **D1 — `.fbk` = a stdlib `zip` of `manifest.json` + `params.json` + `vault.db`.**
  Reuse the vault's shapes rather than invent a crypto container: the encrypted DB
  is a SQLCipher file (verbatim), the KDF params are the existing sidecar JSON
  (verbatim), and `zipfile` (stdlib) frames them into one portable, inspectable
  file. The salt in plaintext `params.json` is the vault's own sidecar model. No
  bespoke framing, no new runtime dependency.
- **D2 — Re-key via SQLCipher `sqlcipher_export()`, HMAC-on, temp-in-memory.**
  Produce the backup DB by `ATTACH DATABASE '<tmp>' AS backup KEY "x'<backup raw
  hex>'"; PRAGMA backup.cipher_use_hmac=ON; SELECT sqlcipher_export('backup');
  DETACH` against the already-open, master-keyed connection, with
  `PRAGMA temp_store=MEMORY` set first (INV-1b). This copies **schema + all tables +
  rows** (including the `schema_version` table, so the version travels — INV-6).
  **Verify-empirically (rule 13):** a throwaway spike proves, on this build's
  pinned `sqlcipher3-binary`, that (i) `sqlcipher_export` exists and round-trips,
  (ii) `PRAGMA rekey` re-keys in place, (iii) the backup DB carries the
  `schema_version` **table**, (iv) HMAC-on makes a wrong key fail page-1, and (v)
  **no plaintext temp/journal** is written. The spike runs **before** any UI
  deliverable (its outcome may change the `Vault.export_to` shape). Fallback if the
  binding lacks `sqlcipher_export`: create a fresh backup-keyed DB and copy schema +
  rows explicitly. Spike outcome recorded in the journal.
- **D3 — Separate backup password, same KDF, fresh salt.** `Argon2id` with the
  pinned params and a **fresh random salt per backup** (`AuthService.new_params`);
  backup floor **8** via `MIN_BACKUP_PASSWORD_LEN` in `services/backup.py`
  (non-UI). Export shows a strong-password warning (off-device threat model). The
  **new master** password at restore follows the unchanged first-run policy
  (non-empty).
- **D4 — Restore rekeys to the new master, then installs atomically.** Open the
  backup DB with the backup key (after INV-6b param re-validation) → migrate up →
  `PRAGMA rekey` to the **new master** key → write a **fresh** master sidecar via
  `new_params()` (new salt; the backup's `params.json` opens the backup DB only,
  **never** becomes the restored vault's sidecar) → move any existing `vault.db` +
  `vault.kdf.json` aside to `*.old` (INV-5) → install the restored pair in a fixed
  order (**`vault.db` first, then `vault.kdf.json`**; a crash between the two leaves
  the mixed state `presence_state()` rejects — the operator recovers from the
  `*.old` pair, INV-5). The shell then unlocks under the new master password.
- **D5 — A new `services/backup.py` `BackupService` owns export/restore.** It
  collaborates with `crypto.py` + `Vault` (adding the reuse helpers in Deliverable
  2) and keeps `AuthService` focused. UI is thin: an Export control in Settings and
  a Restore dialog reachable from the two pre-login surfaces.
- **D6 — Version guard from the manifest (early) + the embedded table
  (authoritative).** `manifest.schema_version` vs `migrations.LATEST_SCHEMA_VERSION`
  is a pre-disk UX refuse (INV-6a); the authenticated `schema_version`-table check
  in `Vault.open` is the real guard (INV-6b/INV-6). `format_version` guards the
  container shape itself (INV-4).

## Data model — the `.fbk` container (a zip; no vault schema change)

| Entry | Plaintext? | Contents |
|-------|-----------|----------|
| `manifest.json` | yes | `format_version` (container shape), `app_version`, `schema_version` (early version gate — INV-6a) |
| `params.json` | yes (salt is public) | the KDF sidecar verbatim (`format_version, memory_kib, time_cost, parallelism, key_len, salt_len, salt_hex`); re-validated on restore (INV-6b) |
| `vault.db` | **no** (AES-256, HMAC on) | the vault re-keyed to `Argon2id(backup password)`; carries the `schema_version` table (INV-6b authoritative check) |

## Out of scope

- **Backup-staleness nudge** (**FIBR-0089**) — first-run copy + "last backup N days
  ago" reminder; and **restore-integrity verification** beyond open-and-HMAC-check
  (**FIBR-0033**, whose dependency re-points from FIBR-0018 to this item).
- **Scheduled / automatic backups**, cloud sync / off-device transport — the user
  places the `.fbk` themselves.
- **Non-encrypted / portable-data export** (CSV is FIBR-0093).
- **Changing master-password policy** (min length, strength meter) and **raising the
  per-backup Argon2 cost** above the vault's — considered, deferred (threat-model note).
- **Theme, PDF-password management, language switcher** — split-out FIBR-0127/0128/0129.

## Exit criteria

- `./scripts/ci-local.sh` green (ruff/format/bandit/pip-audit/gitleaks + mypy 0 +
  `pytest -m "not perf"`).
- Settings exports a portable encrypted `.fbk` (no plaintext financial data in the
  file or in any temp file); the unlock **and** first-run screens restore it with
  the backup password + a new master password, recovering all data (all tables)
  without the old master password; a wrong password, a corrupt/truncated file, a
  below-floor `params.json`, and a newer-version backup are each refused cleanly
  with the on-disk vault unchanged; an existing vault is moved aside (not destroyed)
  on restore; the spec and `security-model.md` T11 agree.
