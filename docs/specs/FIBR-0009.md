**Theme:** P05 built the CSV importer and the whole write pipeline; P06 added OFX
as a second importer that *skips* mapping (self-describing). PDF is the **third**
importer — and the hard one. A CSV is a tidy grid and an OFX file is a
self-describing standard; **a PDF statement is a printed page.** There is no
universal way to know which text on the page is a transaction. This phase makes
the **tractable** version of the problem: many banks print their transactions as
a real **ruled table**, and `pdfplumber` lifts such a table off the page as rows
of cells. So the design is: **extract the transaction table, serialise it to CSV
text, and feed the existing CSV pipeline verbatim** — a PDF is imported as "a CSV
we happened to lift off a page." The genuinely new surface is small: the
extractor, the in-memory decrypt of locked PDFs, an opt-in remembered password
(one new nullable column, a v5 migration), and a wizard branch. **Free-text
(non-table) statements are explicitly out of scope** (a later phase; § Out of
scope).

## Context

The app is layered — **UI → Services → Repositories → SQLCipher** (design.md) —
and its headline flow is **import → categorise → insight**. FIBR-0007 built the
first importer (CSV) and the entire write path; FIBR-0008 added OFX. This phase
adds the **third** importer. Because a PDF's transaction table is
**columns-that-need-mapping** (like CSV) rather than self-describing (like OFX),
PDF reuses the **CSV** half of the pipeline — and it does so by turning the
extracted table back into CSV text (D1), so the reuse is total.

This phase **reuses, unchanged**, the machinery FIBR-0007 shipped:

- **`CsvImporter.parse(text, mapping, exponent)`** — the pure CSV parser. The PDF
  path serialises its chosen table to CSV text and calls this **verbatim** (D1),
  so a PDF amount obeys the identical money contract as a manually-typed one.
- **`ImportService.preview(text, mapping, account_id)`**,
  **`match_profile(header)`**, **`save_profile(name, header, mapping)`**,
  **`signature_for(header)`** — the CSV mapping-profile machinery. A PDF table's
  header feeds these **unchanged**, so a PDF bank layout is remembered per-bank
  exactly like a CSV one (D9), for free.
- **`ImportService.commit_import(preview, period_start, period_end,
  source_filename)`**, the **multiset-delta dedup** (`_dedup`), and the
  `statement_periods` span-dedup — **not touched**; the PDF path produces the same
  `ImportPreview` and calls the same `commit_import`.
- **`parse_transaction`** + **`read_minor_unit_exponent(conn)`**
  (`services/transactions.py`), **`TransactionDraft`** (`models.py`), and the
  **`ParseResult` / `RowError`** value objects (`importers/base.py`).
- **The import wizard** (`ImportWizardWidget`) — the PDF path is a **branch**
  inside `_select_file` (beside the CSV and OFX branches), converging onto the
  existing map/preview steps, not a new screen.
- **`pikepdf`** — already a pinned runtime dependency (ADR-0004, the PDF-export
  encryption layer). This phase exercises its **decryption** half; no new crypto
  dependency for the locked-PDF path.

So the genuinely new code is: the `PdfImporter` extractor, the in-memory decrypt
helper, the v5 migration + two repository accessor methods, the password dialog,
the wizard PDF branch, and one pinned dependency (`pdfplumber`).

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` gates them before code. **D1–D4,
D7–D11, D13, D14 are engineering calls; D5, D6, D12 carry a security/scope
decision** (remembered-credential storage model, credential hygiene, and the
native-bundle surface).

- **D1 — Extract-then-CSV-adapter (the core reuse decision).** The `PdfImporter`
  is an **extractor**, not a second parser: it lifts a table with `pdfplumber`
  and serialises it to CSV **text**, which the *existing* `CsvImporter` /
  `ImportService.preview` consume unchanged. Rejected alternative: a parallel
  `PdfImporter.parse(rows, mapping, exponent)` mirroring `CsvImporter.parse` — it
  would duplicate the ragged-row guard, the date/amount parse, the `RowError`
  collection, and the span logic. Per coding.md § 1.3 + the Rule of Three, at the
  **second** table-shaped importer the right move is to reuse the CSV parser via a
  thin text adapter, not to fork it. Six-month legibility: "a PDF is imported as a
  CSV lifted off a page" is one sentence a future reader holds in their head.
- **D2 — `pikepdf`/qpdf is the decryptor, not `pdfminer`'s `password=`.**
  Empirically **both** decrypt AES-256/R6 (see § Verified API basis:
  `pdfplumber.open(BytesIO, password=…)` succeeded). We deliberately route
  decryption through **`pikepdf` (qpdf)** anyway: qpdf is the industry-standard
  PDF decryptor and handles the widest range of real-bank encryption variants far
  more robustly than `pdfminer`'s crypto, it is **already** a bundled dependency
  (ADR-0004), and the roadmap names it explicitly. `pdfplumber` then only ever
  sees **already-decrypted** bytes — it is never handed a password. Both paths
  keep bytes in memory (INV-2). Routing through `pikepdf` also yields a **clean
  `PasswordError`** to gate the prompt on (D3) — not just qpdf's decrypt
  robustness.
- **D3 — Uniform in-memory normalise through `pikepdf`; prompt only on a real
  user password.** The extractor **always** opens the source bytes with
  `pikepdf.open(BytesIO(raw)[, password])` and re-saves them to an in-memory
  `BytesIO`, then hands **those** plaintext bytes to `pdfplumber`. This one path
  covers all three cases: a **plaintext** PDF (opens, re-saves), an **owner-only**
  encrypted PDF (opens **without** a password; `pikepdf.save` strips the
  encryption), and a **user-password** PDF (`pikepdf.open` raises
  `pikepdf.PasswordError` → the wizard supplies a password and retries).
  **Why gate on `pikepdf`, not `pdfplumber` directly?** `pdfplumber` *can* read
  plaintext and even owner-only PDFs unaided (verified — § Verified API basis), so
  the routing is **not required for correctness**; it is chosen for three real
  reasons: (1) a **clean detection signal** — `pikepdf.open` without a password
  raises the precise `pikepdf.PasswordError` on a user-password PDF, whereas
  `pdfplumber` raises a **broad `PdfminerException`** (verified) that cannot
  cleanly distinguish "needs a password" from "corrupt file", so a
  `pdfplumber`-only prompt trigger is unreliable; (2) **qpdf robustness** — qpdf
  decrypts a wider range of real-bank encryption than `pdfminer`'s crypto (D2);
  and (3) a **single code path** (no branch on encryption type). A **rejected
  alternative** — a separate `is_encrypted` probe deciding whether to decrypt —
  is discarded because a name like `is_encrypted` collides with `pikepdf`'s own
  attribute (`True` for owner-only too), inviting a spurious prompt; the uniform
  rule has no such trap: **prompt iff `pikepdf.PasswordError`**. The one extra
  `pikepdf.save` on an already-plaintext PDF is cheap (bounded by INV-9).
- **D4 — In-memory decrypt is a tested invariant, not a comment.** The extractor
  takes `bytes` and returns tables/bytes via `BytesIO` only. INV-2 is tested
  **three** ways: **(a)** a positive test that an encrypted fixture
  decrypts+extracts purely from bytes; **(b)** a **runtime filesystem sentinel** —
  redirect temp writes to a monitored scratch dir via **both**
  `monkeypatch.setenv("TMPDIR", <scratch>)` (for a C-level write) **and**
  `monkeypatch.setattr(tempfile, "tempdir", <scratch>)` (the env var alone is
  ignored once `tempfile.gettempdir()` has cached — e.g. after any prior `tmp_path`
  fixture — so Python's module attribute must be set too), then snapshot that
  scratch dir **and** the CWD around the decrypt+extract of an encrypted fixture,
  asserting **no new file appeared**. The operation is genuinely disk-free
  (verified), so this catches any stray Python- or C-level temp write however
  spelled, including one buried inside `pikepdf`/`pdfplumber`; and **(c)** a
  source-token guard that the module contains no
  `tempfile` / `NamedTemporaryFile` / `.write_bytes(` / `os.write` /
  `open(…, "w"/"wb")` disk-write token. **(b) is the load-bearing test**; (c) is a
  cheap necessary-not-sufficient backstop (a token allowlist alone is gameable).
- **D5 — Remembered password: stored plaintext inside the encrypted vault, no
  second layer.** The vault is SQLCipher (AES-256, keyed by the Argon2id-derived
  master key). A value stored in a vault table **is** encrypted at rest by that
  key — so "store it encrypted in the vault" (roadmap) is satisfied by a plain
  column value; adding a second, app-level encryption of the password on top would
  be redundant ceremony (over-engineering — coding.md § 1.1, shortest correct implementation) with no threat it
  answers (anyone who can read the column has already unlocked the vault, i.e.
  has the master password — the same trust boundary as every transaction). The
  remembered password is therefore protected **exactly as strongly as the
  financial data itself**, and only when the user opts in.
- **D6 — Credential hygiene: the password column is kept off the `Account`
  object.** The v5 column is read/written by **dedicated** repository methods
  (`get_pdf_password(account_id)` / `set_pdf_password(account_id, value|None)`),
  and is **not** added to the `Account` dataclass nor its main `SELECT`. So a
  remembered credential never rides along with the account listings that populate
  combos, tables, or a log line — it is fetched only at the moment a PDF import
  needs it (INV-11). (This also keeps the `Account(*row)` positional-construction
  contract and every account call-site unchanged — INV-8.)
- **D7 — Multi-table chooser lives on the map step, default = largest table.**
  When >1 candidate table is extracted (D8), a chooser (a `QComboBox`, shown only
  for >1 — the same "visible iff >1" pattern the OFX statement chooser uses,
  though **that** one sits on the *preview* step, not the map step) sits above the
  column combos on the **map** step. Selecting a table does three things: (i)
  re-serialises **that** table (`table_to_text`) into the wizard's `self._text` /
  `self._header` that the reused `preview` consumes — so Preview always imports the
  **chosen** table, not the default; (ii) re-runs `match_profile` on the new
  header (each candidate has its own signature) to re-fill or clear the mapping
  combos; (iii) repopulates the combo *items* from that header. The default
  selection is the table with the most **data** rows (the transactions table,
  empirically, dwarfs a summary/header table), **ties broken by the page of the
  group's first member, then table order**. Because the user still maps columns, a
  wrong table pick is fully recoverable. No auto-commit on a guessed table.
  **Chooser hygiene (both directions):** the PDF table chooser **and** the OFX
  statement chooser (`_ofx_statement_combo`, which lives on the *preview* step) are
  cleared + hidden at the **top of `_select_file`, before the format dispatch** —
  so **every** pick (CSV / OFX / PDF) starts from a clean chooser state and no
  prior pick's chooser lingers on the shared map/preview steps. This **hoists** the
  currently CSV-branch-only OFX reset (`import_wizard.py:237-239`) ahead of the
  `_looks_like_ofx` / `_looks_like_pdf` dispatch, closing **both** leak directions
  (OFX→PDF and PDF→CSV/OFX), which a one-sided reset would miss.
- **D8 — Tables grouped across pages by identical header.** `extract_tables()` is
  per-page, but a statement's transaction table routinely continues across pages
  under a **repeated header row**. After extraction, tables whose **header row**
  (row 0, whitespace-folded) is identical are **grouped** into one logical
  candidate table — the first group member keeps its header; subsequent members
  contribute their data rows with any repeated header row dropped. So a 3-page
  statement yields **one** transactions candidate, not three. A page's tables with
  a unique header stay separate candidates. A continuation page whose header is
  **not** repeated is **not** stitched to the prior page's table (each stays a
  separate candidate — § Out of scope); the repeated-header case above is the one
  this phase handles.
- **D9 — A PDF layout is remembered per-bank via the existing profile machinery.**
  The serialised table's header row is a `list[str]` — exactly what
  `signature_for` / `match_profile` / `save_profile` key on. So a saved PDF
  mapping profile works identically to a CSV one, keyed by the PDF table's header
  signature; a second import from the same bank auto-applies it (FIBR-0007
  INV-10a, reused). No new profile concept.
- **D10 — Generalise `read_file_bytes` to a format-neutral size-capped read.**
  FIBR-0008's `read_file_bytes` + `_MAX_OFX_BYTES` are OFX-named but format-generic
  (a size-capped byte read). Rename `_MAX_OFX_BYTES` → `_MAX_IMPORT_BYTES`
  (`import_.py:42/129`), reword `read_file_bytes`'s docstring + its "too large to
  import as OFX" message to be format-neutral, update the `_MAX_OFX_BYTES` mention
  in `ofx_importer.py:37`, and the OFX test's `monkeypatch.setattr(import_mod,
  "_MAX_OFX_BYTES", …)` (`test_ofx_import.py:589`), and the doc reference in
  `tests/features/ofx_import/spec.md:28` — so OFX **and** PDF share one capped read (coding.md § 1.3 **clause 2** — refactor to cover the new case; the
  existing OFX call-site benefits, its behaviour unchanged). The OFX import suite
  must stay green over the rename.
- **D11 — The password prompt is a small custom `QDialog`.** A masked
  (`QLineEdit.Password`) field + a "Remember this password for <account>"
  `QCheckBox` (unchecked by default) + OK/Cancel. Re-shown on a wrong password
  (INV-3); Cancel abandons the import cleanly. (A bare `QInputDialog.getText`
  cannot carry the remember checkbox, so a purpose-built dialog is warranted.)
- **D12 — `pdfplumber` pinned at 0.11.10; its native transitive surface is a
  bundle obligation.** `pdfplumber==0.11.10` (latest, verified) pulls
  `pdfminer.six` (→ `cryptography` (Rust, native) + `charset-normalizer`),
  `Pillow` (native), and `pypdfium2` (whose native PDFium binary `libpdfium.so`
  ships in the **separate top-level `pypdfium2_raw` package** installed by the
  same `pypdfium2` wheel — verified). This is the heaviest single dependency
  addition in the project and, like FIBR-0008's `lxml`, it is a
  **vendored-native-lib blind spot** (`dependencies.md § 2`) that must travel into
  the FIBR-0003 self-contained bundle. The FIBR-0003 build smoke-test is a **hard
  DoD item** for this phase, and the build script must `--collect-all
  pypdfium2_raw` (**the package that actually holds `libpdfium.so`** — a plain
  `--collect-all pypdfium2` would miss it), plus `--collect-all pypdfium2` and
  `--collect-all PIL` (their native payloads are data/binaries PyInstaller does
  not follow by `import` alone). Pinned per dependencies.md (latest-version
  policy).
- **D13 — Blank / duplicate header cells are made unique before mapping.** An
  extracted table's header row (row 0) can contain blank cells (`""`/`None`) or
  repeated labels — and `signature_for` / `match_profile` (reused unchanged, D9)
  **reject** a duplicate-named header with a `ValueError`
  (`services/import_.py:48-49`). So before serialising (D1), the extractor
  rewrites the header cells to a **deterministic, unique** form: a blank or
  would-be-duplicate cell becomes a positional placeholder (`"Column {n}"`,
  1-based over the table's columns); of two identical labels the **first** keeps
  its label and later ones become placeholders. The rewrite then **guarantees**
  the final header set is unique: each generated `"Column {n}"` is checked against
  **all other** final header cells (not just already-emitted ones — so a
  placeholder colliding with an untouched real `"Column 1"` label is caught), and
  on any collision `_2`, `_3`, … is appended until unique. Already-unique non-blank
  labels are otherwise left untouched. This
  keeps the CSV signature/profile machinery reused **verbatim**, gives the user
  mappable column names (they map by position anyway), and means a real bank PDF
  never trips the CSV-framed duplicate-header `ValueError`. The rewrite is
  header-only; data cells are untouched.
- **D14 — Extraction is synchronous inline, like the OFX/CSV parse.** The
  `pikepdf` normalise + `pdfplumber` extraction runs **inline** in the wizard's
  `_select_pdf` — exactly as the OFX parse runs inline (`import_wizard.py:281`) and
  the CSV read (`:241`) — **not** on a background thread. `ui/_worker.py`'s
  `DeriveWorker` is **KDF-specific**
  (`DeriveWorker(password: bytearray, params: KdfParams, parent=None)` →
  `derive_raw`) and is **not** a generic job runner, so it is **not** reused; no
  new worker is
  added. Extraction is bounded by INV-9's byte/page/row caps, so a pathological
  input is **refused** rather than left to hang the UI; a real statement (a few
  pages, well under the caps) extracts fast enough that a worker is unwarranted
  for the MVP (YAGNI). A background worker for extraction is **deferred**
  (§ Out of scope) if a real statement ever proves slow. No latency SLO beyond the
  INV-9 caps.

## Invariants (testable)

The table under *At a glance* is the canonical list (INV-1..INV-11). Sub-invariants:

- **INV-1a** (header uniquification, D13) — a candidate table whose header row has
  **blank** cells, **duplicate** labels, **and** a literal `"Column 1"`-style label
  serialises to a **collision-free unique** header, so the reused `signature_for` /
  `match_profile` never raise their duplicate-header `ValueError`
  (`services/import_.py:48-49`) on a real bank PDF. First occurrence of a duplicate
  keeps its label; blanks/later duplicates become re-suffixed placeholders until
  unique.
- **INV-7a** — Picking a `.pdf` routes to the PDF branch (`_looks_like_pdf`: a
  `.pdf` extension, else a bounded head-sniff — the first 512 bytes, `lstrip`ped,
  tested for the `%PDF-` magic (an ASCII literal — case-exact, unlike
  `_looks_like_ofx`'s `.upper()`d OFX sniff) — matching `_looks_like_ofx`'s bounded-sniff shape
  (`import_wizard.py:268`, which reads the same 512-byte head); never the whole
  file, so the size cap cannot be bypassed by the sniff).
- **INV-7b** — An **encrypted** PDF with **no** stored password shows the password
  dialog; the correct password proceeds; a **wrong** password re-shows the dialog
  (INV-3) and does not crash or advance.
- **INV-7c** — A **>1-table** PDF shows the table chooser defaulted to the largest
  table; switching tables repopulates the mapping combos (INV-6/D7).
- **INV-7d** — After extraction the PDF path lands on the **map step** (unlike OFX,
  which skips it). An exact-signature profile match auto-fills the mapping combos
  and, for a **single** candidate table, jumps to preview (FIBR-0007 INV-10a
  reused). **With >1 candidate table the map step is always shown** (chooser
  visible, INV-6) even on a profile match — the profile pre-fills the combos but
  the user confirms which table; no table is ever auto-committed (upholds INV-6's
  "no table silently assumed").
- **INV-7e** — **Import** inserts exactly the new rows + records the period; a
  second import of the same statement previews all-duplicate and inserts zero
  (INV-10).
- **INV-7f** — With "remember" checked, the password is stored (this spec's INV-4)
  and a subsequent PDF import into that account **auto-applies** it with **no** dialog;
  a stored password that no longer works falls back to the prompt (INV-3).

The security invariants INV-2 (in-memory decrypt), INV-3 (wrong-password
re-prompt), INV-4 (opt-in remembered password), INV-9 (resource bounds +
inertness), and INV-11 (no-secret-logged) each carry a dedicated headless test
(no Qt needed) built on `pikepdf`-encrypted fixtures.

## Out of scope (deferred to named phases)

- **Free-text (non-table) PDF statements.** Statements printed as flowing lines
  with no ruled table are **not** importable in this phase (INV-5 shows a friendly
  "try CSV/OFX" message). A later phase could add per-bank line-pattern (regex)
  rules; explicitly **not** built now (YAGNI; the table path covers the common
  case and reuses everything).
- **Scanned / image-only PDFs (OCR).** No text layer → no extractable table; same
  friendly message. OCR is out of scope for a local, dependency-light app.
- **Multi-page transaction tables without a repeated header.** D8 stitches
  continuation pages **only** when the header row repeats. A statement whose
  transaction table flows onto later pages with **no** repeated header is not
  stitched (each page's table is a separate candidate); a later phase could add
  header-less continuation stitching.
- **A background thread for extraction.** Extraction is synchronous inline (D14),
  bounded by the INV-9 caps. If a real statement ever proves slow enough to bother
  the UI, moving extraction onto a background `QThread` is a later phase — the MVP
  does not add one (YAGNI), and `ui/_worker.py`'s KDF worker is not reused.
- **Automatic table detection tuning** (custom `table_settings` per bank). The MVP
  uses `pdfplumber`'s default line-based detection + the user's table choice (D7);
  per-bank extraction tuning is deferred.
- **PDF *export* password-locking** (the *encryption* half of ADR-0004) — that is
  **FIBR-0013** (P11 locked PDF export), a separate phase. This phase only
  **decrypts** input PDFs.

## Verified API basis

Empirically verified in the project venv before drafting (global rule § 13; the
probe built a synthetic ruled-table PDF with `reportlab`, extracted it, encrypted
it with `pikepdf` AES-256/R6, and round-tripped the decrypt):

- **Versions:** `pdfplumber 0.11.10`, `pikepdf 10.9.1`.
- **`pdfplumber.open(io.BytesIO(bytes))`** works (no path needed); `.pages` is the
  page list; **`page.extract_tables()`** returns `list[table]` where a table is
  `list[row]` and a row is `list[cell]` — cells are `str`, an empty cell is `""`
  (and can be `None` under other table settings — the adapter treats both as
  empty, D1). The synthetic header + 3 data rows extracted exactly, including a
  bank's separate **"Money Out" / "Money In"** columns (→ the CSV debit/credit
  mapping path). `.metadata` is a dict (`Author`/`Creator`/`CreationDate`/…).
- **`pikepdf.open(io.BytesIO(raw))` with no password on an encrypted file → raises
  `pikepdf.PasswordError`** (module `pikepdf._core`, exposed as
  `pikepdf.PasswordError`); **with `password="wrong"` → the same
  `pikepdf.PasswordError`** (INV-3's signal).
- **In-memory decrypt:** `pikepdf.open(io.BytesIO(enc), password="secret")` opens
  (`.is_encrypted` True); **`pdf.save(io.BytesIO())`** yields decrypted bytes that
  `pdfplumber` re-extracts — **no filesystem path involved** (INV-2/D4).
- **`pikepdf.open(...).is_encrypted`** is `False` for a plaintext PDF (D3).
- **Owner-only encryption** (empty user password, non-empty owner password),
  re-verified against the pinned stack:
  - `pdfplumber.open(io.BytesIO(raw))` **without** a password **does extract**
    owner-only PDFs — `len(pages) == 1`, real text, and a **gridded** table all
    return. (A table with **no** ruled lines yields zero detected tables
    *regardless of encryption* — `pdfplumber`'s default table detection is
    line-based; unrelated to the crypto. This artifact produced an earlier
    mistaken "cannot extract" reading — see loop log.)
  - `pikepdf.open(io.BytesIO(raw))` opens owner-only **without** a password
    (`.is_encrypted` True), and `pdf.save(io.BytesIO())` (no `encryption=`)
    **strips** it — `.is_encrypted` is `False` on the saved bytes and `pdfplumber`
    extracts. So D3's save-strip of owner-only is confirmed.
  - Net: routing owner-only through `pikepdf` is **not required for correctness**
    (`pdfplumber` reads it directly) — it is chosen for the uniform single path +
    the clean `PasswordError` detection (D3).
- **User-password detection signal:** `pdfplumber.open(io.BytesIO(user_pw_raw))`
  **without** a password raises `pdfplumber.utils.exceptions.PdfminerException` (a
  broad catch-all), whereas `pikepdf.open` raises the precise
  `pikepdf.PasswordError` — the reason the design gates the prompt on `pikepdf`
  (D3), not `pdfplumber`.
- Note (informational, not the chosen path): `pdfplumber.open(io.BytesIO(enc),
  password="secret")` **also** decrypts AES-256/R6 via `pdfminer.six`'s
  `cryptography` backend — confirming both routes work; D2 still selects `pikepdf`
  for robustness + the clean detection signal.
