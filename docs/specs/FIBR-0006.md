**Theme:** every transaction will eventually be classified by *where
the money went*. This phase lays down the classification tree — a
**self-referential `categories` table** whose top level is the **Type**
(Income / Expenditure), whose second level is the **Category** (Salary,
Bills, Medical, …), and whose (future) third level is the
**Sub-category** — plus a category-management screen exposing the first
two levels. The tree is seeded on first-run and on upgrade with the two
Types and a sensible set of default categories under each, so the app
is useful out of the box. The data model already supports a third
level (`parent_id` allows arbitrary depth), so adding Sub-categories
later needs **no schema migration**, only UI. This phase does **not**
link transactions to categories — that link, and the auto/manual
assignment that fills it, is owned by **P08 (FIBR-0010, rules engine +
manual override)** (D10).

## Context

FIBR-0005 (P03) shipped the account aggregate **and** the machinery
this phase reuses wholesale:

- **The forward-only migration runner** (`migrations.py`):
  `LATEST_SCHEMA_VERSION`, an ordered `_MIGRATIONS` registry, and
  `run_migrations(conn)` that applies each pending step inside a
  **runner-owned** `try: BEGIN … COMMIT except: ROLLBACK; raise`
  (the explicit `BEGIN` is the step's first statement, because with
  the vault's `isolation_level=""` the driver does not implicitly
  `BEGIN` around DDL). It runs on **both** `Vault.create()` (after the
  v1 baseline, before the sidecar) and `Vault.open()` (after the HMAC
  probe) — so a brand-new vault and an upgraded vault converge on the
  same latest shape (FIBR-0005 D1/D2), and the migration path is
  exercised on every first run.
- **`PRAGMA foreign_keys = ON`** in `Vault._connect` (FIBR-0005 D4),
  already enforced on every connection — which is what makes this
  phase's `categories.parent_id REFERENCES categories(id)`
  self-reference real.
- **`SchemaVersionError`** (`errors.py`) for a vault newer than this
  build — reused unchanged (a v3 build refuses a v4+ vault).

Per design.md the app is layered — **UI → Services → Repositories →
SQLCipher** — and the "Tables" list already names `categories`
(self-referential tree, `parent_id`). The glossary defines the
"Category tree" as *"Type (Income / Expenditure) → Category →
(optional, future) Sub-category. Stored self-referentially so depth can
grow; v1's UI shows two levels."* So this phase adds **no** new
architecture — it fills in the categories aggregate that was always
planned, mirroring the FIBR-0005 account aggregate almost one-to-one
(repository → service → manager screen), and adds one more migration
step to the runner that already exists.

## Design decisions baked into this spec

These are the non-obvious calls; `/cold-eyes` and the user sign-off
gate them before code. **D1–D5 and D7 are engineering calls made in
this spec; D6, D8, D9, and D10 are product decisions carried by the
roadmap scope + the seed-set choice (rich default set) confirmed for
this phase.**

1. **D1 — First-run reuses the create-then-migrate path unchanged.**
   `Vault.create()` already writes the v1 baseline, commits, then calls
   `run_migrations(conn)` (vault.py:96) *before* the sidecar. Adding a
   v2→v3 step to `_MIGRATIONS` means a fresh vault now lands at **v3**
   with the categories tree seeded — with **no change to `create()`**.
   A brand-new vault and an upgraded (v2) vault converge on the
   identical v3 shape, so the seed + schema are defined in **one place**
   (the v2→v3 migration), never duplicated (coding.md § 1.3), and the
   migration is continuously exercised on first run (D1 rationale from
   FIBR-0005, applied again).

2. **D2 — The v2→v3 migration is one atomic, idempotent, forward-only
   step.** Unlike v1→v2, it does **not** rebuild any table — it only
   `CREATE`s `categories` and `INSERT`s seed rows — so there is no
   `DROP`/`RENAME` to lose. But it is still wrapped in the runner's own
   `try: BEGIN … COMMIT except: ROLLBACK; raise` (the explicit `BEGIN`
   the step's first statement) for the same reason: with
   `isolation_level=""` the driver does **not** implicitly `BEGIN`
   around the `CREATE TABLE` (DDL), so an un-wrapped failure between the
   `CREATE` and the seed `INSERT`s could leave an empty-but-present
   `categories` table (a half-seeded v2.5 wreck). Wrapping the whole
   step makes it all-or-nothing: either v3 with the tree fully seeded,
   or still v2 with no `categories` table at all. Idempotency is
   inherent in `run_migrations`' version loop (`range(current+1,
   LATEST+1)` is empty once `current == 3`), exactly as for v1→v2 — the
   step is never re-run against an already-v3 vault, so the seed is
   never duplicated.

3. **D3 — The tree is *purely* self-referential; the two Types are
   seeded root rows, categories are their children, sub-categories
   (future) are grandchildren.** `categories.parent_id` alone encodes
   the whole hierarchy: `parent_id IS NULL` marks a **Type root**
   (exactly two: Income, Expenditure); a category is a child of a root;
   a future Sub-category is a child of a category. "Depth can grow"
   (glossary) is literal — a third level is just deeper rows, needing
   **no** schema change. The management UI (D9) exposes the first two
   levels only. This is the shortest correct model (one table, one
   `parent_id`, no special-case columns for level) (coding.md § 1.1).

4. **D4 — The two Type roots carry a stable `kind` token; descendants
   carry `kind = NULL`.** Later phases (P08 rules, P10 dashboard) need
   to know a category's **Type** — is it Income or Expenditure? The two
   roots are identified **structurally** by a non-translated `kind`
   token (`income` / `expenditure`) stored on the root row, **not** by
   their display name — so the roots stay identifiable across
   localisation and any rename, mirroring FIBR-0005 D5's token-vs-label
   split. A **descendant** category has `kind = NULL`; its Type is its
   root ancestor's kind (resolved by walking `parent_id` to the root —
   at most two hops in a three-level tree). Storing `kind` on the roots
   **only** (not denormalised onto every row) keeps the tree the single
   source of truth: moving a category between Types (D9 re-parent) is a
   one-row `parent_id` change with no subtree kind-rewrite. The
   root-walk resolver itself is **not built this phase** — no
   deliverable here needs it (the UI groups by `parent_id`, not by a
   resolved kind); P08/P10 add it when they first aggregate by Type
   (YAGNI, coding.md § 1.1).

5. **D5 — `CategoryKind` is a closed enum stored as a stable token,
   exactly like `AccountType`.** `class CategoryKind(StrEnum)` has two
   members — `income`, `expenditure` — whose `.value` is the stored
   token. It is what the seed writes to the two roots and what any later
   code branches on; it is **never translated** (design.md i18n). The
   human-readable **label** shown as the two Type headers ("Income",
   "Expenditure") is produced separately via `tr()` in the UI, keyed on
   the kind token. So — unlike a free-form category name (user data) —
   the Type labels are a closed set localised by translation, not by the
   user renaming a stored string; the roots therefore need no
   user-facing stored name (their stored `name` is a non-translated seed
   token, D8, never shown).

6. **D6 — Deleting a category is *blocked*, not cascaded, when it would
   lose structure.** The safest default for a classification tree the
   user has invested in: (a) a **Type root** cannot be deleted
   (`ProtectedCategoryError`) — there are always exactly two Types, the
   structural analogue of FIBR-0005 D7's "last account can't be
   deleted"; (b) a category that **still has children** cannot be
   deleted (`CategoryHasChildrenError`, nothing removed) — the user
   removes the sub-categories first, mirroring FIBR-0005 D6's block-in-
   use stance; (c) a **childless, non-root** category deletes cleanly.
   Cascade-delete of a subtree was considered and rejected for this
   phase (silent multi-row loss on financial classification data). Note
   there is **no** in-use-by-transaction guard this phase, because no
   transaction references a category yet (D10) — P08 adds that guard
   alongside the `transactions.category_id` column it introduces.

7. **D7 — Name uniqueness is scoped to siblings, not global.** A
   category name must be unique **among the children of the same
   parent** (case-insensitive, trimmed) — so "Insurance" can exist under
   both Income and Expenditure, but not twice under Expenditure. This
   matches how the tree reads to a user (each branch is its own
   namespace) and is enforced at the **service** layer with a friendly
   message (a single enforcement point; no DB `UNIQUE(parent_id, name)`
   constraint — mirrors FIBR-0005's service-layer name uniqueness).

8. **D8 — The tree is seeded with a rich default set; seed names are
   data, not UI strings.** The migration seeds the two Type roots and
   **16 default categories** (list under § Data model), so the app is
   useful out of the box. As with FIBR-0005's "Default" account (D8),
   the seed **category** names are written to the DB by the migration
   and are therefore **not** run through `tr()` (translating stored data
   would make it locale-dependent, violating D5); the user renames /
   deletes them in the manager to localise or tailor. The two **root**
   names are likewise non-translated seed tokens, but — unlike category
   names — are never shown (their display label is `tr()`-derived from
   `kind`, D5).

9. **D9 — The management UI exposes exactly two levels (Type →
   Category); it can re-parent a category between Types.** The screen is
   a two-level tree: the two Type headers, each with its categories.
   Adding a category chooses its Type (which root it hangs under);
   editing a selected category can rename it **and** move it to the
   other Type (a `parent_id` change — mirrors FIBR-0005's account
   edit, which changes name **and** type). A **Type root** is never an
   *editable or deletable* row — it is a structural header; selecting one
   only disables the Update/Delete actions (INV-7f). Adding
   or editing the **third** level (Sub-categories) is **not** exposed in
   v1's UI (the data model supports it; the UI is a later enhancement).

10. **D10 — Linking a transaction to a category is out of scope; it
    belongs to P08.** This phase does **not** add `transactions.
    category_id`, and does **not** touch `TransactionRepository`,
    `TransactionService`, or the add-transaction form. The transaction→
    category link — and the auto-assignment (rules) plus manual override
    that fill it — is owned by **P08 (FIBR-0010)**, which adds the
    nullable `category_id` FK via its own forward migration step (whatever
    schema version is current then — intervening import phases may add
    their own) at the moment it first needs it. Imports (P05–P07) land transactions with
    no category; they are "uncategorised" until P08 runs. This keeps
    P04 surgical (a new table + a new screen, transactions untouched)
    and puts the link where the code that reads and writes it lives.

## Invariants (testable)

> These `INV-N` are **local to this spec**. Tests live in
> `tests/features/categories/` (spec.md + `test_categories.py`) unless
> noted, use `tmp_path` for every on-disk vault, and never touch the
> network or real financial data (testing.md § 6). GUI-touching
> invariants use the `pytest-qt` `qtbot` fixture (enabled since
> FIBR-0004). The one schema-version assertion in the FIBR-0004 vault
> suite is updated in place (see Deliverables).

- **INV-1 — Category model & CRUD round-trip.**
  `CategoryRepository.add(parent_id, name)` inserts one row (a child,
  always with `kind = NULL` — the two roots are seeded only by the
  migration, D4/D8) and returns its new integer id; `.list_all()` returns every category as a
  `Category(id, parent_id, name, kind, created_at)` record ordered
  deterministically (**by `parent_id` — NULLs first — then name
  case-insensitively, then id**, so the two roots sort ahead of their
  children and siblings sort stably); `.children_of(parent_id)` returns
  just that parent's direct children (and `.children_of(None)` the two
  roots); `.get(id)` returns the matching `Category` or `None`;
  `.update(id, name, parent_id)` overwrites name + parent in place;
  `.delete(id)` removes exactly that row. Deleting or updating a **missing** id
  is an idempotent no-op (zero rows affected, no raise). A round-trip
  (add → get) returns the same `name`, `parent_id`, `kind`, and a
  well-formed ISO-8601 `created_at`. Source: design.md "Repository
  layer".

- **INV-2 — Kind is a closed, non-translated set on the roots only.**
  `CategoryKind` has exactly the two members `income`, `expenditure`;
  their stored values are those exact lowercase tokens. After
  migration, the **two root rows** (`parent_id IS NULL`) carry these
  two kinds (one each, distinct), stored **verbatim** (a value read back
  equals the token written — no case-folding, no translation). Every
  **non-root** category (a seeded default, or one the service adds)
  carries `kind = NULL` — the service's `add_category` always inserts
  `NULL` for a child's kind (a category's Type is its root ancestor's
  kind, D4; the resolver is deferred, D4). INV-2 asserts token storage
  on roots + `NULL` on descendants (headless); the two Type **labels'**
  mapping onto the kinds is owned by INV-7a (it needs the UI). Source:
  D4/D5; roadmap FIBR-0006; design.md i18n.

- **INV-3 — Name validation & sibling-uniqueness.** `add_category` /
  `update_category` raise `ValueError` when the name is empty or
  whitespace-only (after `str.strip()`), or when it duplicates a
  **sibling's** name — another child of the **same `parent_id`** —
  case-insensitively (trimmed). On `update`, the category's *own* row is
  excluded from the sibling check (re-saving an unchanged name is
  allowed), and the check is run against the **target** parent (so a
  re-parent that would collide with a name already under the destination
  Type is refused). A valid, unique, trimmed name is accepted and stored
  trimmed. Source: D7; design.md "Error handling".

- **INV-4 — v2→v3 migration: forward-only, atomic, idempotent,
  transactions untouched.** Opening a **v2** vault (the FIBR-0005 shape:
  `schema_version = 2`, `accounts`, `transactions` with `account_id`)
  runs `run_migrations`, which:
  - creates the `categories` table (§ Data model),
  - inserts the **two Type roots** — (`name "Income"`, `kind income`,
    `parent_id NULL`, ISO-8601 `created_at`) and (`name "Expenditure"`,
    `kind expenditure`, `parent_id NULL`, `created_at`) — capturing each
    root's id,
  - inserts the **default categories** (§ Data model) as children of the
    matching root (`parent_id = root.id`, `kind NULL`, `created_at`),
  - sets `schema_version` to 3,
  - **leaves `transactions` and `accounts` unchanged** (asserted by
    comparing the pre/post row set + column shape of `transactions` and
    the pre/post `accounts` row set — the v2→v3 step touches neither).

  Four guarantees hold, each a test leg:
  - **Atomic.** The whole v2→v3 step runs inside the runner's own
    `try: BEGIN … COMMIT except: ROLLBACK; raise` (D2). A forced failure
    midway — the test wraps `conn.execute` to raise on the **first**
    statement whose text contains `INSERT INTO categories` (letting
    `BEGIN` and the `CREATE TABLE` through, and forwarding the runner's
    own `ROLLBACK`), i.e. immediately after `categories` is created and
    before the first seed row lands — leaves the vault, **on the same
    connection (before any reopen), still at v2 with no `categories`
    table** (the `ROLLBACK` undoes the `CREATE` inside the same
    transaction), never half-seeded, and re-openable — a subsequent
    unlock re-runs the migration cleanly to v3 (matching the FIBR-0005
    rollback test's same-connection framing, test_accounts.py:272–291). Raising on the
    *first* category `INSERT` (rather than a later one) is deliberate: it
    proves the `CREATE TABLE` itself is inside the rolled-back
    transaction. This "first `INSERT INTO categories`" text discriminator
    is the migration's wedge point — the analogue of FIBR-0005 INV-4's
    single-statement `RENAME` discriminator; because every seed `INSERT`
    contains the substring `INSERT INTO categories` (while the
    `CREATE TABLE` and `UPDATE schema_version` do not), the proxy fires
    on the first one it sees — a root insert — which is exactly the wedge
    we want.
  - **Idempotent.** Re-running `run_migrations` on a **v3** vault makes
    no change (version stays 3; the roots and defaults are not
    duplicated) — inherent in the version loop.
  - **Baseline-complete.** A brand-new first-run vault ends at exactly
    **v3** with the tree seeded (D1 — same create-then-migrate path),
    because `SCHEMA_VERSION` (= 1) create() writes still equals the
    first migration step's *from*-version, and the runner walks 1→2→3.
  - **Forward-only.** A vault whose recorded version exceeds
    `LATEST_SCHEMA_VERSION` (now = 3) is refused with the existing
    `SchemaVersionError` (unchanged from FIBR-0005), not downgraded.

  Source: design.md "Persistence"; FIBR-0005 (the runner).

- **INV-5 — The tree is well-formed.** After migration:
  `children_of(None)` returns **exactly two** roots, with distinct
  kinds `income` and `expenditure` and `parent_id IS NULL`. Every
  seeded default category has a `parent_id` equal to one of the two
  roots' ids (no orphan, no default that is itself a root). With
  `PRAGMA foreign_keys = ON` (inherited from FIBR-0005 D4), inserting a
  category whose `parent_id` does **not** exist raises
  `sqlcipher3.dbapi2.IntegrityError` (the self-referential FK backstop),
  and the row is not written. The service **also** refuses
  `add_category` with a `None` (root-less) parent: a `NULL` `parent_id`
  is FK-exempt (§ Verified API basis), so without this guard
  `add_category(None, …)` would mint a **third** root and defeat "exactly
  two roots" — roots are migration-only (D4/D8). Two headless legs lock
  this: adding a category under a **non-existent** parent id raises
  `IntegrityError` (no row written), and adding one with a **`None`**
  parent raises `ValueError` (no row written). The FK-violation leg runs
  on a **`Vault`-opened** connection (where `Vault._connect` has set
  `PRAGMA foreign_keys = ON`, vault.py:142), **not** on the raw
  v2-fixture connection, which does not enable the pragma — otherwise the
  `IntegrityError` would not fire. `update_category` carries the
  **symmetric** guard, since it too can defeat "exactly two roots": it
  rejects a `None` destination parent with **`ValueError`** (which would
  mint a third root — symmetric with `add_category`), and refuses editing
  a row that is **itself a root** (the *edited row*'s own `parent_id IS
  NULL`) with **`ProtectedCategoryError`** (a root cannot be renamed or
  demoted to a child). The two conditions are distinct — a `None`
  *destination* parent vs. a root *subject* row — so their exceptions
  never overlap; a call that is both (editing a root **and** passing
  `None`) hits the root-subject guard first (`ProtectedCategoryError`).
  Two more headless legs lock these (each asserting its named exception
  type). Source: roadmap FIBR-0006; design.md "Persistence".

- **INV-6 — Delete guard (protect roots + block-with-children).**
  `CategoryService.delete_category(id)`:
  - raises `ProtectedCategoryError` (nothing removed) when the target is
    a **Type root** (`parent_id IS NULL`) — the two Types are
    structural and permanent (D6);
  - raises `CategoryHasChildrenError` — a type **distinct** from
    `ProtectedCategoryError` — when the target is a non-root category
    that **still has ≥ 1 child** (`children_of(id)` is non-empty),
    checked in the service **before** any `DELETE` (D6);
  - **succeeds**, removing exactly that row, for a **childless, non-root**
    category;
  - on a **missing** id, **falls through both guards** to the
    repository's zero-row no-op (not a root — `get` is `None`; no
    children) — no raise; consistent with INV-1's missing-id no-op.

  (finbreak is single-user and single-threaded, so nothing writes
  between the guard checks and the `DELETE`.) Asserted headless (seed
  present; confirm a root is refused, a category with a hand-added child
  is refused, a childless category deletes, a missing id is a no-op).
  Source: user decision D6.

- **INV-7 — Categories-manager UI round-trip** (`qtbot`; sub-cases per
  testing.md § 4):
  - **INV-7a** — the tree shows exactly **two** top-level Type nodes
    with translated labels ("Income", "Expenditure"), each label
    mapping back to its stored `kind` token (the label↔token assertion
    INV-2 delegates here). Because the Type `QComboBox` carries the
    root's **id** as `itemData` (not the kind token), the test resolves
    label→kind via the top-level tree node's `kind` role, not off the
    combo (unlike the accounts suite, whose combo stores the token
    directly). The seeded default categories appear as children under
    the correct Type.
  - **INV-7b** — entering a name, choosing a Type, and pressing **Add**
    makes the category appear as a child of that Type in the tree.
  - **INV-7c** — selecting a category node loads its name + Type into
    the form (the add/**edit** form reused for editing).
  - **INV-7d** — **Update selected** renames the category and, if the
    Type picker was changed, **re-parents** it under the other Type
    (INV-1/INV-3 through the UI), and the tree reflects the move.
  - **INV-7e** — a childless, non-root category **Delete**s and
    disappears from the tree.
  - **INV-7f** — a **Type root** is never a target: selecting a root
    node **disables** the Update and Delete buttons (so a root cannot be
    renamed, re-parented, or removed through the form), re-asserting
    INV-6's root protection from the UI.

  All strings via `self.tr(...)`; layout via Qt layout managers and a
  `QTreeWidget` (RTL-safe, coding.md § 5.2). Source: glossary
  "Category tree"; design.md "Category manager".

- **INV-8 — No secret/network regression in the new modules.** The
  files this phase adds introduce **no** network import (`socket`,
  `http`, `urllib`, `requests`, `ftplib`, …) anywhere under
  `src/finbreak/`, and log **no** secret. Two legs (mirroring FIBR-0005
  INV-8):
  - **No-network (automatic).** The vault-suite's existing **static
    import scan** already walks **all** of `src/finbreak/`, so it covers
    the new modules with no new test.
  - **No-secret-log (new leg).** A `caplog` capture over a category
    **add → update → delete** cycle asserts no password/key bytes appear
    (category names are user data, not secrets, and may appear in
    lifecycle lines; no transaction amount is logged — none is handled
    here).

  Source: security-model INV-8, INV-9.

## Deliverables

## Data model (v3)

The v2→v3 migration adds one table and leaves the rest untouched:

- `categories(id INTEGER PRIMARY KEY, parent_id INTEGER REFERENCES
  categories(id), name TEXT NOT NULL, kind TEXT, created_at TEXT NOT
  NULL)` — one row per node. `parent_id IS NULL` marks a **Type root**
  (exactly two); a non-NULL `parent_id` is a category (child of a root)
  or a future sub-category (child of a category). `kind` is a
  `CategoryKind` token (`income` / `expenditure`) on the **roots only**,
  `NULL` on every descendant (D4). The repository `SELECT` and the
  `Category` dataclass share this exact column order — `id, parent_id,
  name, kind, created_at` — so `Category(*row)` stays aligned (`id` and
  `parent_id` are adjacent `int` columns, so swapping them would compile
  but silently corrupt — hence the fixed order; FIBR-0005 `Transaction`
  parity). Sibling-uniqueness of `name` (same
  `parent_id`, case-insensitive) is enforced at the **service** layer
  (INV-3), not by a DB constraint. The self-referential FK is enforced
  by `PRAGMA foreign_keys = ON` (FIBR-0005 D4).
- `transactions`, `accounts`, `settings` — **unchanged** (no
  `category_id` this phase, D10).
- `schema_version` — the single row now holds `3`.

**Seed (written by the v2→v3 migration, as non-translated data, D8):**

- Two **Type roots**: `Income` (`kind = income`), `Expenditure`
  (`kind = expenditure`).
- Default **Income** categories: Salary, Sales, Interest, Gifts,
  Lottery, Other income.
- Default **Expenditure** categories: Groceries, Fast food, Bills &
  utilities, Rent / Mortgage, Transport, Medical, Entertainment,
  Clothing, Insurance, Other expenditure.

(16 default categories — 18 rows including the two Type roots; the exact
list is a product default the user edits in the manager — the roadmap's
named examples — salary, sales, fast food, bills, medical, lottery — are
all present.)

Forward-only: a later phase adds a new migration step for
`transactions.category_id` when it first assigns categories (P08); this
one is never reversed.

## Verified API basis

Pins the idioms the INVs assert against, so the contract cites real
APIs (global rule § 13). To be **re-verified empirically at
implementation time** against the pinned `sqlcipher3-binary` 0.6.0 and
PySide6 6.11.1. The migration mechanics (explicit `BEGIN` first under
`isolation_level=""`; FK-ON self-reference) are already proven by
FIBR-0005's shipped `_migrate_to_v2` — this phase reuses them.

- **SQLite self-referential FK** — `parent_id INTEGER REFERENCES
  categories(id)` in the same table is valid SQLite; with `PRAGMA
  foreign_keys = ON` (already set per connection, FIBR-0005 D4),
  inserting a row whose `parent_id` has no matching `id` raises
  `IntegrityError` (subclass of `DatabaseError`) (INV-5). A `NULL`
  `parent_id` is exempt from the FK check (the two roots) — standard
  SQL FK semantics; correspondingly, `children_of(None)` must query
  `WHERE parent_id IS NULL` (not `= ?` bound to `None`, which matches no
  row in SQL) to return the two roots.
- **v2→v3 migration** — a plain `CREATE TABLE` + seed `INSERT`s (no
  table rebuild), wrapped in the runner's explicit `BEGIN … COMMIT` /
  `ROLLBACK` per D2 — the same mechanism `_migrate_to_v2` already uses
  and the vault tests already prove. Written as **discrete**
  `conn.execute()` statements (not `executescript`), so the
  atomic-rollback test's per-statement proxy can intercept the first
  category `INSERT` (INV-4).
- **`enum.StrEnum`** (stdlib, 3.11+) — `class CategoryKind(StrEnum)`
  gives string-valued members whose `.value` is the stored token; the
  modern idiom over `class CategoryKind(str, Enum)` (coding.md § 1.5),
  identical to `AccountType`.
- **PySide6 6.11.1 `QTreeWidget`** — two top-level `QTreeWidgetItem`
  Type nodes, each with category children; `item.data(col, role)` /
  `setData` carry the row's id + `parent_id` + `kind` behind the
  display text (mirroring `AccountsWidget`'s `QListWidget` item-data
  roles). `QComboBox.addItem(label, userData=root_id)` + `currentData()`
  carries the chosen Type's root id behind a `tr()` label. Standard
  `self.tr(...)` + layout-manager idioms (coding.md § 5.2).
