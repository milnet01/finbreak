## 1. Problem & intent

finbreak ships today as a **single AppImage** (FIBR-0054) attached to the
GitHub release, plus a Windows `.exe`. That serves users who download-and-run,
but it is not a **first-class Linux install**: it does not appear in a
software centre, it is not on a distro update channel, and it needs the user to
find the GitHub release page, download the blob, `chmod +x`, and trust FUSE.

FIBR-0155 publishes finbreak through the **openSUSE Build Service** (OBS,
`build.opensuse.org`, where the user has an account). OBS builds **native
packages** (RPM for openSUSE/Fedora, deb for Debian/Ubuntu) from one source
recipe and publishes them to `software.opensuse.org` plus per-distro download
repos. The outcome the user wants: **install finbreak the normal way for your
distro, see it in your app menu / software centre, and get updates through your
distro's package manager** — the "app store" experience.

This is **complementary** to, not a replacement for, the AppImage (FIBR-0054)
and the deferred macOS `.dmg` + Flatpak/Flathub route (FIBR-0130): AppImage = portable
download; OBS = distro-integrated install; Flatpak = sandboxed cross-distro.

## 2. Scope

**In scope**
- An **OBS project + package** (`packaging/obs/`, source-tracked in this repo)
  that builds native RPM + deb for the **four buildable targets** in § 3.1
  (plus openSUSE Leap as a documented glibc gap).
- A **PyInstaller-frozen payload** installed under `/usr/lib/finbreak/`
  (§ 3.2), with a **`/usr/bin/finbreak`** launcher (§ 3.4).
- **Reverse-DNS desktop integration** (§ 3.3): a checked-in
  `io.github.milnet01.finbreak.desktop`, the hicolor icon set, and an AppStream
  `io.github.milnet01.finbreak.metainfo.xml`.
- **Metainfo `<release>` mirrors CHANGELOG** (§ 3.7), synced by `bump.json`
  `todos` reusing the `changelog-writer` subagent, so the store listing shows
  real notes.
- A **`[project.scripts]` console entry point** in `pyproject.toml` (§ 3.4) — the
  documented, testable `finbreak.__main__:main` launcher target.
- **Offline, reproducible OBS builds** (§ 3.6): the pinned wheel closure is
  vendored (an `_service`-fetched source tarball + `pip download`), so the OBS
  build root needs no network.
- A **`security-model.md` note** that the distro build makes **zero** outbound
  requests by default (§ 3.8) — the self-updater is inert (`detect_installer()`
  → `None`), tightening INV-8 for this build.
- **Tests** by the FIBR-0096 script-scrape pattern (§ 4): assert the recipe /
  desktop / metainfo invariants without a real OBS build.

**Out of scope**
- **Getting into the official distro archives** (Fedora/Debian main). This is
  the author's **OBS home project** repo (`home:milnet01` or a devel project),
  the accepted venue for a bundled-runtime third-party app. Official-archive
  submission (which would force the distro-dep model § 7 rejects) is a much
  larger, separate effort.
- **Flatpak/Flathub** (FIBR-0130, deferred) and **macOS** packaging.
- **Sharing the security-critical native stack with distro libraries** — see
  § 7 for why SQLCipher/qpdf/pdfium stay bundled (correctness + availability).
- **Auto-submitting releases to OBS from CI** — the first cut is a documented
  manual `osc` submit tied to the tag; automation is a follow-up (§ 3.9).
- Changing the app's **behaviour**. The source changes are narrow and
  packaging-driven: (1) an additive `[project.scripts]` console entry point
  (§ 3.4), and (2) a **one-line** change to the runtime app-id
  (`app.setDesktopFileName(...)`, `src/finbreak/app.py:43`) so the Wayland
  `app_id` matches the installed reverse-DNS desktop file (§ 3.3) — a
  correctness fix, not a behaviour change. Everything else is packaging + docs.

  the Settings checkbox is disabled via `is_update_supported()`
  (`update_installer.py:404-407`). Wired at `main_window.py:210-211`.
- **INV-8** (`docs/security-model.md:244-255`): "the shipped app makes exactly
  one kind of outbound request — an opt-in, off-by-default update flow …
  confined to `services/update_fetch.py`." Enforced by
  `test_INV8_no_network_imports_under_src` (allowlist = only
  `services/update_fetch.py`).
- **Two DIFFERENT Qt-lib sets — do not conflate them (§ 3.5):**
  - **Unfrozen-venv set** (`scripts/ci-setup.sh:34` —
    `libgl1 libegl1 libglib2.0-0 libxkbcommon0 libdbus-1-3 libfontconfig1`): what
    `pytest-qt` / `python -m finbreak` needs to load PySide6 **from a venv** in
    CI. NOT the frozen artifact's external needs.
  - **Frozen-artifact set:** the clean-room proof
    (`scripts/build-smoke.sh:95-96`) installs **only `libgl1 libegl1`** on the
    Python-free host — glib/xkbcommon/dbus/fontconfig/freetype/harfbuzz + the
    SQLCipher/qpdf natives **travel inside the bundle**. The build-time set that
    PyInstaller *collects into* the bundle is the wider
    `scripts/_build-smoke-in-container.sh:30-33` list. → **`Requires:` = libGL/
    libEGL only; `BuildRequires:` = the wider collect-set** (§ 3.5).
- **Payload glibc floor (`pyproject.toml:21-26`):** the pinned wheel closure's
  glibc floor is **2.34** (PySide6's `manylinux_2_34` wheel governs); the
  frozen artifact's *effective* floor is the **build image's** glibc (today
  `python:3.12-slim-bookworm`, ~2.36). **A target whose build-root glibc is
  below 2.34 cannot install the wheels** — this gates the target matrix (below).
- **App-ID already reserved:** `docs/standards/naming.md:175-177` prescribes
  `io.github.<owner>.finbreak.desktop`, `io.github.<owner>.finbreak.metainfo.xml`,
  and `finbreak.spec` "at the ADR-0007 packaging step". ROADMAP.md pre-scopes
  this item.

**Concrete target matrix.** The **glibc ≥ 2.34** floor (above) is the gating
constraint, not the Python version — a build root below 2.34 cannot install the
PySide6 wheel:

| Family | Target repo | build-root glibc | Verdict |
|---|---|---|---|
| openSUSE | Tumbleweed | ~2.40 (rolling) | ✅ OBS home turf; the primary listing |
| Fedora | latest stable | ≥ 2.38 | ✅ second RPM family |
| Debian | 13 (trixie) | 2.41 | ✅ matches the CI clean-room `debian:13-slim` |
| Ubuntu | 24.04 LTS | 2.39 | ✅ largest desktop base |
| openSUSE | **Leap 15.6** | **2.31** | ❌ **below the 2.34 floor** — see below |

**Four buildable targets**, plus openSUSE **Leap** as a **known gap**: Leap 15.6
ships glibc **2.31** (< 2.34), so the manylinux wheels will not install in its
OBS build root — the `pip install --no-index` step (§ 3.6) fails outright. This
is a **hard wheel-floor limitation**, not something the frozen-payload model
fixes. Resolution: target **Leap 16** once released (glibc ≥ 2.38, clears the
floor); until then Leap users use the AppImage (FIBR-0054) or Tumbleweed. This
is a **change from the user's original "Tumbleweed + Leap" target choice** and
is surfaced to the user at spec sign-off. (A from-source PySide6 build to lower
the floor is out of scope, § 2.)

### 3.2 Payload model — a frozen runtime, not distro-shared deps

**Decision: the package installs a PyInstaller-frozen `--onedir` runtime tree
under `/usr/lib/finbreak/`**, built **per target** inside OBS. It bundles its
own Python ≥ 3.12 and every pinned native (SQLCipher, qpdf, pdfium, Qt) in-tree
— conceptually the same self-contained runtime the AppImage ships, but in
**onedir** (directory-tree) form for a native `/usr` layout, not the AppImage's
**onefile** self-extracting blob (§ 3.5 scopes the new build work this implies).

Why this and not the "purist" distro-dependency model (`Requires: python3-pyside6,
python3-pikepdf, python3-sqlcipher, …`):

1. **Correctness (decisive for this app).** finbreak is correctness-critical —
   it encrypts people's money data. `sqlcipher3-wheels` is a **specific fork**
   bundling SQLCipher 4.12.0 with `SQLCIPHER_COMPAT = 4`; substituting a
   distro `python3-sqlcipher3` binding (different upstream, possibly different
   SQLCipher version / KDF defaults) risks **vault incompatibility** and is not
   something to discover in the field. The frozen payload runs the **same pinned
   source closure the gate tests** (same wheel versions, same natives); the
   binaries are **re-linked per target** (§ 3.6), so builds share the dependency
   closure, not a byte-identical artifact.
2. **Availability.** `pypdfium2` (Google's pdfium, via pdfplumber) and the
   `laggykiller` SQLCipher fork are **not known to be distro-packaged** across
   the RPM/deb targets — a distro-dep model would likely be a **blocker** on at
   least one target and a per-family dep-name matrix on the rest. *(This
   availability claim is the crux of the decision and MUST be confirmed by an
   actual `zypper se` / `dnf search` / OBS package-search at implementation time,
   per rule 13 — recorded as an implementation pre-check, not asserted here.)*
3. **Interpreter + glibc independence.** finbreak needs Python **≥ 3.12** and
   glibc **≥ 2.34** (§ 3.1). Bundling Python means the *installed* package needs
   no distro python3.12 at all (a venv-on-system-python model, § 7, would need
   it and would drop targets that default lower). The glibc floor still applies
   at **build** time and is what gates the target matrix (§ 3.1) — bundling does
   not lift it.
4. **Shared flags, one new mode.** The freeze targets `__main__.py` via
   `--paths /src/src` exactly as the gated `build-smoke.sh` does (no package
   `pip install .` — the finbreak package is *not* installed; only the vendored
   **deps** are, § 3.6), reusing its collect/hidden-import flags (rule 3). The
   **only** new build surface is the output mode: `--onedir` vs the proven
   `--onefile` (`_build-smoke-in-container.sh:100,121`; the "do NOT `pip install
   /src`" rationale is at `:46-49`). § 3.5 scopes that; it is validated by the
   recipe's `%check` self-test, not this repo's CI (unless `build-smoke.sh
   --build` is extended — § 6). The `[project.scripts]` entry point (§ 3.4) is a
   pyproject *declaration* verified by INV-8 in the normal pytest gate — it plays
   no part in the OBS freeze.

The accepted cost is a **larger, self-contained package** that does not share
libraries with the distro and does not receive distro security updates for its
bundled deps. That is the standard, accepted tradeoff for a **third-party OBS
home-project** app (it is not entering the official archive, § 2), it is
already how the AppImage ships, and the **FIBR-0096 SBOM** discloses the bundled
closure per release. Distro-native **update delivery still works** — a new
finbreak version is a new package in the OBS repo, pulled by
`zypper up`/`apt upgrade`.

This is **not** "wrap the AppImage" (the rejected option): there is no `.AppImage`
blob, no FUSE. The RPM/deb lays the freeze out in the native `/usr` filesystem
with a real launcher, `.desktop`, icons, and metainfo — a proper distro package
whose payload happens to be a frozen runtime. To be precise: the payload's
**build provenance** (manylinux wheels frozen by the same PyInstaller flags) is
close to the AppImage's; the difference is **packaging format** — a native
install with desktop/AppStream/store integration vs a portable single-file blob.
That format difference is the whole point of the user's "native, not
AppImage-wrap" choice.

### 3.3 Desktop integration — reverse-DNS app-ID

Three checked-in assets (source of truth in `packaging/obs/`), installed to the
standard XDG locations by the RPM/deb:

- **`io.github.milnet01.finbreak.desktop`** → `/usr/share/applications/`.
  Per `naming.md:175-177`. Key fields:
  - `Name=finbreak` (the display name — **mandatory** per the Desktop Entry
    spec; a `.desktop` without it is invalid and software centres refuse it).
  - `Exec=finbreak` (the § 3.4 launcher; **not** an absolute freeze path).
  - `Icon=io.github.milnet01.finbreak` (themed icon name = app-ID).
  - `StartupWMClass=finbreak` — the **X11** launcher-association key, which must
    equal the window's actual X11 `WM_CLASS`. **Wayland and X11 derive the
    identity from DIFFERENT Qt calls — this is the correctness subtlety:**
    - **Wayland** `app_id` = `QGuiApplication::desktopFileName()`. The app
      currently sets the bare `finbreak` (`app.py:43`), which would **not** match
      this reverse-DNS `.desktop` basename → duplicate generic icon. **Fix:** § 2's
      scoped one-liner changes `app.py:43` to
      `setDesktopFileName("io.github.milnet01.finbreak")`, so the Wayland `app_id`
      = the `.desktop` basename. (On Wayland `StartupWMClass` is ignored.)
    - **X11** `WM_CLASS` is derived by Qt's xcb backend from
      `QCoreApplication::applicationName()` (**not** `desktopFileName()`) and used
      **verbatim** when non-empty — Qt's xcb `wmClass()` only upper-cases the
      first letter in the *fallback* branch (when `applicationName()` is empty and
      it derives a name from the executable basename). Since `app.py:42` sets
      `setApplicationName("finbreak")` (non-empty, left unchanged, so nothing
      depending on the app name shifts), the X11 `WM_CLASS` class-string is
      **`finbreak`** (lowercase). Hence **`StartupWMClass=finbreak`**, not the
      app-ID and not a capitalised form.
    - **Belt-and-suspenders:** the implementer confirms the live string once with
      `xprop WM_CLASS` on the built app (§ 5), guarding against a Qt-version quirk;
      INV-3 asserts `StartupWMClass` equals `app.py`'s `applicationName` argument
      verbatim, so a drift breaks the test.
  - `Categories=Office;Finance;` (a personal-finance app; the AppImage's default
    `Utility;` is wrong for a software-centre listing).
  - `Terminal=false`, `Type=Application`.
- **Hicolor icons** → `/usr/share/icons/hicolor/<size>x<size>/apps/io.github.milnet01.finbreak.png`
  for each size present in `assets/icon/finbreak-{16,24,32,48,64,128,256,512}.png`
  (renamed to the app-ID at install). Enables the software-centre + menu icon.
- **`io.github.milnet01.finbreak.metainfo.xml`** →
  `/usr/share/metainfo/`. AppStream component (`desktop-application`) with:
  `<id>io.github.milnet01.finbreak</id>`, name, summary, a `<description>`,
  `<metadata_license>` (CC0-1.0 for the metainfo) + `<project_license>MIT</project_license>`,
  `<url type="homepage">`, `<launchable type="desktop-id">io.github.milnet01.finbreak.desktop</launchable>`,
  screenshots, and a `<releases>` block (§ 3.7). It **must pass**
  `appstreamcli validate` (INV-4) — this is what software.opensuse.org and GNOME
  Software / KDE Discover consume for the listing.

### 3.4 The `/usr/bin/finbreak` launcher + entry point

Two coordinated pieces:

1. **`pyproject.toml`** gains
   `[project.scripts]` → `finbreak = "finbreak.__main__:main"` (a **console**
   entry point). `main()` has real CLI behaviour — `--self-test` with an exit
   code, unknown-arg → exit 2 (`__main__.py:17-34`) — for which `console_scripts`
   is the idiomatic choice. (On this spec's POSIX targets `console_scripts` and
   `gui_scripts` stubs are functionally identical; the Windows `.exe` is frozen
   from `__main__.py` directly and doesn't consume this table, so the choice is
   purely about CLI-entry idiom.) This is the canonical, **testable** launch-target
   declaration (a test asserts it resolves to `__main__.main`). `main(argv=None)`
   already falls back to `sys.argv[1:]`, so it works whether invoked by the
   entry-point stub (no explicit argv) or the § 3.4.2 shell wrapper (`"$@"`).
2. **The packaged `/usr/bin/finbreak`** is a tiny shell wrapper the RPM/deb
   installs, `exec`-ing the frozen entry inside `/usr/lib/finbreak/` (the
   onedir's bootloader), **not** a `#!/usr/bin/python3` script — because the
   payload is frozen, not a site-packages install (§ 3.2). It passes `"$@"`
   through, so `finbreak --self-test` works from the installed package (INV-2,
   reusing the FIBR-0003 native-stack proof as the package's post-install smoke).

`main()` already rejects unknown args with exit 2 and handles `--self-test`
(`__main__.py:17-34`) — no change needed there.

### 3.5 The RPM `.spec` and `debian/` recipes

Both drive the **same three steps**: (1) freeze the payload, (2) lay it out
under `/usr/lib/finbreak/`, (3) install the § 3.3 desktop assets + § 3.4
launcher. Version comes from `__version__` via the `_service` (§ 3.9), never
hand-typed (INV-6).

- **`packaging/obs/finbreak.spec`** (RPM — openSUSE + Fedora):
  - `Version:` injected by the OBS `set_version` service from the tarball.
  - **Two distinct lib stanzas (§ 3.1):** `BuildRequires:` carries the wider
    PyInstaller **collect-set** (the `_build-smoke-in-container.sh:30-33` list,
    so the freeze bundles glib/xkbcommon/dbus/fontconfig/harfbuzz), plus the
    **unversioned** `python3` + `python3-devel` + `pip` (**not** a literal
    `python3.NN` — the generic package floats to the target's own default
    interpreter, § 3.6, so no per-target version conditional is needed).
    `Requires:` carries **only the host-left libs — libGL/libEGL** (the
    `build-smoke.sh:95-96` clean-room proof); the rest
    travel in-bundle, so `Requires`-ing them would be over-broad and risks a
    wrong per-distro package name breaking dependency resolution for a lib that
    was never needed. (Bundled `.so`s are `dlopen`'d, invisible to RPM's auto
    scanner — but the fix is to require libGL/libEGL explicitly, not the whole
    set.)
  - **openSUSE and Fedora use different RPM names** (openSUSE `Mesa-libGL1` /
    `Mesa-libEGL1` vs Fedora `mesa-libGL` / `mesa-libEGL`; and for the build-set
    `libglib-2_0-0` vs `glib2`, etc.), so both stanzas use
    **`%if 0%{?suse_version}` / `%if 0%{?fedora}` conditional blocks**, not one
    unconditional list.
  - **Suppress RPM's auto-machinery over the bundled tree:**
    `%global debug_package %{nil}` (the foreign stripped `.so`s have no
    debuginfo to split) and
    `%global __requires_exclude_from ^%{_prefix}/lib/finbreak/.*$` +
    `__provides_exclude_from` — so `rpmbuild`'s `find-requires`/`find-provides`
    don't auto-generate bogus deps/provides from the hundreds of bundled
    libraries (a standard bundled-runtime-RPM measure).
  - `%build`: `pip install --no-index --find-links vendor/` the **deps only**
    (into a build venv on the target's default `python3`), then the **onedir**
    PyInstaller freeze targeting `__main__.py` via `--paths` (§ 3.2 point 4 —
    shared collect flags, `--onedir` the one new mode; the finbreak package is
    *not* pip-installed).
  - `%install`: copy the onedir tree to `%{buildroot}%{_prefix}/lib/finbreak/`,
    the launcher to `/usr/bin/finbreak`, and the desktop/icon/metainfo assets.
  - `%files`: the three XDG assets + the payload tree + launcher.
  - **`%post`/`%postun`:** the openSUSE/Fedora icon-cache + desktop-database
    scriptlet macros (`%icon_theme_cache_post`/`_postun`,
    `%desktop_database_post`/`_postun`) so the menu entry + icon appear without a
    manual cache refresh (RPM-specific; the `debian/` side's `dh` +
    `desktop-file-utils` triggers handle this automatically).
  - `%check`: run the self-test against the **staged onedir bootloader** by its
    build-root path — `QT_QPA_PLATFORM=offscreen
    %{buildroot}%{_prefix}/lib/finbreak/finbreak --self-test` — **not** a bare
    `finbreak` on `$PATH` (the package isn't installed at `%check` time, so `$PATH`
    wouldn't resolve to the staged freeze). **`offscreen` is required** — OBS build
    roots are headless (no X/Wayland), the same guard the repo already uses at
    `tests/conftest.py:17` and `scripts/build-smoke.sh:112`. This is the **sole
    automated gate** on the new onedir build path (§ 3.2 point 4).
- **`packaging/obs/debian/`** (deb — Debian + Ubuntu): `control`
  (`Build-Depends:` = the wider collect-set apt names **+ the unversioned
  `python3`/`python3-dev`** — floats to each target's default interpreter, so no
  per-target version pin is needed even though Debian 13 defaults to 3.13 and
  Ubuntu 24.04 to 3.12; `Depends:` = **libgl1 libegl1 only** — same split as the
  RPM), `rules` (the same freeze→layout→install via `dh`, running the buildroot-
  path `offscreen` self-test), `changelog` (§ 3.7), `install`, `compat`,
  `source/format`. `dpkg-shlibdeps` is likewise scoped away from the bundled tree
  (only libGL/libEGL declared).

### 3.6 Offline, reproducible OBS builds — vendored wheels

OBS build roots are **network-isolated**. The pinned wheel closure is therefore
**vendored**, not fetched during the build:

- The OBS package's **`_service`** (§ 3.9) fetches the tagged source tarball
  (`obs_scm`/`tar_scm` on `v{VERSION}`) **and** the wheel closure into `vendor/`.
  **Two axes govern wheel selection, not one:**
  - **glibc** — the pinned closure floor is `manylinux_2_34` (§ 3.1); one
    manylinux set clears every buildable target (all ≥ 2.38).
  - **CPython ABI** — `PySide6` + `cryptography` ship `abi3` (version-agnostic),
    and `pypdfium2` ships `py3-none` (its `libpdfium.so` is ctypes-loaded, no
    C-API), but `pikepdf`, `sqlcipher3-wheels`, and `pdfplumber`'s `Pillow` ship
    **per-minor** (`cp312`, `cp313`, …) wheels. The four buildable targets do
    **not** share one default `python3` (Ubuntu 24.04 → 3.12; Debian 13 / Fedora
    / Tumbleweed → 3.13). So the `_service` vendors **both `cp312` and `cp313`**
    manylinux_2_34 sets (`pip download --only-binary=:all: --python-version 3.12`
    and `… 3.13`, plus the platform/abi flags), **and** the PEP 517 build
    requirement (`setuptools>=77`, `pyproject.toml:2`). The build uses the
    target's **own default `python3`** and `pip install --no-index --find-links
    vendor/` picks the matching-ABI wheels — **zero network**, provably the
    pinned versions for the exact-pinned deps. *(Reproducibility caveat: the two
    range-specified deps — `beautifulsoup4`, `certifi` (§ 3.1) — resolve to
    whatever `pip download` sees at vendor time, so two vendorings of the same
    tag months apart could differ by a bs4/certifi patch. Neither is in the
    correctness-critical stack; exact-pin them if bit-reproducibility is later
    required.)*
  - *(If a target's default `python3` minor is ever outside `{3.12, 3.13}`, its
    ABI set must be added to the vendor step — a manual-checklist confirmation,
    § 5.)*
- **The glibc ≥ 2.34 floor (§ 3.1) applies here:** on a build root below 2.34
  (Leap 15.6, excluded), pip's tag matcher rejects the manylinux wheel and this
  step fails — which is exactly why the target matrix drops that target rather
  than discovering it in a red OBS build.
- Each buildable target's freeze is produced **on that target's own toolchain**,
  so it links that distro's Qt/libc — a genuine per-target-native result (the
  binaries differ per target by design; they are not byte-identical to one
  reference, § 3.2 point 1).

### 3.7 AppStream `<release>` mirrors CHANGELOG

The metainfo `<releases>` block and the `debian/changelog` are **generated from
`CHANGELOG.md`**, not maintained by hand (drift is the failure mode the whole
FIBR-0096/`bump.json` lockstep discipline exists to prevent):

- **Metainfo `<release>` = a `bump.json` `todos` entry reusing `changelog-writer`
  Mode A.** The `changelog-writer` agent has exactly two modes — **Mode A**
  (CHANGELOG section → metainfo `<release>` XML) and **Mode B** (commit range →
  CHANGELOG) — and the `/bump` skill's template already carries the "Prepend
  `<release version=…>` to the AppStream metainfo XML" todo that dispatches to
  it. FIBR-0155 adds that todo, keyed on `io.github.milnet01.finbreak.metainfo.xml`.
  Reuse-before-rewrite (rule 3) — no new metainfo script.
- **`debian/changelog` = a separate, mechanical `bump.json` `todos` entry** —
  **not** `changelog-writer` (it has no deb-changelog mode). The todo emits the
  stanza directly and **non-interactively** (`finbreak ({VERSION}) unstable;
  urgency=medium` + the CHANGELOG bullets + a `-- name <email>  date` trailer
  from `DEBFULLNAME`/`DEBEMAIL`), **not** via interactive `dch` — so a bump needs
  no TTY. The identity env vars match the release key's owner.
- **One-time seed at implementation:** the two files are *created* by FIBR-0155,
  so their **first** `<release>` / stanza for the **current** version (0.1.16,
  whose dated section already exists at `CHANGELOG.md`) is backfilled once at
  implementation (a manual `changelog-writer` Mode A run + a hand-written deb
  stanza) — the `todos` above only fire on *future* bumps. Without this backfill
  INV-6 fails the moment the suite runs (§ 6 lists it).
- INV-6's version check includes: the metainfo's newest `<release version>`
  equals `__version__`.

### 3.8 security-model.md impact — zero outbound in distro builds

The distro package is launched from `/usr/bin/finbreak` with **no `$APPIMAGE`
env** and is **not** a frozen Windows exe, so `detect_installer()` → `None`
(§ 3.1): the self-updater is **inert** and the Settings "check for updates"
checkbox is **disabled**. Therefore the distro build makes **zero** outbound
requests by default — a **strict subset** of INV-8's "exactly one opt-in flow".

- This is covered **without a source change** by the existing runtime gating.
  The spec adds a **note** under INV-8 (`security-model.md`) recording that the
  distro build's outbound surface is empty, and a **test** (§ 4) asserting
  `detect_installer()` returns `None` when `$APPIMAGE` is unset and `sys.frozen`
  is falsy (belt-and-suspenders: proves a future refactor can't silently arm
  the updater in a distro install).
- No new networking dependency is introduced; the package inherits the
  no-network-import allowlist unchanged.

### 3.9 Version / release-automation tie-in

- **Version flows one way:** git tag `v{VERSION}` → OBS `_service`
  (`tar_scm` `versionformat` / `set_version`) → `.spec`/`debian` `Version:`.
  Never hand-entered (INV-6).
- **First cut = documented manual submit.** After a release is tagged + the
  GitHub release exists, the maintainer runs a documented `osc` flow
  (`osc service run` to refresh vendored sources, `osc commit`) to trigger the
  OBS rebuild. **CI auto-submit to OBS is out of scope** (§ 2) — a follow-up
  once the manual path is proven on a real build.
- The `/bump` recipe gains the metainfo/changelog sync step (§ 3.7) so a bump
  keeps the store notes in lockstep with the other five version-bearing files.
  The metainfo + `debian/changelog` become **two new version-bearing files**, so
  `bump.json`'s **`post_check` is extended** to assert their newest version equals
  the bumped `__version__` too — catching a mis-fired sync at bump time, not just
  downstream in INV-6's pytest gate.

## 4. Invariants

Most invariants are **script-scrape** over the recipes/desktop/metainfo (the
FIBR-0096 pattern); **INV-5 and INV-8 are runtime assertions** (monkeypatch /
import). None needs a real OBS build. Coverage limits are stated per-invariant.

- **INV-1 — Frozen payload; minimal runtime deps; no distro-shared security
  stack.** The recipes install a self-contained payload under
  `/usr/lib/finbreak/`; neither the `.spec` `Requires:` nor `debian/control`
  `Depends:` names a distro SQLCipher, qpdf, pdfium, or PySide package (the
  security-critical stack stays bundled, § 3.2). The **runtime** dep stanza is
  the minimal host-left set (libGL/libEGL only, § 3.5 — not the wider build-time
  collect-set), and the `.spec` has both a `%if 0%{?suse_version}` and a
  `%if 0%{?fedora}` branch. *Test:* grep the recipes; assert the absent
  (bundled-stack) names against an **enumerated blocklist committed in the test
  spec** (so the coverage boundary is explicit, not implicit); assert
  `Requires:`/`Depends:` is the libGL/libEGL set; assert both `%if` branches
  exist.
- **INV-2 — Installed launcher works + self-tests.** `/usr/bin/finbreak` exists,
  is executable, `exec`s the frozen entry with `"$@"` passthrough; a `%check` /
  `debian/rules` step runs `finbreak --self-test` (FIBR-0003 sentinel).
  *Test:* scrape both recipes for the launcher install + the `--self-test` gate.
- **INV-3 — Identity matches on BOTH Wayland and X11.** The `.desktop` basename,
  `Icon=`, metainfo `<id>`, and installed icon names are all
  `io.github.milnet01.finbreak`; `Exec=finbreak`, and a mandatory `Name=` is
  present. The two identity fields are checked against their **distinct** Qt
  sources (§ 3.3): (a) **Wayland** — `app.py`'s `setDesktopFileName(...)` argument
  equals the `.desktop` basename; (b) **X11** — `StartupWMClass=` equals `app.py`'s
  `setApplicationName(...)` argument **verbatim** (`finbreak`; Qt uses a non-empty
  `applicationName` unchanged). *Test:* parse the `.desktop` + metainfo; read
  **both** `app.py` arguments and assert each match — so reverting the
  `setDesktopFileName` change (breaks Wayland) or mis-setting `StartupWMClass`
  (breaks X11) fails the test. *(Coverage limit: the implementer also confirms the
  live `xprop WM_CLASS` string once, § 5, guarding a Qt-version quirk.)*
- **INV-4 — Metainfo validates.** The metainfo passes `appstreamcli validate`
  (or `appstream-util validate-relax`). *Test + coverage limit:* the validators
  are **not** installed by `ci-setup.sh` and are absent from the CI image, so
  this test is **skip-if-absent** and is effectively a **manual / pre-OBS-submit**
  check (also an exit criterion, § 5) — it is NOT automatically gate-covered.
  (The version-equals-`__version__` assertion lives in INV-6, which IS
  gate-covered.)
- **INV-5 — Self-updater inert in a distro launch.** On Linux, with `$APPIMAGE`
  unset, `detect_installer()` returns `None` and `is_update_supported()` is
  `False` (the `sys.frozen` branch is Windows-gated, `update_installer.py`, so
  only `$APPIMAGE` participates on Linux). *Test:* monkeypatch `$APPIMAGE`
  unset on a Linux `sys.platform`; assert both.
- **INV-6 — Version single-source (metainfo AND deb changelog).** The `.spec`
  `Version:` derives from the tag via `_service` (no hard-coded literal), and the
  newest version in **both** the metainfo `<release version>` **and** the top
  `debian/changelog` stanza equals `__version__` (`__init__.py:7`). *Test:* grep
  the `.spec` for a hard-coded `Version:` digit-string (assert it is the service
  placeholder); read `__init__.py`, the metainfo, and `debian/changelog`'s first
  stanza, and assert all three versions match — so a skipped seed or a
  silently-failed bump todo (in either file) is caught by the gate, not left to a
  manual check.
- **INV-7 — Offline build.** The build step uses
  `pip install --no-index --find-links vendor/` — no `pip install` in the build
  phase lacks `--no-index`. *Test:* scrape the recipe/`_service`. *Coverage
  limit:* that the **vendored wheels match the pins** cannot be script-scraped
  (it needs a live `pip download` + version diff) — it is a **manual
  pre-OBS-submit checklist item** (§ 5), not a CI-gated invariant.
- **INV-8 — Console entry point declared.** `pyproject.toml` `[project.scripts]`
  maps `finbreak` → `finbreak.__main__:main`, and that attribute is importable
  + callable. *Test:* load the project metadata; import + assert. *(This is
  **this spec's** INV-8 — distinct from `security-model.md`'s INV-8, the one
  opt-in outbound flow, referenced in § 3.8.)*

## 7. Alternatives considered

- **Distro-shared dependencies** (`Requires: python3-pyside6, python3-pikepdf,
  python3-sqlcipher, python3-pdfplumber, …`; app installed as a noarch wheel).
  *Rejected:* the security-critical SQLCipher binding is a **specific fork**
  (correctness risk to swap, § 3.2.1); `pypdfium2`/pdfium is **not known to be
  distro-packaged** (likely blocker, § 3.2.2); and version skew fights the
  deliberate pins. This is the model the **official distro archive** would demand
  — hence archive submission is explicitly out of scope (§ 2).
- **Bundled venv on the distro's system python3** (ship a private venv of the
  pinned wheels, `#!/usr/bin/python3.12` launcher). *Rejected:* it needs each
  target to ship python ≥ 3.12 as the *default* system interpreter — a real
  constraint for older stable releases — whereas a bundled Python (§ 3.2) needs
  none; and it still must bundle the in-wheel natives anyway (same § 3.1 glibc
  floor, same pdfium/SQLCipher-fork availability gap), so it carries the
  frozen-payload cost **without** the interpreter-independence. PyInstaller-onedir
  bundles Python once.
- **Wrap the frozen AppImage in an RPM/deb** (the option the user explicitly
  rejected). *Rejected by user decision;* also technically inferior for a store
  listing (FUSE dependency, a blob under `/opt`, no clean `/usr` layout).
- **Flatpak/Flathub instead** (FIBR-0130). *Deferred, complementary* — a
  different distribution channel (sandboxed, cross-distro), not a substitute for
  native distro packages + the openSUSE portal the user asked for.
