## Design decisions baked into this spec

**D5 (import-as-flow) and D8/D9 (the provenance data model + backfill) carry the
load-bearing calls;** the rest are UI-structure/idiom.

- **D1 ‚Äî The four views become persistent tabs in a `QTabWidget`; that widget is
  the single "live" content, destroyed on lock.** Replace the
  build-fresh-widget-per-view model (for Home/Statements/Accounts/Categories)
  with **one `QTabWidget`** built once per unlocked session by
  `_build_workspace()` and installed as `self._live` via the **existing**
  `_set_live` (so lock's `_clear_live` destroys the whole workspace ‚Äî INV-3, no
  new teardown path). Navigation (tab bar / View menu / toolbar) calls
  `workspace.setCurrentIndex(i)` ‚Äî it **does not** rebuild a tab. Rejected alt:
  keep destroy-on-swap and fake tabs with buttons (loses real tab affordance,
  reinvents `QTabWidget`); a second persistent stack outside the security
  teardown (risks a live data tab surviving a lock).
- **D2 ‚Äî `_show_home` / `_open_accounts` / `_open_categories` become
  tab-switches; the `done`-returns-to-Home wiring retires for these three.**
  Because Accounts/Categories are now always-present tabs, their `done` "back to
  Home" affordance is meaningless (INV-12) ‚Äî the shell no longer connects
  `done ‚Üí _show_home`; it builds them with `show_done=False`. `_show_home(...)`
  is replaced by "switch to `tab_home`" (+ an explicit `home.refresh()` where a
  mutation requires it, INV-11).
- **D3 ‚Äî Navigation actions switch tabs; Manual entry stays a dialog, Import
  stays a flow.** `action_home` / `action_accounts` / `action_categories` (menu
  **and** toolbar, one `QAction` each) + `action_statements` (menu only) call
  `workspace.setCurrentIndex`. `action_manual_entry` opens the modal
  `ManualEntryDialog` over the workspace (workspace stays alive underneath); on
  its `committed` signal the shell refreshes the Home tab + count and switches to
  it (INV-11). `action_import` runs the import flow (D5).
- **D4 ‚Äî Cross-tab freshness via `currentChanged` + targeted post-mutation
  refresh.** The workspace connects `QTabWidget.currentChanged` to refresh the
  **activated** tab (Home/Statements re-read; Accounts/Categories re-read) ‚Äî so
  data changed elsewhere shows when a tab is next viewed, with **no** need to
  wire every mutation to every tab (coding.md ¬ß 1.1). Mutations that must reflect
  *without* a tab switch ‚Äî manual-entry commit, import `done`, statement delete ‚Äî
  additionally refresh Home + the status count explicitly (INV-11). Re-reading
  the vault per tab activation is cheap at this app's data sizes; a caching layer
  would be premature.
- **D5 ‚Äî Import is a full-content flow that replaces the workspace and returns to
  it, NOT a tab.** An import is a multi-step wizard that wants the full content
  area; making it a tab (a persistent half-filled wizard) is wrong. So
  `_open_import` **`_set_live`s the `ImportWizardWidget`** ‚Äî which (via
  `_clear_live`) **destroys the workspace** while importing (a *good* thing for
  INV-3: fewer decrypted rows alive) ‚Äî and on the wizard's `done` signal
  **rebuilds a fresh workspace and lands on the Statements tab** (so the just-
  imported statement is visible), refreshing Home + count. This is the *exact*
  FIBR-0051 destroy-on-swap pattern; the only new part is "rebuild workspace +
  select Statements" in the `done` handler instead of "show Home". A lock while
  the wizard is showing destroys the wizard (it is `self._live`), INV-3 intact.
- **D6 ‚Äî Home toolbar action + a hand-authored house SVG.** Add `action_home`
  (`objectName` `action_home`, already exists on the `View` menu from FIBR-0051 ‚Äî
  **the same single `QAction`** is added to the toolbar too, no duplicate) with a
  new `ui/icons/home.svg` (monochrome, neutral mid-tone, the FIBR-0051 D7 style),
  first in the toolbar. `ToolButtonTextUnderIcon` keeps the text label (the
  meaning-carrier). The new icon travels into the FIBR-0003 bundle as package
  data exactly like the FIBR-0051 glyphs (Deliverable 10).
- **D7 ‚Äî Window geometry persisted via `QSettings`, in a plain INI outside the
  vault.** `paths.window_settings_path()` returns `data_dir() / "window.ini"`
  (a new sibling of the vault, **not** encrypted ‚Äî geometry is non-sensitive and
  must load *before* unlock). The shell reads it in `__init__` (after building
  chrome, before showing) via `QSettings(str(path), QSettings.Format.IniFormat)`:
  `restoreGeometry(bytes)` (size + position), `restoreState(bytes)` (toolbar
  state), and `setCurrentIndex(last_tab)` **applied when the workspace is built**
  (the tab index is meaningless while locked). It **writes** them in an overridden
  `closeEvent` (geometry + state + current tab, then `super().closeEvent`) and on
  each `currentChanged` (so the last tab survives a crash). Tests inject an
  explicit path via monkeypatching `paths.window_settings_path`, mirroring how
  the vault tests inject `vault_path`. Rejected alt: native-format `QSettings`
  (org/app registry/plist) ‚Äî an explicit INI is easier to point at a `tmp_path`
  and to reason about ("it's this one file").
- **D8 ‚Äî v5‚Üív6: a nullable `transactions.statement_period_id` FK, stamped at
  import.** One migration step `_migrate_to_v6` (the same atomic-`BEGIN` /
  `ADD COLUMN` / `UPDATE schema_version` / `COMMIT` skeleton as `_migrate_to_v5`,
  **plus** the D9 backfill `UPDATE`s inside the same transaction):
  `ALTER TABLE transactions ADD COLUMN statement_period_id INTEGER REFERENCES
  statement_periods(id)`. A nullable `ADD COLUMN` with a plain (non-cascade)
  `REFERENCES` is **verified** valid SQLite (SQLite requires the added FK column
  default to be `NULL` ‚Äî which it is). `commit_import` is **reordered** so the
  `statement_periods` row is created (or its id looked up when the span already
  exists) **before** the transaction batch, and `add_batch` stamps every row with
  that id. `add` (manual) stays `NULL`. Rejected alt: `ON DELETE CASCADE` on the
  column ‚Äî the explicit two-step delete (INV-9) is clearer, testable step by step,
  matches the codebase's explicit-SQL convention, and the plain FK then *blocks*
  an unsafe period-only delete (a safety net cascade would silently remove).
- **D9 ‚Äî A one-time, ambiguity-guarded backfill links statements imported before
  v6.** `_migrate_to_v6`, after adding the column, **stamps existing transactions
  it can attribute unambiguously** ‚Äî a transaction under **exactly one** period
  (same account, date in span) is stamped; one under **zero or ‚â•2** periods stays
  `NULL`. The overlap exclusion lives **in the query**, not a follow-up prose
  rule ‚Äî for each `statement_periods` row `p` (bound params shown as `:name`):
  ```sql
  UPDATE transactions SET statement_period_id = :p_id
   WHERE account_id = :p_account
     AND occurred_on BETWEEN :p_start AND :p_end
     AND statement_period_id IS NULL
     AND NOT EXISTS (          -- skip a date also covered by another period of this account
           SELECT 1 FROM statement_periods q
            WHERE q.account_id = transactions.account_id
              AND q.id <> :p_id
              AND transactions.occurred_on BETWEEN q.period_start AND q.period_end)
  ```
  So an overlap never mis-attributes (the exact bug INV-9d guards). The loop is
  order-independent: the `IS NULL` guard stops an already-claimed row being
  re-touched, and a date under ‚â•2 periods is skipped by *every* period's
  `NOT EXISTS`. This links a user's existing
  single-statement imports (the common case, no overlap) so their count + delete
  work, while never cross-attributing between overlapping statements. **Residual,
  documented caveat:** a *manual* transaction dated inside an imported statement's
  period is indistinguishable from that statement's rows and *will* be linked to
  it (and thus removed if that statement is later deleted) ‚Äî mitigated by INV-10's
  count-naming confirmation. This is inherent to reconstructing provenance that
  was never recorded; new imports (post-v6) are exact. The backfill runs **inside
  `_migrate_to_v6`'s atomic transaction** (all-or-nothing with the column add).
