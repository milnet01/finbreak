icon toolbar, status bar, and a **single-content-at-a-time** central
`QStackedWidget` (Home *or* Accounts *or* Categories *or* Import, each destroyed
on swap). The user now wants a **tabbed workspace** — Home · Statements ·
Accounts · Categories as **persistent tabs** — so they can see *what has been
imported* and move between views without losing their place, plus a **Home
toolbar button** and **remembered window geometry** (size, position, last tab)

came from* — a datum it does not currently keep — so this phase adds a
**nullable `transactions.statement_period_id`** column (v5→v6) and stamps it at
import time, with a one-time backfill for statements imported before the column
existed.

## Context

- A menubar (`File` · `View` · `Help` · `Donate`), an icon toolbar
  (`Manual entry` · `Import` · `Accounts` · `Categories` · `Lock`), a central
  `QStackedWidget` (`self._content`) holding `placeholder_welcome` /
  `placeholder_locked` plus **one** dynamic content widget (`self._live`), and a
  status bar with a permanent transaction-count `QLabel`.
- A state machine: `_show_first_run` / `_show_unlock` / `_enter_unlocked` /
  `_lock` / `_show_home` / `_open_manual_entry` / `_open_accounts` /
  `_open_categories` / `_open_import`, with `_set_live(widget)` / `_clear_live()`

  `_open_categories` / `_open_import` build a fresh `AccountsWidget` /
  `CategoriesWidget` / `ImportWizardWidget` and `_set_live` it; each content

- `service.on_auto_lock = self._lock`; `_lock` closes any dialog, `lock()`s,
  `_clear_live()`s, shows 🔒 Locked, disables the vault chrome, re-opens
  `UnlockDialog`.

## Design decisions baked into this spec

**D5 (import-as-flow) and D8/D9 (the provenance data model + backfill) carry the
load-bearing calls;** the rest are UI-structure/idiom.

- **D1 — The four views become persistent tabs in a `QTabWidget`; that widget is
  the single "live" content, destroyed on lock.** Replace the
  build-fresh-widget-per-view model (for Home/Statements/Accounts/Categories)
  with **one `QTabWidget`** built once per unlocked session by
  `_build_workspace()` and installed as `self._live` via the **existing**
  `_set_live` (so lock's `_clear_live` destroys the whole workspace — INV-3, no
  new teardown path). Navigation (tab bar / View menu / toolbar) calls
  `workspace.setCurrentIndex(i)` — it **does not** rebuild a tab. Rejected alt:
  keep destroy-on-swap and fake tabs with buttons (loses real tab affordance,
  reinvents `QTabWidget`); a second persistent stack outside the security
  teardown (risks a live data tab surviving a lock).
- **D2 — `_show_home` / `_open_accounts` / `_open_categories` become
  tab-switches; the `done`-returns-to-Home wiring retires for these three.**
  Because Accounts/Categories are now always-present tabs, their `done` "back to
  Home" affordance is meaningless (INV-12) — the shell no longer connects
  `done → _show_home`; it builds them with `show_done=False`. `_show_home(...)`
  is replaced by "switch to `tab_home`" (+ an explicit `home.refresh()` where a
  mutation requires it, INV-11).
- **D3 — Navigation actions switch tabs; Manual entry stays a dialog, Import
  stays a flow.** `action_home` / `action_accounts` / `action_categories` (menu
  **and** toolbar, one `QAction` each) + `action_statements` (menu only) call
  `workspace.setCurrentIndex`. `action_manual_entry` opens the modal
  `ManualEntryDialog` over the workspace (workspace stays alive underneath); on
  its `committed` signal the shell refreshes the Home tab + count and switches to
  it (INV-11). `action_import` runs the import flow (D5).
- **D4 — Cross-tab freshness via `currentChanged` + targeted post-mutation
  refresh.** The workspace connects `QTabWidget.currentChanged` to refresh the
  **activated** tab (Home/Statements re-read; Accounts/Categories re-read) — so
  data changed elsewhere shows when a tab is next viewed, with **no** need to
  wire every mutation to every tab (coding.md § 1.1). Mutations that must reflect
  *without* a tab switch — manual-entry commit, import `done`, statement delete —
  additionally refresh Home + the status count explicitly (INV-11). Re-reading
  the vault per tab activation is cheap at this app's data sizes; a caching layer
  would be premature.
- **D5 — Import is a full-content flow that replaces the workspace and returns to
  it, NOT a tab.** An import is a multi-step wizard that wants the full content
  area; making it a tab (a persistent half-filled wizard) is wrong. So
  `_open_import` **`_set_live`s the `ImportWizardWidget`** — which (via
  `_clear_live`) **destroys the workspace** while importing (a *good* thing for
  INV-3: fewer decrypted rows alive) — and on the wizard's `done` signal
  **rebuilds a fresh workspace and lands on the Statements tab** (so the just-
  imported statement is visible), refreshing Home + count. This is the *exact*
  FIBR-0051 destroy-on-swap pattern; the only new part is "rebuild workspace +
  select Statements" in the `done` handler instead of "show Home". A lock while
  the wizard is showing destroys the wizard (it is `self._live`), INV-3 intact.
- **D6 — Home toolbar action + a hand-authored house SVG.** Add `action_home`
  (`objectName` `action_home`, already exists on the `View` menu from FIBR-0051 —
  **the same single `QAction`** is added to the toolbar too, no duplicate) with a
  new `ui/icons/home.svg` (monochrome, neutral mid-tone, the FIBR-0051 D7 style),
  first in the toolbar. `ToolButtonTextUnderIcon` keeps the text label (the
  meaning-carrier). The new icon travels into the FIBR-0003 bundle as package
  data exactly like the FIBR-0051 glyphs (Deliverable 10).
- **D7 — Window geometry persisted via `QSettings`, in a plain INI outside the
  vault.** `paths.window_settings_path()` returns `data_dir() / "window.ini"`
  (a new sibling of the vault, **not** encrypted — geometry is non-sensitive and
  must load *before* unlock). The shell reads it in `__init__` (after building
  chrome, before showing) via `QSettings(str(path), QSettings.Format.IniFormat)`:
  `restoreGeometry(bytes)` (size + position), `restoreState(bytes)` (toolbar
  state), and `setCurrentIndex(last_tab)` **applied when the workspace is built**
  (the tab index is meaningless while locked). It **writes** them in an overridden
  `closeEvent` (geometry + state + current tab, then `super().closeEvent`) and on
  each `currentChanged` (so the last tab survives a crash). Tests inject an
  explicit path via monkeypatching `paths.window_settings_path`, mirroring how
  the vault tests inject `vault_path`. Rejected alt: native-format `QSettings`
  (org/app registry/plist) — an explicit INI is easier to point at a `tmp_path`
  and to reason about ("it's this one file").
- **D8 — v5→v6: a nullable `transactions.statement_period_id` FK, stamped at
  import.** One migration step `_migrate_to_v6` (the exact atomic-`BEGIN` /
  `ADD COLUMN` / `UPDATE schema_version` / `COMMIT` shape of `_migrate_to_v5`):
  `ALTER TABLE transactions ADD COLUMN statement_period_id INTEGER REFERENCES
  statement_periods(id)`. A nullable `ADD COLUMN` with a plain (non-cascade)
  `REFERENCES` is **verified** valid SQLite (SQLite requires the added FK column
  default to be `NULL` — which it is). `commit_import` is **reordered** so the
  `statement_periods` row is created (or its id looked up when the span already
  exists) **before** the transaction batch, and `add_batch` stamps every row with
  that id. `add` (manual) stays `NULL`. Rejected alt: `ON DELETE CASCADE` on the
  column — the explicit two-step delete (INV-9) is clearer, testable step by step,
  matches the codebase's explicit-SQL convention, and the plain FK then *blocks*
  an unsafe period-only delete (a safety net cascade would silently remove).
- **D9 — A one-time, ambiguity-guarded backfill links statements imported before
  v6.** `_migrate_to_v6`, after adding the column, **stamps existing transactions
  it can attribute unambiguously**: for each `statement_periods` row, `UPDATE
  transactions SET statement_period_id = <period.id> WHERE account_id =
  <period.account_id> AND occurred_on BETWEEN <period.period_start> AND
  <period.period_end> AND statement_period_id IS NULL` — **but skipped for any
  transaction whose (account, date) is covered by more than one period** (so an
  overlap never mis-attributes). Concretely: pre-compute, per account, the dates
  covered by ≥2 periods and exclude them; a transaction under exactly one period
  is stamped, one under zero or ≥2 stays `NULL`. This links a user's existing
  single-statement imports (the common case, no overlap) so their count + delete
  work, while never cross-attributing between overlapping statements. **Residual,
  documented caveat:** a *manual* transaction dated inside an imported statement's
  period is indistinguishable from that statement's rows and *will* be linked to
  it (and thus removed if that statement is later deleted) — mitigated by INV-10's
  count-naming confirmation. This is inherent to reconstructing provenance that
  was never recorded; new imports (post-v6) are exact. The backfill runs **inside
  `_migrate_to_v6`'s atomic transaction** (all-or-nothing with the column add).
- **D10 — Delete lives in a new `StatementService`, not `ImportService`.** The
  Statements tab's read (`list_statements`) + delete (`delete_statement`) are a
  distinct responsibility from the import pipeline; a small
  `services/statements.py` keeps `ImportService` focused (it is already large) and
  gives the read/delete their own testable seam. `StatementService` composes
  `StatementPeriodRepository` + `TransactionRepository` + `AccountRepository`
  (for names). Rejected alt: bolt onto `ImportService` (mixes import-flow with
  statement-management; coding.md § 1.1 favours the focused split).
- **D11 — `StatementsWidget` is a plain read+delete tab.** A `QTableWidget`
  (account · period · file · imported · count, columns by fixed index) + a
  `Delete selected` `QPushButton` (`button_delete_statement`, disabled when no row
  is selected) + a `changed` signal emitted after a successful delete. No add /
  edit (statements are created by importing). `_format_amount` is **not** needed
  (no money column — a count, not an amount). It exposes `statement_count()` and
  the selected row's period id for testing.
- **D12 — `AccountsWidget` / `CategoriesWidget` gain `show_done=True` (default).**
  When `False`, the `Done` button + its `done` connection are **not built** (the
  signal attribute stays, unfired). Default `True` keeps every existing
  construction site (and their tests) working unchanged (INV-12). This is the
  single edit to those two widgets.
- **D13 — No new pip dependency; a schema migration but no new native lib.** The
  house SVG is a static asset (package data, like FIBR-0051 D7). `QSettings` /
  `QTabWidget` / `QScreen` are core Qt (already bundled). So there is **one**
  schema-version test ripple (`== 5` → `== 6`, D8) and **no** new bundle surface;
  the FIBR-0003 build smoke's role is again only to prove the (one new) icon
  package-data travels (Deliverable 10 / DoD #2).
- **D14 — The FIBR-0051 `app_shell` test ripple is named, not discovered late.**
  Reshaping the content model (single-live-widget-per-view → one persistent
  tab-workspace) **changes** several FIBR-0051 `app_shell` assertions that keyed
  on the old model: INV-6a (Accounts action *builds+shows* `AccountsWidget` in
  the stack → now *switches to* the Accounts **tab**, widget instance stable);
  INV-9 (manual-entry commit *builds a fresh* `HomeView` → now *refreshes + shows*
  the Home **tab**); INV-4a (lock destroys the *content widget* → now destroys the
  *workspace `QTabWidget`*); and any assertion that `_live` is a `HomeView`
  (→ it is the workspace, whose *current tab* is Home). These are **edited** in

## Verified API basis (empirically checked 2026-07-09, § 13)

- **SQLite/SQLCipher:** `ALTER TABLE transactions ADD COLUMN statement_period_id
  INTEGER REFERENCES statement_periods(id)` succeeds (nullable FK column);
  `PRAGMA foreign_keys = ON` is set on every connection (`vault.py:142`); a
  direct `DELETE` of a `statement_periods` row with stamped children raises
  `IntegrityError`; the ordered two-step delete (children then parent) removes
  only stamped rows and leaves `NULL`-stamped (manual) rows; `cursor.lastrowid`
  gives the new period id.
- **Qt (PySide6 6.11.1):** `QSettings(path, IniFormat)` round-trips `QByteArray`
  + int and writes a plain INI file; `QMainWindow.saveGeometry/restoreGeometry`
  (a 66-byte `QByteArray`, restores `True`) + `saveState/restoreState`;
  `QTabWidget.addTab`/`setCurrentIndex`/`currentIndex`/`currentChanged`/`tabText`;
  `QGuiApplication.primaryScreen().availableGeometry()` +
  `frameGeometry().moveCenter(...)` for centering.
- **Data-layer seams (read 2026-07-09):** `commit_import`
  (`services/import_.py:173`) inserts the batch then the period row — reordered
  here; `TransactionRepository.add_batch` (`:62`) / `add` (`:23`);
  `StatementPeriodRepository.add`/`exists`/`list_for_account`
  (`repositories/statement_periods.py`); `Transaction`/`list_all`'s explicit
