# FIBR-0029 — Password hint shown before unlock

> **Status:** Draft — contract for FIBR-0029 (Kind: feature, lanes:
> crypto, ux). Source: user-request-2026-07-01; design forks resolved by
> user 2026-07-20 (see § 3). Depends on FIBR-0004 (unlock dialog) and the
> plaintext `window.ini` store (FIBR-0095 precedent). Cold-eyes: § 7
> (project cadence `--max-loops 7`). Touches `security-model.md` (a new
> plaintext artefact) — that edit runs through `/cold-eyes` too (§ 14).

## 1. Problem & intent

Users forget master passwords. finbreak has **no backdoor** (T11) — a
forgotten master is unrecoverable except via the encrypted backup. An
optional **hint** is a cheap memory aid that reduces how often users hit
that wall. It is **a memory aid, not a recovery method**.

The hint must be visible **before** the vault is decrypted (that's when
you've forgotten the password), so it **cannot** live in the encrypted
DB. It lives in the plaintext `window.ini` sibling — the same store that
already holds pre-unlock state (unlock-throttle counters, FIBR-0095;
window geometry). **Anyone with device access can read it.** Two
consequences drive the whole design: (a) the user must be **warned** it
is stored unencrypted, and (b) the hint must be **enforced not to be, or
contain, the password** — otherwise the "hint" is a plaintext copy of the
secret.

## 2. Scope

**In scope**
- An optional, user-set, plaintext password hint stored in `window.ini`.
- **Set** it from Settings via a "Set password hint…" button that
  requires the **current master password** (§ 3.2).
- **Show** it on the unlock screen behind a "Show hint" button (hidden
  until clicked, § 3.3).
- Enforcement that the hint is never equal to, nor contains, the password
  (§ 3.4).
- A new `AuthService.verify_password` primitive (§ 3.5).
- Record the new plaintext artefact + enforcement in `security-model.md`
  (§ 3.6).

**Out of scope**
- Any *recovery* of the password from the hint (it's a reminder only).
- A password-strength meter or a "change master password" feature (none
  exists today; not needed here).
- Per-vault hint namespacing — finbreak is **single-vault per OS user**
  (`paths.vault_path()`), so the hint is one global `window.ini` key.
- Encrypting or obfuscating the hint (pointless — it must be readable
  pre-unlock; the mitigation is the enforcement + the warning, not
  secrecy).

## 3. Design

### 3.1 Storage (plaintext window.ini)

A single key in the existing plaintext store
(`paths.window_settings_path()` → `window.ini`), mirroring the
FIBR-0095 throttle adapter (`ui/_unlock_throttle.py`):
- Key: `hint/text` (string). Absent/empty/whitespace-only ⇒ **no hint
  set**.
- I/O lives in a small adapter `ui/_password_hint.py` (QSettings
  read/write), the ui analogue of `_unlock_throttle.py`:
  `read_hint() -> str` (returns `""` when unset; **callable pre-unlock,
  needs no key**), `write_hint(text: str)`, `clear_hint()`. Each write
  calls `.sync()` (same as the throttle adapter, so a same-process read
  sees it).
- Rationale for splitting I/O into a ui adapter: `AuthService` (services
  layer) must not import Qt-`ui`; the unlock dialog (ui) reads the hint
  pre-unlock and Settings (ui) writes it, so QSettings I/O belongs in ui.
  The **pure validation** (§ 3.4) lives in a services module so it is
  testable without Qt — the same split as `services/unlock_throttle.py`
  (math) vs `ui/_unlock_throttle.py` (I/O).

### 3.2 Setting the hint (Settings → confirm password → save)

Follows the Settings "Export backup…" pattern exactly (recon Q5): the
`SettingsDialog` shows a **"Set password hint…"** button in a form row
(`form.addRow(self.tr("Password hint"), self._set_hint)`) and emits a
`set_hint_requested` Signal; the **shell** (`main_window.py`) owns the
sub-dialog and the work (SettingsDialog holds no AuthService ref, like
backup).

The shell opens a small `SetHintDialog` (modelled on
`ui/backup_export.py`'s validation-gated dialog):
- A **hint** field (plain `QLineEdit`, max 100 chars) — pre-filled with
  the current hint if one is set (so "edit" and "set" are one flow).
- A **current-password** field (`QLineEdit`, `EchoMode.Password`).
- A visible **plaintext-storage warning** ("This hint is saved
  unencrypted and can be read by anyone with access to this device. Never
  put your password in it.").
- An "Empty the hint field + Save" path clears the hint (`clear_hint()`).

On Save the shell:
1. `AuthService.verify_password(pw)` — reject with an inline error if the
   password is wrong (§ 3.5). This both **authorizes** the change and
   yields the verified plaintext to check against.
2. `validate_hint(hint, pw)` (§ 3.4) — reject with an inline error naming
   the reason (too long / equals-or-contains the password).
3. `write_hint(hint)` (or `clear_hint()` if the field is empty).
4. Wipe the password buffer (§ 3.5 discipline).

### 3.3 Showing the hint (unlock screen, reveal-on-click)

`ui/unlock.py` `UnlockDialog` (recon Q1): after the error label
(`unlock.py:80`), add a flat **"Show hint"** button modelled on the
existing `_restore_button` (`:66-70`). At dialog build, call
`read_hint()`:
- **Empty/unset** ⇒ the "Show hint" button is **not added** (no affordance
  when there's nothing to show).
- **Set** ⇒ clicking "Show hint" reveals the hint text in a label below
  the button (and the button hides itself, or toggles to "Hide hint").
  The hint is read **once at build**, pre-unlock, needing no key (INV-3).

The hint is **display-only** and never affects unlock logic, the password
field, or the throttle.

### 3.4 Enforcement — the hint is never the password (pure, testable)

`services/password_hint.py` (no Qt, no I/O) — the pure contract:

```python
MAX_HINT_LEN = 100

def validate_hint(hint: str, password: bytearray) -> None:
    """Raise HintPolicyError if the hint is unusable. A blank hint
    (empty / whitespace-only) is caller-handled as 'clear', not passed
    here."""
```

Rules (raise `HintPolicyError` with a specific message on any):
- **Length:** `len(hint) > MAX_HINT_LEN` → too long.
- **Equality:** `hint.strip()` equals the password (compared
  case-**insensitively** after strip) → "the hint may not be your
  password".
- **Containment:** the password appears **inside** the hint
  (case-insensitively) → "the hint may not contain your password". To
  avoid a pathological short-password case (a 1–3 char password would
  reject almost any hint), the **containment** check applies only when
  `len(password) >= 4`; the **equality** check always applies. *(A weak
  short password is its own problem; we don't let it make the hint UI
  unusable, but we never allow an exact copy.)*

The comparison decodes the `bytearray` password to `str` for the check,
then the caller wipes the buffer. Case-insensitive is deliberately
**stricter** than case-sensitive (a hint that is the password in another
case still substantially leaks it).

### 3.5 `AuthService.verify_password` (new primitive)

Recon Q2: the vault is already open in Settings, `self._key`
(`auth.py:172`) holds the raw Argon2id key, and there is **no** verify
primitive. Add:

```python
def verify_password(self, password: bytearray) -> bool:
    """True iff `password` derives the key this vault was unlocked with.
    Re-runs the pinned KDF and compares in constant time. Requires the
    vault to be unlocked (self._key present)."""
```

- `raw = derive_raw(password, self.load_params())` (`auth.py:151/243`),
  then `hmac.compare_digest(raw, self._key)` (constant-time), then wipe
  `raw`. Returns the bool.
- Buffer discipline mirrors `derive_raw`/`validate_first_run`: the caller
  owns `password` and wipes it in a `finally`; `verify_password` wipes its
  own derived `raw`.
- If called while locked (`self._key is None`) it raises
  `VaultLockedError` (consistent with other key-requiring methods) — but
  the Set-hint flow is only reachable unlocked.

### 3.6 security-model.md impact (a new plaintext artefact)

Recon Q7. Two edits, consistent with the existing A1 plaintext note
(`security-model.md:26`) and INV-10 (`:240`):
- **A1 asset row** parenthetical: add the **password hint** to the list
  of things deliberately in plaintext `window.ini`, noting it is
  **user-authored** (so it can contain whatever the user types) and is
  **enforced not to be/contain the password** (INV below).
- **A new invariant INV-11** (§ 5): *"A stored password hint is never
  equal to, nor contains, the master password"* — enforced by
  `validate_hint` at set-time, behind a mandatory `verify_password` check
  so the comparison is against the real password. Falsifiable by test.
- No new threat row is required (the hint introduces no new *attacker*
  capability beyond A1's already-stated "plaintext window.ini is readable
  with device access"); the honest disclosure lives in A1 + INV-11.

## 4. Invariants

- **INV-1 — Hint is optional and off by default.** A fresh vault has no
  hint; the unlock screen shows **no** "Show hint" affordance until one is
  set. *Test:* fresh `window.ini` → `read_hint() == ""`; UnlockDialog has
  no visible show-hint button.
- **INV-2 — Hint never equals or contains the password.** `validate_hint`
  rejects a hint equal to (case-insensitively, stripped) or — for
  passwords ≥ 4 chars — containing the password. *Test:* password
  `"Groceries2026"`; hints `"groceries2026"`, `"see Groceries2026 note"`
  both raise `HintPolicyError`; `"Spar loyalty year"` passes.
- **INV-3 — Hint is readable pre-unlock, with no key.** `read_hint()`
  returns the stored string without the vault being open (it reads
  `window.ini`). *Test:* write a hint, lock/never-unlock, `read_hint()`
  returns it.
- **INV-4 — Setting a hint requires the correct current password.** The
  Set-hint flow calls `verify_password`; a wrong password blocks the save
  and writes nothing. *Test:* wrong password → `write_hint` not called,
  stored hint unchanged; correct password → saved.
- **INV-5 — `verify_password` is correct and constant-time.** Returns
  `True` only for the unlocking password; uses `hmac.compare_digest`;
  wipes its derived key. *Test:* correct→True, wrong→False; (source
  backstop) asserts `compare_digest` is used, not `==`.
- **INV-6 — Length cap.** A hint longer than `MAX_HINT_LEN` (100) is
  rejected. *Test:* 101-char hint raises; 100-char passes.
- **INV-7 — Clearing works.** Saving an empty/whitespace-only hint field
  removes the hint (`clear_hint()`), and the show-hint affordance
  disappears. *Test:* set then clear → `read_hint() == ""`.
- **INV-8 — Password buffers are wiped.** The Set-hint flow wipes the
  entered password `bytearray` after use (verify + validate), in a
  `finally`. *Test:* the buffer is zeroed after the flow (mirror the
  existing `validate_first_run` wipe test).
- **INV-9 — Hint is display-only.** Showing/reading the hint never
  mutates the password field, the unlock result, or the throttle state.
  *Test:* reveal the hint, assert the password field + throttle counters
  are untouched.

## 5. Exit criteria (falsifiable)

1. **Set → show round-trip.** With a created vault: open Settings → Set
   password hint… → enter hint + correct password → Save; then open the
   unlock dialog → "Show hint" reveals exactly that hint (INV-1/3/4).
   Falsifier: hint absent, or shown without a set, or a wrong password
   still saving.
2. **Enforcement holds.** `validate_hint` rejects equal / containing /
   over-length hints and accepts a safe one (INV-2/6); the wrong-password
   path saves nothing (INV-4). Falsifier: a hint equal to or containing
   the password is accepted, or a wrong password saves.
3. **Pre-unlock readability.** `read_hint()` returns the stored hint with
   the vault locked (INV-3). Falsifier: it needs the key or returns "".
4. **verify_password.** Correct password → True, wrong → False,
   constant-time compare, derived key wiped (INV-5). Falsifier: a wrong
   password returns True, or a non-constant-time `==` is used.
5. **Clear + default-off.** No hint on a fresh vault; clearing removes it
   and the affordance (INV-1/7). Falsifier: a stale hint survives a
   clear, or an affordance shows with no hint.
6. **security-model.md updated + cold-eyes-clean.** A1 lists the hint as a
   plaintext artefact; INV-11 states the never-equals/contains rule; the
   doc edit passes its own `/cold-eyes` (§ 3.6, § 14). Falsifier: the
   plaintext artefact is undocumented.
7. **Gate green.** `./scripts/ci-local.sh` passes (ruff, format, bandit,
   pip-audit, gitleaks, pytest, mypy), including the new
   `tests/features/password_hint/` suite (spec.md + tests) and any unlock
   / settings tests touched.
