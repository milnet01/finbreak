**Theme:** on Windows, make the FIBR-0054 updater actually *install*. Today
`detect_installer()` returns `None` off an AppImage (INV-7 there), so a frozen
Windows `.exe` sees the prompt path only if wired — and it is not: Windows is
inert. This feature wires it. The whole check → Ed25519-verify → **Later / Skip /
Update now** flow is already cross-platform; only the **install hand-off** is
platform-specific, and the `Installer` protocol was built (D6 of FIBR-0054) as the
exact plug-in point for this — the module docstring already names `WindowsInstaller`.

**Why a helper process (the crux).** A running Windows `.exe` is **locked by the
OS** — it cannot be overwritten in place while running, and this is a PyInstaller
**onefile** build (it has extracted itself to `%TEMP%\_MEIxxxxxx`, cleaned on
exit). So the Linux trick — `os.replace` the file *then* relaunch, all in-process
— cannot be copied. Instead (user-approved 2026-07-13): finbreak wipes its key,
spawns a **detached helper**, and exits; the helper **waits until no process is
running the `.exe` image** (releasing the file lock, letting the onefile bootloader
clean its `_MEIxxxxxx` dir — D3 pins that the wait is by exe **image path**, not a
PID), **moves the verified new `.exe` over the old path** (retrying, since a released
lock can lag), then
**relaunches it**. This is the Windows twin of the FIBR-0122 `/bin/sh` waiter, and
it watches the same `_MEI`-teardown race (D4).

## Invariants

| INV | What it pins |
|-----|--------------|
| **INV-1** | **Platform detection is exact and fail-inert.** `detect_installer()` returns a `WindowsInstaller(Path(sys.executable))` **only** when `sys.platform == "win32"` **and** `getattr(sys, "frozen", False)` is truthy (a real frozen `.exe`); a non-frozen `python -m finbreak` run on Windows, and every non-Windows run, are unaffected (AppImage on a real AppImage, else `None`). Off a resolved installer the feature is inert — no startup check, no prompt, Settings checkbox disabled + tooltipped, and the **Help → Check-for-updates** action reports *"Automatic updates aren't available for this build of finbreak."* rather than a misleading "up to date" (already the shell's `self._installer is None` behaviour, `main_window.py`; parity with FIBR-0054 INV-7). Tests monkeypatch `sys.platform` / `sys.frozen` / `sys.executable` and assert the returned type (or `None`). |
| **INV-2** | **Asset selection is installer-driven, not hardcoded.** The picker matches the release asset ending in the **active installer's** `asset_suffix()` — `-x86_64.exe` for `WindowsInstaller`, `-x86_64.AppImage` for `AppImageInstaller` — plus the asset named exactly that `+ ".sig"`; absent-or-duplicate either → `None` (fail safe, parity with FIBR-0054 D14). A release carrying **both** a `.AppImage` and an `.exe` asset picks the `.exe` under a Windows installer and the `.AppImage` under a Linux one. With **no** installer, `check_for_update` returns `None` (nothing is installable, so nothing is offered — this removes the previously-hardcoded AppImage suffix). |
| **INV-3** | **The Windows swap is out-of-process; the running `.exe` is never overwritten in place.** `WindowsInstaller.apply(new_file, on_before_exec)` does **not** move any file itself. It runs `on_before_exec()` (key-wipe), spawns a **detached** helper, and `os._exit(0)`s. The helper (a PowerShell `-Command` one-liner with a **fully-specified argv**, D3) waits (bounded, ~60 s) until **no live process is running the target `.exe` image** — a poll on the exe *image path*, not a single PID (D3 explains why the PID approach is fragile for a onefile process-tree + Windows PID-recycling). This image-free wait is the guarantee; the short `Move-Item` retry only absorbs post-wait AV/indexer lag. It then **retries** `Move-Item -Force` of the verified `new_file` over `target_path()` on a fixed schedule (**5 tries × 0.5 s** provisional, tuned on Windows). It relaunches `target_path()` **only if the image actually became free** (a final image-free re-check after the wait); if the ~60 s cap elapsed with a process *still running the image*, the helper **aborts** — **unlinks the staged temp**, no move, no relaunch — so it can never spawn a second instance over a still-running first, and doesn't orphan the temp. **Degraded outcome of an abort:** because `apply` already `os._exit`'d the (GUI-owning) child and wiped the key, an abort leaves the user with **no *usable* finbreak** — only the wedged onefile parent bootloader that tripped the abort (its GUI child already exited) — and a wiped key, until they manually relaunch the old `.exe`. Accepted (a wedged-past-60 s old instance is already an anomaly). Tests assert the **command** and **env** builders (pure string/list construction) and the `apply` ordering with `Popen` + `os._exit` monkeypatched; the live swap is proved on Windows (§ To verify empirically). |
| **INV-4** | **(Security.) The derived key is wiped before this process hands off — and a spawn failure never strands it wiped.** Parity with FIBR-0054 INV-6, with one deliberate, documented asymmetry: on Linux the `os.replace` runs **in `apply`, before** the wipe; on Windows there is **no in-process replace**, so `apply` **spawns the detached helper FIRST, then wipes, then `os._exit(0)`**. The helper blocks until this `.exe` image frees (i.e. until this process exits), so it does nothing before the wipe — but ordering the wipe **after a *successful* spawn** means a spawn failure (e.g. AV/AppLocker denies `powershell.exe`) is **non-destructive**: the temp is unlinked, the key stays intact, the session survives, and an `UpdateError` surfaces — rather than a `NoReturn` method throwing past `os._exit` and leaving a live GUI with a wiped key. The wipe is still before the *relaunch* (which the helper does only after we exit), so INV-4's real requirement holds. (The Linux twin, whose `os.replace` has already committed by the time it spawns, guards its spawn too: on a `Popen` failure it hard-exits anyway so the swapped-in AppImage is picked up by a manual relaunch — never a wiped-key zombie.) Tests assert spawn-then-wipe-then-exit ordering on Windows, no in-process swap, and that a spawn failure leaves the key un-wiped + the temp cleaned. |
| **INV-5** | **(Robustness.) The relaunch resets the onefile bootloader; no POSIX loader fixups.** `_windows_relaunch_env()` returns `dict(os.environ)` with **`PYINSTALLER_RESET_ENVIRONMENT=1`** set — PyInstaller 6.10+'s official restart signal, so the fresh onefile bootloader re-extracts instead of colliding with the old process's soon-cleaned `_MEIxxxxxx` dir (the Windows analogue of the FIBR-0122 `_MEI` race). **The flag must reach the *relaunched* `.exe`, which the helper starts one hop away** (`Start-Process`, not an in-place `exec` like Linux): the env is set on the helper's `Popen`, and `Start-Process` inherits its parent (the helper's) environment, so the flag propagates to the new finbreak — this inheritance is the mechanism, and it is a named empirical check (§ To verify empirically). It does **not** touch `LD_LIBRARY_PATH` / `LD_PRELOAD` (POSIX-only loader concerns) or the AppImage runtime markers — those are Linux-specific. Test: the env builder sets exactly the reset flag and hand-edits no private `_PYI_*` var. |
| **INV-6** | **(Robustness.) A non-creatable install directory fails safely, before the point of no return.** The verified download is staged as a temp **in `target_path().parent`** (`download_and_verify`, extended only to take the temp's file-extension from the installer — INV-2/D2) — that staging is a *dir-creatability* probe: a read-only / full install dir raises at download time, is cleaned up, and is surfaced (`UpdateError`) with the original `.exe` byte-for-byte intact and the key **not** wiped (parity with FIBR-0054 INV-5 for the intact binary, **FIBR-0054 INV-6** for the un-wiped key). **This probe proves the dir accepts a *new* file — it does NOT prove the running `.exe` itself can be *overwritten*** (a different right: delete/replace on that specific file, plus a released lock). So two residuals remain, both **after** the key-wipe + exit and both **non-destructive** *to data* (the old `.exe` stays; the update re-offers next launch): (a) a *transient* lock defeating the helper's retry-capped `Move-Item`; (b) a *permanent* permission/ACL mismatch (the file is protected, or the location needs elevation). **Both fail *silently* — unlike Linux, where a failed swap raises `UpdateError` in-process *before* handoff, the Windows move happens after this process is gone, so the helper relaunches the *old* `.exe`, which has no knowledge the swap failed and surfaces no error.** For residual (b) specifically, that means every future **"Update now"** wipes the key (a forced re-login) and makes **zero** progress — a silent no-op loop whose escape is **"Skip this version"** (or turning the
auto-update opt-in off). (The launch-time *check* itself never wipes the key — only the `apply` path reached by an explicit "Update now" does — so the loop costs a re-login only when the user actively retries, not on every launch.) This is an accepted limitation, documented here rather than mitigated (a signal-back channel would need the helper to write a status the next launch reads — out of scope). On **either** a move that ultimately fails **or** a timeout-abort (INV-3 — the image never freed), the helper **best-effort-unlinks the staged temp** (`Remove-Item -Force`) so a chronically-wedged instance doesn't accrete `finbreak-update-*.exe` files in the dir. Elevation-required locations (e.g. `C:\Program Files\`) are **out of scope** — finbreak is a portable single-`.exe` with no installer, expected in a user-writable location; there the swap simply won't apply. Documented accepted difference from Linux (which can keep the live session on a failed swap). |
| **INV-7** | **(Security.) The Windows download rides the same Ed25519 gate.** The release `.exe` is signed by `scripts/sign-release.py` (**unchanged** — it signs any file's exact bytes → `<name>.sig`), and `download_and_verify` verifies that raw 64-byte Ed25519 signature over the exact downloaded `.exe` bytes against the committed public key (`update_key.public_key()`), exactly as for the AppImage. A one-byte tamper of the `.exe` **or** the `.sig` → `UpdateVerificationError`, temps deleted, nothing installed. No new crypto — the FIBR-0054 verify path is reused verbatim; this INV pins that the picker + installer feed the `.exe` bytes through it. |
| **INV-8** | **No new network surface; the confinement holds.** Windows adds **no** networked module — the check + download reuse `services/update_fetch.py`, still the **sole** `urllib` importer. `WindowsInstaller` opens no socket and makes no request (it only spawns a local helper). The FIBR-0054 INV-12 `_network_offenders` source scan stays green with the new code present. |
| **INV-9** | **No new UI, no new blocking.** The prompt + progress path is the unchanged cross-platform `ui/update_dialog.py` / `ui/_update_worker.py` (no `.exec(`, auto-lock-safe — FIBR-0054 INV-9 owns this). This feature adds **zero** UI surface: the same dialog, driven by the same shell wiring, now resolves a `WindowsInstaller` instead of `None` on a frozen Windows build. |

## Design decisions

- **D1 — `WindowsInstaller` lives in the existing `services/update_installer.py`.**
  No new module: it is the *same* platform seam FIBR-0054 D6 built for exactly this.
  We add the `WindowsInstaller` class, the `_windows_relaunch_command` /
  `_windows_relaunch_env` module-level helpers (siblings of the AppImage ones), and
  extend `detect_installer()` with the frozen-Windows branch. `UpdateService` and
  the UI are untouched by D1 (they talk to the `Installer` protocol).

- **D2 — The `Installer` protocol gains `asset_suffix() -> str`; the picker is
  installer-driven.** `AppImageInstaller.asset_suffix()` → `"-x86_64.AppImage"`;
  `WindowsInstaller.asset_suffix()` → `"-x86_64.exe"` (matching the actual release
  asset name `finbreak-<v>-x86_64.exe`). `update.py`'s `_select_assets(assets,
  suffix)` takes the suffix as a parameter instead of the module constant
  `_APPIMAGE_SUFFIX`. `check_for_update` short-circuits to `None` **at the very top —
  before the opt-in gate and before any fetch — when `self._installer is None`**
  (nothing is installable on this platform, so there is nothing to offer and no
  reason to touch the network); otherwise it reads the suffix from
  `self._installer.asset_suffix()`. This is the minimal change that makes one picker
  serve both platforms and deletes the hardcoded AppImage assumption.
  **Two honesty-of-naming clean-ups that ride along (this feature is what makes the
  plumbing cross-platform, so the rename is in-lane, not drive-by):** (i)
  `download_and_verify` stages its download temp with the extension **derived from
  the active installer's suffix** — `Path(installer.asset_suffix()).suffix` yields
  `.exe` on Windows / `.AppImage` on Linux (no extra protocol method needed) — rather
  than the hardcoded `".AppImage"`, so a Windows download isn't a misleadingly-named
  `*.AppImage` temp; the `.sig` temp keeps its `.sig` extension. (ii)
  `UpdateInfo.appimage_url` → **`asset_url`** and the `appimage_tmp` local →
  `asset_tmp`, since the field now legitimately carries a `.exe` URL on Windows.
  **Ripple:** the existing `_select_assets` unit tests change signature (now pass a
  suffix); the `appimage_url` reads are **entirely inside `services/update.py`** (the
  `UpdateInfo` dataclass definition, its construction in `check_for_update`, and
  `download_and_verify`) plus `tests/features/auto_update/` — the UI never reads the
  field (`ui/_update_worker.py` passes the whole `UpdateInfo` to `download_and_verify`;
  `ui/main_window.py` only holds the type), so the rename's blast radius is
  `update.py` + tests. Deliberate refactors, surfaced in Test ripple, not silent drift.

- **D3 — The helper is a detached PowerShell `-Command` one-liner with a fully-pinned
  argv.** Rationale: Windows PowerShell 5.1 ships on every supported Windows (10/11);
  `-Command` executes inline and is **not** gated by the `.ps1` script execution
  policy; and `Get-Process` / `Start-Sleep` / `Move-Item` / `Start-Process` are
  clean, first-class primitives (vs a `tasklist`-poll `cmd` batch).

  **The exact argv `_windows_relaunch_command` returns** (mirroring the Linux twin's
  concrete `["/bin/sh", "-c", script]`, `update_installer.py:96`): a **resolved
  absolute** PowerShell path — `%SystemRoot%\System32\WindowsPowerShell\v1.0\powershell.exe`
  (from `os.environ["SystemRoot"]`, falling back to `shutil.which("powershell")`) —
  **not** a bare `"powershell"`, because a `DETACHED_PROCESS` child can run with a
  stripped `PATH` and fail to launch. Flags: `-NoProfile -NonInteractive -Command
  <script>`.

  **Which process the helper waits on (the correctness crux) — by image path, not a
  PID.** PyInstaller **onefile on Windows is a two-process tree** — a parent
  *bootloader* (`finbreak.exe`) that extracts `_MEIxxxxxx` and spawns a *child* that
  runs the app; the on-disk `finbreak.exe` write-lock is held while **either** runs,
  and the parent does the `_MEIxxxxxx` cleanup on exit. So waiting on a single PID is
  fragile: `os.getpid()` is only the child, `os.getppid()` is the bootloader **but
  Windows recycles PIDs**, so a PID that names the bootloader at spawn time can name an
  unrelated process a moment later. The helper therefore **waits by the exe *image
  path***: it polls until **no** live process is running `target_path()`, which is both
  **tree-agnostic** (waits for parent *and* child) and **recycling-proof** (it matches
  the file, not a number). This "wait until the image is free" is the load-bearing
  guarantee; the short `Move-Item` retry (below) only absorbs the brief AV/indexer
  lock-lag *after* the image is free — it does **not** rescue a wait that returned too
  early. **That the child's `os._exit` lets the parent finish `_MEIxxxxxx` cleanup and
  release the lock within the ~60 s cap is a HARD empirical gate** (§ To verify
  empirically). *(The helper is `powershell.exe`, never `finbreak.exe`, so it doesn't
  match its own poll; the relaunched instance is started only after the wait, so it
  can't either.)*

  **The script body:** a bounded poll —
  `while (Get-Process -ErrorAction SilentlyContinue | Where-Object { $_.Path -eq
  <exe> }) { Start-Sleep -Milliseconds 200 }`, capped at ~60 s (≈300 iterations) →
  **a final image-free re-check**: if a process running `<exe>` is **still present**
  (the cap elapsed on a wedged old instance), **abort** — `Remove-Item -Force
  <new_file>` so the staged temp isn't orphaned, then no move + no relaunch — so the
  helper can never `Start-Process` a *second* live instance over a still-running,
  still-locked first (the timeout edge INV-3 pins) → else a **5 tries × 0.5 s**
  `Move-Item -LiteralPath <new> -Destination <exe> -Force` retry (AV/indexer lock lag),
  and if that retry loop ultimately fails, `Remove-Item -Force <new_file>` too (INV-6
  no-accretion) → `Start-Process -FilePath <exe>` (relaunch even if the move ultimately
  failed — the old `.exe` is still there, INV-6). Paths are PowerShell-single-quote-escaped
  (double any embedded `'`).

  It is launched via `subprocess.Popen(..., creationflags=..., close_fds=True)` so it
  outlives finbreak's exit. **The Windows-only creationflags are referenced via
  `getattr(subprocess, "DETACHED_PROCESS", 0) | getattr(subprocess,
  "CREATE_NEW_PROCESS_GROUP", 0)` (add `CREATE_NO_WINDOW` the same way)** — those
  attributes **do not exist on Linux**, so a bare `subprocess.DETACHED_PROCESS` would
  raise `AttributeError` when the ordering test drives `apply()` on the Linux CI gate;
  the `getattr(..., 0)` fallback makes the flags a harmless `0` off-Windows so the
  monkeypatched `Popen` is still reached (Test ripple). **Diagnostics (Linux parity,
  `update_installer.py:111-122,183-199`):** the helper's `stdout`/`stderr` are
  redirected to the shared `update-relaunch.log` handle (`_relaunch_log_handle()`), so
  that under `--windowed` (no console) a failed live swap on the user's remote Windows
  box still leaves a post-mortem trail — without this the two-cycle validation has
  nothing to inspect. The argv is **our own** — the resolved PowerShell path,
  `sys.executable`, and our staged temp path, never user input — so the `subprocess`
  `bandit` B404/B603 suppressions mirror the Linux waiter's, with the constraint named
  in the comment.

- **D4 — Relaunch env = reset flag only.** `_windows_relaunch_env()` copies
  `os.environ` and sets `PYINSTALLER_RESET_ENVIRONMENT=1`. No `LD_*` restoration
  (POSIX-only), no AppImage-marker drop (Linux-only). The reset flag is what stops
  the relaunched onefile bootloader from treating itself as a worker of the exited
  parent and reusing the deleted `_MEIxxxxxx` dir. It is set on the **helper's**
  `Popen` env; the relaunched `.exe` is a further `Start-Process` hop, which inherits
  the helper's environment (INV-5) — so the flag reaches the new finbreak without an
  explicit re-set.

- **D5 — Add the Ed25519 `.sig` sidecar to the (already-attached) release `.exe`
  (a documented release step, no new code).** The `.exe` is **already a published
  release asset** — `v0.1.9` ships `finbreak-0.1.9-x86_64.exe` (built by
  `windows-build.yml` / `build-windows-exe.py` as a CI artifact, then attached to the
  release via a manual `gh release upload`). What is **missing** for the updater is
  the **`.exe.sig`** — `v0.1.9` has an `AppImage.sig` but no `exe.sig`, so
  `download_and_verify` would have nothing to verify. So D5's remaining work is: at
  release time, Ed25519-sign the release `.exe` with `scripts/sign-release.py`
  (**unchanged** — it signs any file's exact bytes → `<name>.sig`) and attach the
  resulting `finbreak-<v>-x86_64.exe.sig` next to the `.exe`. This is added to the
  release runbook (the `.claude/bump.json` release note / a release doc), not to
  product code — nothing to build beyond documenting + performing the sign-and-attach.

- **D6 — The two-cycle caveat is identical to Linux.** The *running* (old) version
  performs each relaunch, so the swap+relaunch only *executes* on the update
  **after** the one that first ships FIBR-0131. Ship it, then live-prove on the
  user's Windows machine at the next release. Unit tests cover everything provable
  off-Windows; the live swap is a manual Windows leg (§ To verify empirically), the
  same posture as [FIBR-0054](FIBR-0054.md)'s D8 relaunch mechanism (the two-cycle
  caveat itself is spelled out inline above; it was first hit by FIBR-0122's Linux
  relaunch fixes).

## Test ripple

**New legs (all runnable on Linux — pure logic + monkeypatched platform):**
- `detect_installer()` platform matrix: `win32 + frozen` → `WindowsInstaller`;
  `win32` not-frozen → `None`; non-win32 unaffected (monkeypatch `sys.platform`,
  `sys.frozen`, `sys.executable`).
- `asset_suffix()` on both installers returns the expected suffix.
- Asset-picker matrix: a release with **both** `.AppImage` and `.exe` assets →
  `.exe` under a Windows-suffix call, `.AppImage` under a Linux one; missing `.sig`
  → `None`; duplicate suffix → `None`.
- `check_for_update` with `installer is None` → `None` and **never calls the
  fetcher at all** (the short-circuit is above the opt-in gate — INV-2/D2; assert
  the injected fake fetcher records zero calls).
- `_windows_relaunch_command`: the argv[0] is an **absolute** `powershell.exe` path
  (not bare `powershell`) with `-NoProfile -NonInteractive -Command`; the script polls
  by **image path** (`Get-Process` + a `$_.Path -eq <exe>` filter — no `-Id`/PID), does
  a final image-free re-check with an **abort branch** that `Remove-Item -Force`s the
  staged temp, runs a **bounded** `Move-Item -Force` retry loop, and `Start-Process`es
  the exe; a path with a space / embedded `'` is single-quote-escaped. The retry
  count/interval + the poll cap (`5 × 0.5 s`, ~60 s provisional, D3) are **tuned on
  Windows**, so the test asserts the *shape* of the poll + retry, **not** the literal
  numbers — it must not couple the suite to values the spec expects to change.
- `_windows_relaunch_env`: sets `PYINSTALLER_RESET_ENVIRONMENT=1`, edits no `_PYI_*`.
- `WindowsInstaller.apply` ordering: with `Popen` + `os._exit` monkeypatched, the
  key-wipe callback fires **before** the spawn, the spawn is detached, and no
  in-process file move occurs; a staging failure path leaves the wipe uncalled.
- INV-7 (Ed25519 gate over `.exe` bytes): the verify path is the **unchanged**,
  byte-agnostic FIBR-0054 `download_and_verify` (its existing one-byte-tamper →
  `UpdateVerificationError` leg covers any payload), so the *new* `.exe`-specific
  coverage is the asset-picker leg feeding an `.exe` URL/bytes through it (above) — no
  duplicate crypto leg is added; the transitive coverage is stated so it isn't a gap.
- Confinement: the existing `_network_offenders` scan (FIBR-0054 INV-12) stays green.

**Changed legs (deliberate refactor — D2):**
- The existing `_select_assets` tests move from the module-constant suffix to the
  parameterised `_select_assets(assets, suffix)`; the `_APPIMAGE_SUFFIX` module
  constant is replaced by `AppImageInstaller.asset_suffix()` (it does not survive as a
  standalone constant), so any test naming it is re-pointed there.
- Tests reading `UpdateInfo.appimage_url` move to `asset_url` (the `update.py`
  construction + `download_and_verify` round-trip legs; the UI has no such reads).

## Out of scope

- **Authenticode / SmartScreen "trusted publisher" signing** → **FIBR-0133**
  (SignPath Foundation OSS program, blocked on external approval). The same release
  `.exe` gains that second signature there; the Ed25519 gate here is independent and
  does not wait for it. An unsigned self-swapping `.exe` is what Defender/SmartScreen
  distrusts most, so the user may see a one-time warning until FIBR-0133 lands —
  acknowledged, not a blocker (sequencing decision 2026-07-14).
- **Rollback / post-update health-check / update watchdog** — accepted risk, parity
  with FIBR-0054's Out-of-scope (single-key, no A/B slots).
- **Elevation-required install locations** (e.g. `C:\Program Files\`) — finbreak is a
  portable single-`.exe` with no installer, expected in a user-writable folder; in a
  protected location the swap fails non-destructively (key wiped, old `.exe` intact,
  update re-offers) rather than prompting for elevation (INV-6).
- **macOS installer** — `detect_installer()` still returns `None` on macOS; the
  `.app`/`.dmg` mechanism is FIBR-0130 territory.
- **Changing how the `.exe` is frozen** — the `--onefile` (FIBR-0015) + `--windowed`
  (FIBR-0132) freeze is reused as-is; this feature consumes its output, it does not
  alter it.

## To verify empirically (Windows machine — the manual live leg)

Grounded from the Linux precedent; confirm on the user's Windows test box before /
at the first post-ship release:
- `sys.frozen` truthy and `sys.executable` == the `.exe` path inside the actual
  onefile build (asserted indirectly by the freeze; confirm at runtime).
- **(Correctness-critical, HARD gate) The image-path wait actually detects teardown.**
  Confirm `Get-Process | Where-Object { $_.Path -eq <exe> }` sees the finbreak
  processes (own-user, so `.Path` is readable) and that the poll returns empty **only
  after** the onefile parent bootloader has finished `_MEIxxxxxx` cleanup and released
  the on-disk `.exe` lock — i.e. the child's `os._exit` reliably leads to a fully-free
  image within the ~60 s cap. (Waiting by image path is deliberately chosen over a PID
  wait so the onefile two-process tree and Windows PID-recycling can't defeat it — D3.)
- The **abort-on-timeout path** works against the image-path predicate: the final
  `Get-Process | Where-Object { $_.Path -eq <exe> }` re-check after the poll cap
  correctly reports **still-present** when an old instance is wedged past ~60 s (→
  abort: `Remove-Item` the temp, no move, no relaunch — INV-3) and **empty** when the
  image freed (→ proceed to the move). This is the leg that proves INV-3's "never a
  second instance over a still-running first," so it must exercise the *image-path*
  check the helper ships, not a PID.
- `Move-Item -Force` over a just-exited onefile `.exe` succeeds within the **5 × 0.5 s**
  retry window (tune count/interval); AV/indexer lock lag is the reason for the retry.
- `PYINSTALLER_RESET_ENVIRONMENT=1` set on the helper's `Popen` env **reaches the
  relaunched `.exe`** through the `Start-Process` hop (env inheritance, INV-5/D4) —
  i.e. the new instance re-extracts `_MEIxxxxxx` cleanly and does not collide.
- `subprocess.Popen(creationflags=DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP)`
  yields a helper that **outlives** finbreak's `os._exit` (the whole point).
- `--windowed` (no console) does not suppress the helper spawn (Popen makes its own
  process; `CREATE_NO_WINDOW` added if a console flashes).
- The frozen Windows `.exe` (~85 MB per the FIBR-0015 build) is **within the reused
  `_MAX_UPDATE_BYTES` = 200 MiB download cap** (`update.py`) — confirm the actual
  release `.exe` size so the cap needs no change (a `.exe` over the cap would fail
  `download_and_verify` on every Windows update).
- The live **swap + relaunch** end-to-end on the update *after* the ship (two-cycle).
