> **Source:** user-request-2026-07-13 (design brainstorm: **a separate helper
> process does the swap** — the Windows analogue of the FIBR-0122 `/bin/sh`

> over the exact bytes** that gates the AppImage (INV-7). Windows "unknown
> publisher" (Authenticode/SmartScreen) is **out of scope → FIBR-0133**; the
> integrity gate here does not depend on it.

**Theme:** on Windows, make the FIBR-0054 updater actually *install*. Today
`detect_installer()` returns `None` off an AppImage (INV-7 there), so a frozen
Windows `.exe` sees the prompt path only if wired — and it is not: Windows is
inert. This feature wires it. The whole check → Ed25519-verify → **Later / Skip /
Update now** flow is already cross-platform; only the **install hand-off** is
platform-specific, and the `Installer` protocol was built (D6 of FIBR-0054) as the
exact plug-in point for this — its docstring already names `WindowsInstaller`.

**Why a helper process (the crux).** A running Windows `.exe` is **locked by the
OS** — it cannot be overwritten in place while running, and this is a PyInstaller
**onefile** build (it has extracted itself to `%TEMP%\_MEIxxxxxx`, cleaned on
exit). So the Linux trick — `os.replace` the file *then* relaunch, all in-process
— cannot be copied. Instead (user-approved 2026-07-13): finbreak wipes its key,
spawns a **detached helper**, and exits; the helper **waits for finbreak to fully
exit** (releasing the file lock, letting the onefile bootloader clean its
`_MEIxxxxxx` dir), **moves the verified new `.exe` over the old path**, then
**relaunches it**. This is the Windows twin of the FIBR-0122 `/bin/sh` waiter, and
it watches the same `_MEI`-teardown race (D4).

## Invariants

| INV | What it pins |
|-----|--------------|
| **INV-1** | **Platform detection is exact and fail-inert.** `detect_installer()` returns a `WindowsInstaller(Path(sys.executable))` **only** when `sys.platform == "win32"` **and** `getattr(sys, "frozen", False)` is truthy (a real frozen `.exe`); a non-frozen `python -m finbreak` run on Windows, and every non-Windows run, are unaffected (AppImage on a real AppImage, else `None`). Off a resolved installer the feature is inert — no check, no prompt, Settings checkbox disabled + tooltipped (parity with FIBR-0054 INV-7). Tests monkeypatch `sys.platform` / `sys.frozen` / `sys.executable` and assert the returned type (or `None`). |
| **INV-2** | **Asset selection is installer-driven, not hardcoded.** The picker matches the release asset ending in the **active installer's** `asset_suffix()` — `-x86_64.exe` for `WindowsInstaller`, `-x86_64.AppImage` for `AppImageInstaller` — plus the asset named exactly that `+ ".sig"`; absent-or-duplicate either → `None` (fail safe, parity with FIBR-0054 D14). A release carrying **both** a `.AppImage` and an `.exe` asset picks the `.exe` under a Windows installer and the `.AppImage` under a Linux one. With **no** installer, `check_for_update` returns `None` (nothing is installable, so nothing is offered — this removes the previously-hardcoded AppImage suffix). |
| **INV-3** | **The Windows swap is out-of-process; the running `.exe` is never overwritten in place.** `WindowsInstaller.apply(new_file, on_before_exec)` does **not** move any file itself. It runs `on_before_exec()` (key-wipe), spawns a **detached** helper, and `os._exit(0)`s. The helper (a PowerShell `-Command` one-liner, D3) waits for the old PID to exit (bounded, ~60 s), then `Move-Item -Force`s the verified `new_file` over `target_path()` — **retrying** a few times over a short window because AV / Search-indexer can re-grab a just-exited `.exe` — then relaunches `target_path()`. Tests assert the **command** and **env** builders (pure string/list construction) and the `apply` ordering with `Popen` + `os._exit` monkeypatched; the live swap is proved on Windows (§ To verify empirically). |
| **INV-4** | **(Security.) The derived key is wiped before this process hands off.** Parity with FIBR-0054 INV-6, with one deliberate, documented asymmetry: on Linux the `os.replace` runs **in `apply`, before** the wipe; on Windows there is **no in-process replace**, so the order is: `download_and_verify` verifies + stages the new `.exe` into `target_path().parent` → `apply` runs `on_before_exec()` (wipe) → spawns the helper → `os._exit(0)`; the physical move happens in the helper **after** this process is gone. A failure *before* the wipe (staging into a non-writable dir — caught at download time, INV-6) leaves the key intact and the session usable. Tests assert wipe-then-spawn-then-exit ordering, and that no swap call is made in-process. |
| **INV-5** | **(Robustness.) The relaunch resets the onefile bootloader; no POSIX loader fixups.** `_windows_relaunch_env()` returns `dict(os.environ)` with **`PYINSTALLER_RESET_ENVIRONMENT=1`** set — PyInstaller 6.10+'s official restart signal, so the fresh onefile bootloader re-extracts instead of colliding with the old process's soon-cleaned `_MEIxxxxxx` dir (the Windows analogue of the FIBR-0122 `_MEI` race). It does **not** touch `LD_LIBRARY_PATH` / `LD_PRELOAD` (POSIX-only loader concerns) or the AppImage runtime markers — those are Linux-specific. Test: the env builder sets exactly the reset flag and hand-edits no private `_PYI_*` var. |
| **INV-6** | **(Robustness.) A non-writable install directory fails safely, before the point of no return.** The verified `.exe` is staged as a temp **in `target_path().parent`** (`download_and_verify`, unchanged) — that staging *is* the writability probe: a read-only / full install dir raises at download time, is cleaned up, and is surfaced (`UpdateError`) with the original `.exe` byte-for-byte intact and the key **not** wiped (parity with FIBR-0054 INV-5). The only residual is a *transient* lock defeating the helper's retry-capped `Move-Item` **after** exit — non-destructive: the old `.exe` remains, the helper still relaunches (the old version), and the update re-offers next launch. Documented accepted difference from Linux (which can keep the live session on a failed swap). |
| **INV-7** | **(Security.) The Windows download rides the same Ed25519 gate.** The release `.exe` is signed by `scripts/sign-release.py` (**unchanged** — it signs any file's exact bytes → `<name>.sig`), and `download_and_verify` verifies that raw 64-byte Ed25519 signature over the exact downloaded `.exe` bytes against the committed public key (`update_key.public_key()`), exactly as for the AppImage. A one-byte tamper of the `.exe` **or** the `.sig` → `UpdateVerificationError`, temps deleted, nothing installed. No new crypto — the FIBR-0054 verify path is reused verbatim; this INV pins that the picker + installer feed the `.exe` bytes through it. |
| **INV-8** | **No new network surface; the confinement holds.** Windows adds **no** networked module — the check + download reuse `services/update_fetch.py`, still the **sole** `urllib` importer. `WindowsInstaller` opens no socket and makes no request (it only spawns a local helper). The FIBR-0054 INV-12 `_network_offenders` source scan stays green with the new code present. |
| **INV-9** | **No new UI, no new blocking.** The prompt + progress path is the unchanged cross-platform `ui/update_dialog.py` / `ui/_update_worker.py` (no `.exec(`, auto-lock-safe — FIBR-0054 INV-9 owns this). This feature adds **zero** UI surface: the same dialog, driven by the same shell wiring, now resolves a `WindowsInstaller` instead of `None` on a frozen Windows build. |

## Design decisions

- **D1 — `WindowsInstaller` lives in the existing `services/update_installer.py`.**
  No new module: it is the *same* platform seam FIBR-0054 D6 built for exactly this.
  We add the `WindowsInstaller` class, the `_windows_relaunch_command` /
  `_windows_relaunch_env` module-level helpers (siblings of the AppImage ones), and
  extend `detect_installer()` with the frozen-Windows branch. `UpdateService` and
  the UI are untouched by D1 (they talk to the `Installer` protocol).

- **D2 — The `Installer` protocol gains `asset_suffix() -> str`; the picker is
  installer-driven.** `AppImageInstaller.asset_suffix()` → `"-x86_64.AppImage"`;
  `WindowsInstaller.asset_suffix()` → `"-x86_64.exe"` (matching the actual release
  asset name `finbreak-<v>-x86_64.exe`). `update.py`'s `_select_assets(assets,
  suffix)` takes the suffix as a parameter instead of the module constant
  `_APPIMAGE_SUFFIX`; `check_for_update` reads it from `self._installer.asset_suffix()`
  and returns `None` up front when `self._installer is None`. This is the minimal
  change that makes one picker serve both platforms and deletes the hardcoded
  AppImage assumption. **Ripple:** the existing `_select_assets` unit tests change
  signature (now pass a suffix) — a deliberate refactor, surfaced in Test ripple,
  not silent drift.

- **D3 — The helper is a detached PowerShell `-Command` one-liner.** Rationale:
  Windows PowerShell 5.1 ships on every supported Windows (10/11); `-Command`
  executes inline and is **not** gated by the `.ps1` script execution policy; and
  `Wait-Process` / `Move-Item` / `Start-Process` are clean, first-class primitives
  (vs a `tasklist`-poll `cmd` batch). The script:
  `Wait-Process -Id <pid> -Timeout 60 -ErrorAction SilentlyContinue` (returns
  cleanly if the PID is already gone or the cap elapses) → a bounded retry loop of
  `Move-Item -LiteralPath <new> -Destination <exe> -Force` (short sleeps between
  tries, for AV/indexer lock lag) → `Start-Process -FilePath <exe>` (always
  relaunch, even if the move didn't take — the old `.exe` is still there, INV-6).
  Paths are PowerShell-single-quote-escaped (double any embedded `'`). It is
  launched via `subprocess.Popen(..., creationflags=DETACHED_PROCESS |
  CREATE_NEW_PROCESS_GROUP, close_fds=True)` so it outlives finbreak's exit. The
  argv is **our own** — `sys.executable` and our staged temp path, never user input
  — so the `subprocess` `bandit` B404/B603 suppressions mirror the Linux waiter's,
  with the constraint named in the comment.

- **D4 — Relaunch env = reset flag only.** `_windows_relaunch_env()` copies
  `os.environ` and sets `PYINSTALLER_RESET_ENVIRONMENT=1`. No `LD_*` restoration
  (POSIX-only), no AppImage-marker drop (Linux-only). The reset flag is what stops
  the relaunched onefile bootloader from treating itself as a worker of the exited
  parent and reusing the deleted `_MEIxxxxxx` dir.

- **D5 — Promote the `.exe` to a signed release asset (a documented release step,
  no new code).** The `.exe` is frozen by `windows-build.yml` / `build-windows-exe.py`
  today only as a CI artifact. At release time it is renamed to
  `finbreak-<v>-x86_64.exe`, Ed25519-signed with `scripts/sign-release.py`
  (unchanged), and **both** `finbreak-<v>-x86_64.exe` + `.exe.sig` are attached to
  the GitHub release next to the AppImage. This is added to the release runbook
  (the `.claude/bump.json` release note / a release doc), not to product code —
  `sign-release.py` already signs an arbitrary artifact, so there is nothing to
  build here beyond documenting + performing the attach.

- **D6 — The two-cycle caveat is identical to Linux.** The *running* (old) version
  performs each relaunch, so the swap+relaunch only *executes* on the update
  **after** the one that first ships FIBR-0131. Ship it, then live-prove on the
  user's Windows machine at the next release. Unit tests cover everything provable
  off-Windows; the live swap is a manual Windows leg (§ To verify empirically),
  same posture as FIBR-0054 (see [[appimage-update-relaunch-caveat]]).

## Test ripple

**New legs (all runnable on Linux — pure logic + monkeypatched platform):**
- `detect_installer()` platform matrix: `win32 + frozen` → `WindowsInstaller`;
  `win32` not-frozen → `None`; non-win32 unaffected (monkeypatch `sys.platform`,
  `sys.frozen`, `sys.executable`).
- `asset_suffix()` on both installers returns the expected suffix.
- Asset-picker matrix: a release with **both** `.AppImage` and `.exe` assets →
  `.exe` under a Windows-suffix call, `.AppImage` under a Linux one; missing `.sig`
  → `None`; duplicate suffix → `None`.
- `check_for_update` with `installer is None` → `None` (and never calls the fetcher
  beyond the version compare — matches INV-1 opt-in behaviour).
- `_windows_relaunch_command`: contains `Wait-Process`, the pid, a bounded
  `Move-Item -Force` retry, `Start-Process`, and single-quote-escapes a path with a
  space / embedded quote.
- `_windows_relaunch_env`: sets `PYINSTALLER_RESET_ENVIRONMENT=1`, edits no `_PYI_*`.
- `WindowsInstaller.apply` ordering: with `Popen` + `os._exit` monkeypatched, the
  key-wipe callback fires **before** the spawn, the spawn is detached, and no
  in-process file move occurs; a staging failure path leaves the wipe uncalled.
- Confinement: the existing `_network_offenders` scan (FIBR-0054 INV-12) stays green.

**Changed legs (deliberate refactor — D2):**
- The existing `_select_assets` tests move from the module-constant suffix to the
  parameterised `_select_assets(assets, suffix)` (or an installer-driven wrapper).
- Any test asserting the Linux-only `_APPIMAGE_SUFFIX` constant is re-pointed at
  `AppImageInstaller.asset_suffix()`.

## Out of scope

- **Authenticode / SmartScreen "trusted publisher" signing** → **FIBR-0133**
  (SignPath Foundation OSS program, blocked on external approval). The same release
  `.exe` gains that second signature there; the Ed25519 gate here is independent and
  does not wait for it. An unsigned self-swapping `.exe` is what Defender/SmartScreen
  distrusts most, so the user may see a one-time warning until FIBR-0133 lands —
  acknowledged, not a blocker (sequencing decision 2026-07-14).
- **Rollback / post-update health-check / update watchdog** — accepted risk, parity
  with FIBR-0054's Out-of-scope (single-key, no A/B slots).
- **macOS installer** — `detect_installer()` still returns `None` on macOS; the
  `.app`/`.dmg` mechanism is FIBR-0130 territory.
- **Changing how the `.exe` is frozen** — the `--onefile --windowed` freeze
  (FIBR-0015) is reused as-is; this feature consumes its output, it does not alter it.

## To verify empirically (Windows machine — the manual live leg)

Grounded from the Linux precedent; confirm on the user's Windows test box before /
at the first post-ship release:
- `sys.frozen` truthy and `sys.executable` == the `.exe` path inside the actual
  onefile build (asserted indirectly by the freeze; confirm at runtime).
- `Wait-Process -Id N -Timeout 60 -ErrorAction SilentlyContinue` returns cleanly
  when N is already gone **and** when it times out (never throws).
- `Move-Item -Force` over a just-exited onefile `.exe` succeeds within the retry
  window (tune count/interval); AV/indexer lock lag is the reason for the retry.
- `subprocess.Popen(creationflags=DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP)`
  yields a helper that **outlives** finbreak's `os._exit` (the whole point).
- `--windowed` (no console) does not suppress the helper spawn (Popen makes its own
  process; `CREATE_NO_WINDOW` added if a console flashes).
- The live **swap + relaunch** end-to-end on the update *after* the ship (two-cycle).
