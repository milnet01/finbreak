# FIBR-0013 — Password-protected PDF report export (P11)

> **Status:** ✍️ **DRAFT — awaiting `/cold-eyes`** (project cap **7** loops,
> `finbreak/CLAUDE.md`). Not yet cleared for code.
> **Phase:** P11 — Password-protected PDF export (the standing next phase item
> after FIBR-0012; order locked by the user 2026-07-13: FIBR-0013 → 0014 → 0015).
> **Kind:** implement.
> **Source:** planned (ROADMAP FIBR-0013) + a design brainstorm approved by the
> user 2026-07-13. User choices captured in that brainstorm:
> (1) three sections — **Summary**, **Charts**, **Transactions** — each
> **independently** toggleable (roadmap "section selection");
> (2) **account selection** — all accounts or a chosen **subset** (1, 2, …),
> beyond Home's single-or-all selector;
> (3) with multiple accounts, **combined totals + a per-account summary line**
> (charts and the transaction list pool the selection; the Summary section adds a
> one-line income/spending/net per selected account);
> (4) the export password is **OPTIONAL** (user 2026-07-12) — **blank ⇒ a normal
> unencrypted PDF**, non-empty ⇒ `pikepdf` AES-256 lock, with a **minimum length
> enforced only when set** (user 2026-07-13);
> (5) entry point on **both** the File menu **and** the toolbar;
> (6) a **Light / Dark** PDF theme choice, default **Light** (user 2026-07-13).
> **Security:** the optional export password is a **user-chosen** secret entered at
> export time; it is **never persisted** and lives only on the dialog field for the
> duration of the export call (mirrors FIBR-0009 `PasswordDialog`, not the
> remember-for-account widening). The unencrypted PDF bytes **never touch disk**
> when a password is set (INV-2, mirrors FIBR-0009 INV-2); the only file written is
> the final (encrypted-or-plain) output at the user-chosen path. All new work
> reads/aggregates metadata inside the already-encrypted vault, reachable only
> while unlocked (every new surface is vault-dependent chrome). **No network.**
> **Governed by:** [`docs/decisions/0004-qt-native-pdf-over-weasyprint.md`](../decisions/0004-qt-native-pdf-over-weasyprint.md)
> (ADR-0004, the Qt-PDF-engine + `pikepdf`-encrypt decision this realises),
> [ADR-0008](../decisions/0008-qtcharts-for-reporting.md) (the QtCharts the export
> re-renders offscreen), [ADR-0006](../decisions/0006-transfer-detection-suggest-confirm.md)
> (transfers excluded from Summary/Charts — the same exclusion primitive FIBR-0012
> consumes), and `docs/standards/{coding,naming,testing,commits}.md`.
> **Roadmap:** [ROADMAP.md → FIBR-0013](../../ROADMAP.md).
> Builds directly on [FIBR-0012](FIBR-0012.md) (the **`ReportingService`** +
> `ReportPrefs` + `resolve_period` period model + the `_donut_wedges` /
> `build-chart` idioms this re-renders to PDF, and the persisted period pref the
> dialog pre-fills from), [FIBR-0011](FIBR-0011.md) (the
> `confirmed_transfer_txn_ids()` exclusion the Summary/Charts total against),
> [FIBR-0009](FIBR-0009.md)/[FIBR-0050](FIBR-0050.md) (the in-memory `pikepdf`

**Theme (plain language):** You can save a PDF report of your finances. You pick
the **date range**, which **accounts** to include (all, or just some), and which
**parts** to include — a **summary** (income / spending / net), the **charts**
(the spending donut + the month-by-month trend), and the **list of
transactions**. You can leave it as a normal PDF anyone can open, **or** set a
password to lock it (minimum 8 characters if you do — and a locked PDF cannot be
recovered if you forget the password). You choose a **Light** or **Dark** look,
Light by default (better for printing).

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **Password is optional; the lock is real when set.** An **empty** password field yields a normal **unencrypted** PDF. A **non-empty** password encrypts the output with `pikepdf` **AES-256** (`Encryption(user=pw, owner=pw, R=6)`): the file then **opens with that password and is refused without it** (a round-trip assertion). | user 2026-07-12/13; ADR-0004 |
| INV-2 | **No plaintext PDF to disk.** When a password is set, the unencrypted PDF bytes exist **only in memory** (`QBuffer` → `pikepdf.open(BytesIO)` → encrypted `BytesIO`); the **only** file written to disk is the final (encrypted-or-plain) output at the user-chosen path, and `export()` is the **sole** writer. On **any** failure, no partial/half-written and no accidentally-**unencrypted** fallback file is left. Verified **structurally** (not by a runtime sentinel): `render_pdf_bytes(options) -> bytes` takes **no path** and the encrypt step is pure in-memory `BytesIO`, so no plaintext code path can reach disk. | FIBR-0009 INV-2 (+ security-model INV-4); coding.md § 2 |
| INV-3 | **Sections are opt-in.** The PDF contains **exactly** the ticked sections (Summary / Charts / Transactions), in that fixed order; an unticked section is entirely absent. **≥ 1** section is required (Export disabled otherwise, INV-14). | roadmap "section selection" |
| INV-4 | **Account selection + combined totals.** The report covers the **selected account set**: `account_ids` is **`None` ⇒ all accounts**, otherwise the exact chosen subset. The set is **never empty** — INV-14 requires ≥ 1 account, and the dialog's **"All accounts"** toggle emits **`None`**, not a materialised full set (D7). Summary tiles, the donut, the trend, and the transaction list all aggregate **that set**; the combined figures equal the **sum of the per-account figures** (accounts are disjoint). | user 2026-07-13 |
| INV-5 | **Per-account summary line.** When the Summary section is included, it shows the combined income / spending / net **and** a one-line income / spending / net for **each** selected account, **ordered by account name** and prefixed with that name, each computed by the same `ReportingService.summary` over a single-account set (`frozenset({id})`). When `account_ids` is `None` (all), the per-account lines cover **every** account. | user 2026-07-13 |
| INV-6 | **Transfers excluded from Summary + Charts; the transaction list is complete and marks them.** Summary and the donut/trend exclude every `confirmed_transfer_txn_ids()` id (carried from FIBR-0012 INV-1). The **Transactions** section lists **every** in-range transaction for the selected accounts (transfers included — they are real ledger entries), **uncapped**, ordered by date; each **confirmed-transfer** row is **marked** (a `⇄ Transfer` tag in the category cell) so the reader sees exactly which rows Summary/Charts omit, and a footnote states that Summary/Charts exclude internal transfers so the two can differ without contradiction. | ADR-0006; FIBR-0012 INV-1; user 2026-07-13 |
| INV-7 | **Period model reused.** The dialog offers the **same five modes** as Home (previous-month / current-month / specific-month / year-to-date / specific-year) via the shared `ReportPrefs` + `resolve_period`; it is **pre-filled** from Home's current selection (which itself restores the FIBR-0012 INV-2 persisted period pref). The report's period and 12-month trend span resolve exactly as FIBR-0012 INV-3 / INV-6. | FIBR-0012 INV-2/3/6 |
| INV-8 | **Charts are the same charts — built once.** The donut and trend in the PDF are produced by **shared builder functions** used by **both** `HomeView` and the export (no duplicated chart code). For the PDF they are **rendered offscreen to a raster image** and embedded via `QTextDocument.addResource(ImageResource, …)`. | coding.md § 1.3 (reuse); ADR-0004/0008 |
| INV-9 | **Theme is explicit, not palette-derived.** The chart builders and the HTML template take **explicit colours** (a `ChartTheme` / a light/dark colour set), so an offscreen render does **not** depend on any live widget palette. Default **Light**; **Dark** available. `HomeView` passes its live palette colours through the same builders (no behaviour change on screen). | user 2026-07-13; testing.md § 6 (determinism) |
| INV-10 | **Money stays exact.** Every figure is computed on integer `amount_minor` through the reused `read_minor_unit_exponent` / `to_display_decimal`; only the rendered string crosses to `Decimal`/`float`. Storage and computation stay exact (carried from FIBR-0012 INV-13). | coding.md § 5 |
| INV-11 | **Vault-dependent, no network, i18n, bundle-travel.** The export action is **disabled while locked** — it lives on the File menu + toolbar, both of which `_set_vault_chrome_enabled` disables as whole widgets (its child actions inherit the state; no per-action wiring). No import under `src/` reaches the network (`test_INV8_no_network_imports_under_src` still passes). Every user string `tr()`; every widget in a layout manager. The **encrypt-export** native path (`QPdfWriter` + `pikepdf.Encryption`) travels into the frozen bundle, proven by a `--self-test` leg. | coding.md § 5.2; security-model INV-8; FIBR-0003 |
| INV-12 | **Failures are friendly and safe.** A vault auto-lock mid-export (`VaultLockedError`), a **write error** (unwritable path, permission denied, or no free space — all surfaced by one clear "couldn't write the PDF there" message), or a `pikepdf` encryption error each leave **no** partial/unencrypted file (INV-2) and keep the dialog open so the user can retry or cancel. **Cancelling** the save dialog is a clean **no-op**, not a failure (D9). The export runs under a wait cursor. | coding.md § 2 |
| INV-13 | **An empty period still exports.** A period/account selection with no matching rows produces a **valid** PDF (zero-valued Summary, the donut's empty-state placeholder, an empty-but-headed Transactions table) — never an error or an empty file. | FIBR-0012 D9 (empty donut) |
| INV-14 | **Export gating.** The dialog's **Export…** button is enabled **iff** ≥ 1 section ∧ ≥ 1 account ∧ (the password field is empty **OR** (its length ≥ `MIN_EXPORT_PASSWORD_LEN` = **8** ∧ it equals the Confirm field)); a disabled button shows a one-line reason. The password is used **verbatim** — not trimmed (it may contain spaces); the length check counts the entered characters. The helper text states blank ⇒ no password, and that a set password needs ≥ 8 characters and is unrecoverable. | user 2026-07-13; MIN_EXPORT_PASSWORD_LEN (D7) |

## Context

Verified against source 2026-07-13 (see **Verified code basis** for exact
signatures):

- **`ReportingService(vault)`** (`services/reporting.py`) exposes `summary(prefs,
  account_id, today=None) -> Summary`, `spending_by_category(prefs, account_id,
  today=None) -> list[CategorySpend]` (full, uncapped; the `None`/Uncategorised
  bucket appended last), `monthly_trend(prefs, account_id, today=None) ->
  list[MonthlyTotal]` (exactly 12 points), `base_currency() -> str`,
  `transaction_count() -> int`. It builds the transfer-exclusion set once per call
  (`_excluded()` → `TransferDetectionService.confirmed_transfer_txn_ids()`) and
  drops those ids from every figure. **Today** it takes a **single** `account_id:
  int | None` (`None` = all); D4 widens this to a set.
- **`ReportingRepository(conn).rows_in_range(start_iso, end_iso, account_id)`**
  returns `(id, occurred_on, amount_minor, category_id)` tuples; the account
  predicate is `(? IS NULL OR account_id = ?)`. D4 widens it to an `IN` set.
- **`ReportPrefs`** (frozen `mode` + optional `year`/`month`) and the **pure**
  `resolve_period(prefs, today)` / `resolve_trend_months(prefs, today)` are reused
  verbatim; `MODE_*` tokens name the five modes. `AuthService.report_prefs()`
  returns the persisted `ReportPrefs` (defensively parsed, previous-month default).
- **Charts (`ui/home.py`).** `_donut_wedges(spending, uncat_label, other_label)`
  (module-level) is the ≤ 8-wedge/Other-collapse render list. `HomeView._render_donut`
  builds a `QPieSeries` (hole 0.4) and `_render_trend` a `QBarSeries` with income /
  expenditure `QBarSet`s (+ `QBarCategoryAxis` / `QValueAxis`); both call
  `_themed_chart(series)` which sets a transparent background and pulls text colour
  from `self.palette().text().color()`. D3 extracts these into shared,
  palette-free builders.
- **Transaction list (`services/transactions.py`).** `TransactionService(vault)
  .list_transactions() -> list[tuple[Transaction, Decimal, str, str]]` = each row +
  display `Decimal` + account name + **leaf** category name (`""` for
  uncategorised), whole-vault. `TransactionsView` filters this **in Python**
  (`_matches`) by date-range / account / category — the export reuses the same read
  and filters to the period + selected account set (D6). `Transaction` carries
  `id, account_id, occurred_on` (ISO `YYYY-MM-DD`), `amount_minor`, `description`, …
- **Accounts.** `AccountService(vault).list_accounts() -> list[Account]`;
  `Account` carries `id`, `name`, `type`, `created_at` (only `id`/`name` are used
  here — `id` as the checkbox value, `name` as its label). This feeds the dialog's
  account checkbox list.
- **In-memory `pikepdf` idiom (`importers/pdf_importer.py`).**
  `_normalise_to_plaintext` opens `pikepdf.open(io.BytesIO(raw), password=…)` and
  re-saves to a `BytesIO` — the **decrypt** direction. The export uses the
  **encrypt** direction: `pdf.save(out, encryption=pikepdf.Encryption(user=pw,
  owner=pw, R=6))`. Same "bytes never touch disk" property.
- **Shell (`ui/main_window.py`).** `_make_action(object_name, text, icon_name,
  handler) -> QAction`; a single `QAction` added to **both** `_menu_file` and
  `_toolbar` (the existing pattern, comment at the toolbar loop). `_open_settings`
  is the template for a File-menu dialog: read data from the vault, hand it to a
  dialog that holds **no** vault ref, connect `saved`/`rejected`, `_open_dialog(…,
  defer=False)`. `_set_vault_chrome_enabled(enabled)` enables/disables the
  **`_toolbar` + `_menu_file` + `_menu_view`** widgets as a whole (`setEnabled`) —
  their child actions inherit the state, so there is **no per-action list** to join
  (D8). `_build_workspace` constructs `HomeView(ReportingService(self._service
  .vault), AccountService(self._service.vault), self._service, self._amount_prefs)`
  — the shell already has the handles the export needs. `_home_tab` exposes its current selection (D7 adds two
  small public accessors for the pre-fill).

## Design decisions

- **D1 — A new headless render service, `services/pdf_export.py`.** `ExportOptions`
  (frozen dataclass: `prefs: ReportPrefs`, `account_ids: frozenset[int] | None`,
  `sections: frozenset[str]` over `{"summary","charts","transactions"}`, `theme:
  str` ∈ `{"light","dark"}`, `password: str | None`, `today: date | None`) +
  `PdfExportService(vault)`. Public surface: `export(options, out_path: Path) ->
  None` (gather → build → render → optionally encrypt → **write once**) and the
  lower-level `render_pdf_bytes(options) -> bytes` (returns the final
  encrypted-or-plain bytes, no disk) so encryption is assertable in a pure unit
  test. It composes the existing `ReportingService` / `TransactionService` /
  `AccountService`; it owns **no widget**, but it does drive Qt render classes
  (`QTextDocument`, `QPdfWriter`, offscreen `QChart`), so it runs on the GUI thread
  under a `QApplication` (tests already run under the offscreen platform). Naming
  per naming.md § 4 (module) / coding.md § 4 (class).
- **D2 — Render via the Qt PDF engine, encrypt via `pikepdf` (ADR-0004).** Build
  the report as a `QTextDocument` from an HTML string; render it to an in-memory
  PDF with `QPdfWriter` writing to a `QBuffer` (A4, sensible margins). If
  `password` is set, `pikepdf.open` the bytes and re-`save` with
  `Encryption(user, owner, R=6)` (AES-256) to a second `BytesIO`. Charts are
  embedded as raster images via `document.addResource(QTextDocument.ResourceType
  .ImageResource, QUrl("finbreak:donut.png"), QImage)` referenced by
  `<img src="finbreak:donut.png">` (the Qt resource idiom, not data URIs). Exact
  `QPdfWriter`/`addResource`/`Encryption` call shapes are flagged **To verify
  empirically**. **Rendered strings** (all through `tr()`): the document title
  `tr("finbreak — Financial Report")`; a generated line
  `tr("Generated {date} · Accounts: {names} · {currency}")`; a period line
  `tr("Period: {label}")`; the three section headings `tr("Summary")` /
  `tr("Charts")` / `tr("Transactions")`; the Transactions column headers
  `tr("Date") / tr("Description") / tr("Category") / tr("Amount")`.
- **D3 — Extract the chart builders into a shared, palette-free module
  (`ui/charts.py`).** `build_donut_chart(spending, uncat_label, other_label,
  theme: ChartTheme) -> QChart` and `build_trend_chart(trend, income_label,
  spending_label, theme) -> QChart`, where `ChartTheme` carries the explicit
  colours (`text`, `positive`, `negative`, the donut palette, background). Move
  `_donut_wedges` + the palette constants there. `HomeView._render_donut/_render_trend`
  refactor to call the shared builders with a `ChartTheme` derived from its **live
  palette** (`palette().text().color()`, the FIBR-0012 positive/negative colours),
  so on-screen behaviour is unchanged (reuse per coding.md § 1.3, a rule-of-three
  second call-site: HomeView + export). The export builds a **Light** or **Dark**
  `ChartTheme` independent of any widget.
- **D4 — Widen `ReportingService` to an account *set*.** Change
  `summary`/`spending_by_category`/`monthly_trend` and
  `ReportingRepository.rows_in_range` from `account_id: int | None` to
  `account_ids: frozenset[int] | None` (**`None` ⇒ all accounts**; the caller never
  passes an **empty** set — INV-4/INV-14 — so the query treats `None` as the only
  "all" sentinel), with a **safely parameterised** `IN` clause (integer
  placeholders only; `None` ⇒ no account predicate). `HomeView`
  wraps its single selection (`None if account_id is None else
  frozenset({account_id})`) at its call sites. This keeps the money query — with
  its tested sort / `None`-last / transfer-exclusion — as the **single** code path
  for both one-account, all-accounts, and the export's arbitrary subset (a
  refactor-to-cover, coding.md § 1.3, not a duplicated aggregation). Ripple is
  bounded to `HomeView.refresh` and the reporting test suite (documented under
  Deliverables → test ripple).
- **D5 — Per-account summary lines.** The export calls `summary(prefs,
  frozenset({account_id}))` once per selected account for the per-account lines and
  `summary(prefs, account_ids)` for the combined figures. The two are consistent by
  construction (INV-4: combined = Σ per-account, accounts disjoint).
- **D6 — Transactions section reuses `list_transactions()`.** Filter the
  whole-vault 4-tuples in Python to `occurred_on ∈ [start, end]` ∧ `account_id ∈
  account_ids` (or all), mirroring `TransactionsView._matches`; sort by
  `occurred_on`. Render date / description / category-name / amount, formatted with
  the reused amount + date display prefs (FIBR-0083/0105). Transfers are **not**
  filtered out here (INV-6): a row whose id is in `confirmed_transfer_txn_ids()` has
  its category cell rendered as `tr("⇄ Transfer")` (the marker is keyed on the
  **transfer-id set**, not the category name, so a real category literally named
  "Transfer" is unaffected), and the section carries a footnote
  `tr("Summary and charts exclude money moved between your own accounts (transfers).")`.
- **D7 — The export dialog, `ui/export_dialog.py`.** `ExportDialog(QDialog)` built
  with the account list + the current `ReportPrefs` + selected account handed in
  (holds **no** vault ref, like `SettingsDialog`). Controls: a **period** selector
  (the five modes + the specific-month/-year pickers, reusing Home's control
  layout), an **Accounts** checkbox list with an **"All accounts"** master toggle,
  three **section** checkboxes, a **Theme** Light/Dark radio pair (default Light),
  and the **password** + **confirm** fields (masked, with a Show toggle) + the
  helper text. **Account master-toggle state machine:** "All accounts" starts
  **ticked**; while ticked, the individual account rows are disabled and
  `options().account_ids` is **`None`**. Unticking "All accounts" enables the rows
  (all pre-ticked) and `account_ids` becomes the `frozenset` of ticked ids;
  unticking rows narrows that set; if the user re-ticks **every** row it stays a
  full `frozenset` (not coerced back to `None`) — only the master toggle yields
  `None`; if the user unticks the **last** row the Export button disables (INV-14,
  ≥ 1 account). The password min length is the module constant
  **`MIN_EXPORT_PASSWORD_LEN = 8`** (referenced by INV-14); the password is compared
  **verbatim** (not `.strip()`-ed). `_update_export_enabled()` enforces INV-14 on
  every field change. On accept the
  dialog exposes `options() -> ExportOptions`. Two small public accessors added to
  `HomeView` — `current_prefs() -> ReportPrefs` and `selected_account_id() -> int |
  None` — provide the pre-fill (the private `_current_prefs`/`_selected_account_id`
  promoted / thinly wrapped).
- **D8 — Entry point: one `QAction` on File menu + toolbar.** `self._action_export
  = _make_action("action_export", tr("Export report as PDF…"), "export",
  self._open_export)`, added to `_menu_file` (after Import statement, before the
  separator) **and** appended to the toolbar tuple. It is **vault-gated by
  placement** — no per-action wiring: because it lives on `_menu_file` + the
  toolbar, and `_set_vault_chrome_enabled` disables those whole widgets while locked
  (their child actions inherit the disabled state), the action is inert while locked
  (INV-11). (`_make_action` sets no accelerator, so a disabled container cannot be
  bypassed by a shortcut.) A new `ui/icons/export.svg` glyph (theme-aware via `toolbar_icon`,
  matching the existing hand-authored toolbar SVGs; travels via the package-data
  glob + PyInstaller `--add-data`, FIBR-0051).
- **D9 — `_open_export` handler + save location.** Mirrors `_open_settings`: build
  `ExportDialog` seeded from `AccountService(vault).list_accounts()` +
  `_home_tab.current_prefs()`/`selected_account_id()`; on the dialog's accept
  signal, pop a native `QFileDialog.getSaveFileName` (default filename
  `finbreak-report-<period-label>.pdf`, `.pdf` filter), then run
  `PdfExportService(vault).export(options, path)` under a wait cursor inside a
  `try/except` (INV-12). If the user **cancels** the save dialog (empty path
  returned), the handler is a clean **no-op** and the export dialog stays open (not
  a failure). Success → status-bar confirmation; failure → a `QMessageBox` and the
  dialog stays open. Dialog lifecycle via `_open_dialog`/`_teardown_dialog`.
  **`<period-label>`** is derived deterministically from the resolved `ReportPrefs`
  mode:

  | Mode (`MODE_*`) | `<period-label>` | Example |
  |-----------------|------------------|---------|
  | `PREVIOUS_MONTH` / `CURRENT_MONTH` / `SPECIFIC_MONTH` | `YYYY-MM` (the period's end month) | `2026-06` |
  | `YEAR_TO_DATE` | `YYYY-ytd` | `2026-ytd` |
  | `SPECIFIC_YEAR` | `YYYY` | `2026` |
- **D10 — Theme default Light.** `ExportDialog` defaults the Light radio; the
  service maps `"light"`/`"dark"` to a `ChartTheme` + the HTML body/text/table
  colours. Light = white page, dark text (print-friendly); Dark = the app's dark
  panel look. No dependence on the OS/app theme at export time (INV-9).
- **D11 — Synchronous render (simplicity-first).** The export runs synchronously on
  the GUI thread under a wait cursor. **Budget:** a single period's report renders
  in **≲ 1 s** for a vault on the order of the current dogfood size (~700
  transactions; a full-year specific-year period is the worst case). The
  transaction list is uncapped (INV-6), so the cost scales with the in-range row
  count; the synchronous choice holds while that count stays in the **low
  thousands** (≲ ~5 000 rows). Beyond that the render still completes (no cap) but
  moving `render_pdf_bytes` to a `_worker` (the FIBR-0054 worker pattern, otherwise
  reserved for **network**) becomes the follow-up — noted, not built (coding.md
  § 1.1, no speculative scaffolding). *(The ≲ 1 s / ~5 000-row figures are design
  targets to confirm on the real vault during TDD, not measured guarantees.)*
- **D12 — `--self-test` leg proves the encrypt-export path travels.** Add a leg to
  `_selftest.py` that builds a one-page PDF with `QPdfWriter` and encrypts it with
  `pikepdf.Encryption(user=…, owner=…, R=6)` in memory, then re-opens it with the
  password and asserts success — proving the `QtGui` PDF writer **and** the
  `pikepdf` **encrypt** code path (distinct from the import-side decrypt) are both
  present in the frozen bundle (FIBR-0003). QtCharts already has a self-test leg
  (FIBR-0012 INV-11).
- **D13 — Transfers in the list: show all, marked (user 2026-07-13).** The
  Transactions section shows **all** in-range transactions; each **confirmed-transfer**
  row is marked with a `⇄ Transfer` tag in the category cell (the
  `confirmed_transfer_txn_ids()` set is already built for INV-6, so the marking is
  free), and a footnote clarifies Summary/Charts exclude internal transfers. This
  keeps the list a **complete** ledger while making the Summary/Charts exclusion
  **visible and reconcilable** — the money-clarity the app is held to. The marker
  text goes through `tr()`; a real user category literally named "Transfer" is
  distinguished by the marker being applied on the **transfer-id set**, not the
  category name.

## Exit criteria

1. From an unlocked vault, File → Export report as PDF… (and the toolbar button)
   opens the dialog pre-filled from Home; the action is disabled while locked.
2. Exporting with the password **blank** produces a PDF that opens with no
   password; exporting **with** a password produces one that opens **only** with
   it (both proven by a `pikepdf` round-trip test).
3. Ticking any subset of sections and any subset of accounts produces a PDF with
   exactly those sections and combined-over-that-set figures, plus a per-account
   summary line per selected account; Summary/Charts exclude confirmed transfers,
   the transaction list is complete.
4. Light and Dark themes each render a valid, correctly-coloured PDF, charts
   included.
5. A locked-vault-mid-export / disk-error / encryption-error each show a friendly
   message and leave no partial or unencrypted file.
6. Full gate green (`ruff`/format/`bandit`/`pip-audit`/`gitleaks`/`mypy`/`pytest`),
   the network-AST scan still passes, and the `--self-test` encrypt-export leg
   passes (and the FIBR-0003 build smoke, if run).

## To verify empirically during TDD

Per global CLAUDE.md rule 13 — these API shapes are **assumed** and MUST be
confirmed against PySide6 6.x / pikepdf before being relied on (not asserted as
fact in code without a test):

1. **`QPdfWriter` lives in `PySide6.QtGui`** and is a `QPagedPaintDevice` that
   `QTextDocument.print_(writer)` accepts; page size set via `writer.setPageSize(
   QPageSize(QPageSize.PageSizeId.A4))` and margins via `setPageMargins`. Confirm no
   `QtPrintSupport` import is needed for the `QPdfWriter` path.
2. **Writing to memory:** `QPdfWriter(QBuffer)` produces valid PDF bytes in the
   buffer after the `QTextDocument` is printed (confirm the buffer must be opened
   `WriteOnly` and the writer flushed/destroyed before reading).
3. **`QTextDocument.addResource(QTextDocument.ResourceType.ImageResource,
   QUrl(...), QImage)`** embeds an image referenced by `<img src="...">` in the
   HTML, and the image survives into the PDF (vs. needing a `QTextCursor.insertImage`
   or a data-URI fallback).
4. **Offscreen chart raster:** the chosen path to turn a `QChart` into a `QImage`
   without a shown window — `QChartView(chart)` + `resize` + `grab()` **or** a
   `QGraphicsScene.render` onto a `QPainter`/`QImage` — including whether an event
   loop spin (`QApplication.processEvents`) is needed for layout before grabbing.
5. **`pikepdf.Encryption(user=pw, owner=pw, R=6)`** is the exact AES-256 kwargs
   shape for `pdf.save(out, encryption=…)`, and `pikepdf.open(BytesIO, password=pw)`
   round-trips it (wrong/absent password → `pikepdf.PasswordError`).
