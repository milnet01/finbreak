**Theme:** the roadmap bullet asks for an **optional** updater: on a newer version,
prompt the user with **Later** (re-ask next launch) / **Skip this version** (never
re-prompt for it) / **Update now** (download → install → relaunch). We build the
**full flow for Linux AppImage** — the one platform where the install hand-off is
simplest (the running AppImage knows its own path via `$APPIMAGE`, so "install" is
*replace one file and re-exec*) and where the user can dogfood it end-to-end on
their own machine. The feature ships in **two phases** because the updater cannot
be built or tested until a real, signed release exists to update *from*:

- **Phase 1 — a real Linux release.** Give the app a real version (`0.1.0`), build
  a **real** release AppImage (not the FIBR-0003 self-test stub), sign it
  (Ed25519), and publish GitHub Release `v0.1.0` with the AppImage + its signature.
  This is the Linux slice of P13/FIBR-0015 pulled forward, and reusable there.
- **Phase 2 — the in-app updater.** The opt-in check, the prompt, the
  signature-verified download, the atomic swap, and the relaunch — built and tested
  against Phase 1's release.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | **(Security.) Opt-in, off by default — the check is the *only* deliberate network egress.** No outbound call happens unless the plaintext flag `check_for_updates` is `true` in the `window.ini` sibling (`paths.window_settings_path()`). An **absent** key ⇒ off (a fresh install never phones home). `UpdateService.is_enabled()` returns `False` for absent/`"false"`/malformed. A test asserts: absent → no check runs (the injected fetcher is never called); explicitly enabled → it is. | security-model INV-8 (amended: this is the single consented exception) |
| INV-2 | **(Security.) The check needs no vault and runs while locked, touching no decrypted data.** The launch **check** reads only the plaintext INI, the compiled-in `__version__`, and the public GitHub Releases API — never the vault key, never a decrypted row — and runs on the still-locked shell (before or without any unlock). The **offer prompt** it produces is deferred to *after* unlock (D15) to avoid colliding with the modal unlock dialog, but that is a UX choice, not a vault dependency: the check + download + install never read the vault. A test drives the check with a locked `AuthService` and asserts it completes and never calls a vault-connection accessor. | FIBR-0004 INV-3 (key lifetime); security-model INV-1 |
| INV-3 | **A newer version is offered only for a strictly-greater, well-formed, non-skipped tag (fail-safe parse).** `check_for_update()` parses the release `tag_name` and the current `__version__` per the **version grammar (D13)**, selects the AppImage + `.sig` assets per the **asset predicate (D14)**, and takes `notes` from the release JSON `body` (shown inline in the prompt). It returns an `UpdateInfo` **only if** `latest > current` (D13 comparison) **and** `latest` is not the skipped version **and both** the AppImage asset **and** its matching `.sig` asset are present. A malformed `tag_name` (e.g. `"v1.2-rc1"`, `"latest"`), a malformed `__version__`, a missing/duplicate asset, or `latest <= current` ⇒ returns `None` (no offer). Tests: newer→offer; equal→None; older→None; malformed tag `"v1.2-rc1"`→None; short-vs-long `"0.1"` vs `"0.1.0"`→equal (D13 zero-pad, so no offer); an `int()`-quirk tag `"1_0.0"`→None (the `isdigit` guard, not bare `int()`); missing `.sig` asset→None; skipped==latest→None. | roadmap bullet (offer, don't force) |
| INV-4 | **(Security — the core gate.) Only an Ed25519-signed download is ever installed.** `download_and_verify(info)` downloads the AppImage **and** its detached `.sig`, then verifies the signature over the **exact downloaded bytes** with `cryptography`'s `Ed25519PublicKey.verify`, using the **public key committed at `services/update_key.py`**. On **any** verification failure (bad signature, truncated file, wrong key, absent `.sig`) it deletes the temp file and raises `UpdateVerificationError` — **nothing is installed**. A test signs a blob with a throwaway private key whose public half is monkeypatched in → verify passes; a one-byte tamper of the blob **or** the signature → `UpdateVerificationError` and the temp file is gone. | security-model INV-2/INV-8; ADR-0007 |
| INV-5 | **The swap is atomic; the running binary is never left half-written.** The download lands in a temp file **in the same directory as `$APPIMAGE`** (`installer.target_path().parent`, D6 — same filesystem, so the rename is atomic); only **after** INV-4 passes does the installer `chmod 0o755` it and `os.replace(temp, appimage_path)`. Any failure before the `os.replace` leaves the original `$APPIMAGE` byte-for-byte intact (the temp is deleted). A disk-full (`ENOSPC`) or read-only-directory error during the download / `chmod` / `os.replace` is caught, the temp deleted, and surfaced per INV-11 with the original `$APPIMAGE` intact. The "same-filesystem ⇒ atomic rename" property is a **code-review checklist item** (like INV-13's `tr()` leg), not something a unit test proves; the test asserts only that the target holds the new bytes + is executable after a success, and that a pre-`os.replace` raise leaves the original untouched. | ADR-0007 |
| INV-6 | **(Security.) The derived key is wiped after a successful swap, before the relaunch.** The relaunch (a detached `subprocess.Popen` + `os._exit`, D8) does **not** run Qt's `aboutToQuit`, so `AppImageInstaller.apply(new_file, on_before_exec)` invokes the key-wipe callback (`AuthService.on_about_to_quit`) **after a successful `os.replace`** and **before** the spawn — so no decrypted key survives into the relaunched process, while a swap that fails first (ENOSPC/read-only, INV-5) leaves the key intact and the session usable (INV-11). Tests assert the order (wipe → relaunch → exit) with the spawn + `os._exit` monkeypatched; that a failing `os.replace` leaves the wipe **uncalled**; and that the relaunch is detached (`start_new_session=True`) with `PYINSTALLER_RESET_ENVIRONMENT=1` + the stale AppImage vars dropped. | FIBR-0004 INV-3 |
| INV-7 | **(Security/robustness.) Off an AppImage the whole feature is inert.** `detect_installer()` returns an `AppImageInstaller` **only** when `$APPIMAGE` is set and points at an existing file; otherwise `None`. With no installer: no launch check runs, no prompt appears, and the Settings checkbox is **disabled with an explanatory tooltip**. So a `python -m finbreak` run, a Flatpak, or a future un-wired Windows build never half-updates. Tests: `$APPIMAGE` unset → `detect_installer() is None` and `is_update_supported()` is `False`; the Settings checkbox is disabled. | ADR-0007 (per-platform mechanism) |
| INV-8 | **Skip persists; Later does not; neither carries sensitive data.** "Skip this version" writes `skipped_update_version = <version>` to the plaintext INI; a subsequent check for that same version returns `None` (INV-3). "Later" writes nothing — the next launch re-checks and re-prompts. Both keys live in `window.ini` (never the vault) and hold only a version string / a boolean. A test: skip `0.1.1`, re-run the check against a release advertising `0.1.1` → `None`; against `0.1.2` → offered. | roadmap bullet (Later/Skip semantics) |
| INV-9 | **(Security.) The prompt + progress are non-blocking and auto-lock-safe.** `UpdateDialog` is opened via the shell's tracked `_open_dialog(dialog, defer=False)` (`self._dialog`, `setModal(True)`, `show()` — non-blocking, **never** `exec()`), shown only post-unlock (D15). If an idle auto-lock fires while it is open, `_lock` step 1 `_teardown_dialog()` destroys it (it is `self._dialog`, FIBR-0051 INV-4b); the update path holds no vault reference, and any in-flight download worker's result is dropped when the dialog is gone, so nothing crashes. A test (mirroring the FIBR-0065 dialog-lifecycle grep) asserts `ui/update_dialog.py` contains no `.exec(`; a `qtbot` test opens the prompt, captures it, drives `_lock`, and asserts it is destroyed. | FIBR-0051 INV-4b; FIBR-0065 INV-1 |
| INV-10 | **(Security.) The download is resource-bounded.** The fetch is capped by a maximum byte size (`_MAX_UPDATE_BYTES`) and a socket timeout; a response exceeding the cap is abandoned (temp deleted, `ValueError`) before it can exhaust disk, and a stalled connection times out rather than hanging the check. Concrete bounds: `_MAX_UPDATE_BYTES = 200 * 1024 * 1024` (200 MiB — an AppImage is ~tens of MB) and a **30 s** socket timeout. The API JSON response is likewise read under `_MAX_API_BYTES` (~1 MiB), not unbounded. Mirrors the `_MAX_IMPORT_BYTES` resource-cap pattern (FIBR-0009 / security-model INV-5b). A test feeds an over-cap stream and asserts the abort. | security-model INV-5b |
| INV-11 | **Any network/API failure is silent and safe.** A DNS/connection error, a non-200 status, a timeout, or unparseable JSON during the **check** is caught, logged (no traceback to the user), and results in **no prompt** — the app proceeds exactly as if up to date. Only an explicit **Update now** surfaces an error dialog — for **any** of {signature mismatch (INV-4), oversize / timeout (INV-10), `ENOSPC` / read-only-dir at swap (INV-5), a mid-download connection drop} — and stays on the current version. A test makes the injected fetcher raise and asserts `check_for_update()` returns `None` without propagating; a second asserts a verify failure on **Update now** raises `UpdateVerificationError` (surfaced, not swallowed) with the current version unchanged. | roadmap bullet (never nag); security-model (no-network default preserved on failure) |
| INV-12 | **(Security.) Network code is confined to one allowlisted module; the ban holds everywhere else.** The **only** file under `src/finbreak/` permitted to import `urllib` is `services/update_fetch.py`; `socket`, `http`, `requests`, and `ftplib` stay banned in **every** file, including that one (`urllib.request` over the default-TLS `https://` endpoint is the entire network surface). `test_INV8_no_network_imports_under_src` is amended so its allowlist keys on the **relative path** `services/update_fetch.py` (via `py.relative_to(package_root).as_posix()`, **not** the basename `update_fetch.py`) and permits **only** `urllib` there; every other banned name stays banned there, and `urllib` stays banned in every other file. To test the amended rule without editing shipped source, the per-file AST scan is **factored into a helper** `_network_offenders(rel_path: str, source: str) -> list[str]`; a test feeds it a planted `import socket` at `services/update_fetch.py` (still flagged) and a planted `import urllib` at another relative path (still flagged), plus a `urllib` import at `services/update_fetch.py` (**not** flagged — the one allowed case). | security-model INV-8 (amended) |
| INV-13 | **(i18n.) Every visible string in the new dialog is translatable; the dialog is RTL-safe.** All `UpdateDialog` labels/buttons/title use `self.tr("literal")` (no `self.tr(variable)`); the new Settings checkbox text is a `tr()` literal. Enforced the FIBR-0051/FIBR-0055 way: the runnable `test_INV10_no_fixed_geometry_in_new_ui` source-scan (globs all `ui/*.py` for fixed geometry) covers `ui/update_dialog.py` (use layout managers, no fixed geometry); `tr()`-wrapping is a review-checklist item per `coding.md § 5.2`, not a grep. | `coding.md § 5.2`; FIBR-0051 D8/INV-10 |
| INV-14 | **(Phase 1.) A published release round-trips through the signature gate.** The signing script produces, for the release AppImage, a `.sig` that `services/update.py`'s verifier accepts against the committed public key; a tampered AppImage is rejected. The **private key is never committed** (only the 32-byte public key is, as a base64 constant in `services/update_key.py`); the private key path is git-ignored. A test signs a fixture blob with a test key and verifies the round-trip; the `test_INV8`-style repo scan asserts no private-key material is tracked. | security-model INV-2 |
| INV-15 | **(Phase 1.) The release AppImage is the real app and launches Python-free.** `scripts/build-release-appimage.sh` freezes the **GUI entry** (`python -m finbreak`, **not** `--self-test`) and the FIBR-0003 clean-room proof (launch in a Python-free container) still passes for it — so what users download actually runs. | FIBR-0003 (clean-room bundling proof); ADR-0007 |

## Design decisions

- **D1 — Ed25519 via the already-bundled `cryptography` library; no minisign tool,
  no new runtime dependency.** The integrity gate is a detached **Ed25519**
  signature over the AppImage bytes. The app verifies it with
  `Ed25519PublicKey.from_public_bytes(...).verify(sig, data)` from `cryptography`
  (already in the bundle) — so **no external tool at runtime** and **no new install
  weight**. `cryptography` is promoted from transitive to an explicit
  `pyproject` runtime dep (we now import it directly; `dependencies.md`). The `.sig`
  is the **raw 64-byte** Ed25519 signature (binary), our own minimal convention —
  both the signer and verifier are ours, so the minisign container format
  (base64 + trusted-comment lines) would be parsing for no benefit. *Rejected:*
  shelling out to a `minisign`/`gpg` binary (can't assume it exists on the user's
  box or inside the bundle); a bare SHA-256 checksum (protects only against
  corruption, not a tampered release — wrong gate for a security app).

- **D2 — Simple full-file download + replace, not zsync/AppImageUpdate delta.** For
  a small, infrequently-updated app, downloading the whole AppImage and swapping it
  is dead simple and robust; delta updates (`zsync`) add a dependency and moving
  parts for a bandwidth saving that does not matter here (YAGNI). Surfaced, chosen
  deliberately.

- **D3 — Check once, on launch — no background polling.** One bounded outbound call
  per launch, only when opted in. No timer-driven polling (that multiplies the
  egress surface and the no-network posture). Predictable and auditable.

- **D4 — The opt-in flag + skipped-version live in the plaintext `window.ini`, NOT
  the vault.** The check must run **while locked** (before any unlock), so its
  settings cannot live in the encrypted vault (which needs the key to read). This is
  the deliberate mirror-image of FIBR-0055's auto-lock timeout (a *security control*,
  stored **in** the vault, read on unlock): the update prefs are **non-sensitive**
  (a boolean + a version string) and needed **pre-unlock**, exactly like the window
  geometry FIBR-0052 already keeps there. `UpdateService` owns the read/write via
  `QSettings(paths.window_settings_path(), IniFormat)`.

- **D5 — The toggle is surfaced in the existing Settings dialog (unlocked), value
  persisted to the plaintext INI.** No new pre-unlock settings UI. Default off; a
  user opts in once they reach **File → Settings…** (vault-dependent chrome). From
  the next launch the check reads the plaintext flag and runs even before unlock.
  `SettingsDialog` gains a checkbox; the shell persists it via
  `UpdateService.set_enabled(...)` on Save (the dialog holds no `UpdateService`
  network reference — it just reports the checkbox state), so the vault-scoped
  `AuthService.set_auto_lock_minutes` path (FIBR-0055) is untouched.

- **D6 — A platform seam: `Installer` protocol + `AppImageInstaller` +
  `detect_installer()`.** `Installer` (a `typing.Protocol`) is
  `can_self_update() -> bool`, `target_path() -> Path` (the file to replace — the
  running `$APPIMAGE`; its `.parent` is where `download_and_verify` stages the temp so
  `os.replace` is atomic, INV-5), and `apply(new_file: Path, on_before_exec:
  Callable[[], None]) -> NoReturn`.
  `AppImageInstaller` implements the AppImage swap+relaunch; `detect_installer()`
  returns it only when `$APPIMAGE` is set (else `None` → INV-7). **This is the
  Windows plug-in point** — a future `WindowsInstaller` (download an installer /
  swap-on-exit helper) registers here with **no change** to `UpdateService` or the
  UI. Windows is *designed for*, not built (Out of scope).

- **D7 — The launch check runs on a `QThread` worker (mirroring `DeriveWorker`).**
  `ui/_update_worker.py::UpdateCheckWorker(QThread)` calls
  `UpdateService.check_for_update()` off the GUI thread and emits `found(object)`
  (carrying an `UpdateInfo`) / `none()` / `failed(object)`; parented,
  `finished→deleteLater`, re-entrancy-safe. (The sibling `DownloadWorker` — Deliverable
  13 — runs the download the same way.)
  The shell arms it via `QTimer.singleShot(0, self._maybe_check_for_update)` from
  `__init__`, gated on `is_update_supported()` **and** `UpdateService.is_enabled()`.
  On `found`, the result is **stored in `self._pending_update`** and the prompt is
  shown post-unlock per D15 (not immediately, to avoid the modal collision).

- **D8 — Relaunch via a detached `subprocess.Popen` + `os._exit`; key wiped between a
  successful swap and the relaunch.** `AppImageInstaller.apply(new_file,
  on_before_exec)` does: `chmod 0o755` the verified temp → `os.replace` over
  `$APPIMAGE` → **`on_before_exec()`** (the key-wipe) → spawn the swapped image
  **detached** (`subprocess.Popen([appimage_path], start_new_session=True,
  env=_relaunch_env(), stdio=DEVNULL)`) → `os._exit(0)`. `_relaunch_env()` sets
  **`PYINSTALLER_RESET_ENVIRONMENT=1`** (PyInstaller 6.10+'s official "restart the
  frozen app" signal — the new onefile bootloader then re-extracts instead of
  treating itself as a worker subprocess and reusing the deleted extraction dir) and
  drops the stale AppImage `APPDIR`/`APPIMAGE`/`ARGV0` so the outer runtime re-mounts
  cleanly. The shell passes `on_before_exec=self._service.on_about_to_quit`, so the
  key is wiped **only after a successful replace** and **before** the spawn — a failed
  swap never wipes it, leaving the session usable (INV-6/INV-11), and there is no
  un-wiped parent window (this process is already wiped when it spawns + exits).
  `apply` is typed `NoReturn` (`os._exit` ends the process). **Field-corrected:** the
  original design used an in-place `os.execv`, which shipped in v0.1.0–0.1.2 and
  **failed on the live 0.1.2→0.1.3 self-update** ("closed but didn't reopen") — an
  in-place re-exec can't replace the running image's busy FUSE mount, and the onefile
  bootloader mistakes the re-exec for a worker subprocess. *Rejected:* wiping *before*
  `apply` (a failed swap would then strand a locked live session); an in-place
  `os.execv` (the shipped-then-reverted approach above); relying on `aboutToQuit`
  (the hard exit bypasses it).

- **D9 — stdlib `urllib.request` for the API + download; no HTTP-client
  dependency.** The GitHub Releases API
  (`https://api.github.com/repos/milnet01/finbreak/releases/latest`) and the asset
  download use `urllib.request` with an explicit timeout and the INV-10 byte cap.
  Confined to `services/update_fetch.py` (D12). No `requests`/`httpx` — a new
  networked dependency would both bloat the bundle and widen the banned surface.

- **D10 — Amend `security-model.md` INV-8 to record the single consented egress.**
  INV-8 is rewritten from "no outbound call of any kind" to: "the shipped app makes
  **exactly one** kind of outbound call — an **opt-in, off-by-default** update check
  to the GitHub Releases API, confined to `services/update_fetch.py`; no other
  network access exists, and the check is never made without explicit user consent."
  The `pip-audit`/CI note is unchanged. This is a doc change tracked as a
  deliverable, not a silent drift.

- **D11 — Releases are published as full (non-prerelease) releases so
  `/releases/latest` returns them.** GitHub's `releases/latest` endpoint **excludes**
  prereleases; marking `v0.1.0` "pre-release" would make the check find nothing.
  They are early versions, not prereleases — published as normal releases. (An
  honest "preview" framing lives in the release notes, not the prerelease flag.)

- **D12 — All `urllib` use is isolated to `services/update_fetch.py`, the sole
  allowlisted networked module.** Keeping the network surface in one small,
  clearly-named file (whose only job is `fetch_latest_release()` +
  `download(...)`) makes the egress trivially auditable and lets the amended
  `test_INV8` allowlist name exactly one file for `urllib` only (INV-12).

- **D13 — Version grammar + comparison (fail-safe, dependency-free).** A version is
  exactly `N(.N)*` — one or more dot-separated non-negative integer segments — with
  an optional leading `v`/`V` stripped first. Parsing splits on `.`; a segment is
  valid only if `segment.isascii() and segment.isdigit()` (**not** bare `int()`,
  which quietly accepts `"1_0"`, `" 1"`, `"+1"`, and unicode digits); **any** segment
  failing that guard (a pre-release suffix like `-rc1`, a word like `latest`, `"1_0"`,
  an empty segment from a trailing/`..` dot) makes the parse return `None`, and the
  caller treats that version as **unusable** — no offer,
  never a crash (INV-3). Comparison **zero-pads the shorter tuple** to equal length,
  so `(0,1)` and `(0,1,0)` compare **equal** and `(0,1,0) < (0,1,1)`. This is
  deliberately stricter than PEP 440 (no epochs / pre / post / local segments)
  because our own tags are plain `vMAJOR.MINOR.PATCH` (D11) — a tiny hand-rolled
  parser suffices and avoids pulling in `packaging`. *Rejected:* `packaging.version`
  (a new dependency for a format we fully control).

- **D14 — Asset-selection predicate.** From the release JSON `assets[]`, the
  **AppImage** is the asset whose `name` ends `-x86_64.AppImage`; its **signature**
  is the asset named exactly that AppImage name **plus** `.sig` (e.g.
  `finbreak-0.1.0-x86_64.AppImage` → `finbreak-0.1.0-x86_64.AppImage.sig`).
  `appimage_url` / `sig_url` are those two assets' `browser_download_url`. If
  **either** is absent — or **more than one** asset matches the AppImage suffix —
  `check_for_update()` returns `None` (fail safe, INV-3). The signing script
  (Deliverable 4) writes `<artifact>.sig`, so this naming holds by construction.

- **D15 — The offer is shown *after* unlock; the prompt is a standard tracked modal
  (no collision with first-run/unlock).** The **check** (network only) runs at launch
  and needs no vault (D7/INV-2). But the first-run/unlock dialog is
  **application-modal** and, like the update prompt, would be shown on the same first
  event-loop turn into the **single-slot** `self._dialog` — two app-modals cannot
  coexist. So the worker's `found` result is **held** in `self._pending_update`, and a
  single `_maybe_show_pending_offer()` decides when to show it: it opens the prompt
  **only** when the shell is **unlocked** (a `self._unlocked` bool set `True` in
  `_enter_unlocked`, `False` in `_lock`) **and** idle (`self._dialog is None`), then
  **clears `self._pending_update`** (shown at most once per launch). It is called (a)
  from the **end of `_enter_unlocked`** (after `_teardown_dialog` + `_build_workspace`,
  so it never tears down the prompt it just opened), **and** (b) from the worker's
  `found` handler (the common case: the network usually returns *after* the user has
  unlocked). If the shell is still locked, **or** busy with another tracked dialog
  (e.g. Settings) when `found` arrives, the offer stays pending and is retried at the
  next `_enter_unlocked` — so the prompt never collides with another app-modal. (In
  the unlocked-but-busy case, the retry is only reached on the **next launch's** check
  — the busy-dialog close slots are deliberately *not* hooked, since `_teardown_dialog`
  runs on many paths incl. mid-`_lock`; an accepted minor limitation, as the check
  re-runs every launch.)
  `Skip`, `Later`, and `Update now` each **also** clear `self._pending_update`, so a
  Skipped or deferred version is **not** re-offered again this launch (Later = re-ask
  **next launch**, per the theme); `Skip` additionally persists the version (INV-8).
  The prompt is opened via the existing **`_open_dialog(dialog, defer=False)`** tracked
  path (`self._dialog`,
  `setModal(True)`, `show()` — non-blocking, never `exec()`), **not** `show_modal`:
  `UpdateDialog` has **three** custom signals (`later`/`skip`/`update_now`) wired to
  shell slots, and on **Update now** it must **stay open** in the "Downloading…" busy
  state (it does not `accept()`/`reject()` until the install relaunches or an error
  surfaces), which the vanilla `show_modal(accepted→on_accept, finished→deleteLater)`
  contract does not fit. Being `self._dialog`, an auto-lock tears it down via `_lock`
  step 1 for free (INV-9). This keeps the *security* property ("the update needs no
  vault") while the *UX* is "you see the offer once you're in" — the user's chosen
  behaviour (2026-07-10). *Rejected:* an offer on the still-locked screen (would
  require de-modalizing the unlock dialog — out of scope); checking only after unlock
  (drops the no-vault-needed check property).

## Out of scope (this cut)

- **Windows / macOS / Flatpak installers** — the `Installer` seam is built and
  Windows is *designed for* (D6), but only `AppImageInstaller` is implemented. On
  every non-AppImage package the feature is inert (INV-7). A `WindowsInstaller` is a
  follow-up (the user's "Windows too, later").
- **Delta / zsync updates** — full-file replace only (D2).
- **Background/periodic polling** — check on launch only (D3).
- **Auto-download without asking / silent updates** — always prompt; never force
  (roadmap bullet).
- **A full multi-platform release pipeline (P13/FIBR-0015/FIBR-0016)** — Phase 1
  builds only the Linux AppImage slice; the Windows `.exe`, macOS `.app`, and Flathub
  manifest remain FIBR-0015/0016.
- ~~**In-app changelog rendering**~~ — **now shipped** (2026-07-11, post-release
  enhancement): the prompt shows the release `body` **inline** (markdown-rendered,
  read-only) instead of linking out to the browser. No extra egress — the body
  already rides in the release JSON the check fetches.
- **Key rotation / multi-key trust** — a single baked-in public key; rotation is a
  future concern (would ship a new key in a new signed release).
- **Rollback / post-relaunch health-check (accepted risk).** The Ed25519 signature
  proves the new AppImage is **authentic**, not that it **runs**. Once `os.replace`
  swaps it in there is no automatic revert if the new binary fails to launch. This is
  an **accepted risk** for the first cut (single-key, no watchdog); a future
  enhancement could keep a `<appimage>.bak` and restore it if the relaunched process
  doesn't check in. Surfaced deliberately, not silently dropped.
