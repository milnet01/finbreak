> **Status:** 🚧 **DRAFT — awaiting `/cold-eyes`** (global rule § 14: loop until a
> pass returns zero verified findings before any code). Not cleared for code.
> **Phase:** enhancement of P07 (PDF import). **Kind:** feature.
> **Source:** user-request-2026-07-05 (the user supplied five real Standard Bank
> statement types + confirmed the design; see § Verified basis).
> **Security: NO new surface.** This phase adds **no** dependency, **no** schema
> change, and **no** new credential path. It **reuses** FIBR-0009's in-memory
> `pikepdf` decrypt (security-model INV-4, decrypted bytes never on disk), the
> `pikepdf.PasswordError` re-prompt, the opt-in remembered password

> unchanged. The one security-relevant obligation is inherited, not new: the
> statement password is **never** logged (INV-11 re-asserts it for the new code
> path). **Test fixtures are 100 % synthetic** — no real statement, account
> number, or ID number is committed (§ Fixtures).
> **Roadmap:** [ROADMAP.md → FIBR-0050](../../ROADMAP.md).
> **Governed by:** `docs/standards/{coding,naming,testing,commits,dependencies}.md`;
> [ADR-0005](../decisions/0005-csv-mapping-profiles.md) (this phase is a further
> **specialisation** of the importer interface — like OFX, a Standard Bank
> statement is **self-describing**, so it **skips** the CSV mapping step);
> builds directly on [FIBR-0009](FIBR-0009.md) (the PDF decrypt + extraction
> machinery it reuses) and [FIBR-0008](FIBR-0008.md) (the OFX importer whose
> `parse(bytes) -> ParseResult` shape + `preview_ofx` seam this phase mirrors).

**Theme.** FIBR-0009 shipped a **generic** PDF path: lift a **ruled table** off
the page (`pdfplumber.extract_tables()`), serialise it to CSV, feed the CSV
pipeline. That path is right for a bank whose statement is a clean ruled grid —
but it is **fragile** on real Standard Bank statements, empirically:

| Real SB statement | Generic table path (FIBR-0009) |
|---|---|
| Home Loan | ✅ extracts row-aligned |
| Savings (Puresave) | ✅ extracts row-aligned |
| Personal Loan (Revolving Credit) | ✅ extracts row-aligned |
| **Current account** | ❌ **collapses** — all transactions into one cell |
| **Credit card** | ❌ two-columns-per-line, no grid lines — unreadable |

Same bank, same visual language, but the **table geometry** is unreliable (the
Current account has the same column shape as Savings yet collapses; a denser
statement could regress the same way). The **printed text**, however, is stable
across every layout — it is what a human reads. So this phase adds **one
Standard Bank reader** that parses the **text lines** (not the grid) and feeds
the existing preview → dedup → commit pipeline. It is **one reader for all
account types**, not one-per-type (the user's explicit constraint): the account
families differ only in a small, enumerated set of line-grammar knobs handled
inside the single module.

## Context

FIBR-0009 (P07) reused, unchanged, the FIBR-0007 CSV pipeline and added a generic
PDF **table** extractor + the in-memory decrypt. This phase reuses **both** — the
FIBR-0007 write pipeline **and** FIBR-0009's decrypt/caps — and adds a Standard
Bank **text** reader beside the generic table extractor. It reuses, unchanged:

- **`_normalise_to_plaintext(raw, password)`** (`importers/pdf_importer.py`) — the
  in-memory `pikepdf` decrypt + save-strip (INV-2 of FIBR-0009). The SB reader
  calls it verbatim and opens the plaintext with `pdfplumber` to read **text**
  (`page.extract_text()`) instead of tables.
- **`parse_transaction(occurred_on, amount: Decimal, description, exponent)`** +
  **`read_minor_unit_exponent`** (`services/transactions.py`) — the money
  contract. The SB reader builds a signed `Decimal` and calls it **exactly** as
  `OfxImporter._parse_statement` does (`ofx_importer.py:91`).
- **`ParseResult` / `RowError` / `TransactionDraft`** (`importers/base.py`,
  `models.py`) — the uniform importer output.
- **`ImportService._preview_from_result` / `commit_import` / `_dedup`** — the
  dedup-analysis + atomic write tail. The wizard calls the shared preview seam
  (D5) so the SB path reaches the **identical** `commit_import` as CSV/OFX.
- **The import wizard's PDF branch + password loop** (`ui/import_wizard.py`,
  `ui/password_dialog.py`) — the `pikepdf.PasswordError` catch, stored-password
  auto-try, and re-prompt are reused; the SB branch is a **pre-check** inside the
  existing `_select_pdf`, not a new screen.
- **The `accounts.statement_pdf_password` column + accessors** (v5, FIBR-0009) —
  the remembered-password path is reused wholesale; **no schema change** here.

So the genuinely new code is: the `StandardBankImporter` (text reader + the pure
line-grammar helpers), the `detect_standard_bank` sniff, one small wizard
pre-check branch, and a format-neutral rename of `preview_ofx` (D5). **No new
runtime dependency** (`reportlab`, used to build synthetic fixtures, is a
probe/test-authoring tool, never shipped — like FIBR-0009's use of it) and **no
migration**.

## The three layout families (empirical grammar)

All grammar below is transcribed from the user's **real** statements via
`pdfplumber.page.extract_text()` (§ Verified basis). Every example line is a
verbatim extraction.

### Family A — transactional (Current, Savings, Revolving-Credit)

Column header (may wrap across 2–3 text lines): `Details │ Service Fee │ Debits │
Credits │ Date │ Balance` (Savings shows it on one line; Current and RCP wrap;
RCP orders the middle pair `Credits / Debits`). Transactions are **right-anchored**:

```
BALANCE BROUGHT FORWARD 05 13 1,427.41          ← anchor: MM DD balance, NO amount
LOTTERY WINNINGS 12.10 05 15 1,439.51           ← credit: amount 12.10  (delta +12.10)
VAS00216222570                                  ← continuation → fold into prior desc
PAYSHAP PAYMENT TO 300.00- 05 16 1,139.51       ← debit: amount 300.00- (delta −300.00)
GARDEN SERVI                                    ← continuation → fold
BILL PAYMENT 1,871.46- 05 20 730.55-            ← balance goes negative (overdraft)
```

Parse rule (per line, from the **right**): `balance` = last token (number, optional
trailing `-`); `DD`,`MM` = the two integer tokens before it; `amount` = the token
before **that** **iff** it is a number (optional trailing `-`) — **absent** ⇒ the
line is the balance anchor (brought-forward), not a transaction; `description` =
everything before. **Signed amount = balance[i] − balance[i−1]** (INV-6); the
printed amount cross-checks its magnitude + sign.

RCP is identical but **European numbers** (`239.206,04-`) and a bi-annual span that
crosses a year boundary (INV-8 year-rollover). RCP fee lines carry a `*##` marker
(`LOAN SERVICE FEE *## 13,80-`) and reference continuation lines
(`"302393463 @09,250%"`, `302393463 - RCC`) — all folded/stripped by INV-9.

### Family B — Home Loan

Column header (wraps): `Posting Date │ Effective Date │ [Description] │ Cash Fee │
Debit │ Credit │ Balance`. Dates are **full ISO**; there is **no printed
per-amount sign**:

```
2025-03-01 BROUGHT FORWARD 58,175.07                     ← anchor: 1 date, no amount
2025-03-01 2025-03-01 Service Fee HL * 69.00 58,244.07   ← delta +69.00 (debt up)
2025-03-05 2025-03-05 Autobank Transfer 25,000.00 33,987.80 ← delta −25,000 (payment)
```

Parse rule: from the **left**, take 1–2 ISO dates (`YYYY-MM-DD`); the **posting**
(first) date is `occurred_on`. From the **right**, `balance` = last number,
`amount` = second-to-last number (absent on the anchor row); `description` = the
middle. **Sign is balance-delta only** (INV-6) — the sole family with no printed
sign to cross-check, so the delta *is* the amount.

### Family C — Credit card

Page 2 carries the transactions in **two columns per line**; header `Date
Description Amount Date Description Amount`. There is **no per-transaction running
balance** — only a brought-forward and a closing figure:

```
20 Apr 26 Balance Brought Forward 2,704.14                     ← opening (skip as txn)
Credits Credits                                                ← section header (skip)
28 Apr 26 Fund Transfers -5,000.00 20 May 26 Fund Transfers -4,504.51  ← 2 credits
11 May 26 Fund Transfers -1,400.00                             ← 1 txn (odd row)
Debit Debit                                                    ← section header (skip)
21 Apr 26 Chinacashandcarry 350.00 9 May 26 Checkers Sixty60 415.94    ← 2 debits
24 Apr 26 Patreon* Membership 5.75 95.41 ...                   ← desc holds "5.75"; amt 95.41
Closing balance 1,983.25                                       ← closing (skip)
```

Parse rule: within the bounded table region, **split each line at each `DD Mon YY`
date** to recover 1–2 transaction segments; a segment is `date description
amount`, `amount` = the **last** numeric token (so a price inside the description,
"Patreon* Membership 5.75", is kept in the description). **Sign = the printed
figure, flipped** to the budget view (D3): a purchase `350.00` → `−350.00`
(spent), a credit `-5,000.00` → `+5,000.00`. Order-independent (no balance chain).
A segment whose description is "Balance Brought Forward" / "Closing balance", or
a line that is a section header / the masked "Account …" line, is skipped
(INV-9). Checksum: `brought_forward + Σ(printed) == closing` (INV-10).

## Design decisions

- **D1 — One text reader, family-dispatched; not one importer per statement
  type.** `StandardBankImporter.parse` detects the family
  (`detect_standard_bank`) and dispatches to one of three **pure** line-grammar
  functions (`_parse_family_a` / `_parse_family_b` / `_parse_family_c`) in the
  **same** module. Rejected: a module per account type (5+ files) — the families
  share the number-format, year-inference, continuation-fold, checksum, and
  `parse_transaction` machinery, so per-type files would duplicate it (coding.md
  § 1.3; the user's explicit "no per-bank-per-type importers"). A future *other*
  bank would be a sibling module implementing the same `parse(bytes) -> ParseResult`
  — **not built now** (YAGNI; the user banks only at Standard Bank).
- **D2 — Read the text layer, not the table geometry.** The reader uses
  `page.extract_text()` (stable across all five layouts) rather than
  `extract_tables()` (empirically collapses the Current account, unreadable on the
  credit card). This is the core reason the phase exists; the generic table path
  stays as the **fallback** for unrecognised layouts (INV-2).
- **D3 — Budget-view signs: balance-delta for A/B, flipped-print for C.** The
  user chose "money leaving = negative everywhere" (purchases negative). For the
  balance families this is the natural balance-delta (INV-6). For the credit card
  (no per-row balance) it is `−(printed)`, because the card's own convention is
  "a purchase increases what you owe" (positive) — flipping yields spent-money-
  negative. The **loan caveat** (a loan fee reads positive under balance-delta) is
  documented (INV-6) and a loan-sign toggle deferred, not silently resolved
  (global rule § 8 — surfaced to the user at design time; § Out of scope).
- **D4 — Detection is conservative: legal marker AND a family header signature.**
  `detect_standard_bank(full_text)` requires the SB legal string ("Standard Bank
  of South Africa", present in **every** statement's footer — the masthead is an
  **image**, not text, so the footer legal line is the reliable marker) **and**
  one family's column-header signature (Family A: a line with `Debits` + `Credits`
  + `Date` + `Balance`; Family B: `Posting` + `Effective` + `Balance` + ISO-dated
  rows; Family C: `Date Description Amount Date Description Amount`). Missing either
  ⇒ `None` ⇒ generic fallback (INV-2). Order of checks: C, then B, then A (C and B
  have the most specific headers). A statement matching **no** family header — even
  with the legal marker — falls through (an un-characterised SB layout is safer on
  the generic path than mis-parsed).
- **D5 — Rename `preview_ofx` → `preview_result` (format-neutral seam).**
  `preview_ofx(result, account_id)` is already a one-line delegate to
  `_preview_from_result` (`import_.py:145-150`). Rename it `preview_result` and
  update the single OFX call-site + the OFX test, so **OFX and SB** share one
  wizard-facing preview entry for a pre-built `ParseResult` (coding.md § 1.3
  clause 2 — refactor to cover the new case; the OFX call-site benefits, behaviour
  unchanged). `_preview_from_result` stays private. The OFX import suite must stay
  green over the rename.
- **D6 — The wizard SB pre-check sits inside the existing `_select_pdf`, before
  the generic table extraction.** After `_normalise_to_plaintext` (reused; the
  password loop already wraps it), the branch runs `detect_standard_bank` on the
  concatenated page text. On a hit: `StandardBankImporter.parse` → `ParseResult`
  → `preview_result` → **preview step** (skip the map step + table chooser, like
  the OFX branch). On a miss: the **existing** FIBR-0009 table path runs unchanged.
  The password prompt / stored-password / re-prompt loop is **shared** — it wraps
  the normalise, so both the SB parse and the generic extract sit inside one
  `PasswordError` retry (no duplication).
- **D7 — `occurred_on` is the posting date (Family B); the effective date is not
  imported.** Standard Bank's Home Loan prints both a Posting and an Effective
  date (usually equal or ±1 day). The **posting** date is always present (the
  anchor row has only it) and is the ledger date, so it is `occurred_on`; the
  effective date is dropped (no field for it, and it does not affect dedup). A
  later phase could surface both if a user asks (out of scope).
- **D8 — Year inference is period-anchored with rollover (Family A).** The bare
  `MM DD` gets its year from the "Statement from `<D Month YYYY>` to `<D Month
  YYYY>`" line: seed the year at the start endpoint's year and increment it the
  first time a transaction's month number is **less than** the previous
  transaction's (one span never covers >12 months, so a single rollover suffices).
  If the period line is absent/unparseable (it is present on A and C; B has none),
  fall back to deriving the span from the parsed dates themselves (Family B's dates
  are already full ISO, so no inference is needed there). The period line also
  yields `period_start`/`period_end` for the `ParseResult` (INV-8); when absent,
  they default to the min/max parsed date (as the CSV path does for a span-less
  file).
- **D9 — Number format is detected once per statement and required consistent.**
  Scan the statement's money tokens; the decimal separator is the punctuation
  before the final two digits (`.` ⇒ US, `,` ⇒ European). All tokens must agree;
  a statement with **both** `1,234.56` and `1.234,56` shapes is refused with a
  friendly `ValueError` (a real SB statement is internally consistent — mixed
  shapes signal a mis-read, INV-10 territory). Parsing strips the thousands
  separator and normalises the decimal to `.` before `Decimal(...)`.
- **D10 — The checksum is a hard parse gate, not a warning (INV-10).** After
  parsing, the reader asserts the balance identity (A/B: opening + Σ signed ==
  closing; C: brought_forward + Σ printed == closing) to **exact** 2-decimal
  equality. A mismatch means a row was mis-parsed (a continuation misread as a
  transaction, a dropped row, a number-format slip) — the reader raises a friendly
  `ValueError` ("this statement didn't add up — its running balance and
  transactions disagree; try the generic table import or your bank's CSV/OFX")
  rather than importing a silently-wrong figure (global rule § 1; coding.md § 2).
  The opening/closing balances are read from the anchor row + the last
  transaction's balance (A/B) or the "Balance Brought Forward"/"Closing balance"
  lines (C).
- **D11 — Continuation folding is bounded to the table region (INV-9).** The
  table region is `[column-header line, first terminator line)` where a terminator
  is any of the enumerated post-table markers ("Please verify", "The Standard Bank
  of South Africa", "VAT Summary", "Interest Calculation", "Limit Structure",
  "Fee Structure", "Account Summary", "End of Statement", "***End of Statement***",
  "*## These fees", "## These fees", the credit-card advisory "* * * * *"). Only
  lines **inside** the region are parsed; a non-transaction line inside it folds
  into the preceding transaction's description. Footer prose can therefore never
  fold. A per-page region is recomputed on each page (the header repeats).
- **D12 — Multi-page continuation via the repeated brought-forward (INV-9).**
  Transactions flow across pages; each continuation page repeats "BALANCE BROUGHT
  FORWARD" whose balance equals the prior page's closing. The reader concatenates
  pages in order and treats each brought-forward as a **balance re-anchor** (not a
  transaction), so the delta chain is seamless and no row is double-counted. This
  is a stronger guarantee than FIBR-0009 D8 (which only stitched a repeated
  **header**); here the repeated **balance** stitches the ledger.

## Invariants (testable)

The *At a glance* table is canonical (INV-1..INV-13). Sub-invariants:

- **INV-3a** — a Family-A description containing an embedded `MM DD`-shaped number
  ("INTEREST ON OVERDRAFT UP TO 08 24") parses with that text **kept in the
  description**, because only the fixed right tail is peeled (INV-3).
- **INV-5a** — a Family-C description containing an embedded price ("Patreon*
  Membership 5.75") keeps the price in the description; the **amount** is the last
  numeric token (`95.41`) (INV-5).
- **INV-6a** — a Home-Loan row (no printed sign) gets the correct sign purely from
  the balance delta: a fee is `+`, a payment is `−` (INV-4/6).
- **INV-6b** — a Family-A row's balance-delta magnitude+sign equals its printed
  amount; a deliberately corrupted fixture (printed amount ≠ delta) surfaces a
  `RowError`, not a silent wrong draft (INV-6).
- **INV-7a** — a European-number fixture (`1.910,76-`) parses to the same minor
  units a US-number fixture (`1,910.76-`) of the same value does (INV-7).
- **INV-8a** — a Family-A fixture whose transactions cross a Dec→Jan boundary
  assigns the correct years via rollover (INV-8).
- **INV-10a** — a fixture whose transactions do **not** reconcile to the printed
  closing balance raises the friendly checksum `ValueError` (INV-10).
- **INV-12a** — the detector returns `None` for a non-SB PDF (a plain gridded CSV-
  style PDF) and for an SB-legal-marker PDF with no recognised family header, and
  the wizard then uses the generic path (INV-2).

The security invariant INV-11 (in-memory decrypt reuse, no secret logged) and the
resource-bound INV-13 each carry a headless test on a `pikepdf`-encrypted
synthetic fixture; the pure line-grammar functions are unit-tested on **synthetic
text strings** (no PDF needed).

## Fixtures (synthetic — no real data)

**Hard rule: not one byte of the user's real statements, account numbers, or ID
number is committed.** All fixtures are **synthetic** SB-shaped PDFs generated
with `reportlab` (a probe/authoring tool, **not** a project or test dependency —
identical to FIBR-0009's `reportlab` usage), committed as small binary blobs:

- One **Family-A** blob (US numbers, single page) + one **European-number**
  Family-A blob (INV-7a) + one **year-rollover** Family-A blob (INV-8a).
- One **Family-B** (Home Loan, ISO dates, no printed sign) blob (INV-6a).
- One **Family-C** (credit card, two-column, sections) blob (INV-5/5a).
- One **multi-page** Family-A blob with a repeated brought-forward (D12/INV-9).
- One **checksum-fail** blob (a transaction row's amount altered so the balance
  no longer reconciles) (INV-10a) and one **printed-≠-delta** blob (INV-6b).
- An **encrypted** variant produced in-test by `pikepdf`-encrypting a plaintext
  blob (INV-11/INV-12 password path).

The pure line-grammar functions are additionally unit-tested on **synthetic text
strings** built inline in the test (no PDF), so the grammar is covered without a
fixture. Every fixture uses invented merchants, amounts, and a fake account
number (never `5239 **** **** 9556`, never the real ID).

## Out of scope (deferred)

- **A loan-sign toggle** (present a loan fee as negative). INV-6's balance-delta
  reads a loan fee positive (debt up); flipping loan accounts to a
  "cost = negative" view is a **user preference**, deferred to a follow-up once the
  user has seen real loan data and decided (global rule § 8 — surfaced, not
  silently chosen).
- **Non-Standard-Bank statements.** Other banks use CSV/OFX or the FIBR-0009
  generic table path; no other bank profile is built (YAGNI).
- **Un-characterised SB layouts** (a statement type the user does not hold, or a
  future SB redesign) — falls through to the generic path (INV-2), not force-parsed.
- **The effective date** (Family B imports the posting date only, D7).
- **Investment-account statements** — the user will export CSV for those (no PDF
  sample exists); handled by the existing CSV path, not here.
- **Scanned / image-only PDFs (OCR)** — no text layer (inherited from FIBR-0009).

## Verified basis

Empirically verified against the user's **real** Standard Bank statements in the
project venv before drafting (global rule § 13), password supplied by the user and
passed via an environment variable, **never** logged/written/committed:

- **Versions:** `pdfplumber 0.11.10`, `pikepdf 10.9.1` (the pinned stack).
- **All five statements are user-password-encrypted** with the same ID; the
  reused `_normalise_to_plaintext` decrypts each in memory and `pdfplumber`
  extracts text.
- **Family A right-anchored grammar** reproduced on Current, Savings, RCP; the
  **balance-delta** equals the printed signed amount on every sampled row,
  including across a balance sign-flip into overdraft (`BILL PAYMENT 1,871.46- …
  730.55-`, delta `−1,871.46`).
- **Family B** (Home Loan) prints **no per-amount sign**; the balance delta is the
  only sign source (`Service Fee … 69.00 … 58,244.07` after `58,175.07` ⇒ `+69.00`;
  `Autobank Transfer 25,000.00 … 33,987.80` after `58,987.80` ⇒ `−25,000.00`).
- **Family C** (credit card) is two-columns-per-line with `Credits`/`Debit`
  sections; credits print negative (`Fund Transfers -5,000.00`), purchases
  positive (`Chinacashandcarry 350.00`); a price can sit inside a description
  (`Patreon* Membership 5.75 95.41`, amount `95.41`); brought-forward `2,704.14`
  and closing `1,983.25` bound the checksum.
- **European numbers** confirmed on the Revolving-Credit statement
  (`239.206,04-`, `1.910,76-`), US on all others (`1,427.41`).
- **Period line** "Statement from `<D Month YYYY>` to `<D Month YYYY>`" present on
  A and C (incl. RCP's year-crossing "19 August 2023 to 17 February 2024"); Home
  Loan has none (its ISO dates need no inference).
- **The masthead is an image** (no "Standard Bank" text on page 1 of Current); the
  reliable text marker is the footer "The Standard Bank of South Africa Limited"
  (present on every statement) — the basis for D4's legal-marker sniff.
- **The generic FIBR-0009 table path** was re-run on all five: Home Loan / Savings
  / RCP extract row-aligned; the **Current account collapses** into one cell and
  the **credit card** yields two mega-cells — the empirical motivation for this
  phase (§ Theme table).
