> [FIBR-0008](FIBR-0008.md) (the OFX importer's per-statement `ParseResult`
> mirrored — SB is one account per PDF, so `parse` returns a bare `ParseResult |
> None` rather than OFX's `list[(OfxAccountInfo, ParseResult)]` — and its
> `preview_ofx` seam renamed — D5).

**Theme.** FIBR-0009 shipped a **generic** PDF path: lift a **ruled table** off
the page (`pdfplumber.extract_tables()`), serialise it to CSV, feed the CSV
pipeline. That path is right for a bank whose statement is a clean ruled grid —
but it is **fragile** on real Standard Bank statements, empirically:

| Real SB statement | Generic table path (FIBR-0009) |
|---|---|
| Home Loan | ✅ extracts row-aligned |
| Savings (Puresave) | ✅ extracts row-aligned |
| Revolving-Credit (RCP) loan | ✅ extracts row-aligned |
| Money Market (investment) | ~ (not re-probed via tables; text is clean) |
| **Current account** | ❌ **collapses** — all transactions into one cell |
| **Credit card** | ❌ two-columns-per-line, no grid lines — unreadable |

Same bank, same visual language, but the **table geometry** is unreliable (the
Current account has the same column shape as Savings yet collapses; a denser
statement could regress the same way). The **printed text**, however, is stable
across every layout — it is what a human reads. So this phase adds **one Standard
Bank reader** that parses the **text lines** (not the grid) and feeds the existing
preview → dedup → commit pipeline. It is **one reader for all account types**, not
one-per-type (the user's explicit constraint): the account families differ only in
a small, enumerated set of line-grammar knobs handled inside the single module.

## At a glance — invariants

Families: **A** = transactional (Current / Savings / Revolving-Credit); **B** =
Home Loan; **D** = Money Market / investment; **C** = credit card. A, B and D are
**running-balance** families (each row prints a running balance — regardless of how
many other columns it has); C has no per-row balance.

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | `StandardBankImporter.parse(pdf_bytes, exponent, password) -> ParseResult \| None` produces the **same** `ParseResult` (drafts + period; SB import is all-or-nothing, so `errors` is always empty on success — INV-11) an OFX statement does, and the wizard feeds it through the **public** `preview_result` (the renamed `preview_ofx`, D5) → the unchanged `_preview_from_result` → `_dedup` → `commit_import` tail — so an SB amount obeys the **identical** money + dedup contract (`parse_transaction`, reused verbatim) as CSV/OFX/manual. | INV-7 | **Sign.** For the balance families (A, B, D): **imported signed amount = the printed amount's magnitude carrying the sign of the running-balance delta** `balance[i] − balance[i−1]` (both balance signs honoured, incl. an overdraft `730.55-`). The delta sign is authoritative (it is the only sign source for Home Loan); where the amount also prints a sign (A trailing `-`, D leading `-`), it must **agree** with the delta sign. For credit card (no per-row balance): **`−(printed signed)`** (D3). **Documented loan consequence:** on a **loan** a fee raises the balance and so imports **positive** (debt up); a loan-sign toggle is deferred (§ Out of scope) | § Verified basis (delta reproduced on real data); coding.md § 2 |
| INV-8 | **Both South-African number formats parse**, detected **per statement** and required **consistent**: US `1,427.41` (`,` thousands, `.` decimal) and European `239.206,04` (`.` thousands, `,` decimal — the Revolving-Credit statement). Detection reads, over the statement's **money tokens only** (amount/balance figures — after stripping the sign **wherever it prints** (trailing for Families A/RCP, leading for Family B's negative balances and Family C credits / Family D) and any `R` prefix; **excluding** interest-rate / reference tokens like `@09,250%` or `7.050%`), the decimal separator as the punctuation preceding the final two digits; a statement mixing both separators-as-decimal is refused with a friendly `ValueError` (D9) | § Verified basis (the RCP European sample); coding.md § 2 |
| INV-9 | **Dates resolve to a full ISO `YYYY-MM-DD`.** Family A's bare `MM DD` gets its **year inferred** from the "Statement from `<D Month YYYY>` to `<D Month YYYY>`" period: transactions are walked in statement order from the start year, and the year **increments** whenever a transaction's month number is **less than** the previous transaction's (under SB's chronological print order a month decrease is always a year wrap — so a Nov→Feb gap with no Dec/Jan activity still rolls, which a strict `12→1` test would miss; D8). Families B (`YYYY-MM-DD`), C (`D[D] Mon YY`, 2-digit year) and D (`YYYY MM DD`) carry explicit years — no inference | § The four layout families |
| INV-10 | **Only transaction rows become drafts; the printed closing figure is captured when present (case-insensitive markers).** A "balance brought forward" / "brought forward" row (matched case-insensitively — it prints as "BALANCE BROUGHT FORWARD" / "BROUGHT FORWARD" / "Balance brought forward" / "Balance Brought Forward" across families) is a **balance anchor**, not a transaction (it seeds the delta chain + gives `opening_balance`, incl. the per-page repeat on a continuation page, D12). The statement's **independently-printed closing figure**, **where the family prints one** (Current "Balance at date of statement"; Revolving-Credit "Balance outstanding at date of statement"; Family B the "CLOSING BALANCE" ledger row; Family D "Balance as at …"; Family C "Closing balance" — **Savings prints none**, only a calendar "Month-end Balance" that is **not** used), is **captured** as the completeness target (D10) — not imported as a transaction. Section headers ("Credits"/"Debit"), the masked "Account …" line, and the whole post-table region (VAT Summary / Interest Calculation / Interest payment details / Limit-or-Fee Structure / Account Summary / legal footer / "End of Statement") are **skipped**. A line with no date+amount tail (a `VAS…`/`*****…` reference, a second-line payee) is **folded into the preceding transaction's description**. The table region is bounded (column-header line → first terminator line, recomputed per page) so footer prose can never fold into a description | § The four layout families; coding.md § 2 |
| INV-11 | **A parsed statement is integrity-checked; SB import is all-or-nothing.** The **primary correctness gate** for the balance families (A/B/D) is per-row `\|delta\| == printed magnitude` **and** the printed sign (where one prints) agreeing with the delta sign (INV-7b) — non-tautological (it compares the balance-derived delta against the separately-printed amount), catching any mis-parsed row and any mid-statement dropped row (its delta then spans two transactions). A **completeness gate** additionally fires **where the family prints an independent closing figure** (INV-10): `opening_balance + Σ(signed printed magnitudes) == that printed closing` — catching trailing-row truncation (a dropped tail leaves Σ short of the **independent** closing; using the last transaction's own balance would telescope to a tautology). Credit card (C, no running balance): the sole gate `brought_forward + Σ(printed signed) == printed "Closing balance"`. **Coverage limits (stated honestly, global rule § 1):** a statement lacking an independent closing (Savings — single page, distinctive per-page terminators) relies on the per-row gate only, so a clean **trailing** truncation is not separately caught (a multi-page Current/RCP would only reach this state if SB renamed its exact closing marker — `_capture_closing` returning `None` — which D2's text-stability premise treats as out of scope); and a **compensating adjacent run (two or more)** of dropped rows whose amounts sum to zero passes both gates. Any gate failure, or any `parse_transaction` rejection, raises **one** friendly `ValueError` ("this statement didn't add up — try your bank's CSV or OFX export") — **no partial import** (the message points only at CSV/OFX, not the generic table path: detection is a deterministic gate, so re-picking the same PDF re-detects SB and hits the same failure — the generic path is unreachable for a recognised-but-failing SB statement). **Quiet-month exception:** a recognised statement with **zero** transactions that reconciles (`opening == closing`, or, absent a printed closing, an anchor with no rows) returns an **empty-draft `ParseResult`** (period only, like a quiet-month OFX — FIBR-0008 D14), so the period is recorded (D13) | coding.md § 2; global rule § 1 (no silent wrong result) |
| INV-12 | **Reused security spine, no new surface.** The wizard decrypts **once** (the shared `_decrypt_pdf` password loop, D6) yielding plaintext bytes that both the SB reader and the generic extractor consume (their internal `_normalise_to_plaintext` is a cheap idempotent re-save on already-plaintext bytes); decrypted bytes are **never** on disk (security-model INV-4); `pikepdf.PasswordError` re-prompts (FIBR-0009 INV-3); the byte/page/row caps hold; and **no secret is logged** — a sentinel password appears in **no** log record (`caplog`) and **no** exception message (`str(exc)`). No new network import; the module is `eval`/shell-free (`bandit`) | FIBR-0009 INV-2/INV-3/INV-9; security-model INV-4/INV-8/INV-9 |
| INV-13 | Import-wizard SB round-trip (`qtbot`): picking a recognised SB `.pdf` → **if encrypted**, the existing password dialog (a wrong password re-prompts, FIBR-0009 INV-3; a remembered password auto-applies) → **straight to preview** with **no** mapping step and **no** table chooser (unlike the generic PDF path) → **Import** inserts exactly the new rows + records the period; a second import of the same statement previews all-duplicate and adds zero (reused multiset-delta dedup) | FIBR-0008 INV-7 (OFX skips mapping); FIBR-0009 INV-7; FIBR-0007 INV-5/6/7 |
| INV-14 | PDF is **untrusted, bounded input** (reused): the SB `parse` **re-checks** `_MAX_PDF_PAGES` itself (it bypasses `candidate_tables`, where FIBR-0009's page-cap lives — `pdf_importer.py:141` — so the check must be repeated after its own `pdfplumber.open`) and refuses over-long input with a whole-statement transaction cap reusing FIBR-0009's `_MAX_PDF_ROWS` (100 000); the byte cap `_MAX_IMPORT_BYTES` is enforced by the service's `read_file_bytes` (called by the wizard) before the bytes are read. Extracted text is **inert** — a cell that looks like a formula/path round-trips as literal `str` through `parse_transaction`; extraction is synchronous inline (FIBR-0009 D14), bounded, so a pathological input is refused, not left to hang the UI (the inertness is upheld by the **reused** `parse_transaction`, whose CSV/OFX suites already test formula/path-looking cells; no new SB fixture needed for it) | FIBR-0009 INV-9/D14; security-model INV-5a/INV-5b |

## Context

FIBR-0009 (P07) reused, unchanged, the FIBR-0007 CSV pipeline and added a generic
PDF **table** extractor + the in-memory decrypt. This phase reuses **both** — the
FIBR-0007 write pipeline **and** FIBR-0009's decrypt/caps — and adds a Standard
Bank **text** reader beside the generic table extractor. It reuses, unchanged:

- **`_normalise_to_plaintext(raw, password)`** (`importers/pdf_importer.py:165`) —
  the in-memory `pikepdf` decrypt + save-strip (FIBR-0009 INV-2), exposed to the
  wizard via a thin public `PdfImporter.decrypt_to_plaintext` (D6). The SB reader
  opens the plaintext with `pdfplumber` to read **text** (`page.extract_text()`).
- **`parse_transaction(occurred_on: str, raw_amount: str | Decimal, description:
  str, exponent: int)`** (`services/transactions.py:35`) + **`read_minor_unit_exponent`**
  (same module, ~`:24`) — the money contract. The SB reader builds a signed
  `Decimal` and calls `parse_transaction` with the **same argument shape** as
  `OfxImporter._parse_statement` (`ofx_importer.py:91`); unlike OFX (which collects
  per-row `RowError`s and continues) the SB reader is all-or-nothing and **raises**
  on any rejection (INV-11).
- **`ParseResult` / `RowError` / `TransactionDraft`** (`importers/base.py`,
  `models.py`) — the uniform importer output.
- **`ImportService.preview_result` (renamed from `preview_ofx`, D5) /
  `_preview_from_result` / `commit_import` / `_dedup`** — the dedup-analysis +
  atomic write tail. The wizard calls `preview_result` so the SB path reaches the
  **identical** `commit_import` as CSV/OFX.
- **The import wizard's PDF branch + password loop** (`ui/import_wizard.py`,
  `ui/password_dialog.py`) — the `pikepdf.PasswordError` catch, stored-password
  auto-try, and re-prompt are reused (refactored into `_decrypt_pdf`, D6); the SB
  branch is a **pre-check** inside the existing `_select_pdf`, not a new screen.
- **The `accounts.statement_pdf_password` column + accessors** (v5, FIBR-0009) —
  the remembered-password path is reused wholesale; **no schema change** here.

So the genuinely new code is: the `StandardBankImporter` (text reader + the pure
line-grammar helpers), the `detect_standard_bank` sniff, the `_decrypt_pdf`
refactor + the small `decrypt_to_plaintext` accessor, the wizard SB pre-check, and
the format-neutral rename of `preview_ofx` (D5). **No new runtime dependency**
(`reportlab`, used to author synthetic fixtures, is a probe/test tool, never
shipped — as in FIBR-0009) and **no migration**.

## The four layout families (empirical grammar)

All grammar below is transcribed from the user's **real** statements via
`pdfplumber.page.extract_text()` (§ Verified basis). Each **balance-family** (A/B/D)
worked run below is a **contiguous** set of real extracted lines that chains
exactly. The **credit-card** run (Family C) is an explicitly-captioned **partial**
listing (elided rows) — it has no running balance to chain, so it makes no
reconciliation claim.

### Family A — transactional (Current, Savings, Revolving-Credit "RCP")

Column header (may **wrap across 2–3 text lines**): `Details │ Service Fee │ Debits
│ Credits │ Date │ Balance` (Savings shows it on one line; Current and the
Revolving-Credit (RCP) statement wrap; RCP orders the middle pair `Credits /
Debits`). Transactions are **right-anchored**; a contiguous run:

```
BALANCE BROUGHT FORWARD 05 13 1,427.41          ← anchor: MM DD balance, NO amount
LOTTERY WINNINGS 12.10 05 15 1,439.51           ← +12.10   (1,439.51 − 1,427.41)
VAS00216222570                                  ← continuation → fold into prior desc
PAYSHAP PAYMENT TO 300.00- 05 16 1,139.51       ← −300.00  (1,139.51 − 1,439.51)
FEE: PAYSHAP PAYMENT ## 7.00- 05 16 1,132.51    ← −7.00    (1,132.51 − 1,139.51)
```

Overdraft (balance crosses to negative) — a contiguous run showing the sign-flip:

```
LOTTERY WINNINGS 8.40 05 18 1,140.91            ← +8.40    (1,140.91 − 1,132.51)
BILL PAYMENT 1,871.46- 05 20 730.55-            ← −1,871.46 (−730.55 − 1,140.91)
FEE BILL PAYMENT ## 1.50- 05 20 732.05-         ← −1.50    (−732.05 − −730.55)
```

Parse rule (per line, from the **right**): `balance` = last token (number, optional
trailing `-`); `DD`,`MM` = the two integer tokens before it; `amount` = the token
before **that** **iff** it is a number (optional trailing `-`) — **absent** ⇒ the
line is a **balance anchor** (it is amount-less **and** its description matches the
case-insensitive "brought forward" marker, INV-10), not a transaction; `description`
= everything before. (An amount-less line with a `MM DD`+balance tail that does
**not** match the marker is a mis-parse → the all-or-nothing `ValueError`, never a
silent re-anchor.) Sign per INV-7. RCP is identical but **European numbers**
(`239.206,04-`) and a bi-annual span crossing a year boundary (INV-9 rollover); its
fee lines carry a `*##` marker and reference continuation lines
(`"302393463 @09,250%"`, `302393463 - RCC`) — folded/stripped by INV-10.

### Family B — Home Loan

Column header (wraps): `Posting Date │ Effective Date │ [Description] │ Cash Fee │
Debit │ Credit │ Balance`. Dates are **full ISO**; the amount magnitude prints but
carries **no sign**; a contiguous run:

```
2025-03-01 BROUGHT FORWARD 58,175.07                        ← anchor: 1 date, no amount
2025-03-01 2025-03-01 Service Fee HL * 69.00 58,244.07      ← +69.00   (delta, debt up)
2025-03-03 2025-03-02 Insurance Premium 743.73 58,987.80    ← +743.73
2025-03-05 2025-03-05 Autobank Transfer 25,000.00 33,987.80 ← −25,000.00 (delta, payment)
```

Parse rule: from the **left**, take 1–2 ISO dates (`YYYY-MM-DD`); the **posting**
(first) date is `occurred_on` (D7). From the **right**, `balance` = last number,
`amount` = second-to-last number (absent on the anchor row); `description` = the
middle. **Sign is the balance delta** (INV-7) — the only family with no printed
sign to cross-check, so the magnitude cross-check `|delta| == printed amount`
(INV-7b) is the mis-parse gate.

### Family D — Money Market / investment

Column header: `Date │ Transaction description │ Withdrawals │ Deposits │ Interest
rate │ Balance`, with a `ccyy mm dd` sub-header. Dates are **full space-separated
`YYYY MM DD`**; amounts and the running balance are **`R`-prefixed** and carry a
**leading** sign; a contiguous run:

```
2026 03 26 Balance brought forward R709,297.74              ← anchor: full date, no amount
2026 03 28 Ib transfer to -R4,200.00 R705,097.74            ← −4,200.00 (withdrawal)
*****9000740 06H54 *****5235                                ← continuation → fold
2026 03 28 Ib transfer to -R500.00 R704,597.74             ← −500.00
2026 03 30 Ib transfer to -R15,000.00 R689,597.74          ← −15,000.00
2026 03 31 Credit interest R4,282.14 R693,879.88           ← +4,282.14 (interest); … more rows …
… (later, after all rows) …
Balance as at 26 June 2026 R623,457.48                     ← the printed CLOSING figure
                                                              (INV-10 capture, INV-11 checksum target)
```

(The five lines above are a **contiguous prefix** that chains exactly; the "Balance
as at" figure `R623,457.48` is the statement's final printed closing — the INV-11
checksum target — after the elided intervening rows, **not** the running balance of
the last row shown.) Parse rule: from the **left**, `YYYY MM DD` (three integer
tokens) → `occurred_on`; from the **right**, `balance` = last money token, `amount`
= the token before it **iff** present (absent on the anchor). Money tokens are
`[-]R<number>`: strip the `R` (and the leading `-`) before number parsing (INV-8).
Sign is the printed leading sign, cross-checked against the balance delta
(INV-7/7b). **Page 2's
"Interest payment details" table is not transactions** — its header is not the
transaction header and "Balance as at …" / "Interest payment details" / "VAT
Summary" terminate the region (INV-10), so its rows are never parsed as
transactions.

### Family C — credit card

Page 2 carries the transactions in **two columns per line**; header `Date
Description Amount Date Description Amount`. There is **no per-transaction running
balance** — only a brought-forward and a closing figure:

```
20 Apr 26 Balance Brought Forward 2,704.14                     ← opening (skip as txn)
Credits Credits                                                ← section header (skip)
28 Apr 26 Fund Transfers -5,000.00 20 May 26 Fund Transfers -4,504.51  ← 2 credits
11 May 26 Fund Transfers -1,400.00                             ← 1 txn (odd row)
Debit Debit                                                    ← section header (skip)
21 Apr 26 Chinacashandcarry 350.00 9 May 26 Checkers Sixty60 415.94    ← 2 debits
24 Apr 26 Patreon* Membership 5.75 95.41 ...                   ← desc keeps "5.75"; amt 95.41
Closing balance 1,983.25                                       ← closing (skip)
```

Parse rule: within the bounded table region, **split each line at each
line-position `D[D] Mon YY` date** (a 1–2-digit day, so `9 May 26` matches) to
recover 1–2 transaction segments; a segment is `date description amount`, `amount`
= the **last** numeric token (so a price inside the description, "Patreon*
Membership 5.75", is kept in the description). **Sign = the printed figure,
flipped** (D3): a purchase `350.00` → `−350.00` (spent), a credit `-5,000.00` →
`+5,000.00`. Order-independent (no balance chain). A segment whose description is
"Balance Brought Forward" / "Closing balance", a section-header line, the masked
"Account …" line, or a region line with **zero** dates (folds into the prior
segment's description; a zero-date line at region **start**, with no prior segment,
is dropped) is handled by INV-10. **De-interleave risk (counter-example):**
if a *merchant description* itself contained a `D[D] Mon YY` substring, the split
would over-segment — mitigated by (a) anchoring the split only at line-leading /
post-amount date positions and (b) the genuine Family-C checksum (INV-11) failing
loudly on any mis-split **that changes the amount total** rather than importing a
wrong split. Checksum:
`brought_forward + Σ(printed) == closing` (INV-11).

## Design decisions

- **D1 — One text reader, family-dispatched; not one importer per statement
  type.** `StandardBankImporter.parse` detects the family
  (`detect_standard_bank`) and dispatches to one of four **pure** line-grammar
  functions (`_parse_family_a/b/c/d`) in the **same** module. Rejected: a module
  per account type (6+ files) — the families share the number-format,
  year-inference, continuation-fold, checksum, and `parse_transaction` machinery,
  so per-type files would duplicate it (coding.md § 1.3; the user's explicit "no
  per-bank-per-type importers"). A future *other* bank would be a sibling module

- **D7 — `occurred_on` is the posting date (Family B); the effective date is not
  imported.** The Home Loan prints a Posting and an Effective date (usually equal
  or ±1 day). The **posting** date is always present (the anchor row has only it)
  and is the ledger date, so it is `occurred_on`; the effective date is dropped (no

- **D8 — Year inference is period-anchored, rolling on any month decrease
  (Family A only).** The bare `MM DD` gets its year from the "Statement from `<D
  Month YYYY>` to `<D Month YYYY>`" line: seed the year at the start endpoint's
  year and seed the **initial "previous month" from the period-start month** (NOT
  the brought-forward anchor's date — the anchor can be dated in the *prior* period,
  e.g. a December brought-forward on a January-starting statement, which would
  otherwise false-roll the first January row to seed+1). Walking transactions in
  statement (chronological) order, increment the year whenever the **current month
  number is less than the previous transaction's**. Under SB's chronological print
  order a month decrease is always a year wrap, so this rolls correctly even across
  a **gap** (a dormant account with a Nov transaction then Feb — `11 → 2`, one roll —
  which a strict `12 → 1` test would miss, silently stamping the Feb rows with the
  prior year). An SB statement span is **≤ 12 months** (verified: the widest sampled
  is the bi-annual Revolving-Credit statement), so at most one roll occurs; a
  same-month reordering (no decrease) never trips it. Families B/C/D carry explicit
  years and need no inference (D's dates are full `YYYY MM DD`, B's are ISO, C's
  carry a 2-digit year). **Statement period** (`period_start`/`period_end`): for
  **A/C** the printed "Statement from … to …" line is **authoritative** — used
  whenever present, regardless of how many transactions parsed (the direct analogue
  of OFX's embedded DTSTART/DTEND, which `OfxImporter` treats as authoritative —
  `ofx_importer.py:108`). For **B/D** (no period line) the span is the **min/max
  parsed transaction date** when drafts exist, falling back to the statement "Date"
  line as a zero-width span for a quiet month (D13; as the CSV path does for a
  span-less file).
- **D9 — Number format is detected once per statement and required consistent.**
  Over the statement's **money tokens** (amount/balance figures only — after
  stripping the sign wherever it prints (**a trailing `-` for Families A/RCP**,
  **a leading `-` for Family B's negative balances, Family C credits, and Family
  D**) and any `R` prefix (Family D), and **excluding** interest-rate / reference tokens such as
  `@09,250%`, `7.050%`, `SBSARCP STD …`), the decimal separator is the punctuation
  before the final two digits (`.` ⇒ US, `,` ⇒ European). All money tokens must
  agree; a statement with **both** `1,234.56` and `1.234,56` shapes is refused with
  a friendly `ValueError`. (SB always prints amounts to exactly 2 decimals, so
  "punctuation before the final two digits" is well-defined.) **Detection runs in
  `parse` before family-dispatch** — a light scan gathering money-shaped tokens
  (a run of digits/`,`/`.` with a 2-digit tail, optionally `R`-prefixed and
  `-`-suffixed/prefixed) over the table region — so `fmt` is known when each
  `_parse_family_*(…, fmt)` is called; the helper then re-isolates its columns and
  parses each with `_parse_amount(token, fmt)`, which strips the `R` prefix, a `-`
  wherever it appears (returning the unsigned **magnitude**; the printed sign is read
  separately for INV-7b), the thousands separator, and normalises the decimal to `.`
  before `Decimal(...)`.
- **D10 — Per-row correctness gate always; completeness gate against an INDEPENDENT
  closing where printed; all-or-nothing (INV-11).** The mistake to avoid:
  checksumming `opening + Σ(signed) == last-transaction's-running-balance`
  **telescopes to a tautology** (it can never fail) and misses trailing truncation.
  The **primary** gate is therefore per-row `|delta| == printed magnitude` (+ sign
  agreement where printed) to exact 2-decimal equality (INV-7b) — non-tautological,
  catching a mis-parsed row or a mid-statement drop. A **completeness** gate
  `opening + Σ(signed printed magnitudes) == printed closing figure` fires **only
  where the family prints an independent closing** (`_capture_closing`, INV-10):
  **Current** "Balance at date of statement"; **Revolving-Credit** "Balance
  outstanding at date of statement"; **Home Loan** "CLOSING BALANCE" row; **Money
  Market** "Balance as at …". **Savings prints no such line** (only a calendar
  "Month-end Balance", explicitly **not** the closing) → the per-row gate is its
  sole gate; a missing closing figure on Savings is **expected, not an error**.
  `opening_balance` comes from the brought-forward anchor. Family C (no running
  balance): the single gate `brought_forward + Σ(printed signed) == printed "Closing
  balance"`. Any gate failure, or any `parse_transaction` rejection ⇒ **one** friendly
  `ValueError`, import **nothing**, directing the user to their bank's CSV or OFX
  export (global rule § 1; coding.md § 2). **Quiet-month exception** (D13): zero
  transactions that reconcile ⇒ an empty-draft `ParseResult` (period only).
- **D11 — Continuation folding is bounded to the table region (INV-10).** The table
  region is `[column-header line, first terminator line)` where a terminator is any
  enumerated post-table marker ("Please verify", "The Standard Bank of South
  Africa", "VAT Summary", "Interest Calculation", "Interest payment details",
  "Balance as at", "CLOSING BALANCE" (the Home-Loan ledger closing row — a terminator
  so it is neither imported as a transaction nor folded into the prior description),
  "Balance at date of statement" + "Balance outstanding at date of statement" (the
  Current/RCP closings — normally on a summary page outside the region, listed
  defensively so a short single-page variant printing one in-region still terminates
  rather than folds), "Limit Structure", "Fee Structure", "Account Summary", "End of Statement",
  "***End of Statement***", "*## These fees", "## These fees", the credit-card
  advisory "* * * * *"). Only lines **inside** the region are parsed; a
  non-transaction line inside it folds into the preceding transaction's
  description. Footer/interest-schedule prose can therefore never fold. The region
  is recomputed per page (the header repeats).
- **D12 — Multi-page continuation via the repeated brought-forward (INV-10).**
  Transactions flow across pages; each continuation page repeats "BALANCE BROUGHT
  FORWARD" whose balance equals the prior page's closing. The reader concatenates
  pages in order and treats each brought-forward as a **balance re-anchor** (not a
  transaction), so the delta chain is seamless and no row is double-counted — a
  stronger guarantee than FIBR-0009 D8 (which stitched a repeated **header**; here
  the repeated **balance** stitches the ledger).
- **D13 — Degenerate inputs behave predictably, never crash.** A recognised-family
  PDF with **zero** transactions that **reconciles** (`opening == printed closing`,
  or — for a family with no printed closing — an anchor with no rows) returns an
  **empty-draft `ParseResult`** carrying the period, so the wizard's quiet-month path
  records the statement span (matching the OFX behaviour it already ships —
  `import_wizard.py:562-564`, FIBR-0008 D14). The period comes from the "Statement
  from … to …" line for A/C; for **B/D (no period line)** a zero-transaction
  statement has no transaction dates to min/max, so the span falls back to the
  statement **"Date"** line (Home Loan prints `Date 2026 02 28`, Money Market
  `Date 26 June 2026` — two shapes, so the fallback parses both) as a zero-width
  period. A row whose
  amount is unparseable, a description empty after folding (which `parse_transaction`
  rejects), a zero-transaction statement that does **not** reconcile, or a closing
  figure absent on a family that **always prints one** (B/D/C; Family A's Savings
  legitimately prints none, so its absence there is **not** an error —
  `_verify_checksum` is family-aware) folds into D10's single friendly `ValueError`
  (all-or-nothing); the wizard shows the message and the import
  is abandoned cleanly. A **non-SB** PDF returns `None` from `parse` (INV-2), not an
  error.

## Invariants (sub-invariants)

The *At a glance* table is canonical (INV-1..INV-14). Sub-invariants:

- **INV-3a** — a Family-A description containing an embedded `MM DD`-shaped number
  ("INTEREST ON OVERDRAFT UP TO 08 24") parses with that text **kept in the
  description**, because only the fixed right tail is peeled (INV-3).
- **INV-6a** — a Family-C description containing an embedded price ("Patreon*
  Membership 5.75") keeps the price in the description; the **amount** is the last
  numeric token (`95.41`) (INV-6).
- **INV-7a** — a Home-Loan row (no printed sign) gets the correct sign purely from
  the balance delta: a fee is `+`, a payment is `−` (INV-4/7).
- **INV-7b** — a balance-family row's `|delta|` equals its printed amount magnitude
  (and, where a sign prints, the signs agree); a deliberately corrupted fixture
  (printed amount ≠ delta) trips the all-or-nothing checksum `ValueError` (INV-11),
  never a silent wrong draft (INV-7).
- **INV-8a** — a European-number fixture (`1.910,76-`) parses to the same minor
  units a US-number fixture (`1,910.76-`) of the same value does (INV-8).
- **INV-9a** — two Family-A fixtures: one crossing a Dec→Jan boundary, and one with
  a **Nov→Feb gap** (no Dec/Jan activity), both assign correct years via the
  month-decrease rollover (INV-9).
- **INV-11a** — the checksum `ValueError` fires and nothing imports for: a
  balance-family (A/B/D) fixture with a corrupted **printed amount** (trips both the
  per-row cross-check and the completeness sum) **and** one with a **trailing row
  dropped** (Σ falls short of the independent printed closing); **and** a Family-C
  fixture whose `brought_forward + Σ(printed) ≠ closing` (the credit-card gate).
- **INV-2a** — `detect_standard_bank` returns `None` for a non-SB PDF (a plain
  gridded CSV-style PDF) **and** for an SB-legal-marker PDF with no recognised
  family header (→ the wizard uses the generic path, INV-2); and, given a fixture
  whose header block satisfies **two** family signatures, the most-specific-first
  order (C→D→B→A) selects the correct family (D4).

The security invariant INV-12 (in-memory decrypt reuse, no secret logged) and the
resource-bound INV-14 each carry a headless test on a `pikepdf`-encrypted synthetic
fixture; the pure line-grammar helpers are unit-tested on **synthetic text
strings** (no PDF needed).
