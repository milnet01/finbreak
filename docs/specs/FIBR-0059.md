**Theme:** The Statements tab (FIBR-0052) lists every imported statement and can
**delete** one. This adds the third verb the user asked for: **change which
account a logged statement belongs to**. Picking "Change account" on a selected
statement opens an account picker; confirming atomically re-points **both** the
`statement_periods` row **and every transaction stamped with it**
(`statement_period_id = id`) to the chosen account — one service-owned
`BEGIN…COMMIT` that mirrors `delete_statement`, leaving manual and other
statements' rows untouched and `ROLLBACK`-ing to a re-openable vault on failure.

## At a glance — invariants

| INV | Guarantee | Upholds |
|-----|-----------|---------|
| INV-1 | The move is **atomic**: the period row's `account_id` **and** every stamped transaction's `account_id` change together, or neither — one service-owned `BEGIN…COMMIT`; any failure `ROLLBACK`s to a re-openable vault | coding.md § 2; mirrors FIBR-0052 INV-9 |
| INV-2 | **Isolation**: only the target statement's rows move — manual (`NULL`-stamped) transactions and every **other** statement's rows (on any account) are untouched | FIBR-0052 INV-8 provenance |
| INV-3 | **Span-collision guard**: re-assigning to an account that already has a *different* statement for the **identical** `(period_start, period_end)` span is **refused** with a friendly message and **no** change (else two same-span periods on one account break the import span-dedup and duplicate rows) | FIBR-0007 INV-6 |
| INV-4 | `reassign_account` returns the **number of transactions moved** (parity with `delete_statement`'s deleted count) | — |
| INV-5 | Re-assigning to the statement's **current** account is a safe **no-op** (the UI skips the call; the service is idempotent if called anyway) | coding.md § 2 |
| INV-6 | The transactions move **verbatim** — NOT re-deduped against the target account's existing rows (a statement is a coherent unit; deduping on move could silently drop the statement's own rows) | — |
| INV-7 | **UI enablement**: "Change account" is disabled with no selection, enabled on a single-row selection (mirrors *Delete selected*) | FIBR-0052 INV-10 |
| INV-8 | After a successful move the Statements list **re-lists** (the row's Account column shows the new account) and a **distinct `reassigned` signal** fires (NOT `changed`, whose shell handler shows "Statement deleted"), so the shell refreshes Home + the status count and shows a move-specific message | FIBR-0052 INV-10/INV-11 |
| INV-9 | The picker **preselects the statement's current account** (a safe default — confirming without changing is the INV-5 no-op) | — |
| INV-10 | An idle **auto-lock** firing while the picker/confirm is open raises `VaultLockedError`, which is **caught** — the click never crashes the slot (the next unlock rebuilds a fresh tab) | FIBR-0052 INV-3 / the delete path |
| INV-11 | **No schema change** — reuses the v6 tables + the `statement_period_id` stamp; all strings `tr()`, every widget in a layout manager | coding.md § 5.2 |

## Context

`StatementService` (FIBR-0052) already owns the Statements tab's read
(`list_statements`, a `JOIN accounts` for the name + a `LEFT JOIN` grouped
`COUNT` of stamped transactions) and its atomic `delete_statement` (one owned `BEGIN`: delete the
stamped children **then** the parent period, `ROLLBACK` on failure). The v6
schema stamps each imported transaction with its `statement_period_id`
(`NULL` for manual entries), which is exactly the key this feature moves rows by.
`StatementsWidget` is the tab: a `QTableWidget` (Account · Period · File ·
Imported · count) + a *Delete selected* button gated on a single selection + a
`changed` signal the shell already wires to a Home + status-count refresh whose
handler ends with a hard-coded `_status(tr("Statement deleted"))` — so the move
emits its **own** `reassigned` signal rather than reuse `changed` (D6a).

## Design decisions

- **D1 — Service method, mirroring `delete_statement`.**
  `StatementService.reassign_account(period_id: int, new_account_id: int) -> int`.
  **The D2 span guard runs first, as a pure read + refuse, BEFORE any `BEGIN`** —
  so a refusal raises with no transaction open (INV-1's rollback guarantee has no
  hole). Reads before `BEGIN` are safe: the vault opens with `isolation_level=""`
  (FIBR-0004, `vault.py`), so the driver issues no implicit `BEGIN` **before a plain
  `SELECT`**, and the two guard reads (`get`, `id_for_span`) open no transaction —
  the explicit `BEGIN` then cannot raise "cannot start a transaction within a
  transaction". Then one owned
  `conn.execute("BEGIN")`: re-point the period row's
  `account_id`, then re-point every transaction stamped with `period_id`;
  `conn.commit()`; on any exception `conn.rollback()` and re-raise (a re-openable
  vault, INV-1). Returns the transaction rowcount moved (INV-4). The two writes are
  ordered period-then-transactions (order is immaterial here — both are plain
  `UPDATE`s, no FK create/delete — but a fixed order reads clearly).

- **D2 — Span-collision guard (INV-3).** Read the moving period's span via a new
  `StatementPeriodRepository.get(period_id)`. The repo method honestly returns
  `StatementPeriod | None`; the service **narrows** it via `cast` (the
  `period_id` is always a currently-listed statement — the UI selection
  guarantees it exists), mirroring `AccountService`'s
  `cast(Account, repo.get(...))` narrow-an-Optional-when-existence-is-guaranteed
  convention (there the row is just-written; here it is the UI-selected
  statement) — not a runtime `None`-branch.
  Then look up `id_for_span(new_account_id, period_start, period_end)` and refuse
  **only when it returns an id other than the moving period's own** — i.e.
  `existing not in (None, period_id)` — raising `ValueError("that account already
  has a statement for this period …")`. **The self-exclusion is load-bearing:**
  when `new_account_id` is the statement's *current* account the lookup returns
  `period_id` itself (the period is still on that account, pre-write), which must
  **not** refuse — that is the INV-5 same-account no-op, not a collision. A
  *different* id means the target account already has another statement for that
  exact span. This mirrors the import's per-account span-uniqueness (FIBR-0007
  INV-6): two same-span periods on one account would make `id_for_span`'s
  `LIMIT 1` ambiguous and let a later import stamp onto the wrong one, duplicating
  rows.

- **D3 — Move verbatim, no re-dedup (INV-6).** A statement is a coherent set of
  rows the user is relocating; re-deduping against the target account's existing
  rows could silently delete the statement's own transactions (a match with a
  pre-existing manual row) — surprising and lossy. The move re-points all stamped
  rows unchanged. A rare coincidental duplicate with a pre-existing manual entry
  is the user's to resolve; the INV-3 span guard prevents the *systematic*
  duplication (a whole same-span statement colliding).

- **D4 — `StatementRow` gains `account_id`.** The view row (FIBR-0052) carries
  the joined account **name** but not its **id**; the picker needs the current
  `account_id` to preselect it (INV-9) and the widget needs it to skip a
  same-account no-op (INV-5). `list_statements`' `SELECT`/`GROUP BY` add
  `p.account_id` (right after `a.name`), and `StatementRow` gains
  `account_id: int` in that position. The only construction site is
  `list_statements` (`StatementRow(*row)`), so the positional ripple is contained.

- **D5 — Commit-free repository writes** (mirroring the delete path's
  commit-free `delete`/`delete_for_statement`, invoked inside the service's owned
  transaction):
  - `StatementPeriodRepository.set_account(period_id, account_id)` — one
    `UPDATE statement_periods SET account_id = ? WHERE id = ?`, commit-free.
  - `StatementPeriodRepository.get(period_id) -> StatementPeriod | None` — the
    single-row read for the D2 span lookup.
  - `TransactionRepository.reassign_account(statement_period_id, account_id) -> int`
    — one `UPDATE transactions SET account_id = ? WHERE statement_period_id = ?`,
    returning `rowcount`, commit-free.

- **D6 — UI: "Change account" + an account picker.** A second button on the
  Statements tab, `button_reassign_statement` (`self.tr("Change account")`),
  disabled with no selection and enabled on a single selection by the **existing**
  `_on_selection_changed` (extended to gate both buttons). `_on_reassign` opens
  `AccountPickerDialog(accounts, current_account_id, parent)`; on `Accepted`, if
  the chosen id **differs** from the statement's current `account_id` (INV-5 — a
  same-account pick is skipped, no service call), call `reassign_account`, then
  `refresh()` + emit a **new `reassigned` signal** (INV-8). It must be a distinct
  signal, NOT the `changed` reuse: the shell's `changed` handler
  (`_on_statement_changed`, `main_window.py`) ends with a hard-coded
  `_status(tr("Statement deleted"))`, which would misreport a successful move.
  `VaultLockedError` is caught and swallowed (INV-10, as in `_on_delete`); the
  INV-3 `ValueError` is caught and shown as a **`tr()`-composed**
  `QMessageBox.warning` (the widget renders its own translated text — the
  exception string is a control-flow signal, not display text, honouring INV-11).
  The widget builds `AccountService(service.vault)` (as the import wizard does) for
  the account list.

- **D6a — Shell wiring (small, new).** `main_window.py` connects the new
  `StatementsWidget.reassigned` signal to a `_on_statement_reassigned` slot that
  runs the **same** Home + status-count refresh as `_on_statement_changed` (extract
  the shared `_home_tab.refresh()` + `_refresh_count(...)` into one helper both
  call) but shows a move-specific `_status(tr("Statement account changed"))`. A
  move doesn't change the *total* transaction count (rows change account, not
  existence), so the count refresh is a numeric no-op — harmless, kept for
  symmetry; the status message is the real user-visible effect (INV-8). Scope the
  extraction to exactly the two statement handlers (`_on_statement_changed` +
  `_on_statement_reassigned`); the other shell sites that repeat the `refresh()` +
  `_refresh_count()` pair (`_on_import_done`, the tab refreshes) are **left
  untouched** here — folding them in is a separate debt-sweep, out of this scope.

- **D7 — `AccountPickerDialog` (new `ui/account_picker.py`).** A small `QDialog`
  (one dialog per file, matching `ui/{settings,password_dialog,unlock,first_run}.py`):
  a labelled `QComboBox` of accounts (label = name, `userData` = id) populated
  from the **passed `accounts` list** — the *widget* fetches it via
  `AccountService.list_accounts()` (D6); the dialog itself takes no service, only
  `(accounts, current_account_id, parent)`. The combo is preselected to
  `current_account_id` (`findData` with the `if index >= 0` guard, as
  `ui/settings.py`'s auto-lock combo does), plus an OK/Cancel `QDialogButtonBox`.
  Exposes `selected_account_id() -> int` (`currentData()` — always a valid id: a
  vault has **≥1 account** from creation onward (the v1→v2 migration seeds a
  "Default" account, and `AccountService.delete_account` refuses the last one with
  `LastAccountError`), so the combo is never empty). The statement's **current** account is always in the
  list — the period FK-references it and an account carrying data cannot be deleted
  — so `findData(current_account_id) >= 0` always holds and the preselect never
  silently falls through to index 0 (INV-9 can't misfire). With exactly **one**
  account the picker offers only the current one → confirming is the INV-5 no-op
  (harmless, not a dead-end; not disabled). All strings `tr()`, widgets in layouts
  (INV-11).

## Invariants

Restated with their enforcement point (the table above is the index).

- **INV-1 (atomic move)** — `reassign_account` wraps both `UPDATE`s in one owned
  `BEGIN…COMMIT`; `except: conn.rollback(); raise`. Tested by wedging the second
  write to fail and asserting neither table changed and the connection is still
  usable after the rollback (the `test_INV9b` post-rollback assertion — "usable
  after rollback", not a literal close/reopen).
- **INV-2 (isolation)** — the transaction `UPDATE` is keyed on
  `statement_period_id = ?`, so `NULL`-stamped manual rows and other periods'
  rows (matched by their own id) never move. Tested with a manual row + a second
  statement present, asserting both untouched.
- **INV-3 (span guard)** — refuses when `id_for_span(new_account_id, start, end)`
  returns an id **other than `period_id`** (the D2 self-exclusion); no write
  occurs. Tested two ways: (a) give the target account a same-span statement →
  `ValueError` + both statements unchanged; (b) a same-account call
  (`id_for_span` returns `period_id`) does **not** refuse — the INV-5 no-op.
- **INV-4 (moved count)** — returns the transactions `UPDATE` `rowcount`; a
  statement with zero stamped transactions moves the period row only and
  returns 0.
- **INV-5 (same-account no-op)** — the widget compares the chosen id to
  `StatementRow.account_id` and skips; the service, if called with the current
  account, is not refused by the D2 guard (self-exclusion) and performs idempotent
  `UPDATE`s (same value), returning the count. (SQLite `UPDATE … WHERE …` counts
  **matched** rows, not only changed ones, so the same-account rowcount is the
  transaction count, not 0.)
- **INV-6 (verbatim move)** — no dedup; all stamped rows re-pointed. Tested by
  moving a statement whose rows duplicate a target-account manual row and
  asserting the count is unchanged (nothing dropped).
- **INV-7 (enablement)** — `_on_selection_changed` sets both buttons'
  `setEnabled(selected_row is not None)`.
- **INV-8 (refresh + signal)** — `_on_reassign` calls `refresh()` then
  `reassigned.emit()` (the distinct signal, D6/D6a); the row's Account column then
  shows the new name, and the shell shows `tr("Statement account changed")` — NOT
  the `changed` handler's "Statement deleted". Tested: the row's account changes
  and a `reassigned` spy fires once.
- **INV-9 (preselect current)** — the dialog `findData(current_account_id)` with
  the standard `if index >= 0` guard.
- **INV-10 (auto-lock safety)** — `except VaultLockedError: return` around the
  service call, as in `_on_delete`. Tested by stubbing `AccountPickerDialog` to
  return `Accepted` + a target id and **monkeypatching
  `widget._statements.reassign_account`** to raise `VaultLockedError` (the widget
  owns its own `StatementService`, so patch the bound method — not a stand-in
  vault, which is the service-level INV-1 seam) — the slot returns without
  crashing.
- **INV-11 (no schema change / i18n)** — no migration; `LATEST_SCHEMA_VERSION`
  stays 6; every string `tr()`, every widget in a layout manager.

## Data model — no schema change

Reuses the FIBR-0052 v6 schema verbatim: `statement_periods.account_id`,
`transactions.account_id`, and `transactions.statement_period_id` (the
provenance stamp) already exist. This feature only issues `UPDATE`s against them;
`LATEST_SCHEMA_VERSION` stays **6** and there is no migration.

## Out of scope

- **Editing a statement's period dates, source filename, or individual
  transactions** — this cut moves only the *account*. Row-level editing is a
  separate future item.
- **Merging two statements** (moving into an account that already has the same
  span) — deliberately **refused** (INV-3), not merged; merge is its own feature.
- **Re-dedup on move** (INV-6) — out by design.
- **A generic "edit statement" dialog** — the single "Change account" action is
  the whole deliverable.

## Exit criteria

1. All INV-1..INV-11 covered by `tests/features/statements/` and green (INV-11's
   i18n half is convention/ruff-enforced, not a unit test — see Deliverable 8).

3. `/audit` 0 actionable; `/indie-review` clean (or all findings folded).
4. Manual sanity: import a statement to the wrong account, "Change account" to the
   right one, confirm the Statements row + the account transaction counts move.
