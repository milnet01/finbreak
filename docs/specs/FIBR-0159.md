## 1. Problem & intent

finbreak installs today as a GitHub-release AppImage, a Windows `.exe`, and —
since FIBR-0155 — native RPMs/debs via the openSUSE Build Service (OBS). OBS
covers openSUSE + Fedora well, but each distro family is a **separate build
target**, and OBS does not surface in the two app stores most Linux desktop
users actually browse: **GNOME Software** and **KDE Discover**.

**Flathub** (`flathub.org`) is the de-facto cross-distro app store: a *single*
submission lists finbreak in GNOME Software + KDE Discover on openSUSE, Fedora,
Ubuntu, Debian, Mint, and every other Flatpak-enabled distro. Unlike the
official distro archives (which forbid bundling and need a maintainer sponsor —
impractical for finbreak's deliberately-bundled runtime), Flathub is
**self-publish** and embraces the self-contained, **sandboxed** model, so it
fits finbreak's architecture directly.

The outcome the user wants: a user on *any* distro opens their graphical Software
centre, searches "finbreak", clicks Install, and runs it — with the app
**sandboxed** (no network, no access to the rest of their files) as the finance
security story demands.

This is **complementary** to OBS (FIBR-0155) and the AppImage (FIBR-0054):
AppImage = portable download; OBS = distro-native RPM/deb install; **Flatpak =
sandboxed, cross-distro, in the app store**. A separate build pipeline —
`flatpak-builder` + a manifest, not rpm/deb (§ 3).

## 2. Scope

**In scope**
- A **`flatpak-builder` manifest** `packaging/flatpak/io.github.milnet01.finbreak.yaml`
  on the **freedesktop runtime** (§ 3.1), installing finbreak + its pinned wheel
  closure into `/app` (§ 3.2).
- A **pinned pip-source module** (`packaging/flatpak/python3-deps.yaml`,
  generated by `flatpak-pip-generator`) — every dependency wheel/sdist pinned by
  URL + **sha256**, so the Flathub network-free build resolves offline (§ 3.6).
- A **minimal, network-free, portal-only sandbox** (`finish-args`, § 3.4) — the
  security contract: **no `--share=network`, no `--filesystem=…`**.
- **Reuse of the FIBR-0155 desktop assets** (§ 3.3): the checked-in
  `io.github.milnet01.finbreak.desktop`, `…​.metainfo.xml`, and the hicolor icon
  set are the single source of truth, installed by the Flatpak too — **no new
  desktop/metainfo files** (rule 3).
- A **local build+test recipe** (`packaging/flatpak/flatpak-build.sh`) and a
  **maintainer runbook** (`packaging/flatpak/README.md`) covering the submission
  flow to `github.com/flathub/flathub` (§ 3.8).
- **Tests** by the FIBR-0155 script-scrape pattern (§ 4): assert the manifest /
  sandbox / pin invariants without a real Flathub build, plus the one runtime
  assertion (updater inert under Flatpak).

**Out of scope**
- **The Snap Store** (roadmap follow-up after Flathub — a `snapcraft.yaml`).
- **Getting into the official distro archives** (bundling policy — same rationale
  as FIBR-0155 § 2).
- **Auto-submitting releases** from CI. First cut is a documented manual
  `flatpak-builder` build + a PR; automating the update PR is a follow-up (§ 3.8).
- **Any change to the app's behaviour or source.** Recon (§ 3.5) confirms the
  self-updater is inert under Flatpak, the file-dialog *code* uses the native
  chooser, and data lives in an XDG location. **No `src/` edit is *expected*** (the
  key contrast with FIBR-0155, which needed the one-line Wayland app-id fix —
  already shipped). Two **portal behaviours** are gated on the § 5 smoke test
  (§ 3.5): (a) the native dialog *routing* to the portal with the pip-bundled Qt —
  whose contingency, if it fails, is a **manifest** change (a
  `QT_QPA_PLATFORMTHEME` `--env=` / plugin), still no `src/` edit; and (b) the two
  export services' sibling-temp-then-`os.replace` write to the *user-chosen*
  (portal) path, which the document portal may forbid — **its** contingency is a
  **scoped `src/` change** to `pdf_export.py` + `backup.py`'s write-path (write to
  the portal file directly, or stage app-private and stream-copy), bounded to
  those two functions. Nothing else in `src/` is touched.

## 3. Design

### 3.1 Runtime — freedesktop `25.08`, not KDE / not the PySide BaseApp

**Decision: build on `org.freedesktop.Platform` / `org.freedesktop.Sdk`, branch
`25.08`** (the latest stable freedesktop runtime at spec time, released 2025-09;
the § 5 pre-submit check re-confirms it is current + installable at submit time,
so this number isn't load-bearing). finbreak
**pip-installs its own `PySide6==6.11.1` wheel**, which **bundles its own Qt6** —
so it needs *no Qt from the runtime*, only the base system libraries
(glibc, libGL/libEGL, fontconfig, xkbcommon, dbus) the freedesktop runtime
provides. The manylinux wheels' glibc floor (`manylinux_2_34` = glibc 2.34,
`pyproject.toml:21-26`) is comfortably cleared by the `25.08` runtime.

Why **not** the two Qt-provider alternatives (full rationale § 7):

1. **KDE runtime (`org.kde.Platform`) is rejected.** Its branches track Qt
   versions, and there is **no `6.11` branch** to match `PySide6==6.11.1` (the KDE
   runtime lags upstream Qt point releases — verify current branches at
   `invent.kde.org/packaging/flatpak-kde-runtime` before locking, § 5). Worse,
   layering a pip PySide6 wheel (with its bundled Qt) over the runtime's Qt yields
   **two Qt copies** in one process — a plugin/symbol-clash risk that defeats the
   pinned-closure guarantee.
2. **The PySide BaseApp (`io.qt.PySide.BaseApp`) is rejected.** Its newest branch
   is **6.10** (no 6.11), and it bakes a *fixed* PySide/Qt build tied to the KDE
   runtime — i.e. **not** finbreak's pinned `6.11.1`. Using it would fork the
   correctness-critical stack away from what the gate tests.

The freedesktop-runtime + pip-wheel route is the **only clean path to the exact
pinned PySide6** today, and it mirrors FIBR-0155's decision exactly: run the
**same pinned wheel closure the gate tests**, never a substituted Qt/SQLCipher.

### 3.2 Payload — the pinned wheel closure, pip-installed into `/app`

**Decision: the manifest pip-installs the exact pinned closure from
`pyproject.toml` into `/app`**, then pip-installs the finbreak package itself from
its tagged git source. Two module groups:

1. **`python3-deps`** — the dependency closure (§ 3.6), each source pinned by
   sha256, installed with
   `pip3 install --no-build-isolation --no-index --prefix=${FLATPAK_DEST}`.
   The native, ABI-sensitive pieces (`PySide6==6.11.1`, `sqlcipher3-wheels==0.5.7`,
   `pikepdf==10.9.1`, `cryptography==49.0.0`, plus the transitive natives
   `argon2-cffi-bindings`, `Pillow`, `pypdfium2_raw`) travel as manylinux wheels;
   the pure-Python wrappers `argon2-cffi==25.1.0` and `pdfplumber==0.11.10`
   themselves ship as `py3-none-any` (their native code is in those separate
   bindings — matching § 3.6's attribution); `ofxparse==0.21` (sdist-only, no PyPI
   wheel) builds from its sdist.
2. **`finbreak`** — the app, built from `github.com/milnet01/finbreak` at the
   release tag (commit-pinned, § 3.8), with
   `pip3 install --no-build-isolation --no-index --prefix=${FLATPAK_DEST} .`
   (deps already satisfied in `/app`). This installs the **`/app/bin/finbreak`
   console script** from `pyproject.toml`'s `[project.scripts]`
   (`finbreak = "finbreak.__main__:main"`, `pyproject.toml:74-75`, shipped by
   FIBR-0155), which becomes the Flatpak `command:` (§ 3.3). **The desktop assets
   (§ 3.3) are installed by this same module's `build-commands`** — `install
   -Dm644 packaging/obs/io.github.milnet01.finbreak.{desktop,metainfo.xml} …` and
   the renamed icon loop — operating on **this module's own cloned source tree**
   (`packaging/obs/` is part of the finbreak checkout). This is why the assets
   need no manifest-level `type: file` disk reference: a submitted Flathub manifest
   stands alone at its repo root with no `packaging/obs/` beside it, so a
   `../obs/…` path would build locally yet break on submission — the module
   build-commands reach the assets through the git clone, which survives
   submission (§ 3.3).

**Why pip-install, not the FIBR-0155 PyInstaller freeze.** Flatpak already
provides the isolation + bundled runtime that PyInstaller gives a bare RPM/deb
payload, so freezing is redundant here — a normal `pip install` into `/app` is
the idiomatic Flatpak build and needs no PyInstaller. **FIBR-0155's deliberate
"do NOT `pip install /src`" rule does not apply:** that rule is
*PyInstaller-specific* (it freezes a `--paths`-referenced source tree rather than
installing the package, and the read-only `/src` mount can't take an egg-info —
`_build-smoke-in-container.sh:48-50`); with no freeze step, installing the package
normally (`pip install .`, a bare dot, on the writable cloned tree) is correct. The **payload is the same pinned closure** either way — the
correctness guarantee (§ 3.1) is preserved; only the packaging mechanism differs
(pip-into-`/app` vs freeze-into-`/usr/lib`).

`setuptools>=77` (finbreak's own `[build-system]` requirement, `pyproject.toml:2`)
is needed to build the `finbreak` tree in module 2; it is provided by the
freedesktop Sdk, but is **pinned into `python3-deps` as a belt-and-suspenders
source** if the Sdk's version is below the floor (a § 5 check). This is a
**distinct** concern from FIBR-0155's setuptools note: FIBR-0155 never installs
the finbreak package (it freezes `__main__.py`), so it needed setuptools only as
*PyInstaller's* own install-time dependency (floor ≥ 42) and explicitly disclaimed
finbreak's `[build-system]` floor — here the `pip install .` in module 2 makes
finbreak's own `>=77` floor the live constraint.

### 3.3 Desktop integration — reuse the FIBR-0155 assets verbatim

The three XDG assets FIBR-0155 checked in are **already Flathub-shaped** and are
reused as the single source of truth (rule 3 — no new files):

- **`packaging/obs/io.github.milnet01.finbreak.desktop`** → installed to
  `${FLATPAK_DEST}/share/applications/`. `Icon=io.github.milnet01.finbreak`
  already matches the app-ID (Flathub requires this). `Exec=finbreak` is the
  console script (§ 3.2); Flatpak rewrites the host-facing `Exec` to
  `flatpak run io.github.milnet01.finbreak` automatically.
- **The hicolor icon set** `assets/icon/finbreak-{16,24,32,48,64,128,256,512}.png` → installed
  renamed to `io.github.milnet01.finbreak.png` under
  `${FLATPAK_DEST}/share/icons/hicolor/<size>x<size>/apps/` (the same rename the
  RPM/deb does, FIBR-0155 § 3.3).
- **`packaging/obs/io.github.milnet01.finbreak.metainfo.xml`** → installed to
  `${FLATPAK_DEST}/share/metainfo/`. It already carries every field Flathub
  requires — `<developer>`, `<content_rating type="oars-1.1"/>`,
  `<project_license>`/`<metadata_license>`, `<screenshots>`, and a `<releases>`
  block whose newest `<release>` mirrors CHANGELOG (kept in lockstep by `/bump`,
  FIBR-0155 § 3.7). It **must pass `appstreamcli validate`** (INV-5; a Flathub
  gate).
- The Flatpak **`command:`** is `finbreak`, and the manifest **app-ID** (manifest
  filename + `id`) is `io.github.milnet01.finbreak` — matching the runtime Wayland
  `app_id` (`QGuiApplication.setDesktopFileName("io.github.milnet01.finbreak")`,
  `src/finbreak/app.py:48`, shipped by FIBR-0155) so the window associates with
  the launcher.

**Asset ownership note:** these three assets now serve *both* OBS and Flatpak.
They stay in `packaging/obs/` as the single source; the **`finbreak` module's
`build-commands` install them from its own git-cloned source tree** (§ 3.2 item 2)
— **not** a manifest-level `type: file` disk path (a `../obs/…` reference builds
locally but breaks once the manifest is submitted standalone to Flathub, § 3.2).
Hoisting them to a shared `packaging/` location is a deliberate **non-goal** here
(it would churn the just-shipped FIBR-0155 recipes for no functional gain) — noted
as a future refactor (§ 7).

### 3.4 Sandbox — the minimal, network-free, portal-only contract

This is the security core. finbreak is local-only; the sandbox is the **minimal
set** that lets a Qt app draw a window and open/save files **through the portal**,
and nothing more:

```
finish-args:
  - --socket=wayland          # Wayland display
  - --socket=fallback-x11     # X11 only when Wayland is absent
  - --share=ipc               # X11 shared-memory (XShm); harmless, standard
  - --device=dri              # GPU accel for Qt; falls back to software render
```

`--device=dri` is the one entry not strictly required: finbreak is a plain
`QWidgets` app (no OpenGL/QML in `src/`), so Qt renders fine on software
(llvmpipe) without it — it is included as conventional Qt-Flatpak boilerplate for
smoother compositing, and the § 5 build+launch smoke test confirms the app starts
with it (and, as a minimality check, without it). Everything else is load-bearing.

**Deliberately absent — the security story:**
- **NO `--share=network`.** Network is off by default in a Flatpak sandbox;
  omitting the flag *enforces* no-network. finbreak makes zero **app-initiated**
  network requests in this build (§ 3.5) — the sandbox makes that a hard
  guarantee, not a policy. ("App-initiated" is the same scope as
  `security-model.md` INV-8: it excludes the donate/report-issue links, which
  hand a URL to the *host* browser via `QDesktopServices.openUrl` /the OpenURI
  portal — `main_window.py:1346`, a user-initiated egress from the browser, not
  the sandboxed app process, and needing no `--share=network`.)
- **NO `--filesystem=…` of any kind** (no `home`, no `host`, no `xdg-*`). File
  import (CSV/OFX/PDF) and PDF/backup export are designed to go through the
  **xdg-desktop-portal file chooser**, which grants access to *only the specific
  file the user picks*, via the document portal. finbreak's code never forces the
  non-native dialog (§ 3.5, source-confirmed), but that the native chooser
  actually **routes to the portal with the pip-bundled Qt** (§ 3.1's own Qt, not
  the runtime's) is **gated on the § 5 portal smoke test** — see § 3.5 for the
  contingency (a `QT_QPA_PLATFORMTHEME=xdgdesktopportal` `--env=` and/or bundling
  the portal platform-theme plugin) if it doesn't fire automatically. No
  filesystem permission is needed or wanted.
- The app's own data (the encrypted vault + KDF sidecar, plus `window.ini` and the
  update-check `QSettings` prefs) lives in
  `QStandardPaths.AppDataLocation` (`src/finbreak/paths.py:3,25-26`), which Flatpak
  transparently redirects into the app's private
  `~/.var/app/io.github.milnet01.finbreak/` — **no permission needed**.

The result: a finance app that **cannot reach the network and cannot read any
file the user did not explicitly hand it** — a materially stronger posture than
the AppImage or the RPM/deb, and exactly the security story the user asked for.

### 3.5 App-source change — source-confirmed vs portal-gated

Four paths a sandbox could affect. The **updater** and the **XDG data path** are
**fully confirmed in current source** (rule 13 — grep/Read-verified). The
**file-dialog** and **export** paths are confirmed *at the code level* (the app
uses the native chooser and an atomic write), but each carries a **portal
behaviour that cannot be verified from the repo** — it depends on the pip-bundled
Qt (§ 3.1) and the live portal, not on `src/` — so both are **gated on the § 5
portal smoke test**: (a) the native dialog actually routing to the portal (file
*open*), and (b) the atomic write-then-rename surviving a portal `SaveFile` path
(file *save*).

- **Self-updater is inert under Flatpak.** `detect_installer()`
  (`src/finbreak/services/update_installer.py:386-401`) returns an installer
  **only** for a frozen Windows `.exe` or a real AppImage (`$APPIMAGE` set + file
  exists); its docstring (`:392`) explicitly names **"a Flatpak"** as a `None`
  case. So under Flatpak `is_update_supported()` is `False` (`:404-407`) and the
  Settings "check for updates" checkbox is disabled (the `self._installer is not
  None` gate, `main_window.py:787`, FIBR-0155 § 3.1). Outbound is zero
  **unconditionally** under Flatpak — both the silent startup check and a manual
  "Check for updates" short-circuit on `self._installer is None`
  (`update.py:193`, `main_window.py:1208`), not merely "off by default". **No
  build-time gating is needed** — the roadmap's "gate the update check off in the
  Flatpak build" is already true by construction, so this spec adds no such step.
- **File dialogs — code confirmed, portal-routing gated.** Every dialog is a
  static `QFileDialog.getOpenFileName` / `getSaveFileName`
  (`main_window.py:829,882`, `backup_restore.py:108`, `import_wizard.py:328`,
  `backup_verify.py:160`) with **no `DontUseNativeDialog` anywhere** in `src/`
  (a bypass would break the no-`--filesystem` design) — that much is
  source-confirmed. **Gated:** whether that native chooser routes through the
  xdg-desktop-portal depends on the **pip-bundled Qt** carrying + activating the
  portal integration (the runtime-Qt "it just works" folklore, § 3.1, does *not*
  transfer to a bundled Qt). The § 5 file-open smoke test confirms it; if it
  doesn't fire, the contingency is a `QT_QPA_PLATFORMTHEME=xdgdesktopportal`
  `--env=` in `finish-args` and/or ensuring the portal platform-theme plugin
  travels in the bundled Qt — a bounded manifest change, **not** a broadened
  filesystem permission.
- **Data path is XDG.** `QStandardPaths.AppDataLocation` (§ 3.4) — auto-redirected
  into the sandbox; no hard-coded `$HOME`/`~/.local/share` path exists in `src/`.
  (The **vault** write lands here — an app-private, writable location — so it is
  unaffected by the export risk below, which concerns only *user-chosen* paths.)
- **⚠ Export write-path — the second gated portal risk (not yet confirmed).**
  Both export services write the final file to the **user-chosen** save location
  using an
  atomic *sibling-temp-then-`os.replace`* pattern:
  `pdf_export.py:161` (`out_path.with_name(out_path.name + ".part")` →
  `os.replace`) and `backup.py:113` (`dest.with_name(dest.name + ".tmp")` →
  `os.replace`). Under a **no-`--filesystem`** sandbox the portal `SaveFile`
  result is a document-portal FUSE path (`/run/user/<uid>/doc/<id>/<name>`) whose
  directory exposes **only the one granted file** — so creating a *sibling*
  `.part`/`.tmp` there and renaming over the target **may fail**, breaking PDF
  export and encrypted-backup export. This cannot be confirmed without a live
  Flatpak build, so it is **NOT asserted as recon-verified** (rule 13): it is a
  **§ 5 pre-submit smoke test** (do a real export through the portal, not just
  "a dialog opens"). **Contingency:** if the test shows breakage, a **scoped
  source change to those two write-paths** comes into scope (§ 2) — write directly
  to the portal file, or stage in an app-private dir and stream-copy — preserving
  the INV-2/INV-12 "no partial/unencrypted file" guarantee as the implementation
  allows. Nothing else in `src/` is affected. (The security-model.md INV-8 note
  for this build's empty outbound surface is § 3.7's action, not repeated here.)

### 3.6 Offline, reproducible sources — sha256-pinned wheels

Flathub builds run **with no network** and require **every source pinned by a
publicly-resolvable URL + sha256** (`--share=network` in `build-args` is
non-functional on Flathub's builders). The pinned closure is therefore expressed
as pinned pip sources:

- **`packaging/flatpak/python3-deps.yaml` is generated by `flatpak-pip-generator`**
  (from `github.com/flatpak/flatpak-builder-tools`), driven by finbreak's exact
  pins. It emits a top-level module for the closure — for a multi-package closure
  (finbreak's is ~10+) the generator nests one sub-module per package under a
  `modules:` list rather than a single flat `sources:` array — where **every**
  `sources[]` entry is a wheel/sdist file with its `url` + `sha256` (fetched into
  the build dir), and each build-command runs `pip3 install --no-index` against
  those **local files by path** — so no package index is ever consulted (the
  sha256-pinned local files are the only inputs). *(INV-3's scraper therefore
  walks the nested `modules[].sources[]`, not just a top-level `sources:` — § 4.)*
- **Manylinux binary wheels are opt-in in the generator:** the correctness-critical
  stack (PySide6, cryptography, sqlcipher3-wheels, pikepdf, pdfplumber's Pillow)
  ships **only** as manylinux wheels, so generation passes `--prefer-wheels` (which
  requires `--runtime=org.freedesktop.Sdk//25.08` so the generator resolves against
  the right ABI). `ofxparse==0.21` needs no flag — it is sdist-only and the
  generator prefers its sdist by default, building it from source.
- A helper **`packaging/flatpak/generate-pip-sources.sh`** records the exact
  generator invocation (deps read from `pyproject.toml`, so a bump regenerates the
  pins from one source of truth) — the Flatpak analogue of FIBR-0155's
  `vendor-wheels.sh`, and re-run on the same trigger (dependency-closure change).
- **CPython ABI:** unlike OBS (which spans distro pythons cp312/cp313/cp314 —
  see `packaging/obs/vendor-wheels.sh`, whose `PY` list vendors all three; note
  FIBR-0155 § 3.6's prose predates the cp314 addition and reads dual-ABI), the
  Flatpak targets **one** interpreter — the freedesktop Sdk's own `python3` — so
  **exactly one ABI's wheel set** is pinned (the Sdk's python minor, confirmed at
  § 5). This is simpler than the OBS multi-ABI vendoring.
- **Failure mode — no wheel for the Sdk's python minor.** The correctness-critical
  per-minor-ABI deps (`pikepdf`, `sqlcipher3-wheels`, `pdfplumber`'s `Pillow`)
  publish a wheel per CPython minor; if the Sdk's `python3` minor (or a mid-branch
  bump within the runtime's 2-year life) has **no** manylinux wheel for a pinned
  exact version, `flatpak-pip-generator --prefer-wheels` has nothing to pin for
  that dep and generation fails. **Fallback, in order:** (a) pin the manifest to
  an Sdk sub-version whose python minor is covered; (b) bump the affected dep pin
  to a release that ships that minor's wheel (a coordinated `pyproject.toml`
  change, re-gated by the full test suite — never a silent Flatpak-only drift);
  (c) add a from-source build module for that one dep. This is a § 5 pre-submit
  check ("the build IS the proof"), with the fallback recorded so a failed
  generation isn't a dead end.

**Reproducibility caveat (inherited from FIBR-0155 § 3.6):** two deps are
range-pinned, not `==` — `beautifulsoup4>=4.9,<5` and `certifi>=2024.2.2`
(`pyproject.toml:44,64`). The generator resolves them to whatever is current at
generation time; neither is in the correctness-critical stack. The **generated
sha256 pins freeze them** for any given manifest, so a built manifest is
reproducible; only a *regeneration* months apart could bump a bs4/certifi patch.

### 3.7 Flathub-build outbound surface — a note, not a change

The updater being inert under Flatpak (the mechanism is § 3.5, not restated here)
means **zero app-initiated outbound requests** — a strict subset of
`security-model.md`'s INV-8 ("exactly one opt-in flow"), and the sandbox (§ 3.4)
additionally makes app networking *unreachable* at the OS level. **This section's
action** (mirroring FIBR-0155 § 3.8's OBS note): add a note under
`security-model.md` INV-8 recording that the Flatpak build's app-initiated

### 3.8 Submission + release tie-in

- **First submission = a PR to `github.com/flathub/flathub`.** New apps go on the
  **`new-pr` base branch** (not `master`): fork, branch from `new-pr`, add the
  manifest (+ `python3-deps.yaml`) at the repo root, open a PR titled
  `Add io.github.milnet01.finbreak`. The reviewer checks it builds entirely from
  pinned source, passes the Flathub linter, and has valid metainfo. On merge,
  Flathub creates the app's own repo and builds/hosts it. (Runbook: § 6 README.)
- **App-ID `io.github.milnet01.finbreak` is Flathub-valid:** reverse-DNS, ≥4
  components for a GitHub-hosted app (`io.github.<user>.<app>`), mapping to the
  real `github.com/milnet01/finbreak` repo.
- **Git source is commit-pinned.** The `finbreak` module's `git` source pins the
  release tag **and its commit sha** (Flathub requires an immutable commit, not a
  floating tag) — the version still flows one way from the git tag `v{VERSION}`
  (INV-4; no hand-typed version in the manifest).
- **Ongoing releases:** each new finbreak version is a new PR to the app's Flathub
  repo bumping the `finbreak` module's tag/commit and regenerating
  `python3-deps.yaml` only if the closure changed. The `/bump` recipe already
  keeps the metainfo `<release>` current (FIBR-0155 § 3.7); this adds a runbook
  step, not new automation (auto-PR is out of scope, § 2).

## 4. Invariants

Most are **script-scrape** over the manifest / deps file (the FIBR-0155 pattern);
**INV-6 is a runtime assertion**. None needs a real Flathub build. Coverage
limits are stated per-invariant.

- **INV-1 — freedesktop runtime, pinned branch, no distro/KDE Qt.** The manifest's
  `runtime` is `org.freedesktop.Platform` and `sdk` is `org.freedesktop.Sdk`, both
  on a **pinned** `runtime-version` (not floating), and the manifest names neither
  `org.kde.Platform` nor a PySide/Qt BaseApp. *Test:* parse the manifest YAML;
  assert `runtime`/`sdk`/`runtime-version`; assert the KDE/BaseApp strings are
  absent.
- **INV-2 — Sandbox is network-free + filesystem-free (the security contract).**
  `finish-args` contains **no** `--share=network` and **no** `--filesystem=` entry
  of any form. *Test:* parse `finish-args`; assert every entry is in an
  **enumerated allowlist committed in the test spec** (`--socket=wayland`,
  `--socket=fallback-x11`, `--share=ipc`, `--device=dri`), so *adding* any network
  or filesystem permission fails the test (the coverage boundary is explicit, not
  a substring blocklist).
- **INV-3 — Every pip source is sha256-pinned (offline/reproducible).** In
  `python3-deps.yaml`, every `sources[]` entry of `type: file`/`archive` carries a
  `sha256`, and no build-command adds `--share=network` or omits `--no-index`.
  *Test:* parse the deps file (**recursing into nested `modules[].sources[]`**,
  § 3.6); assert each source has a 64-hex `sha256`; scrape the build-commands for
  `--no-index`. *Coverage limit:* that the pinned hashes *match the current pins*
  needs a live regeneration + diff — a **manual pre-submit checklist item** (§ 5),
  not CI-gated.
- **INV-4 — Identity + version single-source.** The manifest `id`, its filename,
  the `command`, the `.desktop` basename, the metainfo `<id>`, and
  `app.py:48`'s `setDesktopFileName` argument are all
  `io.github.milnet01.finbreak`; `command: finbreak` resolves to the
  `[project.scripts]` console entry; and the `finbreak` module's git source pins an
  **immutable commit**, not a hand-typed version literal. *Test:* the
  **Flatpak-specific legs** — parse the manifest, assert `id`/`command`; **assert
  the `finbreak` module's git `sources[]` carries a non-empty 40-hex `commit`**
  (the § 3.8 immutable-commit pin, so a `tag:`-only source fails); read `app.py` +
  `pyproject.toml [project.scripts]` and assert the matches; assert **no literal
  semver string is hand-typed** anywhere in the manifest (it has no RPM-style
  `Version:` field — version rides the git `tag`/`commit` only). The
  **`.desktop`-basename, metainfo-`<id>`, and `Icon=` legs are already covered** by
  FIBR-0155's shipped `obs_packaging` INV-3 (which asserts all three equal the
  app-ID) — reused, not duplicated (rule 3); this invariant assumes those assets
  are the same reused files (§ 3.3).
- **INV-5 — Metainfo validates for Flathub.** The reused metainfo passes
  `appstreamcli validate`. *Test + coverage limit:* `appstreamcli validate` is
  **not** in the CI image (as in FIBR-0155 INV-4), so this is **skip-if-absent /
  manual pre-submit** (§ 5). The `.desktop` `Icon=`/app-ID match is **not**
  re-asserted here — it is INV-4's reused `obs_packaging` leg. *(This reuses the
  FIBR-0155 metainfo, already validated there — this invariant guards against a
  Flatpak-side metainfo regression, e.g. a malformed `<release>` from a bad
  `/bump` sync.)*
- **INV-6 — Self-updater inert under a Flatpak launch.** With `$APPIMAGE` unset
  and `sys.frozen` falsy on Linux, `detect_installer()` returns `None` and
  `is_update_supported()` is `False`. *Test:* monkeypatch a Linux `sys.platform`
  with `$APPIMAGE` unset; assert both. *(This is the **same** predicate FIBR-0155's
  obs_packaging INV-5 asserts — rule 3: the Flatpak feature test **reuses** that
  assertion rather than duplicating the logic; it is listed here because it is
  load-bearing for § 3.4/3.7's security claim.)*
- **INV-7 — PySide6 pin is the single Qt source.** The manylinux `PySide6` wheel
  pinned in `python3-deps.yaml` is version **`6.11.1`**, equal to
  `pyproject.toml`'s `PySide6==6.11.1` (`:28`) — i.e. the runtime is *not* the Qt
  provider and the wheel pin is not drifted. *Test:* grep the deps file for the
  PySide6 source filename/version; assert it equals the pyproject pin. *Coverage
  limit:* this scrapes the **committed** `python3-deps.yaml` (§ 6 commits it);
  regeneration drift is caught by INV-3's manual pin-diff (§ 5).

## 7. Alternatives considered

- **KDE runtime (`org.kde.Platform`) providing Qt.** *Rejected:* no `6.11` branch
  to match `PySide6==6.11.1`; and a pip PySide6 wheel over the runtime's Qt puts
  **two Qt copies** in-process (plugin/symbol clash), defeating the pinned-closure
  correctness guarantee (§ 3.1). finbreak's whole architecture is "run the exact
  pinned stack, not a substituted Qt" — the KDE runtime is the opposite model.
- **The PySide BaseApp (`io.qt.PySide.BaseApp`).** *Rejected:* its newest branch is
  6.10 (no 6.11), and it ships a *fixed* PySide/Qt build — forking the
  correctness-critical stack off the gate-tested `6.11.1` pin. (The FIBR-0159
  ROADMAP bullet floated "a recent freedesktop runtime + the PySide6 BaseApp" as a
  candidate; the research finding above superseded it in favour of the freedesktop
  + pip-wheel path, which the same bullet offered as the alternative.)
- **Reuse the FIBR-0155 PyInstaller freeze inside the Flatpak.** *Rejected as
  redundant:* Flatpak already supplies the isolation + bundled runtime PyInstaller
  gives a bare RPM/deb, so freezing adds a build step and a second bundling
  mechanism for no gain. A plain `pip install` into `/app` is the Flatpak idiom and
  yields the **same pinned closure** (§ 3.2).
- **Grant `--filesystem=home` (or `xdg-download`) instead of relying on the
  portal.** *Rejected on security grounds:* it would let a finance app read the
  user's whole home (or Downloads) — exactly the exposure the portal exists to
  avoid. The portal grants access to *only the picked file*, and Qt uses it
  automatically (§ 3.4, § 3.5). Broad filesystem access is the single worst
  default for this app class.
- **Hoist the shared desktop/metainfo/icon assets to a common `packaging/`
  location.** *Deferred (not rejected):* the right long-term shape once two
  packaging backends share them, but it would churn the just-shipped FIBR-0155
  recipes for zero functional gain now. This spec references them in place
  (§ 3.3); a later refactor can hoist them.
- **Snap Store instead of / before Flathub.** *Deferred, complementary* (roadmap
  follow-up) — a different store; Flathub has the widest cross-distro reach, so it
  goes first.
