# FIBR-0060 — Window size + Center window on Wayland — closed 2026-07-09

**Kind:** fix (my own code — FIBR-0052 assumed X11; the sole-author rule owns
it). User-reported on KDE Wayland: the window size/position weren't remembered
and Window → Center window did nothing. **The user pointed me at their
SystemManager project, which had already solved Center-on-Wayland** — I reviewed
its technique and adopted it.

## Root cause

FIBR-0052 (INV-5 geometry persistence, INV-6 Center window) used X11 semantics:
`restoreGeometry` / `move()` set the window **position**, which a Wayland
compositor ignores (it owns placement), and `restoreGeometry`'s **size** is
unreliable before the first map. So on Wayland size wasn't restored and Center
was a silent no-op. The FIBR-0052 tests only asserted the QSettings round-trip
under the offscreen QPA, never real WM behaviour, so they passed while the
feature was broken for the user.

## What changed (`ui/main_window.py`)

- **Platform detection.** `_is_wayland()` (`platformName()` startswith
  `wayland`), `_kde_wayland()` (Wayland AND `XDG_CURRENT_DESKTOP` contains
  `KDE`), `_center_supported()` (not Wayland, or KDE Wayland) — module functions
  so tests can monkeypatch them.
- **Remember the size (both platforms).** `_save_geometry` now also writes a
  bare `window_size` key beside the opaque `saveGeometry` blob. On Wayland
  `_restore_geometry` restores **only the size** via `resize()` (a size request
  the compositor honours) — the Qt equivalent of SystemManager's GTK
  `set_default_size`. Off Wayland it keeps `restoreGeometry` (size + position).
  `_reset_layout` clears `window_size` too.
- **Center window on KDE Wayland (the SystemManager technique).** `_center_kwin`
  drives **KWin's scripting D-Bus API** — `loadScript` / `start` / `unloadScript`
  over **QtDBus** — of a tiny script that centres our PID-matched window in its
  work area. Ported to QtDBus (rather than SystemManager's `dbus-send`
  subprocess) so there is no external-binary dependency — cleaner for a bundled
  Qt app, and no bandit subprocess finding. Best-effort: any failure (no D-Bus /
  KWin, sandboxed, ImportError) leaves the window put. Off Wayland → plain
  `move()`. Non-KDE Wayland → the action is **disabled with an explaining
  tooltip** (no app-usable placement API), and `_center_window` is a safe no-op
  (`_reset_layout` still calls it).

## What was learned

- **Center window is NOT impossible on Wayland** — the roadmap bullet assumed it
  was. KWin (KDE) exposes a scripting D-Bus API that *can* place a window; the
  user's SystemManager project proved it on this exact machine. Verifying the
  claim (the interface is reachable; QtDBus drives it) beat accepting the
  can't-be-done framing. Sway (`swaymsg`) and Hyprland (`hyprctl`) have similar
  IPC — a future enhancement; KDE is the user's platform and the one shipped.
- **Live verification mattered.** Beyond the monkeypatched unit tests, I drove a
  real Qt window on the user's KDE Wayland session through `_center_kwin` and
  read the geometry back from KWin: it centred **exactly** (work-area centre
  offset `dcx=0 dcy=0`). The offscreen QPA can't exercise the real branch, so
  the end-to-end check is what actually proves the fix.

## What was deferred

- **Position-restore on launch on Wayland** — left to the compositor (System
  Manager also accepts this; its `move()` restore is an X11-only no-op on
  Wayland). Size is remembered; position is compositor-placed.
- **Sway / Hyprland Center backends** — no user on them; the action degrades to
  disabled-with-tooltip. Noted for a future enhancement.
- **Flatpak D-Bus permission** — a KWin-scripting Flatpak build will need
  `--talk-name=org.kde.KWin`; a FIBR-0015 packaging concern, not needed for the
  source / AppImage run.
