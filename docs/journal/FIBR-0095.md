# FIBR-0095 — Failed-unlock throttling (exponential backoff)

**Shipped 2026-07-18** (self-directed, autonomous "build it all" run).
Retires the duplicate **FIBR-0031**. Tag `FIBR-0095-complete`.

## What shipped

After a wrong master password in the interactive unlock dialog, the next
attempt is delayed on a **capped exponential schedule** — 1s, 2s, 4s, 8s,
16s, 30s (cap) — before it is accepted. The attempt count and last-failure
time persist in the plaintext `window.ini`, so closing and reopening the app
does **not** reset the delay (the trivial bypass). A correct password always
clears the counter, and the delay is capped, so the legitimate owner is never
locked out.

**Scope (honest):** soft defence-in-depth against *interactive* guessing
through the app UI only. It is **not** a security boundary — the count lives
in an attacker-writable plaintext file, and the offline-cracking path on a
copied vault is defended solely by Argon2id (security-model INV-2). Added as
**security-model INV-10**.

## Design (three layers)

- `services/unlock_throttle.py` — Qt-free pure core: `backoff_delay_seconds`
  (exponent clamped at `CAP_N=6` **before** the power, so a tampered huge
  `fail_count` can't build a giant integer and hang the app), and
  `remaining_lockout_seconds` (fail-safe on missing/malformed/naive/future
  `last_fail`), plus a frozen `ThrottleState`.
- `ui/_unlock_throttle.py` — `UnlockThrottle` QSettings adapter over
  `window.ini`; defensive `int()`/`fromisoformat()` coercion; `.sync()` after
  every write; reset on success only.
- `ui/unlock.py` — authoritative entry gate in `_on_unlock` (re-reads the
  persisted state on **every** submit; the 1-Hz countdown `QTimer` is purely
  cosmetic), `record_failure` + countdown in `_show_failure`, `reset()` on the
  success path. The countdown disables **submit-only** via `_set_submit_enabled`
  so Cancel/Restore stay usable — `_set_busy` (which disables them during
  derivation) is untouched, preserving FIBR-0004 INV-2f.
