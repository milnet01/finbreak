# FIBR-0010 — P08 rules engine + manual override + learning

Closed 2026-07-10. Spec: [`docs/specs/FIBR-0010.md`](../specs/FIBR-0010.md)
(`/cold-eyes`-converged at loop 6). Tag `FIBR-0010-complete`.

## What changed

The transaction → category link deferred from P04, plus the engine that fills it.

- **Schema v6 → v7** (`_migrate_to_v7`, one atomic step): two nullable
  `transactions` columns — `category_id` (plain `REFERENCES categories(id)`, no
  `ON DELETE`) and `category_source` (`'rule'` | `'manual'` | `NULL`) — and a new
  `categorization_rules` table.
- **The engine.** `finbreak/text.normalise_text` (the shared
  collapse-whitespace-then-casefold primitive, now reused by import dedup *and* the
  matcher). `services/categorization.categorize` (pure first-match by priority) and
  `recategorize_auto_rows(conn)` (commit-free; recompute every **auto** row, write
  only rows that change, return the changed count). The **golden rule**: a row is
  **manual** (frozen) or **auto** (rule-set or none); the engine recomputes autos
  and never touches a manual row — enforced by the NULL-safe read predicate
  `category_source IS NULL OR category_source <> 'manual'`.
- **`CategorizationService`** — rules CRUD, top-priority insert (new rules win),
  adjacent-swap reorder, `set_manual_category`, `would_categorize`, `apply_rules`.
- **Import runs the rules** inside its own transaction (`commit_import` calls
  `recategorize_auto_rows` before commit), so a fresh import arrives categorised.
- **Delete-category cascade** (`CategoryService.delete_category`), one owned
  transaction: clear + reset the category's transactions to auto, delete its rules,
  delete the category, re-apply — with `delete_blast_radius(id)` feeding a
  confirmation that names how many transactions and rules are affected.
- **UI:** a Home **Category** column + right-click *Set category…* (sets a row
  manual) with the **learning** offer (correct an auto row → offered a rule); a
  **Rules** tab (add / edit / delete / move / Apply); a leaf `CategoryPickerDialog`;
  the net-new category-delete blast-radius confirm.

TDD: `tests/features/categorisation/` (45 tests, INV-1..16 + edges) red → the 14
deliverables to green.

## What was learned

- **The spec's literal `repo.delete(id)` in the cascade would have broken INV-7.**
  `CategoryRepository.delete` commits per-write, which would end the service-owned
  transaction early — so the category delete is issued **in-line** commit-free.
  Surfaced during implementation (never silently drifted); the spec D8 was
  annotated to match. The INV-7 rollback test (monkeypatch `recategorize_auto_rows`
  to raise) is what pins this — it asserts *nothing* changed, which only holds if
  steps 1–3 hadn't already committed.
- **The auto-lock/event-loop interaction was subtle.** The learning offer's dialog
  pumps the event loop, so an idle auto-lock can fire mid-flow and the trailing
  `refresh()` then reads a locked vault. The first-cut INV-14 test missed it (it
  patched a *bound service method* to raise, leaving the real vault unlocked so
  `refresh()` still worked). The cold review caught it; the fix wraps set + learn +
  refresh in one guard, and a regression test now patches `list_transactions`
  (what `refresh()` calls) to raise so the guarded path is genuinely exercised.
- **`set_category`'s conditional `UPDATE … WHERE … (col IS NOT ? OR …)`** with a
  summed rowcount is what makes idempotent-apply (INV-13) honest *and* lets
  `auto_rows()` stay a lean `(id, description)` shape — the change-detection lives
  in the write, not a wider read.

## What was deferred

- **Grouping the leaf pickers by root** — the category selectors are flat combos;
  visual grouping under the Type root is UX polish (annotated in spec D13).
- **Per-account rules, regex, amount/direction matching** — out of scope
  (text-substring only, user decision 2026-07-09); a clean future seam.
- **Live cross-tab refresh** — Home refreshes on navigation.
