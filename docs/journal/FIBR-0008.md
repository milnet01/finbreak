## What changed

- **The second importer.** A pure `OfxImporter` (`importers/ofx_importer.py`)
  turns OFX **bytes** + the currency exponent into one `ParseResult` per
  statement (`ofx.accounts`), reusing `parse_transaction` per transaction so an
  OFX amount obeys the identical money contract as CSV/manual. OFX is
  self-describing, so — unlike CSV — there is **no `ColumnMapping`**: the signed
  `<TRNAMT>` passes straight through (D3, no `<TRNTYPE>` inversion), `<DTPOSTED>`
  → ISO date, description = payee-else-memo (D5), and the coverage period is the
  embedded `<DTSTART>`/`<DTEND>` with the `''`-missing-span falsiness guard (D4).
- **The D15 error model** (the phase's subtlest call): ofxparse parses
  all-or-nothing per statement, so a **structural** malformation (bad
  `<DTPOSTED>`, empty `<TRNAMT>`) aborts the whole statement → the single D7
  boundary catch surfaces a friendly `ValueError`; a **post-parse**
  `parse_transaction` rejection (zero / over-precise amount, blank description)
  becomes a collected `RowError` while sibling rows still import.
- **`importers/base.py`** — `ParseResult` + `RowError` moved here (D12),
  re-exported from `csv_importer` so its public surface is unchanged (INV-9).
- **`ImportService` reuse seam (D2)** — extracted `_preview_from_result`
  (dedup-analysis + period-carry), added `preview_ofx` (just that tail) and the
  size-capped `read_file_bytes` (`_MAX_OFX_BYTES` 16 MiB, D13). `commit_import`,
  `_dedup`, `signature_for`, `save_profile`, `match_profile` **byte-for-byte
  unchanged** — verified by the git diff and the untouched CSV suite.
- **`models.py`** — `OfxAccountInfo(account_id, account_type)` value object (D8).
- **The wizard OFX branch** (`ui/import_wizard.py`, D10) — extension +
  bounded-512-byte content-sniff detection; the `_show_preview` reuse seam
  (owns the load-bearing `self._preview` stash); a multi-account statement
  chooser (`_ofx_statement_combo`, shown only for > 1 account, D8); the D14
  quiet-month predicate relaxation (`drafts > 0 or period_start is not None`,
  proven CSV-equivalent); the file-dialog filter widened to `*.ofx`/`*.qfx`.
  The CSV path is behaviourally unchanged.
- **Resource bounds (D13/INV-10)** — a file over `_MAX_OFX_BYTES` is refused
  before its bytes are read; a file whose total transaction count exceeds
  `_MAX_OFX_TRANSACTIONS` (100 000) is refused whole-file.
- **`pyproject.toml`** — `ofxparse==0.21` (pulls `beautifulsoup4` + native
  `lxml` + `six`); ofxparse parses with the stdlib `html.parser` (lxml is loaded
  via bs4 but not used as the parser). mypy per-module ignore for stub-less
  ofxparse. **No schema change, no migration** (D9) — reuses the v4 tables.
- **A fifth self-test leg** (`_selftest.py` `_check_ofxparse`) — parses a tiny
  OFX, proving ofxparse + its native tree travel and *function* in the bundle.
  Needed because `--self-test` never imports the app, so the existing smoke
  would freeze ofxparse in but never exercise it (mirrors FIBR-0004's argon2
  leg). A matching `test_INV1_selftest_fail_names_ofxparse` bundling unit test.
- **Tests:** `tests/features/ofx_import/` (33 tests, INV-1..10 incl. INV-1a–d /
  INV-7a–f). No conftest lift, **no schema-version ripple** (D9) — the
  distinguishing simplification from FIBR-0007.

## What was learned

- **`--self-test` doesn't import the app, so the bundle smoke wouldn't have
  exercised ofxparse at all** — the existing smoke only ran Qt/SQLCipher/qpdf/
  argon2. A new native dep needs its own self-test leg to be meaningfully
  proven in the Python-free bundle (the FIBR-0004 argon2 precedent, now codified
  for ofxparse).
- **The build-smoke dependency list had silently drifted.**
  `_build-smoke-in-container.sh` installed a **hand-maintained** dep list that
  never got `argon2-cffi` when FIBR-0004 added it — so the frozen bundle had
  been missing argon2 since then, undetected because FIBR-0005/0006/0007 were
  pure-Python phases that didn't re-run the slow opt-in smoke. Re-running it for
  FIBR-0008's native lxml surfaced the latent `ModuleNotFoundError: argon2`.
  Root-cause fix (no workaround): install the runtime deps read straight from
  `pyproject [project].dependencies` (single source of truth, can't drift) and
  `--collect-all` the native-bearing packages. Lesson: a build-input list that
  duplicates pyproject is a drift trap — derive it, don't restate it.
- **A pre-existing FIBR-0007 mypy error surfaced** by the full-tree run
  (`_do_import` passed `str | None` where `commit_import` wants `str`) — fixed
  with a precondition `assert` (the helper's call-sites always import a row, so
  the period is present). FIBR-0007's "mypy 0" close must have run a narrower
  scope; the whole tree is clean now.
- **Exact-format reuse removed a whole error class again** — OFX needs no schema
  change, no migration, and no mapping profile, so this phase is the smallest
  since the security spine: a new format really was "add one importer, nothing
  else changes" (design.md), exactly as FIBR-0007's design-ahead intended.

## What was deferred

- **FIBR-0042 (fix) — timezone-bearing `<DTPOSTED>` day-shift.** A `<DTPOSTED>`
  carrying a time + timezone is normalised to UTC by ofxparse, so an
  evening-in-a-negative-offset transaction can roll to the next calendar day
  (mis-assigning its day/period and potentially defeating cross-source dedup).
  Out of the FIBR-0008 contract (D4/INV-1b specify date-only, and every fixture
  is date-only, so nothing shipped is wrong) and blocked by ofxparse discarding
  the original offset — it needs a product decision (local vs UTC authoritative)
  + raw-DTPOSTED handling. Filed by the `/indie-review` lane-1 finding.
