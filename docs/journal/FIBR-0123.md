# FIBR-0123 — Group category pickers by Income/Expenditure type

**Closed 2026-07-13 by `/close-phase`.** Kind: `ux`. Tag `FIBR-0123-complete`.

## What changed

Shipped by TDD in 6 red→green slices, plus one indie-review LOW fixed inline:

- **`services/categorization.py`** — new `leaf_categories_grouped()`: groups the
  assignable categories into `income`-then-`expenditure` sections, name-sorted
  (casefold) within each, omitting an empty section. Type is resolved by
  **ascending `parent_id` to the kind-bearing root** (depth-safe, D3), so a
  3-level branch's grandchild still groups under its ancestor root. The tuple key
  is the untranslated `CategoryKind` token — labelling stays in the UI layer.
- **`ui/_widgets.py`** — new `category_type_labels()` (one shared Income/
  Expenditure label map under a fixed `"CategoryTypeLabels"` translate context)
  and `add_grouped_categories()` (a disabled **bold** section header per non-empty
  section + a `Name (Type)` row per category; the resting selection skips a header
  onto the first enabled row but leaves an already-selectable current alone).
- **`ui/category_picker.py`, `ui/rules.py`** — both dialogs now take the
  `leaf_categories_grouped()` shape (D4). `RuleEditDialog`'s OK-gate moved from
  `count() > 0` to `currentData() is not None` (D5) — header-safe, still disabled
  with zero leaves (preserves FIBR-0079).
- **`ui/transactions.py`** — all three dialog call sites switched to grouped
  input; the category **filter** intersects the grouped list with the categories
  present in the visible rows (special rows `All categories` + `Uncategorised`
  stay above the headers so `findData(None)` resolves to Uncategorised, not a
  header). `_unique_categories()` removed.
- **`ui/categories.py`** — the manager's inline label dict replaced by the shared
  `category_type_labels()` (INV-4, one label source, no drift); the now-orphaned
  `CategoryKind` import removed.
- **CHANGELOG** — a `Changed` entry (grouped category pickers).

Tests added across `tests/features/{categorisation,transactions_tab,categories}/`
covering every invariant (INV-1 order/sort/omission, INV-2 no-drop multiset +

## What was learned

- **The service permits parent-chain cycles.** `CategoryService.update_category`
  accepts any existing parent (the UI only ever offers roots, so it's not
  user-reachable), so a direct `A→B→A` reparent is constructible. The new ascent
  would have looped forever on such corrupt data — the indie-review LOW. Now
  guarded with a `seen`-id set that raises `ValueError` (fail loud, never hang) —
  the right mode for a money app. Reproduced first: the regression test builds the
  cycle and hangs the ascent without the guard.
- **`cast`, not `assert`, for known-non-None narrowing in `src/`.** The codebase
  keeps `src/` assert-free (bandit B101); `cast(str, root.kind)` narrows the
  ascent's `str | None` result without a runtime guard, matching the existing
  `cast(CategorizationRule, repo.get(...))` idiom.
- **Two cold review lanes cost far less than a `/cold-eyes` loop** for code — a
  single bounded adversarial pass per lane + verify (~177k subagent tokens total),
  vs cold-eyes re-dispatching N lanes *per convergence loop*. (User observation.)
