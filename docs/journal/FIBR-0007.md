  `ParseResult` (transaction drafts + collected per-row `RowError`s + the

## What was learned

- **Exact-signature design removed a whole error class.** Applying a profile
  only on an exact header match (D4) makes "a stale profile mis-maps a changed
  export" unreachable — its mapped columns are present by construction — so no
  `ProfileMismatchError` (dropped as dead code at cold-eyes loop 2). Simpler and
  safer than a guard-and-recover path.
- **bandit B608 on f-string SQL, even with a literal constant.** A
  `f"SELECT {_COLUMNS} …"` where `_COLUMNS` is a module constant still trips
  B608. Root-caused (not `# nosec`) by inlining the literal column lists —
  matching the `AccountRepository`/`CategoryRepository` convention. B101 asserts
  similarly root-caused to `cast` (the `AccountService` convention).
- **Explicit `BEGIN` + `executemany` is safe** under the vault's
  `isolation_level=""`: the pre-insert reads don't open a transaction, so the
  explicit `BEGIN` doesn't double-begin — verified by the core-logic reviewer
  against the migration precedent.
