# FIBR-0148 — Deleting a statement must not lose transactions a remaining overlapping statement also covers

**Closed:** 2026-07-17 · **Kind:** fix (data integrity) · **Tag:** `FIBR-0148-complete`

## What changed

- **The bug.** Import dedup is a multiset delta: a duplicate transaction is
  dropped at import and the single stored copy is stamped to whichever statement
  imported it *first* (`transactions.statement_period_id`, single-valued). When
  two statements' periods overlapped, the shared rows lived under only one of
  them, so deleting that one destroyed rows the other, still-present statement
  legitimately covered — silent money-record loss.
- **The fix (direction (a) — ownership hand-off).** `StatementService.delete_statement`
  now runs three ordered steps in one `owned_transaction`:
  1. `TransactionRepository.hand_off_covered(period_id)` — a correlated `UPDATE`
     that re-stamps each row a *remaining* same-account statement also covers to
     that statement (deterministic owner: `ORDER BY q.period_start, q.id LIMIT 1`);
  2. `delete_for_statement(period_id)` — delete the rows still stamped `period_id`
     (nothing else covers them);
  3. delete the period row.
  The hand-off `UPDATE`'s `EXISTS` guard shares the `SET` sub-query's predicate
  **byte-for-byte**, so a row nothing else covers is never re-stamped to `NULL`
  (the money-critical NULL trap). It mirrors the v5→v6 backfill idiom
  (`_migrate_to_v6`) with the polarity flipped (`NOT EXISTS` → `EXISTS`, outer
  `WHERE … IS NULL` → `= :pid`, `SET` gains the deterministic pick).
- **No schema change.** Column, FK, and index already existed; `delete_statement`'s
  return value stays "rows actually deleted" (orphans), preserving the INV-9
  contract's meaning.
- **Tests.** New `tests/features/statements/` FIBR-0148 block, INV-1…INV-9,
  reproduce-first — the overlap-delete data-loss test fails on the pre-fix code.
  The existing non-overlapping INV-9a/b/c/d delete tests are unchanged (their
  fixtures never overlap).

## What was learned

- **Span-overlap is a faithful proxy for "the other statement listed it."** A bank
  statement lists every transaction in its period; import only dropped the shared
  rows because they already existed. So handing a row to any remaining statement
  whose period covers its date is correct, not a guess — the same assumption the
  v6 backfill already relied on.
- **The hand-off is *safer* for transfer-pair cascades than the old delete.** The
  review noted that because covered rows are no longer deleted, their
  `ON DELETE CASCADE` transfer-pair rows are preserved rather than lost — an
  incidental correctness gain.
- **A "deterministic" test can fail to test its own key.** The first INV-5 draft
  put the earliest-`period_start` statement at the lowest `id` too, so a broken
  `ORDER BY id` alone would still pass. Folded a fix so the earlier-starting
  statement carries the *higher* id — a genuine `(period_start, id)` falsifier.

## What was deferred

- **FIBR-0149** (planned, `ux`) — the delete confirm dialog still says "delete its
  N transaction(s)? This cannot be undone" with N = the full linked count, which
  overstates the loss in an overlap delete (some survive). Spec D5 deferred the
  wording deliberately; the review sharpened it, so it is now tracked as a
  follow-up (compute the would-orphan count + reword + add the overlap-path
  confirmation test).
