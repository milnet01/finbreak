# FIBR-0005 — P03 multiple accounts per profile (journal)

Closed 2026-07-02. Money now lives in accounts: several accounts per
profile, each tagged with a type, every transaction linked to one — plus
the first forward-only schema-migration runner (the machinery every later
schema change reuses).

## What changed

- **Data model & migration.** `models.py` gains `AccountType(StrEnum)` (7
  tokens) + the `Account` record and an `account_id` on `Transaction`.
  `migrations.py` (new) is the forward-only runner: a runner-owned
  `try: BEGIN … COMMIT except: ROLLBACK; raise` v1→v2 step that creates
  `accounts`, seeds one "Default" account, rebuilds `transactions` with a
  `NOT NULL account_id REFERENCES accounts(id)` (SQLite table-rebuild, not
  `ALTER ADD COLUMN`), backfills every prior row to Default, and bumps the
  version. `vault.py` runs it on both `create()` and `open()`, enables
  `PRAGMA foreign_keys = ON`, and cleans up a half-open vault on failure.
- **Repositories & services.** `repositories/accounts.py` (CRUD + `count`),
  `services/accounts.py` (name/type validation with own-name-exclusion on
  update, and the delete guard: block-in-use + keep-≥1). `transactions`
  repo/service gain the required `account_id` and an id→name join for
  display.
- **UI (PySide6).** `ui/accounts.py` (new) — the Accounts manager:
  add / **edit (Update selected)** / delete, a type picker carrying the
  language-neutral token behind a `tr()` label. `ui/main_window.py` gains
  the account picker + an Account column + a "Manage accounts…" button;
  `app.py` routes the new view.
- **Errors.** `AccountInUseError`, `LastAccountError`, `SchemaVersionError`
  (all distinct types so guarantees are assertable by exception type).

## What was learned

- **The migration transaction must be the runner's own.** Under
  `isolation_level=""` sqlcipher3 does not auto-`BEGIN` around DDL, so an
  un-wrapped `DROP`/`RENAME` autocommits and the rollback silently wouldn't
  fire. Verified empirically before coding: an explicit `BEGIN` as the
  step's first statement owns the transaction, and a forced failure at
  `RENAME` rolls the `DROP` back to a re-openable v1 vault.
- **A "recoverability" test is not an "atomicity" test.** The first INV-4
  rollback test reopened the vault (which re-ran and *completed* the
  migration) and only checked the rows survived — so a no-rollback
  regression would still pass. The review caught it; the test now asserts,
  on the same connection before any reopen, `schema_version == 1`, accounts
  absent, and rows intact.
- **A "built + tested" service method can still be unreachable.**
  `update_account` was fully tested headless but never wired to a button,
  so the spec's "add/edit form" deliverable was unmet and a mistyped
  account name was permanent (delete is block-only). The review surfaced
  the spec-vs-code gap; wiring the edit path (+ INV-7f) closed it.
- **Fix the tool signal, don't suppress it.** A mypy `annotation-unchecked`
  note was two test bodies going unchecked — fixed at root (add `-> None`)
  rather than allowlisted; the allowlist stays empty.

## What was deferred

- **Selecting a row overwrites in-progress form input** — an accepted MVP
  UX edge (matches the D8 single shared add/edit form); no dirty-guard. Not
  a contract violation; captured here rather than fixed.
- **Editing an existing transaction's account, reassign/bulk-move,
  delete-time reassignment** (D9/D6) — a later transaction-editing phase.
- **Per-account currency / opening balance / institution, archiving,
  user-ordering** — model is name + type only; a later enhancement.

## False positives confirmed

- None. `docs/audit-allowlist.md` remains empty — every review finding
  across two rounds was fixed inline. The mypy `annotation-unchecked`
  note was a real (if trivial) signal, root-caused rather than suppressed.
