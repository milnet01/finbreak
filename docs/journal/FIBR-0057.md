# FIBR-0057 — Import wizard mis-links the target account — closed 2026-07-09

**Kind:** fix (my own code — the sole-author rule owns it). User-reported: an
imported SBSA credit-card statement landed on "Current" instead of "Credit Card".

## Root cause

`ui/import_wizard.py` snapshotted the destination account **at file-select**
(`_select_file`: `self._account_id = self._account_combo.currentData()`) and baked
it into the `ImportPreview`; `commit_import` writes `preview.account_id`. The
account picker lived **only on the pick step**, which is left behind once a file is
chosen — so there was no way to see or change the destination before the
irreversible Import. A user who clicked the prominent "Choose a statement file…"
button before touching the account dropdown silently imported under the default
(first) account.

## What changed

- **Preview-step destination picker (the fix).** `_build_preview_step` now carries
  an "Import into account" combo (`_confirm_account_combo`), the **single source of
  truth** for the committed account. It is seeded from the pick-step choice at
  file-select (blocked, so seeding fires no re-dedup) and read live via a new
  `_target_account_id()` by **every** preview path (CSV auto-match + mapping, OFX
  single/multi, generic PDF, Standard-Bank PDF) and the per-account PDF-password
  lookup. Changing it on the preview step re-runs the dedup and refreshes the
  counts, so the shown counts and the committed rows land on the chosen account.
- **`ImportService.retarget(preview, account_id)`** (new) re-runs the multiset
  dedup of an already-parsed preview under a different account — drafts, per-row
  errors and the coverage span are account-independent, only the delta + baked
  `account_id` change. Extracted `_build_preview` as the single construction point
  shared with `_preview_from_result` (no behaviour change for existing callers).
- **The pick-step combo stays.** The PDF-password step (`_decrypt_pdf`) needs an
  account *before* a preview exists (per-account stored password), so the account
  must be selectable up front; the pick-step combo seeds the confirm combo.
- **Cold-review fold (MEDIUM):** a remembered locked-PDF password is persisted
  under the provisional file-select account (INV-7f persists on decrypt, before a
  preview exists). On a **re-targeted** import the rows commit elsewhere, so the
  password was orphaned on the wrong account. `_carry_stored_pw_to_committed_account`
  now also stores it on the committed account — conservatively (fills an empty slot
  only, never clobbers; leaves the provisional account's own password intact).

## What was learned

- **The account genuinely can't move entirely to the preview step.** The FIBR-0009
  per-account stored PDF password is looked up during decrypt, which runs before
  any preview — so a pick-step account is a hard requirement, not redundancy. The
  clean shape is *pick-step seeds → preview-step confirms/corrects*, with one live
  accessor as the source of truth.
- **A cross-feature interaction the diff didn't obviously touch.** Making the
  account correctable re-opened FIBR-0009's remember-password contract: password
  target and row target could now diverge. The cold review caught it; the fix is a
  post-commit carry-over, keeping INV-7f's persist-on-decrypt timing intact.
