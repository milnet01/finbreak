# FIBR-0055 — Settings screen (configurable auto-lock timeout) — closed 2026-07-09

Near-term user request (2026-07-09): add a Settings menu whose first control is a
user-set auto-lock timeout. Pulled the Settings-screen scaffold + timeout portion
forward from FIBR-0014/P12; FIBR-0014 keeps the rest (encrypted backup, theme
system + toggle, stored-PDF-password management, i18n host). **Kind:** feature.

## What changed

- **`repositories/settings.py` (new)** — `SettingsRepository(connection)` with
  `get(key) -> str | None` and `set(key, value)` (SQLite `ON CONFLICT(key)` upsert +
  commit). A generic key→value seam over the v1 `settings` table (Rule of Three:
  `base_currency` + `minor_unit_exponent` + the new `auto_lock_minutes`). The two
  existing raw reads were deliberately left alone (stay-in-lane).
- **`services/auth.py`** — `AUTO_LOCK_MINUTES` → `DEFAULT_AUTO_LOCK_MINUTES` (=10) +
  `ALLOWED_AUTO_LOCK_MINUTES = (1, 5, 10, 15, 30)`; `auto_lock_minutes()` (a
  **guarded parse** — absent / non-integer / out-of-set all fall back to the secure
  default) and `set_auto_lock_minutes(n)` (**validate → persist → re-arm**, in that
  order); `_arm_timer` now reads the configured value, so first-run / unlock / every
  change picks it up.
- **`ui/settings.py` (new)** — `SettingsDialog`: an auto-lock combo (per-value
  `tr()` labels, `userData` = int minutes, preselected via `findData`) + a read-only
  base-currency `QLabel`. Save calls the service and emits `saved`; the shell owns
  the close (the `ManualEntryDialog` tracked-modal pattern). The dialog holds **no**
  vault / `TransactionService` reference — the shell reads the currency and passes it.
- **`ui/main_window.py`** — a `File → Settings…` action (vault-dependent chrome,
  disabled while locked) + `_open_settings` / `_on_settings_saved` wiring.
- **No schema change** — the `settings` table has existed since v1 (FIBR-0004);
  absence of the `auto_lock_minutes` row means the 10-minute default, so
  pre-FIBR-0055 vaults keep working and gain the row only on first change.

## What was learned

- **The spec's INV-7 `self._dialog is None` claim was wrong — TDD caught it.** The
  idle-auto-lock sequence (`MainWindow._lock`) **re-opens the `UnlockDialog` into
  `self._dialog`** as its last step, so after an auto-lock `_dialog` is the fresh
  unlock dialog, not `None`. The real security property is that the *SettingsDialog*
  is destroyed — the test captures the dialog ref and asserts
  `shiboken6.isValid(dialog)` is `False` after a `DeferredDelete` flush; the spec was
  corrected to match (no silent drift).
- **A disabled `QMenu` does not disable its child `QAction`s** — `QAction.isEnabled()`
  keeps its own flag. Cold-eyes loop 3 caught this: INV-6's test asserts at the menu
  level (`_menu_file.isEnabled()`), mirroring the house `test_INV6c` idiom.
- **A "Never auto-lock" option was deliberately excluded** (D6) — it would defeat the
  security spine of a decrypted financial vault. A warning-gated version is a later
  explicit follow-up.
