# FIBR-0006 — P04: Type → Category tree (closed 2026-07-02)

## What changed

- **Category data model.** A self-referential `categories(id, parent_id, name,
  kind, created_at)` table. The two **Type roots** (Income / Expenditure) have
  `parent_id IS NULL` and carry a `CategoryKind` token (`income` /
  `expenditure`); every descendant has `kind = NULL` — a category's Type is its
  root ancestor. `Category` dataclass + `CategoryKind(StrEnum)` in `models.py`.
- **v2→v3 migration** (`migrations.py`): `LATEST_SCHEMA_VERSION = 3`, a new
  `_migrate_to_v3` step that `CREATE`s `categories` and seeds the two roots + 16
  default categories, wrapped in the runner-owned `BEGIN … COMMIT`/`ROLLBACK`
  (explicit `BEGIN` first, discrete `execute` calls) so it is atomic — a failure
  mid-seed rolls back to v2 with no `categories` table. Runs on both
  `Vault.create()` (first-run → v3) and `Vault.open()` (upgrade), reusing the
  FIBR-0005 runner unchanged. `transactions`/`accounts` untouched.
- **Repository + service.** `CategoryRepository` (CRUD, `children_of(None)` →
  the two roots via `WHERE parent_id IS NULL`); `CategoryService` — sibling-name
  validation (non-empty, unique within a parent, ci), the root-count guards
  (add/update reject a `None` parent → `ValueError`; editing a root →
  `ProtectedCategoryError`), and the delete guard (root → `ProtectedCategoryError`;
  has-children → `CategoryHasChildrenError`; childless leaf deletes; missing id
  no-op). Two new error types in `errors.py`.
- **UI.** `CategoriesWidget` — a `QTreeWidget` showing the two Type roots + their
  categories, a shared add/edit form (name + a Type picker carrying the root id),
  and per-selection root protection (selecting a Type root disables Update +
  Delete). A "Manage categories…" button + `manage_categories` signal on
  `MainWindow`; `AppShell` routes it as a normal stacked view (auto-lock-safe),
  mirroring the Accounts view.
- **Deferred by design (D10):** transactions gain **no** `category_id` this
  phase — the transaction→category link and its auto/manual assignment are owned
  by P08 (FIBR-0010). P04 stays surgical: a new table + screen, transactions
  untouched.
- **Test infra.** Lifted the raw-v1-vault builder (`build_v1_vault` + `_params`
  + `_PW` + a `keyed_connection` helper) into `tests/conftest.py`, shared by the
  accounts + categories migration suites. Rippled the four schema-version
  assertions `== 2` → `== 3` (vault suite ×1, accounts suite ×3) and renamed the
  two `…_v2…` accounts tests to `…_v3…` / `…_at_latest`. New
  `tests/features/categories/` (spec.md + 22 tests, INV-1..8).

## What was learned

- The `LATEST_SCHEMA_VERSION` bump ripples further than the naive "one vault-suite
  assertion" — the accounts suite has three post-migration version assertions and
  two version-named tests that also move. Cold-eyes loop 1 caught this before code;
  the spec's Tests section enumerates all four flips + two renames, which made the
  ripple a mechanical edit.
- The atomic-rollback wedge for a create-and-seed migration must raise on the
  **first** `INSERT INTO categories` (not a later one) so the test proves the
  `CREATE TABLE` itself is inside the rolled-back transaction — a stronger guarantee
  than the FIBR-0005 `RENAME` wedge.
- Bandit's B608 fires on any f-string in `.execute()`, even with a hardcoded column
  constant — the codebase convention (literal SQL) sidesteps it cleanly; no `# nosec`
  needed.

## What was deferred

- Transaction→category link, auto/manual category assignment → **P08 (FIBR-0010)**.
- The third UI level (sub-categories), category ordering/icons/colours, and
  cycle-prevention on arbitrary re-parenting → later enhancements (the two-level
  UI can't create a non-root parent, so no cycle can arise this phase).
- Aggregation by category/Type + the root-walk kind resolver → **P10** (dashboard).
