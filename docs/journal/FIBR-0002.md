# FIBR-0002 — `.gitignore` + secret-leak guard (journal)

**Closed:** 2026-07-01 · **Kind:** chore · **Tag:** `FIBR-0002-complete`

## What changed

- Extended `.gitignore` (did not rewrite it) with three grouped,
  commented sections: financial data / local vaults
  (`*.db`, `*.sqlite`, `*.sqlite3` + the sidecar globs `*.db-*`,
  `*.sqlite-*`, `*.sqlite3-*`), tooling caches (`.pytest_cache/`,
  `.ruff_cache/`, `.mypy_cache/`), and packaging / build output
  (`build/`, `dist/`, `*.egg-info/`, `*.dmg`, `*.AppImage`, `*.flatpak`,
  `.flatpak-builder/`). Closes the `.gitignore` half of
  security-model **T6**; `gitleaks` (FIBR-0001) stays the content backstop.
- Added `tests/features/gitignore/` (`spec.md` + `test_gitignore.py`,
  23 parametrized cases) enforcing INV-1..INV-3 via
  `git check-ignore --no-index`, run against a copy of the repo
  `.gitignore` inside a throwaway git repo.

## What was learned

- **`git check-ignore` has two silent traps** the spec's cold-eyes loop
  caught before implementation: it reports every *tracked* file as
  "not ignored" (needs `--no-index`), and a directory pattern like
  `build/` won't match a bare `build` query (needs an inside-path like
  `dist/finbreak.exe`). A naive test would have looked green while
  proving nothing.
- **A live SQLite/SQLCipher vault leaves sidecars** (`-wal` / `-shm` /
  `-journal`) that hold the same data; `*.db` alone doesn't cover them,
  so `*.db-*` (etc.) were added — a MEDIUM cold-eyes finding.
- **Test in isolation, not in-place.** pytest/ruff drop their own
  `.gitignore` (`*`) inside their cache dirs, which masks whether *our*
  rule ignores them — so the test copies the repo `.gitignore` into a
  fresh repo to test that file's rules alone. `/indie-review` then found
  a related coupling: the fixture must also neutralize the developer's
  **global** git excludes (`-c core.excludesFile=/dev/null`), else a
  global `*.py` rule would false-red INV-3. Fixed inline during close.
