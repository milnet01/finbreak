# FIBR-0052 — P07.6: tabbed main window + statement provenance & delete

Closed 2026-07-09 by `/close-phase`. Kind: implement. Builds on FIBR-0051.

## What changed

- **Tabbed workspace.** The FIBR-0051 single-content `QStackedWidget` model is
  reshaped into a persistent `QTabWidget` (`ui/main_window.py`) with four fixed
  tabs — `tab_home` · `tab_statements` · `tab_accounts` · `tab_categories` —
  mirrored under the View menu and reachable via toolbar buttons. Navigation is a
  tab **switch**, not a rebuild (stable instances, INV-2a). Import remains a flow
  that replaces the workspace and rebuilds it (landing on Statements) on `done`.
- **Home toolbar button + house glyph** (`ui/icons/home.svg`), first in the
  toolbar (Home · Manual entry · Import · Accounts · Categories · Lock).
- **Window menu + geometry.** A vault-independent `Window` menu (Center window /
  Reset layout, enabled while locked) + size/position/state/last-tab persisted to
  a plain INI at `paths.window_settings_path()` (`window.ini`), **outside** the
  encrypted vault, restored before unlock. `_DEFAULT_WINDOW_SIZE = QSize(1000,
  700)`.
- **Statement provenance (v5→v6).** A nullable `transactions.statement_period_id
  INTEGER REFERENCES statement_periods(id)` column (`_migrate_to_v6`).
  `commit_import` is **reordered** to resolve/create the period row first, then
  stamp the batch with its id (`add_batch(rows, statement_period_id)`); manual
  entry stays `NULL`. A one-time, ambiguity-guarded backfill (`NOT EXISTS` overlap
  guard) links pre-v6 rows under exactly one period; overlaps stay `NULL`. The
  backfill runs inside the migration's atomic transaction.
- **Statements tab (read + delete).** `StatementService` (`services/statements.py`)
  — `list_statements()` (a `LEFT JOIN` + `COUNT` so a zero-linked statement still
  lists with count 0) and `delete_statement()` (one owned `BEGIN`: children then
  parent, `COMMIT`/`ROLLBACK`). `StatementsWidget` (`ui/statements.py`) lists
  imports + a Delete action that confirms (naming the count) and refreshes Home +
  the status count. New repo methods: `StatementPeriodRepository.add()->int` /
  `id_for_span` / `list_all` / `delete`; `TransactionRepository.delete_for_statement`.
  New `StatementRow` model.
- **`show_done` flag** on `AccountsWidget`/`CategoriesWidget` (default `True`) —
  the tab-mode build passes `False` to drop the now-redundant "back to Home" Done
  button; the `done` signal stays (unfired), so no existing caller/test breaks.
- **Tests.** 27 `tests/features/statements/` (INV-1..INV-13 + sub-invariants), the
  FIBR-0051 `app_shell` ripple (nav/lock/manual-entry re-homed to the tab model),
  and the v5→v6 schema-version ripple (`== 5` → `== 6`, name flips, a new v5→v6
  upgrade test) across vault/accounts/categories/import_/ofx_import/pdf_import.
  Gate green **324 passed / 1 skipped**, mypy 0; `/audit` 0. Validated end-to-end
  against a real Standard Bank credit-card PDF in a throwaway vault (53 txns,
  exact count, clean delete) — PDF/password never committed.

## What was learned

- **The `commit_import` reorder is required for FK correctness, not just for the
  stamp.** With `PRAGMA foreign_keys = ON` and an immediate FK, the child
  `add_batch` rows can only reference a parent `statement_periods` row that already
  exists in the same uncommitted transaction — so period-first ordering is
  load-bearing, not cosmetic.
- **The reorder silently drained an existing atomicity test.** The FIBR-0007
  `test_INV7_atomic_write_rolls_back_both_tables` wedged on the period INSERT,
  which is now the *first* write — so it fired before any transaction row and no
  longer proved "rollback undoes the batch." Caught by cold review; the wedge was
  rewired to fail on the `add_batch` executemany. A reorder can invalidate a test
  that still passes green.
- **Offscreen QPA does not byte-round-trip `saveGeometry`.** Width/x are adjusted
  by frame margins; height/y are the portable witnesses that restoration (not
  defaulting) occurred — the INV-5a geometry test asserts on those + the persisted
  blob.
- **A nested `QMessageBox.question` is not shell-tracked.** An idle auto-lock can
  fire while the delete confirmation is open, closing the vault under it; the fix
  catches the specific `VaultLockedError` on the post-confirm delete.

## What was deferred

- A dedicated per-statement transaction **view** (double-click a statement → its
  transactions) — cheap now the stamp exists, not requested; a near-term follow-up.
- **Undo** of a delete (confirmed but not undoable).
- The Home **dashboard** (income/expenditure summary + breakdown) — **FIBR-0012**,
  blocked on P08 (category link) + P09 (transfer detection).
- **Re-unlock tab restore** (INFO): `_initial_tab` is captured once at construction,
  so a lock→unlock lands on the startup tab, not the pre-lock tab. Within the
  contract (D7 restores "when the window is constructed"); a small future nicety.

findings; every `/indie-review` finding was actionable and fixed inline.
