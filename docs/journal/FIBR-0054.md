# FIBR-0054 — Optional in-app auto-update (Linux / AppImage first)

Closed 2026-07-14 by `/close-phase`. Kind: feature. Tag: `FIBR-0054-complete`.

## What changed

The app's first-ever, opt-in (off-by-default) network egress: a launch-time
update check against the GitHub Releases API, an Ed25519-signature-gated
download, an atomic AppImage swap, a key-wipe, and a detached relaunch — built in
two phases (a real signed Linux release, then the in-app updater). The feature
was implemented + released back at **v0.1.0** (2026-07-11) and has shipped and
been field-proven through **v0.1.9**; this record covers the deferred *close*.

- **Services / security core:** `services/update.py` (`UpdateService`:
  opt-in/skip prefs on the plaintext INI, `check_for_update`, the
  signature-verified `download_and_verify`), `services/update_fetch.py` (the sole
  `urllib` module — https-only, byte-capped, timeout-bounded), `update_installer.py`
  (`Installer` protocol + `AppImageInstaller` swap/relaunch + `detect_installer`),
  `update_key.py` (committed Ed25519 public key), `errors.py`
  (`UpdateVerificationError`).
- **UI:** `ui/update_dialog.py` (Later / Skip / Update-now + inline release notes +
  busy state), `ui/_update_worker.py` (check + download `QThread`s),
  `ui/main_window.py` (launch check, D15 pending-offer lifecycle, Update-now →
  download → apply), the Settings "Check for updates on startup" checkbox.
- **Field fixes folded during the feature's life** (pre-close, already released):
  the in-place `os.execv` relaunch → detached `subprocess.Popen` +
  `PYINSTALLER_RESET_ENVIRONMENT` (v0.1.4); the `_relaunch_env` `LD_LIBRARY_PATH`
  restore from the `*_ORIG` vars so the `/bin/sh` waiter runs against system libs
  (FIBR-0122, v0.1.8). See [[appimage-update-relaunch-caveat]].
- **Live proof:** the user's **v0.1.8 → v0.1.9** self-update downloaded, verified,
  swapped, **and auto-relaunched the new version** (commit `8e4a298`) — the true
  post-v0.1.8 two-cycle test, which was the last gate before this close.

## Close (steps 5–9)

`/audit` (semgrep/ruff/bandit/gitleaks, scope full) + two cold `/indie-review`
lanes (services-security; ui-wiring-tests) over the auto-update surface.

**Findings — all fixed inline** (project standing directive: no FP## deferral),
commit `67132c1`:

- **MEDIUM** — `ui/main_window.py` `_on_download_failed` lacked the auto-lock guard
  its `_on_download_ready` sibling has. A download that *failed* after an idle
  auto-lock tore the busy prompt down would `_teardown_dialog()` the re-opened
  UnlockDialog and pop a stray "Update failed" warning over the lock screen. Now
  captures the prompt and mirrors the `self._dialog is prompt and
  shiboken6.isValid(prompt)` guard — silent when the prompt is gone (INV-9/INV-11).
  Two regression tests added (live-prompt warns + closes; torn-down stays silent).
- **LOW** — `services/update.py`: the two `_stage_temp` (`mkstemp`) calls sat
  *outside* the `try`, so a read-only / full `$APPIMAGE`-dir staging failure
  escaped as a raw `OSError` (not `UpdateError`) and could orphan the first temp.
  Staged inside the `try` now, with an `_unlink(Path | None)` helper; the verified
  path returns from inside the `try` (avoids a bandit-B101 `assert` for the
  `Path | None` narrowing).
- **LOW (test)** — `test_D15_skip_persists_and_no_reprompt_after_relock` never
  actually re-locked/unlocked (it called `_maybe_show_pending_offer()` directly).
  Tightened to a real `_lock()` → `unlock()` → `_enter_unlocked()` round-trip so
  the `_enter_unlocked` tail is what proves the silence.

## What was learned

- The `ready` path was carefully guarded against auto-lock teardown; the sibling
  `failed` path was not — the asymmetry between two nearly-parallel handlers was
  the tell. Worth grepping for "one branch guards, the sibling doesn't" whenever a
  worker has both a success and a failure slot.
- Both cold lanes independently judged the security spine sound: no path installs
  an unverified binary, the swap is atomic, the key-wipe ordering holds, and the
  network surface is confined + bounded. The single MEDIUM was a UX/robustness
  glitch, not a security hole.

## False positives confirmed

- **allowlist-001** — semgrep `dynamic-urllib-use-detected` ×2 at
  `services/update_fetch.py` (the two `urlopen` calls). This is the one module the
  design deliberately allows to use `urllib` (INV-12/D9); URLs are
  GitHub-API-sourced, https-only, byte-capped, and Ed25519-verified before
  install. Suppressed inline (`# nosemgrep`, beside the existing `# nosec B310`).
