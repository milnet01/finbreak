# FIBR-0138 — Expandable dashboard drill-down (journal)

Closed 2026-07-14 by `/close-phase`. Spec: `docs/specs/FIBR-0138.md`
(`/cold-eyes`-converged loop 5, 5 loops × 3 cold lanes).

## What changed

- **`models.py`** — two frozen dataclasses: `DrillNode(label, amount, count,
  children)` (one recursive read model for every tree level) and `DrillLabels(income,
  spending, transfers, uncategorised)` (the four `tr()`-ed fixed strings the caller
  injects, since the non-`QObject` service can't translate — D2/INV-9).
- **`text.py`** — `merchant_name(description) -> str`, a pure + **total** best-guess
  shop-name cleanup (strip leading noise prefixes longest-first as whole
  words/phrases; drop ≥3-digit and majority-digit tokens; strip edge punctuation;
  title-case, else fall back to the trimmed raw). stdlib `re` only — no new dependency.
- **`repositories/reporting.py`** — `drill_rows_in_range`, a deliberate sibling of
  `rows_in_range` returning the 5-tuple `(id, occurred_on, amount_minor, category_id,
  description)` — one added `description` column, no `account_id` (the scope is already
  in the `WHERE`). The three hot aggregations keep their lean 4-tuple.
- **`services/reporting.py`** — `ReportingService.drill_down(prefs, account_ids,
  today=None, *, labels)` returning exactly `[Income, Spending, Transfers]`. One
  "group by top-of-chain" category algorithm (climb `parent_id` to the child-of-a-root,
  so no row is ever dropped and the branch total equals its tile — INV-1); a per-branch
  `_category_node` closure summing own bucket + descendants and mixing child-category
  and merchant nodes; the account-pair transfers branch. Every sibling list sorted by
  the INV-7 `(magnitude desc, label asc, uniform-string-key asc)` key. Spending
  accumulates `-amount_minor` (positive magnitude), so `to_display_decimal(sum)` equals
  `Summary.expenditure` exactly (`scaleb` is an exact shift — no rounding drift).
- **`ui/home.py`** — the dashboard content wrapped in a `QScrollArea`; a
  `dashboard_drilldown` `QTreeWidget` (sorting off — the service pre-sorts);
  `refresh()` builds the `DrillLabels` via `tr()` and renders after the trend;
  `_render_drilldown` / `_add_node` insert the tree, thread the branch colour, and
  append `×N` only to merchant/account-pair nodes (structurally: a node whose children
  are all leaves).
- **Tests** — `tests/features/dashboard_drilldown/{spec.md, test_dashboard_drilldown.py}`,
  41 legs covering INV-1..INV-9.

## What was learned

- **`to_display_decimal` is `Decimal(minor).scaleb(-exp)`** — an exact decimal-point
  shift, so per-row-scale-then-sum equals sum-then-scale with no rounding drift. That is
  what lets the drill sum display Decimals of children and still match the tile exactly.
- **A category cycle is reachable.** `CategoryService.update_category` blocks
  re-parenting a *root* but has **no descendant-cycle guard**, so a user can re-parent a
  category under its own child (X→Y→X). The sibling `categorization.type_of` already
  fails loud on that; the drill instead stays **total** — a visited set in *both*
  `top_of_chain` and `category_node` breaks the cycle and keeps every row counted, so a
  corrupt tree can't hang or overflow the always-rendered dashboard. (The missing
  `CategoryService` guard is a pre-existing latent issue — see Deferred.)
- **The `×N`-only-on-merchant/pair rule needs no `kind` field** — those nodes are
  exactly the ones whose children are all individual-transaction leaves, so structure
  alone distinguishes them from category/top nodes.
