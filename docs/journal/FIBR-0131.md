# FIBR-0131 — Windows in-app auto-update (closed 2026-07-14 by /close-phase)

## What changed

Made the FIBR-0054 updater actually *install* on Windows, behind the `Installer`
seam FIBR-0054 built for exactly this (its module docstring already named
`WindowsInstaller`).

- **`services/update_installer.py`** — `WindowsInstaller` (behind the existing
  `Installer` protocol): `apply` **spawns** a detached PowerShell helper, **then**
  wipes the key, **then** `os._exit`s — no in-process move (the running `.exe` is
  OS-locked). `_windows_relaunch_command` is the Windows twin of the `/bin/sh`
  waiter: it waits by the exe **image path** (`Get-Process | Where-Object { $_.Path
  -eq $exe }`, **not** a PID — tree-agnostic across the onefile parent/child tree and
  proof against Windows PID-recycling), **aborts** (removes the staged temp, no
  relaunch) if the image never frees within ~60 s, else retries `Move-Item` then
  `Start-Process`. Absolute `powershell.exe` resolved from `%SystemRoot%`;
  single-quote-escaped paths; Windows-only `creationflags` `getattr`-guarded so
  `apply` loads + runs on the Linux gate. `_windows_relaunch_env` sets only
  `PYINSTALLER_RESET_ENVIRONMENT=1`. `detect_installer()` → `WindowsInstaller` on a
  frozen Windows `.exe`; `asset_suffix()` added to the protocol + both installers.
- **`services/update.py`** — `_select_assets(assets, suffix)` is installer-driven;
  `check_for_update` short-circuits to `None` (zero fetcher calls) with no installer
  and sources the suffix from the installer; `download_and_verify` stages the temp
  with the installer-derived extension; `UpdateInfo.appimage_url` → `asset_url`.
- **UI / docs** — generalised the AppImage-only Settings tooltip + `main_window`
  call-site comments; `security-model.md` update-flow clause amended to cover the
  `.exe` under the **same** Ed25519 gate (Authenticode = separate FIBR-0133);
  `.claude/bump.json` release runbook gained the `.exe` sign-and-attach-`.sig` step;
  `tests/features/auto_update/spec.md` coverage table gained the `FIBR-0131 INV-N`
  rows.
- **Tests** — +32 `tests/features/auto_update/` legs (detect matrix, per-platform
  picker, image-path command builder, `apply` spawn→wipe→exit ordering + spawn
  failure, installer-derived temp, installer-None short-circuit, Linux spawn-failure
  guard). Existing `_select_assets`/`appimage_url` legs re-pointed. Gate green
  **877 / 1**, mypy 0.

## What was learned

- **Wait by image path, not a PID.** The spec brainstormed a `getppid()` PID wait
  through several cold-eyes loops; loop 4 replaced it with an image-path poll, which
  is simultaneously tree-agnostic (a onefile build is a parent-bootloader→child-app
  tree; the on-disk `.exe` lock is held while either runs) *and* immune to Windows
  PID-recycling. This dissolved a whole recurring class of correctness questions.
- **Spawn before wipe.** The close-phase review (both lanes) caught that wiping the
  key *before* the `Popen` meant a spawn failure (AV/AppLocker blocking
  `powershell.exe`) threw out of a `NoReturn` method and left a live GUI with a
  wiped key. Reordered to spawn→wipe→exit — the helper blocks until this process
  exits, so ordering the wipe after a *successful* spawn is safe and makes a spawn
  failure non-destructive. The Linux twin was guarded the same way (a `Popen`
  failure after the committed `os.replace` now still `os._exit`s).

## What was deferred / pending

- **Live Windows swap+relaunch — a two-cycle manual verification** on the user's
  Windows box, and the same "the running old version does the relaunch" caveat as
  Linux ([[appimage-update-relaunch-caveat]]): it only *executes* on the update
  **after** the one that first ships this code. It also needs a release that
  attaches the **Ed25519 `.exe.sig`** — v0.1.9 shipped `finbreak-0.1.9-x86_64.exe`
  but no `.sig`, so the Windows updater is inert until a release carries it
  (documented in `bump.json`). This is the honest post-ship gate, not a blocker to
  closing the implementation phase (mirrors FIBR-0054's posture).
- **Authenticode / SmartScreen publisher signing** → **FIBR-0133** (SignPath,
  blocked on external approval). Orthogonal to this integrity gate.

## False positives confirmed

- `/audit` surfaced 3 `bandit` **B101 (assert_used)** hits in
  `tests/features/auto_update/test_auto_update.py` — the standard "asserts in test
  code" pattern. The gate runs `bandit -r src` (tests deliberately out of scope), so
  these are non-actionable and outside the gated path; no allowlist entry needed.
