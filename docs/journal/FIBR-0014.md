# FIBR-0014 — Encrypted backup export / restore

**Closed 2026-07-13 by `/close-phase`.** Kind: `implement`. Security: HIGH. Tag
`FIBR-0014-complete`.

## The D2 spike (run before any production code)

Per the spec's rule-13 requirement, a throwaway spike
(`scratchpad/d2_spike.py`, not shipped) proved the SQLCipher mechanics
empirically on the pinned `sqlcipher3-binary==0.6.0` (**SQLCipher 4.12.0
community / openssl**) before a line of code:

- `sqlcipher_export` exists and round-trips schema + all rows, **including the
  `schema_version` table** (so the version travels).
- `PRAGMA rekey` re-keys in place — old key then fails, new key opens, data intact.
- HMAC-on ⇒ a wrong key fails page-1 (fail-closed).
- With `temp_store=MEMORY`, **no plaintext** (a seeded sentinel) spills to any
  temp/journal file beside the backup DB.
- **Key finding:** `cipher_compatibility` / `cipher_default_compatibility` are
  **write-only** pragmas — reading them returns `None`. So restore must *apply*
  the recorded level explicitly and never read it back. Proven functionally: with
  the process default forced to 3, the compat-4 backup opens **only** with an
  explicit `cipher_compatibility=4`.

The spike outcomes matched the spec's D2 assumptions exactly — no `export_to`
shape change was needed.

## What was learned

- **The cipher-compat pragmas are write-only.** The whole INV-13 portability
  story rests on *applying* the recorded level (never reading it); the spike
  caught this before it became a "read returns None" bug in restore.
- **`bytearray(key)` at a call site is a silent key leak.** Passing
  `bytearray(backup_key)` to `open`/`rekey` (which only read `key.hex()`) makes an
  un-wiped copy outside the `finally` wipe set — the cold-review HIGH. Pass the
  original; the `finally` then wipes the only reference.
- **`json.loads` raises `UnicodeDecodeError`, not `JSONDecodeError`, on non-UTF-8
  bytes**, and a corrupt DEFLATE entry raises `zlib.error` (not `OSError`). Both
  escaped the `BackupError` normalisation on crafted input until the fold-in — a
  reminder that "normalise every failure" needs the *actual* exception taxonomy,
  not the obvious one.
- **Two app-modal dialogs can't share the one `_dialog` slot.** Opening the
  export/restore dialog without tearing down its originator left the old modal
  trapping input after success — the shell's single-slot invariant, easy to break
  from a new entry point.

## What was deferred

None blocking. Documented out-of-scope items (unchanged from the spec): pruning
the `*.old` copies a restore leaves behind (manual for now); raising the
per-backup Argon2 cost above the vault's; and pinning the **live** vault's
SQLCipher `cipher_compatibility` (the same cross-version concern INV-13 solves for
the `.fbk`, recommended as a companion hardening).
