# FIBR-0059 — Edit a logged statement's account — closed 2026-07-09

**Kind:** feature. User request 2026-07-09: after an SBSA credit-card statement
imported to "Current" (the FIBR-0057 mis-link), correct an already-logged
statement's account **in place**, without deleting + re-importing.

## What changed

- **`StatementService.reassign_account(period_id, new_account_id) -> int`** —
  atomically re-points `statement_periods.account_id` **and** every transaction
  stamped with it (`statement_period_id = id`) to the chosen account, returning
  the transactions moved. One service-owned `BEGIN … COMMIT` mirroring
  `delete_statement`; `ROLLBACK` to a re-openable vault on any failure.
- **Span-collision guard (INV-3).** A pure read + refuse that runs **before**
  `BEGIN` (so a refusal opens no transaction): `id_for_span(new_account_id, …)`
  with the **`period_id` self-exclusion** (`existing not in (None, period_id)`).
  A same-account move returns `period_id` → excluded → the INV-5 no-op; a real
  collision (the target already has that span) → a different id → `ValueError`,
  which would otherwise let a later import duplicate rows.
- **Distinct `reassigned` signal.** The shell's `changed` handler hard-codes
  `_status(tr("Statement deleted"))`, so a move emits its **own** signal, wired to
  `_on_statement_reassigned` → `tr("Statement account changed")` via a shared
  `_refresh_after_statement_change` helper (extracted from the delete handler).
- **`AccountPickerDialog`** (new `ui/account_picker.py`) — a small `QDialog`
  preselecting the current account; the Change-account button on the Statements
  tab (both buttons now gated on selection). `deleteLater`'d after use (review
  fold). `VaultLockedError` caught (auto-lock mid-move); the collision `ValueError`
  shown as a widget-composed `tr()` warning, not the raw exception.
- **`StatementRow` += `account_id`** (after `account_name`; `list_statements`'
  `SELECT`/`GROUP BY` extended); repos `get()` / `set_account()` /
  `reassign_account()` (all commit-free). **No schema change** — reuses the v6
  provenance stamp; `LATEST_SCHEMA_VERSION` stays 6.

## What was learned

- **Reusing a signal is a trap when its handler bakes in a message.** The obvious
  "emit `changed`, get the refresh for free" would have reported "Statement
  deleted" on a successful move — cold-eyes loop 2 caught it. A distinct signal +
  a shared refresh helper is the honest split.
- **The span-guard self-exclusion is load-bearing.** Without `!= period_id`, the
  guard false-fires on the same-account no-op (loop-1 CRITICAL). Per-account
  span-uniqueness (FIBR-0007) makes `id_for_span`'s `LIMIT 1` unambiguous, so the
  guard reads exactly one candidate.
- **Move verbatim, don't re-dedup.** A statement is a coherent unit; deduping on
  move could silently drop its own rows against a coincidental target manual entry.
  The span guard prevents the *systematic* duplication (a whole same-span
  statement); a lone coincidental dup is the user's to resolve.
